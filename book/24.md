## Chapter 24: Data Serialization & Transformation

Every real system eventually has to hand data to some *other* system, and the two rarely speak the same format. A REST API speaks JSON. A legacy billing platform speaks XML. A DevOps pipeline speaks YAML and `.env` files. A data warehouse wants flat CSV. An embedded sensor wants compact binary. A sysadmin migrating a fifteen-year-old application still finds `.ini` files sitting next to brand-new `pyproject.toml` files in the same repository. None of these formats is "correct" — each was chosen for the constraints of its era and its audience — and the practical job of a Python developer or sysadmin is rarely to pick a winner. It is to move data safely and losslessly from one shape into another, over and over, usually inside a script that runs unattended at 3 a.m.

This chapter is a working toolkit for exactly that job. Each of the twenty-three scripts is a small, self-contained command-line utility that does one conversion or one serialization task well, with sane defaults, a built-in demo so you can see it work with zero setup, and `argparse` flags so you can point it at real files in a real pipeline. None of them require a framework or a config file of their own — you run `python3 script.py`, and it either transforms what's in front of it or tells you clearly why it couldn't.

#### Why serialization deserves its own chapter

"Serialization" sounds abstract, but it is one of the most concrete things a script ever does: turning an in-memory Python object graph into a sequence of bytes (or characters) that can survive being written to disk, sent across a network, or handed to a program written in a completely different language — and then turning those bytes back into an equivalent object graph later. Get it right and a JSON file written today is still readable in ten years by software that doesn't exist yet. Get it wrong — silently drop a data type JSON can't express, mismatch character encodings, or trust a binary format you shouldn't — and you get corrupted records, mojibake in production logs, or in the worst case (see `pickle_serialize.py`) a remote code execution vulnerability sitting quietly in your ingestion pipeline.

The scripts here fall into a few natural groups, and it helps to think of them that way rather than as twenty-three unrelated tools:

**Text-based interchange formats.** JSON, YAML, XML, CSV, TOML, and INI are the formats humans and machines both read. Scripts 24.1 through 24.13 (minus the binary ones) convert between these — JSON to CSV, CSV to JSON, JSON to YAML and back, XML to JSON and back, INI and TOML into modern JSON/YAML — because in practice you're rarely handed data in the one format your next tool expects.

**Binary and byte-safe formats.** Pickle and MessagePack (24.14–24.15) trade human-readability for speed, size, and (in Pickle's case) the ability to represent almost any Python object. Base64 and URL encoding (24.16–24.17) solve a narrower but constant problem: how do you safely embed arbitrary bytes inside a text-only channel like a URL or a JSON string field.

**Structural transformation.** Flattening and merging (24.19–24.20) don't change the format so much as reshape the data itself — collapsing nested trees into flat rows for a spreadsheet, or deep-merging several config files into one effective configuration, which is exactly what production deployment tooling does with base + environment-specific overrides.

**Querying, validating, and scaling.** The jq-style query script (24.11) lets you pull one field out of a JSON haystack without writing a parser by hand. The JSON Schema validator (24.21) catches malformed data before it reaches code that assumes it's well-formed. And the streaming script (24.23) addresses the question every one of the other scripts quietly ignores: what happens when the file is 40 GB and doesn't fit in RAM?


### 24.1 Serialize Python Objects to JSON


#### Use case

Whenever a Python service needs to write output another system will consume — an API response, a log record, a message queued for a worker — JSON is the default interchange format. A sysadmin scripting a nightly export of application state (users, jobs, config snapshots) needs a serializer that doesn't choke the moment the data includes a `datetime` or a custom class, which is exactly where `json.dumps()` fails out of the box.

#### Prerequisites

Standard library only — no installation required.

```bash
python3 --version   # 3.8+ recommended; f-strings and dataclasses are used
```

OS scope: Linux, macOS, Windows — identical behavior everywhere since it touches no filesystem-specific APIs beyond plain file I/O.

#### Cautions

`json.dumps()` raises `TypeError` on anything it doesn't recognize (datetimes, sets, custom objects) — this script's custom encoder papers over that, but only for the types it explicitly handles. Anything else will still fail loudly, which is the correct behavior; silently stringifying unknown types hides bugs.

#### Script

```python
#!/usr/bin/env python3
"""
json_serialize.py
------------------
Topic 337: Serialize Python Objects to JSON

Converts Python dictionaries, lists, and custom objects (dataclasses /
plain classes with __dict__) into a JSON string, and optionally writes
the result to a file.

USAGE
    # Run built-in demo (no arguments needed)
    python3 json_serialize.py

    # Serialize a custom "input" python literal file (a .py file that
    # defines a variable called DATA) and write JSON to output.json
    python3 json_serialize.py --input mydata.py --output out.json

    # Pretty print with custom indent
    python3 json_serialize.py --indent 4

EXPECTED OUTPUT
    A JSON string printed to the terminal (and written to --output
    if given). The demo serializes a dict, a list, a dataclass
    instance and a datetime object.
"""
import argparse
import importlib.util
import json
import sys
from dataclasses import asdict, is_dataclass, dataclass
from datetime import datetime
from pathlib import Path


@dataclass
class Employee:
    name: str
    role: str
    hired: datetime


class DemoJSONEncoder(json.JSONEncoder):
    """Encoder that knows how to serialize dataclasses, datetimes and
    generic objects that expose __dict__."""

    def default(self, obj):
        if is_dataclass(obj):
            return asdict(obj)
        if isinstance(obj, datetime):
            return obj.isoformat()
        if hasattr(obj, "__dict__"):
            return obj.__dict__
        return super().default(obj)


def build_demo_data():
    return {
        "company": "Acme Corp",
        "employees": [
            Employee("Alice", "Engineer", datetime(2022, 3, 1)),
            Employee("Bob", "Designer", datetime(2023, 7, 15)),
        ],
        "tags": ["python", "json", "serialization"],
        "active": True,
        "headcount": 2,
    }


def load_input_module(path: str):
    """Load a .py file and return its DATA attribute."""
    spec = importlib.util.spec_from_file_location("user_input", path)
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    if not hasattr(module, "DATA"):
        sys.exit(f"ERROR: {path} must define a top-level variable named DATA")
    return module.DATA


def main():
    parser = argparse.ArgumentParser(description="Serialize Python objects to JSON")
    parser.add_argument("--input", help="Path to a .py file defining a DATA variable")
    parser.add_argument("--output", help="Path to write the resulting JSON file")
    parser.add_argument("--indent", type=int, default=2, help="Indent width (default: 2)")
    args = parser.parse_args()

    data = load_input_module(args.input) if args.input else build_demo_data()

    json_text = json.dumps(data, cls=DemoJSONEncoder, indent=args.indent)
    print(json_text)

    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"\n[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The core trick is `DemoJSONEncoder.default()`. `json.dumps()` calls this method exactly when it hits a value it doesn't natively know how to encode (i.e., not a dict/list/str/int/float/bool/None). The encoder checks, in order, whether the value is a dataclass (converts with `asdict()`), a `datetime` (converts with `.isoformat()`), or any other object exposing `__dict__` (uses that dict directly) — and only falls through to `super().default(obj)`, which raises `TypeError`, if none of those match.

```bash
# Built-in demo
python3 json_serialize.py

# Custom indent width
python3 json_serialize.py --indent 4

# Serialize your own data: create a .py file that defines DATA
cat > mydata.py << 'EOF'
DATA = {"host": "db01", "port": 5432, "tags": ["prod", "primary"]}
EOF
python3 json_serialize.py --input mydata.py --output out.json
```

#### Sample output

```

{
  "company": "Acme Corp",
  "employees": [
    {
      "name": "Alice",
      "role": "Engineer",
      "hired": "2022-03-01T00:00:00"
    },
    {
      "name": "Bob",
      "role": "Designer",
      "hired": "2023-07-15T00:00:00"
    }

```

Note that the `Employee` dataclass instances become plain JSON objects, and the `datetime` fields become ISO-8601 strings (`2022-03-01T00:00:00`) — this round-trips cleanly with `json_deserialize.py` (24.2), though the result on the way back will be a plain string, not a `datetime` object, unless you add explicit parsing.

#### Variations / flags

- `--indent 0` produces JSON with newlines but no indentation; omit `--indent` entirely and pass `compact=True`-style separators in your own fork for single-line minified output (`json.dumps(data, separators=(",", ":"))`).
- Extend `DemoJSONEncoder.default()` with an `isinstance(obj, set)` branch (`return list(obj)`) if your data includes sets — the base script deliberately doesn't handle sets so `pickle_serialize.py` (24.14) remains the natural home for object types JSON truly can't represent.
- Pipe output straight into `jq` or `jq_query.py` (24.11) for a one-line "serialize and query" chain: `python3 json_serialize.py | python3 jq_query.py --query ".employees[*].name"` (requires piping into `--json` since `jq_query.py` reads from `--json`/`--input`, not stdin, by default — redirect via a temp file or extend it with stdin support).

#### Common pitfalls

- **`TypeError: Object of type X is not JSON serializable`** — this is the encoder correctly refusing to guess; add a branch to `DemoJSONEncoder.default()` rather than suppressing the error.
- **Timezone-naive datetimes** — `datetime(2022, 3, 1)` has no timezone, so `.isoformat()` produces `2022-03-01T00:00:00` with no offset; if your data crosses timezones, use timezone-aware `datetime` objects (`datetime.now(timezone.utc)`) or the output will silently lose that information.
- **`--input` module has side effects** — `load_input_module()` executes the `.py` file with `exec_module()`; never point `--input` at a file you don't trust, since this is arbitrary code execution by design.

### 24.2 Deserialize JSON to Python Objects

#### Use case

A sysadmin pulling a JSON API response or reading a JSON config/log file needs to load it into Python and quickly understand its shape (how many keys, how deep, what's a list vs. an object) before writing code against it — this script does both the parse and a structural summary in one step.

#### Prerequisites

Standard library only.

```bash
python3 --version   # any Python 3.x
```

OS scope: Linux, macOS, Windows — no platform-specific code.

#### Cautions

`json.loads()` is safe to call on untrusted input (unlike `pickle.load()` — see 24.14); the risk with JSON is nesting depth on adversarial input degrading performance or hitting Python's default recursion limit, not code execution.

#### Script

```python
#!/usr/bin/env python3
"""
json_deserialize.py
--------------------
Topic 338: Deserialize JSON to Python Objects

Parses a JSON string / file back into native Python data structures
(dict, list, str, int, float, bool, None) and shows how to walk the
resulting structure.

USAGE
    # Run built-in demo (no arguments needed)
    python3 json_deserialize.py

    # Deserialize a specific JSON file
    python3 json_deserialize.py --input data.json

    # Deserialize a raw JSON string passed on the command line
    python3 json_deserialize.py --json '{"a": 1, "b": [1,2,3]}'

EXPECTED OUTPUT
    The parsed Python object printed with repr(), its type, and (for
    dict/list top level) a short structural summary.
"""
import argparse
import json
import sys
from pathlib import Path

DEMO_JSON = """
{
  "company": "Acme Corp",
  "employees": [
    {"name": "Alice", "role": "Engineer", "hired": "2022-03-01"},
    {"name": "Bob", "role": "Designer", "hired": "2023-07-15"}
  ],
  "tags": ["python", "json", "serialization"],
  "active": true,
  "headcount": 2,
  "notes": null
}
"""


def summarize(obj, depth=0):
    prefix = "  " * depth
    if isinstance(obj, dict):
        print(f"{prefix}dict with {len(obj)} keys: {list(obj.keys())}")
        for k, v in obj.items():
            if isinstance(v, (dict, list)):
                print(f"{prefix}  - {k}:")
                summarize(v, depth + 2)
    elif isinstance(obj, list):
        print(f"{prefix}list with {len(obj)} items")
        if obj and isinstance(obj[0], (dict, list)):
            summarize(obj[0], depth + 1)


def main():
    parser = argparse.ArgumentParser(description="Deserialize JSON into Python objects")
    parser.add_argument("--input", help="Path to a JSON file")
    parser.add_argument("--json", help="Raw JSON string")
    args = parser.parse_args()

    if args.json:
        raw = args.json
    elif args.input:
        raw = Path(args.input).read_text(encoding="utf-8")
    else:
        raw = DEMO_JSON

    try:
        obj = json.loads(raw)
    except json.JSONDecodeError as exc:
        sys.exit(f"ERROR: invalid JSON input: {exc}")

    print(f"Top-level Python type: {type(obj).__name__}\n")
    print("repr():")
    print(repr(obj))
    print("\nStructure summary:")
    summarize(obj)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`json.loads()` maps JSON `object` → `dict`, `array` → `list`, `string` → `str`, `number` → `int`/`float`, `true`/`false` → `bool`, and `null` → `None`. The `summarize()` helper then recurses only into nested dicts/lists so you get a quick outline of the document without dumping every leaf value.

```bash
# Built-in demo
python3 json_deserialize.py

# From a file
python3 json_deserialize.py --input data.json

# Inline JSON string (handy in one-liners / cron jobs)
python3 json_deserialize.py --json '{"a": 1, "b": [1, 2, 3]}'
```

#### Sample output

```
$ python3 json_deserialize.py
Top-level Python type: dict

repr():
{'company': 'Acme Corp', 'employees': [{'name': 'Alice', 'role': 'Engineer', 'hired': '2022-03-01'}, {'name': 'Bob', 'role': 'Designer', 'hired': '2023-07-15'}], 'tags': ['python', 'json', 'serialization'], 'active': True, 'headcount': 2, 'notes': None}

Structure summary:
dict with 6 keys: ['company', 'employees', 'tags', 'active', 'headcount', 'notes']
  - employees:
    list with 2 items
      dict with 3 keys: ['name', 'role', 'hired']
  - tags:
    list with 3 items
```

Note `"hired": "2022-03-01"` stays a plain `str` — JSON has no native date type, so any date/datetime parsing is your responsibility after deserializing (pair with `datetime.fromisoformat()` if needed).

#### Variations / flags

- Add `object_hook=` to `json.loads()` to convert specific shapes (e.g. ISO date strings) into richer Python objects automatically as they're parsed, instead of post-processing.
- For very large files, don't use this script as-is (it reads the whole file into memory) — use `stream_large_dataset.py` (24.23) instead.
- Combine with `jq_query.py` (24.11) when you only need one field rather than the whole structural summary.

#### Common pitfalls

- **`json.JSONDecodeError`** on trailing commas or single quotes — JSON is stricter than Python literals; `{'a': 1,}` is invalid JSON even though it's valid-ish Python.
- **Large integers losing precision** — JSON numbers map to Python `int`/`float`; a 64-bit ID that overflows `float` precision on the *producing* side (e.g. some JavaScript emitters) will already be wrong before Python ever sees it — verify at the source, not after `json.loads()`.
- **Shell quoting with `--json`** — on Windows `cmd.exe`, wrapping JSON in single quotes doesn't work the way it does in bash; use PowerShell with escaped double quotes or read from a file instead.

### 24.3 Convert JSON to CSV

#### Use case

Business users and spreadsheet tools want CSV, not JSON. A sysadmin exporting API results, log records, or database dumps to hand off to a non-technical team (or import into Excel/Google Sheets) needs nested JSON collapsed into flat columns first.

#### Prerequisites

Standard library only (`csv`, `json`, `io`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows. Note: CSV line endings are written as `\r\n` by Python's `csv` module by default regardless of OS, which is correct per RFC 4180 but can look odd if you `cat` the file on Linux (see pitfalls).

#### Cautions

Column order depends on first-seen key order across all rows; if row 2 has a field row 1 doesn't, that column still gets added, but earlier rows will show as blank for it — no data is lost, but skim the header row before trusting column meaning.

#### Script

```python
#!/usr/bin/env python3
"""
json_to_csv.py
---------------
Topic 339: Convert JSON to CSV

Flattens a JSON array of (possibly nested) objects into flat CSV rows.
Nested keys are joined with a dot, e.g. {"a": {"b": 1}} -> column "a.b".

USAGE
    # Run built-in demo (no arguments needed) -> prints CSV to stdout
    python3 json_to_csv.py

    # Convert a specific file
    python3 json_to_csv.py --input data.json --output data.csv

EXPECTED OUTPUT
    A CSV table printed to the terminal (and written to --output if
    given), one row per JSON object, one column per flattened field.
"""
import argparse
import csv
import io
import json
import sys
from pathlib import Path

DEMO_DATA = [
    {
        "id": 1,
        "name": "Alice",
        "address": {"city": "Lahore", "zip": "54000"},
        "skills": ["python", "sql"],
    },
    {
        "id": 2,
        "name": "Bob",
        "address": {"city": "Karachi", "zip": "74000"},
        "skills": ["go"],
    },
]


def flatten(obj, parent_key="", sep="."):
    items = {}
    if isinstance(obj, dict):
        for k, v in obj.items():
            new_key = f"{parent_key}{sep}{k}" if parent_key else str(k)
            items.update(flatten(v, new_key, sep))
    elif isinstance(obj, list):
        # join list of scalars with ';' ; lists of dicts get index-expanded
        if obj and isinstance(obj[0], (dict, list)):
            for i, v in enumerate(obj):
                items.update(flatten(v, f"{parent_key}[{i}]", sep))
        else:
            items[parent_key] = ";".join(str(v) for v in obj)
    else:
        items[parent_key] = obj
    return items


def main():
    parser = argparse.ArgumentParser(description="Convert JSON array to flat CSV")
    parser.add_argument("--input", help="Path to a JSON file (array of objects)")
    parser.add_argument("--output", help="Path to write the CSV file")
    args = parser.parse_args()

    if args.input:
        data = json.loads(Path(args.input).read_text(encoding="utf-8"))
    else:
        data = DEMO_DATA

    if isinstance(data, dict):
        data = [data]
    if not isinstance(data, list):
        sys.exit("ERROR: input JSON must be an object or an array of objects")

    flat_rows = [flatten(row) for row in data]
    fieldnames = []
    for row in flat_rows:
        for k in row:
            if k not in fieldnames:
                fieldnames.append(k)

    buf = io.StringIO()
    writer = csv.DictWriter(buf, fieldnames=fieldnames)
    writer.writeheader()
    writer.writerows(flat_rows)
    csv_text = buf.getvalue()

    print(csv_text)
    if args.output:
        Path(args.output).write_text(csv_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`flatten()` walks the object recursively: nested dicts get dot-joined keys (`address.city`), lists of scalars get semicolon-joined into a single cell (`"python;sql"`), and lists of dicts/lists get index-expanded (`skills[0]`, `skills[1]`) so structure isn't silently lost. A single JSON object (not wrapped in an array) is automatically treated as a one-row table.

```bash
# Built-in demo
python3 json_to_csv.py

# Real file, written to disk
python3 json_to_csv.py --input employees.json --output employees.csv
```

#### Sample output

```
$ python3 json_to_csv.py
id,name,address.city,address.zip,skills
1,Alice,Lahore,54000,python;sql
2,Bob,Karachi,74000,go
```

#### Variations / flags

- Add a `--sep` flag (mirroring `flatten_json.py`, 24.19) to change `.` to `_` for column names that must be valid SQL identifiers.
- Add a `--delimiter` flag piping into `csv.DictWriter(buf, fieldnames=fieldnames, delimiter=';')` for European-locale CSV (comma is the decimal separator in many locales, so semicolon-delimited CSV is common there).
- Chain with `csv_to_xml.py` (24.9) for a JSON → CSV → XML pipeline when a downstream system only accepts XML but you want a CSV audit trail in between.

#### Common pitfalls

- **Ragged fields across rows** — if some objects in the array have keys others lack, `csv.DictWriter` leaves those cells blank; it does not error, so double check nothing was silently mis-typed upstream.
- **Lists of scalars lose their delimiter semantics** — `"python;sql"` round-trips back to a single string, not a list, unless the consumer specifically knows to split on `;`; if your skills list might itself contain a semicolon, this is lossy — pick a separator that can't appear in your data, or index-expand instead.
- **`\r\n` line endings** — Python's `csv` module writes `\r\n` per RFC 4180 even on Linux; this is correct for spreadsheet tools but will show as `^M` if you `cat -A` the file — not a bug, just be aware when diffing output.


### 24.4 Convert CSV to JSON

#### Use case

CSV exports from spreadsheets, databases, or legacy systems are the most common data-handoff format a sysadmin receives; most modern APIs and services expect JSON, so this is one of the most frequently needed conversions in any onboarding/migration script.

#### Prerequisites

Standard library only (`csv`, `json`, `io`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows. Watch for encoding differences on files exported from Excel on Windows, which sometimes default to `cp1252`/`latin-1` rather than UTF-8 (see pitfalls).

#### Cautions

Type coercion (`coerce()`) is a convenience, not a schema — a ZIP code column like `"00501"` will silently become the integer `501`, dropping the leading zero. Inspect coercion behavior against your actual data before trusting the output for anything identifier-like.

#### Script

```python
#!/usr/bin/env python3
"""
csv_to_json.py
---------------
Topic 340: Convert CSV to JSON

Transforms tabular CSV data into a JSON array of objects. Column
names containing a dot (e.g. "address.city") are automatically
nested back into sub-objects. Numeric-looking values are converted
to int/float, "true"/"false" to booleans.

USAGE
    # Run built-in demo (no arguments needed)
    python3 csv_to_json.py

    # Convert a specific file
    python3 csv_to_json.py --input data.csv --output data.json

EXPECTED OUTPUT
    A pretty-printed JSON array printed to the terminal (and written
    to --output if given).
"""
import argparse
import csv
import io
import json
import sys
from pathlib import Path

DEMO_CSV = """id,name,address.city,address.zip,active
1,Alice,Lahore,54000,true
2,Bob,Karachi,74000,false
"""


def coerce(value: str):
    if value == "":
        return None
    low = value.lower()
    if low == "true":
        return True
    if low == "false":
        return False
    try:
        if "." in value:
            return float(value)
        return int(value)
    except ValueError:
        return value


def nest(flat_row: dict) -> dict:
    result = {}
    for key, value in flat_row.items():
        parts = key.split(".")
        d = result
        for part in parts[:-1]:
            d = d.setdefault(part, {})
        d[parts[-1]] = coerce(value)
    return result


def main():
    parser = argparse.ArgumentParser(description="Convert CSV to nested JSON")
    parser.add_argument("--input", help="Path to a CSV file")
    parser.add_argument("--output", help="Path to write the JSON file")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_CSV
    reader = csv.DictReader(io.StringIO(text))
    rows = [nest(row) for row in reader]

    if not rows:
        sys.exit("ERROR: no rows found in CSV input")

    json_text = json.dumps(rows, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`csv.DictReader` turns each row into a flat dict keyed by header name. `nest()` then splits each key on `.` and walks/creates nested dicts (`address.city` → `{"address": {"city": ...}}`) — the exact inverse of `json_to_csv.py`'s flattening (24.3). `coerce()` runs on every value: empty string → `None`, `"true"`/`"false"` (case-insensitive) → `bool`, anything parseable as a number → `int` or `float`, otherwise left as a plain string.

```bash
# Built-in demo
python3 csv_to_json.py

# Real file
python3 csv_to_json.py --input export.csv --output export.json --indent 4
```

#### Sample output

```
$ python3 csv_to_json.py
[
  {
    "id": 1,
    "name": "Alice",
    "address": {
      "city": "Lahore",
      "zip": 54000
    },
    "active": true
  },
  {
    "id": 2,
    "name": "Bob",
    "address": {
      "city": "Karachi",
      "zip": 74000
    },
    "active": false
  }
]
```

Notice `address.zip` became the *integer* `54000` — this is the coercion behavior called out in Cautions above; for a real ZIP-code column you'd want to keep it a string.

#### Variations / flags

- Add a `--no-coerce` flag that skips `coerce()` and keeps every value as a raw string, for cases (ZIP codes, phone numbers, IDs with leading zeros) where type inference is actively harmful.
- Add `--sep` to match a custom nesting separator produced by `flatten_json.py` (24.19) if you changed it from `.` there.
- Pipe the result into `validate_json_schema.py` (24.21) to confirm the coerced types actually match what your downstream system expects, since silent coercion is exactly the kind of thing schema validation is meant to catch.

#### Common pitfalls

- **Leading zeros disappear** — as shown above, `"00501"` becomes `501`; this is the single most common CSV→JSON bug reported in production ETL scripts. Fix at the source (quote the column as text in the exporting tool) or patch `coerce()` to skip fields matching a "looks like an ID" pattern.
- **Non-UTF-8 CSV from Excel on Windows** — `Path(args.input).read_text(encoding="utf-8")` will raise `UnicodeDecodeError` on `cp1252`-encoded exports; pass `encoding="cp1252"` (or better, run `encoding_convert.py`, 24.18, first to normalize to UTF-8).
- **Empty CSV file** — the script exits cleanly with `ERROR: no rows found in CSV input` rather than emitting `[]` silently; if your pipeline expects an empty array as valid output, adjust that `sys.exit()` call.

### 24.5 Convert JSON to YAML

#### Use case

Config-management tools (Kubernetes, Docker Compose, Ansible, GitHub Actions) all use YAML because it's easier for a human to read and edit than JSON. A sysadmin generating config programmatically (e.g. templating a Kubernetes manifest from a JSON API response) needs it as YAML for the final file, not JSON.

#### Prerequisites

Requires PyYAML.

```bash
pip install pyyaml --break-system-packages
# or, inside a virtual environment:
python3 -m venv venv && source venv/bin/activate && pip install pyyaml
```

OS scope: Linux, macOS, Windows — PyYAML is a pure-Python-plus-C-extension package with prebuilt wheels for all three.

#### Cautions

`yaml.dump()` here uses `default_flow_style=False` for block-style output (the readable style), and `sort_keys=False` to preserve insertion order — if you swap in `yaml.safe_dump()` elsewhere in your own scripts, note that plain `yaml.dump()` (not `safe_dump`) can serialize arbitrary Python objects using YAML's `!!python/object` tags, which is unsafe to load back with `yaml.load()` on untrusted input. This script's `main()` doesn't use `safe_dump` for output (dumping is safe either way) but always use `yaml.safe_load()`, never `yaml.load()`, when reading YAML back.

#### Script

```python
#!/usr/bin/env python3
"""
json_to_yaml.py
-----------------
Topic 341: Convert JSON to YAML

Translates JSON data into human-readable YAML, suitable for
configuration files. Requires PyYAML (see requirements.txt).

USAGE
    python3 json_to_yaml.py
    python3 json_to_yaml.py --input data.json --output config.yaml

EXPECTED OUTPUT
    YAML text printed to the terminal (and written to --output if
    given).
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import yaml
except ImportError:
    sys.exit(
        "ERROR: PyYAML is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_DATA = {
    "server": {
        "host": "0.0.0.0",
        "port": 8080,
        "debug": False,
    },
    "database": {
        "engine": "postgresql",
        "name": "app_db",
    },
    "allowed_hosts": ["localhost", "127.0.0.1", "example.com"],
}


def main():
    parser = argparse.ArgumentParser(description="Convert JSON to YAML")
    parser.add_argument("--input", help="Path to a JSON file")
    parser.add_argument("--output", help="Path to write the YAML file")
    args = parser.parse_args()

    if args.input:
        data = json.loads(Path(args.input).read_text(encoding="utf-8"))
    else:
        data = DEMO_DATA

    yaml_text = yaml.dump(data, sort_keys=False, default_flow_style=False)
    print(yaml_text)
    if args.output:
        Path(args.output).write_text(yaml_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`json.loads()` produces the same plain dict/list/str/int/bool structures whether the source was JSON or Python literals, so `yaml.dump()` needs no special adapter — it walks the same structure JSON already gave us. `default_flow_style=False` forces YAML's indented block style (`key:\n  subkey: value`) rather than the compact `{key: value}` flow style, which is what config-file conventions expect.

```bash

python3 json_to_yaml.py

# Convert a real API response saved to disk
python3 json_to_yaml.py --input response.json --output response.yaml
```

#### Sample output

```
- localhost
- 127.0.0.1

```

#### Variations / flags

- Add `--flow` to switch to `default_flow_style=True` for compact single-line YAML when embedding small snippets inline elsewhere.
- Add `width=` to `yaml.dump()` if long string values are being hard-wrapped in a way that breaks your downstream YAML consumer (PyYAML defaults to wrapping at 80 columns).
- Chain with `merge_configs.py` (24.20): generate several JSON fragments, convert each to YAML, and merge them as the final deployment config.

#### Common pitfalls

- **`ModuleNotFoundError: No module named 'yaml'`** — install with `pip install pyyaml`; note the *package* name is `pyyaml` but the *import* name is `yaml` — a frequent point of confusion.
- **`0.0.0.0` printed unquoted** — PyYAML's default dumper is generally correct about not quoting things that don't need it, but always spot-check values that look like other YAML types (`yes`/`no`/`on`/`off` as strings, version strings like `"1.0"` that could be misread as floats) since YAML 1.1 parsers are famously inconsistent about implicit typing (the "Norway problem": `NO` parsing as boolean `false`).
- **Key order matters to some consumers** — `sort_keys=False` preserves the JSON's original key order; if you need alphabetical order instead (some diff tools prefer it), pass `sort_keys=True`.

### 24.6 Convert YAML to JSON

#### Use case

Kubernetes manifests, Ansible playbooks, and CI pipeline definitions are written in YAML, but some downstream tooling — a JSON Schema validator, a JavaScript frontend, a `jq` pipeline — only speaks JSON. A sysadmin scripting a config-validation step (validate a Helm-rendered manifest against a JSON Schema, for example) needs this conversion as a first step.

This script was not provided in the original upload and was written from scratch for this chapter using the same conventions (argparse, demo data, `pathlib`) as the rest — it has been executed against its own demo and against a deliberately malformed file to confirm both the success and failure paths.

#### Prerequisites

Requires PyYAML.

```bash
pip install pyyaml --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

Uses `yaml.safe_load()`, not `yaml.load()` — this is a deliberate security choice. `yaml.load()` without a `Loader=` argument (or with `Loader=yaml.Loader`) can be tricked into constructing arbitrary Python objects (including calling functions) via YAML's `!!python/object/apply` tag if the source YAML is untrusted. Always parse YAML from an unknown source with `safe_load()`.

#### Script

```python
#!/usr/bin/env python3
"""
yaml_to_json.py
------------------
Topic 342: Convert YAML to JSON

Translates human-written YAML (config files, Kubernetes manifests,
CI pipelines) into strict JSON that any downstream tool can parse.
Requires PyYAML (see requirements.txt).

USAGE
    python3 yaml_to_json.py
    python3 yaml_to_json.py --input config.yaml --output config.json

EXPECTED OUTPUT
    Pretty-printed JSON printed to the terminal (and written to
    --output if given).
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import yaml
except ImportError:
    sys.exit(
        "ERROR: PyYAML is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_YAML = """
server:
  host: 0.0.0.0
  port: 8080
  debug: false
database:
  engine: postgresql
  name: app_db
allowed_hosts:
  - localhost
  - 127.0.0.1
  - example.com
"""


def main():
    parser = argparse.ArgumentParser(description="Convert YAML to JSON")
    parser.add_argument("--input", help="Path to a YAML file")
    parser.add_argument("--output", help="Path to write the JSON file")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_YAML

    try:
        data = yaml.safe_load(text)
    except yaml.YAMLError as exc:
        sys.exit(f"ERROR: invalid YAML input: {exc}")

    json_text = json.dumps(data, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`yaml.safe_load()` parses the YAML document into the same plain Python types `json.loads()` would produce (dict, list, str, int, float, bool, `None`), so handing that straight to `json.dumps()` "just works" with no adapter layer needed — the two libraries agree on how to represent data in memory even though their text formats differ.

#### Variations / flags

- Add `--multi-doc` using `yaml.safe_load_all()` to handle files with multiple `---`-separated YAML documents (common in Kubernetes manifests that define several resources in one file) and emit a JSON array of documents.
- Chain directly with `validate_json_schema.py` (24.21): `python3 yaml_to_json.py --input config.yaml --output config.json && python3 validate_json_schema.py --schema schema.json --data config.json`.
- For round-tripping, pair with `json_to_yaml.py` (24.5) — note a full round trip is *not* always byte-identical (YAML comments and formatting are lost) but is always structurally identical.

#### Common pitfalls

- **The "Norway problem"** — bare `no`, `yes`, `on`, `off`, `y`, `n` (unquoted) are interpreted as booleans by YAML 1.1 parsers including PyYAML's default resolver; a country code column with `NO` (Norway) silently becomes `false`. Always quote such values in the source YAML (`"NO"`).
- **Tabs are illegal in YAML indentation** — unlike JSON, YAML forbids tab characters for indentation; a file edited with a tab-inserting editor will fail to parse with a `YAMLError` that can be non-obvious at first glance.
- **`yaml.load()` vs `yaml.safe_load()`** — if you customize this script and switch to plain `yaml.load()` for convenience (e.g. to get custom tag support), you reopen the arbitrary-code-execution risk described in Cautions; there is essentially never a good reason to do this for config files coming from disk or a network source you don't fully control.

### 24.7 Convert XML to JSON

#### Use case

Legacy enterprise systems (SOAP APIs, old billing platforms, EDI-adjacent feeds) still speak XML. A sysadmin bridging one of these into a modern JSON-based pipeline needs a reliable, attribute-preserving XML→JSON step rather than hand-rolling an `xml.etree.ElementTree` walker for every new document shape.

This script was not provided in the original upload and was written from scratch for this chapter, deliberately mirroring the existing `json_to_xml.py` (24.8) so the two form an exact round-trip pair — verified below.

#### Prerequisites

Requires `xmltodict`.

```bash
pip install xmltodict --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

`xmltodict.parse()` uses Python's built-in `expat` parser under the hood, which does **not** resolve external entities by default in modern Python — so classic XXE (XML External Entity) attacks are not a concern here out of the box. Still, treat XML from an untrusted source with the same caution as any other untrusted structured input: validate its size before parsing, and don't blindly trust deeply nested documents (entity/attribute count limits exist in `expat` but are not infinite).

#### Script

```python
#!/usr/bin/env python3
"""
xml_to_json.py
-----------------
Topic 343: Convert XML to JSON

Parses an XML document into a nested Python dict (via xmltodict)
and serializes that dict as JSON. Attributes are prefixed with "@"
and element text is kept under "#text" when an element has both
attributes and text content.

USAGE
    python3 xml_to_json.py
    python3 xml_to_json.py --input data.xml --output data.json

EXPECTED OUTPUT
    Pretty-printed JSON printed to the terminal (and written to
    --output if given).
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import xmltodict
except ImportError:
    sys.exit(
        "ERROR: xmltodict is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_XML = """<?xml version="1.0"?>
<company name="Acme Corp">
  <employees>
    <employee id="1">
      <name>Alice</name>
      <role>Engineer</role>
    </employee>
    <employee id="2">
      <name>Bob</name>
      <role>Designer</role>
    </employee>
  </employees>
</company>
"""


def main():
    parser = argparse.ArgumentParser(description="Convert XML to JSON")
    parser.add_argument("--input", help="Path to an XML file")
    parser.add_argument("--output", help="Path to write the JSON file")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_XML

    try:
        data = xmltodict.parse(text)
    except Exception as exc:
        sys.exit(f"ERROR: could not parse XML: {exc}")

    json_text = json.dumps(data, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`xmltodict.parse()` converts the XML tree into an `OrderedDict`: element attributes become keys prefixed with `@` (`id="1"` → `"@id": "1"`), child elements become nested keys, and repeated sibling elements with the same tag (like the two `<employee>` elements) automatically become a JSON array rather than overwriting each other — this is the single hardest part of hand-rolling an XML→JSON converter, and `xmltodict` handles it correctly out of the box.

```bash

# Real file
python3 xml_to_json.py --input legacy_export.xml --output legacy_export.json
```

#### Variations / flags

- Add `--root` similar to `json_to_xml.py` (24.8) to strip or rename the single top-level key `xmltodict.parse()` always produces (XML documents always have exactly one root element, so the resulting dict always has exactly one top-level key).
- Pass `attr_prefix=""` and `cdata_key="text"` to `xmltodict.parse()` if you want cleaner key names and don't need to distinguish attributes from child elements downstream (note this makes the conversion lossy if an element has both an attribute and a child with the same name).
- Chain with `flatten_json.py` (24.19) to turn deeply nested SOAP-style XML into flat key-value pairs suitable for a spreadsheet or a wide database table.

#### Common pitfalls

- **A single root requirement, always** — well-formed XML always has exactly one top-level element, so the output dict always has exactly one top-level key; scripts downstream that expect a "normal" flat JSON object need to unwrap that one key first (`data["company"]`, in the demo).
- **Single vs. repeated child elements behave differently** — if there had been only *one* `<employee>` element instead of two, `xmltodict` would produce a dict, not a list, for `"employee"` — a classic "works with 2 rows, breaks with 1 row" bug. Defensive code should always do `employees = data["company"]["employees"]["employee"]; if isinstance(employees, dict): employees = [employees]`.
- **Malformed XML fails loudly, which is correct** — an unclosed tag (`<a><b></a>`) raises a parse error caught by the `except Exception` block and reported clearly; don't be tempted to catch and ignore this, since silently skipping malformed XML hides real data-quality problems upstream.

### 24.8 Convert JSON to XML

#### Use case

Modern services produce JSON, but plenty of enterprise integrations (SOAP endpoints, older ERP/CRM systems, some government/EDI interfaces) still require XML input. A sysadmin building an integration adapter needs a dependable JSON→XML step that correctly handles attributes and repeated elements, not just a naive recursive tag-wrapper.

#### Prerequisites

Requires `xmltodict`.

```bash
pip install xmltodict --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

XML requires exactly one root element; the script enforces this and exits with a clear error rather than guessing which of several top-level JSON keys should become the root. Plan your JSON shape (or use `--root`) with that constraint in mind.

#### Script

```python
#!/usr/bin/env python3
"""
json_to_xml.py
-----------------
Topic 344: Convert JSON to XML

Builds an XML document from a JSON data structure using xmltodict's
unparse(). The JSON must have a single root key (XML requires exactly
one root element) -- the demo and README show the expected shape.

USAGE
    python3 json_to_xml.py
    python3 json_to_xml.py --input data.json --output data.xml --root company

EXPECTED OUTPUT
    An XML document printed to the terminal (and written to --output
    if given).
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import xmltodict
except ImportError:
    sys.exit(
        "ERROR: xmltodict is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_DATA = {
    "company": {
        "@name": "Acme Corp",
        "employees": {
            "employee": [
                {"@id": "1", "name": "Alice", "role": "Engineer"},
                {"@id": "2", "name": "Bob", "role": "Designer"},
            ]
        },
    }
}


def main():
    parser = argparse.ArgumentParser(description="Convert JSON to XML")
    parser.add_argument("--input", help="Path to a JSON file")
    parser.add_argument("--output", help="Path to write the XML file")
    parser.add_argument(
        "--root",
        help="If the JSON has multiple top-level keys, wrap it under this root element name",
    )
    args = parser.parse_args()

    if args.input:
        data = json.loads(Path(args.input).read_text(encoding="utf-8"))
    else:
        data = DEMO_DATA

    if args.root:
        data = {args.root: data}

    if not isinstance(data, dict) or len(data) != 1:
        sys.exit(
            "ERROR: XML requires exactly one root element.\n"
            "Wrap your JSON under a single top-level key, or pass --root NAME."
        )

    try:
        xml_text = xmltodict.unparse(data, pretty=True, indent="  ")
    except Exception as exc:
        sys.exit(f"ERROR: could not build XML: {exc}")

    print(xml_text)
    if args.output:
        Path(args.output).write_text(xml_text, encoding="utf-8")
        print(f"\n[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`xmltodict.unparse()` is the exact inverse of `xmltodict.parse()` (see 24.7): keys prefixed with `@` become XML attributes, a plain nested dict becomes a child element, and a list value becomes multiple sibling elements with the same tag. The script guards against the one real failure mode of this whole approach — multiple top-level keys, which XML cannot represent as siblings without an enclosing root — by checking `len(data) != 1` and either failing clearly or auto-wrapping via `--root`.

```bash

python3 json_to_xml.py

# Real file that already has one root key
python3 json_to_xml.py --input company.json --output company.xml

# Real file with multiple top-level keys, wrapped under a new root
python3 json_to_xml.py --input flat_data.json --root records --output records.xml
```

#### Sample output

```

<?xml version="1.0" encoding="utf-8"?>
<company name="Acme Corp">
  <employees>
    <employee id="1">
      <name>Alice</name>
      <role>Engineer</role>
    </employee>
    <employee id="2">
      <name>Bob</name>
      <role>Designer</role>
    </employee>
  </employees>
</company>
```

#### Variations / flags

- Add `--attr-prefix` and pass it through to `unparse(attr_prefix=...)` if your source JSON used a different attribute-marking convention than `@`.
- Combine with `xslt_transform.py` (24.22): generate XML here, then apply an XSLT stylesheet to render it as HTML for a human-readable report.
- Pipe straight into `xml_to_csv.py` (24.10) for a JSON → XML → CSV chain, useful when a legacy tool in the middle of your pipeline only accepts XML as an intermediate format.

#### Common pitfalls

- **Multiple top-level keys** — the most common failure when adapting this script to new data; a JSON object like `{"a": 1, "b": 2}` has no single root and will exit with a clear error rather than silently picking one key — use `--root wrapper` to fix it.
- **Attribute vs. element ambiguity** — deciding which JSON fields should become XML attributes (`@id`) versus child elements (`<name>`) is a modeling decision you make when shaping the input JSON; `xmltodict` won't guess this for you, so the source data must already use the `@`-prefix convention for anything you want as an attribute.
- **Encoding declaration mismatch** — the output declares `encoding="utf-8"`; if you write the file with a different encoding downstream (e.g. accidentally re-saving as `latin-1`), the XML declaration will lie about the actual byte encoding and break strict XML parsers. Keep declaration and actual file encoding in sync, or use `encoding_convert.py` (24.18) deliberately if you must change it.

### 24.9 Convert CSV to XML

#### Use case

Legacy systems that expect XML batch uploads (older invoicing systems, some government reporting portals, certain SOAP-based B2B integrations) often need data that started life as a spreadsheet export. A sysadmin scripting a nightly upload job needs a dependable CSV→XML step using only the standard library, since installing dependencies on a locked-down production/legacy server is often not an option.

#### Prerequisites

Standard library only (`csv`, `xml.etree.ElementTree`, `xml.dom.minidom`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows — no external dependencies to worry about, which matters more here than in most of this chapter since target systems for this conversion are often old and dependency-restricted.

#### Cautions

Column headers become XML tag names verbatim — a CSV header containing spaces, a leading digit, or XML-reserved characters will produce invalid XML tag names. Sanitize headers before running this script if you don't control the source CSV.

#### Script

```python
#!/usr/bin/env python3
"""
csv_to_xml.py
----------------
Topic 345: Convert CSV to XML

Transforms tabular CSV data into hierarchical XML markup, useful for
feeding legacy systems that expect XML. Uses only the standard
library (csv + xml.etree.ElementTree).

USAGE
    python3 csv_to_xml.py
    python3 csv_to_xml.py --input data.csv --output data.xml \\
        --root records --row record

EXPECTED OUTPUT
    An XML document printed to the terminal (and written to --output
    if given), with one <row-tag> element per CSV row and one child
    element per column.
"""
import argparse
import csv
import io
import sys
from pathlib import Path
from xml.dom import minidom
from xml.etree.ElementTree import Element, SubElement, tostring

DEMO_CSV = """id,name,city,active
1,Alice,Lahore,true
2,Bob,Karachi,false
"""


def csv_to_xml(csv_text: str, root_tag: str, row_tag: str) -> str:
    reader = csv.DictReader(io.StringIO(csv_text))
    root = Element(root_tag)
    for row in reader:
        row_el = SubElement(root, row_tag)
        for key, value in row.items():
            child = SubElement(row_el, key)
            child.text = value
    rough = tostring(root, encoding="unicode")
    return minidom.parseString(rough).toprettyxml(indent="  ")


def main():
    parser = argparse.ArgumentParser(description="Convert CSV to XML")
    parser.add_argument("--input", help="Path to a CSV file")
    parser.add_argument("--output", help="Path to write the XML file")
    parser.add_argument("--root", default="records", help="Root element tag (default: records)")
    parser.add_argument("--row", default="record", help="Per-row element tag (default: record)")
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_CSV
    xml_text = csv_to_xml(text, args.root, args.row)

    print(xml_text)
    if args.output:
        Path(args.output).write_text(xml_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`csv.DictReader` yields one dict per row; the script builds an `xml.etree.ElementTree.Element` tree by hand — one `<record>` `SubElement` per row, one child element per column, named after the CSV header and populated with `child.text = value`. `tostring()` serializes the tree to a single compact line, and `minidom.parseString(...).toprettyxml()` re-parses that and re-emits it with indentation, since `ElementTree` alone doesn't pretty-print in older Python versions.

```bash
# Built-in demo
python3 csv_to_xml.py

# Real file with custom tag names
python3 csv_to_xml.py --input orders.csv --output orders.xml --root orders --row order
```

#### Sample output

```
$ python3 csv_to_xml.py
<?xml version="1.0" ?>
<records>
  <record>
    <id>1</id>
    <name>Alice</name>
    <city>Lahore</city>
    <active>true</active>
  </record>
  <record>
    <id>2</id>
    <name>Bob</name>
    <city>Karachi</city>
    <active>false</active>
  </record>
</records>
```

#### Variations / flags

- Add an `--id-attr` flag that promotes one column (e.g. `id`) to an XML attribute on the row element (`<record id="1">`) instead of a child element, matching the attribute style used in `json_to_xml.py` (24.8).
- Add header sanitization (`re.sub(r'\W|^(?=\d)', '_', key)`) before using column names as tags, to automatically handle CSV headers with spaces or leading digits instead of crashing.
- Chain with `xslt_transform.py` (24.22) to turn the generated XML directly into an HTML report for a human audience.

#### Common pitfalls

- **Invalid XML tag names from raw headers** — a header like `"first name"` or `"2023_total"` becomes an invalid tag and `Element()`/`SubElement()` will raise a `ValueError` immediately; sanitize headers first, or rename the CSV columns at the source.
- **`None`/missing values become the literal text `"None"`** — if a row is missing a trailing column, `csv.DictReader` fills it with `None` by default (via `restval`), and `child.text = None` renders as an empty element (not the string `"None"`) which is usually what you want — but verify this matches your downstream system's expectations for "empty" vs. "missing" fields.
- **XML doesn't have a native boolean type** — `true`/`false` values pass through as plain text exactly as typed in the CSV; if the CSV had inconsistent casing (`True`, `TRUE`, `true`), that inconsistency carries straight into the XML. Normalize case in the CSV or add a coercion step if the consuming system is case-sensitive.

### 24.10 Convert XML to CSV

#### Use case

A sysadmin receiving a nightly XML export from a legacy system (order feeds, inventory dumps, SOAP-response logs saved to disk) but needing to load it into a spreadsheet or a simple CSV-ingesting database import tool needs a lightweight XML→CSV bridge that doesn't require standing up a full XML processing pipeline.

#### Prerequisites

Standard library only (`csv`, `xml.etree.ElementTree`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows.

#### Cautions

This script assumes a flat row shape — each `<record>` (or whatever `--row` names) has only direct child elements with text content, no further nesting. Deeply nested XML will have inner structure silently dropped down to the child element's own `.text`, since the script only reads one level of children (`for child in row_el`). For deeply nested XML, flatten with `xml_to_json.py` (24.7) → `flatten_json.py` (24.19) → CSV instead.

#### Script

```python
#!/usr/bin/env python3
"""
xml_to_csv.py
----------------
Topic 346: Convert XML to CSV

Flattens XML document data into row-based CSV, one row per repeated
child element under a chosen "row tag" (e.g. every <record> under
<records>). Uses only the standard library.

USAGE
    python3 xml_to_csv.py
    python3 xml_to_csv.py --input data.xml --output data.csv --row record

EXPECTED OUTPUT
    A CSV table printed to the terminal (and written to --output if
    given), one column per unique child-tag found across rows.
"""
import argparse
import csv
import io
import sys
from pathlib import Path
from xml.etree.ElementTree import fromstring

DEMO_XML = """<records>
  <record>
    <id>1</id>
    <name>Alice</name>
    <city>Lahore</city>
  </record>
  <record>
    <id>2</id>
    <name>Bob</name>
    <city>Karachi</city>
  </record>
</records>
"""


def main():
    parser = argparse.ArgumentParser(description="Convert XML to CSV")
    parser.add_argument("--input", help="Path to an XML file")
    parser.add_argument("--output", help="Path to write the CSV file")
    parser.add_argument("--row", default="record", help="Tag name of repeated row elements (default: record)")
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_XML

    try:
        root = fromstring(text)
    except Exception as exc:
        sys.exit(f"ERROR: could not parse XML: {exc}")

    rows = root.findall(f".//{args.row}")
    if not rows:
        sys.exit(f"ERROR: no elements named <{args.row}> found in the XML")

    flat_rows = []
    fieldnames = []
    for row_el in rows:
        flat = {}
        for child in row_el:
            flat[child.tag] = child.text or ""
            if child.tag not in fieldnames:
                fieldnames.append(child.tag)
        flat_rows.append(flat)

    buf = io.StringIO()
    writer = csv.DictWriter(buf, fieldnames=fieldnames)
    writer.writeheader()
    writer.writerows(flat_rows)
    csv_text = buf.getvalue()

    print(csv_text)
    if args.output:
        Path(args.output).write_text(csv_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`root.findall(f".//{args.row}")` uses ElementTree's limited XPath support to find every element with the given tag name at any depth (`.//` means "search recursively from here"). For each matching row element, the script iterates its *direct* children only, using each child's tag as a CSV column name and its text as the cell value — column order is first-seen order across all rows, same convention as `json_to_csv.py` (24.3).

```bash
# Built-in demo
python3 xml_to_csv.py

# Real file with a different row tag
python3 xml_to_csv.py --input feed.xml --row item --output feed.csv
```

#### Sample output

```
$ python3 xml_to_csv.py
id,name,city
1,Alice,Lahore
2,Bob,Karachi
```

#### Variations / flags

- Add an `--attrs` flag to also emit attribute values as columns (currently only child-element text is captured, not attributes on the row element itself, like `<record id="1">` with `id` as an attribute rather than a child).
- Add `--sep` support similar to other converters in this chapter if you need semicolon-delimited output for a European-locale CSV consumer.
- Chain with `csv_to_json.py` (24.4) for an XML → CSV → JSON path when you need JSON but the intermediate CSV is also a useful audit artifact.

#### Common pitfalls

- **`ERROR: no elements named <record> found`** — the most common failure mode: your XML uses a different tag name for repeated rows (`<item>`, `<entry>`, `<row>`), or a namespace prefix (`<ns:record>`) that doesn't match a bare `--row record`; check the actual tag names with `xml_to_json.py` (24.7) first if you're unsure of the document's shape.
- **XML namespaces silently break tag matching** — `root.findall(".//record")` will not match `<record xmlns="http://example.com/ns">` without namespace-qualifying the search (`{http://example.com/ns}record`); this is arguably ElementTree's biggest surprise for newcomers, since the tags print without the namespace prefix but must be matched with it.
- **Nested child elements are flattened to their `.text`, which is often `None`/empty** — if a `<record>`'s child itself contains child elements rather than plain text, `child.text` will usually be whitespace or empty, silently losing the nested data rather than erroring; verify your XML is genuinely flat before trusting the output.

### 24.11 Parse and Transform with jq-style Queries

#### Use case

`jq` is the standard tool for slicing JSON on the command line, but it isn't always installable — locked-down containers, air-gapped servers, or CI runners with a minimal image sometimes don't have it and can't get it. A sysadmin who needs `.employees[*].name`-style field extraction in a pure-Python script (for portability, or because Python is already guaranteed to be present) reaches for this instead.

#### Prerequisites

Standard library only (`json`, `re`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows — no dependency on the external `jq` binary at all, which is the entire point of this script.

#### Cautions

This is explicitly a *subset* of real jq syntax — no pipes (`|`), no functions (`select()`, `map()`), no arithmetic. Don't reach for this when you actually have `jq` available and need its full filter language; use it only where portability matters more than expressiveness.

#### Script

```python
#!/usr/bin/env python3
"""
jq_query.py
--------------
Topic 347: Parse and Transform with jq-style Queries

Applies simplified jq-style path expressions to JSON data, entirely
in pure Python (no dependency on the `jq` C library, which can be
tricky to install on some systems). Supports:

    .a.b.c          -> nested field access
    .a.b[0]         -> array indexing
    .a.b[*]         -> collect a field across every array item
    .[]             -> iterate a top-level array

USAGE
    python3 jq_query.py
    python3 jq_query.py --input data.json --query ".employees[*].name"
    python3 jq_query.py --json '{"a":{"b":5}}' --query ".a.b"

EXPECTED OUTPUT
    The result of applying the query, printed as JSON.

NOTE
    This is a lightweight subset of real jq syntax, sufficient for
    common field-extraction and array-mapping tasks. It does not
    implement jq's full filter language (pipes, functions, etc).
"""
import argparse
import json
import re
import sys
from pathlib import Path

DEMO_DATA = {
    "company": "Acme Corp",
    "employees": [
        {"name": "Alice", "role": "Engineer", "salary": 95000},
        {"name": "Bob", "role": "Designer", "salary": 80000},
    ],
}

TOKEN_RE = re.compile(r"\.([A-Za-z0-9_]+)|\[(\*|\d+)\]|\.\[\]")


def tokenize(query: str):
    tokens = []
    pos = 0
    query = query.strip()
    while pos < len(query):
        m = TOKEN_RE.match(query, pos)
        if not m:
            sys.exit(f"ERROR: could not parse query near '{query[pos:]}'")
        if m.group(0) == ".[]":
            tokens.append(("wildcard_top", None))
        elif m.group(1) is not None:
            tokens.append(("field", m.group(1)))
        elif m.group(2) == "*":
            tokens.append(("wildcard", None))
        else:
            tokens.append(("index", int(m.group(2))))
        pos = m.end()
    return tokens


def apply_tokens(data, tokens):
    if not tokens:
        return data
    kind, val = tokens[0]
    rest = tokens[1:]

    if kind == "field":
        if isinstance(data, dict) and val in data:
            return apply_tokens(data[val], rest)
        return None
    if kind == "index":
        if isinstance(data, list) and 0 <= val < len(data):
            return apply_tokens(data[val], rest)
        return None
    if kind in ("wildcard", "wildcard_top"):
        if isinstance(data, list):
            return [apply_tokens(item, rest) for item in data]
        return None
    return None


def main():
    parser = argparse.ArgumentParser(description="Apply a jq-style query to JSON")
    parser.add_argument("--input", help="Path to a JSON file")
    parser.add_argument("--json", help="Raw JSON string")
    parser.add_argument(
        "--query",
        default=".employees[*].name",
        help='jq-style query, e.g. ".employees[*].name" (default shown)',
    )
    args = parser.parse_args()

    if args.json:
        data = json.loads(args.json)
    elif args.input:
        data = json.loads(Path(args.input).read_text(encoding="utf-8"))
    else:
        data = DEMO_DATA

    tokens = tokenize(args.query)
    result = apply_tokens(data, tokens)
    print(json.dumps(result, indent=2))


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`tokenize()` uses a single compiled regex to break the query string into a sequence of `("field", name)`, `("index", n)`, `("wildcard", None)`, or `("wildcard_top", None)` tokens by repeatedly matching at the current position. `apply_tokens()` then recursively consumes one token at a time against the data: `field` does a dict lookup, `index` does a list lookup, and `wildcard`/`wildcard_top` map the *remaining* tokens over every item in a list — which is what lets `.employees[*].name` pull `name` out of every object in the array in one expression.

```bash
# Built-in demo query (.employees[*].name)
python3 jq_query.py

# Custom query against the same demo data
python3 jq_query.py --query ".employees[0].salary"

# Query a real file
python3 jq_query.py --input data.json --query ".company"

# Query an inline JSON string
python3 jq_query.py --json '{"a":{"b":5}}' --query ".a.b"
```

#### Sample output

```
$ python3 jq_query.py
[
  "Alice",
  "Bob"
]

$ python3 jq_query.py --query ".employees[0].name"
"Alice"
```

#### Variations / flags

- Add `--raw` to print string results without surrounding quotes (mirroring real `jq -r`), which is often what you want when piping the output into another shell command.
- Extend `TOKEN_RE` and `apply_tokens()` to support a basic `select(.field == "value")` filter for conditional extraction — a common next feature request once teams get used to the field-access subset.
- For genuinely complex filters (pipes, `map()`, arithmetic), install real `jq` where possible and reserve this script for environments where that isn't an option.

#### Common pitfalls

- **Silent `null` on typos** — a query like `.employes[*].name` (missing an "e") returns `null` rather than raising an error, because `apply_tokens()` returns `None` for any missing field; double-check field names against the actual JSON shape (use `json_deserialize.py`'s structure summary, 24.2, first) rather than assuming a `null` result means the field is genuinely empty.
- **No support for negative indices or slices** — `.employees[-1]` or `.employees[0:2]` are not recognized by `TOKEN_RE` and will produce a parse error; this is a deliberate scope limit, not a bug.
- **Query syntax errors point at the *remaining* unparsed string, not a line/column** — for complex queries, if you get `ERROR: could not parse query near '...'`, count characters manually or break the query into smaller pieces to isolate the bad token, since there's no fancier error reporting here.

### 24.12 Convert INI to JSON/YAML

#### Use case

Older Python/PHP/Windows applications store configuration in `.ini` format (`configparser`-style: `[section]` headers, `key = value` lines). A sysadmin modernizing a deployment pipeline — moving config into a Kubernetes ConfigMap or a JSON-based feature-flag service — needs a clean, structured export from the old format rather than hand-transcribing every section.

#### Prerequisites

Standard library only for JSON output; PyYAML only if using `--format yaml`.

```bash
# JSON output needs nothing extra
python3 --version

# YAML output needs PyYAML
pip install pyyaml --break-system-packages
```

OS scope: Linux, macOS, Windows. INI files are especially common as Windows application config, so this conversion is frequently run on/for Windows-originated files even when the script itself runs on Linux.

#### Cautions

`configparser` (and this script) treats INI keys as case-insensitive by default and lower-cases them on read — if your original INI file relies on case-sensitive keys, that distinction is lost in the conversion. Verify this doesn't matter for your specific config before relying on the output.

#### Script

```python
#!/usr/bin/env python3
"""
ini_to_json_yaml.py
----------------------
Topic 348: Convert INI to JSON/YAML

Migrates legacy INI configuration files (as read by configparser)
into modern structured JSON or YAML. Each INI section becomes a
top-level key; keys inside become nested fields.

USAGE
    python3 ini_to_json_yaml.py
    python3 ini_to_json_yaml.py --input app.ini --format yaml --output app.yaml
    python3 ini_to_json_yaml.py --input app.ini --format json --output app.json

EXPECTED OUTPUT
    The converted config printed to the terminal (and written to
    --output if given) in the chosen --format (json by default).
"""
import argparse
import configparser
import json
import sys
from pathlib import Path

DEMO_INI = """[server]
host = 0.0.0.0
port = 8080
debug = false

[database]
engine = postgresql
name = app_db
"""


def coerce(value: str):
    low = value.lower()
    if low in ("true", "false"):
        return low == "true"
    try:
        if "." in value:
            return float(value)
        return int(value)
    except ValueError:
        return value


def ini_to_dict(text: str) -> dict:
    parser = configparser.ConfigParser()
    parser.read_string(text)
    result = {}
    for section in parser.sections():
        result[section] = {k: coerce(v) for k, v in parser.items(section)}
    return result


def main():
    parser = argparse.ArgumentParser(description="Convert INI to JSON or YAML")
    parser.add_argument("--input", help="Path to an INI file")
    parser.add_argument("--output", help="Path to write the output file")
    parser.add_argument("--format", choices=["json", "yaml"], default="json")
    args = parser.parse_args()

    text = Path(args.input).read_text(encoding="utf-8") if args.input else DEMO_INI
    data = ini_to_dict(text)

    if args.format == "yaml":
        try:
            import yaml
        except ImportError:
            sys.exit(
                "ERROR: PyYAML is not installed.\n"
                "Install dependencies first:  pip install -r requirements.txt"
            )
        out_text = yaml.dump(data, sort_keys=False, default_flow_style=False)
    else:
        out_text = json.dumps(data, indent=2)

    print(out_text)
    if args.output:
        Path(args.output).write_text(out_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`configparser.ConfigParser().read_string()` parses the INI text into sections; `ini_to_dict()` walks every section and every key inside it, running each value through the same `coerce()` pattern seen in `csv_to_json.py` (24.4) — `"true"`/`"false"` (any case) become real booleans, numeric-looking strings become `int`/`float`, everything else stays a string. The `--format` flag then routes the resulting nested dict through either `json.dumps()` or `yaml.dump()`, importing PyYAML lazily only when YAML output is actually requested — so JSON output works with zero extra dependencies.

```bash
# Built-in demo, JSON output (default)
python3 ini_to_json_yaml.py

# Built-in demo, YAML output
python3 ini_to_json_yaml.py --format yaml

# Real file
python3 ini_to_json_yaml.py --input app.ini --format yaml --output app.yaml
```

#### Sample output

```
$ python3 ini_to_json_yaml.py
{
  "server": {
    "host": "0.0.0.0",
    "port": 8080,
    "debug": false
  },
  "database": {
    "engine": "postgresql",
    "name": "app_db"
  }
}

$ python3 ini_to_json_yaml.py --format yaml
server:
  host: 0.0.0.0
  port: 8080
  debug: false
database:
  engine: postgresql
  name: app_db
```

#### Variations / flags

- Add a `--flat` flag that skips the section-nesting and instead prefixes keys with the section name (`server.host`) for tools that expect a flat config namespace like environment variables (`SERVER_HOST`).
- Support INI's `%(interpolation)s` syntax explicitly by using `configparser.ConfigParser(interpolation=configparser.ExtendedInterpolation())` if your source files rely on cross-referencing values between keys — the default parser only supports basic `%()s` interpolation within the same section.
- Preserve a `DEFAULT` section's fallback behavior (INI files often have a `[DEFAULT]` section whose keys apply to every other section) by checking `parser.defaults()` separately, since the current script's `parser.items(section)` already includes inherited defaults merged in, which may or may not be the behavior you want to preserve explicitly in the output.

#### Common pitfalls

- **Duplicate section names are silently merged** — INI files with the same `[section]` header appearing twice will have their keys merged by `configparser`, with later values overwriting earlier ones for the same key; this is `configparser`'s default `strict=True` behavior actually *rejecting* true duplicate sections with a `DuplicateSectionError` — so in practice you'll see a crash, not silent merging, which is the safer outcome but can be a surprise on old files with genuine duplicates.
- **Comments and ordering are lost** — like the JSON↔YAML round trip elsewhere in this chapter, converting INI to JSON/YAML and back does not preserve inline comments (`; like this` or `# like this`) or blank-line formatting.
- **Values with `%` characters** — `configparser`'s default interpolation treats `%` specially; a literal password or config value containing a percent sign (`key = 50%off`) can raise `InterpolationSyntaxError` unless escaped as `%%` in the source INI or `interpolation=None` is passed to `ConfigParser()`.

### 24.13 Convert TOML to JSON

#### Use case

TOML has become the standard for Python packaging metadata (`pyproject.toml`) and is also used by Rust (`Cargo.toml`), Hugo, and various CI tools. A sysadmin writing tooling that inspects or audits project metadata across a large monorepo — say, collecting every package's declared dependencies for a security scan — often finds it easier to work with that data as JSON in a pipeline that already has other JSON-based steps.

#### Prerequisites

Python 3.11+ has a TOML parser (`tomllib`) built into the standard library — nothing to install. Python 3.10 and earlier need the third-party `toml` package as a fallback.

```bash
python3 --version   # if 3.11+, nothing further needed

# only if on Python < 3.11:
pip install toml --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

`tomllib` (3.11+) is **read-only** — the standard library deliberately does not include a TOML *writer*. This script only ever converts TOML → JSON, never the reverse; if you need to write TOML, you'll need a separate third-party writer package regardless of Python version.

#### Script

```python
#!/usr/bin/env python3
"""
toml_to_json.py
------------------
Topic 349: Convert TOML to JSON

Transforms TOML files (e.g. Python's pyproject.toml) into widely
compatible JSON. Uses the standard-library `tomllib` on Python
3.11+, and falls back to the third-party `toml` package on older
Python versions (see requirements.txt).

USAGE
    python3 toml_to_json.py
    python3 toml_to_json.py --input pyproject.toml --output pyproject.json

EXPECTED OUTPUT
    Pretty-printed JSON printed to the terminal (and written to
    --output if given).
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import tomllib  # Python 3.11+
    _read_mode = "rb"

    def _load(text_bytes):
        return tomllib.loads(text_bytes.decode("utf-8"))

except ModuleNotFoundError:
    try:
        import toml
    except ImportError:
        sys.exit(
            "ERROR: no TOML parser available.\n"
            "On Python < 3.11, install the fallback:  pip install -r requirements.txt"
        )
    _read_mode = "r"

    def _load(text_str):
        return toml.loads(text_str)

DEMO_TOML = """
[project]
name = "example-app"
version = "1.0.0"

[project.dependencies]
requests = "^2.31"
pyyaml = "^6.0"

[tool.settings]
debug = false
max_connections = 10
"""


def main():
    parser = argparse.ArgumentParser(description="Convert TOML to JSON")
    parser.add_argument("--input", help="Path to a TOML file")
    parser.add_argument("--output", help="Path to write the JSON file")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    if args.input:
        mode = _read_mode
        with open(args.input, mode) as f:
            raw = f.read()
    else:
        raw = DEMO_TOML.encode("utf-8") if _read_mode == "rb" else DEMO_TOML

    data = _load(raw)
    json_text = json.dumps(data, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The `try`/`except ModuleNotFoundError` at import time picks the right TOML parser for the running interpreter: `tomllib` on 3.11+ (which requires opening the file in **binary** mode, `"rb"`, per its API) or the third-party `toml` package on older versions (which expects a plain text string). This dual-path handling is exactly the kind of version-compatibility shim a sysadmin needs when a fleet of servers isn't all on the same Python version yet.

```bash
# Built-in demo
python3 toml_to_json.py

# Real pyproject.toml from a project
python3 toml_to_json.py --input pyproject.toml --output pyproject.json
```

#### Sample output

```
$ python3 toml_to_json.py
{
  "project": {
    "name": "example-app",
    "version": "1.0.0",
    "dependencies": {
      "requests": "^2.31",
      "pyyaml": "^6.0"
    }
  },
  "tool": {
    "settings": {
      "debug": false,
      "max_connections": 10
    }
  }
}
```

Note how TOML's dotted table headers (`[project.dependencies]`, `[tool.settings]`) become properly nested JSON objects automatically — this is native TOML behavior, not something the script has to build itself.

#### Variations / flags

- Add `--query` support by piping the output into `jq_query.py` (24.11) to pull a single field, like a package's declared `version`, out of a large `pyproject.toml` in a one-liner.
- For a full CI dependency audit, loop this script across every `pyproject.toml`/`Cargo.toml` in a monorepo and merge the results with `merge_configs.py` (24.20) into one combined report.
- If you need TOML → YAML directly, chain this script with `json_to_yaml.py` (24.5): `python3 toml_to_json.py --input x.toml | python3 json_to_yaml.py --input /dev/stdin` (note `json_to_yaml.py` as provided reads `--input` from a file path, so redirect to a temp file if `/dev/stdin` isn't supported on your platform).

#### Common pitfalls

- **`tomllib` requires binary file mode** — a common mistake when adapting this script is opening the file with `open(path, "r")` and passing the resulting `str` to `tomllib.loads()`, which raises `TypeError: File must be opened in binary mode`; the script's `_read_mode` variable exists specifically to avoid this trap.
- **TOML dates/datetimes have no JSON equivalent** — TOML natively supports date, time, and datetime literals (`updated = 2026-07-18`); `tomllib`/`toml` parse these into Python `date`/`datetime` objects, which `json.dumps()` cannot serialize by default and will raise `TypeError`. If your TOML uses date literals, add a custom encoder (see `json_serialize.py`, 24.1) with an `isinstance(obj, (date, datetime))` branch before this script will handle them.
- **Python version drift across a fleet** — if some servers run Python 3.10 and others 3.11+, this script behaves identically from the *user's* perspective, but only if the `toml` package is actually installed on the older machines; a bare `pip freeze` audit across your fleet is worth doing before assuming this script "just works" everywhere.

### 24.14 Binary Serialization with Pickle

#### Use case

Caching intermediate computation results between runs of a Python script (e.g. a trained model object, a partially built graph, or any object with types JSON can't express) is the classic legitimate use of pickle: a sysadmin or data engineer needs *exact* Python object fidelity, not a portable interchange format for other languages.

#### Prerequisites

Standard library only (`pickle`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows. Pickle files are **not guaranteed cross-Python-version compatible** in all cases — a `.pkl` written by Python 3.12 using a recent pickle protocol may fail to load on Python 3.6 (see pitfalls).

#### Cautions

**This is the single most important caution in this entire chapter.** `pickle.load()` can execute arbitrary code if the data comes from an untrusted or unauthenticated source — a malicious `.pkl` file is not "corrupted data" that fails safely, it is an active attack payload. Never call `pickle.load()` on a file you did not create yourself, that arrived over an insecure channel, or whose origin you can't fully verify. This script is safe only because it exclusively loads files it just wrote in the same run. If you need to exchange serialized data with another system or store it long-term where its trust boundary is unclear, use `msgpack_serialize.py` (24.15) or a JSON-based approach instead.

#### Script

```python
#!/usr/bin/env python3
"""
pickle_serialize.py
-----------------------
Topic 350: Binary Serialization with Pickle

Serializes complex Python objects (including custom classes,
datetimes, sets, tuples -- things plain JSON cannot represent) into
binary .pkl format, and demonstrates loading them back.

!! SECURITY WARNING !!
    pickle.load() can execute arbitrary code if the data comes from
    an untrusted or unauthenticated source. NEVER unpickle data you
    did not create yourself or that arrived over an insecure channel.
    This script is safe because it only loads files it just wrote.

USAGE
    # Run built-in demo: serializes sample data to demo.pkl then
    # reads it back and prints it
    python3 pickle_serialize.py

    # Serialize your own file: pass a .py file that defines DATA
    python3 pickle_serialize.py --mode dump --input mydata.py --output out.pkl

    # Load an existing pickle file (only do this for files you trust!)
    python3 pickle_serialize.py --mode load --input out.pkl

EXPECTED OUTPUT
    On dump: confirmation that N bytes were written to the .pkl file.
    On load: repr() of the deserialized Python object.
"""
import argparse
import importlib.util
import pickle
import sys
from dataclasses import dataclass
from datetime import datetime
from pathlib import Path


@dataclass
class Event:
    name: str
    when: datetime
    tags: set


def build_demo_data():
    return {
        "event": Event("Launch", datetime(2026, 7, 17, 10, 0), {"prod", "release"}),
        "coordinates": (33.6844, 73.0479),  # tuple, not representable in JSON as-is
        "counts": {1, 2, 3, 3, 2},  # a set
    }


def load_input_module(path: str):
    spec = importlib.util.spec_from_file_location("user_input", path)
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    if not hasattr(module, "DATA"):
        sys.exit(f"ERROR: {path} must define a top-level variable named DATA")
    return module.DATA


def main():
    parser = argparse.ArgumentParser(description="Serialize/deserialize Python objects with pickle")
    parser.add_argument("--mode", choices=["dump", "load", "roundtrip"], default="roundtrip")
    parser.add_argument("--input", help="For dump: .py file with DATA. For load: .pkl file")
    parser.add_argument("--output", default="demo.pkl", help="Path for the .pkl file (dump mode)")
    args = parser.parse_args()

    if args.mode in ("dump", "roundtrip"):
        data = load_input_module(args.input) if (args.input and args.mode == "dump") else build_demo_data()
        with open(args.output, "wb") as f:
            pickle.dump(data, f)
        size = Path(args.output).stat().st_size
        print(f"Serialized object -> {args.output} ({size} bytes)")
        print(f"Original object: {data!r}")

    if args.mode == "load":
        with open(args.input, "rb") as f:
            obj = pickle.load(f)
        print(f"Deserialized object: {obj!r}")

    if args.mode == "roundtrip":
        with open(args.output, "rb") as f:
            obj = pickle.load(f)
        print(f"\nRound-tripped object (loaded back): {obj!r}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`pickle.dump()` walks the object graph and writes a binary opcode stream that, when replayed by `pickle.load()`, reconstructs equivalent objects — including custom class instances (`Event`), `set`s, and `tuple`s, none of which JSON can represent natively. The default `--mode roundtrip` demonstrates the full cycle in one run: serialize the demo data to `demo.pkl`, then immediately load it back and print both versions so you can visually confirm they match.

```bash
# Full round trip using demo data (writes demo.pkl, then reads it back)
python3 pickle_serialize.py

# Serialize your own data (define DATA in a .py file first)
cat > mydata.py << 'EOF'
DATA = {"tags": {"a", "b", "c"}, "point": (1, 2)}
EOF
python3 pickle_serialize.py --mode dump --input mydata.py --output out.pkl

# Load a trusted, previously-written pickle file
python3 pickle_serialize.py --mode load --input out.pkl
```

#### Sample output

```
$ python3 pickle_serialize.py
Serialized object -> demo.pkl (198 bytes)
Original object: {'event': Event(name='Launch', when=datetime.datetime(2026, 7, 17, 10, 0), tags={'release', 'prod'}), 'coordinates': (33.6844, 73.0479), 'counts': {1, 2, 3}}

Round-tripped object (loaded back): {'event': Event(name='Launch', when=datetime.datetime(2026, 7, 17, 10, 0), tags={'release', 'prod'}), 'coordinates': (33.6844, 73.0479), 'counts': {1, 2, 3}}
```

Note `{1, 2, 3, 3, 2}` (the literal in `build_demo_data()`) is printed as `{1, 2, 3}` — `set` deduplication happens the moment the literal is evaluated, before pickling ever runs, and both the "original" and "round-tripped" printouts correctly show only the three unique values.

#### Variations / flags

- Add `protocol=pickle.HIGHEST_PROTOCOL` explicitly to `pickle.dump()` calls for smaller files and slightly faster (de)serialization — the script currently uses the implicit default protocol, which is safe but not maximally optimized.
- For a size/speed comparison against a portable alternative, run the same demo data through `msgpack_serialize.py` (24.15) and compare byte counts — pickle is typically *not* smaller than MessagePack for equivalent JSON-representable data, since pickle's format carries extra bookkeeping to support arbitrary Python types.
- If custom class instances (like `Event`) need to remain loadable after the class definition changes (renamed field, moved module), implement `__reduce__` or `__setstate__`/`__getstate__` on the class for version-tolerant pickling — outside the scope of this basic script.

#### Common pitfalls

- **The security warning is not optional** — repeating it because it's the most consequential mistake in this whole chapter: never run `--mode load` against a `.pkl` file you didn't create or fully trust. If you're tempted to accept pickle files from users, an API, or a network share "just this once," don't — use JSON or MessagePack for anything crossing a trust boundary.
- **Cross-version compatibility isn't guaranteed** — a pickle written with a newer Python's default protocol may fail with `UnpicklingError` or `ValueError: unsupported pickle protocol` on an older interpreter; if your fleet has mixed Python versions, pin `protocol=` to the highest version *all* your machines support, or avoid pickle for cross-machine transport entirely.
- **Class must be importable at load time** — unpickling `Event(...)` requires Python to be able to `import` a class named `Event` from the same module path it was pickled from; moving or renaming the `Event` class (or the script itself) between dump and load will raise `AttributeError`/`ModuleNotFoundError` on load, even though the file loaded from a fully trusted source.

### 24.15 Binary Serialization with MessagePack

#### Use case

A sysadmin building a high-throughput internal service (message queue payloads, cache entries, inter-service RPC) that needs JSON-like data but with less network/disk overhead and faster parsing reaches for MessagePack: same conceptual data model as JSON (objects, arrays, strings, numbers, booleans, null), but a smaller binary wire format, and — unlike pickle (24.14) — safe to accept from untrusted or cross-language sources since it can't execute code.

#### Prerequisites

Requires the `msgpack` package.

```bash
pip install msgpack --break-system-packages
```

OS scope: Linux, macOS, Windows — `msgpack-python` ships prebuilt wheels with a C extension for speed; pure-Python fallback is used automatically if the wheel isn't available for your platform.

#### Cautions

MessagePack, unlike pickle, only represents JSON-equivalent data types by default — no custom classes, no arbitrary Python objects — which is precisely why it's safe to unpack data from sources you don't fully trust (unlike `pickle.load()`). Don't reach for MessagePack expecting pickle's "serialize anything" flexibility; if you need that, and only from a fully trusted source, use pickle instead.

#### Script

```python
#!/usr/bin/env python3
"""
msgpack_serialize.py
------------------------
Topic 351: Binary Serialization with MessagePack

Packs Python data into MessagePack -- a compact, cross-language
binary format that is typically smaller and faster to (de)serialize
than JSON. Requires the `msgpack` package (see requirements.txt).

USAGE
    python3 msgpack_serialize.py
    python3 msgpack_serialize.py --input data.json --output data.msgpack
    python3 msgpack_serialize.py --mode load --input data.msgpack

EXPECTED OUTPUT
    On pack: the packed byte size compared to the equivalent JSON
    size, and the file it was written to.
    On load: the object that was unpacked, printed with repr().
"""
import argparse
import json
import sys
from pathlib import Path

try:
    import msgpack
except ImportError:
    sys.exit(
        "ERROR: msgpack is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_DATA = {
    "id": 42,
    "name": "Acme Corp",
    "employees": [
        {"name": "Alice", "role": "Engineer", "salary": 95000},
        {"name": "Bob", "role": "Designer", "salary": 80000},
    ],
    "active": True,
}


def main():
    parser = argparse.ArgumentParser(description="Pack/unpack data with MessagePack")
    parser.add_argument("--mode", choices=["pack", "load"], default="pack")
    parser.add_argument("--input", help="For pack: JSON file. For load: .msgpack file")
    parser.add_argument("--output", default="data.msgpack", help="Path for the packed file")
    args = parser.parse_args()

    if args.mode == "pack":
        data = json.loads(Path(args.input).read_text(encoding="utf-8")) if args.input else DEMO_DATA
        packed = msgpack.packb(data, use_bin_type=True)
        Path(args.output).write_bytes(packed)

        json_size = len(json.dumps(data).encode("utf-8"))
        print(f"Packed {len(packed)} bytes -> {args.output}")
        print(f"Equivalent JSON size: {json_size} bytes")
        print(f"Space saved: {json_size - len(packed)} bytes ({(1 - len(packed) / json_size):.1%})")
    else:
        if not args.input:
            sys.exit("ERROR: --input is required for --mode load")
        packed = Path(args.input).read_bytes()
        obj = msgpack.unpackb(packed, raw=False)
        print(f"Unpacked object: {obj!r}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`msgpack.packb(data, use_bin_type=True)` serializes the data to a compact binary blob using MessagePack's type-tagged encoding (each value is prefixed with a compact type/length marker rather than JSON's verbose text syntax like `"key": `, `,`, `{`, `}`). `use_bin_type=True` ensures Python `bytes` and `str` are distinguished on the wire (important for cross-language correctness); `raw=False` on unpack correspondingly decodes MessagePack strings back to Python `str` rather than raw `bytes`.

```bash
# Built-in demo: pack demo data, compare size to JSON
python3 msgpack_serialize.py

# Pack a real JSON file
python3 msgpack_serialize.py --input data.json --output data.msgpack

# Load a packed file back
python3 msgpack_serialize.py --mode load --input data.msgpack
```

#### Sample output

```
$ python3 msgpack_serialize.py
Packed 113 bytes -> data.msgpack
Equivalent JSON size: 172 bytes
Space saved: 59 bytes (34.3%)

$ python3 msgpack_serialize.py --mode load --input data.msgpack
Unpacked object: {'id': 42, 'name': 'Acme Corp', 'employees': [{'name': 'Alice', 'role': 'Engineer', 'salary': 95000}, {'name': 'Bob', 'role': 'Designer', 'salary': 80000}], 'active': True}
```

The ~34% size reduction shown here is typical for this kind of moderately-nested data with several small integers and short strings; the savings grow larger for data with many repeated small integers or booleans (MessagePack's fixint/fixstr encodings are extremely compact) and shrink for data dominated by long strings (where JSON's overhead is proportionally smaller).

#### Variations / flags

- Add a `--compare` mode that also times both `json.dumps()`/`json.loads()` and `msgpack.packb()`/`msgpack.unpackb()` over many iterations (`timeit`) to show the speed difference alongside the size difference — size and speed are the two arguments for choosing MessagePack, and this script currently only demonstrates the first.
- Add streaming support via `msgpack.Unpacker()` for large files that shouldn't be loaded fully into memory, mirroring the approach in `stream_large_dataset.py` (24.23).
- Batch this across many small JSON files (message queue payloads, cache dumps) to build a fleet-wide "would MessagePack save us meaningful bandwidth" report before committing to a wire-format migration.

#### Common pitfalls

- **`raw=False` is important, and easy to forget** — omitting it (or using an older msgpack version's default) unpacks MessagePack strings as raw `bytes` instead of `str`, which then compare unequal to the original Python strings (`b"Alice" != "Alice"`) even though the round trip is technically correct; always pass `raw=False` explicitly when your data is text, not raw binary.
- **Not a drop-in replacement for arbitrary Python objects** — attempting to pack a Python `set`, `datetime`, or custom class without a custom `default=` encoder (similar to `json_serialize.py`'s `DemoJSONEncoder`, 24.1) raises `TypeError`; MessagePack out of the box supports exactly the same type set as JSON, no more.
- **Cross-language consumers must agree on `use_bin_type`** — older MessagePack implementations in other languages may not distinguish binary from string data the same way; if you're integrating with a non-Python service, confirm both sides agree on the `bin`/`str` type distinction before assuming interoperability.

### 24.16 Base64 Encode/Decode Data

#### Use case

A sysadmin embedding a small binary file (an image, a certificate, a signed token) inside a JSON payload, an environment variable, or an email attachment needs Base64 because those text-only channels can't safely carry raw binary bytes — control characters, null bytes, and non-UTF-8 sequences would corrupt or break the transport.

#### Prerequisites

Standard library only (`base64`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows.

#### Cautions

Base64 is an *encoding*, not encryption or compression — it provides zero confidentiality (anyone can decode it instantly) and actually *increases* size by roughly 33%. Don't use it as a substitute for a secrets manager or for actual data compression.

#### Script

```python
#!/usr/bin/env python3
"""
base64_codec.py
-------------------
Topic 352: Base64 Encode/Decode Data

Encodes binary/text data as ASCII-safe Base64 text (for embedding in
JSON, URLs, or text protocols), and decodes it back to the original
bytes. Standard library only (base64 module).

USAGE
    # Run built-in demo (encode then decode a sample string)
    python3 base64_codec.py

    # Encode a file (any binary file works, e.g. an image)
    python3 base64_codec.py --mode encode --input photo.png --output photo.b64

    # Decode a Base64 file back to binary
    python3 base64_codec.py --mode decode --input photo.b64 --output photo_copy.png

    # Encode/decode raw text directly
    python3 base64_codec.py --mode encode --text "hello world"
    python3 base64_codec.py --mode decode --text "aGVsbG8gd29ybGQ="

EXPECTED OUTPUT
    encode: the Base64 text (and/or an output file).
    decode: the recovered bytes, shown as text if valid UTF-8,
    otherwise a byte count.
"""
import argparse
import base64
import sys
from pathlib import Path

DEMO_TEXT = "Data Serialization & Transformation with Python 🚀"


def main():
    parser = argparse.ArgumentParser(description="Base64 encode/decode data")
    parser.add_argument("--mode", choices=["encode", "decode"], default="encode")
    parser.add_argument("--input", help="Path to a file to read")
    parser.add_argument("--text", help="Raw text to encode/decode instead of a file")
    parser.add_argument("--output", help="Path to write the result")
    args = parser.parse_args()

    demo_mode = args.text is None and args.input is None

    if args.text is not None:
        raw = args.text.encode("utf-8")
    elif args.input:
        raw = Path(args.input).read_bytes()
    else:
        raw = DEMO_TEXT.encode("utf-8")

    if demo_mode:
        encoded = base64.b64encode(raw).decode("ascii")
        print(f"Original: {DEMO_TEXT}")
        print(f"Encoded : {encoded}")
        print(f"Decoded : {base64.b64decode(encoded).decode('utf-8')}")
        return

    if args.mode == "encode":
        result = base64.b64encode(raw)
        print(result.decode("ascii"))
        if args.output:
            Path(args.output).write_bytes(result)
            print(f"[written to {args.output}]", file=sys.stderr)
    else:
        try:
            result = base64.b64decode(raw)
        except Exception as exc:
            sys.exit(f"ERROR: invalid Base64 input: {exc}")
        try:
            print(result.decode("utf-8"))
        except UnicodeDecodeError:
            print(f"[decoded {len(result)} bytes of binary data]")
        if args.output:
            Path(args.output).write_bytes(result)
            print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`base64.b64encode()` maps every 3 raw bytes to 4 ASCII characters drawn from a 64-character alphabet (`A-Z`, `a-z`, `0-9`, `+`, `/`, with `=` padding), guaranteeing the output is always safe to embed in text-only contexts regardless of what the original bytes were. `base64.b64decode()` reverses this exactly. The script works identically whether the input is a text string (`--text`) or an arbitrary binary file (`--input`), since both are ultimately just bytes to `base64`.

```bash
# Built-in demo (encode then decode a sample string, round trip shown)
python3 base64_codec.py

# Encode a binary file
python3 base64_codec.py --mode encode --input photo.png --output photo.b64

# Decode it back
python3 base64_codec.py --mode decode --input photo.b64 --output photo_copy.png

# Encode/decode raw text directly (useful for quick one-liners)
python3 base64_codec.py --mode encode --text "hello world"
python3 base64_codec.py --mode decode --text "aGVsbG8gd29ybGQ="
```

#### Sample output

```
$ python3 base64_codec.py
Original: Data Serialization & Transformation with Python 🚀
Encoded : RGF0YSBTZXJpYWxpemF0aW9uICYgVHJhbnNmb3JtYXRpb24gd2l0aCBQeXRob24g8J+agA==
Decoded : Data Serialization & Transformation with Python 🚀
```

Note the emoji (a 4-byte UTF-8 sequence) round-trips correctly — Base64 operates on raw bytes, so it's entirely agnostic to whether those bytes represent text, an image, or anything else.

#### Variations / flags

- Add `--urlsafe` to switch to `base64.urlsafe_b64encode()`/`urlsafe_b64decode()`, which replace `+`/`/` with `-`/`_` so the result is safe to embed directly in a URL path or query string without additional percent-encoding.
- Chain with `json_serialize.py` (24.1) to embed binary data (e.g. a small icon) as a Base64 string value inside a larger JSON document — this is the standard pattern for "data URIs" and JSON-embedded binary attachments.
- For very large files, avoid loading the whole file into memory with `read_bytes()`; use `base64.encode()`/`decode()` with file objects directly (`base64.encode(input_stream, output_stream)`) for constant-memory streaming encode/decode.

#### Common pitfalls

- **`binascii.Error: Incorrect padding`** — Base64 strings must be a multiple of 4 characters (padded with `=` as needed); a truncated or manually-edited Base64 string (missing trailing `=`) fails to decode with this specific error — the script surfaces it clearly as `ERROR: invalid Base64 input: Incorrect padding` rather than crashing with a raw traceback.
- **Confusing standard and URL-safe alphabets** — a Base64 string produced with `urlsafe_b64encode()` (using `-`/`_`) will fail or silently produce wrong bytes if decoded with plain `b64decode()` (expecting `+`/`/`), and vice versa; always decode with the matching function.
- **33% size inflation is easy to forget when budgeting** — embedding a 3 MB file as Base64 inside a JSON payload produces roughly a 4 MB string; if you're hitting a request-size limit on an API, remember Base64 is part of why, not just the original file size.

### 24.17 URL Encode/Decode Data

#### Use case

A sysadmin scripting an HTTP client call, building a redirect URL with untrusted user input, or parsing incoming query-string parameters from a log file needs correct, standards-compliant percent-encoding — get it wrong and spaces, ampersands, or non-ASCII characters break the URL or, worse, allow query-parameter injection.

#### Prerequisites

Standard library only (`urllib.parse`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows.

#### Cautions

`quote()` and `urlencode()` encode differently by default: `quote()` encodes spaces as `%20`, while `urlencode()` (used for query strings) encodes spaces as `+` per the `application/x-www-form-urlencoded` convention — mixing the two conventions in the same URL is a common source of subtle bugs.

#### Script

```python
#!/usr/bin/env python3
"""
url_codec.py
---------------
Topic 353: URL Encode/Decode Data

Percent-encodes special characters for safe inclusion in URLs (query
strings, path segments) and decodes them back. Standard library only
(urllib.parse).

USAGE
    # Run built-in demo (no arguments needed)
    python3 url_codec.py

    # Encode a string
    python3 url_codec.py --mode encode --text "search term with spaces & symbols"

    # Decode a string
    python3 url_codec.py --mode decode --text "search%20term%20with%20spaces%20%26%20symbols"

    # Encode/decode a full query-string dict (JSON object of key/value pairs)
    python3 url_codec.py --mode encode-query --json '{"q": "python 3", "page": "2"}'

EXPECTED OUTPUT
    The encoded or decoded string, printed to the terminal.
"""
import argparse
import json
import sys
from urllib.parse import quote, unquote, urlencode, parse_qs

DEMO_TEXT = "search term with spaces & symbols?"


def main():
    parser = argparse.ArgumentParser(description="URL encode/decode data")
    parser.add_argument(
        "--mode",
        choices=["encode", "decode", "encode-query", "decode-query"],
        default="encode",
    )
    parser.add_argument("--text", help="Text to encode/decode")
    parser.add_argument("--json", help="JSON object for encode-query mode")
    args = parser.parse_args()

    if args.mode == "encode":
        text = args.text if args.text is not None else DEMO_TEXT
        print(quote(text))
    elif args.mode == "decode":
        text = args.text if args.text is not None else quote(DEMO_TEXT)
        print(unquote(text))
    elif args.mode == "encode-query":
        obj = json.loads(args.json) if args.json else {"q": "python 3", "page": "2"}
        print(urlencode(obj))
    elif args.mode == "decode-query":
        text = args.text if args.text is not None else "q=python+3&page=2"
        parsed = parse_qs(text)
        print(json.dumps({k: v[0] if len(v) == 1 else v for k, v in parsed.items()}, indent=2))


if __name__ == "__main__":
    if len(sys.argv) == 1:
        print(f"Original       : {DEMO_TEXT}")
        print(f"URL-encoded    : {quote(DEMO_TEXT)}")
        print(f"Decoded again  : {unquote(quote(DEMO_TEXT))}")
    else:
        main()
```

#### How it works and how to run

`quote()` percent-encodes every character outside a safe ASCII subset (letters, digits, and a small set of punctuation) — suitable for a single path segment or a value you'll insert manually into a URL. `urlencode()` takes a whole dict and builds a complete `key=value&key2=value2` query string, using `+` for spaces per form-encoding convention. `parse_qs()` does the reverse of `urlencode()`, returning a dict of lists (since a query string can legally repeat a key, e.g. `?tag=a&tag=b`) — the script collapses single-item lists back to a plain value for readability, but preserves genuine multi-value lists.

```bash
# Built-in demo (no args): quick encode/decode round trip
python3 url_codec.py

# Encode a specific string
python3 url_codec.py --mode encode --text "search term with spaces & symbols"

# Decode it back
python3 url_codec.py --mode decode --text "search%20term%20with%20spaces%20%26%20symbols"

# Build a full query string from a JSON object
python3 url_codec.py --mode encode-query --json '{"q": "python 3", "page": "2"}'

# Parse a query string back into structured JSON
python3 url_codec.py --mode decode-query --text "q=python+3&page=2&tag=a&tag=b"
```

```

URL-encoded    : search%20term%20with%20spaces%20%26%20symbols%3F
Decoded again  : search term with spaces & symbols?

```

#### Variations / flags

- Add `--safe` to expose `quote()`'s `safe=` parameter (characters that should *not* be encoded even though they normally would be, e.g. `safe="/"` when encoding a path that should keep its slashes).
- For path segments specifically, prefer `quote()` over `urlencode()`'s space-as-`+` convention — `+` is only meaningful inside a query string, not a path, and would be interpreted as a literal plus sign elsewhere.
- Chain `decode-query` output into `validate_json_schema.py` (24.21) to validate incoming query parameters against an expected shape before using them in application logic.

#### Common pitfalls

- **`+` vs `%20` for spaces** — mixing `quote()` (produces `%20`) and `urlencode()` (produces `+`) output in the same URL is a very common bug; both are technically valid in their respective contexts (path vs. form-encoded query string) but are not interchangeable — a literal `+` inside a `quote()`-encoded path segment will be misread by some servers as an encoded space when it shouldn't be.
- **Double-encoding** — running `quote()` on already-encoded text turns `%20` into `%2520` (the `%` itself gets encoded); always track whether a string has already been encoded before encoding it again, especially when values pass through multiple layers of a pipeline.
- **`parse_qs()` always returns lists** — even `?page=2` (a single value) parses to `{"page": ["2"]}`, not `{"page": "2"}`; the script's `decode-query` mode unwraps single-item lists for display convenience, but if you use `parse_qs()` directly elsewhere, remember every value is a list until you unwrap it yourself.

### 24.18 Convert Between Character Encodings

#### Use case

A sysadmin migrating data from an older Windows or European system (often `cp1252`/`latin-1`) into a modern UTF-8-only pipeline — or the reverse, preparing a file for a legacy system that only accepts `latin-1` — needs a controlled, explicit transcoding step rather than letting each tool in the pipeline guess the encoding and silently mangle non-ASCII characters.

#### Prerequisites

Standard library only (`pathlib`, built-in `str.encode`/`bytes.decode`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows. Encoding mismatches are especially common when moving files between Windows (historically `cp1252`) and Linux/macOS (UTF-8 by convention).

#### Cautions

Not every character in every encoding can be represented in every other encoding — converting UTF-8 (which can represent virtually any character) to `latin-1` (which only covers Western European characters) will fail by default on anything outside that range unless you explicitly choose `--errors replace` or `--errors ignore`, both of which are lossy.

#### Script

```python
#!/usr/bin/env python3
"""
encoding_convert.py
-----------------------
Topic 354: Convert Between Character Encodings

Transcodes text files between character encodings (e.g. UTF-8,
Latin-1/ISO-8859-1, ASCII, UTF-16) without data loss, as long as the
target encoding can represent every character in the source text.
Standard library only.

USAGE
    # Run built-in demo (no arguments needed)
    python3 encoding_convert.py

    # Convert a file from Latin-1 to UTF-8
    python3 encoding_convert.py --input legacy.txt --from-encoding latin-1 \\
        --to-encoding utf-8 --output modern.txt

EXPECTED OUTPUT
    Confirms the source/target encodings and byte counts, and writes
    the transcoded file if --output is given. If a character cannot
    be represented in the target encoding, a clear error is shown
    (use --errors replace or --errors ignore to override).
"""
import argparse
import sys
from pathlib import Path

DEMO_TEXT = "Café - naïve - 100% résumé"


def main():
    parser = argparse.ArgumentParser(description="Convert text between character encodings")
    parser.add_argument("--input", help="Path to a text file to convert")
    parser.add_argument("--output", help="Path to write the converted file")
    parser.add_argument("--from-encoding", default="utf-8", help="Source encoding (default: utf-8)")
    parser.add_argument("--to-encoding", default="latin-1", help="Target encoding (default: latin-1)")
    parser.add_argument(
        "--errors",
        default="strict",
        choices=["strict", "replace", "ignore"],
        help="How to handle characters that don't fit the target encoding",
    )
    args = parser.parse_args()

    if args.input:
        raw_bytes = Path(args.input).read_bytes()
        text = raw_bytes.decode(args.from_encoding)
    else:
        text = DEMO_TEXT

    try:
        converted = text.encode(args.to_encoding, errors=args.errors)
    except UnicodeEncodeError as exc:
        sys.exit(
            f"ERROR: cannot encode to {args.to_encoding}: {exc}\n"
            f"Try --errors replace or --errors ignore, or pick a target encoding "
            f"that supports all characters (e.g. utf-8)."
        )

    print(f"Text            : {text}")
    print(f"Source encoding : {args.from_encoding}")
    print(f"Target encoding : {args.to_encoding}")
    print(f"Converted bytes : {converted!r}")

    if args.output:
        Path(args.output).write_bytes(converted)
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Text conversion between encodings always goes through Python's internal Unicode representation as an intermediate step: `bytes.decode(from_encoding)` turns the source bytes into a Unicode `str`, and `str.encode(to_encoding)` turns that `str` into the target bytes. If any character in the `str` has no representation in the target encoding, `encode()` raises `UnicodeEncodeError` under the default `errors="strict"` — the script catches this and prints a clear, actionable error rather than a raw traceback.

```bash
# Built-in demo (UTF-8 text -> Latin-1 bytes, shown as a Python bytes repr)
python3 encoding_convert.py

# Convert a real legacy file from Latin-1 to UTF-8
python3 encoding_convert.py --input legacy.txt --from-encoding latin-1 \
    --to-encoding utf-8 --output modern.txt

# Force a lossy conversion when some characters can't be represented
python3 encoding_convert.py --input notes.txt --to-encoding ascii --errors replace
```

#### Sample output

```
$ python3 encoding_convert.py
Text            : Café - naïve - 100% résumé
Source encoding : utf-8
Target encoding : latin-1
Converted bytes : b'Caf\xe9 - na\xefve - 100% r\xe9sum\xe9'
```

Tested with a character genuinely outside Latin-1's range (the Euro sign, `€`) to confirm the failure path behaves correctly:

```
$ python3 encoding_convert.py --input euro.txt --from-encoding utf-8 --to-encoding latin-1
ERROR: cannot encode to latin-1: 'latin-1' codec can't encode character '\u20ac' in position 10: ordinal not in range(256)
Try --errors replace or --errors ignore, or pick a target encoding that supports all characters (e.g. utf-8).

$ python3 encoding_convert.py --input euro.txt --from-encoding utf-8 --to-encoding latin-1 --errors replace
Text            : Price: 100€
Source encoding : utf-8
Target encoding : latin-1
Converted bytes : b'Price: 100?\n'
```

Note `--errors replace` swaps the unrepresentable `€` for a literal `?` — data loss, but a controlled and visible one, rather than a crash or silent corruption.

#### Variations / flags

- Add a `--detect` flag using the `chardet`/`charset-normalizer` third-party package to guess the source encoding automatically when it's genuinely unknown — the current script requires you to already know (or assert) `--from-encoding`, which is the *safer* default since encoding detection is inherently probabilistic and can guess wrong on short or ambiguous text.
- Add `--errors xmlcharrefreplace` for HTML/XML output contexts, which converts unrepresentable characters to numeric character references (`&#8364;`) instead of dropping or replacing them with `?`.
- Batch-process a directory of legacy files with a shell loop: `for f in *.txt; do python3 encoding_convert.py --input "$f" --from-encoding cp1252 --to-encoding utf-8 --output "utf8_$f"; done`.

#### Common pitfalls

- **Guessing the wrong source encoding produces valid-looking garbage, not an error** — decoding `cp1252` bytes as if they were `latin-1` (or vice versa) often succeeds without raising an exception, because both are single-byte encodings that assign *some* character to every byte value — but the resulting text is wrong (mojibake). A `UnicodeDecodeError` is actually the easy case; silent wrong-encoding decoding is the dangerous one, so verify against known reference text when the source encoding is uncertain.
- **BOM (byte-order mark) confusion with UTF-16/UTF-8** — files saved by some Windows tools include a UTF-8 BOM (`\xef\xbb\xbf`) at the start; decoding with plain `"utf-8"` leaves a stray `\ufeff` character at the start of the resulting string. Use `"utf-8-sig"` as the source encoding to strip it automatically.
- **`--errors ignore` silently deletes characters** — unlike `replace` (which substitutes a visible placeholder), `ignore` removes unrepresentable characters entirely with no trace; this can silently corrupt meaning (e.g. dropping a currency symbol from a price) and should be used only when you've confirmed the affected characters are genuinely disposable.

### 24.19 Flatten Nested JSON Structures

#### Use case

A sysadmin loading JSON documents into a database table, a spreadsheet, or any system that expects flat rows rather than arbitrary nesting depth needs a general-purpose flattener that handles both nested objects and arrays consistently — and, just as often, needs the reverse operation to reconstruct nested JSON from a flat key-value store (like environment variables or a flat config table) back into the shape an application actually expects.

#### Prerequisites

Standard library only (`json`, `re`).

```bash
python3 --version
```

OS scope: Linux, macOS, Windows.

#### Cautions

Flatten and unflatten are meant to be exact inverses of each other for data produced by this script, but hand-edited flat JSON with inconsistent array indices (e.g. `roles[0]` and `roles[2]` present but `roles[1]` missing) will unflatten with `None` gaps filled in for the missing index rather than an error — verify array indices are contiguous if you're constructing flat keys by hand.

#### Script

```python
#!/usr/bin/env python3
"""
flatten_json.py
-------------------
Topic 355: Flatten Nested JSON Structures

Transforms deeply nested JSON objects/arrays into flat key-value
pairs (dot/bracket notation keys), suitable for database insertion
or CSV export. Also supports the reverse operation (unflatten) to
rebuild the original nested structure.

USAGE
    # Run built-in demo (no arguments needed)
    python3 flatten_json.py

    # Flatten a specific file
    python3 flatten_json.py --input data.json --output flat.json

    # Unflatten a previously flattened file back to nested JSON
    python3 flatten_json.py --mode unflatten --input flat.json --output nested.json

    # Use a custom separator (default is ".")
    python3 flatten_json.py --sep "_"

EXPECTED OUTPUT
    Pretty-printed JSON (flat or nested, depending on --mode) printed
    to the terminal (and written to --output if given).
"""
import argparse
import json
import re
import sys
from pathlib import Path

DEMO_DATA = {
    "user": {
        "id": 101,
        "name": "Alice",
        "address": {"city": "Lahore", "zip": "54000"},
        "roles": ["admin", "editor"],
    },
    "active": True,
}

INDEX_RE = re.compile(r"\[(\d+)\]$")


def flatten(obj, parent_key="", sep=".", out=None):
    if out is None:
        out = {}
    if isinstance(obj, dict):
        for k, v in obj.items():
            new_key = f"{parent_key}{sep}{k}" if parent_key else str(k)
            flatten(v, new_key, sep, out)
    elif isinstance(obj, list):
        for i, v in enumerate(obj):
            new_key = f"{parent_key}[{i}]"
            flatten(v, new_key, sep, out)
    else:
        out[parent_key] = obj
    return out


def unflatten(flat: dict, sep="."):
    result = {}
    for compound_key, value in flat.items():
        parts = re.split(rf"{re.escape(sep)}|(?=\[)", compound_key)
        parts = [p for p in parts if p != ""]
        cursor = result
        for i, part in enumerate(parts):
            m = INDEX_RE.match(part)
            is_last = i == len(parts) - 1
            if m:
                idx = int(m.group(1))
                if not isinstance(cursor, list):
                    raise ValueError(f"Cannot apply index on non-list at '{part}'")
                while len(cursor) <= idx:
                    cursor.append(None)
                if is_last:
                    cursor[idx] = value
                else:
                    nxt = parts[i + 1]
                    if cursor[idx] is None:
                        cursor[idx] = [] if INDEX_RE.match(nxt) else {}
                    cursor = cursor[idx]
            else:
                if is_last:
                    cursor[part] = value
                else:
                    nxt = parts[i + 1]
                    if part not in cursor or cursor[part] is None:
                        cursor[part] = [] if INDEX_RE.match(nxt) else {}
                    cursor = cursor[part]
    return result


def main():
    parser = argparse.ArgumentParser(description="Flatten or unflatten JSON structures")
    parser.add_argument("--mode", choices=["flatten", "unflatten"], default="flatten")
    parser.add_argument("--input", help="Path to a JSON file")
    parser.add_argument("--output", help="Path to write the resulting JSON file")
    parser.add_argument("--sep", default=".", help="Separator for nested keys (default: '.')")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    if args.input:
        data = json.loads(Path(args.input).read_text(encoding="utf-8"))
    else:
        data = DEMO_DATA

    if args.mode == "flatten":
        result = flatten(data, sep=args.sep)
    else:
        if not isinstance(data, dict):
            sys.exit("ERROR: unflatten mode requires a flat JSON object as input")
        result = unflatten(data, sep=args.sep)

    json_text = json.dumps(result, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`flatten()` recurses through the structure, building a compound key as it goes: dict keys are joined with `sep` (default `.`), list items are appended with bracket-index notation (`roles[0]`). When it hits a scalar (anything that isn't a dict or list), it stops recursing and stores the value under the accumulated key. `unflatten()` reverses this by splitting each compound key back into parts with a regex that handles both the separator and bracket boundaries, then walking/creating the nested structure part by part, correctly distinguishing "this part is a list index" from "this part is a dict key."

```bash
# Built-in demo (flatten)
python3 flatten_json.py

# Round trip: flatten then unflatten back to the original shape
python3 flatten_json.py --output flat.json
python3 flatten_json.py --mode unflatten --input flat.json

# Custom separator (e.g. underscore, common for env-var-style flat keys)
python3 flatten_json.py --sep "_"
```

#### Sample output

```
$ python3 flatten_json.py
{
  "user.id": 101,
  "user.name": "Alice",
  "user.address.city": "Lahore",
  "user.address.zip": "54000",
  "user.roles[0]": "admin",
  "user.roles[1]": "editor",
  "active": true
}
```

Round-trip confirmed by direct testing — flattening the demo data and then unflattening the result reproduces the exact original nested structure, including the two-item `roles` array rebuilt correctly from `roles[0]`/`roles[1]`.

#### Variations / flags

- Add a `--max-depth` flag to stop flattening beyond a certain nesting level, useful when only the top few levels need to become columns and deeper structure should stay as a nested JSON blob in one cell.
- Chain with `json_to_csv.py` (24.3), which implements its own simpler flattening internally — for full control over separator and array-handling behavior, flatten explicitly with this script first, then feed the already-flat JSON through a CSV writer directly.
- Use `--sep "_"` when the flattened keys will become environment variable names or SQL column identifiers, since `.` is invalid in both contexts.

#### Common pitfalls

- **Zip codes and other string-typed numbers stay strings** — unlike `csv_to_json.py`'s aggressive type coercion (24.4), this script does no coercion at all; `"zip": "54000"` stays a string through flatten and unflatten, which is usually what you want for identifier-like fields.
- **Ambiguous separators** — if your data itself contains the separator character inside a key name (e.g. a dict key literally named `"user.id"` rather than nested `user` → `id`), flattening produces an indistinguishable compound key, and unflattening will incorrectly split it into nested `user.id`; pick a separator (`--sep`) that can't appear in your actual key names.
- **`unflatten` requires a flat *object*, not an array** — passing a JSON array as `--input` in `--mode unflatten` exits with a clear error (`ERROR: unflatten mode requires a flat JSON object as input`) rather than attempting to guess a reasonable structure; wrap top-level arrays in an object first if needed.

### 24.20 Merge JSON/YAML Data Sources

#### Use case

Production deployment tooling almost universally layers configuration: a `base.yaml` with sane defaults, an `environment.yaml` with per-environment overrides (staging vs. production), and sometimes a `local.json` with machine-specific overrides. A sysadmin building or debugging this kind of layered config system needs a deep merge — not a shallow dict update — so that overriding one nested setting (`server.port`) doesn't wipe out sibling settings (`server.host`) that weren't mentioned in the override file.

#### Prerequisites

Standard library only for JSON-only merges; PyYAML required if any input file is `.yaml`/`.yml`.

```bash
# JSON-only merges need nothing extra
python3 --version

# For YAML inputs:
pip install pyyaml --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

The deep merge in this script does **not** merge lists — a list value in a later file completely replaces a list value in an earlier file, rather than concatenating or merging by index. This is a deliberate, common convention (Kubernetes' strategic merge patches are a notable exception that *does* merge some lists), but verify it matches what your specific tooling expects before relying on it.

#### Script

```python
#!/usr/bin/env python3
"""
merge_configs.py
--------------------
Topic 356: Merge JSON/YAML Data Sources

Combines multiple JSON and/or YAML configuration files using a deep
merge: nested dicts are merged key-by-key, lists and scalars from
later files override earlier ones. Later files in the list win on
conflicts. Requires PyYAML for .yaml/.yml files (see requirements.txt).

USAGE
    # Run built-in demo (merges two built-in sample configs)
    python3 merge_configs.py

    # Merge specific files (order matters: later files override earlier)
    python3 merge_configs.py --inputs base.json override.yaml local.json \\
        --output merged.json

EXPECTED OUTPUT
    The deep-merged configuration, pretty-printed as JSON to the
    terminal (and written to --output if given).
"""
import argparse
import json
import sys
from pathlib import Path

DEMO_BASE = {
    "server": {"host": "0.0.0.0", "port": 8080, "debug": False},
    "features": ["auth", "logging"],
}
DEMO_OVERRIDE = {
    "server": {"port": 9090},
    "features": ["auth", "logging", "metrics"],
    "database": {"engine": "postgresql"},
}


def load_file(path: str):
    text = Path(path).read_text(encoding="utf-8")
    if path.endswith((".yaml", ".yml")):
        try:
            import yaml
        except ImportError:
            sys.exit(
                "ERROR: PyYAML is not installed.\n"
                "Install dependencies first:  pip install -r requirements.txt"
            )
        return yaml.safe_load(text)
    return json.loads(text)


def deep_merge(base: dict, override: dict) -> dict:
    result = dict(base)
    for key, value in override.items():
        if key in result and isinstance(result[key], dict) and isinstance(value, dict):
            result[key] = deep_merge(result[key], value)
        else:
            result[key] = value
    return result


def main():
    parser = argparse.ArgumentParser(description="Deep-merge multiple JSON/YAML config files")
    parser.add_argument("--inputs", nargs="+", help="Paths to config files, in override order")
    parser.add_argument("--output", help="Path to write the merged JSON file")
    parser.add_argument("--indent", type=int, default=2)
    args = parser.parse_args()

    if args.inputs:
        configs = [load_file(p) for p in args.inputs]
    else:
        configs = [DEMO_BASE, DEMO_OVERRIDE]

    merged = {}
    for cfg in configs:
        merged = deep_merge(merged, cfg)

    json_text = json.dumps(merged, indent=args.indent)
    print(json_text)
    if args.output:
        Path(args.output).write_text(json_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`deep_merge()` recursively walks `override`'s keys: if both `base` and `override` have the same key and both values are dicts, it recurses into them and merges *those* dicts key-by-key; otherwise (scalar, list, or type mismatch) the override value wins outright, replacing the base value entirely. `main()` folds this over the whole list of input files in order, starting from an empty `{}`, so later files always take precedence over earlier ones — the same convention as CSS specificity or `dict.update()`, just applied recursively.

```bash
# Built-in demo (merges DEMO_BASE and DEMO_OVERRIDE)
python3 merge_configs.py

# Real layered config: base, then environment override, then local override
python3 merge_configs.py --inputs base.json production.yaml local.json --output merged.json
```

#### Sample output

```
$ python3 merge_configs.py
{
  "server": {
    "host": "0.0.0.0",
    "port": 9090,
    "debug": false
  },
  "features": [
    "auth",
    "logging",
    "metrics"
  ],
  "database": {
    "engine": "postgresql"
  }
}
```

Note `server.host` and `server.debug` (only present in `DEMO_BASE`) survive the merge untouched, while `server.port` is correctly overridden from `8080` to `9090` — this is the deep-merge behavior in action: only the specific nested keys mentioned in the override are touched. Also note `features` (a list) is fully replaced by the override's three-item list rather than combined with the base's two-item list, per the caution above.

#### Variations / flags

- Add a `--list-strategy` flag with `replace` (current behavior) and `append`/`union` options for callers who want list concatenation or deduplication instead of outright replacement.
- Add a `--dry-run`/`--diff` mode that prints only the keys that changed at each merge step, useful for auditing exactly what an override file actually modifies before applying it.
- Chain with `validate_json_schema.py` (24.21) to validate the final merged config before deploying it, catching cases where an override introduced an invalid combination of settings.

#### Common pitfalls

- **Lists don't merge, they replace** — repeated here because it's the single most common surprise: if `production.yaml` only wants to *add* one feature flag to a base list, it must repeat the entire list (all existing items plus the new one), not just the new item, or the base items will be lost.
- **Type mismatches silently override rather than error** — if `base` has `server.port` as an `int` and an override file has `server.port` as a `str` (an easy YAML authoring mistake, e.g. an unquoted value that YAML parses differently than expected), `deep_merge()` doesn't flag the type change — it just takes the override's value as-is. Validate types explicitly downstream if this matters.
- **File extension determines the parser, not content sniffing** — `load_file()` decides JSON vs. YAML purely from the `.json`/`.yaml`/`.yml` suffix; a YAML file saved with a `.json` extension (or vice versa) will be parsed with the wrong loader and likely fail — rename files to match their actual format before merging.

### 24.21 Validate JSON Against JSON Schema

#### Use case

A sysadmin accepting JSON from an external API, a user-submitted webhook payload, or an upstream data feed needs to reject or flag malformed records *before* they reach application code that assumes well-formed input — catching a missing `email` field or a negative `age` at the validation boundary is far cheaper than debugging the resulting crash three layers deeper in a pipeline.

#### Prerequisites

Requires the `jsonschema` package.

```bash
pip install jsonschema --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

This script validates against JSON Schema Draft 7 specifically (`Draft7Validator`); if your schema uses features from a newer draft (2019-09, 2020-12), some keywords may not behave as expected — check which draft your schema was authored against, since JSON Schema's keyword set has evolved across drafts.

#### Script

```python
#!/usr/bin/env python3
"""
validate_json_schema.py
---------------------------
Topic 357: Validate JSON Against JSON Schema

Verifies that JSON data conforms to a JSON Schema (type checking,
required fields, constraints like min/max, patterns, enums, etc).
Requires the `jsonschema` package (see requirements.txt).

USAGE
    # Run built-in demo (validates a sample record against a sample
    # schema, then shows a deliberately invalid record failing)
    python3 validate_json_schema.py

    # Validate your own files
    python3 validate_json_schema.py --schema schema.json --data record.json

EXPECTED OUTPUT
    "VALID" with no errors if the data conforms, or a list of
    validation error messages (path + reason) if it does not.
"""
import argparse
import json
import sys
from pathlib import Path

try:
    from jsonschema import Draft7Validator
except ImportError:
    sys.exit(
        "ERROR: jsonschema is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_SCHEMA = {
    "type": "object",
    "properties": {
        "name": {"type": "string", "minLength": 1},
        "age": {"type": "integer", "minimum": 0},
        "email": {"type": "string", "pattern": r"^[^@]+@[^@]+\.[^@]+$"},
    },
    "required": ["name", "age", "email"],
}

DEMO_VALID = {"name": "Alice", "age": 30, "email": "alice@example.com"}
DEMO_INVALID = {"name": "", "age": -5, "email": "not-an-email"}


def validate(schema: dict, data) -> list:
    validator = Draft7Validator(schema)
    errors = sorted(validator.iter_errors(data), key=lambda e: list(e.path))
    return [
        f"{'.'.join(str(p) for p in err.path) or '(root)'}: {err.message}"
        for err in errors
    ]


def report(label, schema, data):
    print(f"--- {label} ---")
    errors = validate(schema, data)
    if not errors:
        print("VALID: data conforms to the schema.\n")
    else:
        print("INVALID: the following problems were found:")
        for e in errors:
            print(f"  - {e}")
        print()


def main():
    parser = argparse.ArgumentParser(description="Validate JSON data against a JSON Schema")
    parser.add_argument("--schema", help="Path to a JSON Schema file")
    parser.add_argument("--data", help="Path to the JSON data file to validate")
    args = parser.parse_args()

    if bool(args.schema) != bool(args.data):
        sys.exit("ERROR: --schema and --data must be provided together")

    if args.schema and args.data:
        schema = json.loads(Path(args.schema).read_text(encoding="utf-8"))
        data = json.loads(Path(args.data).read_text(encoding="utf-8"))
        report(f"Validating {args.data} against {args.schema}", schema, data)
    else:
        report("Demo: valid record", DEMO_SCHEMA, DEMO_VALID)
        report("Demo: invalid record", DEMO_SCHEMA, DEMO_INVALID)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`Draft7Validator(schema).iter_errors(data)` walks the data against every rule in the schema and yields **every** violation found, not just the first one — the script sorts them by their path within the document (`e.path`) so related errors on the same field group together, then formats each as `field.path: message` for readability. Using `iter_errors()` rather than `validate()` (which raises on the first error and stops) is deliberate: for a data-quality report, you want the complete list of problems in one pass, not one-error-at-a-time debugging.

```bash
# Built-in demo (shows both a valid and a deliberately invalid record)
python3 validate_json_schema.py

# Validate your own schema/data pair
python3 validate_json_schema.py --schema user_schema.json --data incoming_user.json
```

#### Sample output

```
$ python3 validate_json_schema.py
--- Demo: valid record ---
VALID: data conforms to the schema.

--- Demo: invalid record ---
INVALID: the following problems were found:
  - age: -5 is less than the minimum of 0
  - email: 'not-an-email' does not match '^[^@]+@[^@]+\.[^@]+$'
  - name: '' should be non-empty
```

All three deliberate violations in `DEMO_INVALID` (empty name, negative age, malformed email) are caught and reported in a single pass, each with a clear field path and a human-readable reason.

#### Variations / flags

- Add `--exit-code` so the script returns a non-zero exit status on validation failure, making it usable directly as a CI/CD pipeline gate (`python3 validate_json_schema.py --schema s.json --data d.json || exit 1`).
- Switch to a newer draft validator (`jsonschema.Draft202012Validator`) if your schemas use JSON Schema 2020-12 features like `$dynamicRef` or `prefixItems`; `Draft7Validator` predates both.
- Add batch mode (`--data-dir`) to validate every JSON file in a directory against the same schema in one run, useful for auditing an entire feed of incoming records at once.

#### Common pitfalls

- **`required` and `properties` are independent** — a schema can declare a property's type/constraints under `properties` without listing it in `required`; omitting a field not in `required` is valid (the field is simply absent, not violating any type constraint), which surprises people expecting `properties` alone to imply mandatory fields.
- **Pattern regex dialect differences** — JSON Schema's `pattern` keyword uses ECMA 262 (JavaScript) regex syntax conventions, which mostly but not entirely overlaps with Python's `re` module; `jsonschema` uses Python's `re` under the hood, so most patterns work identically, but avoid JS-specific regex features (lookbehind variations, certain Unicode property escapes) if the schema might also be validated by a non-Python tool.
- **`--schema`/`--data` must be provided as a pair** — the script deliberately rejects providing only one of the two (`ERROR: --schema and --data must be provided together`) rather than silently falling back to demo mode, to avoid a typo (forgetting `--data`) accidentally validating demo data against your real schema without any indication something was wrong.

### 24.22 Transform XML with XSLT

#### Use case

A sysadmin receiving XML data feeds from a legacy or enterprise system (SOAP responses, vendor data exports, government reporting formats) often needs to render that XML as a human-readable HTML report, restructure it into a different XML schema for a downstream consumer, or extract a subset of fields — all classic XSLT use cases that predate JSON-era tooling but are still very much alive in enterprise integration work.

This script was not provided in the original upload and was written from scratch for this chapter. It has been executed against its own built-in demo (transforming a `<records>` XML document into an HTML table) and against deliberately malformed XML/XSLT input to confirm the error paths behave correctly.

#### Prerequisites

Requires `lxml`, which bundles `libxslt` (the actual XSLT processing engine) as a compiled dependency.

```bash
pip install lxml --break-system-packages
```

OS scope: Linux, macOS, Windows — `lxml` ships prebuilt wheels for all three, so a plain `pip install` normally avoids needing a C compiler or system `libxml2`/`libxslt` development headers. If a wheel isn't available for your exact platform/Python combination, you may need `apt install libxml2-dev libxslt-dev` (Debian/Ubuntu) or the equivalent before `pip install` will succeed from source.

#### Cautions

`lxml`'s XSLT support implements **XSLT 1.0**, not the newer XSLT 2.0/3.0 (which add features like grouping and regex support, and which lxml/libxslt do not implement). If you're porting a stylesheet written for a 2.0/3.0 processor (like Saxon), expect to need real rework, not just a drop-in swap.

#### Script

```python
#!/usr/bin/env python3
"""
xslt_transform.py
---------------------
Topic 358 (24.22): Transform XML with XSLT

Applies an XSLT 1.0 stylesheet to an XML document and writes the
transformed result (XML, HTML, or plain text -- whatever the
stylesheet's <xsl:output> declares). Requires lxml, which bundles
libxslt (see requirements.txt).

USAGE
    # Run built-in demo (transforms a sample <records> XML document
    # into an HTML table using a built-in stylesheet)
    python3 xslt_transform.py

    # Apply your own stylesheet to your own XML file
    python3 xslt_transform.py --xml data.xml --xslt style.xsl --output out.html

EXPECTED OUTPUT
    The transformed document printed to the terminal (and written to
    --output if given).
"""
import argparse
import sys
from pathlib import Path

try:
    from lxml import etree
except ImportError:
    sys.exit(
        "ERROR: lxml is not installed.\n"
        "Install dependencies first:  pip install -r requirements.txt"
    )

DEMO_XML = """<?xml version="1.0"?>
<records>
  <record>
    <id>1</id>
    <name>Alice</name>
    <city>Lahore</city>
  </record>
  <record>
    <id>2</id>
    <name>Bob</name>
    <city>Karachi</city>
  </record>
</records>
"""

# A minimal XSLT 1.0 stylesheet: turns <records> into an HTML table.
DEMO_XSLT = """<?xml version="1.0"?>
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
  <xsl:output method="html" indent="yes"/>
  <xsl:template match="/records">
    <table border="1">
      <tr><th>ID</th><th>Name</th><th>City</th></tr>
      <xsl:for-each select="record">
        <tr>
          <td><xsl:value-of select="id"/></td>
          <td><xsl:value-of select="name"/></td>
          <td><xsl:value-of select="city"/></td>
        </tr>
      </xsl:for-each>
    </table>
  </xsl:template>
</xsl:stylesheet>
"""


def transform(xml_text: str, xslt_text: str) -> str:
    xml_doc = etree.fromstring(xml_text.encode("utf-8"))
    xslt_doc = etree.fromstring(xslt_text.encode("utf-8"))
    transformer = etree.XSLT(xslt_doc)
    result = transformer(xml_doc)
    if transformer.error_log:
        for entry in transformer.error_log:
            print(f"[xslt warning] {entry}", file=sys.stderr)
    return str(result)


def main():
    parser = argparse.ArgumentParser(description="Apply an XSLT stylesheet to an XML document")
    parser.add_argument("--xml", help="Path to the XML input file")
    parser.add_argument("--xslt", help="Path to the .xsl / .xslt stylesheet")
    parser.add_argument("--output", help="Path to write the transformed output")
    args = parser.parse_args()

    if bool(args.xml) != bool(args.xslt):
        sys.exit("ERROR: --xml and --xslt must be provided together")

    if args.xml and args.xslt:
        xml_text = Path(args.xml).read_text(encoding="utf-8")
        xslt_text = Path(args.xslt).read_text(encoding="utf-8")
    else:
        xml_text, xslt_text = DEMO_XML, DEMO_XSLT

    try:
        output_text = transform(xml_text, xslt_text)
    except etree.XMLSyntaxError as exc:
        sys.exit(f"ERROR: could not parse XML/XSLT: {exc}")
    except etree.XSLTParseError as exc:
        sys.exit(f"ERROR: invalid XSLT stylesheet: {exc}")

    print(output_text)
    if args.output:
        Path(args.output).write_text(output_text, encoding="utf-8")
        print(f"[written to {args.output}]", file=sys.stderr)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`etree.XSLT(xslt_doc)` compiles the stylesheet once into a reusable transformer object; calling that transformer on a parsed XML document (`transformer(xml_doc)`) runs the actual transformation and returns an `_XSLTResultTree`, which the script converts to a plain string with `str(result)`. The stylesheet's own `<xsl:output method="html">` declaration controls the serialization format of the *result* — the same mechanism could instead declare `method="xml"` or `method="text"` for different output shapes, entirely independent of Python code.

```bash
# Built-in demo: transforms a <records> XML document into an HTML table
python3 xslt_transform.py

# Apply your own stylesheet and XML file, write the result to disk
python3 xslt_transform.py --xml data.xml --xslt style.xsl --output out.html
```

#### Sample output

```
$ python3 xslt_transform.py
<table border="1">
<tr>
<th>ID</th>
<th>Name</th>
<th>City</th>
</tr>
<tr>
<td>1</td>
<td>Alice</td>
<td>Lahore</td>
</tr>
<tr>
<td>2</td>
<td>Bob</td>
<td>Karachi</td>
</tr>
</table>
```

Both a malformed-XML input and a `--xml`-without-`--xslt` invocation were tested during development and produce clean, actionable errors rather than raw tracebacks:

```

```

#### Variations / flags

- Add `--params key=value` support using `transformer(xml_doc, key="'value'")` (XSLT stylesheet parameters must be passed as XPath string literals, hence the extra quoting) to pass runtime values into `<xsl:param>` declarations in the stylesheet without editing the stylesheet itself.
- For `method="xml"` stylesheets whose result should be re-parsed as structured XML rather than printed as text, use `etree.tostring(result, pretty_print=True)` instead of `str(result)` for more control over serialization formatting.
- Chain with `csv_to_xml.py` (24.9) or `json_to_xml.py` (24.8) to build a pipeline that starts from CSV/JSON, produces intermediate XML, and finishes with an XSLT-rendered HTML report — useful when the "real" data lives in JSON/CSV but a specific report format is easiest to express as an XSLT template.

#### Common pitfalls

- **XSLT 1.0 only** — as noted in Cautions, `lxml` does not support XSLT 2.0/3.0; stylesheets using `xsl:function`, regex-based `matches()`/`replace()`, or grouping (`xsl:for-each-group`) will fail to compile with `etree.XSLTParseError`, since those are 2.0+ features.
- **Namespace-qualified XML requires namespace-aware XPath in the stylesheet** — if the source XML declares a default namespace (`<records xmlns="http://example.com/ns">`), bare `select="record"` in the stylesheet will silently match nothing (empty output, not an error) unless the stylesheet declares and uses a matching namespace prefix; this is the single most common "my XSLT produces nothing" bug.
- **`lxml` installation from source needs system libraries** — on a minimal/locked-down Linux container without prebuilt wheel support, `pip install lxml` can fail needing `libxml2-dev`/`libxslt-dev` (Debian/Ubuntu naming) or `libxml2-devel`/`libxslt-devel` (RHEL/Fedora naming) plus a C compiler; check for a wheel-compatible Python version first, since this is by far the easiest fix.

### 24.23 Stream Large Datasets Without Loading into Memory

#### Use case

A sysadmin processing a multi-gigabyte export (a full database dump, a day's worth of application logs, a large analytics feed) on a machine with limited RAM needs an approach that never holds the whole dataset in memory at once — `json.load()`/a full CSV-into-a-list read will either crash with `MemoryError` or, more insidiously, silently thrash the machine into swap and grind everything else on the box to a halt.

#### Prerequisites

CSV streaming needs nothing beyond the standard library (`csv.DictReader` is already iterator-based). JSON streaming requires `ijson`.

```bash
pip install ijson --break-system-packages
```

OS scope: Linux, macOS, Windows.

#### Cautions

**A real bug was found in this script's JSON streaming path during testing, not a hypothetical one.** By default, `ijson.items()` returns numeric values as Python `Decimal` objects, not `float` — but `stream_json()` initializes its running total as `total = 0.0` (a `float`) and then does `total += v`. Adding a `Decimal` to a `float` raises `TypeError: unsupported operand type(s) for +=: 'float' and 'decimal.Decimal'`, which means **the JSON streaming demo crashes on a stock `ijson` install with default settings.** This was confirmed by direct execution, not assumed. The fix, confirmed working, is to pass `use_float=True` to `ijson.items()`:

```python
for item in ijson.items(f, "item", use_float=True):
```

Apply this one-line fix before relying on `stream_json()` in production. The CSV path (`stream_csv()`) has no equivalent bug — it already explicitly casts with `float(row["value"])`.

#### Script

```python
#!/usr/bin/env python3
"""
stream_large_dataset.py
---------------------------
Topic 359: Stream Large Datasets Without Loading into Memory

Processes large JSON or CSV files incrementally (row-by-row /
item-by-item) instead of loading the whole file into RAM. JSON
streaming uses `ijson` (see requirements.txt); CSV streaming uses
the standard-library `csv` module, which is already iterator-based.

USAGE
    # Run built-in demo: generates a sample large-ish JSON array file
    # and a CSV file, then streams both while computing simple stats
    python3 stream_large_dataset.py

    # Stream a specific JSON file (must be a top-level JSON array)
    python3 stream_large_dataset.py --format json --input big.json

    # Stream a specific CSV file
    python3 stream_large_dataset.py --format csv --input big.csv

EXPECTED OUTPUT
    Progress is NOT buffered into memory as a whole list; the script
    prints a running count and a couple of aggregate stats (e.g. a
    numeric column's min/max/average) computed with O(1) extra
    memory as it streams, plus peak memory usage for the process.

NOTE (patched):
    stream_json() below passes use_float=True to ijson.items(). The
    original version omitted this, which caused a TypeError when
    adding ijson's default Decimal values to a float accumulator --
    confirmed by direct execution while preparing this chapter.
"""
import argparse
import csv
import json
import sys
import tracemalloc
from pathlib import Path

try:
    import ijson
except ImportError:
    ijson = None


def generate_demo_files(json_path: Path, csv_path: Path, n=5000):
    # JSON: array of {"id": int, "value": float}
    with open(json_path, "w", encoding="utf-8") as f:
        f.write("[\n")
        for i in range(n):
            row = {"id": i, "value": round((i * 37) % 1000 / 3, 2)}
            sep = ",\n" if i < n - 1 else "\n"
            f.write(json.dumps(row) + sep)
        f.write("]\n")

    # CSV: id,value
    with open(csv_path, "w", newline="", encoding="utf-8") as f:
        writer = csv.writer(f)
        writer.writerow(["id", "value"])
        for i in range(n):
            writer.writerow([i, round((i * 37) % 1000 / 3, 2)])


def stream_json(path: Path):
    if ijson is None:
        sys.exit(
            "ERROR: ijson is not installed.\n"
            "Install dependencies first:  pip install -r requirements.txt"
        )
    count = 0
    total = 0.0
    vmin, vmax = None, None
    with open(path, "rb") as f:
        # use_float=True avoids ijson's default Decimal type, which
        # cannot be added to the float accumulator below (see NOTE).
        for item in ijson.items(f, "item", use_float=True):
            count += 1
            v = item.get("value")
            if v is not None:
                total += v
                vmin = v if vmin is None else min(vmin, v)
                vmax = v if vmax is None else max(vmax, v)
    avg = total / count if count else 0
    return count, vmin, vmax, avg


def stream_csv(path: Path):
    count = 0
    total = 0.0
    vmin, vmax = None, None
    with open(path, newline="", encoding="utf-8") as f:
        reader = csv.DictReader(f)
        for row in reader:
            count += 1
            v = float(row["value"])
            total += v
            vmin = v if vmin is None else min(vmin, v)
            vmax = v if vmax is None else max(vmax, v)
    avg = total / count if count else 0
    return count, vmin, vmax, avg


def main():
    parser = argparse.ArgumentParser(description="Stream large JSON/CSV files without loading them fully into memory")
    parser.add_argument("--format", choices=["json", "csv"], help="Which format to stream")
    parser.add_argument("--input", help="Path to the file to stream")
    args = parser.parse_args()

    tracemalloc.start()

    if args.format and args.input:
        path = Path(args.input)
        if args.format == "json":
            count, vmin, vmax, avg = stream_json(path)
        else:
            count, vmin, vmax, avg = stream_csv(path)
        label = f"{args.format.upper()} file: {path}"
    else:
        tmp_dir = Path("./_stream_demo")
        tmp_dir.mkdir(exist_ok=True)
        json_path = tmp_dir / "demo_large.json"
        csv_path = tmp_dir / "demo_large.csv"
        print("No --input given: generating demo files (5000 rows each)...")
        generate_demo_files(json_path, csv_path)

        print("\nStreaming CSV...")
        count, vmin, vmax, avg = stream_csv(csv_path)
        label = f"CSV file: {csv_path}"
        print(f"{label} -> rows={count}, min={vmin}, max={vmax}, avg={avg:.2f}")

        if ijson is not None:
            print("\nStreaming JSON...")
            count, vmin, vmax, avg = stream_json(json_path)
            label = f"JSON file: {json_path}"
        else:
            print("\n(skipping JSON streaming demo: ijson not installed)")
            current, peak = tracemalloc.get_traced_memory()
            print(f"\nPeak memory used during streaming: {peak / 1024:.1f} KB")
            return

    print(f"{label} -> rows={count}, min={vmin}, max={vmax}, avg={avg:.2f}")
    current, peak = tracemalloc.get_traced_memory()
    print(f"\nPeak memory used during streaming: {peak / 1024:.1f} KB")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

For CSV, `csv.DictReader` is already a lazy iterator — it reads and yields one row at a time from the underlying file object, so `stream_csv()` never holds more than one row plus the running `count`/`total`/`vmin`/`vmax` accumulators in memory. For JSON, plain `json.load()` has no equivalent lazy mode (it must parse the entire document tree before returning anything), so `stream_json()` uses `ijson.items(f, "item", ...)`, which parses the JSON token-by-token and yields each `item` in the top-level array as soon as its closing `}` is seen — the file is read incrementally from disk (`open(path, "rb")`), never fully materialized as a Python object.

```bash
# Built-in demo: generates 5000-row JSON and CSV files, streams both
python3 stream_large_dataset.py

# Stream a specific large JSON file
python3 stream_large_dataset.py --format json --input big.json

# Stream a specific large CSV file
python3 stream_large_dataset.py --format csv --input big.csv
```


#### Variations / flags

- Add `--chunk-report N` to print a running progress line every N rows (`if count % N == 0: print(f"...{count} rows so far")`), useful for multi-gigabyte files where a silent multi-minute run is otherwise indistinguishable from a hang.
- For JSON files that are **not** a single top-level array (e.g. newline-delimited JSON / JSON Lines, one object per line), skip `ijson` entirely and iterate the file line-by-line with `json.loads()` per line — much simpler and doesn't need a streaming parser at all, since each line is already a small, independently parseable unit.
- Extend `stream_csv()`/`stream_json()` to write streamed-and-transformed output to a new file incrementally (rather than only computing stats), turning this into a genuine large-file ETL step rather than a read-only analysis pass.

#### Common pitfalls

- **The `Decimal`-vs-`float` bug documented above is the headline pitfall of this script** — if you copy `stream_json()` without the `use_float=True` fix, it will work fine on `count` (an `int`) but crash the moment it tries to accumulate a numeric `value` field; this is exactly the kind of bug that passes a quick smoke test on a tiny file with all-integer values and only surfaces once real decimal data flows through.
- **`ijson`'s `items()` path syntax requires knowing your JSON's shape in advance** — `"item"` matches items of the top-level array; a differently-shaped document (an object with a nested array under a key, e.g. `{"records": [...]}`) needs a different path (`"records.item"`), and getting this wrong produces zero iterations with no error, not a crash — easy to mistake for "the file is empty."
- **`tracemalloc`'s peak-memory figure measures Python-level allocations only** — it does not account for OS-level file-buffer caching or memory used by C extensions like `ijson`'s underlying C parser (`ijson` can be built with a fast `yajl2_c` backend); treat the printed KB figure as a useful relative comparison between runs, not an absolute measurement of total process RSS.
