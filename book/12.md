## Chapter 12: Database Automation

Every system administrator eventually inherits a database — sometimes a single SQLite file sitting quietly on a server, sometimes a sprawling MySQL cluster backing a production application, sometimes a PostgreSQL instance that nobody remembers setting up. Databases don't run themselves. They need to be connected to, queried, backed up, restored, migrated, and fed data from a hundred different CSV exports handed over by finance, marketing, or a vendor's "integration team." Doing any of this by hand, over and over, is how 2 a.m. mistakes happen.

This chapter is about removing the human from the loop wherever it's safe to do so. Instead of opening a GUI tool, clicking through a wizard, and hoping you remembered every step, you'll write small, dependable Python scripts that do the same job every single time — and that you can hand off, schedule, or version-control like any other piece of infrastructure code.

We'll work across the three database engines you're most likely to encounter in day-to-day sysadmin work:

- **SQLite** — the zero-configuration, single-file database that ships with Python's standard library. No server, no daemon, no credentials to leak. Perfect for local tooling, embedded apps, and quick scripts.
- **MySQL** — the workhorse relational database behind a huge share of the web. You'll need a client library (`mysql-connector-python` or `PyMySQL`) and a running server to connect to.
- **PostgreSQL** — the engine of choice for teams that want strict standards compliance, advanced data types, and rock-solid transactional guarantees. We'll use `psycopg2` to talk to it.

The scripts in this chapter are deliberately built the same way the automation scripts in earlier chapters were: they're runnable out of the box (most will generate sample data or a sample database if you don't point them at a real one), they use `argparse` so every behavior is controllable from the command line, and they print clear `[info]`, `[success]`, `[warning]`, and `[error]` messages so you always know what happened.

By the end of this chapter you'll be able to:

- Open a connection to SQLite, MySQL, or PostgreSQL from a script, with credentials handled safely instead of hard-coded.
- Run arbitrary SQL — both one-off queries and multi-statement `.sql` files — and capture the results programmatically.
- Take consistent, restorable backups of a database, whether that means copying a SQLite file safely or shelling out to `mysqldump`/`pg_dump`.
- Restore a database from a backup file without guessing at the right incantation each time.
- Export any table (or every table) to CSV for reporting, auditing, or handing off to a non-technical colleague.
- Import a CSV file into a database table, including handling type inference and duplicate rows.
- Apply schema changes safely and repeatably using a lightweight migration runner, instead of running ad-hoc `ALTER TABLE` statements by hand.
- Understand and configure connection pooling so a busy script or service doesn't exhaust a database's connection limit under load.

A note on scope: this chapter favors the Python standard library (`sqlite3`) wherever possible, and only reaches for third-party packages (`mysql-connector-python`, `psycopg2-binary`, `SQLAlchemy`) when a task genuinely requires talking to a network database server. Every script names its dependencies up front, tells you how to install them, and notes which operating systems it has been designed for. Where a script needs credentials, it reads them from command-line flags or environment variables — never hard-coded into the file — and every script that can destroy data (backups excluded, restores very much included) says so clearly before you run it.


### 12.1 Connect to SQLite

##### Use case

A sysadmin needs a fast, dependency-free way to check that a `.sqlite`/`.db` file is reachable and readable before handing it to a bigger script — for example, confirming a nightly job's database file wasn't corrupted, or peeking at what tables exist on a server you've just SSH'd into.

##### Prerequisites

`sqlite3` ships in the Python standard library, so there is nothing to install for the connection itself. No server process, no credentials, no network port.

```bash
# No installation required — sqlite3 is built into Python 3
python3 --version   # confirm Python 3.x is available
```

OS scope: Linux, Windows, and macOS all behave identically since SQLite is just a local file format.

##### Cautions

- SQLite does not enforce a schema on write for most column types unless you explicitly declare constraints — don't assume the file is "clean" just because it opens.
- Opening a SQLite file that another process is actively writing to can raise `sqlite3.OperationalError: database is locked`; the script below retries briefly but isn't a substitute for proper write coordination.

##### Script

```python
#!/usr/bin/env python3
"""
sqlite_connect.py
==================
Open a SQLite database file, verify the connection works, and print a
summary of the tables it contains (name, column list, row count).

Usage
-----
    python sqlite_connect.py --file company.db

If --file is omitted, a small sample database is created first so the
script works end-to-end without any setup.

Requirements
------------
No third-party packages required (uses the built-in sqlite3 module).
"""

import argparse
import os
import sqlite3


def create_sample_database(path: str) -> None:
    conn = sqlite3.connect(path)
    cur = conn.cursor()
    cur.execute("""
        CREATE TABLE IF NOT EXISTS employees (
            id INTEGER PRIMARY KEY,
            name TEXT NOT NULL,
            department TEXT,
            salary REAL
        )
    """)
    cur.executemany(
        "INSERT INTO employees (name, department, salary) VALUES (?, ?, ?)",
        [
            ("Ayesha Siddiqui", "Finance", 95000),
            ("Kamran Sheikh", "Operations", 71000),
            ("Bilal Ahmed", "Sales", 62000),
        ],
    )
    conn.commit()
    conn.close()
    print(f"[info] No --file given, created a sample database at: {path}")


def connect_and_inspect(path: str, timeout: float) -> None:
    if not os.path.exists(path):
        print(f"[error] File not found: {path}")
        return

    try:
        conn = sqlite3.connect(path, timeout=timeout)
        conn.row_factory = sqlite3.Row
    except sqlite3.Error as exc:
        print(f"[error] Could not open database: {exc}")
        return

    print(f"[success] Connected to: {path}")

    cur = conn.cursor()
    cur.execute("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name")
    tables = [row["name"] for row in cur.fetchall()]

    if not tables:
        print("[info] Database opened successfully but contains no tables.")
        conn.close()
        return

    print(f"Tables found: {tables}\n")
    for table in tables:
        cur.execute(f"PRAGMA table_info({table})")
        columns = [f"{row['name']} ({row['type']})" for row in cur.fetchall()]
        cur.execute(f"SELECT COUNT(*) FROM {table}")
        row_count = cur.fetchone()[0]
        print(f"--- Table: {table} ---")
        print(f"  Columns:   {columns}")
        print(f"  Row count: {row_count}\n")

    conn.close()


def main():
    parser = argparse.ArgumentParser(description="Connect to a SQLite database and print a schema summary.")
    parser.add_argument("--file", default=None, help="Path to the .sqlite/.db file to connect to")
    parser.add_argument("--timeout", type=float, default=5.0, help="Seconds to wait on a locked database before failing")
    args = parser.parse_args()

    path = args.file
    if path is None:
        path = "sample.db"
        create_sample_database(path)

    connect_and_inspect(path, args.timeout)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`sqlite3.connect()` opens (or silently creates) the file at the given path — there's no separate "does this exist" check built into the driver, which is why the script checks `os.path.exists()` first when a real file is expected. `sqlite_master` is SQLite's built-in catalog table; querying it is the standard way to list every table without needing admin privileges. `PRAGMA table_info(table)` returns each column's name and declared type.

```bash
# Inspect a real database file
python sqlite_connect.py --file /var/data/company.db

# Increase the lock-wait timeout for a busy database
python sqlite_connect.py --file /var/data/company.db --timeout 15

# Run with no arguments to see it work against a generated sample
python sqlite_connect.py
```

##### Sample output

```bash
$ python sqlite_connect.py
[info] No --file given, created a sample database at: sample.db
[success] Connected to: sample.db
Tables found: ['employees']

--- Table: employees ---
  Columns:   ['id (INTEGER)', 'name (TEXT)', 'department (TEXT)', 'salary (REAL)']
  Row count: 3
```

##### Variations / flags

- Add `--table employees` plus a `LIMIT 5` preview query if you want row-level output rather than just counts.
- Add `--readonly` and open with `sqlite3.connect(f"file:{path}?mode=ro", uri=True)` when you want to guarantee the script can never write to production data.
- Wrap the connect call in a retry loop (`for attempt in range(3): ... time.sleep(1)`) for scripts that run against files another process writes to frequently.

##### Common pitfalls

- **Relative paths**: running the script from a different working directory silently creates a *new* empty database instead of failing, because `sqlite3.connect()` creates the file if it doesn't exist. Always use absolute paths in production automation.
- **File permissions**: on Linux, a database file owned by another user with `600` permissions will raise `sqlite3.OperationalError: unable to open database file` — check `ls -l` before assuming the driver is broken.
- **Network filesystems**: SQLite's file-locking model is unreliable over NFS/SMB shares; avoid connecting to a `.db` file that lives on one from multiple hosts.

### 12.2 Connect to MySQL

##### Use case

A sysadmin needs to confirm that a MySQL server is reachable and that a given set of credentials actually work — for example, after rotating a database password, or before deploying a script that depends on that server being up.

##### Prerequisites

```bash
pip install mysql-connector-python
```

You also need a running MySQL (or MariaDB) server to connect to; this script does not start one for you. OS scope: Linux, Windows, and macOS are all supported by `mysql-connector-python` since it's a pure client library with no OS-specific binary dependency.

##### Cautions

- Never hard-code passwords into a script that will be committed to version control. This script reads credentials from environment variables or command-line flags, with environment variables preferred.
- A connection left open (not closed) will hold a slot against the server's `max_connections` limit — always close or use a context manager.

##### Script

```python
#!/usr/bin/env python3
"""
mysql_connect.py
=================
Connect to a MySQL/MariaDB server and print server version, connected
database, and a list of tables in that database (if any).

Usage
-----
    python mysql_connect.py --host localhost --user admin --database shop

Credentials can also be supplied via environment variables instead of
flags: MYSQL_HOST, MYSQL_USER, MYSQL_PASSWORD, MYSQL_DATABASE.
Environment variables are used as defaults; explicit flags override them.

Requirements
------------
    pip install mysql-connector-python
"""

import argparse
import getpass
import os
import sys

import mysql.connector
from mysql.connector import Error


def build_connection_kwargs(args) -> dict:
    return {
        "host": args.host or os.environ.get("MYSQL_HOST", "localhost"),
        "port": args.port,
        "user": args.user or os.environ.get("MYSQL_USER"),
        "password": args.password or os.environ.get("MYSQL_PASSWORD"),
        "database": args.database or os.environ.get("MYSQL_DATABASE"),
        "connection_timeout": args.timeout,
    }


def connect_and_inspect(kwargs: dict) -> None:
    if not kwargs.get("user"):
        print("[error] No user supplied. Use --user or set MYSQL_USER.")
        sys.exit(1)

    if kwargs.get("password") is None:
        kwargs["password"] = getpass.getpass(f"MySQL password for {kwargs['user']}: ")

    try:
        conn = mysql.connector.connect(**kwargs)
    except Error as exc:
        print(f"[error] Could not connect to MySQL server: {exc}")
        sys.exit(1)

    print(f"[success] Connected to MySQL server at {kwargs['host']}:{kwargs['port']}")

    cur = conn.cursor()
    cur.execute("SELECT VERSION()")
    version = cur.fetchone()[0]
    print(f"  Server version: {version}")

    if kwargs.get("database"):
        cur.execute("SHOW TABLES")
        tables = [row[0] for row in cur.fetchall()]
        print(f"  Database:       {kwargs['database']}")
        print(f"  Tables:         {tables if tables else '(none)'}")
    else:
        print("  [info] No --database given, skipping table listing.")

    cur.close()
    conn.close()


def main():
    parser = argparse.ArgumentParser(description="Connect to a MySQL server and print a connection summary.")
    parser.add_argument("--host", default=None, help="MySQL host (default: MYSQL_HOST env var, or localhost)")
    parser.add_argument("--port", type=int, default=3306, help="MySQL port")
    parser.add_argument("--user", default=None, help="MySQL username (default: MYSQL_USER env var)")
    parser.add_argument("--password", default=None, help="MySQL password (default: MYSQL_PASSWORD env var, or prompt)")
    parser.add_argument("--database", default=None, help="Database to connect to (default: MYSQL_DATABASE env var)")
    parser.add_argument("--timeout", type=int, default=10, help="Connection timeout in seconds")
    args = parser.parse_args()

    connect_and_inspect(build_connection_kwargs(args))


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script builds its connection arguments from flags first, falling back to environment variables, and only prompts interactively for a password as a last resort — this keeps the script usable both in an interactive terminal and inside a non-interactive cron job or CI pipeline (where you'd export `MYSQL_PASSWORD` beforehand). `SHOW TABLES` is MySQL's equivalent of SQLite's `sqlite_master` query.

```bash
# Fully explicit
python mysql_connect.py --host db.internal --user report_user --database shop

# Using environment variables (good for automation / cron)
export MYSQL_HOST=db.internal
export MYSQL_USER=report_user
export MYSQL_PASSWORD='s3cret'
export MYSQL_DATABASE=shop
python mysql_connect.py

# Non-default port (e.g. a Docker container mapped to 3307)
python mysql_connect.py --host localhost --port 3307 --user root
```

##### Sample output

```bash
$ python mysql_connect.py --host localhost --user report_user --database shop
MySQL password for report_user:
[success] Connected to MySQL server at localhost:3306
  Server version: 8.0.36
  Database:       shop
  Tables:         ['customers', 'orders', 'products']
```

##### Variations / flags

- Add `--ssl-ca /path/to/ca.pem` and pass it through to `mysql.connector.connect(ssl_ca=...)` when connecting to a managed database that requires TLS (e.g. AWS RDS, Cloud SQL).
- Add a `--dry-run` flag that only validates the arguments (host/user present) without actually opening a socket, useful for validating a config file in CI before deploying it.
- Wrap `connect_and_inspect` in a `try/finally` that always calls `conn.close()`, and add a `--retries` flag with exponential backoff for flaky network links.

##### Common pitfalls

- **`Access denied for user`**: usually means either the password is wrong or the user doesn't have a grant for the host you're connecting from — check `SELECT user, host FROM mysql.user;` on the server.
- **`Can't connect to MySQL server`**: the server may be bound to `127.0.0.1` only; connecting via a public hostname will fail even though `localhost` works on the server itself. Check `bind-address` in `my.cnf`.
- **Missing driver**: forgetting `pip install mysql-connector-python` produces a plain `ModuleNotFoundError`, not a MySQL-specific error — easy to misdiagnose as a server problem.
- **Firewalls**: port 3306 is often blocked by default on cloud instances; confirm with `telnet host 3306` or `nc -zv host 3306` before blaming the script.

#### 12.3 Connect to PostgreSQL

##### Use case

A sysadmin needs to verify that a PostgreSQL instance is up, accepting connections, and that a given role can reach a given database — a common first check after a server reboot, a `pg_hba.conf` change, or before running a migration.

##### Prerequisites

```bash
pip install psycopg2-binary
```

`psycopg2-binary` bundles its own `libpq` build so you don't need PostgreSQL's client headers installed system-wide, which makes it the easiest option on all three major OSes. (For production deployments, the non-binary `psycopg2` package built against your system `libpq` is generally preferred, but `psycopg2-binary` is fine for scripting and tooling.)

##### Cautions

- PostgreSQL's default `pg_hba.conf` on many distributions restricts connections to `peer`/`trust` for local Unix-socket connections only — a script connecting over TCP (`host=localhost`) may be rejected even from the same machine until that file is updated.
- As with MySQL, avoid hard-coding passwords; use environment variables or a `.pgpass` file.

##### Script

```python
#!/usr/bin/env python3
"""
postgres_connect.py
=====================
Connect to a PostgreSQL server and print server version, connected
database, and a list of user tables in the "public" schema.

Usage
-----
    python postgres_connect.py --host localhost --user postgres --database shop

Credentials can also be supplied via environment variables:
PGHOST, PGUSER, PGPASSWORD, PGDATABASE (the standard libpq names).

Requirements
------------
    pip install psycopg2-binary
"""

import argparse
import getpass
import os
import sys

import psycopg2


def build_connection_kwargs(args) -> dict:
    return {
        "host": args.host or os.environ.get("PGHOST", "localhost"),
        "port": args.port,
        "user": args.user or os.environ.get("PGUSER"),
        "password": args.password or os.environ.get("PGPASSWORD"),
        "dbname": args.database or os.environ.get("PGDATABASE", "postgres"),
        "connect_timeout": args.timeout,
    }


def connect_and_inspect(kwargs: dict) -> None:
    if not kwargs.get("user"):
        print("[error] No user supplied. Use --user or set PGUSER.")
        sys.exit(1)

    if kwargs.get("password") is None:
        kwargs["password"] = getpass.getpass(f"PostgreSQL password for {kwargs['user']}: ")

    try:
        conn = psycopg2.connect(**kwargs)
    except psycopg2.OperationalError as exc:
        print(f"[error] Could not connect to PostgreSQL server: {exc}")
        sys.exit(1)

    print(f"[success] Connected to PostgreSQL server at {kwargs['host']}:{kwargs['port']}")

    cur = conn.cursor()
    cur.execute("SELECT version()")
    version = cur.fetchone()[0]
    print(f"  Server version: {version}")

    cur.execute("""
        SELECT table_name FROM information_schema.tables
        WHERE table_schema = 'public' ORDER BY table_name
    """)
    tables = [row[0] for row in cur.fetchall()]
    print(f"  Database:       {kwargs['dbname']}")
    print(f"  Tables (public): {tables if tables else '(none)'}")

    cur.close()
    conn.close()


def main():
    parser = argparse.ArgumentParser(description="Connect to a PostgreSQL server and print a connection summary.")
    parser.add_argument("--host", default=None, help="PostgreSQL host (default: PGHOST env var, or localhost)")
    parser.add_argument("--port", type=int, default=5432, help="PostgreSQL port")
    parser.add_argument("--user", default=None, help="PostgreSQL username (default: PGUSER env var)")
    parser.add_argument("--password", default=None, help="PostgreSQL password (default: PGPASSWORD env var, or prompt)")
    parser.add_argument("--database", default=None, help="Database to connect to (default: PGDATABASE env var, or 'postgres')")
    parser.add_argument("--timeout", type=int, default=10, help="Connection timeout in seconds")
    args = parser.parse_args()

    connect_and_inspect(build_connection_kwargs(args))


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`psycopg2.connect()` accepts the same parameter names libpq uses internally (`host`, `port`, `user`, `password`, `dbname`), which is why the script reads the matching `PG*` environment variables `psql` itself understands — a script written this way behaves consistently with the `psql` command line tool your team already knows. `information_schema.tables` is the SQL-standard catalog view (as opposed to Postgres-specific `pg_catalog` views), filtered to the `public` schema to exclude system tables.

```bash
# Fully explicit
python postgres_connect.py --host db.internal --user report_user --database shop

# Using standard libpq environment variables
export PGHOST=db.internal
export PGUSER=report_user
export PGPASSWORD='s3cret'
export PGDATABASE=shop
python postgres_connect.py

# Connecting to a non-default port (e.g. a second Postgres instance)
python postgres_connect.py --host localhost --port 5433 --user postgres
```

##### Sample output

```bash
$ python postgres_connect.py --host localhost --user report_user --database shop
PostgreSQL password for report_user:
[success] Connected to PostgreSQL server at localhost:5432
  Server version: PostgreSQL 16.2 on x86_64-pc-linux-gnu, compiled by gcc...
  Database:       shop
  Tables (public): ['customers', 'orders', 'products']
```

##### Variations / flags

- Add `--sslmode require` and pass through as `psycopg2.connect(sslmode=args.sslmode, ...)` for managed Postgres services that mandate TLS.
- Add `--schema` to inspect a non-`public` schema, changing the `WHERE table_schema = %s` filter accordingly.
- Wrap the connection in `with psycopg2.connect(**kwargs) as conn:` for scripts that run multiple short-lived queries, since psycopg2 handles transaction commit/rollback automatically on context-manager exit.

##### Common pitfalls

- **`FATAL: password authentication failed`**: often means `pg_hba.conf` requires `md5`/`scram-sha-256` but the role was created with no password, or the wrong auth method is configured for that host/user combination — check `pg_hba.conf` and reload with `pg_ctl reload` or `SELECT pg_reload_conf();`.
- **`FATAL: no pg_hba.conf entry for host`**: the server is reachable but its host-based access control rules don't allow this client IP/user/database combination — this is a server-config issue, not a Python issue.
- **Peer vs TCP auth confusion**: `psql` run locally often authenticates via the `peer` method using your OS username, which behaves differently from a Python script connecting over TCP with an explicit `user`/`password` — don't assume "it works in `psql`" means it will work here.
- **`could not translate host name`**: DNS resolution failure for the `--host` value; confirm with `getent hosts <host>` or `nslookup <host>`.

### 12.4 Execute SQL Queries

##### Use case

A sysadmin needs to run a one-off query (or a saved `.sql` script containing several statements) against a database from the command line — for example, checking how many rows changed status overnight, or running a cleanup script as part of a maintenance window — without opening an interactive DB shell.

##### Prerequisites

```bash
# No installation required for the SQLite mode — uses the built-in sqlite3 module
python3 --version
```

This script targets SQLite by default because it requires no server. The same pattern (open connection → `cursor.execute()` → `fetchall()`/`commit()`) applies directly to MySQL (`mysql.connector`) and PostgreSQL (`psycopg2`) connections from sections 12.2 and 12.3 — swap the `sqlite3.connect(...)` call for the appropriate driver's `connect()` call and everything else is unchanged.

##### Cautions

- This script executes whatever SQL it's given, including `DELETE`, `DROP TABLE`, and `UPDATE` statements with no `WHERE` clause. There is no confirmation prompt built in — treat `--file` and `--sql` inputs with the same care you'd give a `rm -rf`.
- `executescript()` (used for multi-statement `.sql` files) implicitly commits any open transaction before running, which can surprise you if you were relying on transactional isolation.

##### Script

```python
#!/usr/bin/env python3
"""
execute_query.py
==================
Execute a single SQL statement (--sql) or a multi-statement .sql file
(--file) against a SQLite database, printing SELECT results as a table
and reporting rows affected for INSERT/UPDATE/DELETE.

Usage
-----
    python execute_query.py --db shop.db --sql "SELECT * FROM customers LIMIT 5"
    python execute_query.py --db shop.db --file cleanup.sql
    python execute_query.py --db shop.db --sql "SELECT * FROM orders" --output results.csv

If --db is omitted, a small sample database is created first.

Requirements
------------
No third-party packages required (uses the built-in sqlite3 module).
"""

import argparse
import csv
import os
import sqlite3
import sys


def create_sample_database(path: str) -> None:
    conn = sqlite3.connect(path)
    conn.execute("""
        CREATE TABLE IF NOT EXISTS customers (
            id INTEGER PRIMARY KEY,
            name TEXT NOT NULL,
            city TEXT,
            active INTEGER DEFAULT 1
        )
    """)
    conn.executemany(
        "INSERT INTO customers (name, city, active) VALUES (?, ?, ?)",
        [
            ("Hassan Raza", "Lahore", 1),
            ("Ayesha Noor", "Karachi", 1),
            ("Bilal Farooq", "Islamabad", 0),
        ],
    )
    conn.commit()
    conn.close()
    print(f"[info] No --db given, created a sample database at: {path}")


def print_select_results(cur: sqlite3.Cursor) -> list:
    columns = [d[0] for d in cur.description]
    rows = cur.fetchall()
    print(" | ".join(columns))
    print("-" * (len(" | ".join(columns))))
    for row in rows:
        print(" | ".join(str(v) for v in row))
    print(f"\n({len(rows)} row(s) returned)")
    return [columns] + [list(r) for r in rows]


def write_csv(data: list, output: str) -> None:
    with open(output, "w", newline="", encoding="utf-8") as f:
        writer = csv.writer(f)
        writer.writerows(data)
    print(f"[success] Results written to CSV: {output}")


def run_single_statement(conn: sqlite3.Connection, sql: str, output: str) -> None:
    cur = conn.cursor()
    try:
        cur.execute(sql)
    except sqlite3.Error as exc:
        print(f"[error] SQL execution failed: {exc}")
        sys.exit(1)

    if cur.description is not None:  # SELECT-like statement
        data = print_select_results(cur)
        if output:
            write_csv(data, output)
    else:
        conn.commit()
        print(f"[success] Statement executed. Rows affected: {cur.rowcount}")


def run_script_file(conn: sqlite3.Connection, path: str) -> None:
    with open(path, "r", encoding="utf-8") as f:
        script = f.read()
    try:
        conn.executescript(script)
    except sqlite3.Error as exc:
        print(f"[error] Script execution failed: {exc}")
        sys.exit(1)
    print(f"[success] Script '{path}' executed successfully (all statements committed).")


def main():
    parser = argparse.ArgumentParser(description="Execute SQL against a SQLite database.")
    parser.add_argument("--db", default=None, help="Path to the SQLite database file")
    parser.add_argument("--sql", default=None, help="A single SQL statement to run")
    parser.add_argument("--file", default=None, help="Path to a .sql file with one or more statements")
    parser.add_argument("--output", default=None, help="Write SELECT results to this CSV path")
    args = parser.parse_args()

    if not args.sql and not args.file:
        print("[error] Provide either --sql or --file.")
        sys.exit(1)

    db_path = args.db
    if db_path is None:
        db_path = "sample.db"
        create_sample_database(db_path)
    elif not os.path.exists(db_path):
        print(f"[error] Database file not found: {db_path}")
        sys.exit(1)

    conn = sqlite3.connect(db_path)

    if args.file:
        run_script_file(conn, args.file)
    else:
        run_single_statement(conn, args.sql, args.output)

    conn.close()


if __name__ == "__main__":
    main()
```

##### How it works and how to run

For a single statement, the script inspects `cursor.description`: it's `None` for statements that don't return rows (`INSERT`/`UPDATE`/`DELETE`/`CREATE`) and populated for anything that does (`SELECT`). That lets one code path handle both read and write queries without the caller having to say which kind it is. For a `.sql` file, `executescript()` runs every statement in the file in order, which is the right tool for maintenance scripts with several sequential steps.

```bash
# Ad-hoc SELECT, printed to the terminal
python execute_query.py --db shop.db --sql "SELECT id, name, city FROM customers WHERE active = 1"

# Ad-hoc UPDATE
python execute_query.py --db shop.db --sql "UPDATE customers SET active = 0 WHERE city = 'Islamabad'"

# Export a query's results straight to CSV
python execute_query.py --db shop.db --sql "SELECT * FROM customers" --output customers.csv

# Run a saved multi-statement maintenance script
python execute_query.py --db shop.db --file cleanup.sql
```

##### Sample output

```bash
$ python execute_query.py --db shop.db --sql "SELECT id, name, city FROM customers WHERE active = 1"
id | name | city
---------------------
1 | Hassan Raza | Lahore
2 | Ayesha Noor | Karachi

(2 row(s) returned)
```

##### Variations / flags

- Add `--params` accepting a JSON array so parameterized queries (`SELECT * FROM customers WHERE city = ?`) can be run safely from the CLI instead of string-formatting user input into SQL.
- Add `--dry-run` for `--file` mode that prints each statement it *would* run (split on `;`) without executing, useful for reviewing a maintenance script before a change window.
- Add `--format json` alongside the existing CSV export for feeding results into another tool.

##### Common pitfalls

- **SQL injection**: never build the `--sql` string by concatenating untrusted input; if a value needs to come from a variable, use `cursor.execute(sql, params)` with `?` placeholders instead of Python string formatting.
- **Silent no-op UPDATE/DELETE**: a `WHERE` clause that matches zero rows won't raise an error — always check the printed "Rows affected" count rather than assuming success from the absence of an exception.
- **Multi-statement files and comments**: `executescript()` handles `--` line comments fine, but a stray unterminated string or missing semicolon will fail the entire script partway through — test `.sql` files against a copy of the database first.
- **Locking**: running a long `--file` script against a SQLite database that another process has open for writing can raise `database is locked`; add a `timeout=` to `sqlite3.connect()` if this happens frequently.

### 12.5 Backup Databases

##### Use case

A sysadmin needs a reliable, scriptable way to take a database backup before a risky change (a migration, a bulk update, a version upgrade) or as part of a scheduled nightly job — one script that handles all three engines consistently instead of remembering three different sets of commands.

##### Prerequisites

```bash
# SQLite backups: no extra install (uses the built-in sqlite3 module's
# online backup API, which is safe to use on a live database)

# MySQL backups: requires the mysqldump client binary (usually installed
# alongside the MySQL server or client package)
sudo apt install mysql-client        # Debian/Ubuntu
brew install mysql-client            # macOS

# PostgreSQL backups: requires the pg_dump client binary
sudo apt install postgresql-client   # Debian/Ubuntu
brew install postgresql              # macOS
```

OS scope: SQLite backup mode works identically everywhere. MySQL/Postgres modes shell out to `mysqldump`/`pg_dump`, so those binaries must be on the system `PATH`; on Windows, install the corresponding client tools and ensure they're added to `PATH` during setup.

##### Cautions

- `mysqldump`/`pg_dump` credentials appear in the process list (`ps aux`) for a brief moment on some systems if passed as plain arguments; this script passes MySQL's password via environment variable (`MYSQL_PWD`) and lets Postgres use `.pgpass`/`PGPASSWORD` for the same reason.
- A backup is only useful if it's tested — periodically restore a backup file (see 12.6) to a scratch database to confirm it's actually valid, not just "a file that exists."

##### Script

```python
#!/usr/bin/env python3
"""
backup_database.py
=====================
Back up a database. Supports three engines:
    sqlite   - uses sqlite3's online backup API (safe on a live DB)
    mysql    - shells out to `mysqldump`
    postgres - shells out to `pg_dump`

Usage
-----
    python backup_database.py --engine sqlite --db shop.db --output backups/shop_2026-07-18.db
    python backup_database.py --engine mysql --db shop --user root --output backups/shop.sql
    python backup_database.py --engine postgres --db shop --user postgres --output backups/shop.dump

If --engine sqlite and --db are both omitted, a sample database is
created first and then backed up, so the script runs end-to-end.

Requirements
------------
    SQLite:     none (standard library)
    MySQL:      mysqldump binary on PATH
    PostgreSQL: pg_dump binary on PATH
"""

import argparse
import datetime
import getpass
import os
import shutil
import sqlite3
import subprocess
import sys


def create_sample_sqlite_db(path: str) -> None:
    conn = sqlite3.connect(path)
    conn.execute("CREATE TABLE IF NOT EXISTS notes (id INTEGER PRIMARY KEY, text TEXT)")
    conn.execute("INSERT INTO notes (text) VALUES ('sample backup target')")
    conn.commit()
    conn.close()
    print(f"[info] No --db given, created a sample SQLite database at: {path}")


def backup_sqlite(db_path: str, output: str) -> None:
    if not os.path.exists(db_path):
        print(f"[error] Database file not found: {db_path}")
        sys.exit(1)

    os.makedirs(os.path.dirname(output) or ".", exist_ok=True)

    source = sqlite3.connect(db_path)
    dest = sqlite3.connect(output)
    with dest:
        source.backup(dest)  # online backup API: safe even if source is being written to
    source.close()
    dest.close()

    size_kb = os.path.getsize(output) / 1024
    print(f"[success] SQLite backup written to: {output} ({size_kb:.1f} KB)")


def backup_mysql(db_name: str, output: str, host: str, port: int, user: str, password: str) -> None:
    os.makedirs(os.path.dirname(output) or ".", exist_ok=True)
    env = os.environ.copy()
    if password:
        env["MYSQL_PWD"] = password  # avoids password appearing in `ps` output

    cmd = ["mysqldump", "-h", host, "-P", str(port), "-u", user, db_name]
    try:
        with open(output, "w", encoding="utf-8") as out_f:
            result = subprocess.run(cmd, stdout=out_f, stderr=subprocess.PIPE, env=env, text=True)
    except FileNotFoundError:
        print("[error] 'mysqldump' not found on PATH. Install the MySQL client tools first.")
        sys.exit(1)

    if result.returncode != 0:
        print(f"[error] mysqldump failed: {result.stderr.strip()}")
        sys.exit(1)

    size_kb = os.path.getsize(output) / 1024
    print(f"[success] MySQL backup written to: {output} ({size_kb:.1f} KB)")


def backup_postgres(db_name: str, output: str, host: str, port: int, user: str, password: str) -> None:
    os.makedirs(os.path.dirname(output) or ".", exist_ok=True)
    env = os.environ.copy()
    if password:
        env["PGPASSWORD"] = password

    cmd = ["pg_dump", "-h", host, "-p", str(port), "-U", user, "-F", "c", "-f", output, db_name]
    try:
        result = subprocess.run(cmd, stderr=subprocess.PIPE, env=env, text=True)
    except FileNotFoundError:
        print("[error] 'pg_dump' not found on PATH. Install the PostgreSQL client tools first.")
        sys.exit(1)

    if result.returncode != 0:
        print(f"[error] pg_dump failed: {result.stderr.strip()}")
        sys.exit(1)

    size_kb = os.path.getsize(output) / 1024
    print(f"[success] PostgreSQL backup written to: {output} ({size_kb:.1f} KB)")


def default_output(engine: str, db_name: str) -> str:
    stamp = datetime.datetime.now().strftime("%Y-%m-%d_%H%M%S")
    ext = {"sqlite": "db", "mysql": "sql", "postgres": "dump"}[engine]
    return os.path.join("backups", f"{db_name}_{stamp}.{ext}")


def main():
    parser = argparse.ArgumentParser(description="Back up a SQLite, MySQL, or PostgreSQL database.")
    parser.add_argument("--engine", choices=["sqlite", "mysql", "postgres"], default="sqlite")
    parser.add_argument("--db", default=None, help="SQLite file path, or MySQL/Postgres database name")
    parser.add_argument("--output", default=None, help="Path for the backup file (default: backups/<db>_<timestamp>.<ext>)")
    parser.add_argument("--host", default="localhost", help="Host for mysql/postgres engines")
    parser.add_argument("--port", type=int, default=None, help="Port for mysql/postgres engines")
    parser.add_argument("--user", default=None, help="Username for mysql/postgres engines")
    parser.add_argument("--password", default=None, help="Password for mysql/postgres engines (prompts if omitted)")
    args = parser.parse_args()

    if args.engine == "sqlite":
        db_path = args.db or "sample.db"
        if args.db is None:
            create_sample_sqlite_db(db_path)
        output = args.output or default_output("sqlite", os.path.splitext(os.path.basename(db_path))[0])
        backup_sqlite(db_path, output)
        return

    if not args.db or not args.user:
        print("[error] --db and --user are required for mysql/postgres engines.")
        sys.exit(1)

    password = args.password
    if password is None:
        password = getpass.getpass(f"Password for {args.user}: ")

    output = args.output or default_output(args.engine, args.db)

    if args.engine == "mysql":
        backup_mysql(args.db, output, args.host, args.port or 3306, args.user, password)
    else:
        backup_postgres(args.db, output, args.host, args.port or 5432, args.user, password)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The SQLite path uses `Connection.backup()` — the "online backup API" built into `sqlite3` — which safely copies the database page-by-page even while it's in use, unlike a naive `shutil.copy()` that can capture a half-written file. The MySQL and PostgreSQL paths shell out to the engines' own official dump tools (`mysqldump`, `pg_dump`) via `subprocess.run()`, because reimplementing a consistent logical backup in pure Python would be reinventing (badly) tools the database vendors already maintain. Postgres backups use the `-F c` "custom" format, which is compressed and required for the selective/parallel restore options covered in 12.6.

```bash
# SQLite — safe to run against a database that's currently in use
python backup_database.py --engine sqlite --db /var/data/shop.db --output backups/shop.db

# MySQL
python backup_database.py --engine mysql --db shop --host db.internal --user backup_user --output backups/shop.sql

# PostgreSQL
python backup_database.py --engine postgres --db shop --host db.internal --user backup_user --output backups/shop.dump

# Run with no arguments to see the SQLite path work against a generated sample
python backup_database.py
```

##### Sample output

```bash
$ python backup_database.py
[info] No --db given, created a sample SQLite database at: sample.db
[success] SQLite backup written to: backups/sample_2026-07-18_143012.db (12.0 KB)
```

##### Variations / flags

- Add a `--compress` flag that pipes `mysqldump`/`pg_dump` output through `gzip` (or uses Postgres's built-in `-F c` compression level) to save space on large databases.
- Add `--schedule` documentation instead of code: schedule this script with `cron` for nightly backups, e.g. `0 2 * * * /usr/bin/python3 /opt/scripts/backup_database.py --engine postgres --db shop --user backup_user --output /backups/shop_$(date +\%F).dump >> /var/log/db_backup.log 2>&1`.
- Add `--retention-days N` to delete backup files in the output directory older than N days after a successful run, so backups don't silently fill the disk.

##### Common pitfalls

- **Permissions on the backup directory**: `mysqldump`/`pg_dump` will fail cryptically if the script can't write to the output path — always confirm the target directory is writable by the account running the backup, especially under cron where the `PATH` and home directory differ from an interactive shell.
- **Missing client binaries under cron**: cron's `PATH` is often much shorter than your login shell's; a script that works fine interactively can fail under cron with `mysqldump: command not found`. Use the full binary path (`/usr/bin/mysqldump`) or set `PATH` explicitly in the crontab.
- **Disk space**: a `pg_dump`/`mysqldump` of a large database can be a large file; always check available disk space before scheduling automated backups, and consider streaming compression.
- **Partial/corrupt backups going unnoticed**: this script checks the subprocess return code, but a `0` exit status doesn't guarantee a *restorable* backup — periodically test restores (12.6) rather than trusting the backup job's log output alone.

### 12.6 Restore Databases

##### Use case

A sysadmin needs to bring a database back from a backup — after a failed migration, accidental data loss, or when standing up a staging environment that should mirror production. This is the highest-stakes script in the chapter: it overwrites whatever is currently in the target database.

##### Prerequisites

```bash
# SQLite: no extra install (standard library)

# MySQL: requires the mysql client binary
sudo apt install mysql-client        # Debian/Ubuntu
brew install mysql-client            # macOS

# PostgreSQL: requires pg_restore (for custom-format dumps from 12.5)
sudo apt install postgresql-client   # Debian/Ubuntu
brew install postgresql              # macOS
```

##### Cautions

- **This script is destructive by design.** Restoring a backup overwrites the target database's current contents. The script requires an explicit `--confirm` flag before it will touch a real (non-sample) target — there is no "undo."
- Always restore to a *new* or *scratch* database first to verify the backup is valid before pointing a restore at anything production actually depends on.

##### Script

```python
#!/usr/bin/env python3
"""
restore_database.py
=====================
Restore a database from a backup file produced by backup_database.py.
Supports three engines:
    sqlite   - copies the backup file over the target file
    mysql    - pipes a .sql dump into `mysql`
    postgres - runs `pg_restore` against a custom-format .dump file

Usage
-----
    python restore_database.py --engine sqlite --backup backups/shop.db --db shop_restored.db --confirm
    python restore_database.py --engine mysql --backup backups/shop.sql --db shop --user root --confirm
    python restore_database.py --engine postgres --backup backups/shop.dump --db shop --user postgres --confirm

The --confirm flag is required for every real restore; running without
it prints what WOULD happen and exits without touching anything.

Requirements
------------
    SQLite:     none (standard library)
    MySQL:      mysql client binary on PATH
    PostgreSQL: pg_restore binary on PATH
"""

import argparse
import getpass
import os
import shutil
import subprocess
import sys


def restore_sqlite(backup_path: str, db_path: str, confirm: bool) -> None:
    if not os.path.exists(backup_path):
        print(f"[error] Backup file not found: {backup_path}")
        sys.exit(1)

    target_exists = os.path.exists(db_path)
    print(f"[plan] Restore SQLite backup '{backup_path}' -> '{db_path}'"
          f"{' (will OVERWRITE existing file)' if target_exists else ' (new file)'}")

    if not confirm:
        print("[info] Dry run only. Re-run with --confirm to actually restore.")
        return

    shutil.copy2(backup_path, db_path)
    size_kb = os.path.getsize(db_path) / 1024
    print(f"[success] SQLite database restored to: {db_path} ({size_kb:.1f} KB)")


def restore_mysql(backup_path: str, db_name: str, host: str, port: int, user: str, password: str, confirm: bool) -> None:
    if not os.path.exists(backup_path):
        print(f"[error] Backup file not found: {backup_path}")
        sys.exit(1)

    print(f"[plan] Restore MySQL dump '{backup_path}' -> database '{db_name}' on {host}:{port}"
          f" (will OVERWRITE matching tables)")

    if not confirm:
        print("[info] Dry run only. Re-run with --confirm to actually restore.")
        return

    env = os.environ.copy()
    if password:
        env["MYSQL_PWD"] = password

    cmd = ["mysql", "-h", host, "-P", str(port), "-u", user, db_name]
    try:
        with open(backup_path, "r", encoding="utf-8") as in_f:
            result = subprocess.run(cmd, stdin=in_f, stderr=subprocess.PIPE, env=env, text=True)
    except FileNotFoundError:
        print("[error] 'mysql' client not found on PATH. Install the MySQL client tools first.")
        sys.exit(1)

    if result.returncode != 0:
        print(f"[error] Restore failed: {result.stderr.strip()}")
        sys.exit(1)

    print(f"[success] MySQL database '{db_name}' restored from: {backup_path}")


def restore_postgres(backup_path: str, db_name: str, host: str, port: int, user: str, password: str, confirm: bool, clean: bool) -> None:
    if not os.path.exists(backup_path):
        print(f"[error] Backup file not found: {backup_path}")
        sys.exit(1)

    print(f"[plan] Restore PostgreSQL dump '{backup_path}' -> database '{db_name}' on {host}:{port}"
          f" (will {'DROP and recreate objects' if clean else 'OVERWRITE matching objects'})")

    if not confirm:
        print("[info] Dry run only. Re-run with --confirm to actually restore.")
        return

    env = os.environ.copy()
    if password:
        env["PGPASSWORD"] = password

    cmd = ["pg_restore", "-h", host, "-p", str(port), "-U", user, "-d", db_name, "--no-owner"]
    if clean:
        cmd.append("--clean")
    cmd.append(backup_path)

    try:
        result = subprocess.run(cmd, stderr=subprocess.PIPE, env=env, text=True)
    except FileNotFoundError:
        print("[error] 'pg_restore' not found on PATH. Install the PostgreSQL client tools first.")
        sys.exit(1)

    if result.returncode != 0:
        print(f"[error] Restore failed: {result.stderr.strip()}")
        sys.exit(1)

    print(f"[success] PostgreSQL database '{db_name}' restored from: {backup_path}")


def main():
    parser = argparse.ArgumentParser(description="Restore a database from a backup file.")
    parser.add_argument("--engine", choices=["sqlite", "mysql", "postgres"], required=True)
    parser.add_argument("--backup", required=True, help="Path to the backup file")
    parser.add_argument("--db", required=True, help="Target SQLite file path, or MySQL/Postgres database name")
    parser.add_argument("--host", default="localhost", help="Host for mysql/postgres engines")
    parser.add_argument("--port", type=int, default=None, help="Port for mysql/postgres engines")
    parser.add_argument("--user", default=None, help="Username for mysql/postgres engines")
    parser.add_argument("--password", default=None, help="Password for mysql/postgres engines (prompts if omitted)")
    parser.add_argument("--clean", action="store_true", help="(postgres only) drop existing objects before restoring")
    parser.add_argument("--confirm", action="store_true", help="Required to actually perform the restore (destructive)")
    args = parser.parse_args()

    if args.engine == "sqlite":
        restore_sqlite(args.backup, args.db, args.confirm)
        return

    if not args.user:
        print("[error] --user is required for mysql/postgres engines.")
        sys.exit(1)

    password = args.password
    if password is None and args.confirm:
        password = getpass.getpass(f"Password for {args.user}: ")

    if args.engine == "mysql":
        restore_mysql(args.backup, args.db, args.host, args.port or 3306, args.user, password, args.confirm)
    else:
        restore_postgres(args.backup, args.db, args.host, args.port or 5432, args.user, password, args.confirm, args.clean)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

Every code path follows the same two-phase pattern: print a `[plan]` line describing exactly what will happen, then stop unless `--confirm` was passed. This mirrors the "dry-run by default" convention used elsewhere in the book and gives you one last readable line to review before anything destructive runs. MySQL restores work by piping the `.sql` text dump into the `mysql` CLI's stdin, exactly as if you'd typed `mysql -u root shop < backup.sql`. PostgreSQL restores use `pg_restore` because the backups from 12.5 use the compressed custom format (`-F c`), which `psql` alone cannot read.

```bash
# Always try a dry run first (no --confirm)
python restore_database.py --engine postgres --backup backups/shop.dump --db shop_staging --user postgres

# SQLite restore to a new file
python restore_database.py --engine sqlite --backup backups/shop.db --db shop_restored.db --confirm

# MySQL restore into an existing (empty) database
python restore_database.py --engine mysql --backup backups/shop.sql --db shop_staging --user root --confirm

# PostgreSQL restore, dropping conflicting objects first
python restore_database.py --engine postgres --backup backups/shop.dump --db shop_staging --user postgres --clean --confirm
```

##### Sample output

```bash
$ python restore_database.py --engine sqlite --backup backups/shop.db --db shop_restored.db
[plan] Restore SQLite backup 'backups/shop.db' -> 'shop_restored.db' (new file)
[info] Dry run only. Re-run with --confirm to actually restore.

$ python restore_database.py --engine sqlite --backup backups/shop.db --db shop_restored.db --confirm
[plan] Restore SQLite backup 'backups/shop.db' -> 'shop_restored.db' (new file)
[success] SQLite database restored to: shop_restored.db (12.0 KB)
```

##### Variations / flags

- Add a `--target-check` step for MySQL/Postgres that runs a lightweight `SELECT 1` connection test against `--db` before restoring, catching typos in the database name before the restore command runs.
- Add `--verify-only` that runs `pg_restore --list` (or greps the `.sql` file for `CREATE TABLE`) to preview what objects a backup contains without restoring anything.
- Schedule periodic "restore drills" with `cron` against a disposable scratch database, and alert if the restore fails — a backup you've never restored is unverified.

##### Common pitfalls

- **Restoring into a non-empty database without `--clean`**: MySQL and Postgres restores can fail partway through with "table already exists" errors if the target database isn't empty and `--clean`/`DROP TABLE` handling isn't used — decide deliberately whether you want a clean restore or a merge.
- **Character set / collation mismatches (MySQL)**: restoring a dump taken with one character set into a database created with a different default can silently mangle non-ASCII text; match `--default-character-set` on both ends.
- **Version skew (PostgreSQL)**: `pg_restore` from a newer client can produce dumps a much older server can't apply, and vice versa; keep client and server major versions reasonably close, or dump in plain SQL format for maximum portability.
- **Forgetting `--confirm` in automation**: if you're calling this script from another script, make sure `--confirm` is being passed intentionally — a dry run inside an automated pipeline will look like a silent no-op success unless you check the output.

### 12.7 Export Database Tables

##### Use case

A sysadmin needs to hand a colleague (or a reporting tool that only speaks CSV) a snapshot of one or more database tables — for a monthly report, an audit request, or as a lightweight alternative to giving someone direct database access.

##### Prerequisites

```bash
# No installation required — uses the built-in sqlite3 and csv modules
python3 --version
```

The same `cursor.execute("SELECT * FROM table")` → `csv.writer` pattern applies unchanged to MySQL or PostgreSQL connections; only the `connect()` call at the top differs (see 12.2/12.3).

##### Cautions

- Exporting an entire large table to CSV loads results a row at a time via `fetchmany()` in this script (not all at once), but the resulting CSV file itself can still be very large — check available disk space for tables with millions of rows.
- CSV has no concept of data types; numbers, dates, and booleans all become plain text on export, which can matter if the CSV is later re-imported (see 12.8) into a database with strict typing.

##### Script

```python
#!/usr/bin/env python3
"""
export_tables.py
===================
Export one table (--table) or every table in a SQLite database
(default) to CSV files.

Usage
-----
    python export_tables.py --db shop.db --table customers --output customers.csv
    python export_tables.py --db shop.db --output-dir exports

If --db is omitted, a small sample database is created first.

Requirements
------------
No third-party packages required (uses the built-in sqlite3 and csv modules).
"""

import argparse
import csv
import os
import sqlite3


def create_sample_database(path: str) -> None:
    conn = sqlite3.connect(path)
    conn.execute("CREATE TABLE IF NOT EXISTS customers (id INTEGER PRIMARY KEY, name TEXT, city TEXT)")
    conn.executemany("INSERT INTO customers (name, city) VALUES (?, ?)", [
        ("Hassan Raza", "Lahore"), ("Ayesha Noor", "Karachi"), ("Bilal Farooq", "Islamabad"),
    ])
    conn.execute("CREATE TABLE IF NOT EXISTS orders (id INTEGER PRIMARY KEY, customer_id INTEGER, total REAL)")
    conn.executemany("INSERT INTO orders (customer_id, total) VALUES (?, ?)", [
        (1, 45.00), (2, 120.50), (1, 30.00),
    ])
    conn.commit()
    conn.close()
    print(f"[info] No --db given, created a sample database at: {path}")


def list_tables(conn: sqlite3.Connection) -> list:
    cur = conn.cursor()
    cur.execute("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name")
    return [row[0] for row in cur.fetchall()]


def export_table(conn: sqlite3.Connection, table: str, output_path: str, batch_size: int = 1000) -> int:
    cur = conn.cursor()
    cur.execute(f"SELECT * FROM {table}")
    columns = [d[0] for d in cur.description]

    os.makedirs(os.path.dirname(output_path) or ".", exist_ok=True)
    row_count = 0
    with open(output_path, "w", newline="", encoding="utf-8") as f:
        writer = csv.writer(f)
        writer.writerow(columns)
        while True:
            batch = cur.fetchmany(batch_size)
            if not batch:
                break
            writer.writerows(batch)
            row_count += len(batch)

    return row_count


def main():
    parser = argparse.ArgumentParser(description="Export SQLite table(s) to CSV.")
    parser.add_argument("--db", default=None, help="Path to the SQLite database file")
    parser.add_argument("--table", default=None, help="Export only this table (default: export all tables)")
    parser.add_argument("--output", default=None, help="Output CSV path when --table is used")
    parser.add_argument("--output-dir", default="exports", help="Directory for per-table CSVs when exporting all tables")
    args = parser.parse_args()

    db_path = args.db
    if db_path is None:
        db_path = "sample.db"
        create_sample_database(db_path)
    elif not os.path.exists(db_path):
        print(f"[error] Database file not found: {db_path}")
        return

    conn = sqlite3.connect(db_path)

    if args.table:
        output = args.output or f"{args.table}.csv"
        row_count = export_table(conn, args.table, output)
        print(f"[success] Table '{args.table}' -> '{output}' ({row_count} rows)")
    else:
        tables = list_tables(conn)
        if not tables:
            print("[info] No tables found in database.")
            conn.close()
            return
        os.makedirs(args.output_dir, exist_ok=True)
        for table in tables:
            output = os.path.join(args.output_dir, f"{table}.csv")
            row_count = export_table(conn, table, output)
            print(f"[success] Table '{table}' -> '{output}' ({row_count} rows)")

    conn.close()


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`export_table()` uses `cursor.fetchmany(batch_size)` in a loop instead of `fetchall()`, so memory usage stays bounded even for a table with millions of rows — the script never holds more than `batch_size` rows in memory at once. The header row comes from `cursor.description`, the same metadata source used in 12.1 and 12.4.

```bash
# Export a single table
python export_tables.py --db shop.db --table customers --output customers.csv

# Export every table in the database into its own CSV
python export_tables.py --db shop.db --output-dir exports

# Run with no arguments to see it work against a generated sample
python export_tables.py
```

##### Sample output

```bash

[success] Table 'customers' -> 'exports/customers.csv' (3 rows)
[success] Table 'orders' -> 'exports/orders.csv' (3 rows)
```

##### Variations / flags

- Add `--where "city = 'Lahore'"` to export a filtered subset of a table instead of the whole thing (`SELECT * FROM {table} WHERE {where_clause}`, with the usual injection caution).
- Add `--batch-size` as a CLI flag instead of a hard-coded default, letting the caller tune memory usage for very wide or very narrow tables.
- Add `--gzip` to compress each output CSV with `gzip.open()` for large exports that need to be transferred or archived.

##### Common pitfalls

- **Table name injection**: table names are interpolated directly into the SQL string (`SELECT * FROM {table}`) since SQLite doesn't support parameterized identifiers — only pass `--table` values from trusted sources, never directly from unvalidated user input.
- **Encoding surprises**: this script writes UTF-8; if the target system (e.g. older Windows Excel) expects a different encoding or a BOM, add `encoding="utf-8-sig"` when opening the output file.
- **NULL vs empty string**: SQLite `NULL` values become an empty CSV field, which is indistinguishable from an actual empty string once exported — document this ambiguity for whoever consumes the file downstream.
- **Very wide tables**: tables with many columns can produce CSV rows that exceed the default field-size expectations of some downstream tools (e.g. certain spreadsheet import wizards); test the first exported file with the actual downstream tool before scheduling ongoing exports.

### 12.8 Import CSV into Databases

##### Use case

A sysadmin regularly receives CSV exports from another system (a vendor, a spreadsheet, another department) that need to land in a database table for querying or joining against existing data — for example, importing a monthly sales export from finance into a reporting database.

##### Prerequisites

```bash
# No installation required — uses the built-in sqlite3 and csv modules
python3 --version
```

##### Cautions

- This script can create a new table with an inferred schema, which is convenient but not a substitute for a reviewed, deliberate schema for anything long-lived — inferred `TEXT`/`INTEGER`/`REAL` types are a best guess based on the first data row.
- Re-running an import against a table that already has rows will duplicate them unless `--mode replace` or a uniqueness constraint is used — decide on your intended behavior (`append`, `replace`, or `upsert`-like dedup) before scheduling this as a repeating job.

##### Script

```python
#!/usr/bin/env python3
"""
import_csv.py
================
Import a CSV file into a SQLite table, creating the table automatically
if it doesn't exist (inferring column types from the first data row).

Usage
-----
    python import_csv.py --file sales.csv --db shop.db --table sales
    python import_csv.py --file sales.csv --db shop.db --table sales --mode replace

If --file is omitted, a sample CSV is generated first.

Requirements
------------
No third-party packages required (uses the built-in sqlite3 and csv modules).
"""

import argparse
import csv
import os
import sqlite3


def create_sample_csv(path: str) -> None:
    content = (
        "order_id,product,quantity,price\n"
        "1001,Wireless Mouse,3,15.00\n"
        "1002,Keyboard,1,30.00\n"
        "1003,USB-C Hub,2,29.00\n"
    )
    with open(path, "w", encoding="utf-8") as f:
        f.write(content)
    print(f"[info] No --file given, created a sample CSV at: {path}")


def infer_type(value: str) -> str:
    if value == "":
        return "TEXT"
    try:
        int(value)
        return "INTEGER"
    except ValueError:
        pass
    try:
        float(value)
        return "REAL"
    except ValueError:
        pass
    return "TEXT"


def ensure_table(conn: sqlite3.Connection, table: str, header: list, sample_row: list, mode: str) -> None:
    cur = conn.cursor()
    cur.execute("SELECT name FROM sqlite_master WHERE type='table' AND name=?", (table,))
    exists = cur.fetchone() is not None

    if exists and mode == "replace":
        cur.execute(f"DROP TABLE {table}")
        exists = False
        print(f"[info] --mode replace: dropped existing table '{table}'")

    if not exists:
        col_defs = []
        for name, sample_value in zip(header, sample_row):
            col_type = infer_type(sample_value)
            safe_name = name.strip().replace(" ", "_")
            col_defs.append(f'"{safe_name}" {col_type}')
        create_sql = f'CREATE TABLE "{table}" (id INTEGER PRIMARY KEY, {", ".join(col_defs)})'
        cur.execute(create_sql)
        conn.commit()
        print(f"[info] Created table '{table}' with inferred schema: {col_defs}")


def import_csv(csv_path: str, db_path: str, table: str, mode: str, batch_size: int = 500) -> None:
    if not os.path.exists(csv_path):
        print(f"[error] CSV file not found: {csv_path}")
        return

    with open(csv_path, newline="", encoding="utf-8") as f:
        reader = csv.reader(f)
        header = next(reader)
        first_row = next(reader, None)
        if first_row is None:
            print("[warning] CSV has a header but no data rows. Nothing to import.")
            return

        conn = sqlite3.connect(db_path)
        ensure_table(conn, table, header, first_row, mode)

        placeholders = ", ".join(["?"] * len(header))
        quoted_cols = ", ".join(f'"{h.strip().replace(" ", "_")}"' for h in header)
        insert_sql = f'INSERT INTO "{table}" ({quoted_cols}) VALUES ({placeholders})'

        cur = conn.cursor()
        rows_written = 0
        batch = [first_row]

        for row in reader:
            batch.append(row)
            if len(batch) >= batch_size:
                cur.executemany(insert_sql, batch)
                rows_written += len(batch)
                batch = []

        if batch:
            cur.executemany(insert_sql, batch)
            rows_written += len(batch)

        conn.commit()
        conn.close()

    print(f"[success] Imported {rows_written} row(s) from '{csv_path}' into table '{table}' ({db_path})")


def main():
    parser = argparse.ArgumentParser(description="Import a CSV file into a SQLite table.")
    parser.add_argument("--file", default=None, help="Path to the input CSV file")
    parser.add_argument("--db", default="import_target.db", help="Path to the target SQLite database")
    parser.add_argument("--table", default="imported_data", help="Target table name")
    parser.add_argument("--mode", choices=["append", "replace"], default="append",
                         help="'append' adds rows to an existing table; 'replace' drops and recreates it")
    args = parser.parse_args()

    path = args.file
    if path is None:
        path = "sample.csv"
        create_sample_csv(path)

    import_csv(path, args.db, args.table, args.mode)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

Type inference looks only at the first data row of the CSV (trying `int`, then `float`, then falling back to `TEXT`), which keeps the script simple but means a column that's numeric in row 1 and has a stray non-numeric value later will fail on insert — a deliberate trade-off for a lightweight import tool rather than a full-blown ETL framework. Rows are inserted in batches via `executemany()` for a large speed improvement over inserting one row at a time, while still capping memory use.

```bash
# Import into a new table, auto-creating the schema
python import_csv.py --file sales.csv --db shop.db --table sales

# Re-import, wiping out the previous contents first
python import_csv.py --file sales.csv --db shop.db --table sales --mode replace

# Run with no arguments to see it work against a generated sample
python import_csv.py
```

##### Sample output

```bash
$ python import_csv.py
[info] No --file given, created a sample CSV at: sample.csv
[info] Created table 'imported_data' with inferred schema: ['"order_id" INTEGER', '"product" TEXT', '"quantity" INTEGER', '"price" REAL']
[success] Imported 3 row(s) from 'sample.csv' into table 'imported_data' (import_target.db)
```

##### Variations / flags

- Add `--unique-column order_id` to build the `INSERT` as `INSERT OR IGNORE`/`INSERT OR REPLACE` against a unique index, giving true upsert-like deduplication on re-import instead of plain `append`.
- Add `--dry-run` that prints the inferred `CREATE TABLE` statement and the first few rows it would insert, without touching the database — useful when handed a CSV of unknown shape.
- Add `--delimiter ';'` and `--encoding latin-1` flags for CSVs exported from tools that don't default to comma-separated UTF-8.

##### Common pitfalls

- **Header/column name collisions with SQL keywords**: a CSV column literally named `order` or `group` will break an unquoted `CREATE TABLE`/`INSERT`; this script quotes every identifier with double quotes specifically to avoid that class of bug — don't remove the quoting if you customize the script.
- **Inconsistent types further down the file**: since type inference only looks at the first row, a later row with `"N/A"` in an otherwise-numeric column will raise a `sqlite3.IntegrityError`/`OperationalError` mid-import; consider validating the whole column first for messy source data.
- **Re-running `append` mode accidentally**: without a uniqueness constraint, running this script twice against the same CSV doubles every row — always confirm `--mode` matches your intent before scheduling repeat imports.
- **Large CSVs and available disk**: SQLite journal files can temporarily grow close to the size of the data being inserted during a large transaction; ensure adequate free disk space beyond just the final database size.

### 12.9 Database Migration Scripts

##### Use case

A sysadmin or small engineering team needs to apply schema changes (adding a column, creating an index, splitting a table) to a database consistently across environments — dev, staging, production — without a heavyweight framework, while still tracking which changes have already been applied so nothing runs twice.

##### Prerequisites

```bash
# No installation required — uses the built-in sqlite3 module
python3 --version
```

For MySQL/PostgreSQL, the same pattern applies: replace the `sqlite3.connect()` call with the appropriate driver from 12.2/12.3, and use a `migrations` tracking table in that engine's dialect (the concept — a table that records which migration files have run — is identical across engines).

##### Cautions

- Migrations should generally be **forward-only** in production: prefer writing a new migration to fix a mistake rather than editing a migration that has already been applied to real environments, since already-run migrations are, by design, never re-executed.
- Always back up the database (12.5) before running a batch of migrations against production for the first time.

##### Script

```python
#!/usr/bin/env python3
"""
db_migrate.py
================
A minimal migration runner for SQLite. Migrations are plain .sql files
in a directory, named with a numeric prefix (0001_create_users.sql,
0002_add_email_index.sql, ...). The runner tracks which migrations
have already been applied in a `_migrations` table and only runs new
ones, in order.

Usage
-----
    python db_migrate.py --db shop.db --migrations-dir migrations --apply
    python db_migrate.py --db shop.db --migrations-dir migrations        # dry run: shows pending migrations only

If --migrations-dir doesn't exist, a sample set of migration files is
created first so the script works end-to-end.

Requirements
------------
No third-party packages required (uses the built-in sqlite3 module).
"""

import argparse
import os
import sqlite3
import sys


SAMPLE_MIGRATIONS = {
    "0001_create_customers.sql": (
        "CREATE TABLE customers (\n"
        "    id INTEGER PRIMARY KEY,\n"
        "    name TEXT NOT NULL,\n"
        "    email TEXT\n"
        ");\n"
    ),
    "0002_add_customers_email_index.sql": (
        "CREATE UNIQUE INDEX idx_customers_email ON customers(email);\n"
    ),
    "0003_add_customers_created_at.sql": (
        "ALTER TABLE customers ADD COLUMN created_at TEXT DEFAULT (datetime('now'));\n"
    ),
}


def create_sample_migrations(migrations_dir: str) -> None:
    os.makedirs(migrations_dir, exist_ok=True)
    for filename, sql in SAMPLE_MIGRATIONS.items():
        with open(os.path.join(migrations_dir, filename), "w", encoding="utf-8") as f:
            f.write(sql)
    print(f"[info] No migrations found, created sample migration files in: {migrations_dir}")


def ensure_migrations_table(conn: sqlite3.Connection) -> None:
    conn.execute("""
        CREATE TABLE IF NOT EXISTS _migrations (
            filename TEXT PRIMARY KEY,
            applied_at TEXT DEFAULT (datetime('now'))
        )
    """)
    conn.commit()


def get_applied(conn: sqlite3.Connection) -> set:
    cur = conn.execute("SELECT filename FROM _migrations")
    return {row[0] for row in cur.fetchall()}


def list_migration_files(migrations_dir: str) -> list:
    files = [f for f in os.listdir(migrations_dir) if f.endswith(".sql")]
    return sorted(files)  # numeric prefix ensures correct ordering


def run_migrations(db_path: str, migrations_dir: str, apply: bool) -> None:
    conn = sqlite3.connect(db_path)
    ensure_migrations_table(conn)
    applied = get_applied(conn)
    all_files = list_migration_files(migrations_dir)
    pending = [f for f in all_files if f not in applied]

    print(f"Database:   {db_path}")
    print(f"Migrations: {migrations_dir}")
    print(f"Applied:    {len(applied)}  |  Pending: {len(pending)}\n")

    if not pending:
        print("[success] Database is up to date. Nothing to do.")
        conn.close()
        return

    for filename in pending:
        status = "will apply" if apply else "pending (dry run)"
        print(f"  [{status}] {filename}")

    if not apply:
        print("\n[info] Dry run only. Re-run with --apply to execute these migrations.")
        conn.close()
        return

    print()
    for filename in pending:
        path = os.path.join(migrations_dir, filename)
        with open(path, "r", encoding="utf-8") as f:
            sql = f.read()
        try:
            conn.executescript(sql)
            conn.execute("INSERT INTO _migrations (filename) VALUES (?)", (filename,))
            conn.commit()
            print(f"[success] Applied: {filename}")
        except sqlite3.Error as exc:
            conn.rollback()
            print(f"[error] Migration '{filename}' failed: {exc}")
            print("[error] Stopping. Fix the migration and re-run; earlier migrations remain applied.")
            conn.close()
            sys.exit(1)

    conn.close()
    print(f"\n[success] All {len(pending)} pending migration(s) applied successfully.")


def main():
    parser = argparse.ArgumentParser(description="Apply pending SQL migrations to a SQLite database in order.")
    parser.add_argument("--db", default="app.db", help="Path to the target SQLite database")
    parser.add_argument("--migrations-dir", default="migrations", help="Directory containing numbered .sql migration files")
    parser.add_argument("--apply", action="store_true", help="Actually run pending migrations (default is dry run)")
    args = parser.parse_args()

    if not os.path.isdir(args.migrations_dir):
        create_sample_migrations(args.migrations_dir)

    run_migrations(args.db, args.migrations_dir, args.apply)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The runner keeps its own bookkeeping table, `_migrations`, inside the target database itself — this is the same core idea used by every popular migration framework (Django, Alembic, Rails' ActiveRecord, Flyway): a table that records which migration filenames have already run, so the tool only ever executes what's new. Filenames are sorted alphabetically, which is why the sample migrations use a zero-padded numeric prefix (`0001_`, `0002_`) — this guarantees execution order matches intent even as the migration count grows past nine or ninety-nine. Each migration runs inside its own commit; if one fails, the script stops immediately rather than attempting later migrations against a schema that didn't get the expected earlier change.

```bash
# Dry run — see what's pending without touching the database
python db_migrate.py --db shop.db --migrations-dir migrations

# Apply all pending migrations, in order
python db_migrate.py --db shop.db --migrations-dir migrations --apply

# Run against a fresh database with no arguments to see the full flow
python db_migrate.py --apply
```

##### Sample output

```bash
$ python db_migrate.py
[info] No migrations found, created sample migration files in: migrations
Database:   app.db
Migrations: migrations
Applied:    0  |  Pending: 3

  [pending (dry run)] 0001_create_customers.sql
  [pending (dry run)] 0002_add_customers_email_index.sql
  [pending (dry run)] 0003_add_customers_created_at.sql

[info] Dry run only. Re-run with --apply to execute these migrations.

$ python db_migrate.py --apply
Database:   app.db
Migrations: migrations
Applied:    0  |  Pending: 3

  [will apply] 0001_create_customers.sql
  [will apply] 0002_add_customers_email_index.sql
  [will apply] 0003_add_customers_created_at.sql

[success] Applied: 0001_create_customers.sql
```

##### Variations / flags

- Add a `--rollback` mode that looks for a matching `_down.sql` file next to each migration (e.g. `0002_add_customers_email_index_down.sql`) and runs the most recently applied one in reverse, for environments that need to undo a bad deploy.
- Add a `--checksum` check that stores a hash of each migration's contents in `_migrations` and warns loudly if an already-applied file's contents have changed since it ran — catching the "someone edited a migration after the fact" mistake mentioned in the cautions above.
- Add `--target 0002_add_customers_email_index.sql` to apply migrations only up to (and including) a specific file, useful for stepping through a large migration set carefully during a change window.

##### Common pitfalls

- **SQLite's limited `ALTER TABLE`**: SQLite only supports a subset of `ALTER TABLE` (adding columns is fine; dropping/renaming columns pre-3.35 requires a rebuild-and-copy dance) — check your SQLite version's capabilities before writing a migration that assumes full `ALTER TABLE` support.
- **Non-idempotent migrations**: a migration that does `CREATE TABLE customers (...)` without `IF NOT EXISTS` will fail loudly (correctly) if run twice by accident outside this tool's bookkeeping — don't rely on `IF NOT EXISTS` as a substitute for the `_migrations` tracking table, since it can mask a migration that only partially applied.
- **Editing an already-applied migration**: because the runner only checks the *filename*, not the file's contents, silently editing a migration that has already run in production will have zero effect there while behaving differently on a fresh environment — always create a new migration file instead.
- **Long-running migrations locking the table**: an `ALTER TABLE` or large `UPDATE` migration on a big production table can hold a lock for longer than expected; test migration duration against a realistic data volume (or a production-sized copy) before running it during business hours.

### 12.10 Database Connection Pooling

##### Use case

A sysadmin is running a script or lightweight service that handles many short-lived database operations concurrently (e.g. a batch job processing hundreds of files in parallel threads, or a small internal web service) and needs to avoid opening a brand-new database connection for every single operation, which is slow and — for network databases — can exhaust the server's `max_connections` limit.

##### Prerequisites

```bash
# No installation required for the demonstration pool below — uses only
# the standard library (sqlite3, queue, threading)

# For MySQL, the connector library ships its own production-grade pool:
pip install mysql-connector-python

# For PostgreSQL, psycopg2 ships a pool implementation too:
pip install psycopg2-binary
```

##### Cautions

- SQLite connections are not safe to share across threads by default (`check_same_thread=True`); this pool works around that by giving each borrowed connection to exactly one thread at a time and setting `check_same_thread=False` on each pooled connection, but you are still responsible for not using a single checked-out connection from two threads simultaneously.
- A pool that's too small serializes work (threads wait for a free connection); a pool that's too large can overwhelm a real database server's `max_connections` setting — size it based on the target database's documented limits, not just the number of threads in your script.

##### Script

```python
#!/usr/bin/env python3
"""
connection_pool.py
=====================
A minimal thread-safe SQLite connection pool, demonstrated by running
several worker threads that each borrow a connection, run a query,
and return it to the pool.

Usage
-----
    python connection_pool.py --db shop.db --pool-size 4 --workers 10

If --db is omitted, a sample database is created first.

Requirements
------------
No third-party packages required (uses the built-in sqlite3, queue,
and threading modules).
"""

import argparse
import os
import queue
import sqlite3
import threading
import time


class SQLiteConnectionPool:
    """A fixed-size pool of SQLite connections, safe to borrow from multiple threads."""

    def __init__(self, db_path: str, pool_size: int):
        self.db_path = db_path
        self._pool = queue.Queue(maxsize=pool_size)
        for _ in range(pool_size):
            conn = sqlite3.connect(db_path, check_same_thread=False, timeout=10)
            self._pool.put(conn)
        self.pool_size = pool_size

    def acquire(self, block_timeout: float = 30.0) -> sqlite3.Connection:
        return self._pool.get(timeout=block_timeout)

    def release(self, conn: sqlite3.Connection) -> None:
        self._pool.put(conn)

    def closeall(self) -> None:
        while not self._pool.empty():
            self._pool.get().close()


def create_sample_database(path: str) -> None:
    conn = sqlite3.connect(path)
    conn.execute("CREATE TABLE IF NOT EXISTS events (id INTEGER PRIMARY KEY, worker TEXT, ts TEXT)")
    conn.commit()
    conn.close()
    print(f"[info] No --db given, created a sample database at: {path}")


def worker_task(pool: SQLiteConnectionPool, worker_id: int, results: list, lock: threading.Lock) -> None:
    start = time.time()
    conn = pool.acquire()
    try:
        conn.execute(
            "INSERT INTO events (worker, ts) VALUES (?, datetime('now'))",
            (f"worker-{worker_id}",),
        )
        conn.commit()
        time.sleep(0.05)  # simulate a small amount of work while holding the connection
    finally:
        pool.release(conn)
    elapsed = time.time() - start
    with lock:
        results.append((worker_id, elapsed))


def run_demo(db_path: str, pool_size: int, worker_count: int) -> None:
    pool = SQLiteConnectionPool(db_path, pool_size)
    print(f"[info] Pool created: {pool_size} connection(s) for {worker_count} worker thread(s)")

    results = []
    lock = threading.Lock()
    threads = [
        threading.Thread(target=worker_task, args=(pool, i, results, lock))
        for i in range(worker_count)
    ]

    started = time.time()
    for t in threads:
        t.start()
    for t in threads:
        t.join()
    total_time = time.time() - started

    pool.closeall()

    avg_wait = sum(r[1] for r in results) / len(results)
    print(f"[success] All {worker_count} workers completed in {total_time:.2f}s using only {pool_size} connection(s)")
    print(f"          Average time per worker (including any queue wait): {avg_wait:.3f}s")


def main():
    parser = argparse.ArgumentParser(description="Demonstrate a thread-safe SQLite connection pool.")
    parser.add_argument("--db", default=None, help="Path to the SQLite database file")
    parser.add_argument("--pool-size", type=int, default=4, help="Number of connections to keep in the pool")
    parser.add_argument("--workers", type=int, default=10, help="Number of concurrent worker threads to simulate")
    args = parser.parse_args()

    db_path = args.db
    if db_path is None:
        db_path = "sample.db"
        create_sample_database(db_path)

    run_demo(db_path, args.pool_size, args.workers)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The pool pre-opens `pool_size` connections at startup and stores them in a `queue.Queue`, which is already thread-safe for exactly this producer/consumer pattern: `acquire()` blocks (up to `block_timeout` seconds) until a connection is available, and `release()` hands it back for the next thread to use. This means the number of *open* database connections stays fixed and small regardless of how many worker threads are running concurrently — ten workers sharing a pool of four connections simply queue for a turn instead of opening ten separate connections.

```bash
# Default: 4 pooled connections serving 10 concurrent workers
python connection_pool.py

# A larger pool for higher concurrency
python connection_pool.py --db shop.db --pool-size 8 --workers 50

# A deliberately small pool to see workers queue for a turn
python connection_pool.py --db shop.db --pool-size 1 --workers 10
```

##### Sample output

```bash
$ python connection_pool.py
[info] No --db given, created a sample database at: sample.db
[info] Pool created: 4 connection(s) for 10 worker thread(s)
[success] All 10 workers completed in 0.16s using only 4 connection(s)
          Average time per worker (including any queue wait): 0.062s
```

##### Variations / flags

- For MySQL, swap this hand-rolled pool for the connector library's built-in one: `mysql.connector.pooling.MySQLConnectionPool(pool_name="mypool", pool_size=5, host=..., user=..., password=..., database=...)`, then call `pool.get_connection()` instead of `pool.acquire()`.
- For PostgreSQL, use `psycopg2.pool.ThreadedConnectionPool(minconn, maxconn, host=..., user=..., ...)`, calling `pool.getconn()`/`pool.putconn(conn)` in place of this script's `acquire()`/`release()`.
- For a longer-running service (rather than a one-shot script), consider `SQLAlchemy`'s engine-level pooling (`create_engine(url, pool_size=5, max_overflow=10)`), which adds health checks (`pool_pre_ping=True`) that automatically discard and replace connections the database has silently closed.

##### Common pitfalls

- **Forgetting to release a connection**: if a worker raises an exception between `acquire()` and `release()` without a `try/finally`, that connection is lost from the pool forever for the life of the process — always release inside a `finally` block, exactly as this script does.
- **Pool size vs. server limits**: a pool sized generously on the client side can still exceed a MySQL/PostgreSQL server's configured `max_connections` if multiple script instances or services share the same database — check the server-side limit, not just what feels reasonable locally.
- **Stale/dead connections**: a connection that's been idle in the pool for a long time can be silently closed by the database server or an intervening firewall/load balancer; production pools (SQLAlchemy, the MySQL/psycopg2 built-in pools) typically offer a "ping before use" or max-idle-time setting — plain hand-rolled pools like the one above don't, and will surface a stale connection as a confusing runtime error on the next query.
- **Thread-safety of the connection itself, not just the pool**: the pool guarantees each connection is only handed to one thread at a time, but if you pass the same borrowed connection object to two different functions running concurrently within that same thread's call stack, you can still cause corruption — the pool solves cross-thread sharing, not misuse within a thread.
