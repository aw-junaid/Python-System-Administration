## Chapter 0: Getting Started with Python for Automation

#### Why This Chapter Exists

Every automation book is tempted to skip the basics and dive straight into
the exciting stuff — scripts that rename ten thousand files, bots that
scrape websites, tools that talk to APIs and shuffle data between systems.
That temptation is understandable, but it's also how readers end up stuck
on page 12 fighting a `ModuleNotFoundError` instead of learning anything
about automation.

This chapter is the runway before takeoff. It exists to make sure that,
regardless of your operating system, your prior experience, or how long
it's been since you last opened a terminal, you and the book are standing
on the same ground. By the end of Chapter 0, you will have:

- A working Python 3.9+ installation, verified and ready to use.
- A clear mental model of virtual environments and why every project in
  this book uses one.
- Comfort installing third-party packages with `pip`.
- A script you wrote yourself, ran from the command line, and — if you're
  on Linux or macOS — made directly executable.
- A refreshed grasp of core Python syntax: variables, types, control flow,
  functions, exceptions, context managers, and modules.
- A working understanding of file paths, previewed subprocess usage, and
  how to structure a script so it behaves like a real command-line tool.
- The ability to read a Python traceback without panicking, and to know
  where to look when the book (or your own code) doesn't have the answer.

#### Who This Chapter Is For

If you already have Python 3.9 or later installed, know what a virtual
environment is, and can explain the difference between `==` and `is` in
your sleep, you can skim this chapter — but skim it. Automation scripts
tend to punish sloppy assumptions about environments and paths more than
typical application code does, because they often run unattended, on
someone else's machine, or on a schedule at 3 a.m. when no one is around
to fix a silent failure. A five-minute refresher now is cheaper than a
confusing bug report later.

If you're newer to Python, or newer to the command line specifically,
read this chapter in full and type out every example. Automation is a
hands-on discipline. You will learn far more by running the code, breaking
it, and fixing it than by reading it passively.

#### What "Automation" Means in This Book

Throughout this book, *automation* refers to using Python to do things a
person would otherwise do by hand: moving and renaming files, filling out
forms, talking to web services, parsing logs, scheduling repetitive tasks,
and orchestrating other command-line tools. This is a deliberately broad
definition, and it means the skills in this chapter — comfortable use of
the terminal, an understanding of exit codes, and a first look at
`subprocess` — matter just as much as the Python language features
themselves. Automation code lives at the boundary between Python and the
operating system, so this chapter spends real time on that boundary.

#### How the Sections Fit Together

The chapter is organized to mirror the actual order you'd follow when
setting up a new machine for this kind of work:

1. **Sections 0.1–0.4** get Python installed, verified, and equipped with
   `pip` and virtual environments — the tools every later chapter assumes
   you already have.
2. **Sections 0.5–0.7** cover the practical mechanics of writing and
   running scripts: choosing an editor, running a file from the command
   line, and turning a script into something that behaves like a native
   command with a shebang line and executable permissions.
3. **Sections 0.8–0.13** are a syntax refresher — not a full introduction
   to Python, but a review of the specific features (control flow,
   functions, exceptions, context managers, modules, and file paths) that
   automation scripts lean on constantly.
4. **Sections 0.14–0.15** give you an early, deliberately partial look at
   `subprocess` and at how to structure a script with a `main()` function
   and meaningful exit codes — full coverage comes later, but you'll see
   the shape of it here first.
5. **Sections 0.16–0.18** step back from the code entirely to cover how to
   read documentation and tracebacks, why this book targets Python 3.9+
   and leaves Python 2 behind, and how to get the most out of the chapters
   that follow, including the companion code repository.

#### A Note on Pace

Don't rush this chapter to get to "the good parts." The good parts — web
scraping, file automation, API integrations, scheduled jobs — all assume
you can install a package without confusion, activate a virtual
environment without thinking about it, and read an error message well
enough to act on it. Chapter 0 is where those assumptions become true.
Take the time now, and every chapter after this one will move faster.

With that said, let's get your environment set up.

### 0.1 Installing Python on Linux, macOS, and Windows

This book targets **Python 3.9 or later**. Before writing a single line of
automation code, you need a working interpreter on your machine. The exact
steps differ by operating system, but the goal is the same everywhere:
get `python3` (or `python` on Windows) available on your command line.

#### Linux

Most Linux distributions ship with Python 3 preinstalled, but often an
older patch version. Check what you have first (see 0.2), then install or
upgrade using your distribution's package manager.

**Debian/Ubuntu:**
```bash
sudo apt update
sudo apt install python3 python3-pip python3-venv
```

**Fedora:**
```bash
sudo dnf install python3 python3-pip
```

**Arch:**
```bash
sudo pacman -S python python-pip
```

If your distribution's repositories only offer an older version than
3.9, consider using [`pyenv`](https://github.com/pyenv/pyenv) to install
a specific version alongside the system Python without disturbing it.
Never remove or replace your system's default Python — many Linux
utilities depend on it internally.

#### macOS

macOS no longer ships Python 2 by default, and modern versions don't
reliably ship Python 3 either. The two common installation paths are:

**Option A — Homebrew (recommended):**
```bash
brew install python
```
This installs a current Python 3 and keeps it updated independently of
Apple's system tools.

**Option B — Official installer:**
Download the macOS installer from
[python.org/downloads](https://www.python.org/downloads/) and run it like
any other application installer.

Avoid relying on `/usr/bin/python3` that may come bundled with Xcode
Command Line Tools for serious project work — it can lag behind current
releases and is tied to Apple's own release schedule.

#### Windows

**Option A — Official installer (recommended for most readers):**
Download the installer from
[python.org/downloads](https://www.python.org/downloads/). During setup,
**check the box labeled "Add python.exe to PATH"** — this single checkbox
prevents the majority of "python is not recognized" errors that new
Windows users encounter.

**Option B — Microsoft Store:**
Python is also available as a Store app, which handles PATH setup
automatically and sandboxes updates, though some automation scripts that
touch low-level system paths behave more predictably with the official
installer instead.

**Option C — Windows Subsystem for Linux (WSL):**
If you'll be following Linux-flavored examples in later chapters (cron
jobs, shell scripting, `chmod`), consider installing WSL and following
the Linux instructions above inside it. Many automation workflows in this
book — especially those involving `subprocess` and shell interop — will
feel more natural in a Linux-like environment.

#### A Quick Sanity Check

Regardless of platform, once installation finishes, open a fresh terminal
window (not one that was already open — it won't see the updated PATH)
and confirm the interpreter is reachable:

```bash
python3 --version
```

On Windows, this is often just `python --version`. If this command fails,
don't move on — resolve it here. Section 0.2 covers verification and the
common `python` vs `python3` naming confusion in more detail.

### 0.2 Verifying Your Installation and Python Version

Once Python is installed, verify it before writing any automation code.
This section also clears up the `python` vs `python3` naming confusion
that trips up nearly every new automation developer at some point.

#### Checking the Version

```bash
python3 --version
```

Expected output looks like:

```
Python 3.11.6
```

This book assumes **3.9 or later**. If your version is older, revisit
0.1 and upgrade before continuing — some syntax used later in the book
(such as certain `pathlib` methods and newer typing syntax) will not run
on older interpreters.

#### `python` vs `python3`

On Linux and macOS, `python3` is the safe, unambiguous command. The bare
`python` command may not exist at all, may point to a Python 2
installation on older systems, or may point to Python 3 depending on how
things were installed. On Windows, the official installer typically
provides `python` directly.

To avoid ambiguity in your own scripts and shell history, this book uses
`python3` in all Linux/macOS examples and `python` in Windows-specific
examples. If you're unsure which command maps to which version on your
system, check both:

```bash
python --version
python3 --version
```

#### Locating the Interpreter

Sometimes `--version` succeeds but you're not sure *which* Python you're
actually running — this matters when multiple installations exist side
by side (e.g., a Homebrew Python and an Anaconda Python on the same Mac).

```bash
which python3        # Linux/macOS
where python          # Windows
```

This prints the full path to the interpreter that will run when you type
the command. If the output surprises you (pointing to somewhere you
didn't expect), your PATH is picking up a different installation than you
intended — worth resolving now rather than debugging later.

#### Verifying `pip` Is Present

Automation work depends heavily on third-party packages, so confirm `pip`
came along with your installation:

```bash
python3 -m pip --version
```

If this fails, see 0.4 for installing or repairing `pip`.

#### A One-Line Health Check

As a final check, run a trivial script directly from the command line to
confirm the interpreter actually executes code, not just reports a
version string:

```bash
python3 -c "print('Python is working')"
```

If that prints `Python is working`, your installation is ready, and you
can move on to virtual environments in 0.3.

### 0.4 Installing Packages with `pip`

`pip` is Python's package installer, and it's how you'll pull in nearly
every third-party library used later in this book — `requests` for HTTP,
`beautifulsoup4` for parsing HTML, `openpyxl` for spreadsheets, and many
others. This section covers the everyday commands you'll use constantly.

#### Installing a Package

With your virtual environment activated (see 0.3):

```bash
pip install requests
```

To install a specific version:

```bash
pip install requests==2.31.0
```

To install at least a minimum version:

```bash
pip install "requests>=2.28"
```

#### Listing What's Installed

```bash
pip list
```

#### Checking Details on a Specific Package

```bash
pip show requests
```

This displays the installed version, location, and dependencies.

#### Uninstalling a Package

```bash
pip uninstall requests
```

#### `requirements.txt`: Recording Your Dependencies

A `requirements.txt` file lists every package a project needs, so anyone
(including future-you, on a different machine) can recreate the exact
environment.

**Generating one from your current environment:**

```bash
pip freeze > requirements.txt
```

This produces a file like:

```
requests==2.31.0
beautifulsoup4==4.12.2
openpyxl==3.1.2
```

**Installing from one:**

```bash
pip install -r requirements.txt
```

This book's companion repository includes a `requirements.txt` for every
chapter that needs one — running this single command inside a fresh
virtual environment gets you fully set up for that chapter.

#### Upgrading `pip` Itself

```bash
python3 -m pip install --upgrade pip
```

Running `pip` through `python3 -m pip` rather than calling `pip`
directly is a good habit — it guarantees you're using the `pip` tied to
the Python interpreter you actually intend, which matters when multiple
Python installations exist on one machine.

#### Upgrading a Package

```bash
pip install --upgrade requests
```

#### A Word of Caution: Global Installs

If you ever see instructions online suggesting `sudo pip install
<package>`, be wary. Installing packages globally with elevated
permissions can conflict with your operating system's own Python tooling
and is almost never necessary. Every example in this book assumes
packages are installed inside an activated virtual environment instead.

#### Where Packages Come From

By default, `pip` installs from the [Python Package Index
(PyPI)](https://pypi.org), the official public repository for
Python packages. Before installing an unfamiliar package, it's worth a
quick look at its PyPI page — checking its maintenance activity, download
counts, and documentation — the same way you'd vet any third-party
dependency in another language's ecosystem.

### 0.5 Choosing an Editor or IDE for Automation Scripts

Automation scripts are usually small-to-medium sized programs that talk
to the file system, the network, and other command-line tools. You don't
need a heavyweight IDE to write them, but a few features make the work
noticeably smoother. This section is a practical guide, not an argument
for one "correct" editor.

#### What Actually Matters for This Kind of Work

- **An integrated terminal.** You'll be running scripts constantly while
  editing them. Switching windows every time breaks flow.
- **Good debugging support.** Being able to set a breakpoint and inspect
  variables saves enormous time when a script behaves unexpectedly partway
  through a long file operation or API call.
- **Linting/formatting hooks.** Catching a typo or an unclosed file handle
  before you run the script is faster than catching it after.
- **Lightweight enough to open instantly.** Automation scripts are often
  short-lived — you write one, run it, tweak it, done. A slow-starting
  editor adds friction to that loop.

#### Recommended Options

**Visual Studio Code (recommended default for this book)**
Free, cross-platform, with an official Python extension providing
IntelliSense, integrated debugging, and a built-in terminal. Most
screenshots and workflow descriptions in this book assume VS Code, but
nothing in the actual Python code depends on it.

**PyCharm (Community Edition)**
A more full-featured, Python-specific IDE with strong refactoring tools
and a polished debugger. A good choice if you're coming from a Java or
other JetBrains-ecosystem background, or if you want deeper static
analysis out of the box.

**Vim / Neovim**
For readers already comfortable in a terminal-based workflow, Neovim with
a plugin like `coc.nvim` or a Python LSP client provides a fast, keyboard-
driven environment well suited to remote server work — common in
automation contexts where scripts run on headless machines.

**Jupyter Notebooks**
Excellent for *exploring* an API response or experimenting with a parsing
approach interactively, but not recommended as the primary way to write
the scripts in this book. Automation code needs to run unattended and
predictably from top to bottom — a notebook's out-of-order execution
model works against that goal. Feel free to prototype in a notebook, then
move the finished logic into a `.py` file.

#### Minimum Setup Regardless of Editor

Whichever tool you choose, configure:

1. **Syntax highlighting for Python** (virtually automatic in all options
   above).
2. **A formatter**, such as `black`, so code style stays consistent
   without manual effort:
   ```bash
   pip install black
   black my_script.py
   ```
3. **A linter**, such as `ruff` or `flake8`, to catch unused imports,
   undefined names, and other easy-to-miss mistakes before runtime.

#### The Real Answer

The best editor for this book is the one you'll actually keep open. Don't
spend your first week evaluating five different tools — pick one from
the list above, move on to 0.6, and revisit your choice later if
something is genuinely getting in your way.

### 0.6 Running a Python Script from the Command Line

Automation scripts are, almost by definition, meant to be run from the
command line — often unattended, sometimes on a schedule, sometimes
triggered by another program. This section covers the basic mechanics.

#### Writing and Running Your First Script

Create a file named `hello.py`:

```python
print("Hello from an automation script")
```

Run it:

```bash
python3 hello.py
```

That's the entire loop you'll repeat thousands of times throughout this
book: edit the file, run it, read the output, repeat.

#### Passing Arguments

Command-line arguments let a script behave differently without editing
its code — essential for reusable automation tools.

```python
# greet.py
import sys

name = sys.argv[1] if len(sys.argv) > 1 else "world"
print(f"Hello, {name}")
```

```bash
python3 greet.py Alice
# Hello, Alice
```

`sys.argv` is a list of strings; `sys.argv[0]` is always the script's own
name. Later chapters introduce the `argparse` module for handling
arguments more robustly — this preview is enough to get started.

#### Running a Script from a Different Directory

You don't need to `cd` into a script's folder to run it — provide a path:

```bash
python3 scripts/cleanup.py
python3 /home/alex/tools/report.py
```

Just be aware that the script's *working directory* (used for relative
file paths inside the script) is wherever you ran the command from, not
where the script file lives. This distinction causes a surprising number
of "file not found" bugs in automation scripts — Section 0.13 covers a
more robust way to handle this using `pathlib`.

#### Passing Input via Standard Input

Many command-line tools are designed to receive data piped in from
another program:

```bash
cat data.txt | python3 process.py
```

Inside the script, `sys.stdin` gives you access to that piped input,
which you'll use in later chapters when chaining automation scripts
together with other Unix tools.

#### Redirecting Output

```bash
python3 report.py > output.txt      # save standard output to a file
python3 report.py 2> errors.txt     # save error output separately
python3 report.py > out.txt 2>&1    # save both to the same file
```

This matters for automation specifically because scheduled or unattended
scripts often have nowhere for their output to go except a log file —
you'll rely on this pattern heavily once cron jobs and scheduled tasks
show up in later chapters.

#### Checking Whether It Worked

After a script finishes, its exit code tells you whether it succeeded:

```bash
python3 my_script.py
echo $?          # Linux/macOS: 0 usually means success
```

```powershell
python my_script.py
echo $LASTEXITCODE   # PowerShell equivalent
```

Exit codes are a small detail with outsized importance in automation,
since they're often the *only* way a scheduler or a parent script knows
whether your script succeeded. Section 0.15 covers how to set them
deliberately.

### 0.7 Making a Script Executable (`#!/usr/bin/env python3` and `chmod +x`)

On Linux and macOS, you can turn a Python script into something that runs
like a native command — no need to type `python3` before it every time.
This section shows how, and explains why it works.

#### Step 1: Add a Shebang Line

The **shebang** is a special first line in a script that tells the shell
which interpreter should run the file. Add this as the very first line:

```python
#!/usr/bin/env python3

print("Running as an executable script")
```

#### Why `/usr/bin/env python3` Instead of a Direct Path

You'll sometimes see older examples using a hardcoded path like
`#!/usr/bin/python3`. This book uses `#!/usr/bin/env python3` instead,
because it looks up `python3` using your shell's `PATH` — the same
`python3` you verified in 0.2 — rather than assuming a fixed install
location. This matters especially when working inside a virtual
environment or on a system where Python lives somewhere nonstandard.

#### Step 2: Make the File Executable

The shebang line only takes effect if the operating system knows the
file is meant to be executed. Grant that permission with `chmod`:

```bash
chmod +x greet.py
```

You can confirm the permission was added:

```bash
ls -l greet.py
```

```
-rwxr-xr-x  1 alex  staff  58 Jul 18 10:03 greet.py
```

The `x` characters in the permission string indicate execute permission
for the owner, group, and others.

#### Step 3: Run It Directly

```bash
./greet.py
```

Note the `./` prefix — it tells the shell to look for the script in the
current directory rather than searching `PATH`, which (correctly) doesn't
include your working directory by default for security reasons.

#### Making It Available Anywhere

If you want to run the script by name alone, from any directory, place it
in a folder that's already on your `PATH` (and drop the `.py` extension,
if you like, for a cleaner command name):

```bash
mv greet.py ~/bin/greet
chmod +x ~/bin/greet
greet
```

(This assumes `~/bin` exists and is included in your `PATH` — check with
`echo $PATH`, and add it in your shell's startup file, e.g., `~/.bashrc`
or `~/.zshrc`, if it isn't.)

#### A Note for Windows Users

Windows doesn't use the shebang/`chmod` mechanism — file execution there
is determined by file association and extension. Windows automation
scripts are typically run with `python script.py` directly, or packaged
as a `.bat` or PowerShell wrapper if you want a bare command name. If
you're following along on Windows, you can skip the executable-permission
mechanics in this section, though it's worth understanding for any
scripts you run inside WSL or deploy to a Linux server later in the book.

#### Why This Matters for Automation

Many of the tools you'll build in later chapters are meant to be invoked
by something else — a cron job, a systemd timer, another script — rather
than run interactively. Those callers generally expect to execute a file
directly, not to know or care that it happens to be Python underneath.
A correct shebang line is what makes that possible.

### 0.8 Python Syntax Refresher: Variables, Types, and Control Flow

This section is a fast review, not a first introduction to programming.
If any of it feels genuinely new rather than familiar, consider working
through a beginner-focused Python resource before continuing, since later
chapters build on this material quickly.

#### Variables and Dynamic Typing

Python variables don't need declared types — the type is attached to the
value, not the name:

```python
count = 5            # int
label = "servers"    # str
ratio = 0.75          # float
active = True         # bool
servers = None        # NoneType
```

Reassigning a variable to a different type is legal, if sometimes
confusing in longer scripts:

```python
count = 5
count = "five"   # allowed, but often a sign of a naming problem
```

#### Core Built-in Types

| Type    | Example              | Notes                              |
|---------|----------------------|-------------------------------------|
| `str`   | `"log.txt"`          | Immutable sequence of characters    |
| `int`   | `42`                  | Arbitrary precision                 |
| `float` | `3.14`                | Standard double-precision           |
| `bool`  | `True` / `False`      | Subclass of `int`                   |
| `list`  | `[1, 2, 3]`           | Mutable, ordered                    |
| `tuple` | `(1, 2, 3)`           | Immutable, ordered                  |
| `dict`  | `{"key": "value"}`    | Mutable key-value mapping           |
| `set`   | `{1, 2, 3}`           | Mutable, unordered, unique elements |

Automation scripts lean heavily on `list` and `dict` in particular —
lists of file paths, dictionaries of configuration values or parsed JSON
responses.

#### f-strings

Formatted string literals are the standard way to build strings in
modern Python, and appear constantly in this book's examples:

```python
name = "backup_job"
count = 12
print(f"{name} processed {count} files")
```

#### Control Flow

**Conditionals:**
```python
if count == 0:
    print("Nothing to process")
elif count < 10:
    print("Small batch")
else:
    print("Large batch")
```

**`for` loops** (Python's for loop iterates over items, not indices):
```python
for filename in ["a.txt", "b.txt", "c.txt"]:
    print(filename)
```

Use `enumerate()` when you need the index too:
```python
for i, filename in enumerate(["a.txt", "b.txt"]):
    print(i, filename)
```

**`while` loops:**
```python
attempts = 0
while attempts < 3:
    attempts += 1
```

**Loop control:**
```python
for item in items:
    if item is None:
        continue      # skip this iteration
    if item == "stop":
        break          # exit the loop entirely
```

#### Truthiness

Automation code frequently checks values for "emptiness" rather than
comparing explicitly:

```python
files = []
if not files:
    print("No files found")
```

Empty strings, empty lists/dicts/sets, `0`, `0.0`, and `None` are all
falsy; nearly everything else is truthy. This is idiomatic Python, but
be careful with numeric values where `0` is a meaningful result rather
than an absence of one.

#### Comments

```python
# This explains why, not what — the code already shows what
retry_count = 3  # API occasionally times out; retry a few times
```

### 0.9 Functions, Arguments, and Return Values

Functions are how automation scripts stay readable as they grow past a
few dozen lines. This section reviews the syntax and a few patterns
you'll see repeatedly in later chapters.

#### Basic Definition and Call

```python
def greet(name):
    return f"Hello, {name}"

message = greet("Alex")
print(message)
```

#### Default Arguments

```python
def connect(host, port=443, timeout=10):
    print(f"Connecting to {host}:{port} (timeout={timeout}s)")

connect("example.com")                  # uses defaults
connect("example.com", port=8080)       # overrides one default
```

Default argument values are evaluated **once**, when the function is
defined — a classic gotcha when the default is a mutable object:

```python
# Don't do this:
def add_item(item, bucket=[]):   # the same list is reused across calls!
    bucket.append(item)
    return bucket

# Do this instead:
def add_item(item, bucket=None):
    if bucket is None:
        bucket = []
    bucket.append(item)
    return bucket
```

#### Positional vs. Keyword Arguments

```python
def resize(width, height):
    ...

resize(800, 600)                 # positional
resize(width=800, height=600)    # keyword — order doesn't matter
resize(height=600, width=800)    # equally valid
```

Keyword arguments make calls to functions with several parameters far
more readable, especially when some are booleans or numbers whose
meaning isn't obvious from the value alone.

#### `*args` and `**kwargs`

Used when a function needs to accept a variable number of arguments —
common in wrapper functions and decorators later in the book:

```python
def run_command(*args, **kwargs):
    print("Positional:", args)
    print("Keyword:", kwargs)

run_command("ls", "-la", cwd="/tmp", check=True)
# Positional: ('ls', '-la')
# Keyword: {'cwd': '/tmp', 'check': True}
```

#### Return Values

A function without an explicit `return` returns `None`:

```python
def log_message(msg):
    print(msg)
    # implicitly returns None

result = log_message("done")
print(result)   # None
```

Functions can return multiple values as a tuple:

```python
def get_dimensions():
    return 1920, 1080

width, height = get_dimensions()
```

#### Type Hints (Previewed)

This book uses type hints in many examples for clarity, though Python
doesn't enforce them at runtime:

```python
def greet(name: str) -> str:
    return f"Hello, {name}"
```

Hints don't change how the function runs — they're documentation that
editors and tools like `mypy` can check for you. Treat them as a helpful
habit rather than a requirement while you're getting comfortable.

#### Docstrings

```python
def parse_log_line(line: str) -> dict:
    """Parse a single log line into a dictionary of fields.

    Args:
        line: A raw line of text from a log file.

    Returns:
        A dictionary with keys like 'timestamp', 'level', and 'message'.
    """
    ...
```

Docstrings matter more in automation code than they might seem to at
first — scripts written to solve a one-off problem have a habit of
outliving that problem and getting reused six months later, often by
someone (including future-you) who has forgotten exactly what a function
does.

### 0.10 Working with Exceptions and `try`/`except`

Automation scripts run into failure constantly and often by design: a
file might not exist, a server might not respond, a config value might be
missing. Handling these situations gracefully — rather than letting the
whole script crash — is one of the most important habits this book will
reinforce.

#### Basic Syntax

```python
try:
    result = 10 / 0
except ZeroDivisionError:
    print("Can't divide by zero")
```

#### Catching Specific Exceptions

Always catch the most specific exception type you can, rather than a bare
`except:`. This avoids accidentally hiding bugs unrelated to the problem
you're actually handling:

```python
try:
    with open("config.json") as f:
        data = f.read()
except FileNotFoundError:
    print("Config file is missing — using defaults")
```

#### Catching Multiple Exception Types

```python
try:
    response = fetch_data(url)
except (ConnectionError, TimeoutError) as e:
    print(f"Network problem: {e}")
```

#### The `else` and `finally` Clauses

```python
try:
    f = open("data.txt")
except FileNotFoundError:
    print("File missing")
else:
    print("File opened successfully")   # runs only if no exception occurred
    f.close()
finally:
    print("Done attempting to open the file")  # always runs
```

`finally` is especially useful for cleanup code — closing a connection or
releasing a lock — that must run whether or not something went wrong.
(Section 0.11 covers `with` statements, which handle this more elegantly
in most cases.)

#### Accessing Exception Details

```python
try:
    value = int("not a number")
except ValueError as e:
    print(f"Conversion failed: {e}")
```

#### Raising Your Own Exceptions

```python
def set_retry_count(n):
    if n < 0:
        raise ValueError("retry count cannot be negative")
    return n
```

#### Custom Exception Classes

For larger automation tools, defining your own exception types makes
error handling in calling code more precise:

```python
class ConfigError(Exception):
    """Raised when a configuration file is invalid or missing required keys."""

def load_config(path):
    if not path.exists():
        raise ConfigError(f"No config file found at {path}")
```

#### A Common Automation Pattern: Fail Loudly, Not Silently

A tempting but risky pattern:

```python
try:
    risky_operation()
except Exception:
    pass   # silently swallows every possible error — avoid this
```

This hides real problems and makes debugging painful weeks later when
something has quietly been failing the whole time. Prefer logging the
error, even if you choose to continue:

```python
try:
    risky_operation()
except Exception as e:
    print(f"Warning: risky_operation failed: {e}")
```

Later chapters replace these `print` calls with the `logging` module for
anything running unattended, but the underlying principle — don't hide
failures from yourself — holds throughout the entire book.


### 0.11 Context Managers and the `with` Statement

Automation scripts constantly open things that need to be closed:
files, network connections, database sessions, subprocess pipes. The
`with` statement is Python's built-in mechanism for guaranteeing that
cleanup happens — even if an error occurs partway through.

#### The Problem It Solves

Without `with`, you'd need to manage cleanup manually:

```python
f = open("data.txt")
try:
    contents = f.read()
finally:
    f.close()
```

This works, but it's easy to forget the `finally`, and it gets repetitive
fast when a script opens dozens of files or connections.

#### The `with` Statement

```python
with open("data.txt") as f:
    contents = f.read()
# f.close() is called automatically here, even if an exception occurred
```

This is the idiomatic way to open files in Python, and you'll see it in
nearly every script in this book that touches the file system.

#### Multiple Context Managers at Once

```python
with open("input.txt") as infile, open("output.txt", "w") as outfile:
    outfile.write(infile.read().upper())
```

#### What "Context Manager" Actually Means

Any object that implements two special methods — `__enter__` and
`__exit__` — can be used with `with`. `__enter__` runs at the start of
the block, `__exit__` runs at the end, regardless of whether the block
succeeded or raised an exception. `open()` is just one built-in example;
you'll encounter others throughout the book, including database
connections, HTTP sessions, and file locks.

#### Writing Your Own Context Manager

For automation code, it's often useful to define your own — for example,
to reliably switch into a directory and back:

```python
import os
from contextlib import contextmanager

@contextmanager
def working_directory(path):
    original = os.getcwd()
    os.chdir(path)
    try:
        yield
    finally:
        os.chdir(original)

with working_directory("/tmp/build"):
    # code here runs with /tmp/build as the current directory
    ...
# automatically restored to the original directory afterward
```

The `@contextmanager` decorator from the standard library's `contextlib`
module is the easiest way to write one of these without manually
implementing `__enter__` and `__exit__`.

#### Why This Matters More in Automation Than in Typical Scripts

Automation code is disproportionately likely to run unattended, which
means there's often no human around to notice a file handle or network
connection was left open. Left unmanaged for long enough, these leaks can
exhaust a system's available file descriptors or connection pool,
causing failures in unrelated parts of a long-running process. Using
`with` consistently — instead of manual `open()`/`close()` pairs — is one
of the cheapest reliability habits available to you.


### 0.12 Understanding Modules, Imports, and `__main__`

As automation projects grow past a single file, you'll need to split code
across multiple files and understand how Python connects them together.
This section covers modules, imports, and the `if __name__ ==
"__main__":` pattern you'll see at the bottom of nearly every script in
this book.

#### What a Module Is

Any `.py` file is a module. If you have a file named `helpers.py`, you
can import it from another file in the same directory:

```python
# helpers.py
def slugify(text):
    return text.lower().replace(" ", "-")
```

```python
# main.py
import helpers

print(helpers.slugify("Automation Report"))
```

#### Import Variations

```python
import helpers                       # access as helpers.slugify(...)
from helpers import slugify          # access as slugify(...) directly
from helpers import slugify as slug  # rename on import
import helpers as h                  # rename the module itself
```

Prefer `import module_name` for clarity in longer scripts — it's
immediately obvious where a function came from when it's called as
`helpers.slugify(...)` rather than bare `slugify(...)`.

#### Standard Library, Third-Party, and Local Imports

By convention (and enforced by tools like `isort`), imports are grouped
in this order:

```python
# Standard library
import os
import sys

# Third-party (installed via pip)
import requests

# Local modules
import helpers
```

#### Packages: Grouping Modules Into Directories

A directory containing an `__init__.py` file (which can be empty) is a
**package** — a way of grouping related modules:

```
myproject/
    __init__.py
    parsing.py
    reporting.py
```

```python
from myproject import parsing
```

As of modern Python, `__init__.py` is technically optional for basic
package discovery, but this book includes it explicitly since it remains
common practice and gives you a clear place for package-level setup code.

#### The `if __name__ == "__main__":` Pattern

Every Python module has a built-in variable called `__name__`. When a
file is run directly, `__name__` is set to `"__main__"`. When that same
file is *imported* by another file, `__name__` is set to the module's
actual name instead.

```python
# report.py
def build_report():
    print("Building report...")

if __name__ == "__main__":
    build_report()
```

Run directly:
```bash
python3 report.py
# Building report...
```

Imported elsewhere:
```python
import report
# nothing prints — build_report() is not called automatically
report.build_report()   # only runs when explicitly called
```

#### Why This Pattern Matters for Automation

This pattern lets a file work two ways at once: as a standalone script
you run from the command line, and as a reusable module other scripts can
import without triggering side effects just by importing it. Nearly
every script in this book ends with this block — it's a small piece of
boilerplate that pays for itself the first time you want to reuse a
function from an old script inside a new one.


### 0.13 File Paths the Right Way: `pathlib` vs `os.path`

Almost every automation script touches the file system, and path
handling is one of the most common sources of bugs that only show up
when a script runs on a different operating system than the one it was
written on. This section covers both of Python's approaches and explains
why this book prefers `pathlib`.

#### The Old Way: `os.path`

`os.path` is a collection of functions for working with path strings:

```python
import os

path = os.path.join("data", "logs", "app.log")
print(path)                       # data/logs/app.log (or data\logs\app.log on Windows)
print(os.path.exists(path))
print(os.path.basename(path))     # app.log
print(os.path.dirname(path))      # data/logs
```

This works fine, and you'll still see it in a lot of existing code and
documentation. Its main drawback is that paths are plain strings, so
combining and manipulating them means calling separate functions for
every operation.

#### The Modern Way: `pathlib`

`pathlib` represents paths as objects, with operations available as
methods and a more readable syntax overall. This book uses `pathlib`
throughout.

```python
from pathlib import Path

path = Path("data") / "logs" / "app.log"
print(path)                    # data/logs/app.log
print(path.exists())
print(path.name)               # app.log
print(path.parent)             # data/logs
print(path.suffix)             # .log
print(path.stem)               # app
```

The `/` operator for joining paths is not a typo — `Path` overloads it
specifically for this purpose, and it works correctly on both Windows
and Unix-like systems, producing the right separator for whichever
platform the script actually runs on.

#### Common Operations

```python
from pathlib import Path

p = Path("reports/2026/summary.txt")

p.exists()          # does it exist on disk?
p.is_file()          # is it a file (not a directory)?
p.is_dir()           # is it a directory?
p.mkdir(parents=True, exist_ok=True)   # create directories as needed
p.read_text()         # read file contents as a string
p.write_text("...")   # write a string to the file
```

#### Finding Files with Patterns

```python
from pathlib import Path

for log_file in Path("logs").glob("*.log"):
    print(log_file)

# recursive search through subdirectories
for py_file in Path(".").rglob("*.py"):
    print(py_file)
```

#### Getting the Current Script's Own Location

A frequent automation need — finding a file relative to the *script*,
not relative to wherever it happens to be run from (see 0.6 for why that
distinction matters):

```python
from pathlib import Path

script_dir = Path(__file__).parent
config_path = script_dir / "config.json"
```

#### Absolute vs. Relative Paths

```python
p = Path("data.txt")
print(p.resolve())    # turns a relative path into an absolute one
```

Prefer resolving paths to absolute form early in a script when the script
will be scheduled or invoked from an unpredictable working directory —
cron jobs and other schedulers are a common source of "it works when I
run it manually but not on schedule" bugs, almost always traceable to an
unexpected working directory.

#### Why This Book Uses `pathlib`

`os.path` remains fully supported and you'll need to recognize it when
reading other people's code, but `pathlib`'s object-oriented approach
produces code that reads more like the operation being described,
handles cross-platform separators automatically, and bundles related
operations (existence checks, reading, writing, globbing) onto the same
object instead of scattering them across separate function calls.

### 0.14 A First Look at `subprocess`

A large share of real-world automation isn't pure Python at all — it's
Python coordinating other command-line tools: `git`, `ffmpeg`, `rsync`,
shell scripts, database CLIs. The `subprocess` module is how Python
launches and communicates with those external programs. This section is
a deliberately partial preview; Chapter 2 covers `subprocess` in full
depth, including streaming output, timeouts, and process pipelines.

#### Running a Command

```python
import subprocess

result = subprocess.run(["ls", "-la"], capture_output=True, text=True)
print(result.stdout)
```

A few things worth noting immediately:

- Commands are passed as a **list of strings**, not a single string —
  each element is one argument, avoiding a whole category of quoting bugs.
- `capture_output=True` captures the command's standard output and
  standard error instead of printing them directly to your terminal.
- `text=True` decodes the output as a string rather than raw bytes.

#### Checking Whether It Succeeded

```python
result = subprocess.run(["python3", "--version"], capture_output=True, text=True)
print(result.returncode)   # 0 usually means success
print(result.stdout)
```

#### Raising an Exception on Failure

```python
try:
    subprocess.run(["false"], check=True)   # "false" always exits with code 1
except subprocess.CalledProcessError as e:
    print(f"Command failed with exit code {e.returncode}")
```

`check=True` makes `subprocess.run` raise an exception automatically if
the command returns a non-zero exit code, rather than silently returning
a result you have to remember to inspect — usually the safer default for
automation code.

#### Passing Arguments Dynamically

```python
filename = "report.pdf"
subprocess.run(["rm", filename], check=True)
```

Because arguments are list elements rather than one concatenated string,
a filename containing spaces or special characters is handled safely
without manual escaping — a common source of bugs (and security issues)
when commands are built as raw strings instead.

#### A Word of Caution: Avoid `shell=True` Unless You Have To

```python
# Generally avoid this pattern:
subprocess.run("ls -la | grep .py", shell=True)
```

`shell=True` lets you use shell features like pipes (`|`) and wildcards
directly, but it also means any untrusted input concatenated into that
string becomes a potential command injection risk. Chapter 2 covers safer
patterns for chaining commands using `subprocess` directly, without
relying on the shell to interpret a string.

#### Why Only a Preview Here

`subprocess` has a lot of surface area — streaming output line by line,
setting timeouts, handling processes that need ongoing input, running
commands in parallel — enough to justify a full chapter. This early
introduction exists so you can recognize the basic `subprocess.run(...)`
call by sight, since a handful of examples between now and Chapter 2 will
use it in passing before it's formally covered.

### 0.15 Structuring a Script: Functions, `main()`, and Exit Codes

This section brings together several earlier topics — functions (0.9),
the `__name__ == "__main__"` pattern (0.12), and exit codes (touched on
in 0.6) — into a standard structure you'll see used for nearly every
script in this book.

#### Why Structure Matters More for Automation

A script that "just works" as a flat sequence of top-level statements is
fine for a five-line throwaway task. But automation scripts tend to
survive longer than expected, get reused, get scheduled, and get read by
other people (or future-you). A small amount of structure up front pays
for itself quickly.

#### The Standard Shape

```python
#!/usr/bin/env python3
"""
cleanup.py — remove log files older than a given number of days.
"""

import sys
from pathlib import Path


def find_old_logs(directory: Path, days: int) -> list[Path]:
    """Return a list of log files older than the given number of days."""
    ...


def delete_files(files: list[Path]) -> int:
    """Delete each file, returning the count successfully removed."""
    ...


def main() -> int:
    directory = Path("logs")
    old_logs = find_old_logs(directory, days=30)

    if not old_logs:
        print("No old logs to remove")
        return 0

    deleted_count = delete_files(old_logs)
    print(f"Deleted {deleted_count} of {len(old_logs)} old log files")

    if deleted_count < len(old_logs):
        return 1   # partial failure

    return 0


if __name__ == "__main__":
    sys.exit(main())
```

#### Why Wrap Everything in `main()`?

Putting the script's logic inside a `main()` function instead of directly
under `if __name__ == "__main__":` keeps the top-level namespace clean,
makes the script importable and testable (you can call `main()` from a
test without also re-running the whole script), and gives you a single
place to manage the overall control flow and return value.

#### Why `sys.exit(main())` Instead of Just Calling `main()`

`sys.exit()` sets the process's exit code — a small integer the calling
environment (a shell, a scheduler, another script) uses to determine
whether the script succeeded. By convention:

- `0` means success
- Any non-zero value means some kind of failure (the specific number can
  encode different failure reasons if useful)

```python
if __name__ == "__main__":
    sys.exit(main())
```

If `main()` returns `0`, the process exits successfully. If it returns
`1`, or any other non-zero integer, the process reports failure — this
is exactly what a calling script, a CI pipeline, or a cron job checks
(often via `$?`, shown in 0.6) to decide whether your automation actually
worked.

#### Handling Uncaught Errors Gracefully

Combine this structure with the exception handling from 0.10 to avoid
ugly, unhelpful tracebacks reaching whoever — or whatever — invoked the
script:

```python
def main() -> int:
    try:
        run_task()
    except Exception as e:
        print(f"Error: {e}", file=sys.stderr)
        return 1
    return 0
```

#### A Structure Worth Adopting Early

You don't need every script to be elaborate, but adopting this
`main()` + exit code shape from your very first automation script means
every later chapter in this book — and every real script you write after
finishing it — starts from a foundation that's already testable,
importable, and predictable for whatever ends up calling it.

### 0.16 Reading Python's Official Docs and Error Tracebacks

Sooner or later, every script in this book will throw an error you
didn't anticipate. Knowing how to read the resulting traceback — and
where to look for authoritative answers — is a skill in its own right,
and arguably more valuable long-term than memorizing any specific syntax.

#### Anatomy of a Traceback

Consider this script:

```python
# broken.py
def divide(a, b):
    return a / b

def process():
    return divide(10, 0)

process()
```

Running it produces:

```
Traceback (most recent call last):
  File "broken.py", line 7, in <module>
    process()
  File "broken.py", line 5, in process
    return divide(10, 0)
  File "broken.py", line 2, in divide
    return a / b
ZeroDivisionError: division by zero
```

Read tracebacks **from the bottom up**:

1. The **last line** (`ZeroDivisionError: division by zero`) tells you
   the actual error type and message — start here.
2. The lines above it show the **call stack** — the chain of function
   calls that led to the error, with the most recent call (where the
   error actually occurred) at the bottom and the original entry point
   at the top.
3. Each frame shows the file, line number, function name, and the exact
   line of code being executed — go straight to that line number in your
   editor.

#### Common Exception Types Worth Recognizing on Sight

| Exception            | Typical Cause                                       |
|-----------------------|------------------------------------------------------|
| `FileNotFoundError`   | A path doesn't exist where expected                  |
| `TypeError`           | An operation applied to the wrong type of value      |
| `ValueError`          | A value of the right type but an invalid value       |
| `KeyError`            | A dictionary key that doesn't exist                  |
| `IndexError`          | A list/tuple index out of range                      |
| `AttributeError`      | Calling a method/attribute an object doesn't have    |
| `ModuleNotFoundError` | An import that isn't installed or is misspelled      |
| `PermissionError`     | The OS denied a file or directory operation           |

Recognizing these by name, before even reading the message, narrows down
the likely cause immediately.

#### Reading Python's Official Documentation

The single most reliable reference for anything covered in this chapter
is [docs.python.org](https://docs.python.org/3/). A few navigation tips:

- The **Language Reference** covers syntax and semantics (how `for`
  loops work, what a generator is).
- The **Library Reference** covers the standard library — `pathlib`,
  `subprocess`, `json`, and everything else imported with `import`.
- Every standard library page includes a **"Changed in version X.Y"**
  note where relevant — check this against your own Python version (0.2)
  if something in a code example doesn't behave as documented.
- Use the search box directly on the docs site rather than a general web
  search when you know roughly what you're looking for (e.g., searching
  "pathlib glob") — it goes straight to the authoritative page rather
  than a possibly outdated blog post.

#### Reading Third-Party Package Documentation

For packages installed via `pip` (0.4), documentation usually lives
either on the package's own site (linked from its PyPI page) or in a
`README` in its source repository. Prefer the version of the docs that
matches the version you actually have installed (`pip show
<package>`), since APIs do change between major versions.

#### When the Traceback Alone Isn't Enough

If a traceback's meaning isn't obvious, add a `print()` statement (or a
debugger breakpoint, if your editor supports one — see 0.5) just before
the failing line to inspect the actual values involved:

```python
def divide(a, b):
    print(f"Dividing {a} by {b}")   # temporary debugging line
    return a / b
```

This simple technique resolves a surprising share of real-world bugs
faster than any other tool — don't underestimate it in favor of more
sophisticated approaches while you're still building confidence.
