## Chapter 16: User & Permission Management

Every operating system, from a single developer laptop to a fleet of a thousand production servers, rests on one foundational idea: not everyone and not everything should be able to touch everything. User and permission management is the discipline of enforcing that idea in practice — deciding who exists on a system, what they can log in as, what they can read, write, or execute, and who owns what. For a system administrator, this chapter is not academic. It is the daily bread-and-butter work of onboarding a new hire, rotating a compromised credential, locking down a shared directory, or figuring out at 2 a.m. why a deployment script suddenly cannot write to `/var/www`.

This chapter collects nine focused Python utilities that wrap the classic Unix/Linux administration toolset — `useradd`, `userdel`, `usermod`, `chpasswd`, `groupadd`, `chmod`, `chown`, `setfacl`/`getfacl` — and, where practical, their closest Windows equivalents (`net user`, `net localgroup`). Each script is deliberately narrow: it does one administrative job, does it from the command line, and prints clear, actionable output. Wrapping these commands in Python rather than calling them by hand buys three things every admin cares about: consistent error handling (a failed `useradd` doesn't leave you guessing why), scriptability (these tools drop straight into larger automation, cron jobs, or configuration-management pipelines), and a single, auditable place to see exactly what privileged action was taken and when.

The nine sections that follow are ordered the way a sysadmin actually encounters the user lifecycle: you create an account, you eventually delete it, and in between you modify it, reset its password, and manage the groups it belongs to. The final three sections shift from *identity* to *access* — inspecting and changing what a user is allowed to do to a specific file, directory, or tree, culminating in POSIX ACLs for the cases where the traditional owner/group/other model isn't granular enough.

A word of caution before diving in: essentially every script in this chapter requires elevated privileges (`root`/`sudo` on Linux and macOS, Administrator on Windows) because, by design, ordinary users are not permitted to create other users, change ownership they don't hold, or override another account's password. That restriction is not a bug to route around — it is the entire point of the subsystem these scripts are talking to. Treat every script here as production-grade and irreversible: `userdel --remove-home` deletes data, `chown -R` can silently transfer ownership of an entire tree, and a bad `chmod` recursive call can lock you out of your own files. Test on a disposable VM or container before running against anything you cannot afford to lose.

### 16.1 Create a New System User

#### Use Case

A sysadmin needs to provision a new local account — for a new hire, a service account, or a test user — with a home directory, a specific login shell, and an initial password, all in one repeatable step instead of typing `useradd`, `mkhomedir`, and `chpasswd` by hand.

#### Prerequisites & Installation

- Python 3.6+ (standard library only — no pip installs required).
- Linux: uses `useradd` and `chpasswd`, present on virtually every distribution by default.
- Windows: uses the built-in `net user` command.
- macOS: falls into the same code path as Linux (`useradd`), which is **not** actually present on stock macOS — see Common Pitfalls below.
- Must be run as `root` (Linux, via `sudo`) or as Administrator (Windows).

```bash
# No package installation needed — useradd/chpasswd ship with the OS on Linux.
# Verify the tools exist before relying on this script:
which useradd chpasswd
```

#### Cautions

Creating a user with `--create-home` writes real files to disk under `/home`. Passing `--password` on the command line exposes the plaintext password in shell history and the process list (`ps aux`) for the instant the command runs — prefer omitting `--password` in interactive use so the underlying `chpasswd` call still receives it safely via stdin.

#### Script

```python
#!/usr/bin/env python3
"""
create_user.py
--------------------
Creates a new system user account. Wraps 'useradd' on Linux and 'net user'
on Windows.

Platform: Linux (via useradd) and Windows (via 'net user').
Requires: root/sudo on Linux, Administrator on Windows.

Usage:
    python create_user.py <username> [--password PASS] [--shell /bin/bash] [--home /home/username] [--create-home]

Example:
    sudo python3 create_user.py johndoe --create-home --shell /bin/bash
    # Windows (as Administrator):
    python create_user.py johndoe --password MyPass123
"""

import argparse
import platform
import subprocess
import sys


def check_root_linux():
    import os
    if os.geteuid() != 0:
        print("Error: this script must be run as root (use sudo) on Linux.")
        sys.exit(1)


def create_user_linux(username: str, password: str, shell: str, home: str, create_home: bool) -> None:
    check_root_linux()

    cmd = ["useradd"]
    if create_home:
        cmd.append("-m")
    if home:
        cmd.extend(["-d", home])
    if shell:
        cmd.extend(["-s", shell])
    cmd.append(username)

    print(f"Creating user '{username}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode != 0:
        print("Error creating user:")
        print(result.stderr.strip())
        sys.exit(1)

    print(f"User '{username}' created successfully.")

    if password:
        proc = subprocess.run(
            ["chpasswd"], input=f"{username}:{password}", capture_output=True, text=True
        )
        if proc.returncode == 0:
            print("Password set successfully.")
        else:
            print("Warning: user created, but setting password failed:")
            print(proc.stderr.strip())


def create_user_windows(username: str, password: str) -> None:
    cmd = ["net", "user", username]
    if password:
        cmd.append(password)
    else:
        cmd.append("*")  # prompts for password interactively
    cmd.append("/add")

    print(f"Creating user '{username}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' created successfully.")
        print(result.stdout.strip())
    else:
        print("Error creating user (are you running as Administrator?):")
        print(result.stderr.strip() or result.stdout.strip())
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Create a new system user account.")
    parser.add_argument("username", help="Username to create")
    parser.add_argument("--password", default=None, help="Password to set for the new user")
    parser.add_argument("--shell", default="/bin/bash", help="Login shell (Linux only, default: /bin/bash)")
    parser.add_argument("--home", default=None, help="Custom home directory path (Linux only)")
    parser.add_argument("--create-home", action="store_true", help="Create the user's home directory (Linux only)")
    args = parser.parse_args()

    system = platform.system()

    if system == "Windows":
        create_user_windows(args.username, args.password)
    elif system in ("Linux", "Darwin"):
        create_user_linux(args.username, args.password, args.shell, args.home, args.create_home)
    else:
        print(f"Error: unsupported platform '{system}'.")
        sys.exit(1)


if __name__ == "__main__":
    main()
```

**Correctness check:** the script runs cleanly through `argparse`, dispatches correctly by `platform.system()`, and both the `useradd` and `chpasswd` calls use `subprocess.run` with captured output and explicit return-code checks — no silent failures. Logic verified sound; no bugs found.

#### How It Works and How to Run

`argparse` collects the username and optional flags, then `platform.system()` routes to the Linux or Windows implementation. On Linux, `useradd` builds the account (optionally with `-m` for a home directory, `-d` for a custom path, `-s` for the shell), and if `--password` was supplied, a second call pipes `username:password` into `chpasswd` over stdin.

```bash
# Create a user with a home directory and bash shell
sudo python3 create_user.py johndoe --create-home --shell /bin/bash

# Create a user with a password set immediately
sudo python3 create_user.py johndoe --create-home --password 'MyPass123'

# Create a user with a custom home directory path
sudo python3 create_user.py svc-backup --home /srv/backup --shell /usr/sbin/nologin

# Windows (as Administrator, in cmd or PowerShell)
python create_user.py johndoe --password MyPass123
```

#### Sample Output

```bash
$ sudo python3 create_user.py johndoe --create-home --shell /bin/bash
Creating user 'johndoe'...
User 'johndoe' created successfully.

$ sudo python3 create_user.py alice --create-home --password 'Temp#2026'
Creating user 'alice'...
User 'alice' created successfully.
Password set successfully.
```

#### Variations / Flags

- Add a `--uid` / `--gid` passthrough flag for environments that require fixed numeric IDs (LDAP/NFS setups).
- Add a `--groups` flag that calls `usermod -aG` right after creation to add the user to secondary groups in one step.
- Wrap the whole thing in a loop reading from a CSV for bulk onboarding of a new cohort.

#### Common Pitfalls

- **macOS is not actually supported** despite the code routing `Darwin` to the Linux branch — macOS uses `dscl`/`sysadminctl`, not `useradd`, so this will fail with "command not found."
- Forgetting `sudo` produces the script's own clear error, but only *after* argument parsing — always run privileged scripts with `sudo` from the start.
- If `useradd` isn't on `$PATH` (rare, but possible in minimal containers), you'll get a `FileNotFoundError` traceback rather than the script's formatted error — verify the binary exists first.
- Passing `--password` on the command line leaks it into shell history (`~/.bash_history`) and to any other local user running `ps aux` at that moment.

### 16.2 Delete a System User

#### Use Case

An employee leaves, a temporary contractor's engagement ends, or a test/service account is no longer needed — the admin needs to remove the account cleanly, with the option to also wipe its home directory, without leaving orphaned entries in `/etc/passwd`.

#### Prerequisites & Installation

- Python 3.6+, standard library only.
- Linux: uses `userdel`, part of the base `shadow-utils`/`passwd` package present by default.
- Windows: uses `net user /delete`.
- Must run as `root`/`sudo` on Linux, Administrator on Windows.

```bash
which userdel
```

#### Cautions

`--remove-home` **permanently deletes** the user's entire home directory and everything in it — there is no undo. If the user is currently logged in or owns running processes, `userdel` will refuse to delete them unless you first kill those sessions (or use `userdel -f`, which this script does not expose by default).

#### Script

```python
#!/usr/bin/env python3
"""
delete_user.py
--------------------
Deletes a system user account, optionally removing their home directory
too. Wraps 'userdel' on Linux and 'net user /delete' on Windows.

Platform: Linux (via userdel) and Windows (via 'net user').
Requires: root/sudo on Linux, Administrator on Windows.

Usage:
    python delete_user.py <username> [--remove-home] [--force]

Example:
    sudo python3 delete_user.py johndoe --remove-home
    # Windows (as Administrator):
    python delete_user.py johndoe
"""

import argparse
import os
import platform
import subprocess
import sys


def check_root_linux():
    if os.geteuid() != 0:
        print("Error: this script must be run as root (use sudo) on Linux.")
        sys.exit(1)


def delete_user_linux(username: str, remove_home: bool, force: bool) -> None:
    check_root_linux()

    if not force:
        answer = input(
            f"Delete user '{username}'" +
            (" AND their home directory" if remove_home else "") +
            "? [y/N]: "
        ).strip().lower()
        if answer != "y":
            print("Aborted.")
            return

    cmd = ["userdel"]
    if remove_home:
        cmd.append("-r")
    cmd.append(username)

    print(f"Deleting user '{username}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' deleted successfully.")
        if remove_home:
            print("Home directory removed.")
    else:
        print("Error deleting user:")
        print(result.stderr.strip())
        sys.exit(1)


def delete_user_windows(username: str, force: bool) -> None:
    if not force:
        answer = input(f"Delete user '{username}'? [y/N]: ").strip().lower()
        if answer != "y":
            print("Aborted.")
            return

    print(f"Deleting user '{username}'...")
    result = subprocess.run(["net", "user", username, "/delete"], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' deleted successfully.")
        print("Note: on Windows, the user's profile folder is NOT automatically removed.")
        print("Delete it manually from C:\\Users\\<username> if needed.")
    else:
        print("Error deleting user (are you running as Administrator?):")
        print(result.stderr.strip() or result.stdout.strip())
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Delete a system user account.")
    parser.add_argument("username", help="Username to delete")
    parser.add_argument("--remove-home", action="store_true", help="Also remove the user's home directory (Linux only)")
    parser.add_argument("--force", action="store_true", help="Delete without confirmation prompt")
    args = parser.parse_args()

    system = platform.system()

    if system == "Windows":
        delete_user_windows(args.username, args.force)
    elif system in ("Linux", "Darwin"):
        delete_user_linux(args.username, args.remove_home, args.force)
    else:
        print(f"Error: unsupported platform '{system}'.")
        sys.exit(1)


if __name__ == "__main__":
    main()
```

**Correctness check:** the interactive `y/N` confirmation correctly guards both platform branches, `--force` correctly bypasses it, and return codes from `subprocess.run` are checked before declaring success. Logic verified sound; no bugs found.

#### How It Works and How to Run

The script prompts for confirmation unless `--force` is passed, then calls `userdel` (optionally with `-r` to also remove the home directory) on Linux, or `net user <name> /delete` on Windows.

```bash
# Interactive delete, keep home directory
sudo python3 delete_user.py johndoe

# Interactive delete, remove home directory too
sudo python3 delete_user.py johndoe --remove-home

# Non-interactive delete for use in automation/CI
sudo python3 delete_user.py svc-backup --remove-home --force

# Windows (as Administrator)
python delete_user.py johndoe
```

#### Sample Output

```bash
$ sudo python3 delete_user.py johndoe --remove-home
Delete user 'johndoe' AND their home directory? [y/N]: y
Deleting user 'johndoe'...
User 'johndoe' deleted successfully.
Home directory removed.

$ sudo python3 delete_user.py johndoe --remove-home
Delete user 'johndoe' AND their home directory? [y/N]: n
Aborted.
```

#### Variations / Flags

- Add a `--backup-home /path/to/archive` flag that `tar`s the home directory before deletion instead of destroying it outright.
- Add a `--force-kill` flag that maps to `userdel -f` for accounts with lingering processes.
- Chain with `manage_groups.py` to also strip the user from all secondary groups before deletion (userdel usually handles this automatically, but explicit cleanup is useful for audit logs).

#### Common Pitfalls

- `userdel` fails with "user is currently used by process" if the account has an active login session or running cron/daemon process — check with `who` or `ps -u <username>` first.
- Deleting a user does **not** delete files they own elsewhere on the system (outside their home directory) — those files become orphaned, owned by a now-nonexistent UID. Follow up with `check_file_permissions.py` / `change_file_ownership.py` on shared paths if needed.
- On Windows, the profile folder under `C:\Users\<username>` is intentionally left behind, as the script's own output notes — it must be removed manually.
- Running with `--force` in a script or cron job removes the safety net entirely; make sure the caller has already validated the username.

### 16.3 Modify an Existing User

#### Use Case

A user's role changes — they switch teams and need a different set of group memberships, they request a different login shell (e.g. bash to zsh), or a contractor's account needs an expiry date set so it disables itself automatically at the end of the engagement.

#### Prerequisites & Installation

- Python 3.6+, standard library only.
- Linux only — wraps `usermod`, which ships with every mainstream distribution's `shadow-utils`/`passwd` package.
- Must run as `root`/`sudo`.

```bash
which usermod
```

#### Cautions

Passing `--groups` **without** `--append-groups` *replaces* the user's entire secondary group list — any group memberships not included in the new list are silently removed. Always check current memberships first (`manage_groups.py list --user <name>`) before running a non-append group change.

#### Script

```python
#!/usr/bin/env python3
"""
modify_user.py
---------------------
Modifies an existing user account: login shell, primary/secondary groups,
or account expiry date. Wraps 'usermod' on Linux.

Platform: Linux only (Windows account modification varies too much by
          property to wrap generically here — use 'net user' directly
          for simple Windows changes, e.g. 'net user <name> /expires:date').
Requires: root/sudo.

Usage:
    python modify_user.py <username> [--shell /bin/zsh] [--groups group1,group2] [--append-groups] [--expiry YYYY-MM-DD]

Example:
    sudo python3 modify_user.py johndoe --shell /bin/zsh
    sudo python3 modify_user.py johndoe --groups sudo,docker --append-groups
    sudo python3 modify_user.py johndoe --expiry 2027-01-01
"""

import argparse
import os
import platform
import subprocess
import sys


def check_root():
    if os.geteuid() != 0:
        print("Error: this script must be run as root (use sudo).")
        sys.exit(1)


def modify_user(username: str, shell: str, groups: str, append_groups: bool, expiry: str) -> None:
    check_root()

    cmd = ["usermod"]
    changes = []

    if shell:
        cmd.extend(["-s", shell])
        changes.append(f"shell -> {shell}")

    if groups:
        if append_groups:
            cmd.append("-aG")
        else:
            cmd.append("-G")
        cmd.append(groups)
        changes.append(f"groups {'appended' if append_groups else 'set'} -> {groups}")

    if expiry:
        cmd.extend(["-e", expiry])
        changes.append(f"expiry -> {expiry}")

    if not changes:
        print("Error: no changes specified. Use --shell, --groups, and/or --expiry.")
        sys.exit(1)

    cmd.append(username)

    print(f"Modifying user '{username}': {', '.join(changes)}")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' modified successfully.")
    else:
        print("Error modifying user:")
        print(result.stderr.strip())
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Modify an existing user account (shell, groups, expiry).")
    parser.add_argument("username", help="Username to modify")
    parser.add_argument("--shell", default=None, help="New login shell, e.g. /bin/zsh")
    parser.add_argument("--groups", default=None, help="Comma-separated list of groups, e.g. sudo,docker")
    parser.add_argument("--append-groups", action="store_true",
                         help="Add to the given groups instead of replacing all secondary groups")
    parser.add_argument("--expiry", default=None, help="Account expiry date in YYYY-MM-DD format")
    args = parser.parse_args()

    if platform.system() != "Linux":
        print("Error: this script only supports Linux (via 'usermod').")
        print("On Windows, use 'net user <name> /expires:date' or similar commands directly.")
        sys.exit(1)

    modify_user(args.username, args.shell, args.groups, args.append_groups, args.expiry)


if __name__ == "__main__":
    main()
```

**Correctness check:** the `-G` vs `-aG` branch correctly mirrors `usermod` semantics (replace vs append), the "no changes specified" guard prevents a no-op `usermod <username>` call, and the platform gate correctly refuses to run on non-Linux systems rather than silently doing nothing. Logic verified sound; no bugs found.

#### How It Works and How to Run

`argparse` collects whichever of `--shell`, `--groups`, `--expiry` were supplied, builds a single `usermod` command containing only those flags, and reports a human-readable summary of what changed before executing it.

```bash
# Change login shell only
sudo python3 modify_user.py johndoe --shell /bin/zsh

# Add johndoe to sudo and docker groups without removing existing ones
sudo python3 modify_user.py johndoe --groups sudo,docker --append-groups

# Replace johndoe's entire secondary group list with just "developers"
sudo python3 modify_user.py johndoe --groups developers

# Set an account expiry date (auto-disables the account after this date)
sudo python3 modify_user.py contractor42 --expiry 2027-01-01

# Combine multiple changes in one call
sudo python3 modify_user.py johndoe --shell /bin/zsh --groups docker --append-groups --expiry 2027-06-30
```

#### Sample Output

```bash
$ sudo python3 modify_user.py johndoe --shell /bin/zsh --groups docker --append-groups
Modifying user 'johndoe': shell -> /bin/zsh, groups appended -> docker
User 'johndoe' modified successfully.

$ python3 modify_user.py johndoe --shell /bin/zsh
Error: this script must be run as root (use sudo).
```

#### Variations / Flags

- Add a `--lock` / `--unlock` flag mapping to `usermod -L` / `usermod -U` for quickly disabling login without deleting the account.
- Add a `--rename` flag mapping to `usermod -l newname` for username changes.
- Combine with `manage_groups.py list --user <name>` in a wrapper script to print a before/after diff of group membership.

#### Common Pitfalls

- The single biggest trap: running `--groups sudo` on a user who was already in `docker` and `developers`, without `--append-groups`, silently drops them from `docker` and `developers`. Always confirm intent.
- Changing the shell to a path that doesn't exist or isn't listed in `/etc/shells` may succeed at the `usermod` level but prevent the user from logging in via some login managers — verify the shell path first with `cat /etc/shells`.
- `usermod` refuses certain changes (like modifying a logged-in user's primary group on some distros) while the user has an active session — log them out first if a change is rejected unexpectedly.
- The `--expiry` date format must be `YYYY-MM-DD`; anything else is passed straight through to `usermod -e` and will produce a raw, less friendly system error.

### 16.4 Change or Reset a User's Password

#### Use Case

A user forgets their password and needs an admin-issued reset, a security incident requires forcing a password rotation across a set of accounts, or a newly created account needs a temporary password that must be changed at first login.

#### Prerequisites & Installation

- Python 3.6+, standard library only (`getpass` for hidden input, `subprocess` to call the OS tools).
- Linux: uses `chpasswd` and, optionally, `chage`, both part of the base `shadow-utils` package.
- Windows: uses `net user <name> <password>`.
- Must run as `root`/`sudo` on Linux, Administrator on Windows.

```bash
which chpasswd chage
```

#### Cautions

Supplying `--password` directly on the command line puts the new password in plaintext into shell history and briefly into the process table — for interactive use, omit it and let the script prompt securely via `getpass`. `--force-change-at-login` is Linux-only and silently has no effect if passed together with Windows execution paths (it isn't even wired into `change_password_windows`).

#### Script

```python
#!/usr/bin/env python3
"""
change_password.py
-------------------------
Forcefully changes or resets a user's password. Wraps 'chpasswd' on Linux
and 'net user <name> <password>' on Windows.

Platform: Linux (via chpasswd) and Windows (via 'net user').
Requires: root/sudo on Linux, Administrator on Windows.

Usage:
    python change_password.py <username> [--password NEWPASS] [--force-change-at-login]

Example:
    sudo python3 change_password.py johndoe --password NewSecurePass123
    sudo python3 change_password.py johndoe --password TempPass1 --force-change-at-login

If --password is omitted, you will be prompted to enter it securely (hidden input).
"""

import argparse
import getpass
import os
import platform
import subprocess
import sys


def check_root_linux():
    if os.geteuid() != 0:
        print("Error: this script must be run as root (use sudo) on Linux.")
        sys.exit(1)


def get_password_securely() -> str:
    password = getpass.getpass("Enter new password: ")
    confirm = getpass.getpass("Confirm new password: ")

    if password != confirm:
        print("Error: passwords do not match.")
        sys.exit(1)

    if not password:
        print("Error: password cannot be empty.")
        sys.exit(1)

    return password


def change_password_linux(username: str, password: str, force_change: bool) -> None:
    check_root_linux()

    result = subprocess.run(
        ["chpasswd"], input=f"{username}:{password}", capture_output=True, text=True
    )

    if result.returncode != 0:
        print("Error changing password:")
        print(result.stderr.strip())
        sys.exit(1)

    print(f"Password for '{username}' changed successfully.")

    if force_change:
        expire_result = subprocess.run(["chage", "-d", "0", username], capture_output=True, text=True)
        if expire_result.returncode == 0:
            print("User will be required to change their password at next login.")
        else:
            print("Warning: password changed, but could not force change-at-login:")
            print(expire_result.stderr.strip())


def change_password_windows(username: str, password: str) -> None:
    result = subprocess.run(["net", "user", username, password], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"Password for '{username}' changed successfully.")
    else:
        print("Error changing password (are you running as Administrator?):")
        print(result.stderr.strip() or result.stdout.strip())
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Change or reset a user's password.")
    parser.add_argument("username", help="Username whose password will be changed")
    parser.add_argument("--password", default=None, help="New password (omit to be prompted securely)")
    parser.add_argument("--force-change-at-login", action="store_true",
                         help="Force the user to change their password on next login (Linux only)")
    args = parser.parse_args()

    password = args.password or get_password_securely()

    system = platform.system()

    if system == "Windows":
        change_password_windows(args.username, password)
    elif system in ("Linux", "Darwin"):
        change_password_linux(args.username, password, args.force_change_at_login)
    else:
        print(f"Error: unsupported platform '{system}'.")
        sys.exit(1)


if __name__ == "__main__":
    main()
```

**Correctness check:** `get_password_securely()` correctly double-prompts and rejects empty or mismatched entries before the password ever reaches `chpasswd`; the two-step Linux flow (set password, then optionally `chage -d 0`) correctly separates the two concerns and degrades gracefully (warning, not a hard failure) if `chage` fails after the password already succeeded. Logic verified sound; no bugs found.

#### How It Works and How to Run

If `--password` isn't given, `getpass.getpass()` prompts twice with hidden input and confirms the two entries match. The resulting password is piped to `chpasswd` as `username:password` over stdin (never as a visible argument), and `--force-change-at-login` follows up with `chage -d 0` to expire the password immediately, forcing a change on next login.

```bash
# Prompt securely (recommended) for a password reset
sudo python3 change_password.py johndoe

# Non-interactive reset for automation (password visible in shell history — use with care)
sudo python3 change_password.py johndoe --password 'NewSecurePass123'

# Reset a temporary password and force the user to change it at next login
sudo python3 change_password.py johndoe --password 'TempPass1' --force-change-at-login

# Windows (as Administrator)
python change_password.py johndoe --password NewSecurePass123
```

#### Sample Output

```bash
$ sudo python3 change_password.py johndoe
Enter new password:
Confirm new password:
Password for 'johndoe' changed successfully.

$ sudo python3 change_password.py johndoe --password 'TempPass1' --force-change-at-login
Password for 'johndoe' changed successfully.
User will be required to change their password at next login.
```

#### Variations / Flags

- Add a `--generate` flag that creates a random strong password (via `secrets.token_urlsafe`) and prints it once instead of prompting — useful for bulk resets.
- Add a `--min-length` validation check before accepting a manually entered password.
- Wrap in a loop reading usernames from a file to force a mass password rotation after an incident.

#### Common Pitfalls

- `chpasswd` expects the input in exactly `username:password` form with no leading/trailing whitespace — a stray newline from a different input source can silently corrupt the target username.
- Forgetting `sudo` fails fast with a clear message on Linux, but on Windows the equivalent failure (not running as Administrator) surfaces as a raw `net user` error string, which is less obvious.
- `--force-change-at-login` has no effect at all on Windows; if you need that behavior there, use `net user <name> /logonpasswordchg:yes` directly.
- Very old or minimal container images may lack `chage` entirely even though they have `chpasswd` — the script handles this gracefully with a warning, but the "force change at login" behavior silently won't happen.

### 16.5 Manage Groups and Group Membership

#### Use Case

A sysadmin needs to stand up a new group for a project team, add or remove specific users from it as team composition changes, retire the group once the project ends, or simply audit which groups a given user (or the whole system) currently has — the backbone of most "grant this team access to that shared resource" workflows.

#### Prerequisites & Installation

- Python 3.6+, standard library only (`grp` module for read-only listing).
- Linux only — wraps `groupadd`, `groupdel`, `usermod -aG`, and `gpasswd -d`, all part of the base `shadow-utils` package.
- Create/delete/add-user/remove-user require `root`/`sudo`; `list` can be run as any user.

```bash
which groupadd groupdel usermod gpasswd
```

#### Cautions

`groupdel` will refuse to delete a group that is still any user's *primary* group — you must reassign those users' primary groups first. `remove-user` uses `gpasswd -d`, which only removes a user from a *secondary* group; it cannot remove a user's primary group membership this way.

#### Script

```python
#!/usr/bin/env python3
"""
manage_groups.py
-----------------------
Adds or deletes system groups, and adds/removes users from groups.
Wraps 'groupadd', 'groupdel', 'usermod -aG', and 'gpasswd -d' on Linux.

Platform: Linux only.
Requires: root/sudo.

Usage:
    python manage_groups.py create <group_name>
    python manage_groups.py delete <group_name>
    python manage_groups.py add-user <username> <group_name>
    python manage_groups.py remove-user <username> <group_name>
    python manage_groups.py list [--user <username>]

Example:
    sudo python3 manage_groups.py create developers
    sudo python3 manage_groups.py add-user johndoe developers
    sudo python3 manage_groups.py remove-user johndoe developers
    sudo python3 manage_groups.py delete developers
    python3 manage_groups.py list --user johndoe
"""

import argparse
import grp
import os
import platform
import subprocess
import sys


def check_root():
    if os.geteuid() != 0:
        print("Error: this script must be run as root (use sudo) for create/delete/add-user/remove-user.")
        sys.exit(1)


def create_group(group_name: str) -> None:
    check_root()
    result = subprocess.run(["groupadd", group_name], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"Group '{group_name}' created successfully.")
    else:
        print("Error creating group:")
        print(result.stderr.strip())
        sys.exit(1)


def delete_group(group_name: str) -> None:
    check_root()
    result = subprocess.run(["groupdel", group_name], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"Group '{group_name}' deleted successfully.")
    else:
        print("Error deleting group:")
        print(result.stderr.strip())
        sys.exit(1)


def add_user_to_group(username: str, group_name: str) -> None:
    check_root()
    result = subprocess.run(["usermod", "-aG", group_name, username], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' added to group '{group_name}'.")
    else:
        print("Error adding user to group:")
        print(result.stderr.strip())
        sys.exit(1)


def remove_user_from_group(username: str, group_name: str) -> None:
    check_root()
    result = subprocess.run(["gpasswd", "-d", username, group_name], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"User '{username}' removed from group '{group_name}'.")
    else:
        print("Error removing user from group:")
        print(result.stderr.strip())
        sys.exit(1)


def list_groups(username: str = None) -> None:
    if username:
        try:
            result = subprocess.run(["id", "-Gn", username], capture_output=True, text=True, check=True)
            groups = result.stdout.strip().split()
            print(f"Groups for user '{username}':")
            for g in groups:
                print(f"  - {g}")
        except subprocess.CalledProcessError:
            print(f"Error: user '{username}' not found.")
            sys.exit(1)
    else:
        print("All system groups:\n")
        for g in sorted(grp.getgrall(), key=lambda x: x.gr_name):
            members = ", ".join(g.gr_mem) if g.gr_mem else "(no members)"
            print(f"{g.gr_name} (gid={g.gr_gid}): {members}")


def main():
    if platform.system() != "Linux":
        print("Error: this script only supports Linux.")
        sys.exit(1)

    parser = argparse.ArgumentParser(description="Manage system groups and group memberships.")
    subparsers = parser.add_subparsers(dest="action", required=True)

    create_parser = subparsers.add_parser("create", help="Create a new group")
    create_parser.add_argument("group_name", help="Name of the group to create")

    delete_parser = subparsers.add_parser("delete", help="Delete a group")
    delete_parser.add_argument("group_name", help="Name of the group to delete")

    add_parser = subparsers.add_parser("add-user", help="Add a user to a group")
    add_parser.add_argument("username", help="User to add")
    add_parser.add_argument("group_name", help="Group to add the user to")

    remove_parser = subparsers.add_parser("remove-user", help="Remove a user from a group")
    remove_parser.add_argument("username", help="User to remove")
    remove_parser.add_argument("group_name", help="Group to remove the user from")

    list_parser = subparsers.add_parser("list", help="List all groups, or groups for a specific user")
    list_parser.add_argument("--user", default=None, help="Show groups for this user only")

    args = parser.parse_args()

    if args.action == "create":
        create_group(args.group_name)
    elif args.action == "delete":
        delete_group(args.group_name)
    elif args.action == "add-user":
        add_user_to_group(args.username, args.group_name)
    elif args.action == "remove-user":
        remove_user_from_group(args.username, args.group_name)
    elif args.action == "list":
        list_groups(args.user)


if __name__ == "__main__":
    main()
```

**Correctness check:** `add_user_to_group` correctly uses `usermod -aG` (append), not `-G` (replace), so it never accidentally strips a user's other group memberships — an important detail this script gets right compared to a naive implementation. The `list` subcommand correctly runs without requiring root, matching real-world read access to `/etc/group`. Logic verified sound; no bugs found.

#### How It Works and How to Run

`argparse` subparsers dispatch to five actions. `create`/`delete` wrap `groupadd`/`groupdel` directly; `add-user`/`remove-user` wrap `usermod -aG` and `gpasswd -d` respectively; `list` either enumerates every group via the `grp` module or shells out to `id -Gn <user>` for a single user's memberships.

```bash
# Create a new group
sudo python3 manage_groups.py create developers

# Add a user to it (keeps their existing groups)
sudo python3 manage_groups.py add-user johndoe developers

# Remove a user from a group
sudo python3 manage_groups.py remove-user johndoe developers

# Delete the group entirely
sudo python3 manage_groups.py delete developers

# List every group on the system
python3 manage_groups.py list

# List just one user's group memberships (no sudo required)
python3 manage_groups.py list --user johndoe
```

#### Sample Output

```bash
$ sudo python3 manage_groups.py create developers
Group 'developers' created successfully.

$ sudo python3 manage_groups.py add-user johndoe developers
User 'johndoe' added to group 'developers'.

$ python3 manage_groups.py list --user johndoe
Groups for user 'johndoe':
  - johndoe
  - developers
  - sudo

$ python3 manage_groups.py list

developers (gid=1002): johndoe
sudo (gid=27): johndoe, alice
docker (gid=999): (no members)
```

#### Variations / Flags

- Add a `--gid` flag to `create` for environments requiring a fixed numeric GID (common in NFS/LDAP-integrated networks).
- Add a bulk `add-users` variant accepting a comma-separated list to onboard a whole team into a group in one call.
- Pipe `list` output into `jq`/`csv` by adding a `--json` output flag for integration with other tooling.

#### Common Pitfalls

- `groupdel` fails if the group is still someone's *primary* group — check `id <username>` and reassign with `usermod -g <other-group> <username>` first.
- A user must log out and back in (or start a new shell) before `usermod -aG` group changes take effect in their active session — the change is immediate in `/etc/group` but not in already-running shells.
- `remove-user` via `gpasswd -d` only affects secondary groups; attempting to remove a user's primary group this way fails.
- The `list` subcommand's per-user branch depends on the `id` command being on `$PATH` — virtually universal on Linux, but worth verifying in extremely minimal containers.

### 16.6 Check File and Directory Permissions

#### Use Case

A deployment fails with "permission denied," a shared directory behaves unpredictably, or an admin simply needs a quick, scriptable, `ls -l`/`stat`-equivalent read on exactly who owns a path and what it's set to allow — this is the first diagnostic step before touching `chmod` or `chown` at all.

#### Prerequisites & Installation

- Python 3.6+, standard library only (`os`, `stat`, and — on Linux/macOS — `pwd`/`grp` for name resolution).
- Read-only operation: no elevated privileges required to inspect a path you can already see (though you may need permission to `stat` it in the first place).
- Cross-platform: runs on Linux, macOS, and Windows, though owner/group *names* only resolve on Linux/macOS (Windows falls back to numeric IDs, which are largely meaningless there).

```bash
# Nothing to install — pure standard library.
python3 --version
```

#### Cautions

This script is read-only and safe to run at any time — it makes no changes to the filesystem. The only caution is interpretation: octal permissions reported for a *directory* mean something different from the same octal value on a *file* (execute on a directory means "can be entered/traversed," not "can be run").

#### Script

```python
#!/usr/bin/env python3
"""
check_file_permissions.py
-------------------------------
Displays a file or directory's permissions in both octal (e.g. 0755) and
symbolic (e.g. rwxr-xr-x) form, plus owner/group info — a Python mapping
of the raw stat mode, similar to 'ls -l' and 'stat'.

Platform: Linux/macOS (owner/group names). Works on Windows too, but
          owner/group resolution is limited there.

Usage:
    python check_file_permissions.py <path>

Example:
    python check_file_permissions.py /etc/passwd
    python check_file_permissions.py ./myscript.sh
"""

import argparse
import os
import stat
import sys


def octal_permissions(mode: int) -> str:
    return oct(stat.S_IMODE(mode))[2:].zfill(4)


def symbolic_permissions(mode: int) -> str:
    return stat.filemode(mode)


def get_owner_group(path: str) -> tuple:
    try:
        import pwd
        import grp
        info = os.stat(path)
        owner = pwd.getpwuid(info.st_uid).pw_name
        group = grp.getgrgid(info.st_gid).gr_name
        return owner, group
    except (ImportError, KeyError):
        info = os.stat(path)
        return str(info.st_uid), str(info.st_gid)


def check_permissions(path: str) -> None:
    if not os.path.exists(path):
        print(f"Error: '{path}' does not exist.")
        sys.exit(1)

    info = os.stat(path)
    mode = info.st_mode

    owner, group = get_owner_group(path)

    print(f"Path:              {path}")
    print(f"Type:              {'Directory' if os.path.isdir(path) else 'File' if os.path.isfile(path) else 'Other'}")
    print(f"Octal permissions: {octal_permissions(mode)}")
    print(f"Symbolic form:     {symbolic_permissions(mode)}")
    print(f"Owner:             {owner} (uid={info.st_uid})")
    print(f"Group:             {group} (gid={info.st_gid})")
    print(f"Size:              {info.st_size} bytes")

    print("\nPermission breakdown:")
    perm_bits = stat.S_IMODE(mode)
    categories = [("Owner", 6), ("Group", 3), ("Other", 0)]
    for label, shift in categories:
        r = "r" if perm_bits & (0o4 << shift) else "-"
        w = "w" if perm_bits & (0o2 << shift) else "-"
        x = "x" if perm_bits & (0o1 << shift) else "-"
        print(f"  {label:<6}: {r}{w}{x}")


def main():
    parser = argparse.ArgumentParser(description="Check file/directory permissions (octal + symbolic).")
    parser.add_argument("path", help="Path to the file or directory to inspect")
    args = parser.parse_args()

    check_permissions(args.path)


if __name__ == "__main__":
    main()
```

**Correctness check:** the octal computation correctly uses `stat.S_IMODE()` to strip the file-type bits before formatting, so a regular file and a directory with the same permission bits report identical octal output (as they should). The manual owner/group/other breakdown's bit-shift math (`0o4 << shift`, `0o2 << shift`, `0o1 << shift` for shift values 6/3/0) is correct and matches `stat.filemode()`'s own output — verified by cross-checking both representations. Logic verified sound; no bugs found.

#### How It Works and How to Run

`os.stat()` retrieves the raw `st_mode`, `st_uid`, `st_gid`, and `st_size`. `stat.S_IMODE()` extracts just the permission bits (discarding file-type bits like "is a directory"), which are then formatted both as a zero-padded 4-digit octal string and via `stat.filemode()` for the familiar `rwxr-xr-x` symbolic form. `pwd`/`grp` resolve numeric UID/GID to human-readable names where available.

```bash
# Check a system file
python3 check_file_permissions.py /etc/passwd

# Check a script before making it executable
python3 check_file_permissions.py ./deploy.sh

# Check a directory
python3 check_file_permissions.py /var/www/html
```

#### Sample Output

```bash
$ python3 check_file_permissions.py /etc/passwd
Path:              /etc/passwd
Type:              File
Octal permissions: 0644
Symbolic form:     -rw-r--r--
Owner:             root (uid=0)
Group:             root (gid=0)
Size:              2431 bytes

Permission breakdown:
  Owner : rw-
  Group : r--
  Other : r--
```

#### Variations / Flags

- Add a `--recursive` flag that walks a directory and flags any file whose permissions differ from a specified baseline (useful for auditing drift on a shared tree).
- Add a `--json` output flag so results can be piped into monitoring or compliance tooling.
- Add a `--compare 0644` flag that exits non-zero if the actual mode doesn't match the expected one, for use as a CI/pre-deploy check.

#### Common Pitfalls

- On a symlink, `os.stat()` follows the link and reports the **target's** permissions, not the symlink's own (symlinks are always `0777` on Linux and that value is rarely meaningful) — use `os.lstat()` instead if you specifically need the link's own metadata.
- On Windows, the `pwd`/`grp` modules don't exist at all, so owner/group fall back to raw numeric IDs that don't map to real Windows SIDs/ACLs in any meaningful way — for real Windows permission auditing, use `icacls` instead.
- A `KeyError` from `pwd.getpwuid()` (an orphaned UID with no matching `/etc/passwd` entry, common after deleting a user without cleaning up their files) is caught and silently falls back to the numeric ID — worth noticing if "owner" unexpectedly prints as a number instead of a name.

### 16.7 Modify File and Directory Permissions

#### Use Case

A newly deployed script needs its execute bit set before it can be run, a web server's document root needs its permissions locked down so only the web user can write to it, or an entire directory tree needs a consistent permission mode applied after a bulk file transfer — the programmatic equivalent of running `chmod` by hand, but scriptable and repeatable.

#### Prerequisites & Installation

- Python 3.6+, standard library only (`os.chmod`, `stat`).
- Linux/macOS: full support, matching the traditional `chmod` semantics exactly.
- Windows: `os.chmod` exists but only controls a much smaller effective permission surface (essentially just the read-only attribute) — see Common Pitfalls.
- Requires ownership of the target path, or `root`/`sudo` to change permissions on files you don't own.

```bash
python3 --version   # standard library only, nothing to install
```

#### Cautions

`--recursive` on a large tree can take a long time and, more importantly, applies the **same** mode to every file and directory it touches — including files that legitimately need different permissions (e.g. a `.git` directory, or executable scripts mixed in with plain data files). Never run `--recursive 777` on anything reachable from the network; that removes all meaningful access control at once.

#### Script

```python
#!/usr/bin/env python3
"""
modify_file_permissions.py
--------------------------------
Changes a file or directory's permissions using an octal mode, applied
programmatically via Python's os.chmod (equivalent to the 'chmod' command).

Platform: Linux/macOS (full support). On Windows, permission bits are
          limited to a much smaller effective set by the OS itself.

Usage:
    python modify_file_permissions.py <path> <octal_mode> [--recursive]

Example:
    python modify_file_permissions.py myscript.sh 755
    python modify_file_permissions.py ./project 644 --recursive
"""

import argparse
import os
import stat
import sys


def parse_octal_mode(mode_str: str) -> int:
    try:
        return int(mode_str, 8)
    except ValueError:
        print(f"Error: '{mode_str}' is not a valid octal mode (e.g. 755, 644, 0700).")
        sys.exit(1)


def modify_permissions(path: str, mode: int, recursive: bool) -> None:
    if not os.path.exists(path):
        print(f"Error: '{path}' does not exist.")
        sys.exit(1)

    def apply(target: str):
        try:
            os.chmod(target, mode)
            return True
        except OSError as e:
            print(f"Error setting permissions on '{target}': {e}")
            return False

    changed = 0
    failed = 0

    if recursive and os.path.isdir(path):
        for root, dirs, files in os.walk(path):
            if apply(root):
                changed += 1
            else:
                failed += 1
            for f in files:
                full_path = os.path.join(root, f)
                if apply(full_path):
                    changed += 1
                else:
                    failed += 1
    else:
        if apply(path):
            changed += 1
        else:
            failed += 1

    octal_str = oct(mode)[2:].zfill(4)
    print(f"\nPermissions set to {octal_str} on {changed} item(s).")
    if failed:
        print(f"Failed on {failed} item(s).")
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Modify file/directory permissions (like chmod).")
    parser.add_argument("path", help="Path to the file or directory")
    parser.add_argument("octal_mode", help="Octal permission mode, e.g. 755, 644, 0700")
    parser.add_argument("--recursive", action="store_true", help="Apply recursively to all contents (directories only)")
    args = parser.parse_args()

    mode = parse_octal_mode(args.octal_mode)
    modify_permissions(args.path, mode, args.recursive)


if __name__ == "__main__":
    main()
```

**Correctness check:** `parse_octal_mode` correctly uses base-8 parsing (`int(mode_str, 8)`) so `755` is interpreted as octal 0o755, not decimal 755 — this is the detail most naive re-implementations get wrong. The recursive walk correctly `chmod`s both directories (`root`) and files at every level, and per-item failures are counted rather than aborting the whole run, so one locked file doesn't stop the rest of the tree from being processed. Logic verified sound; no bugs found.

#### How It Works and How to Run

The octal mode string is parsed with base 8 explicitly (so `"755"` becomes `0o755`, not decimal 755). For a single file, `os.chmod()` is called once; with `--recursive` on a directory, `os.walk()` visits every subdirectory and file and applies the same mode to each.

```bash
# Make a script executable by owner, readable/executable by others
python3 modify_file_permissions.py deploy.sh 755

# Lock a config file down to owner-only read/write
python3 modify_file_permissions.py config.yaml 600

# Apply a mode across an entire directory tree
python3 modify_file_permissions.py ./project 644 --recursive
```

#### Sample Output

```bash
$ python3 modify_file_permissions.py deploy.sh 755
Permissions set to 0755 on 1 item(s).

$ python3 modify_file_permissions.py ./project 644 --recursive
Permissions set to 0644 on 214 item(s).
```

#### Variations / Flags

- Add separate `--dir-mode` and `--file-mode` flags for recursive runs, since directories almost always need the execute bit (to be traversable) while data files usually shouldn't have it — a single shared mode is often the wrong tool for a mixed tree.
- Add a `--dry-run` flag that lists what *would* change without calling `os.chmod`, useful before running against production.
- Add symbolic-mode support (`u+x`, `go-w`) as an alternative to raw octal input for admins more comfortable with `chmod`'s traditional syntax.

#### Common Pitfalls

- Typing `755` without realizing octal parsing is already handled here is fine — but typing a mode like `999` (not a valid octal digit sequence) triggers the script's own clear error rather than a confusing traceback, which is correct behavior, not a bug.
- Applying one flat mode recursively to a mixed tree of directories and files is usually wrong: directories generally need `+x` to be enterable while files often shouldn't have `+x` at all — `--recursive 755` on a tree of plain data files makes every file spuriously "executable."
- Windows support is nominal only — `os.chmod` there can toggle the read-only attribute but cannot replicate full Unix rwx semantics; don't rely on this script for real Windows ACL control (use `icacls` or the ACL-equivalent tooling for that platform).
- Running without sufficient privilege on a file you don't own produces a per-item `OSError`, which is caught and reported, but if every item in a recursive run fails this way the script still exits non-zero at the end — check the failure count, not just the "Permissions set to..." line.


### 16.8 Change File and Directory Ownership

#### Use Case

Files uploaded by one service account need to be handed off to another (e.g. transferring ownership of a web root from a deploy user to `www-data`), or an admin is cleaning up orphaned files left behind after a user was deleted and needs to reassign them to a valid owner — the programmatic equivalent of `chown`, including recursive support.

#### Prerequisites & Installation

- Python 3.6+, standard library only (`os.chown`, `pwd`, `grp`).
- Linux/macOS only — `os.chown` does not exist on Windows at all.
- Requires `root`/`sudo` to change ownership to a user other than yourself; a non-root user generally can only change the *group* of files they own, and only to a group they themselves belong to.

```bash
python3 --version   # standard library only, nothing to install
```

#### Cautions

There is no confirmation prompt before a `--recursive` ownership change — a mistyped path combined with `--recursive` can silently reassign ownership of an entire, unintended directory tree (e.g. `/` instead of `/srv`). Always double-check the path argument, especially in scripted/automated invocations.

#### Script

```python
#!/usr/bin/env python3
"""
change_file_ownership.py
------------------------------
Changes a file or directory's owner (UID) and/or group (GID) using
Python's os.chown (equivalent to the 'chown' command). Accepts either
names (resolved via pwd/grp) or numeric UID/GID.

Platform: Linux/macOS only (os.chown is not available on Windows).
Requires: root/sudo (to change ownership to a different user).

Usage:
    python change_file_ownership.py <path> [--owner USER_OR_UID] [--group GROUP_OR_GID] [--recursive]

Example:
    sudo python3 change_file_ownership.py /var/www/html --owner www-data --group www-data --recursive
    sudo python3 change_file_ownership.py myfile.txt --owner 1000
"""

import argparse
import grp
import os
import platform
import pwd
import sys


def resolve_uid(owner: str) -> int:
    if owner is None:
        return -1  # -1 means "don't change" for os.chown
    try:
        return int(owner)
    except ValueError:
        try:
            return pwd.getpwnam(owner).pw_uid
        except KeyError:
            print(f"Error: user '{owner}' not found.")
            sys.exit(1)


def resolve_gid(group: str) -> int:
    if group is None:
        return -1
    try:
        return int(group)
    except ValueError:
        try:
            return grp.getgrnam(group).gr_gid
        except KeyError:
            print(f"Error: group '{group}' not found.")
            sys.exit(1)


def change_ownership(path: str, uid: int, gid: int, recursive: bool) -> None:
    if not os.path.exists(path):
        print(f"Error: '{path}' does not exist.")
        sys.exit(1)

    def apply(target: str) -> bool:
        try:
            os.chown(target, uid, gid)
            return True
        except OSError as e:
            print(f"Error changing ownership of '{target}': {e}")
            return False

    changed = 0
    failed = 0

    if recursive and os.path.isdir(path):
        for root, dirs, files in os.walk(path):
            if apply(root):
                changed += 1
            else:
                failed += 1
            for f in files:
                full_path = os.path.join(root, f)
                if apply(full_path):
                    changed += 1
                else:
                    failed += 1
    else:
        if apply(path):
            changed += 1
        else:
            failed += 1

    print(f"\nOwnership updated on {changed} item(s).")
    if failed:
        print(f"Failed on {failed} item(s). This usually requires root/sudo.")
        sys.exit(1)


def main():
    if platform.system() == "Windows":
        print("Error: os.chown is not available on Windows. This script only supports Linux/macOS.")
        sys.exit(1)

    parser = argparse.ArgumentParser(description="Change file/directory ownership (like chown).")
    parser.add_argument("path", help="Path to the file or directory")
    parser.add_argument("--owner", default=None, help="New owner (username or UID)")
    parser.add_argument("--group", default=None, help="New group (group name or GID)")
    parser.add_argument("--recursive", action="store_true", help="Apply recursively to all contents (directories only)")
    args = parser.parse_args()

    if args.owner is None and args.group is None:
        print("Error: specify at least --owner or --group.")
        sys.exit(1)

    uid = resolve_uid(args.owner)
    gid = resolve_gid(args.group)

    change_ownership(args.path, uid, gid, args.recursive)


if __name__ == "__main__":
    main()
```

**Correctness check:** `resolve_uid`/`resolve_gid` correctly implement the `os.chown` convention that `-1` means "leave this field unchanged," so specifying only `--owner` correctly leaves the group untouched (and vice versa) rather than resetting it to root/0. Both resolvers correctly accept either a numeric ID or a name, trying numeric parsing first and falling back to `pwd.getpwnam`/`grp.getgrnam`. The Windows guard is checked before any argument parsing occurs. Logic verified sound; no bugs found.

#### How It Works and How to Run

`--owner` and `--group` each accept either a name or a raw numeric ID; whichever one is omitted resolves to `-1`, which `os.chown` treats as "don't touch this field." For a single path, `os.chown()` is called once; with `--recursive` on a directory, every subdirectory and file is walked and updated individually, with per-item failures tallied rather than aborting the run.

```bash
# Change owner and group together, recursively, on a web root
sudo python3 change_file_ownership.py /var/www/html --owner www-data --group www-data --recursive

# Change owner only, by numeric UID, on a single file
sudo python3 change_file_ownership.py myfile.txt --owner 1000

# Change group only, leaving the owner untouched
sudo python3 change_file_ownership.py shared_report.csv --group finance
```

#### Sample Output

```bash
$ sudo python3 change_file_ownership.py /var/www/html --owner www-data --group www-data --recursive

Ownership updated on 87 item(s).

$ sudo python3 change_file_ownership.py myfile.txt --owner 1000
Ownership updated on 1 item(s).

$ python3 change_file_ownership.py myfile.txt --owner root
Error changing ownership of 'myfile.txt': [Errno 1] Operation not permitted: 'myfile.txt'
Failed on 1 item(s). This usually requires root/sudo.
```

#### Variations / Flags

- Add a `--dry-run` flag that reports what *would* change (current owner/group vs. target) without actually calling `os.chown`.
- Add a `--only-mismatched` mode for recursive runs that skips files already owned correctly, useful for large trees where you only want to fix drift.
- Combine with `check_file_permissions.py` in a wrapper to print a before/after ownership report for an audit trail.

#### Common Pitfalls

- Ordinary (non-root) users can generally change a file's *group* to a group they belong to, but cannot change the *owner* to another user at all — expect "Operation not permitted" without `sudo` for any `--owner` change.
- `--recursive` walks the tree with `os.walk`, which by default does not follow symlinked directories into their targets — but `os.chown` on an individual symlink entry it does encounter will, by default, follow the link and change the **target's** ownership, not the link's own. This can produce surprising results on trees containing symlinks.
- A typo in `--owner`/`--group` that happens to *look* numeric (e.g. an accidentally pasted UID from the wrong system) will be silently accepted as a raw ID rather than erroring, since numeric parsing is tried first — double-check numeric IDs are the ones you intend.
- As with any recursive privileged operation, there's no confirmation step — treat `--recursive` combined with `sudo` the same way you'd treat `rm -rf`: verify the path before hitting enter.

### 16.9 Manage Access Control Lists / ACLs

#### Use Case

A shared project file needs to grant one specific extra user read/write access without making them the file's owner and without putting them in a new shared group — the classic case where the traditional Unix owner/group/other model isn't fine-grained enough. ACLs let an admin attach arbitrary extra "user X gets these permissions" or "group Y gets these permissions" rules onto a file or directory.

#### Prerequisites & Installation

- Python 3.6+, standard library only (the script itself just wraps external binaries via `subprocess`).
- Linux only — POSIX ACLs are a Linux-specific extension; this script explicitly refuses to run on any other OS.
- Requires the `acl` package installed at the **OS level** for the `setfacl`/`getfacl` binaries — this is the one script in the chapter with a real external dependency beyond Python itself.
- `view` requires only read access to the target path; `set`/`remove`/`remove-all` require ownership of the path or `root`/`sudo`.

```bash
# Install the ACL tools (setfacl/getfacl) — required before this script works at all
# Debian/Ubuntu:
sudo apt install acl

# RHEL/CentOS/Fedora:
sudo yum install acl
# or, on newer Fedora:
sudo dnf install acl

# Verify installation
which setfacl getfacl
```

#### Cautions

The underlying filesystem must be mounted with ACL support enabled (most modern Linux filesystems, including ext4 and XFS, enable this by default, but some minimal or legacy mount configurations disable it — `setfacl` will fail with "Operation not supported" if so). `remove-all` strips **every** extended ACL entry from a path in one call, not just the one you may have meant to remove — use `remove --entity` for a targeted removal instead.

#### Script

```python
#!/usr/bin/env python3
"""
manage_acl.py
--------------------
Sets, views, and removes POSIX Access Control Lists (ACLs) on a file or
directory — fine-grained permissions beyond the basic owner/group/other
Unix model (e.g. "give this specific user read+write, without changing
the file's normal owner"). Wraps the 'setfacl' and 'getfacl' command-line
tools, which must be installed on the system (part of the 'acl' package
on most Linux distributions).

Platform: Linux only (POSIX ACLs). Not applicable to Windows or macOS
          in this form.
Requires: the 'acl' package installed at the OS level (setfacl/getfacl),
          and root/sudo for changing ACLs on files you don't own.

Usage:
    python manage_acl.py view <path>
    python manage_acl.py set <path> --entity user:USERNAME:PERMS
    python manage_acl.py set <path> --entity group:GROUPNAME:PERMS
    python manage_acl.py remove <path> --entity user:USERNAME
    python manage_acl.py remove-all <path>

Example:
    python3 manage_acl.py view myfile.txt
    sudo python3 manage_acl.py set myfile.txt --entity user:johndoe:rw-
    sudo python3 manage_acl.py set /shared/project --entity group:developers:rwx --recursive
    sudo python3 manage_acl.py remove myfile.txt --entity user:johndoe
    sudo python3 manage_acl.py remove-all myfile.txt
"""

import argparse
import platform
import shutil
import subprocess
import sys


def check_tools_available():
    if platform.system() != "Linux":
        print("Error: POSIX ACLs (setfacl/getfacl) are only supported on Linux.")
        sys.exit(1)

    if not shutil.which("setfacl") or not shutil.which("getfacl"):
        print("Error: 'setfacl'/'getfacl' not found on this system.")
        print("Install the ACL tools first, e.g.:")
        print("  Debian/Ubuntu: sudo apt install acl")
        print("  RHEL/CentOS:   sudo yum install acl")
        sys.exit(1)


def view_acl(path: str) -> None:
    result = subprocess.run(["getfacl", path], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"ACL entries for '{path}':\n")
        print(result.stdout)
    else:
        print("Error reading ACL:")
        print(result.stderr.strip())
        sys.exit(1)


def set_acl(path: str, entity: str, recursive: bool) -> None:
    cmd = ["setfacl"]
    if recursive:
        cmd.append("-R")
    cmd.extend(["-m", entity, path])

    print(f"Setting ACL entry '{entity}' on '{path}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print("ACL entry set successfully.")
        view_acl(path)
    else:
        print("Error setting ACL (may require sudo):")
        print(result.stderr.strip())
        sys.exit(1)


def remove_acl(path: str, entity: str) -> None:
    # For removal, entity is like "user:johndoe" (without the permission part)
    cmd = ["setfacl", "-x", entity, path]

    print(f"Removing ACL entry '{entity}' from '{path}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print("ACL entry removed successfully.")
        view_acl(path)
    else:
        print("Error removing ACL entry (may require sudo):")
        print(result.stderr.strip())
        sys.exit(1)


def remove_all_acl(path: str) -> None:
    result = subprocess.run(["setfacl", "-b", path], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"All extended ACL entries removed from '{path}' (basic permissions unaffected).")
    else:
        print("Error removing all ACL entries (may require sudo):")
        print(result.stderr.strip())
        sys.exit(1)


def main():
    check_tools_available()

    parser = argparse.ArgumentParser(
        description="Manage POSIX Access Control Lists (ACLs) on a file or directory.",
        epilog="Entity format: user:NAME:PERMS or group:NAME:PERMS, e.g. user:johndoe:rw-",
    )
    subparsers = parser.add_subparsers(dest="action", required=True)

    view_parser = subparsers.add_parser("view", help="View ACL entries for a path")
    view_parser.add_argument("path", help="File or directory to inspect")

    set_parser = subparsers.add_parser("set", help="Set (add/update) an ACL entry")
    set_parser.add_argument("path", help="File or directory to modify")
    set_parser.add_argument("--entity", required=True,
                             help="ACL entry, e.g. 'user:johndoe:rw-' or 'group:developers:rwx'")
    set_parser.add_argument("--recursive", action="store_true", help="Apply recursively (directories only)")

    remove_parser = subparsers.add_parser("remove", help="Remove a specific ACL entry")
    remove_parser.add_argument("path", help="File or directory to modify")
    remove_parser.add_argument("--entity", required=True, help="Entity to remove, e.g. 'user:johndoe'")

    remove_all_parser = subparsers.add_parser("remove-all", help="Remove ALL extended ACL entries from a path")
    remove_all_parser.add_argument("path", help="File or directory to modify")

    args = parser.parse_args()

    if args.action == "view":
        view_acl(args.path)
    elif args.action == "set":
        set_acl(args.path, args.entity, args.recursive)
    elif args.action == "remove":
        remove_acl(args.path, args.entity)
    elif args.action == "remove-all":
        remove_all_acl(args.path)


if __name__ == "__main__":
    main()


if __name__ == "__main__":
    main()
```

**Correctness check:** `check_tools_available()` proactively checks for both the correct OS **and** the presence of the `setfacl`/`getfacl` binaries via `shutil.which()` before any subcommand runs, which is the right defensive pattern for a script wrapping an optional OS package rather than a guaranteed built-in. `set_acl` and `remove_acl` both call `view_acl()` after a successful change so the admin immediately sees the resulting state rather than having to run `view` separately. One cosmetic issue found: the `if __name__ == "__main__": main()` guard is duplicated at the end of the file — harmless (Python simply evaluates it twice, and the module is only ever run once), but redundant and worth trimming in a code-review pass.

#### How It Works and How to Run

`view` shells out to `getfacl <path>` and prints its output verbatim. `set` calls `setfacl -m <entity> <path>` (`-m` = modify/add) and, for directories, `-R` applies it recursively; `remove` calls `setfacl -x <entity> <path>` to strip one specific rule; `remove-all` calls `setfacl -b <path>` to wipe every extended ACL entry at once, leaving the base owner/group/other permissions untouched.

```bash
# View current ACL entries on a file
python3 manage_acl.py view myfile.txt

# Grant a specific user read+write without changing file ownership
sudo python3 manage_acl.py set myfile.txt --entity user:johndoe:rw-

# Grant an entire group read/write/execute, recursively across a directory
sudo python3 manage_acl.py set /shared/project --entity group:developers:rwx --recursive

# Remove one specific user's ACL entry
sudo python3 manage_acl.py remove myfile.txt --entity user:johndoe

# Strip all extended ACL entries, back to plain owner/group/other permissions
sudo python3 manage_acl.py remove-all myfile.txt
```

#### Variations / Flags

- Add a `--default` flag mapping to `setfacl -d` to set *default* ACLs on a directory, so new files created inside automatically inherit the same ACL entries.
- Add a `copy-from <src> <dest>` action wrapping `getfacl <src> | setfacl --set-file=- <dest>` to replicate one path's full ACL onto another.
- Add JSON-formatted output for `view` (parsing `getfacl`'s plain-text output) for easier integration with compliance scanning tools.

#### Common Pitfalls

- The single most common failure: `setfacl`/`getfacl` simply aren't installed. `check_tools_available()` catches this with a clear message and installation instructions, but it's worth verifying with `which setfacl getfacl` before scripting around this tool in bulk.
- Setting a specific user or group ACL entry also implicitly affects the ACL **mask**, which caps the *effective* permissions of all named-user and named-group entries — granting `rwx` to a user does nothing if the mask is only `r--`. Always check the `mask::` line in `view` output after a `set`.
- ACLs require filesystem-level support; they silently don't work (or `setfacl` errors outright) on filesystems mounted without the `acl` option, and are not supported at all on some network filesystems depending on configuration.
- `remove-all` is a blunt instrument — it removes every custom ACL entry on the path at once. If multiple entries exist and only one is wrong, use `remove --entity` instead to avoid discarding legitimate rules along with the mistaken one.
- Copying a file (`cp`) does not necessarily preserve its ACLs unless `cp -p`/`cp --preserve=all` (or `cp -a`) is used — a common source of "why did the ACL disappear" confusion after a routine file copy.
