## Chapter 6: Logging & Monitoring

Every system eventually breaks. Disks fill up, processes die at 3 a.m., a bad deploy fills a log file with tracebacks, and nobody notices until a customer complains. The difference between a sysadmin who catches these problems in minutes and one who catches them in days is almost never talent — it's instrumentation. This chapter is about building that instrumentation with nothing more exotic than Python's standard library and a handful of well-known third-party packages.

We start at the foundation: how Python represents a "warning" versus a "log message," and why the built-in `logging` module is the backbone of almost everything else in this chapter. From there we move outward in concentric circles. First, we make logging *structured* — tagging every warning and error with a machine-readable code so it can be filtered, counted, and alerted on instead of just read. Then we make it *durable*, using rotation and compression so a runaway process can't fill a disk with text files, and so six months of history doesn't cost you six months of disk space. Then we make it *observable from anywhere*, shipping records over the network to a central collector instead of leaving them scattered across every box in the fleet.

With logging handled, the chapter pivots to monitoring: watching the live state of a machine rather than its historical record. `psutil` gives us a portable way to ask any Linux, macOS, or Windows host how busy its CPU is, how much RAM is free, how full its disks are, how much traffic is moving across its network interfaces, and whether the processes we care about are actually alive. Each of these checks is small enough to run standalone from a cron job, and composable enough to be folded into a single health-check script that a load balancer or CI pipeline can query with a simple exit code.

Monitoring numbers is only useful if a human eventually sees them, so the second half of the chapter covers notification. We wire the same alert payload into four different channels — email via SMTP, Slack via incoming webhooks, Telegram via its Bot API, and Discord via webhooks — so you can plug whichever one matches your team's existing workflow into any script in this chapter. Finally, we close the loop with audit logging and structured JSON output, the format modern log-aggregation platforms (the ELK stack, Splunk, Grafana Loki, Datadog) expect, turning your logs from disposable text into first-class, queryable data.

None of the scripts in this chapter assume you're running a specific distribution, cloud provider, or orchestration platform. They are deliberately small, readable, and dependency-light, so you can drop them into a cron job, a systemd timer, a CI pipeline, or a container entrypoint with minimal adaptation. Treat them as a starting toolkit rather than a finished product — the real value comes from wiring the pieces together: a monitor script that detects a problem, a logger that records it, and an alert script that makes sure a human finds out before a customer does.

### 6.1 Generate Warning Messages


#### Use case
A sysadmin maintaining internal tooling or automation scripts needs to flag risky-but-non-fatal conditions (a deprecated function still in use, a calculation that silently returned `None` instead of crashing) without stopping execution or polluting application logs. Warnings are the right tool when you want to surface a code smell to a developer or operator without treating it as an operational error.

#### Prerequisites

```bash
# No external packages needed — the warnings module ships with Python's
# standard library (Python 3.x, any OS: Linux, macOS, Windows)
python3 --version
```

#### Cautions
By default, Python's `warnings` module deduplicates identical warnings and only shows the first occurrence per location — use `warnings.simplefilter("always")` if you need every occurrence to appear (e.g., in a monitoring context). Warnings printed to `stderr` are easy to miss if nothing is watching that stream.

#### Script

```python
#!/usr/bin/env python3
"""
generate_warnings.py
---------------------
Demonstrates how to generate warning messages in Python using the
built-in `warnings` module (different from logging warnings).

Usage:
    python3 generate_warnings.py
"""

import warnings


def divide(a, b):
    if b == 0:
        warnings.warn("Division by zero attempted, returning None", RuntimeWarning)
        return None
    return a / b


def deprecated_function():
    warnings.warn(
        "deprecated_function() is deprecated, use new_function() instead",
        DeprecationWarning,
        stacklevel=2,
    )


def main():
    # Make sure warnings are always shown (Python hides duplicates by default)
    warnings.simplefilter("always")

    print("Result:", divide(10, 2))
    print("Result:", divide(10, 0))  # triggers RuntimeWarning

    deprecated_function()  # triggers DeprecationWarning

    # Custom warning category example
    class DiskSpaceWarning(UserWarning):
        pass

    warnings.warn("Disk space is running low", DiskSpaceWarning)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`warnings.warn(message, category)` raises a warning of the given category (`RuntimeWarning`, `DeprecationWarning`, or a custom subclass of `UserWarning`), which Python prints to `stderr` by default, pointing at the exact line that triggered it. `warnings.simplefilter("always")` overrides Python's default "show once per location" behavior so every call is visible — useful when testing or when running inside a monitoring harness that needs to see repeated warnings.

```bash
# Run normally
python3 generate_warnings.py

# Run with all Python warnings visible, even ones this script doesn't emit
python3 -W always generate_warnings.py

# Treat warnings as errors (useful in CI to catch regressions)
python3 -W error generate_warnings.py

# Redirect warnings to a file for later review (they go to stderr)
python3 generate_warnings.py 2> warnings_output.txt
```

#### Sample output

```
Result: 5.0
generate_warnings.py:17: RuntimeWarning: Division by zero attempted, returning None
  warnings.warn("Division by zero attempted, returning None", RuntimeWarning)
Result: None
generate_warnings.py:37: DeprecationWarning: deprecated_function() is deprecated, use new_function() instead
  deprecated_function()  # triggers DeprecationWarning
generate_warnings.py:43: DiskSpaceWarning: Disk space is running low
  warnings.warn("Disk space is running low", DiskSpaceWarning)
```

#### Variations / flags
- Run with `python3 -W ignore generate_warnings.py` to suppress all warnings (useful for demoing a script cleanly, not recommended for production diagnostics).
- Add `warnings.filterwarnings("error", category=DeprecationWarning)` to turn only one category into a hard exception while leaving others as soft warnings.
- Capture warnings programmatically with `warnings.catch_warnings(record=True)` if you want to log them into your regular application logger instead of `stderr`.

#### Common pitfalls
- Confusing this with `logging.warning()` — the `warnings` module is for code-level, one-time-per-location notices aimed at developers; `logging` is for operational records aimed at operators. They are not interchangeable.
- Forgetting that warnings are deduplicated by default, leading to "missing" warnings on the second and later runs of the same code path within a single process.
- Not redirecting `stderr`, so warnings vanish in cron jobs or background services where nobody is watching the terminal.

### 6.2 Python Logging Module


#### Use case
Any long-running script or service — a backup job, a deployment tool, a daemon — needs a record of what it did and when. A sysadmin uses proper logging (instead of scattered `print()` calls) so messages carry a timestamp, severity level, and source, and so the destination (console, file, syslog) can be changed in one place without touching business logic.

#### Prerequisites

```bash
# logging is part of the Python standard library — nothing to install
python3 --version   # any Python 3.x, Linux/macOS/Windows
```

#### Cautions
`logging.basicConfig()` only takes effect the *first* time it's called in a process — calling it again later has no effect unless you pass `force=True`. Also, the default root logger writes to `stderr`; if you expect a file, make sure you configure `filename=` or attach a `FileHandler` explicitly.

#### Script

```python
#!/usr/bin/env python3
"""
logging_basics.py
------------------
Introduction to Python's built-in `logging` module: setting up a logger,
choosing a log level, and formatting messages.

Usage:
    python3 logging_basics.py
"""

import logging

# 1. Basic configuration: level, format, output destination
logging.basicConfig(
    level=logging.DEBUG,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

# 2. Create a named logger (best practice instead of using the root logger)
logger = logging.getLogger("logging_basics_demo")


def main():
    logger.debug("This is a DEBUG message - detailed diagnostic info")
    logger.info("This is an INFO message - confirms things work as expected")
    logger.warning("This is a WARNING message - something unexpected happened")
    logger.error("This is an ERROR message - a serious problem occurred")
    logger.critical("This is a CRITICAL message - the program may be unable to continue")

    # Logging with variables
    user = "ahmad"
    logger.info("User '%s' logged in successfully", user)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`logging.basicConfig()` configures the root logging system once: the minimum severity level to display (`DEBUG` shows everything), the message format, and the timestamp format. `logging.getLogger("name")` returns a named logger that inherits this configuration but tags every line with its own name, which is invaluable once a codebase has more than one module logging at once. The `%s`-style placeholders in `logger.info("...%s...", user)` are lazily formatted — the string is only built if the message actually gets logged, which matters for performance at `DEBUG` level in hot code paths.

```bash
# Run with default (DEBUG and above, so everything prints)
python3 logging_basics.py

# Send output to a file instead of the terminal by editing basicConfig
# to add: filename="app.log", filemode="a"

# Raise the bar so only WARNING and above show (edit level=logging.WARNING)
```

#### Sample output

```
2026-07-17 18:37:10 [DEBUG] logging_basics_demo: This is a DEBUG message - detailed diagnostic info
2026-07-17 18:37:10 [INFO] logging_basics_demo: This is an INFO message - confirms things work as expected
2026-07-17 18:37:10 [WARNING] logging_basics_demo: This is a WARNING message - something unexpected happened
2026-07-17 18:37:10 [ERROR] logging_basics_demo: This is an ERROR message - a serious problem occurred
2026-07-17 18:37:10 [CRITICAL] logging_basics_demo: This is a CRITICAL message - the program may be unable to continue
2026-07-17 18:37:10 [INFO] logging_basics_demo: User 'ahmad' logged in successfully
```

#### Variations / flags
- Change `level=logging.DEBUG` to `logging.INFO`, `WARNING`, `ERROR`, or `CRITICAL` to control the noise floor per environment (verbose in dev, quiet in production).
- Add `filename="app.log"` and `filemode="a"` to `basicConfig()` to persist logs to disk instead of the console.
- Attach multiple handlers (e.g., one `StreamHandler` for the console and one `FileHandler` for disk) to send the same log to two destinations at different levels.

#### Common pitfalls
- Calling `basicConfig()` more than once expecting it to reconfigure things — it silently no-ops after the first call unless `force=True` is passed (Python 3.8+).
- Using the root logger everywhere (`logging.info(...)` directly) instead of named loggers, which makes it impossible to tell which module emitted a given line in a larger codebase.
- Forgetting that `DEBUG`-level logging can be verbose and expensive in production; always raise the threshold before shipping.

### 6.3 Log Warnings and Error Codes

#### Use case
A sysadmin running the same script across dozens of servers needs more than free-text messages — they need consistent codes they can grep for, dashboard, or wire into an alerting rule ("page on-call whenever `E2xx` appears more than 5 times in 10 minutes"). Tagging log lines with codes turns unstructured text into something a monitoring pipeline can reason about.

#### Prerequisites

```bash
# Standard library only — no pip install required
python3 --version   # Python 3.x, Linux/macOS/Windows
mkdir -p logs        # the script creates this automatically, but good to know
```

#### Cautions
Custom fields passed via `extra={...}` must not collide with existing `LogRecord` attribute names (e.g., don't name a field `message` or `args`), or `logging` will raise a `KeyError`. Also remember `logging.basicConfig(level=logging.WARNING)` here means `INFO` and `DEBUG` calls on this logger will be silently dropped.

#### Script

```python
#!/usr/bin/env python3
"""
log_warnings_errors.py
------------------------
Logs WARNING and ERROR level messages, each tagged with a custom error
code, to a log file for later inspection.

Usage:
    python3 log_warnings_errors.py
"""

import logging
import os

LOG_DIR = "logs"
LOG_FILE = os.path.join(LOG_DIR, "warnings_errors.log")

os.makedirs(LOG_DIR, exist_ok=True)

logging.basicConfig(
    level=logging.WARNING,
    format="%(asctime)s [%(levelname)s] [code=%(error_code)s] %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
    filename=LOG_FILE,
    filemode="a",
)

logger = logging.getLogger("warnings_errors_demo")


def log_warning(message, error_code="W000"):
    logger.warning(message, extra={"error_code": error_code})


def log_error(message, error_code="E000"):
    logger.error(message, extra={"error_code": error_code})


def main():
    log_warning("Disk usage above 80%", error_code="W101")
    log_warning("Response time degraded", error_code="W102")
    log_error("Failed to connect to database", error_code="E201")
    log_error("Unhandled exception in worker thread", error_code="E202")

    print(f"Warnings and errors logged to: {LOG_FILE}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The format string embeds `%(error_code)s`, a custom field that doesn't exist on `LogRecord` by default. Passing `extra={"error_code": "W101"}` to `logger.warning(...)` injects that attribute onto the record for this call only, so the formatter can render it. Because `basicConfig` sets `filename=LOG_FILE, filemode="a"`, every run appends to the same file rather than truncating it — handy for building up history across many invocations (e.g., once per cron run).

```bash
# Run once — appends 4 lines to logs/warnings_errors.log
python3 log_warnings_errors.py

# Run repeatedly (e.g., from cron) and the file just keeps growing —
# pair this script with rotate_logs.py or compress_logs.py (6.5 / 6.6)
# so it doesn't grow forever

# Grep for a specific error code family
grep "code=E2" logs/warnings_errors.log
```

#### Sample output

Console:
```
Warnings and errors logged to: logs/warnings_errors.log
```

`logs/warnings_errors.log`:
```
2026-07-17 18:37:10 [WARNING] [code=W101] Disk usage above 80%
2026-07-17 18:37:10 [WARNING] [code=W102] Response time degraded
2026-07-17 18:37:10 [ERROR] [code=E201] Failed to connect to database
2026-07-17 18:37:10 [ERROR] [code=E202] Unhandled exception in worker thread
```

#### Variations / flags
- Adopt a numbering convention, e.g. `W1xx` = resource warnings, `W2xx` = performance warnings, `E2xx` = connectivity errors, `E3xx` = data-integrity errors, so `grep`/regex filters stay simple.
- Feed this log into `monitor_app_logs.py` (6.8) to get real-time alerts the moment a new `E2xx` line appears.
- Swap `filename=LOG_FILE` for a `RotatingFileHandler` (6.5) if this script runs frequently enough that the file could grow large.

#### Common pitfalls
- Forgetting to pass `extra={"error_code": ...}` on *every* call to a logger configured with `%(error_code)s` in its format string — any call without it raises `KeyError: 'error_code'` at log time.
- Setting the module-level threshold to `WARNING` and then wondering why `logger.info(...)` calls elsewhere in the same logger produce nothing — check the level before assuming a bug in the message itself.
- Using the same error code for unrelated failures, which defeats the purpose of tagging — keep a small, documented code registry so codes stay meaningful across the team.

### 6.4 Log Exceptions


#### Use case
When a script catches an exception to keep running (rather than crashing outright), a sysadmin still needs to know exactly what failed and where. `logger.exception()` captures the complete traceback automatically, so post-incident debugging doesn't rely on the failure happening again in front of you.

#### Prerequisites

```bash
# Standard library only
python3 --version   # Python 3.x, Linux/macOS/Windows
mkdir -p logs
```

#### Cautions
`logger.exception()` must be called from *inside* an `except` block (it relies on `sys.exc_info()` still being active) — calling it outside one logs "NoneType: None" as the traceback instead of anything useful. It also always logs at `ERROR` level; there is no `logger.exception()` equivalent for other severities without passing `exc_info=True` manually.

#### Script

```python
#!/usr/bin/env python3
"""
log_exceptions.py
-------------------
Demonstrates how to properly log exceptions (with full traceback) using
the logging module instead of just printing them.

Usage:
    python3 log_exceptions.py
"""

import logging
import os

LOG_DIR = "logs"
LOG_FILE = os.path.join(LOG_DIR, "exceptions.log")

os.makedirs(LOG_DIR, exist_ok=True)

logging.basicConfig(
    level=logging.DEBUG,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
    filename=LOG_FILE,
    filemode="a",
)

logger = logging.getLogger("exceptions_demo")


def risky_division(a, b):
    return a / b


def parse_int(value):
    return int(value)


def main():
    # Example 1: logger.exception() automatically includes the traceback
    try:
        risky_division(10, 0)
    except ZeroDivisionError:
        logger.exception("Division failed")

    # Example 2: logger.error() with exc_info=True does the same thing
    try:
        parse_int("not_a_number")
    except ValueError as e:
        logger.error("Failed to parse integer: %s", e, exc_info=True)

    print(f"Exceptions logged with traceback to: {LOG_FILE}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`logger.exception("Division failed")` is shorthand for `logger.error("Division failed", exc_info=True)` — both write the message at `ERROR` level and append the full traceback (file, line number, and the exception chain) beneath it. The second example shows the explicit form, useful when you want to log at a level other than `ERROR` (e.g., `logger.warning(..., exc_info=True)` for a recoverable failure) while still keeping the traceback.

```bash
# Run once — appends two tracebacks to logs/exceptions.log
python3 log_exceptions.py

# Inspect the result
cat logs/exceptions.log

# Search the log for a specific exception type later
grep "ZeroDivisionError" logs/exceptions.log
```

#### Sample output

Console:
```

```

`logs/exceptions.log` (abridged):
```
2026-07-17 18:37:10 [ERROR] Division failed
Traceback (most recent call last):
  File "log_exceptions.py", line 27, in main
    risky_division(10, 0)
  File "log_exceptions.py", line 20, in risky_division
    return a / b
ZeroDivisionError: division by zero
2026-07-17 18:37:10 [ERROR] Failed to parse integer: invalid literal for int() with base 10: 'not_a_number'
Traceback (most recent call last):
  File "log_exceptions.py", line 32, in main

  File "log_exceptions.py", line 24, in parse_int
```

#### Variations / flags
- Use `logger.warning("...", exc_info=True)` instead of `logger.exception()` when the failure is expected/recoverable and shouldn't be counted as a hard error in your metrics.
- Combine with 6.3's `extra={"error_code": ...}` pattern to tag exceptions with a code as well as a traceback.
- Pipe this log into `monitor_app_logs.py` (6.8) with `--keywords Traceback,ERROR` for real-time exception alerting.

#### Common pitfalls
- Calling `logger.exception()` outside an active `except` block — it will run without error but the traceback section will be empty or say "NoneType: None."
- Swallowing exceptions with a bare `except: pass` and never logging them at all — silent failures are the hardest to diagnose after the fact.
- Logging the exception and then re-raising it without realizing the traceback will be logged *and* printed to the console by the default exception handler, resulting in duplicate-looking output (this is expected and not a bug).

### 6.5 Rotate Log Files


#### Use case
A long-running service (a daemon, a web app, a scheduled job that runs for years) will eventually fill a disk if its log file is never rotated. A sysadmin uses `RotatingFileHandler` so the active log stays small and a bounded number of historical backups are kept automatically, with zero manual intervention.

#### Prerequisites

```bash
# Standard library only (logging.handlers.RotatingFileHandler)
python3 --version   # Python 3.x, Linux/macOS/Windows
mkdir -p logs
```

#### Cautions
`RotatingFileHandler` rotates based on file *size*, checked each time a record is written — it does not proactively check on a timer, so a quiet process that logs rarely won't rotate even if a long time has passed (see `TimedRotatingFileHandler` for time-based rotation instead). On Windows, an open log file that another process also has open can occasionally block the rename step used during rotation.

#### Script

```python
#!/usr/bin/env python3
"""
rotate_logs.py
-----------------
Demonstrates automatic log rotation using RotatingFileHandler.
When the log file reaches a set size, it is rotated (renamed) and a
fresh file is started, keeping a limited number of backups.

Usage:
    python3 rotate_logs.py
"""

import logging
import os
from logging.handlers import RotatingFileHandler

LOG_DIR = "logs"
LOG_FILE = os.path.join(LOG_DIR, "rotating.log")

os.makedirs(LOG_DIR, exist_ok=True)

logger = logging.getLogger("rotate_logs_demo")
logger.setLevel(logging.INFO)

# Rotate when file reaches ~1 KB, keep 3 backup files (rotating.log.1, .2, .3)
handler = RotatingFileHandler(
    LOG_FILE,
    maxBytes=1024,
    backupCount=3,
)
formatter = logging.Formatter("%(asctime)s [%(levelname)s] %(message)s")
handler.setFormatter(formatter)
logger.addHandler(handler)


def main():
    # Write enough log lines to trigger multiple rotations
    for i in range(200):
        logger.info("Log entry number %d - sample log rotation test message", i)

    print(f"Rotating logs written to: {LOG_DIR}/")
    print("Check the folder - you should see rotating.log, rotating.log.1, .2, .3")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`RotatingFileHandler(LOG_FILE, maxBytes=1024, backupCount=3)` writes to `rotating.log` normally, but as soon as writing the next record would push the file past `maxBytes`, it renames the current file to `rotating.log.1` (shifting any existing `.1` to `.2`, `.2` to `.3`, and dropping anything beyond `backupCount`), then starts a fresh `rotating.log`. This confirmed test run produced one active file plus three ~957-byte backups from 200 log lines against a 1 KB threshold.

```bash
# Run once
python3 rotate_logs.py

# Confirm the rotation happened
ls -la logs/rotating.log*

# In production, set maxBytes much higher, e.g. 10 MB:
#   maxBytes=10 * 1024 * 1024, backupCount=5
```

#### Sample output

Console:
```
Rotating logs written to: logs/
Check the folder - you should see rotating.log, rotating.log.1, .2, .3
```

Directory listing after running:
```
-rw-r--r-- 1 root root  87 Jul 17 18:37 logs/rotating.log
-rw-r--r-- 1 root root 957 Jul 17 18:37 logs/rotating.log.1
-rw-r--r-- 1 root root 957 Jul 17 18:37 logs/rotating.log.2
-rw-r--r-- 1 root root 957 Jul 17 18:37 logs/rotating.log.3
```

#### Variations / flags
- Swap `RotatingFileHandler` for `TimedRotatingFileHandler(LOG_FILE, when="midnight", backupCount=7)` to rotate daily instead of by size.
- Increase `maxBytes` to a realistic production size (e.g., 10–50 MB) and `backupCount` to match your retention policy.
- Combine with `compress_logs.py` (6.6) to `.gz`-compress rotated backups (`rotating.log.1`, `.2`, etc.) once they're no longer being written to.
- On Linux, `logrotate` (a system utility, not Python) is a common alternative/complement for rotating logs written by processes you don't control.

#### Common pitfalls
- Setting `maxBytes` too low for real production log volume, causing excessive rotation churn and near-constant file renames.
- Assuming rotation happens on a schedule — it only happens when a write would exceed the size limit, so an idle process may never rotate.
- Forgetting that `backupCount=3` means 3 *backups* (`.1`–`.3`) plus the active file — 4 files total, not 3.

### 6.6 Compress Old Logs

#### Use case
Log files accumulate fast, especially once rotation (6.5) is in place — old, rotated logs are rarely read but still take up space at full size. A sysadmin schedules this script (via cron) to compress anything past a retention age, cutting disk usage dramatically while still keeping the history available if it's ever needed.

#### Prerequisites

```bash
# gzip and shutil are both standard library — no pip install needed
python3 --version   # Python 3.x, Linux/macOS/Windows
```

#### Cautions
This script **deletes the original file** after compressing it (`os.remove(path)`) — make sure `--dir` points at the correct directory before running it, especially with `--days 0`, which compresses everything immediately regardless of age. Also note it only touches files ending in `.log`; anything already `.gz` is skipped, so re-running it is safe.

#### Script

```python
#!/usr/bin/env python3
"""
compress_logs.py
-------------------
Scans a directory for old log files (based on age in days) and
compresses them into .gz archives to save disk space, then removes
the original uncompressed file.

Usage:
    python3 compress_logs.py --dir logs --days 0
    (use --days 0 for demo purposes so it compresses everything immediately;
     in production use something like --days 7)

Arguments:
    --dir   Directory containing log files (default: logs)
    --days  Minimum age in days before a log file is compressed (default: 7)
"""

import argparse
import gzip
import os
import shutil
import time


def compress_file(path):
    gz_path = path + ".gz"
    with open(path, "rb") as f_in, gzip.open(gz_path, "wb") as f_out:
        shutil.copyfileobj(f_in, f_out)
    os.remove(path)
    return gz_path


def compress_old_logs(directory, min_age_days):
    if not os.path.isdir(directory):
        print(f"Directory not found: {directory}")
        return

    now = time.time()
    cutoff = now - (min_age_days * 86400)
    compressed_count = 0

    for filename in os.listdir(directory):
        filepath = os.path.join(directory, filename)

        # Skip directories and already-compressed files
        if not os.path.isfile(filepath) or filename.endswith(".gz"):
            continue
        if not filename.endswith(".log"):
            continue

        file_mtime = os.path.getmtime(filepath)
        if file_mtime <= cutoff:
            gz_path = compress_file(filepath)
            print(f"Compressed: {filepath} -> {gz_path}")
            compressed_count += 1

    print(f"\nDone. {compressed_count} file(s) compressed.")


def main():
    parser = argparse.ArgumentParser(description="Compress old log files into .gz archives")
    parser.add_argument("--dir", default="logs", help="Directory containing log files")
    parser.add_argument("--days", type=int, default=7, help="Minimum age in days")
    args = parser.parse_args()

    compress_old_logs(args.dir, args.days)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script walks every entry in `--dir`, skips anything that isn't a plain file, isn't a `.log` file, or is already `.gz`, then checks each remaining file's modification time (`os.path.getmtime`) against a cutoff computed as "now minus `--days` days." Anything older than the cutoff is streamed through `gzip.open()` in binary mode (via `shutil.copyfileobj`, which handles large files without loading them entirely into memory) and the original is deleted once the `.gz` copy is written.

```bash
# Compress anything older than 7 days (default) in ./logs
python3 compress_logs.py

# Compress everything immediately, useful for testing
python3 compress_logs.py --dir logs --days 0

# Point at a different directory
python3 compress_logs.py --dir /var/log/myapp --days 14

# Schedule it nightly via cron (crontab -e):
0 2 * * * /usr/bin/python3 /opt/scripts/compress_logs.py --dir /var/log/myapp --days 7
```

#### Sample output

Tested against a directory containing one file (`old_test.log`) older than the cutoff:

```
Compressed: logs/old_test.log -> logs/old_test.log.gz

Done. 1 file(s) compressed.
```

Running again immediately afterward (nothing left to compress):
```
Done. 0 file(s) compressed.
```

#### Variations / flags
- Change the file filter from `.log` to a tuple (`filename.endswith((".log", ".txt"))`) if your logs use multiple extensions.
- Add a `--dry-run` flag that only prints what *would* be compressed without calling `compress_file()`, useful for verifying behavior before trusting it with `--days 0` in production.
- Compress into a separate `archive/` subdirectory instead of leaving `.gz` files alongside active logs, to make retention cleanup (deleting archives after N months) simpler.

#### Common pitfalls
- Running with `--days 0` against a directory that also contains logs still being actively written to — those files get compressed and deleted mid-write, which can corrupt or truncate them if another process has them open.
- Pointing `--dir` at the wrong path and silently compressing/deleting files you didn't intend to touch — always dry-run or double check the directory first in production.
- Forgetting that this script does not recurse into subdirectories — only the top level of `--dir` is scanned.

### 6.7 Send Logs to Remote Servers


#### Use case
Once you're running more than a handful of machines, reading logs one server at a time doesn't scale. A sysadmin centralizes logging — sending every host's records to one collector — so incidents that span multiple machines can be correlated in one place instead of SSH-ing into each box individually.

#### Prerequisites

```bash
# Standard library only: logging.handlers, socketserver, pickle, struct
python3 --version   # Python 3.x, Linux/macOS/Windows
# No firewall rule needed for localhost testing; for real deployments,
# open/allow the chosen TCP port (default 9020 in this script) between
# client and server hosts.
```

#### Cautions
`SocketHandler` sends data **unencrypted** and, in this script, uses `pickle` to serialize log records — never expose this port to an untrusted network, since a malicious sender could submit crafted pickle data. For production, put this behind a VPN, an SSH tunnel, or a TLS-terminating proxy, or replace it with a hardened log shipper (rsyslog, Logstash, Fluentd, Filebeat) instead of hand-rolling transport security.

#### Script

```python
#!/usr/bin/env python3
"""
remote_log_sender.py
-----------------------
Sends log records to a remote server over the network using
logging.handlers.SocketHandler (TCP). This is the standard Python
mechanism for shipping logs to a central logging server.

This script includes BOTH:
  1) A simple demo TCP server that receives and prints logs
  2) A client that sends log messages to that server

Usage:
    # Terminal 1 - start the receiving server
    python3 remote_log_sender.py --mode server --port 9020

    # Terminal 2 - run the client to send sample logs
    python3 remote_log_sender.py --mode client --host 127.0.0.1 --port 9020

Note: For real production use, point --host/--port at your actual
central logging server (e.g. rsyslog, Logstash, Graylog) and secure
the connection (e.g. via a VPN or TLS-terminating proxy) since
SocketHandler by itself sends data unencrypted.
"""

import argparse
import logging
import logging.handlers
import pickle
import socketserver
import struct


def run_server(host, port):
    class LogRecordStreamHandler(socketserver.StreamRequestHandler):
        def handle(self):
            while True:
                chunk = self.connection.recv(4)
                if len(chunk) < 4:
                    break
                slen = struct.unpack(">L", chunk)[0]
                chunk = self.connection.recv(slen)
                while len(chunk) < slen:
                    chunk += self.connection.recv(slen - len(chunk))
                obj = pickle.loads(chunk)
                record = logging.makeLogRecord(obj)
                print(f"[REMOTE LOG] {record.levelname}: {record.getMessage()}")

    class LogRecordSocketReceiver(socketserver.ThreadingTCPServer):
        allow_reuse_address = True

    server = LogRecordSocketReceiver((host, port), LogRecordStreamHandler)
    print(f"Log server listening on {host}:{port} (Ctrl+C to stop)")
    server.serve_forever()


def run_client(host, port):
    logger = logging.getLogger("remote_log_sender_client")
    logger.setLevel(logging.DEBUG)

    socket_handler = logging.handlers.SocketHandler(host, port)
    logger.addHandler(socket_handler)

    logger.info("Client started - sending sample log messages")
    logger.warning("This is a sample warning sent to the remote server")
    logger.error("This is a sample error sent to the remote server")
    print(f"Sent 3 sample log messages to {host}:{port}")


def main():
    parser = argparse.ArgumentParser(description="Send/receive logs over TCP to a remote server")
    parser.add_argument("--mode", choices=["server", "client"], required=True)
    parser.add_argument("--host", default="127.0.0.1")
    parser.add_argument("--port", type=int, default=9020)
    args = parser.parse_args()

    if args.mode == "server":
        run_server(args.host, args.port)
    else:
        run_client(args.host, args.port)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`SocketHandler` pickles each `LogRecord` and sends it over a TCP connection prefixed with a 4-byte big-endian length header (`struct.pack(">L", ...)`), which is exactly the wire format Python's own `logging` documentation recommends for this pattern. On the receiving end, `LogRecordStreamHandler.handle()` reads that length header, reads exactly that many bytes, unpickles them back into a dict, and reconstructs a `LogRecord` via `logging.makeLogRecord()` so it can be printed or re-logged through a local logger.

```bash
# Terminal 1: start the receiving server on port 9020
python3 remote_log_sender.py --mode server --port 9020

# Terminal 2: send 3 sample log messages to it
python3 remote_log_sender.py --mode client --host 127.0.0.1 --port 9020

# Point the client at a real remote collector instead of localhost
python3 remote_log_sender.py --mode client --host 10.0.0.5 --port 9020
```

#### Sample output

Server terminal:
```
Log server listening on 127.0.0.1:9020 (Ctrl+C to stop)
[REMOTE LOG] INFO: Client started - sending sample log messages
[REMOTE LOG] WARNING: This is a sample warning sent to the remote server
[REMOTE LOG] ERROR: This is a sample error sent to the remote server
```

Client terminal:
```
Sent 3 sample log messages to 127.0.0.1:9020
```

#### Variations / flags
- Instead of `print()`, have the server re-log incoming records into a local file via a second `logging` configuration, effectively building a mini centralized log store.
- Wrap the socket in TLS using `ssl.wrap_socket()`/`SSLContext` on both ends, or simply tunnel the connection through `ssh -L` for a quick, secure fix without code changes.
- Replace `SocketHandler` with `logging.handlers.SysLogHandler` if your central collector speaks syslog (rsyslog, syslog-ng) instead of this custom pickle protocol.

#### Common pitfalls
- Exposing the receiving port to the public internet — since it unpickles arbitrary data, this is a remote-code-execution risk from any untrusted sender. Bind it to a private network/VPN interface only.
- Forgetting to start the server before the client — the client will simply fail to connect (or silently drop records depending on handler configuration) if nothing is listening on the target port/host.
- Assuming delivery is guaranteed — `SocketHandler` will drop log records if the connection can't be established and no retry/backoff is configured, so it isn't a substitute for a robust log-shipping pipeline in mission-critical setups.

### 6.8 Monitor Application Logs

#### Use case
A sysadmin wants to know the moment a running application starts writing errors, without constantly re-opening the log file by hand. This script watches a log file continuously and prints an `[ALERT]` line the instant a matching keyword appears, forming the foundation of a simple, dependency-free log-based alerting loop.

#### Prerequisites

```bash
# Standard library only (no pip install needed)
python3 --version   # Python 3.x, Linux/macOS/Windows
# Needs an existing log file to tail — e.g. run log_warnings_errors.py (6.3) first
```

#### Cautions
This script follows the file by position, not by inode — if the log file is rotated (renamed/replaced, e.g. by `logrotate` or `RotatingFileHandler`) while this script is running, it will keep reading from the *old* file handle and stop seeing new lines written to the new file. It also runs as an infinite loop and must be stopped manually (Ctrl+C) or run under a process supervisor.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_app_logs.py
----------------------
Tails an application log file in real time (like `tail -f`) and
flags lines containing keywords such as ERROR, CRITICAL, or FAILED.

Usage:
    python3 monitor_app_logs.py --file logs/warnings_errors.log

Arguments:
    --file       Path to the log file to monitor (required)
    --keywords   Comma-separated keywords to flag (default: ERROR,CRITICAL,FAILED,EXCEPTION)
"""

import argparse
import os
import time


def tail_file(filepath, keywords):
    print(f"Monitoring '{filepath}' for keywords: {', '.join(keywords)}")
    print("Press Ctrl+C to stop.\n")

    with open(filepath, "r") as f:
        # Move to the end of the file so we only see new lines
        f.seek(0, os.SEEK_END)

        while True:
            line = f.readline()
            if not line:
                time.sleep(1)
                continue

            for keyword in keywords:
                if keyword.upper() in line.upper():
                    print(f"[ALERT] Matched '{keyword}': {line.strip()}")
                    break


def main():
    parser = argparse.ArgumentParser(description="Tail and monitor a log file for keywords")
    parser.add_argument("--file", required=True, help="Path to log file")
    parser.add_argument(
        "--keywords",
        default="ERROR,CRITICAL,FAILED,EXCEPTION",
        help="Comma-separated keywords to flag",
    )
    args = parser.parse_args()

    if not os.path.isfile(args.file):
        print(f"Log file not found: {args.file}")
        print("Tip: create the file first, or run log_warnings_errors.py to generate one.")
        return

    keywords = [k.strip() for k in args.keywords.split(",")]

    try:
        tail_file(args.file, keywords)
    except KeyboardInterrupt:
        print("\nStopped monitoring.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`f.seek(0, os.SEEK_END)` positions the read cursor at the current end of the file so only *new* lines appended after the script starts are considered (mirroring `tail -f`, not `tail -n +1 -f`). The loop calls `f.readline()` repeatedly; when there's nothing new, it returns an empty string, so the script sleeps 1 second and tries again rather than busy-looping the CPU. Each new line is checked, case-insensitively, against every keyword in `--keywords`.

```bash
# Watch the file produced by log_warnings_errors.py (6.3) for its default keywords
python3 monitor_app_logs.py --file logs/warnings_errors.log

# Watch a different file with custom keywords
python3 monitor_app_logs.py --file /var/log/myapp/app.log --keywords "ERROR,TIMEOUT,REFUSED"

# In another terminal, append a new line to trigger an alert
echo "2026-07-17 12:00:00 [ERROR] Something broke" >> logs/warnings_errors.log
```

#### Sample output

```
Monitoring 'logs/warnings_errors.log' for keywords: ERROR,CRITICAL,FAILED,EXCEPTION
Press Ctrl+C to stop.

[ALERT] Matched 'ERROR': 2026-07-17 12:00:00 [ERROR] Something broke
```

Stopping with Ctrl+C:
```
^C
Stopped monitoring.
```

#### Variations / flags
- Pass `--keywords "OutOfMemory,Segfault,connection refused"` to tailor detection to your application's actual failure vocabulary.
- Replace the `print()` alert with a call into `slack_alert.py` (6.16) or `email_alert.py` (6.15) so matches trigger a real notification instead of only console output.
- Run this under `systemd` (as a `Type=simple` service with `Restart=always`) or `supervisord` so it survives reboots and restarts automatically if it crashes.

#### Common pitfalls
- Running this against a file managed by log rotation without switching to an inode-aware tailing strategy — after rotation, this script keeps watching the old (now-static) file and appears to "stop working" even though the application is still logging normally to the new file.
- Choosing keywords that are too broad (e.g., `"error"` matching words like "error-handling enabled") — review sample matches before wiring this into a paging alert.
- Forgetting this script blocks the terminal indefinitely — always run it in the background (`&`, `screen`, `tmux`, or a service manager) for real use.

### 6.9 Monitor Disk Usage


#### Use case
Running out of disk space is one of the most common causes of unplanned outages — full disks crash databases, block log writes, and fail deployments. A sysadmin runs this check on a schedule (cron) so a slowly filling disk is caught at 80% instead of discovered at 100%.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, works on Linux/macOS/Windows
python3 --version
```

#### Cautions
`psutil.disk_usage(path)` reports usage for the filesystem that *contains* `path`, not necessarily a distinct mounted volume — checking `/` versus `/mnt/data` only gives different numbers if they are genuinely separate mounts. On Windows, use a drive letter (e.g., `C:\\`) instead of `/`.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_disk.py
-------------------
Checks disk usage for a given path and warns if usage exceeds a
threshold percentage. Requires the `psutil` library.

Usage:
    python3 monitor_disk.py --path / --threshold 80
"""

import argparse
import psutil


def check_disk(path, threshold):
    usage = psutil.disk_usage(path)
    percent = usage.percent

    print(f"Disk usage for '{path}': {percent}%")
    print(f"  Total: {usage.total / (1024**3):.2f} GB")
    print(f"  Used:  {usage.used / (1024**3):.2f} GB")
    print(f"  Free:  {usage.free / (1024**3):.2f} GB")

    if percent >= threshold:
        print(f"[WARNING] Disk usage ({percent}%) exceeds threshold ({threshold}%)")
        return False
    else:
        print(f"[OK] Disk usage is within the {threshold}% threshold")
        return True


def main():
    parser = argparse.ArgumentParser(description="Monitor disk usage")
    parser.add_argument("--path", default="/", help="Path/mount point to check")
    parser.add_argument("--threshold", type=float, default=80.0, help="Warning threshold percent")
    args = parser.parse_args()

    check_disk(args.path, args.threshold)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`psutil.disk_usage(path)` returns a named tuple (`total`, `used`, `free`, `percent`) for the filesystem backing `path`, all in bytes except `percent`. The script divides byte counts by `1024**3` to display gigabytes, then compares `percent` against `--threshold` to decide whether to print a warning.

```bash
# Check the root filesystem with the default 80% threshold
python3 monitor_disk.py

# Check a specific mount point with a stricter threshold
python3 monitor_disk.py --path /var --threshold 70

# On Windows, check the C: drive
python3 monitor_disk.py --path C:\ --threshold 85

# Schedule every 15 minutes via cron
*/15 * * * * /usr/bin/python3 /opt/scripts/monitor_disk.py --path / --threshold 80 >> /var/log/disk_check.log 2>&1
```

#### Sample output

```
Disk usage for '/': 46.1%
  Total: 251.97 GB
  Used:  8.55 GB
  Free:  9.98 GB
[OK] Disk usage is within the 80.0% threshold
```

Example when the threshold is exceeded:
```
Disk usage for '/': 91.4%
  Total: 100.00 GB
  Used:  91.40 GB
  Free:  8.60 GB
[WARNING] Disk usage (91.4%) exceeds threshold (80.0%)
```

#### Variations / flags
- Loop over multiple `--path` values in a wrapper script to check several mount points in one run (e.g., `/`, `/var`, `/home`).
- Make the script `sys.exit(1)` on a warning (like `health_check.py`, 6.14) so it can be used as a Nagios/cron health gate rather than only printing.
- Feed a `[WARNING]` result straight into `slack_alert.py` or `email_alert.py` for automatic notification instead of relying on someone reading cron output.

#### Common pitfalls
- Checking `/` on a system where the actual growth is happening on a separate mount (e.g., `/var/log` on its own partition) — you'll get a clean bill of health while the real partition fills up.
- Free vs. used discrepancies on filesystems with reserved blocks (e.g., ext4's reserved root percentage) — `psutil`'s `percent` matches what `df` shows, which already accounts for this, but don't assume "free" bytes are all usable by non-root processes.
- Not accounting for inode exhaustion — a filesystem can report low byte usage yet still fail to create new files if it runs out of inodes; that scenario is invisible to `disk_usage()`.

### 6.10 Monitor CPU Usage

#### Use case
Sustained high CPU usage can signal a runaway process, an inefficient batch job, or a system genuinely undersized for its workload. A sysadmin runs this check periodically to catch CPU pressure before it causes request timeouts or triggers autoscaling costs unexpectedly.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version
```

#### Cautions
`psutil.cpu_percent(interval=...)` **blocks** for the duration of `interval` while it samples — calling it with `interval=0` (or omitting `interval` on the first call) returns `0.0` or a meaningless value based on time since the last call, not a real live reading. Always pass a non-zero interval (e.g., `1` second) for a trustworthy one-shot measurement.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_cpu.py
------------------
Checks current CPU usage and warns if it exceeds a threshold
percentage. Requires the `psutil` library.

Usage:
    python3 monitor_cpu.py --threshold 85 --interval 1
"""

import argparse
import psutil


def check_cpu(threshold, interval):
    percent = psutil.cpu_percent(interval=interval)
    per_core = psutil.cpu_percent(interval=0.5, percpu=True)

    print(f"Overall CPU usage: {percent}%")
    print(f"Per-core usage: {per_core}")

    if percent >= threshold:
        print(f"[WARNING] CPU usage ({percent}%) exceeds threshold ({threshold}%)")
        return False
    else:
        print(f"[OK] CPU usage is within the {threshold}% threshold")
        return True


def main():
    parser = argparse.ArgumentParser(description="Monitor CPU usage")
    parser.add_argument("--threshold", type=float, default=85.0, help="Warning threshold percent")
    parser.add_argument("--interval", type=float, default=1.0, help="Sampling interval in seconds")
    args = parser.parse_args()

    check_cpu(args.threshold, args.interval)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script samples overall CPU usage over `--interval` seconds (blocking for that duration to get an accurate average), then takes a second, separate 0.5-second sample per core with `percpu=True` to show whether load is spread evenly or concentrated on one core (a classic symptom of a single-threaded runaway process). The two samples are independent calls, so the numbers won't always match exactly between the "overall" and "per-core" readings.

```bash
# Default: 85% threshold, 1-second sampling interval
python3 monitor_cpu.py

# Stricter threshold with a longer, smoother sampling window
python3 monitor_cpu.py --threshold 70 --interval 3

# Quick sample for scripting/testing (still blocks briefly)
python3 monitor_cpu.py --interval 0.2

# Run every 5 minutes via cron
*/5 * * * * /usr/bin/python3 /opt/scripts/monitor_cpu.py --threshold 85 >> /var/log/cpu_check.log 2>&1
```

#### Sample output

```
Overall CPU usage: 0.0%
Per-core usage: [0.0]
[OK] CPU usage is within the 85.0% threshold
```

Example under load with a 4-core machine:
```
Overall CPU usage: 92.3%
Per-core usage: [98.1, 95.4, 89.7, 90.0]
[WARNING] CPU usage (92.3%) exceeds threshold (85.0%)
```

#### Variations / flags
- Increase `--interval` (e.g., to 5 or 10 seconds) for a smoother, less noisy average on bursty workloads.
- Add logic to flag a warning only when a *single core* is pegged near 100% while overall usage looks moderate — a common sign of single-threaded bottlenecks.
- Combine with `monitor_services.py` (6.13) to correlate high CPU with a specific process by name.

#### Common pitfalls
- Calling `psutil.cpu_percent()` with no interval and no prior call in the process, which returns `0.0` on the very first invocation — always use an explicit `interval` for a one-shot script like this.
- Interpreting a brief CPU spike from a single sample as sustained load — a single reading captures a moment in time; use multiple samples over minutes for trend detection, not just alerting on one data point.
- Running this inside a container without CPU limits configured correctly — `psutil` inside some container runtimes may report host-level CPU stats rather than the container's actual allotted share, leading to misleading percentages.

### 6.11 Monitor Memory Usage

#### Use case
Memory exhaustion causes the OOM killer to terminate processes unpredictably, and heavy swap usage can silently degrade performance long before a crash occurs. A sysadmin monitors memory continuously so a slow leak in a long-running process is caught days before it becomes an outage.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version
```

#### Cautions
`psutil.virtual_memory().percent` reflects the OS's own accounting of "used" memory, which on Linux typically excludes reclaimable page cache/buffers — this is usually the right number to alert on, but it can look alarmingly high on systems that aggressively cache disk reads even when plenty of memory is actually available to applications.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_memory.py
---------------------
Checks current RAM and swap usage and warns if usage exceeds a
threshold percentage. Requires the `psutil` library.

Usage:
    python3 monitor_memory.py --threshold 80
"""

import argparse
import psutil


def check_memory(threshold):
    mem = psutil.virtual_memory()
    swap = psutil.swap_memory()

    print(f"RAM usage: {mem.percent}%")
    print(f"  Total: {mem.total / (1024**3):.2f} GB")
    print(f"  Used:  {mem.used / (1024**3):.2f} GB")
    print(f"  Available: {mem.available / (1024**3):.2f} GB")
    print(f"Swap usage: {swap.percent}%")

    if mem.percent >= threshold:
        print(f"[WARNING] RAM usage ({mem.percent}%) exceeds threshold ({threshold}%)")
        return False
    else:
        print(f"[OK] RAM usage is within the {threshold}% threshold")
        return True


def main():
    parser = argparse.ArgumentParser(description="Monitor memory (RAM/swap) usage")
    parser.add_argument("--threshold", type=float, default=80.0, help="Warning threshold percent")
    args = parser.parse_args()

    check_memory(args.threshold)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`psutil.virtual_memory()` returns a named tuple including `total`, `used`, `available`, and `percent` — `available` (not simply `total - used`) is the more meaningful figure since it accounts for memory the kernel could reclaim if needed. `psutil.swap_memory()` reports swap separately since heavy swap use is a distinct (and often worse) warning sign than high RAM usage alone.

```bash
# Default 80% threshold
python3 monitor_memory.py

# Stricter threshold for a memory-sensitive host (e.g., a database server)
python3 monitor_memory.py --threshold 65

# Run every 5 minutes via cron
*/5 * * * * /usr/bin/python3 /opt/scripts/monitor_memory.py --threshold 80 >> /var/log/mem_check.log 2>&1
```

#### Sample output

```
RAM usage: 6.0%
  Total: 3.90 GB
  Used:  0.23 GB
  Available: 3.67 GB
Swap usage: 0.0%
[OK] RAM usage is within the 80.0% threshold
```

Example on a memory-pressured host:
```
RAM usage: 88.7%
  Total: 16.00 GB
  Used:  14.19 GB
  Available: 1.81 GB
Swap usage: 42.3%
[WARNING] RAM usage (88.7%) exceeds threshold (80.0%)
```

#### Variations / flags
- Add a separate `--swap-threshold` so swap usage can trigger its own warning independent of RAM usage — high swap with moderate RAM usage is still worth flagging.
- Log results to a file each run (append mode) to build a time series you can graph later without needing a full metrics stack.
- Combine with `monitor_services.py` (6.13) to identify which running process is the likely source of a memory spike.

#### Common pitfalls
- Treating high "used" memory as automatically bad on Linux — page cache inflates `used` in some accounting methods; `available` is the number to trust for "can new processes get memory."
- Ignoring swap entirely and only alerting on RAM — a system with moderate RAM usage but heavy swapping can still perform terribly.
- Running this inside a container without memory limits set, where `psutil` may report the *host's* total memory rather than the container's cgroup limit, making the percentage misleading for containerized workloads.

### 6.12 Monitor Network Usage


#### Use case
Unexpected network saturation — from a misconfigured backup job, a runaway sync process, or an actual DDoS — can degrade every other service on a host. A sysadmin uses this script to get a quick throughput reading and confirm which interfaces are actually up before digging further with heavier tools like `iftop` or `tcpdump`.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version
```

#### Cautions
This script measures traffic across **all** interfaces combined via `psutil.net_io_counters()` (system-wide totals) — it does not break down usage per interface or per process. On a multi-homed server (multiple NICs), a spike on one interface can be masked by low traffic on another when only looking at the combined total.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_network.py
----------------------
Monitors network traffic (bytes sent/received) over a sampling
interval and reports throughput. Requires the `psutil` library.

Usage:
    python3 monitor_network.py --interval 2
"""

import argparse
import time
import psutil


def monitor(interval):
    old = psutil.net_io_counters()
    print(f"Sampling network traffic over {interval} second(s)...")
    time.sleep(interval)
    new = psutil.net_io_counters()

    sent_bytes = new.bytes_sent - old.bytes_sent
    recv_bytes = new.bytes_recv - old.bytes_recv

    sent_rate = sent_bytes / interval / 1024  # KB/s
    recv_rate = recv_bytes / interval / 1024  # KB/s

    print(f"Upload:   {sent_rate:.2f} KB/s ({sent_bytes} bytes total)")
    print(f"Download: {recv_rate:.2f} KB/s ({recv_bytes} bytes total)")

    # Per-interface stats
    print("\nPer-interface status:")
    stats = psutil.net_if_stats()
    for iface, info in stats.items():
        status = "UP" if info.isup else "DOWN"
        print(f"  {iface}: {status}, speed={info.speed}Mbps")


def main():
    parser = argparse.ArgumentParser(description="Monitor network throughput")
    parser.add_argument("--interval", type=float, default=2.0, help="Sampling interval in seconds")
    args = parser.parse_args()

    monitor(args.interval)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script takes two snapshots of `psutil.net_io_counters()` (cumulative byte counters since boot), separated by `--interval` seconds of sleep, then computes the delta to derive a rate in KB/s. `psutil.net_if_stats()` separately reports each interface's link state (`isup`) and negotiated speed, which is useful to confirm, for example, that a NIC didn't silently renegotiate down to 100 Mbps from 1 Gbps.

```bash
# Default 2-second sample
python3 monitor_network.py

# Longer sample for a smoother average during a suspected sustained spike
python3 monitor_network.py --interval 10

# Run every few minutes via cron, logging output for later review
*/5 * * * * /usr/bin/python3 /opt/scripts/monitor_network.py --interval 3 >> /var/log/net_check.log 2>&1
```

#### Sample output

```
Sampling network traffic over 1.0 second(s)...
Upload:   0.22 KB/s (221 bytes total)
Download: 0.20 KB/s (202 bytes total)

Per-interface status:
  lo: UP, speed=0Mbps
  eth0: UP, speed=0Mbps
```

(Note: virtualized/container network interfaces frequently report `speed=0Mbps` because the virtual NIC doesn't negotiate a physical link speed — this is expected, not a fault.)

#### Variations / flags
- Add a `--threshold-mbps` flag and compare the computed rate against it to print a `[WARNING]`, matching the pattern used in `monitor_cpu.py` / `monitor_disk.py`.
- Use `psutil.net_io_counters(pernic=True)` instead of the combined totals to break throughput down per interface on multi-homed hosts.
- Pair with `psutil.net_connections()` (requires elevated privileges on some OSes) if you need to see *which* connections are driving the traffic, not just the aggregate rate.

#### Common pitfalls
- Interpreting a single short sample as representative of sustained throughput — network traffic is bursty; take multiple samples or a longer interval before concluding there's a real problem.
- Forgetting that `speed=0Mbps` on virtual/loopback interfaces is normal, not an error condition, and shouldn't trigger false alarms if you add speed-based checks.
- Running with a very short `--interval` (e.g., 0.1s) on a busy host, where scheduling jitter can make the rate calculation noisy and unreliable.

### 6.13 Monitor Running Services


#### Use case
Task-oriented: "Find and confirm a process by name is alive" — a sysadmin needs to know immediately if a critical daemon has crashed or been killed, rather than discovering it only when users start reporting an outage. This script is the building block for both manual spot-checks and automated service-health cron jobs.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version
```

#### Cautions
Matching is done with a case-insensitive **substring** check against the process name (`process_name.lower() in proc.info["name"].lower()`), not an exact match — checking for `"ssh"` will also match `sshd`, but checking for a very short or generic string (like `"a"`) will produce false positives against unrelated processes. On some OSes, reading process info for processes owned by other users may raise `AccessDenied`, which this script already catches and skips gracefully.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_services.py
-----------------------
Checks whether specific processes/services (by name) are currently
running on the system. Useful for verifying that critical services
(e.g. nginx, mysql, sshd) haven't crashed. Requires `psutil`.

Usage:
    python3 monitor_services.py --services sshd,cron,nginx
"""

import argparse
import psutil


def is_running(process_name):
    for proc in psutil.process_iter(["name"]):
        try:
            if proc.info["name"] and process_name.lower() in proc.info["name"].lower():
                return True
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            continue
    return False


def check_services(service_list):
    print(f"Checking {len(service_list)} service(s)...\n")
    results = {}

    for service in service_list:
        running = is_running(service)
        results[service] = running
        status = "RUNNING" if running else "NOT RUNNING"
        symbol = "[OK]" if running else "[WARNING]"
        print(f"{symbol} {service}: {status}")

    down = [s for s, r in results.items() if not r]
    if down:
        print(f"\n{len(down)} service(s) are down: {', '.join(down)}")
    else:
        print("\nAll checked services are running.")

    return results


def main():
    parser = argparse.ArgumentParser(description="Check whether services/processes are running")
    parser.add_argument(
        "--services",
        default="sshd",
        help="Comma-separated list of process/service names to check",
    )
    args = parser.parse_args()

    service_list = [s.strip() for s in args.services.split(",")]
    check_services(service_list)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`psutil.process_iter(["name"])` iterates over every currently running process, exposing just the `name` field for efficiency (rather than every field of `proc.info`, which is more expensive to collect). For each service you asked about, `is_running()` walks that process list looking for any process whose name contains the target string, catching `NoSuchProcess`/`AccessDenied` for processes that disappear or are inaccessible mid-iteration (a normal race condition on a busy system).

```bash
# Check the default (sshd only)
python3 monitor_services.py

# Check several services at once
python3 monitor_services.py --services sshd,cron,nginx,mysql

# Confirmed test run on this environment (no sshd running here)
python3 monitor_services.py --services python3,sshd

# Run every 2 minutes via cron, alerting only on failure
*/2 * * * * /usr/bin/python3 /opt/scripts/monitor_services.py --services nginx,mysql || /opt/scripts/slack_alert.py --message "Service down!"
```

#### Sample output

```
Checking 2 service(s)...

[OK] python3: RUNNING
[WARNING] sshd: NOT RUNNING

1 service(s) are down: sshd
```

#### Variations / flags
- Make `check_services()` return a non-zero exit code when any service is down (`sys.exit(1)`), so it composes cleanly with `&&`/`||` in shell one-liners and cron jobs, similar to `health_check.py` (6.14).
- Add a `--restart-cmd` flag that runs a shell command (e.g., `systemctl restart nginx`) automatically when a check fails, for basic self-healing (use with caution and good logging).
- Match against `proc.info["exe"]` or `proc.info["cmdline"]` instead of just `name` if you need to distinguish between multiple binaries with similar process names.

#### Common pitfalls
- Overly broad substring matches — checking `--services web` will also match anything with "web" anywhere in its process name, potentially including unrelated helper processes.
- Assuming a process being "RUNNING" means the *service* is healthy — a process can be alive but hung, deadlocked, or failing health checks internally; pair this with an actual functional check (e.g., an HTTP request to the service) for real confidence.
- Running without sufficient privileges to see other users' processes, which can cause false "NOT RUNNING" results for services running under a different system account — run with appropriate privileges (e.g., `sudo`) if needed.

### 6.14 Health Check Scripts

#### Use case
Rather than running four separate monitoring scripts and manually correlating their output, a sysadmin wants one command that answers "is this box healthy, yes or no?" with a script-friendly exit code. This is the script you'd wire into a deployment pipeline's pre-flight check or a load balancer's custom health-check hook.

#### Prerequisites

```bash
pip install psutil --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version
```

#### Cautions
The exit code (`0` for healthy, `1` for unhealthy) is the whole point of this script — make sure any automation consuming it actually checks `$?` (or the equivalent in your orchestration tool) rather than only reading the printed text, or failures will go unnoticed. The `psutil.cpu_percent(interval=1)` call blocks for 1 second, so factor that into tight polling loops or aggressive health-check intervals.

#### Script

```python
#!/usr/bin/env python3
"""
health_check.py
-------------------
A combined system health check script: CPU, RAM, disk, and key
services, all in one report. Exits with a non-zero status code if
any check fails a threshold - handy for cron jobs or CI health
gates. Requires `psutil`.

Usage:
    python3 health_check.py --cpu-threshold 85 --mem-threshold 80 \
        --disk-threshold 80 --disk-path / --services sshd,cron
"""

import argparse
import sys
import psutil


def run_health_check(cpu_threshold, mem_threshold, disk_threshold, disk_path, services):
    print("=" * 50)
    print("SYSTEM HEALTH CHECK")
    print("=" * 50)

    problems = []

    # CPU
    cpu_percent = psutil.cpu_percent(interval=1)
    cpu_ok = cpu_percent < cpu_threshold
    print(f"CPU usage: {cpu_percent}% - {'OK' if cpu_ok else 'WARNING'}")
    if not cpu_ok:
        problems.append(f"CPU usage high: {cpu_percent}%")

    # Memory
    mem = psutil.virtual_memory()
    mem_ok = mem.percent < mem_threshold
    print(f"Memory usage: {mem.percent}% - {'OK' if mem_ok else 'WARNING'}")
    if not mem_ok:
        problems.append(f"Memory usage high: {mem.percent}%")

    # Disk
    disk = psutil.disk_usage(disk_path)
    disk_ok = disk.percent < disk_threshold
    print(f"Disk usage ({disk_path}): {disk.percent}% - {'OK' if disk_ok else 'WARNING'}")
    if not disk_ok:
        problems.append(f"Disk usage high: {disk.percent}%")

    # Services
    for service in services:
        running = any(
            proc.info["name"] and service.lower() in proc.info["name"].lower()
            for proc in psutil.process_iter(["name"])
        )
        print(f"Service '{service}': {'RUNNING' if running else 'NOT RUNNING'}")
        if not running:
            problems.append(f"Service down: {service}")

    print("=" * 50)
    if problems:
        print(f"HEALTH CHECK FAILED - {len(problems)} issue(s) found:")
        for p in problems:
            print(f"  - {p}")
        return False
    else:
        print("HEALTH CHECK PASSED - all systems normal")
        return True


def main():
    parser = argparse.ArgumentParser(description="Run a combined system health check")
    parser.add_argument("--cpu-threshold", type=float, default=85.0)
    parser.add_argument("--mem-threshold", type=float, default=80.0)
    parser.add_argument("--disk-threshold", type=float, default=80.0)
    parser.add_argument("--disk-path", default="/")
    parser.add_argument("--services", default="sshd", help="Comma-separated service names")
    args = parser.parse_args()

    services = [s.strip() for s in args.services.split(",")]

    healthy = run_health_check(
        args.cpu_threshold,
        args.mem_threshold,
        args.disk_threshold,
        args.disk_path,
        services,
    )

    sys.exit(0 if healthy else 1)
```

#### How it works and how to run

This script reuses the same `psutil` primitives from sections 6.9–6.13 (`cpu_percent`, `virtual_memory`, `disk_usage`, `process_iter`) but collects every failure into a `problems` list instead of stopping at the first one, so you get a complete picture in a single run. `sys.exit(0 if healthy else 1)` is the key integration point — any tool that already understands process exit codes (cron, `systemd`, CI runners, load balancer health-check scripts) can consume this without parsing any output at all.

```bash
# Run with defaults
python3 health_check.py

# Custom thresholds and multiple services
python3 health_check.py --cpu-threshold 90 --mem-threshold 85 --disk-threshold 75 --services nginx,mysql,redis

# Use the exit code directly in a shell conditional
python3 health_check.py && echo "Healthy, proceeding with deploy" || echo "Unhealthy, aborting deploy"

# Confirmed test run (single service check against a healthy sandbox)
python3 health_check.py --services python3
```

#### Sample output

Healthy run (tested):

```
==================================================
SYSTEM HEALTH CHECK
==================================================
CPU usage: 2.0% - OK
Memory usage: 6.0% - OK
Disk usage (/): 46.1% - OK
Service 'python3': RUNNING
==================================================
HEALTH CHECK PASSED - all systems normal

```
Exit code: `0`

Unhealthy example:

```

SYSTEM HEALTH CHECK
==================================================
CPU usage: 94.2% - WARNING
Memory usage: 45.0% - OK
Disk usage (/): 91.8% - WARNING
Service 'nginx': NOT RUNNING
==================================================
HEALTH CHECK FAILED - 3 issue(s) found:
  - CPU usage high: 94.2%
  - Disk usage high: 91.8%
  - Service down: nginx
```
Exit code: `1`

#### Variations / flags
- Wire this into a load balancer's HTTP health-check endpoint by wrapping it in a tiny web handler (e.g., Flask) that runs `run_health_check()` and returns HTTP 200/503 based on the boolean result.
- On failure, automatically call `email_alert.py`, `slack_alert.py`, or another notifier (6.15–6.18) with the `problems` list joined into a message body.
- Add a `--json` output mode for easier parsing by other tools/dashboards instead of the human-readable text block.

#### Common pitfalls
- Forgetting to check the exit code in automation — a cron job that doesn't act on `$?` (or logs it) gains nothing over running the individual scripts separately.
- Setting thresholds identical across very different classes of machines (a database server and a stateless web frontend usually warrant different memory/disk thresholds).
- The 1-second blocking CPU sample adds latency to every invocation — if this is called extremely frequently (e.g., every few seconds by a tight health-check loop), that overhead can add up; consider caching or a shorter interval for high-frequency polling.

### 6.15 Email Alerts

#### Use case
Email remains the lowest-common-denominator alert channel — every on-call rotation, ticketing system, and distribution list can receive it. A sysadmin wires this into any monitoring script (disk, CPU, service-down checks) so a human gets notified the moment a threshold is breached, without needing a dedicated alerting platform.

#### Prerequisites

```bash
# smtplib and email are both standard library — no pip install needed
python3 --version   # Python 3.x, Linux/macOS/Windows

# Required environment variables before running:
export SMTP_HOST=smtp.gmail.com
export SMTP_PORT=587
export SMTP_USER=you@gmail.com
export SMTP_PASSWORD=your_app_password
```

#### Cautions
Never hard-code `SMTP_PASSWORD` into the script or commit it to version control — this script deliberately reads credentials from environment variables for that reason. Gmail (and most modern providers) require an **app password**, not your normal account password, once 2-Step Verification is enabled; a regular password will fail authentication with an unhelpful error.

#### Script

```python
#!/usr/bin/env python3
"""
email_alert.py
------------------
Sends an email alert via SMTP. Works with Gmail, Outlook, or any
SMTP provider. Credentials are read from environment variables so
they are never hard-coded in the script.

Required environment variables:
    SMTP_HOST       e.g. smtp.gmail.com
    SMTP_PORT       e.g. 587
    SMTP_USER       your email address / SMTP username
    SMTP_PASSWORD   your email password or app-specific password

Usage:
    export SMTP_HOST=smtp.gmail.com
    export SMTP_PORT=587
    export SMTP_USER=you@gmail.com
    export SMTP_PASSWORD=your_app_password
    python3 email_alert.py --to admin@example.com --subject "Disk Alert" --message "Disk usage is at 90%"

Note (Gmail users): You must create an "App Password" in your Google
account security settings - normal account passwords will not work
if 2-Step Verification is enabled.
"""

import argparse
import os
import smtplib
from email.mime.text import MIMEText


def send_email_alert(to_addr, subject, message):
    host = os.environ.get("SMTP_HOST")
    port = os.environ.get("SMTP_PORT")
    user = os.environ.get("SMTP_USER")
    password = os.environ.get("SMTP_PASSWORD")

    missing = [name for name, val in [
        ("SMTP_HOST", host), ("SMTP_PORT", port),
        ("SMTP_USER", user), ("SMTP_PASSWORD", password)
    ] if not val]

    if missing:
        print(f"Missing required environment variables: {', '.join(missing)}")
        print("Set them before running this script (see the script docstring).")
        return False

    msg = MIMEText(message)
    msg["Subject"] = subject
    msg["From"] = user
    msg["To"] = to_addr

    try:
        with smtplib.SMTP(host, int(port)) as server:
            server.starttls()
            server.login(user, password)
            server.sendmail(user, [to_addr], msg.as_string())
        print(f"Email alert sent to {to_addr}")
        return True
    except Exception as e:
        print(f"Failed to send email: {e}")
        return False


def main():
    parser = argparse.ArgumentParser(description="Send an email alert via SMTP")
    parser.add_argument("--to", required=True, help="Recipient email address")
    parser.add_argument("--subject", default="System Alert", help="Email subject")
    parser.add_argument("--message", required=True, help="Email body text")
    args = parser.parse_args()

    send_email_alert(args.to, args.subject, args.message)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script reads all four SMTP settings from environment variables and fails fast with a clear message listing exactly which ones are missing, rather than an opaque connection error later. It builds a plain-text `MIMEText` message, opens a connection with `smtplib.SMTP(host, port)`, upgrades it to an encrypted connection with `.starttls()`, authenticates, and sends the message — the standard, minimal pattern for SMTP-based alerting.

```bash
export SMTP_HOST=smtp.gmail.com
export SMTP_PORT=587
export SMTP_USER=you@gmail.com
export SMTP_PASSWORD=your_app_password

python3 email_alert.py --to admin@example.com --subject "Disk Alert" --message "Disk usage is at 90% on server1"

# Wire it into a monitoring check
python3 monitor_disk.py --threshold 80 || python3 email_alert.py --to admin@example.com --subject "Disk Warning" --message "Disk usage exceeded 80% on $(hostname)"
```

#### Sample output

Success:
```
Email alert sent to admin@example.com
```

Missing environment variables:
```
Missing required environment variables: SMTP_HOST, SMTP_PASSWORD
Set them before running this script (see the script docstring).
```

Authentication failure:
```
Failed to send email: (535, b'5.7.8 Username and Password not accepted...')
```

#### Variations / flags
- Add a `--cc`/`--bcc` argument and extend `sendmail()`'s recipient list accordingly.
- Switch to `smtplib.SMTP_SSL` on port 465 instead of `STARTTLS` on 587 if your provider requires implicit TLS from the start of the connection.
- Store credentials in a secrets manager (AWS Secrets Manager, HashiCorp Vault, systemd credentials) and export them into the environment at service start time instead of a plain `.env` file, for better security hygiene.

#### Common pitfalls
- Using a regular account password instead of an app password with providers that enforce 2-Step Verification — this is the single most common cause of authentication failures with this script.
- Forgetting `int(port)` conversion issues if `SMTP_PORT` is set to something non-numeric — the script will raise a `ValueError` before even attempting to connect.
- Corporate networks that block outbound port 587/465 — the script will hang or time out with no useful error until the connection attempt itself fails; test connectivity (`telnet smtp.gmail.com 587`) if the script seems to hang.


### 6.16 Slack Alerts


#### Use case
Most engineering teams already live in Slack, so routing monitoring alerts there — instead of (or alongside) email — gets eyes on a problem faster and keeps a searchable history in a shared channel. A sysadmin plugs this script into any check script in this chapter to post real-time alerts without building a Slack app.

#### Prerequisites

```bash
pip install requests --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version

# Setup (one-time, in Slack):
# 1. Go to https://api.slack.com/apps -> Create New App -> From scratch
# 2. Enable "Incoming Webhooks" and add a webhook to your workspace/channel
# 3. Copy the Webhook URL
export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/XXX/YYY/ZZZ"
```

#### Cautions
Treat the webhook URL itself as a secret — anyone who has it can post messages to your channel as the configured app, with no further authentication required. Slack webhooks are also rate-limited (roughly one message per second per webhook); a tight monitoring loop that fires too often can get throttled or silently dropped.

#### Script

```python
#!/usr/bin/env python3
"""
slack_alert.py
------------------
Sends an alert message to a Slack channel using an Incoming Webhook
URL. Requires the `requests` library.

Setup:
    1. Go to https://api.slack.com/apps -> Create New App -> From scratch
    2. Enable "Incoming Webhooks" and add a webhook to your workspace/channel
    3. Copy the Webhook URL (looks like https://hooks.slack.com/services/XXX/YYY/ZZZ)
    4. Set it as an environment variable: export SLACK_WEBHOOK_URL="https://hooks.slack.com/..."

Usage:
    export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/XXX/YYY/ZZZ"
    python3 slack_alert.py --message "Disk usage is at 90% on server1"
"""

import argparse
import os
import requests


def send_slack_alert(message):
    webhook_url = os.environ.get("SLACK_WEBHOOK_URL")

    if not webhook_url:
        print("Missing SLACK_WEBHOOK_URL environment variable.")
        print("Set it before running this script (see the script docstring).")
        return False

    payload = {"text": message}

    try:
        response = requests.post(webhook_url, json=payload, timeout=10)
        if response.status_code == 200:
            print("Slack alert sent successfully.")
            return True
        else:
            print(f"Slack API returned status {response.status_code}: {response.text}")
            return False
    except requests.RequestException as e:
        print(f"Failed to send Slack alert: {e}")
        return False


def main():
    parser = argparse.ArgumentParser(description="Send an alert message to Slack")
    parser.add_argument("--message", required=True, help="Alert message text")
    args = parser.parse_args()

    send_slack_alert(args.message)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Slack Incoming Webhooks accept a simple JSON POST — `{"text": "your message"}` — with no OAuth flow or bot token required, which is what makes them the fastest way to wire an alert into Slack. The script posts with a 10-second timeout (to avoid hanging indefinitely if Slack is unreachable) and checks for HTTP 200 specifically, since that's what Slack returns on success for this endpoint.

```bash
export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/XXX/YYY/ZZZ"

python3 slack_alert.py --message "Disk usage is at 90% on server1"

# Wire into a monitoring check
python3 monitor_cpu.py --threshold 85 || python3 slack_alert.py --message "CPU threshold exceeded on $(hostname)"
```

#### Sample output

Success:
```
Slack alert sent successfully.
```

Missing webhook URL:
```
Missing SLACK_WEBHOOK_URL environment variable.
Set it before running this script (see the script docstring).
```

Invalid/revoked webhook:
```
Slack API returned status 404: no_service
```

#### Variations / flags
- Extend the payload with Slack's Block Kit (`{"blocks": [...]}`) for richer formatting — bolded thresholds, color-coded attachments, or buttons linking to a dashboard.
- Add a `--channel` override if your webhook is configured to allow posting to multiple channels (requires the newer Slack app permission model rather than a classic single-channel webhook).
- Batch multiple alerts into one message (join with newlines) if a check produces several problems at once, to avoid channel spam.

#### Common pitfalls
- Committing the webhook URL to source control — rotate it immediately in Slack's app settings if this happens, since the URL alone is enough to post as your app.
- Firing too many alerts too quickly and hitting Slack's rate limit, which causes some messages to be silently dropped rather than erroring loudly — add basic deduplication/cooldown logic in front of frequent monitoring checks.
- Assuming a 200 response means the message is visible to everyone — if the webhook is scoped to a channel nobody is actively watching, the alert can go unnoticed just as easily as an email in an ignored inbox.

#### Prerequisites

#### Script

```python
#!/usr/bin/env python3
"""
telegram_alert.py
---------------------
Sends an alert message to a Telegram chat using a Telegram Bot.
Requires the `requests` library.

Setup:
    1. Message @BotFather on Telegram, send /newbot, follow prompts

       and look for "chat":{"id": ...}
    4. Set environment variables:
       export TELEGRAM_BOT_TOKEN="123456789:ABC-defGhIJKlmNoPQRstuVWXyz"
       export TELEGRAM_CHAT_ID="123456789"

Usage:
    python3 telegram_alert.py --message "Backup job failed on server1"
"""

import argparse
import os
import requests


def send_telegram_alert(message):
    bot_token = os.environ.get("TELEGRAM_BOT_TOKEN")
    chat_id = os.environ.get("TELEGRAM_CHAT_ID")

    if not bot_token or not chat_id:
        print("Missing TELEGRAM_BOT_TOKEN and/or TELEGRAM_CHAT_ID environment variables.")
        print("Set them before running this script (see the script docstring).")
        return False

    url = f"https://api.telegram.org/bot{bot_token}/sendMessage"
    payload = {"chat_id": chat_id, "text": message}

    try:
        response = requests.post(url, data=payload, timeout=10)
        if response.status_code == 200:
            print("Telegram alert sent successfully.")
            return True
        else:
            print(f"Telegram API returned status {response.status_code}: {response.text}")
            return False
    except requests.RequestException as e:
        print(f"Failed to send Telegram alert: {e}")
        return False


def main():
    parser = argparse.ArgumentParser(description="Send an alert message to Telegram")
    parser.add_argument("--message", required=True, help="Alert message text")
    args = parser.parse_args()

    send_telegram_alert(args.message)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Telegram's Bot API is a plain HTTPS endpoint per method — `sendMessage` here — authenticated purely by embedding the bot token in the URL path. The script posts `chat_id` and `text` as form data (`data=payload`, not `json=payload`, which matches what Telegram's API expects) and checks for HTTP 200 to confirm delivery.

### 6.18 Discord Webhook Notifications


#### Use case
Teams and communities that coordinate primarily in Discord (common for gaming infrastructure, indie/hobbyist projects, and some open-source communities) want monitoring alerts to land in the same place as everything else. A sysadmin uses this script to post directly into a designated alerts channel with no bot hosting required.

#### Prerequisites

```bash
pip install requests --break-system-packages   # third-party package, Linux/macOS/Windows
python3 --version

# Setup (one-time, in Discord):
# 1. Go to your channel -> Edit Channel -> Integrations -> Webhooks
# 2. Click "New Webhook", copy the Webhook URL
export DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/XXX/YYY"
```

#### Cautions
Anyone holding the webhook URL can post arbitrary messages to your channel — treat it as a secret and regenerate it in Discord's channel settings if it leaks. Discord webhooks are rate-limited (Discord's global rate limits apply per webhook); very frequent alerts can be throttled with a 429 response.

#### Script

```python
#!/usr/bin/env python3
"""
discord_webhook.py
----------------------
Sends an alert message to a Discord channel using a Discord Webhook
URL. Requires the `requests` library.

Setup:
    1. In Discord, go to your channel -> Edit Channel -> Integrations -> Webhooks
    2. Click "New Webhook", copy the Webhook URL
    3. Set it as an environment variable:
       export DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/XXX/YYY"

Usage:
    python3 discord_webhook.py --message "CPU usage exceeded 90% on server1"
"""

import argparse
import os
import requests


def send_discord_alert(message):
    webhook_url = os.environ.get("DISCORD_WEBHOOK_URL")

    if not webhook_url:
        print("Missing DISCORD_WEBHOOK_URL environment variable.")
        print("Set it before running this script (see the script docstring).")
        return False

    payload = {"content": message}

    try:
        response = requests.post(webhook_url, json=payload, timeout=10)
        if response.status_code in (200, 204):
            print("Discord alert sent successfully.")
            return True
        else:
            print(f"Discord API returned status {response.status_code}: {response.text}")
            return False
    except requests.RequestException as e:
        print(f"Failed to send Discord alert: {e}")
        return False


def main():
    parser = argparse.ArgumentParser(description="Send an alert message to a Discord channel")
    parser.add_argument("--message", required=True, help="Alert message text")
    args = parser.parse_args()

    send_discord_alert(args.message)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Discord webhooks accept a JSON POST with a `content` field holding the message text and, notably, return **204 No Content** on success rather than 200 — the script checks for both `200` and `204` explicitly to avoid a false failure on the normal success path. Like the Slack and Telegram scripts, it uses a 10-second timeout so a network hiccup fails fast instead of hanging the calling script.

```bash

python3 discord_webhook.py --message "CPU usage exceeded 90% on server1"

# Wire into a monitoring check
python3 monitor_memory.py --threshold 80 || python3 discord_webhook.py --message "Memory threshold exceeded on $(hostname)"
```

```
Set it before running this script (see the script docstring).
```


#### Variations / flags
- Add `"username"` and `"avatar_url"` fields to the payload to override the webhook's display name/avatar per message (e.g., "Disk Monitor Bot" with a warning icon).
- Use Discord's `embeds` array instead of plain `content` for richer, color-coded alert cards with titles and fields.
- Add basic retry-with-backoff logic to handle Discord's `429 Too Many Requests` responses gracefully, respecting the `Retry-After` header it returns.

#### Common pitfalls
- Checking only for status `200` and treating Discord's normal `204` success response as a failure — this is a common first-try bug when adapting Slack-style code for Discord.
- Leaking the webhook URL in a public repository — Discord will not automatically revoke it; you must delete/regenerate it manually in channel settings.
- Sending messages too frequently from a tight monitoring loop and getting rate-limited, which can cause alerts to silently fail during exactly the kind of incident where they matter most — add cooldown/deduplication logic upstream.

### 6.19 Generate Audit Logs

#### Use case
Compliance requirements (SOC 2, ISO 27001, HIPAA, PCI-DSS) and basic incident-response hygiene both demand a durable, tamper-evident record of security-relevant actions: logins, file access, configuration changes, permission grants. A sysadmin builds an audit trail separate from ordinary application logs so security and compliance teams have a clean, purpose-built record to review.

#### Prerequisites

```bash
# json, logging, os, and datetime are all standard library — no pip install needed
python3 --version   # Python 3.x, Linux/macOS/Windows
mkdir -p logs
```

#### Cautions
This demo writes to a local file with no access controls or integrity protection — a real audit log should be append-only, shipped off-host promptly (see 6.7), and ideally protected against tampering (e.g., write-once storage, cryptographic chaining, or a dedicated audit service) since an audit trail an attacker can edit is not trustworthy evidence. Also make sure `details` fields never accidentally capture sensitive data (passwords, tokens, full credit card numbers) that shouldn't be persisted in logs.

#### Script

```python
#!/usr/bin/env python3
"""
audit_log.py
----------------
Generates structured audit log entries (who did what, when) in JSON
Lines format - one JSON object per line, easy to parse or feed into
log analysis tools like the ELK stack or Splunk.

Usage:
    python3 audit_log.py
"""

import json
import logging
import os
from datetime import datetime, timezone

LOG_DIR = "logs"
AUDIT_FILE = os.path.join(LOG_DIR, "audit.log")

os.makedirs(LOG_DIR, exist_ok=True)


class JsonLineFormatter(logging.Formatter):
    def format(self, record):
        entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "event": getattr(record, "event", "unspecified"),
            "user": getattr(record, "user", "unknown"),
            "details": record.getMessage(),
        }
        return json.dumps(entry)


logger = logging.getLogger("audit_log_demo")
logger.setLevel(logging.INFO)

handler = logging.FileHandler(AUDIT_FILE)
handler.setFormatter(JsonLineFormatter())
logger.addHandler(handler)


def audit(event, user, details=""):
    logger.info(details, extra={"event": event, "user": user})


def main():
    # Example audit trail entries
    audit(event="login", user="ahmad", details="User logged in from 192.168.1.10")
    audit(event="file_access", user="ahmad", details="Accessed /etc/passwd")
    audit(event="config_change", user="admin", details="Changed firewall rule: allow 443/tcp")
    audit(event="logout", user="ahmad", details="User logged out")

    print(f"Audit log entries written to: {AUDIT_FILE}")
    print("Each line is a standalone JSON object (JSON Lines format).")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`JsonLineFormatter` is a custom `logging.Formatter` subclass that ignores the usual `%`-style format string entirely and instead builds a Python dict — timestamp (UTC, ISO 8601), level, event type, user, and details — then serializes it with `json.dumps()`. The `audit(event, user, details)` helper wraps `logger.info()` so callers never touch the formatter directly; each call becomes exactly one line of valid, standalone JSON, which is the defining trait of the JSON Lines (`.jsonl`) format used by most log-shipping pipelines.

```bash

python3 audit_log.py

# Inspect the raw JSON Lines output
cat logs/audit.log

# Pretty-print each line for readability
cat logs/audit.log | python3 -m json.tool --json-lines 2>/dev/null || \
  while read -r line; do echo "$line" | python3 -m json.tool; done

# Filter for a specific user with jq (if installed)
cat logs/audit.log | jq 'select(.user == "ahmad")'
```


#### Variations / flags
- Add a `source_ip` or `session_id` field to the `audit()` helper signature for richer correlation across a session.
- Ship this file to a central collector using `remote_log_sender.py` (6.7) or a dedicated log forwarder (Filebeat, Fluent Bit) instead of leaving it only on local disk.
- Rotate and compress the audit log with `rotate_logs.py`/`compress_logs.py` (6.5/6.6) — but check your compliance requirements first, since some regimes mandate a minimum retention period for audit records before they can even be compressed or archived.

#### Common pitfalls
- Logging free-text `details` without sanitizing them, allowing sensitive data (passwords, tokens, session cookies) to end up permanently in the audit trail — treat audit log content with the same care as the data it might reference.
- Storing the audit log with the same permissions/ownership as ordinary application logs, making it just as easy to tamper with — restrict write access and consider append-only storage for genuine audit use cases.
- Treating this local-file demo as production-ready as-is — a real compliance-grade audit trail typically needs centralized, access-controlled, tamper-evident storage, not a single `logs/audit.log` on the host being audited.

### 6.20 Structured JSON Logging


#### Use case
Modern log platforms — the ELK stack, Splunk, Grafana Loki, Datadog — are built to index and query structured fields (`service`, `duration_ms`, `customer_id`), not to regex-parse free-text log lines. A sysadmin building or operating any non-trivial service adopts structured JSON logging early so that filtering, alerting, and dashboarding on specific fields ("show me every request over 500ms for customer X") works out of the box, instead of requiring brittle text parsing rules later.

#### Prerequisites

```bash
# json, logging, os, socket, traceback, datetime — all standard library
python3 --version   # Python 3.x, Linux/macOS/Windows
mkdir -p logs
```

#### Cautions
Structured logging makes it *easier*, not harder, to accidentally log sensitive data into a durable, searchable format — a `context` dict is just as capable of capturing a password or access token as a plain string message would be, so review what gets passed into `log_with_context()` just as carefully as any other log call. Also, JSON Lines output is not human-friendly to read raw in a terminal; pair it with `jq` or a log viewer rather than `cat`-ing it directly in daily use.

#### Script

```python
#!/usr/bin/env python3
"""
json_structured_logging.py
---------------------------
General-purpose structured JSON logging for applications: every log
line is a single JSON object (JSON Lines format) with a consistent
schema, including exception tracebacks when present. This is distinct
from audit_log.py (6.19), which is purpose-built for who-did-what
audit trails; this script is for everyday application/service logs
that need to be machine-parseable by ELK, Splunk, Loki, or Datadog.

Usage:
    python3 json_structured_logging.py
"""

import json
import logging
import os
import socket
import traceback
from datetime import datetime, timezone

LOG_DIR = "logs"
LOG_FILE = os.path.join(LOG_DIR, "app_structured.log")

os.makedirs(LOG_DIR, exist_ok=True)

HOSTNAME = socket.gethostname()


class StructuredJsonFormatter(logging.Formatter):
    """Formats every LogRecord as a single-line JSON object."""

    def format(self, record):
        entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "host": HOSTNAME,
            "message": record.getMessage(),
        }

        # Include any extra structured fields the caller passed in
        for key, value in getattr(record, "context", {}).items():
            entry[key] = value

        # Include exception info (type, message, traceback) if present
        if record.exc_info:
            exc_type, exc_value, exc_tb = record.exc_info
            entry["exception"] = {
                "type": exc_type.__name__ if exc_type else None,
                "message": str(exc_value) if exc_value else None,
                "traceback": traceback.format_exception(exc_type, exc_value, exc_tb),
            }

        return json.dumps(entry)


def build_logger(name="structured_app"):
    logger = logging.getLogger(name)
    logger.setLevel(logging.DEBUG)

    # Avoid attaching duplicate handlers if this is called more than once
    if not logger.handlers:
        handler = logging.FileHandler(LOG_FILE)
        handler.setFormatter(StructuredJsonFormatter())
        logger.addHandler(handler)

    return logger


def log_with_context(logger, level, message, **context):
    """Helper to attach arbitrary structured fields to a log call."""
    logger.log(level, message, extra={"context": context})


def main():
    logger = build_logger()

    # Plain structured info message
    log_with_context(
        logger, logging.INFO, "Service started",
        service="billing-api", version="2.3.1", pid=os.getpid(),
    )

    # Structured warning with request context
    log_with_context(
        logger, logging.WARNING, "Slow database query detected",
        query="SELECT * FROM invoices", duration_ms=842, threshold_ms=500,
    )

    # Structured error with an attached exception + traceback
    try:
        1 / 0
    except ZeroDivisionError:
        logger.error(
            "Failed to calculate invoice total",
            extra={"context": {"invoice_id": "INV-10432", "customer": "acme-corp"}},
            exc_info=True,
        )

    print(f"Structured JSON log entries written to: {LOG_FILE}")
    print("Each line is a standalone JSON object, ready for ELK / Splunk / Loki ingestion.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`StructuredJsonFormatter` overrides `logging.Formatter.format()` to build a Python dict instead of a format string: a fixed core (`timestamp`, `level`, `logger`, `host`, `message`), followed by whatever extra key/value pairs were passed via the `context` dict, followed by a structured `exception` block (type name, message, and full traceback as a list of lines) whenever `record.exc_info` is set. The `log_with_context()` helper hides the `extra={"context": {...}}` plumbing so callers just pass keyword arguments naturally, and `build_logger()` guards against attaching duplicate handlers if called more than once in the same process — a common source of doubled log lines in real applications.

```bash

# Pretty-print with jq (if installed)
cat logs/app_structured.log | jq .

# Filter for warnings and above with jq
cat logs/app_structured.log | jq 'select(.level == "WARNING" or .level == "ERROR")'

# Filter to a specific structured field, e.g. a slow query threshold
cat logs/app_structured.log | jq 'select(.duration_ms != null and .duration_ms > 500)'
```

#### Variations / flags
- Swap the hand-rolled formatter for the popular `python-json-logger` (`pip install python-json-logger`) package if you want a maintained, feature-rich implementation instead of a custom one.
- Add a `request_id`/`trace_id` field populated from context (e.g., via `contextvars`) so every log line from a single request can be correlated across services — the foundation of distributed tracing.
- Combine with `rotate_logs.py` (6.5) by swapping `FileHandler` for `RotatingFileHandler` so the structured log file doesn't grow unbounded in a long-running service.
- Ship the output straight into an ELK/Loki pipeline with Filebeat or Promtail — since each line is already valid JSON, no parsing grok patterns are needed on the ingestion side.

#### Common pitfalls
- Calling `build_logger()` multiple times without the duplicate-handler guard shown here, resulting in every message being logged twice (or more) as handlers pile up across repeated calls.
- Passing non-JSON-serializable objects (like a raw database connection or a custom class instance) into `context`, which will raise a `TypeError` inside `json.dumps()` at log time — stick to primitives (strings, numbers, booleans, lists, dicts) in structured fields.
- Assuming structured logging alone equals good logging — the schema still needs discipline (consistent field names across the codebase, e.g. always `duration_ms` not sometimes `duration` and sometimes `elapsed`) or downstream queries become just as inconsistent as unstructured text was.
