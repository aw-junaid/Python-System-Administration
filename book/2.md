## Chapter 2: System Commands & Process Management

Every system administrator eventually has to stop clicking around a GUI and start talking to the operating system directly — running a command, checking whether it succeeded, watching what it spawned, and cleaning it up when it misbehaves. This chapter is a field guide to doing exactly that with Python instead of a pile of shell one-liners you'll forget by next Tuesday.

Python sits in a useful middle ground here: it's expressive enough to wrap `subprocess`, `os`, and `psutil` into small, reusable, testable tools, but it stays close enough to the OS that you're not fighting an abstraction layer when something goes wrong. The 22 scripts in this chapter cover the full lifecycle of a command or process — launching it, watching it, timing it, scheduling it, and eventually killing or restarting it — plus the environment-detection utilities (OS, user, hostname, uptime) that every one of those tools quietly depends on.

A note on safety before you dive in: several of these scripts can terminate processes, restart services, or execute arbitrary shell input. None of them are toys — they are close to what you'd actually run in production, minus the guardrails a real ops team would wrap around them (audit logging, RBAC, approval gates). Treat every `kill`, `restart`, and `manage_system_service` example as if it were pointed at a machine you care about, because in real use, it eventually will be.

### 2.1 Execute External System Commands

#### Use Case

Any time you need Python to hand off work to the underlying OS — running `ls`, `whoami`, a build tool, a CLI utility — and get back exactly what it printed and how it exited. This is the "hello world" of shelling out from Python, and most of the rest of this chapter builds on it.

#### Prerequisites

Standard library only — no installation required. Works on Linux, macOS, and Windows (the demo commands used here assume a POSIX shell; on Windows, swap in `dir`/`echo` equivalents).

```bash
python3 --version   # 3.6+ is enough; no pip install needed
```

#### Cautions

This script runs whatever command string you give it with `shell=True`, so passing untrusted input (form data, user-supplied text) straight into it opens the door to arbitrary command execution. Only feed it commands you control.

#### Script

```python
#!/usr/bin/env python3
"""
execute_external_command.py

Purpose:
    Execute an external system command (e.g. "ls -la", "whoami", "echo hello")
    and print its stdout, stderr, and exit code.

Usage:
    python execute_external_command.py "echo Hello World"
    python execute_external_command.py "ls -la"

    If no command is given, a safe built-in demo command runs instead.

Expected Output:
    STDOUT: <output of the command>
    STDERR: <error output, if any>
    Exit Code: <integer>

Caution:
    - This script runs whatever command string you pass to it. Do not pass
      commands from untrusted input (e.g. web forms) without validation,
      since this can lead to arbitrary command execution.
    - shell=True is used here for convenience with full command strings;
      for security-sensitive contexts prefer passing arguments as a list
      with shell=False.
"""

import subprocess
import sys


def run_command(command: str) -> None:
    print(f"Running command: {command}\n")
    result = subprocess.run(
        command,
        shell=True,
        capture_output=True,
        text=True
    )
    print("STDOUT:")
    print(result.stdout if result.stdout else "(no output)")
    print("STDERR:")
    print(result.stderr if result.stderr else "(no errors)")
    print(f"Exit Code: {result.returncode}")


def main():
    if len(sys.argv) > 1:
        command = " ".join(sys.argv[1:])
    else:
        # Safe demo command if user provides none
        command = "echo Hello from execute_external_command.py"
    run_command(command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

The script wraps `subprocess.run()` with `shell=True`, which hands the whole string to the system shell for interpretation — meaning pipes, redirects, and wildcards all work as expected.

```bash
python execute_external_command.py "echo Hello World"
python execute_external_command.py "ls -la /tmp"
python execute_external_command.py                 # runs the built-in demo command
```

#### Sample Output

```bash
$ python execute_external_command.py "echo Hello World"
Running command: echo Hello World

STDOUT:
Hello World

STDERR:
(no errors)
Exit Code: 0
```

#### Variations / Flags

- Redirect output straight to a log file by changing the command string itself, e.g. `"ls -la > listing.log"`.
- Add a `--quiet` flag that suppresses STDOUT/STDERR printing and only reports the exit code.
- Swap `capture_output=True` for `stdout=None, stderr=None` if you want the child process to stream directly to your terminal in real time instead of being buffered.

#### Common Pitfalls

- Forgetting to quote multi-word commands on the command line, which splits them into separate `sys.argv` entries unexpectedly.
- Assuming a non-zero exit code always means "error" — some tools (like `grep`) use it to mean "no matches found," which isn't necessarily a failure.
- Running commands that aren't installed on the target machine; missing binaries usually show up as a non-zero exit code with a "command not found" message in STDERR rather than a Python exception, since the shell itself absorbs the error.

### 2.2 Execute Shell Commands Securely

#### Use Case

You need to run a command where part of it comes from a variable, a config file, or (worst case) user input — and you can't risk shell injection. This is the tool for "run this program with these exact arguments," with no shell metacharacter interpretation in between.

#### Prerequisites

Standard library only. Works identically on Linux, macOS, and Windows, since `shell=False` bypasses the platform shell entirely and calls the executable directly.

```bash
python3 --version   # no pip install needed
```

#### Cautions

Because there's no shell in the loop, pipes (`|`), redirects (`>`), and wildcards (`*`) are treated as literal characters, not special syntax — a command that worked with `shell=True` may silently fail or behave differently here. Still validate any command coming from an untrusted source before running it, even without a shell.

#### Script

```python
#!/usr/bin/env python3
"""
execute_shell_secure.py

Purpose:
    Execute a shell command SECURELY by passing arguments as a list
    (shell=False) instead of a raw string, avoiding shell injection risks.

Usage:
    python execute_shell_secure.py ls -la
    python execute_shell_secure.py echo Hello Secure World

    If no arguments are given, a safe built-in demo runs instead.

Expected Output:
    STDOUT: <output of the command>
    STDERR: <error output, if any>
    Exit Code: <integer>

Caution:
    - This script deliberately avoids shell=True to reduce injection risk.
      Because of this, shell features like pipes (|), redirects (>), and
      wildcards (*) will NOT be interpreted -- each argument is passed
      literally to the program.
    - Still validate/whitelist any command coming from an untrusted source
      (e.g. user input in a web app) before running it.
"""

import subprocess
import sys


def run_command_secure(args: list) -> None:
    print(f"Running command securely: {' '.join(args)}\n")
    try:
        result = subprocess.run(
            args,
            shell=False,
            capture_output=True,
            text=True
        )
        print("STDOUT:")
        print(result.stdout if result.stdout else "(no output)")
        print("STDERR:")
        print(result.stderr if result.stderr else "(no errors)")
        print(f"Exit Code: {result.returncode}")
    except FileNotFoundError:
        print(f"Error: command '{args[0]}' not found on this system.")


def main():
    if len(sys.argv) > 1:
        args = sys.argv[1:]
    else:
        # Safe demo command if user provides none
        args = ["echo", "Hello", "from", "execute_shell_secure.py"]
    run_command_secure(args)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

Each command-line argument becomes its own list element passed straight to `subprocess.run(args, shell=False, ...)`, so there's no shell parsing step for an attacker (or a typo) to exploit.

```bash
python execute_shell_secure.py ls -la
python execute_shell_secure.py echo Hello Secure World
python execute_shell_secure.py                       # runs the built-in demo
```

#### Sample Output

```bash
$ python execute_shell_secure.py echo Hello Secure World
Running command securely: echo Hello Secure World

STDOUT:
Hello Secure World

STDERR:
(no errors)
Exit Code: 0
```

#### Variations / Flags

- Add a whitelist check (`if args[0] not in ALLOWED_BINARIES: reject`) before running, for defense in depth on top of `shell=False`.
- Add a `--cwd` flag to run the command in a specific working directory via `subprocess.run(args, cwd=target_dir, ...)`.
- Combine with `run_command_with_timeout.py`'s `timeout=` parameter for a secure-and-bounded version.

#### Common Pitfalls

- Trying to pass shell syntax like `"ls -la | grep py"` as a single string — it will be treated as literal arguments to `ls`, not interpreted as a pipeline, and will likely fail.
- Forgetting that `FileNotFoundError` (not a non-zero exit code) is what you get here when the binary itself doesn't exist, since there's no shell to translate that into a shell-level error message.
- Assuming "secure" means "safe to run anything" — `shell=False` protects against injection, not against a genuinely destructive command you were tricked into running (e.g. `rm -rf` with the wrong path).

### 2.3 Run Scripts Without User Prompts

#### Use Case

You want a script to run completely unattended — no terminal attached, no confirmation dialogs, nothing waiting on a human — and keep running even after the person who launched it logs out. This is the pattern behind long-lived background services started from a shell that then closes.

#### Prerequisites

Standard library only (`os`, `sys`, `time`). **POSIX-only** (Linux/macOS) — it relies on `os.fork()`, which doesn't exist on Windows.

```bash
python3 --version   # no pip install needed; POSIX OS required
```

#### Cautions

Daemonizing detaches the process from your terminal entirely, so you cannot `Ctrl+C` it once it's running — you have to find its PID (printed inside the log file) and kill it explicitly. This is a minimal educational example; production daemons also redirect stdin/stdout/stderr to `/dev/null` and set a proper working directory and umask.

#### Script

```python
#!/usr/bin/env python3
"""
daemonize_process.py

Purpose:
    Demonstrate turning the current script into a background "daemon"
    process on POSIX systems (Linux/macOS) using the classic double-fork
    technique, detaching it from the controlling terminal.

Usage:
    python daemonize_process.py

    This will daemonize itself, then (as the daemon) write a line to
    /tmp/daemonize_process_demo.log every 2 seconds, 5 times, then exit.

Expected Output:
    (Immediately in your terminal:)
    Daemonizing... check /tmp/daemonize_process_demo.log for output.
    (your shell prompt returns right away)

    (Contents of /tmp/daemonize_process_demo.log after a few seconds:)
    Daemon started, PID <number>
    Tick 1
    Tick 2
    Tick 3
    Tick 4
    Tick 5
    Daemon exiting.

Caution:
    - THIS SCRIPT IS POSIX-ONLY (Linux/macOS). It uses os.fork(), which
      does not exist on Windows; running it there will raise an
      AttributeError.
    - Daemonizing detaches the process from your terminal completely;
      you cannot Ctrl+C it. To stop it early, find its PID (printed
      inside the log file) and use kill_process.py from this same
      folder.
    - This is a minimal educational example. Production-grade daemons
      typically also redirect stdin/stdout/stderr to /dev/null or log
      files and set a proper working directory/umask, which this
      script does in simplified form.
"""

import os
import sys
import time

LOG_FILE = "/tmp/daemonize_process_demo.log"


def daemonize():
    if sys.platform.startswith("win"):
        print("Error: daemonize_process.py requires a POSIX OS (Linux/macOS). "
              "os.fork() is not available on Windows.")
        sys.exit(1)

    # First fork
    pid = os.fork()
    if pid > 0:
        # Parent process exits, letting the child continue detached
        print(f"Daemonizing... check {LOG_FILE} for output.")
        sys.exit(0)

    os.setsid()  # start a new session, detaching from controlling terminal

    # Second fork (prevents daemon from acquiring a controlling terminal again)
    pid = os.fork()
    if pid > 0:
        sys.exit(0)

    # Now running as the daemon process
    with open(LOG_FILE, "a") as f:
        f.write(f"Daemon started, PID {os.getpid()}\n")
        f.flush()
        for i in range(1, 6):
            time.sleep(2)
            f.write(f"Tick {i}\n")
            f.flush()
        f.write("Daemon exiting.\n")

    os._exit(0)


def main():
    daemonize()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

The classic double-fork: the first fork lets the original parent exit immediately (returning your shell prompt), `os.setsid()` starts a new session so the child has no controlling terminal, and the second fork prevents the daemon from ever reacquiring one. What's left keeps running fully detached, with no prompts and no attached terminal to block on.

```bash
python daemonize_process.py
tail -f /tmp/daemonize_process_demo.log
```

#### Sample Output

```bash
$ python daemonize_process.py
Daemonizing... check /tmp/daemonize_process_demo.log for output.
$ cat /tmp/daemonize_process_demo.log
Daemon started, PID 48213
Tick 1
Tick 2
Tick 3
Tick 4
Tick 5
Daemon exiting.
```

#### Variations / Flags

- Redirect `os.dup2()` for stdin/stdout/stderr to `/dev/null` for a fully production-style daemon that never touches the launching terminal at all.
- Write the daemon's PID to a dedicated PID file (`/tmp/mydaemon.pid`) on startup, so a companion script can find and stop it without grepping process names.
- On Linux, pair this pattern with `manage_system_service.py` and a real `systemd` unit file instead of a hand-rolled daemon for anything you actually want to survive reboots.

#### Common Pitfalls

- Running this on Windows — `os.fork()` simply doesn't exist there, and the script exits with an error by design rather than crashing unexpectedly.
- Losing track of the daemon's PID once it's detached; without checking the log file or using `monitor_processes.py`, there's no straightforward way to find it again.
- Forgetting that print statements after daemonizing go nowhere useful — once detached, the process has no terminal to print to, which is why this script writes to a log file instead.

### 2.4 Run Scripts With User Prompts

#### Use Case

You're about to run something irreversible — deleting files, restarting a service, wiping a directory — and you want a human to explicitly confirm before it happens, instead of trusting a script to always be run with the right arguments in the right context.

#### Prerequisites

Standard library only (`subprocess`, `sys`). Cross-platform (Linux, macOS, Windows) — it uses `input()`, which behaves the same everywhere.

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script requires an interactive terminal — `input()` will hang or raise an `EOFError` if run inside a cron job, CI pipeline, or anything else with no attached stdin. Don't use this pattern for anything meant to run unattended (see 2.3 for that case instead).

#### Script

```python
#!/usr/bin/env python3
"""
run_script_with_prompt.py

Purpose:
    Demonstrate gating a potentially destructive command behind an
    explicit interactive confirmation prompt, so a script never runs
    something irreversible without a human saying "yes" first.

Usage:
    python run_script_with_prompt.py "rm -rf /tmp/demo_folder"
    python run_script_with_prompt.py "echo Hello World"

    If no command is given, a safe built-in demo command is used.

Expected Output:
    About to run: echo Hello World
    Proceed? [y/N]: y
    STDOUT: Hello World
    Exit Code: 0

Caution:
    - This script BLOCKS waiting for terminal input. It will hang (or
      raise EOFError) if run with no attached stdin, e.g. inside cron,
      CI, or another non-interactive context.
    - Any answer other than exactly "y" or "yes" (case-insensitive) is
      treated as "no" and aborts safely -- this is deliberate, so a
      stray Enter keypress never accidentally confirms.
"""

import subprocess
import sys


def confirm(prompt: str) -> bool:
    try:
        answer = input(f"{prompt} [y/N]: ").strip().lower()
    except EOFError:
        print("\nNo interactive input available; aborting for safety.")
        return False
    return answer in ("y", "yes")


def run_with_confirmation(command: str) -> None:
    print(f"About to run: {command}")
    if not confirm("Proceed?"):
        print("Aborted by user.")
        return

    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    print("STDOUT:", result.stdout.strip() if result.stdout else "(no output)")
    if result.stderr.strip():
        print("STDERR:", result.stderr.strip())
    print(f"Exit Code: {result.returncode}")


def main():
    if len(sys.argv) > 1:
        command = " ".join(sys.argv[1:])
    else:
        command = "echo Hello from run_script_with_prompt.py"
    run_with_confirmation(command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

The script prints the command it's about to run, calls `input()` to block for an explicit answer, and only proceeds to `subprocess.run()` if the answer is exactly `y` or `yes` (case-insensitive). Anything else — including just pressing Enter — aborts safely.

```bash
python run_script_with_prompt.py "echo Hello World"
python run_script_with_prompt.py "rm -rf /tmp/demo_folder"
python run_script_with_prompt.py                          # runs the built-in demo
```

#### Sample Output

```bash
$ python run_script_with_prompt.py "rm -rf /tmp/demo_folder"
About to run: rm -rf /tmp/demo_folder
Proceed? [y/N]: n
Aborted by user.
```

```bash
$ python run_script_with_prompt.py "echo Hello World"
About to run: echo Hello World
Proceed? [y/N]: y
STDOUT: Hello World
Exit Code: 0
```

#### Variations / Flags

- Add a `--yes` / `-y` flag that skips the prompt entirely, for the same script to be usable both interactively and inside automation once you trust it.
- Require typing the exact command back (not just `y`) as a stronger confirmation step for especially destructive actions.
- Add a countdown ("proceeding in 5 seconds unless you press Ctrl+C") as a softer alternative to a hard yes/no gate.

#### Common Pitfalls

- Wiring this into a cron job or CI pipeline, where there's no terminal to answer the prompt — the script will hang until it times out, or raise `EOFError` if stdin is closed entirely.
- Assuming a blank Enter press means "yes" — this script deliberately treats anything other than `y`/`yes` as "no," which is the safer default but can surprise users used to other tools' conventions.
- Piping input non-interactively (e.g. `yes | python run_script_with_prompt.py ...`) to bypass the prompt in automation — this works, but defeats the entire safety purpose of the confirmation gate, so use the `--yes` variation above instead if that's really what you want.

### 2.5 Launch Background Processes

#### Use Case

You need to kick off a long-running command — a server, a watcher, a long job — from inside a larger automation script, and get control back immediately instead of blocking until it finishes.

#### Prerequisites

Standard library only (`subprocess`). Works on Linux, macOS, and Windows (backgrounding behavior differs slightly on Windows, but the non-blocking launch still works via `Popen`).

```bash
python3 --version   # no pip install needed
```

#### Cautions

The background process keeps running even after this script exits — it is not tied to the parent's lifetime. Use `monitor_processes.py` or `kill_process.py` to check on or stop it later using the printed PID, or it will keep consuming resources unnoticed.

#### Script

```python
#!/usr/bin/env python3
"""
launch_background_process.py

Purpose:
    Launch a command as a background process (non-blocking) and
    immediately return control to the script, printing the new PID.

Usage:
    python launch_background_process.py "sleep 30"
    python launch_background_process.py "ping -c 10 127.0.0.1"

    If no command is given, a safe built-in demo runs (a 5-second sleep).

Expected Output:
    Started background process with PID: <number>
    (the parent script exits immediately without waiting)

Caution:
    - The background process keeps running even after this script exits.
      Use monitor_processes.py or kill_process.py (from this same folder)
      to check on or stop it later, using the printed PID.
    - On Windows, backgrounding behaves differently; this script targets
      Linux/macOS (POSIX) behavior primarily but will still launch the
      process non-blocking on Windows via Popen.
"""

import subprocess
import sys


def launch_background(command: str) -> int:
    process = subprocess.Popen(
        command,
        shell=True,
        stdout=subprocess.DEVNULL,
        stderr=subprocess.DEVNULL
    )
    print(f"Started background process with PID: {process.pid}")
    print("The parent script is not waiting for it to finish.")
    return process.pid


def main():
    if len(sys.argv) > 1:
        command = " ".join(sys.argv[1:])
    else:
        command = "sleep 5"
    launch_background(command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`subprocess.Popen()` starts the process and returns immediately, unlike `subprocess.run()` which blocks until completion. Output is discarded to `DEVNULL` here since nothing is around to read it once the parent moves on.

```bash
python launch_background_process.py "sleep 30"
python launch_background_process.py "ping -c 10 127.0.0.1"
python launch_background_process.py                        # 5-second demo sleep
```

#### Sample Output

```bash
$ python launch_background_process.py "sleep 30"
Started background process with PID: 51922
The parent script is not waiting for it to finish.
$
```

#### Variations / Flags

- Capture stdout/stderr to a log file instead of `DEVNULL` (`stdout=open("job.log", "w")`) if you need to inspect the background process's output later.
- Write the returned PID to a file so a separate script can look it up and manage it, similar to how real daemons use PID files.
- Combine with `get_process_info.py` to poll the launched PID's status periodically without blocking.

#### Common Pitfalls

- Losing the PID because it was only printed to a terminal that later closed — always persist it somewhere (a file, a variable in a longer-running parent) if you'll need it again.
- Forgetting that "non-blocking" doesn't mean "supervised" — if the background process crashes, nothing here notices or restarts it; that's what `restart_process.py` and real process supervisors (systemd, supervisord) are for.
- Launching background processes inside a script that then exits entirely (e.g. a cron job) — depending on the OS and shell, the child may or may not survive the parent's exit; test this on your actual target platform before relying on it.

### 2.6 Monitor Running Processes

#### Use Case

You need to see what's actually running on a machine right now — every PID, name, CPU%, and memory% — usually to find the process that's hogging resources before it takes the whole system down. This is the "look before you kill" step ahead of `kill_process.py`.

#### Prerequisites

Requires the third-party `psutil` library. Works on Linux, macOS, and Windows.

```bash
pip install psutil

```

#### Cautions

Listing all system processes may include ones owned by other users, which you may not have permission to inspect fully — those are skipped silently rather than raising an error. The first CPU% reading for any process needs a short warm-up interval to be meaningful; this script handles that automatically.

#### Script

```python
#!/usr/bin/env python3
"""
monitor_processes.py

Purpose:
    List currently running processes with basic info: PID, name,
    CPU usage %, and memory usage %.

Usage:
    python monitor_processes.py
    python monitor_processes.py --filter python

Expected Output:
    A table like:
    PID     NAME                 CPU%     MEM%
    1234    python3              1.2      0.5
    5678    chrome               15.3     4.2
    ...

Caution:
    - Requires the third-party 'psutil' library (see requirements.txt).
    - CPU% on the first call may read as 0.0 for each process; this is
      normal for psutil, which needs a short interval between two
      samples to compute accurate CPU usage (this script handles that
      automatically with a brief warm-up).
    - Listing all system processes may include processes owned by other
      users, which you may not have permission to inspect fully.
"""

import sys
import time

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def monitor(filter_name: str = None) -> None:
    # Warm-up call so cpu_percent readings are meaningful
    for proc in psutil.process_iter(['pid', 'name']):
        try:
            proc.cpu_percent(interval=None)
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            pass
    time.sleep(0.5)

    print(f"{'PID':<8}{'NAME':<25}{'CPU%':<10}{'MEM%':<10}")
    for proc in psutil.process_iter(['pid', 'name']):
        try:
            name = proc.info['name'] or "unknown"
            if filter_name and filter_name.lower() not in name.lower():
                continue
            cpu = proc.cpu_percent(interval=None)
            mem = proc.memory_percent()
            print(f"{proc.info['pid']:<8}{name:<25}{cpu:<10.1f}{mem:<10.2f}")
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            continue


def main():
    filter_name = None
    if "--filter" in sys.argv:
        idx = sys.argv.index("--filter")
        if idx + 1 < len(sys.argv):
            filter_name = sys.argv[idx + 1]
    monitor(filter_name)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`psutil.process_iter()` walks every visible process. The script first "primes" each process's CPU counter with a no-wait call, sleeps briefly, then reads real percentages on the second pass — that's why there's a short pause before the table prints.

```bash
python monitor_processes.py
python monitor_processes.py --filter python
```

#### Sample Output

```bash
$ python monitor_processes.py --filter python
PID     NAME                     CPU%      MEM%
1234    python3                  1.2       0.51
7788    python3                  0.0       0.22
```

#### Variations / Flags

- Add a `--sort-by cpu` or `--sort-by mem` flag and sort the collected rows before printing, instead of printing in iteration order.
- Add a `--top N` flag to only show the N heaviest processes, useful on machines with hundreds of running processes.
- Loop this on an interval (`--watch 2`) for a poor-man's `top`/`htop` inside a script.

#### Common Pitfalls

- Running without elevated privileges and being surprised some processes show `AccessDenied` and get silently skipped — this is expected behavior on Linux/macOS for processes owned by other users, and can be worked around with `sudo` if you truly need full visibility.
- Reading the very first CPU% value before the warm-up sleep completes and interpreting a `0.0` as "idle," when it may just be an artifact of `psutil` needing two samples to compute a delta.
- Assuming MEM% reflects total system memory pressure — it's per-process resident memory as a percentage of total RAM, not a measure of whether the system as a whole is under memory pressure (swap usage isn't reflected here).

### 2.7 Kill a Process

#### Use Case

Task-oriented: "Find and kill a process by name or PID." Once `monitor_processes.py` has told you which process is misbehaving — hung, leaking memory, stuck in a loop — this is the tool that actually terminates it.

#### Prerequisites

Requires the third-party `psutil` library. Works on Linux, macOS, and Windows.

```bash
pip install psutil
```

#### Cautions

This is destructive. Killing the wrong PID can crash applications, corrupt in-progress work, or destabilize your system if you kill a critical system process. Double-check the PID with `monitor_processes.py` before running this for real, and note that killing processes owned by another user or root may require elevated privileges.

#### Script

```python
#!/usr/bin/env python3
"""
kill_process.py

Purpose:
    Terminate (kill) a running process by its PID.

Usage:
    python kill_process.py <PID>
    python kill_process.py 1234

    If no PID is given, the script lists a few current processes so you
    can pick a valid PID to try (nothing is killed in this demo mode).

Expected Output:
    Success: "Process <PID> terminated successfully."
    Failure: an error message explaining why (not found, access denied).

Caution:
    - THIS IS DESTRUCTIVE. Killing the wrong PID can crash applications,
      corrupt in-progress work, or destabilize your system if you kill
      a critical system process.
    - Double check the PID with monitor_processes.py (in this same
      folder) before running this script for real.
    - Killing processes owned by another user or root may require
      elevated privileges (e.g. running with sudo on Linux/macOS).
"""

import sys

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def kill_process(pid: int) -> None:
    try:
        proc = psutil.Process(pid)
        name = proc.name()
        proc.terminate()
        proc.wait(timeout=5)
        print(f"Process {pid} ({name}) terminated successfully.")
    except psutil.NoSuchProcess:
        print(f"Error: no process found with PID {pid}.")
    except psutil.AccessDenied:
        print(f"Error: access denied when trying to kill PID {pid}. "
              f"Try running with elevated privileges.")
    except psutil.TimeoutExpired:
        print(f"Warning: process {pid} did not terminate in time; "
              f"it may need to be force-killed.")


def list_sample_processes(limit: int = 10) -> None:
    print("No PID given. Here are some currently running processes:\n")
    print(f"{'PID':<8}{'NAME'}")
    count = 0
    for proc in psutil.process_iter(['pid', 'name']):
        print(f"{proc.info['pid']:<8}{proc.info['name']}")
        count += 1
        if count >= limit:
            break
    print("\nRun again as: python kill_process.py <PID>")


def main():
    if len(sys.argv) > 1:
        try:
            pid = int(sys.argv[1])
        except ValueError:
            print("Error: PID must be an integer.")
            sys.exit(1)
        kill_process(pid)
    else:
        list_sample_processes()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`psutil.Process(pid).terminate()` sends a graceful termination signal (`SIGTERM` on POSIX), then `wait(timeout=5)` gives the process up to 5 seconds to actually exit before reporting a timeout warning.

```bash
python kill_process.py            # lists sample PIDs, kills nothing
python kill_process.py 1234       # terminates PID 1234
```

#### Sample Output

```bash
$ python kill_process.py
No PID given. Here are some currently running processes:

PID     NAME
1        systemd
612      sshd
1234     python3
...

Run again as: python kill_process.py <PID>
```

```bash
$ python kill_process.py 1234
Process 1234 (python3) terminated successfully.
```

#### Variations / Flags

- Add a `--force` flag that calls `proc.kill()` (sends `SIGKILL`) instead of `proc.terminate()` when a process refuses to exit gracefully.
- Add a `--name <string>` mode (like `restart_process.py` uses) to kill by matching process name instead of requiring an exact PID.
- Add a `--dry-run` flag that only prints what *would* be killed, without calling `terminate()` — useful for double-checking a name-based match before committing to it.

#### Common Pitfalls

- Killing a PID that's been reused by a completely different process since you last checked it — always re-verify with `monitor_processes.py`/`get_process_info.py` immediately before killing, especially in scripts with any delay between lookup and kill.
- Expecting `terminate()` to be instant — some processes trap `SIGTERM` and take time to clean up (or ignore it entirely), which is why the `wait(timeout=5)` and force-kill variation exist.
- Running without sufficient privileges against a system or another user's process and getting `AccessDenied` instead of a clean kill — this typically requires `sudo` on Linux/macOS or an elevated prompt on Windows.

### 2.8 Restart a Process

#### Use Case

A service has wedged itself into a bad state and the fastest fix is to stop it and start it fresh — the classic "turn it off and on again," scripted so it can be triggered automatically or from a larger automation pipeline.

#### Prerequisites

Requires the third-party `psutil` library. Works on Linux, macOS, and Windows.

```bash
pip install psutil
```

#### Cautions

This does NOT gracefully drain connections or save state for you — it force-stops the old process before starting the new one, so any unsaved work in the old process is lost. If matching by `--name`, ALL processes whose name contains that string will be stopped, so use a specific/unique identifier to avoid killing the wrong thing.

#### Script

```python
#!/usr/bin/env python3
"""
restart_process.py

Purpose:
    "Restart" a process by terminating an existing one (matched by PID
    or by name) and relaunching a given command in its place.

Usage:
    python restart_process.py --pid 1234 --command "python my_server.py"
    python restart_process.py --name "my_server.py" --command "python my_server.py"

    If no arguments are given, a safe built-in demo runs: it launches a
    dummy process, then restarts it.

Expected Output:
    Stopping old process (if found)...
    Old process stopped.
    Starting new process...
    New process started with PID: <number>

Caution:
    - This does NOT gracefully drain connections or save state for you;
      it force-stops the old process before starting the new one. Any
      unsaved work in the old process will be lost.
    - If matching by --name, ALL processes whose name contains that
      string will be stopped, so use a specific/unique identifier to
      avoid killing the wrong thing.
"""

import subprocess
import sys
import time

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def stop_by_pid(pid: int) -> None:
    try:
        proc = psutil.Process(pid)
        proc.terminate()
        proc.wait(timeout=5)
        print(f"Old process (PID {pid}) stopped.")
    except psutil.NoSuchProcess:
        print(f"No process with PID {pid} found; nothing to stop.")
    except psutil.AccessDenied:
        print(f"Access denied stopping PID {pid}.")


def stop_by_name(name: str) -> None:
    stopped_any = False
    for proc in psutil.process_iter(['pid', 'name']):
        if name.lower() in (proc.info['name'] or "").lower():
            try:
                proc.terminate()
                proc.wait(timeout=5)
                print(f"Stopped process {proc.info['pid']} ({proc.info['name']}).")
                stopped_any = True
            except (psutil.NoSuchProcess, psutil.AccessDenied):
                continue
    if not stopped_any:
        print(f"No running process matched name '{name}'.")


def start_new(command: str) -> int:
    process = subprocess.Popen(command, shell=True)
    print(f"New process started with PID: {process.pid}")
    return process.pid


def parse_args():
    args = sys.argv[1:]
    pid = None
    name = None
    command = None
    i = 0
    while i < len(args):
        if args[i] == "--pid" and i + 1 < len(args):
            pid = int(args[i + 1]); i += 2
        elif args[i] == "--name" and i + 1 < len(args):
            name = args[i + 1]; i += 2
        elif args[i] == "--command" and i + 1 < len(args):
            command = args[i + 1]; i += 2
        else:
            i += 1
    return pid, name, command


def main():
    pid, name, command = parse_args()

    if not command:
        # Demo mode: launch a dummy sleep process, then restart it
        print("No --command given, running demo mode.")
        demo_command = "sleep 20"
        proc = subprocess.Popen(demo_command, shell=True)
        print(f"Launched demo process PID: {proc.pid}")
        time.sleep(1)
        print("Stopping old process...")
        stop_by_pid(proc.pid)
        print("Starting new process...")
        start_new(demo_command)
        return

    print("Stopping old process (if found)...")
    if pid:
        stop_by_pid(pid)
    elif name:
        stop_by_name(name)
    else:
        print("No --pid or --name given; skipping stop step.")

    print("Starting new process...")
    start_new(command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

The script stops the old process (by exact PID or by substring name match), then unconditionally launches a fresh copy of `--command` with `subprocess.Popen()`. Stop and start are two independent steps, so there's a brief window where the service is down entirely.

```bash
python restart_process.py --pid 1234 --command "python my_server.py"
python restart_process.py --name "my_server.py" --command "python my_server.py"
python restart_process.py                                            # demo mode
```

#### Sample Output

```bash
$ python restart_process.py --name "my_server.py" --command "python my_server.py"
Stopping old process (if found)...
Stopped process 8842 (python3).
Starting new process...
New process started with PID: 9130
```

#### Variations / Flags

- Add a health-check step after `start_new()` (e.g. poll a port or endpoint) and only report success once the new process actually responds, not just once it launches.
- Add a `--graceful-timeout` flag that sends `SIGTERM` and waits longer before escalating to `SIGKILL`, for services that need extra time to shut down cleanly.
- For anything running in production, prefer `manage_system_service.py` and a real service manager (`systemctl restart ...`) over this pattern — this script is for ad hoc / dev scenarios.

#### Common Pitfalls

- Using `--name` with a substring that matches more than intended — e.g. `--name "server"` could match `my_server.py`, `backup_server.py`, and `server_monitor.py` all at once, stopping all of them.
- Assuming the restart is atomic — there's a real gap between the old process stopping and the new one starting where the service is simply down; anything depending on it will see errors during that window.
- Not checking whether `start_new()`'s process actually stayed running — `Popen()` returns as soon as the process launches, even if it crashes half a second later, so the printed PID alone doesn't guarantee success.

### 2.9 Suspend and Resume Processes

#### Use Case

Task-oriented: "Temporarily pause a runaway process without losing its state, then let it continue later." Unlike killing a process, suspending it (SIGSTOP) freezes it in place — memory, open files, and progress all stay intact — so you can free up CPU right now and resume (SIGCONT) exactly where it left off.

#### Prerequisites

Requires the third-party `psutil` library, which wraps `SIGSTOP`/`SIGCONT` on POSIX and the equivalent suspend/resume calls on Windows.

```bash
pip install psutil
```

#### Cautions

A suspended process still holds all its memory and any locks or file handles it had open — other processes waiting on those resources will block indefinitely until you resume it, which can cause cascading stalls elsewhere in the system. Don't suspend a process that other services depend on staying responsive.

#### Script

```python
#!/usr/bin/env python3
"""
suspend_resume_process.py

Purpose:
    Suspend (pause) a running process without terminating it, and later
    resume it exactly where it left off.

Usage:
    python suspend_resume_process.py suspend <PID>
    python suspend_resume_process.py resume <PID>

    If no arguments are given, a safe built-in demo runs: it launches a
    dummy "sleep" process, suspends it, checks its status, then resumes it.

Expected Output:
    Process <PID> suspended.
    Process <PID> resumed.

Caution:
    - A suspended process holds onto all of its memory and any open
      file handles/locks; other processes waiting on those resources
      will block until it is resumed.
    - Suspending a process that's mid-write to a file or database can
      leave dependent systems (health checks, load balancers) treating
      it as unresponsive, so use this deliberately, not casually.
"""

import subprocess
import sys
import time

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def suspend(pid: int) -> None:
    try:
        proc = psutil.Process(pid)
        proc.suspend()
        print(f"Process {pid} suspended.")
    except psutil.NoSuchProcess:
        print(f"Error: no process found with PID {pid}.")
    except psutil.AccessDenied:
        print(f"Error: access denied suspending PID {pid}.")


def resume(pid: int) -> None:
    try:
        proc = psutil.Process(pid)
        proc.resume()
        print(f"Process {pid} resumed.")
    except psutil.NoSuchProcess:
        print(f"Error: no process found with PID {pid}.")
    except psutil.AccessDenied:
        print(f"Error: access denied resuming PID {pid}.")


def main():
    if len(sys.argv) >= 3 and sys.argv[1] in ("suspend", "resume"):
        action, pid = sys.argv[1], int(sys.argv[2])
        suspend(pid) if action == "suspend" else resume(pid)
        return

    # Demo mode
    print("No arguments given, running demo mode.")
    demo = subprocess.Popen("sleep 30", shell=True)
    print(f"Launched demo process PID: {demo.pid}")
    time.sleep(1)

    suspend(demo.pid)
    print(f"Status while suspended: {psutil.Process(demo.pid).status()}")
    time.sleep(2)

    resume(demo.pid)
    print(f"Status after resume: {psutil.Process(demo.pid).status()}")
    demo.terminate()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`psutil.Process.suspend()` sends `SIGSTOP` on POSIX (or calls `SuspendThread`-equivalent APIs on Windows), which freezes the process without ending it. `resume()` sends `SIGCONT` to let it continue exactly where it was frozen.

```bash
python suspend_resume_process.py suspend 4821
python suspend_resume_process.py resume 4821
python suspend_resume_process.py                 # runs the built-in demo
```

#### Sample Output

```bash
$ python suspend_resume_process.py
No arguments given, running demo mode.
Launched demo process PID: 6120
Process 6120 suspended.
Status while suspended: stopped
Process 6120 resumed.
Status after resume: sleeping
```

#### Variations / Flags

- Add a `--for <seconds>` flag that suspends, sleeps, then automatically resumes — useful for a "quiet this process for 30 seconds" one-liner.
- Add a `--name` mode, matching the pattern used in `restart_process.py`, to suspend/resume by process name instead of requiring an exact PID.
- Combine with `monitor_processes.py` to suspend whichever process currently has the highest CPU%, as a manual load-shedding tool.

#### Common Pitfalls

- Suspending a process that's holding a lock another process needs — the second process won't crash, but it will hang silently until the first one is resumed, which can look like an unrelated outage.
- Forgetting a process is suspended and wondering why it looks "hung" in monitoring — `status()` will report `stopped`, which is easy to miss if you're only checking whether the PID still exists.
- Not available at all inside some restricted containers/sandboxes where signal delivery is limited — test `suspend()`/`resume()` in your actual target environment before depending on it.

### 2.10 Retrieve Process Information

#### Use Case

You have a PID — from `monitor_processes.py`, a log file, or a support ticket — and need the full picture: what it's called, whether it's healthy, how much CPU/memory it's using, where its executable lives, when it started, and what command line launched it.

#### Prerequisites

Requires the third-party `psutil` library. Works on Linux, macOS, and Windows.

```bash
pip install psutil
```

#### Cautions

Some fields (executable path, command line) may be unavailable or restricted depending on OS permissions, returning `"N/A"` instead of raising an error — don't mistake that for the process not existing. CPU% and memory are a live snapshot; the values will already be stale by the time you read them.

#### Script

```python
#!/usr/bin/env python3
"""
get_process_info.py

Purpose:
    Retrieve detailed information about a specific process by PID:
    name, status, CPU%, memory usage, executable path, start time,
    and command line.

Usage:
    python get_process_info.py <PID>
    python get_process_info.py 1234

    If no PID is given, this script prints info about its own process
    as a safe demo.

Expected Output:
    PID:          1234
    Name:         python3
    Status:       running
    CPU%:         0.3
    Memory (MB):  25.4
    Executable:   /usr/bin/python3
    Started:      2026-07-16 10:00:00
    Command line: python3 script.py

Caution:
    - Some fields (like executable path or command line) may be
      unavailable or restricted depending on OS permissions, returning
      "N/A" instead of raising an error.
    - Process information is a live snapshot; values like CPU% and
      memory will change moment to moment.
"""

import os
import sys
import datetime

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def print_process_info(pid: int) -> None:
    try:
        proc = psutil.Process(pid)
        with proc.oneshot():
            name = proc.name()
            status = proc.status()
            cpu = proc.cpu_percent(interval=0.3)
            mem_mb = proc.memory_info().rss / (1024 * 1024)
            try:
                exe = proc.exe()
            except (psutil.AccessDenied, psutil.ZombieProcess):
                exe = "N/A"
            started = datetime.datetime.fromtimestamp(
                proc.create_time()
            ).strftime("%Y-%m-%d %H:%M:%S")
            try:
                cmdline = " ".join(proc.cmdline()) or "N/A"
            except (psutil.AccessDenied, psutil.ZombieProcess):
                cmdline = "N/A"

        print(f"PID:          {pid}")
        print(f"Name:         {name}")
        print(f"Status:       {status}")
        print(f"CPU%:         {cpu}")
        print(f"Memory (MB):  {mem_mb:.1f}")
        print(f"Executable:   {exe}")
        print(f"Started:      {started}")
        print(f"Command line: {cmdline}")
    except psutil.NoSuchProcess:
        print(f"Error: no process found with PID {pid}.")
    except psutil.AccessDenied:
        print(f"Error: access denied reading info for PID {pid}.")


def main():
    if len(sys.argv) > 1:
        pid = int(sys.argv[1])
    else:
        pid = os.getpid()
        print("No PID given; showing info for this script's own process.\n")
    print_process_info(pid)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`proc.oneshot()` batches all the underlying system calls together for efficiency, since several of these fields would otherwise each trigger a separate syscall. The 0.3-second `interval` on `cpu_percent()` deliberately blocks briefly to get a real (non-zero) CPU reading rather than psutil's usual "compare to last call" shortcut.

```bash
python get_process_info.py 1234
python get_process_info.py            # shows info about itself
```

#### Sample Output

```bash
$ python get_process_info.py 1234
PID:          1234
Name:         python3
Status:       running
CPU%:         0.3
Memory (MB):  25.4
Executable:   /usr/bin/python3
Started:      2026-07-16 10:00:00
Command line: python3 script.py
```

#### Variations / Flags

- Add a `--json` flag that dumps the same fields as a JSON object, for feeding into logging or monitoring pipelines instead of a human-readable printout.
- Add a `--watch <seconds>` flag that re-prints the same info on a loop, turning this into a lightweight single-process dashboard.
- Add a `--tree` flag using `proc.children(recursive=True)` to also list every descendant process, not just the one PID.

A companion script, `get_parent_pid.py`, answers a narrower but frequently useful question: *who launched this process?* It reports a PID's parent PID (PPID) and the parent's name — handy for tracing whether a process was spawned by a shell, a service manager, or another script.

### 2.11 Get Exit Status of Commands

#### Use Case

Any script that needs to make a decision based on whether a previous step succeeded — "did the backup actually finish," "did the deploy script fail" — needs to read and interpret an exit code, not just eyeball the output.

#### Prerequisites

Standard library only. Works on Linux, macOS, and Windows (the demo command assumes a POSIX shell — swap `ls /nonexistent` for a Windows equivalent if needed).

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script uses `shell=True` for convenience with full command strings, so avoid passing untrusted input directly as a command. Exit code meaning is command-specific: `0` conventionally means success, but the exact meaning of any non-zero value depends entirely on the program you ran.

#### Script

```python
#!/usr/bin/env python3
"""
get_exit_status.py

Purpose:
    Run a command and report its exit status code, along with a plain
    English interpretation (success vs failure).

Usage:
    python get_exit_status.py "ls /nonexistent"
    python get_exit_status.py "echo hi"

    If no command is given, a safe built-in demo runs.

Expected Output:
    Command: ls /nonexistent
    Exit Code: 2
    Result: FAILURE (non-zero exit code)

Caution:
    - Exit code meaning is command-specific: 0 conventionally means
      success, and any non-zero value means some kind of failure, but
      the exact number's meaning depends on the program that was run.
    - This script uses shell=True for convenience with full command
      strings; avoid passing untrusted input directly as a command.
"""

import subprocess
import sys


def get_exit_status(command: str) -> None:
    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    print(f"Command: {command}")
    print(f"Exit Code: {result.returncode}")
    if result.returncode == 0:
        print("Result: SUCCESS (exit code 0)")
    else:
        print("Result: FAILURE (non-zero exit code)")
    if result.stdout.strip():
        print(f"STDOUT: {result.stdout.strip()}")
    if result.stderr.strip():
        print(f"STDERR: {result.stderr.strip()}")


def main():
    if len(sys.argv) > 1:
        command = " ".join(sys.argv[1:])
    else:
        command = "ls /this/path/does/not/exist"
    get_exit_status(command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`subprocess.run()`'s `.returncode` attribute holds the exit code the OS reported for the process. The script maps `0` to a success message and anything else to a generic failure message, printed alongside whatever the command wrote to stdout/stderr.

```bash
python get_exit_status.py "echo hi"
python get_exit_status.py "ls /nonexistent"
python get_exit_status.py                    # runs the built-in failing demo
```

#### Sample Output

```bash
$ python get_exit_status.py "ls /nonexistent"
Command: ls /nonexistent
Exit Code: 2
Result: FAILURE (non-zero exit code)
STDERR: ls: cannot access '/nonexistent': No such file or directory
```

#### Variations / Flags

- Add an `--expect <code>` flag that compares the actual exit code against a specific expected value, rather than only checking zero vs. non-zero.
- Chain this into `run_scheduled_jobs.py` or `execute_startup_script.py`-style sequences, stopping a pipeline as soon as a non-zero code appears.
- Add a lookup table of well-known exit codes (127 = command not found, 126 = not executable, 130 = terminated by Ctrl+C) for friendlier diagnostics on common failures.

#### Common Pitfalls

- Assuming every non-zero code means "something is broken" — tools like `grep` and `diff` use non-zero exit codes to mean "no match" or "files differ," which is informational, not an error.
- Not checking exit codes at all and instead parsing stdout text for the word "error," which is fragile compared to just reading `.returncode`.
- Forgetting that a command that isn't found on the system doesn't raise a Python exception here (because of `shell=True`) — it comes back as a non-zero exit code with a shell-generated message in STDERR instead.

### 2.12 Execute Commands Asynchronously

#### Use Case

You need to kick off a command and keep the rest of your program responsive while it runs — for example, running several independent health checks concurrently without one slow check blocking the others — using Python's native `asyncio` rather than threads or processes managed by hand.

#### Prerequisites

Standard library only (`asyncio`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # 3.7+ recommended for asyncio.run(); no pip install needed
```

#### Cautions

All given commands run at the same time, so make sure they don't compete destructively for the same resource or file if run together. This script waits for ALL commands to finish before exiting (`asyncio.gather`) — it does not "fire and forget."

#### Script

```python
#!/usr/bin/env python3
"""
execute_async_command.py

Purpose:
    Execute a command asynchronously using Python's asyncio, so the
    script can perform other work while the command runs in parallel.

Usage:
    python execute_async_command.py "sleep 2 && echo done1" "sleep 1 && echo done2"

    If no commands are given, two safe built-in demo commands run
    concurrently.

Expected Output:
    Started: sleep 2 && echo done1
    Started: sleep 1 && echo done2
    Finished: sleep 1 && echo done2 -> exit 0
    Finished: sleep 2 && echo done1 -> exit 0
    (Note faster commands can finish before slower ones, proving true
    concurrency.)

Caution:
    - All given commands run at the same time; make sure they don't
      compete destructively for the same resource/file if run together.
    - This script waits for ALL commands to finish before exiting
      (asyncio.gather); it does not "fire and forget".
"""

import asyncio
import sys


async def run_command_async(command: str) -> None:
    print(f"Started: {command}")
    process = await asyncio.create_subprocess_shell(
        command,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE
    )
    stdout, stderr = await process.communicate()
    print(f"Finished: {command} -> exit {process.returncode}")
    if stdout:
        print(f"  stdout: {stdout.decode().strip()}")
    if stderr:
        print(f"  stderr: {stderr.decode().strip()}")


async def main_async(commands):
    await asyncio.gather(*(run_command_async(cmd) for cmd in commands))


def main():
    if len(sys.argv) > 1:
        commands = sys.argv[1:]
    else:
        commands = [
            "sleep 2 && echo done1",
            "sleep 1 && echo done2"
        ]
    asyncio.run(main_async(commands))


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`asyncio.create_subprocess_shell()` launches each command as a non-blocking subprocess; `asyncio.gather()` runs all the `run_command_async()` coroutines concurrently and waits for every one of them to complete before the script exits.

```bash
python execute_async_command.py "sleep 2 && echo done1" "sleep 1 && echo done2"
python execute_async_command.py                                   # runs the built-in demo
```

#### Sample Output

```bash
$ python execute_async_command.py "sleep 2 && echo done1" "sleep 1 && echo done2"
Started: sleep 2 && echo done1
Started: sleep 1 && echo done2
Finished: sleep 1 && echo done2 -> exit 0
  stdout: done2
Finished: sleep 2 && echo done1 -> exit 0
  stdout: done1
```

#### Variations / Flags

- Add a semaphore (`asyncio.Semaphore(n)`) to cap how many commands run concurrently, instead of firing all of them at once regardless of count.
- Swap `asyncio.gather()` for `asyncio.as_completed()` to process and print each command's result as soon as it finishes, rather than only after all commands are done launching their print statements.
- Add per-command timeouts with `asyncio.wait_for()` so one hung command can't stall the entire batch indefinitely.

#### Common Pitfalls

- Assuming async means "runs instantly" — it means "runs concurrently, without blocking the event loop," but the script as written still waits for every command to finish before returning control.
- Forgetting `asyncio.run()` requires Python 3.7+, and that nesting it inside an already-running event loop (e.g. inside a Jupyter notebook) raises a `RuntimeError` — use `await main_async(...)` directly in those contexts instead.
- Running commands with heavy destructive side effects concurrently — e.g. two commands both writing to the same file — since true concurrency means there's no guaranteed ordering between them.

### 2.13 Run Multiple Commands in Parallel

#### Use Case

You have a batch of independent commands — checks across several servers, several file conversions, several test suites — and want them all running at once using real OS-level parallelism, with results collected once everything finishes.

#### Prerequisites

Standard library only (`subprocess`, `concurrent.futures`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # no pip install needed
```

#### Cautions

Commands run truly in parallel as separate OS processes, so if they write to the same file or resource, you may get race conditions or corrupted output — keep each command's side effects isolated. This script waits for every command to finish before printing results; it's not meant for long-lived background services (see 2.5 for that case).

#### Script

```python
#!/usr/bin/env python3
"""
run_parallel_commands.py

Purpose:
    Run multiple independent commands at the same time using a thread
    pool, and collect all of their results once they finish.

Usage:
    python run_parallel_commands.py "sleep 2 && echo A" "sleep 1 && echo B" "echo C"

    If no commands are given, three safe built-in demo commands run
    in parallel.

Expected Output:
    Results (order may vary based on which finishes first):
    [echo C] exit=0 output=C
    [sleep 1 && echo B] exit=0 output=B
    [sleep 2 && echo A] exit=0 output=A

Caution:
    - Commands run truly in parallel (separate OS processes), so if
      they write to the same file or resource, you may get race
      conditions or corrupted output. Keep each command's side effects
      isolated.
    - This script waits for every command to finish before printing
      results; it is not meant for long-lived background services (see
      launch_background_process.py for that use case).
"""

import subprocess
import sys
from concurrent.futures import ThreadPoolExecutor, as_completed


def run_one(command: str):
    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    return command, result.returncode, result.stdout.strip(), result.stderr.strip()


def run_parallel(commands):
    print(f"Running {len(commands)} commands in parallel...\n")
    results = []
    with ThreadPoolExecutor(max_workers=len(commands)) as executor:
        futures = {executor.submit(run_one, cmd): cmd for cmd in commands}
        for future in as_completed(futures):
            results.append(future.result())

    print("Results (order may vary based on which finishes first):")
    for command, exit_code, output, error in results:
        print(f"[{command}] exit={exit_code} output={output}", end="")
        if error:
            print(f" error={error}")
        else:
            print()


def main():
    if len(sys.argv) > 1:
        commands = sys.argv[1:]
    else:
        commands = [
            "sleep 2 && echo A",
            "sleep 1 && echo B",
            "echo C"
        ]
    run_parallel(commands)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

A `ThreadPoolExecutor` submits one `subprocess.run()` call per command; because each call is a real OS process (not just a Python thread), they genuinely run in parallel regardless of Python's GIL. `as_completed()` yields each result as soon as it's ready, so faster commands report before slower ones even if they were submitted later.

```bash
python run_parallel_commands.py "sleep 2 && echo A" "sleep 1 && echo B" "echo C"
python run_parallel_commands.py                                       # built-in demo
```

#### Sample Output

```bash
$ python run_parallel_commands.py "sleep 2 && echo A" "sleep 1 && echo B" "echo C"
Running 3 commands in parallel...

Results (order may vary based on which finishes first):
[echo C] exit=0 output=C
[sleep 1 && echo B] exit=0 output=B
[sleep 2 && echo A] exit=0 output=A
```

#### Variations / Flags

- Cap concurrency with a fixed `max_workers=N` instead of `len(commands)`, so a batch of 200 commands doesn't try to launch 200 processes at once.
- Add a `--fail-fast` flag that cancels remaining futures as soon as one command reports a non-zero exit code.
- Preserve input order in the output by collecting results into a dict keyed by the original list index, instead of printing in completion order.

#### Common Pitfalls

- Running commands with side effects that collide — e.g. two commands both appending to the same log file — since there's no coordination between them once they're launched.
- Setting `max_workers=len(commands)` on a very large batch and accidentally spawning far more OS processes than the machine can comfortably handle at once; cap it explicitly for large lists.
- Mistaking thread-pool parallelism for a guarantee of process isolation on shared state within Python itself — the *commands* run as separate OS processes, but the `results` list and print statements are still coordinated by Python's GIL-protected main thread, which is why this pattern is safe here.

### 2.14 Schedule Delayed Execution

#### Use Case

Task-oriented: "Run this command once, but not right now — in N seconds/minutes." This is the lightweight building block for "remind me," "clean this up later," or "give the other service time to finish starting" scenarios, without pulling in a full scheduler or cron entry for a single one-off delay.

#### Prerequisites

Standard library only (`subprocess`, `threading`, `time`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script blocks the calling process for the entire delay by default — it is meant for short, one-off delays inside a script, not for delays measured in hours or days, which should instead be handed off to a real scheduler (cron, Task Scheduler, or a job queue) that survives the calling process exiting.

#### Script

```python
#!/usr/bin/env python3
"""
schedule_delayed_execution.py

Purpose:
    Run a single command after waiting a fixed delay, either blocking
    the caller until it runs or firing it in the background via a
    daemon thread so the caller can continue in the meantime.

Usage:
    python schedule_delayed_execution.py --delay 5 --command "echo Hello"
    python schedule_delayed_execution.py --delay 5 --command "echo Hello" --background

    If no arguments are given, a safe built-in demo runs a 3-second
    delayed "echo".

Expected Output:
    Waiting 5 second(s) before running: echo Hello
    STDOUT: Hello
    Exit Code: 0

Caution:
    - By default this BLOCKS for the full delay duration; it's meant
      for short, one-off waits inside a script, not hours-long delays.
    - The --background mode fires the command in a daemon thread and
      returns immediately, but that thread (and the scheduled command)
      is killed if the main script exits before the delay elapses.
"""

import subprocess
import sys
import threading
import time


def run_after_delay(delay_seconds: float, command: str) -> None:
    print(f"Waiting {delay_seconds} second(s) before running: {command}")
    time.sleep(delay_seconds)
    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    print("STDOUT:", result.stdout.strip() if result.stdout else "(no output)")
    if result.stderr.strip():
        print("STDERR:", result.stderr.strip())
    print(f"Exit Code: {result.returncode}")


def schedule_background(delay_seconds: float, command: str) -> threading.Thread:
    thread = threading.Thread(
        target=run_after_delay, args=(delay_seconds, command), daemon=True
    )
    thread.start()
    print(f"Scheduled '{command}' to run in {delay_seconds}s (background thread started).")
    return thread


def parse_args():
    args = sys.argv[1:]
    delay = None
    command = None
    background = "--background" in args
    if background:
        args.remove("--background")
    i = 0
    while i < len(args):
        if args[i] == "--delay" and i + 1 < len(args):
            delay = float(args[i + 1]); i += 2
        elif args[i] == "--command" and i + 1 < len(args):
            command = args[i + 1]; i += 2
        else:
            i += 1
    return delay, command, background


def main():
    delay, command, background = parse_args()
    if delay is None or command is None:
        delay, command = 3, "echo Hello from schedule_delayed_execution.py"
        print("No arguments given, running demo mode.")

    if background:
        thread = schedule_background(delay, command)
        thread.join()  # only for demo purposes, so the script doesn't exit early
    else:
        run_after_delay(delay, command)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

In the default (blocking) mode, the script simply `time.sleep()`s for the delay, then runs the command — simple and predictable. In `--background` mode, the delay-and-run logic happens on a daemon thread instead, so the calling code can do other work in the meantime (the demo `.join()`s only so the example doesn't exit before you see the output).

```bash
python schedule_delayed_execution.py --delay 5 --command "echo Hello"
python schedule_delayed_execution.py --delay 5 --command "echo Hello" --background
python schedule_delayed_execution.py                                        # 3s demo
```

#### Sample Output

```bash
$ python schedule_delayed_execution.py --delay 3 --command "echo Hello"
Waiting 3 second(s) before running: echo Hello
STDOUT: Hello
Exit Code: 0
```

#### Variations / Flags

- Add a `--at "HH:MM"` mode that computes the delay from the current time to a specific clock time, instead of only accepting a relative number of seconds.
- Add cancellation support by returning a `threading.Event` that, when set, causes the delayed thread to skip running the command.
- For delays longer than a few minutes, or for anything that must survive the calling script exiting, use `run_scheduled_jobs.py` in a persistent process, or better, hand it off to `cron`/Task Scheduler instead of relying on `time.sleep()`.

#### Common Pitfalls

- Using `--background` mode and letting the main script exit before the delay elapses — daemon threads are killed immediately when the main program exits, so the scheduled command never runs.
- Using this for long delays (hours) in a script meant to be short-lived — the process has to stay alive and consume resources for the entire wait, which is wasteful compared to a real scheduler.
- Assuming `time.sleep()` delays are perfectly precise — under system load, the actual wake-up time can drift slightly past the requested delay; don't rely on this for anything requiring sub-second timing accuracy.

### 2.15 Run Scheduled Jobs

#### Use Case

You need something to run repeatedly on a fixed interval for a bounded number of times — a lightweight in-process stand-in for cron when you're demoing, testing, or don't want the overhead of registering an OS-level scheduled task for something short-lived.

#### Prerequisites

Standard library only (`subprocess`, `time`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script BLOCKS the terminal for the entire duration (roughly `interval * repeats` seconds) — it's meant for short demos/tests, not long-running production job scheduling. For real production scheduling that survives reboots and doesn't require keeping a Python process alive, use native OS schedulers (cron on Linux/macOS, Task Scheduler on Windows).

#### Script

```python
#!/usr/bin/env python3
"""
run_scheduled_jobs.py

Purpose:
    Run a command repeatedly on a fixed interval (like a simple
    in-process cron job), for a limited number of repetitions.

Usage:
    python run_scheduled_jobs.py --interval 5 --repeats 3 --command "echo tick"

    If no arguments are given, a safe built-in demo runs "echo tick"
    every 2 seconds, 3 times total.

Expected Output:
    [Run 1/3] echo tick
    STDOUT: tick
    [Run 2/3] echo tick
    STDOUT: tick
    [Run 3/3] echo tick
    STDOUT: tick
    Scheduled job finished after 3 run(s).

Caution:
    - This script BLOCKS the terminal for the entire duration
      (interval * repeats seconds approximately); it's meant for short
      demos/tests, not long-running production job scheduling.
    - For real production scheduling that survives reboots and doesn't
      require keeping a Python process alive, use native OS schedulers:
      cron (Linux/macOS) or Task Scheduler (Windows) — see
      execute_startup_scripts.py and manage_system_services.py for
      related concepts.
    - Press Ctrl+C to stop early if you set a large --repeats value.
"""

import subprocess
import sys
import time


def run_scheduled(interval_seconds: float, repeats: int, command: str) -> None:
    for i in range(1, repeats + 1):
        print(f"[Run {i}/{repeats}] {command}")
        result = subprocess.run(command, shell=True, capture_output=True, text=True)
        print("STDOUT:", result.stdout.strip() if result.stdout else "(no output)")
        if result.stderr.strip():
            print("STDERR:", result.stderr.strip())
        if i < repeats:
            time.sleep(interval_seconds)
    print(f"Scheduled job finished after {repeats} run(s).")


def parse_args():
    args = sys.argv[1:]
    interval = None
    repeats = None
    command = None
    i = 0
    while i < len(args):
        if args[i] == "--interval" and i + 1 < len(args):
            interval = float(args[i + 1]); i += 2
        elif args[i] == "--repeats" and i + 1 < len(args):
            repeats = int(args[i + 1]); i += 2
        elif args[i] == "--command" and i + 1 < len(args):
            command = args[i + 1]; i += 2
        else:
            i += 1
    return interval, repeats, command


def main():
    interval, repeats, command = parse_args()
    if interval is None or repeats is None or command is None:
        interval, repeats, command = 2, 3, "echo tick"
        print("No arguments given, running demo mode.")
    try:
        run_scheduled(interval, repeats, command)
    except KeyboardInterrupt:
        print("\nStopped early by user (Ctrl+C).")


if __name__ == "__main__":
    main()
```

#### How It Works & Running

A simple `for` loop runs the command, prints its result, and sleeps for `interval` seconds before the next iteration — skipping the final sleep so the script doesn't wait needlessly after the last run.

```bash
python run_scheduled_jobs.py --interval 5 --repeats 3 --command "echo tick"
python run_scheduled_jobs.py                                          # 2s x 3 demo
```

#### Sample Output

```bash
$ python run_scheduled_jobs.py --interval 2 --repeats 3 --command "echo tick"
[Run 1/3] echo tick
STDOUT: tick
[Run 2/3] echo tick
STDOUT: tick
[Run 3/3] echo tick
STDOUT: tick
Scheduled job finished after 3 run(s).
```

#### Variations / Flags

- Add a `--forever` flag that removes the `repeats` bound and loops until `Ctrl+C`, for a longer-lived (but still Python-process-dependent) watcher.
- Add `--stop-on-failure` so the loop breaks the moment any run returns a non-zero exit code, similar to `execute_startup_script.py`'s behavior.
- For anything that needs to survive the terminal closing or the machine rebooting, generate a real crontab entry (`*/5 * * * * your_command`) instead of relying on this script staying alive.

#### Common Pitfalls

- Setting a large `--repeats` with a long `--interval` and expecting it to run in the background — it blocks the calling terminal/process the entire time, so combine it with `launch_background_process.py` or `nohup` if you need it detached.
- Forgetting the loop is strictly time-based, not "run every N seconds regardless of how long the command itself takes" — if the command takes longer than `interval` to run, the actual cadence between runs will drift later than expected.
- Relying on this in place of cron for anything genuinely long-running or business-critical; it has no persistence, no logging infrastructure, and no recovery if the Python process itself crashes mid-run.

### 2.16 Execute Startup Scripts

#### Use Case

Task-oriented: "Run a sequence of setup steps in order, and stop immediately if any of them fails." This is the pattern behind application startup routines, environment bootstrapping, and — at a much larger scale — real init systems like systemd.

#### Prerequisites

Standard library only (`subprocess`). Works on Linux, macOS, and Windows (the demo commands assume a POSIX-friendly shell).

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script stops at the FIRST failing command (non-zero exit code) to avoid continuing in a broken state — later steps are never attempted once one fails. This is a demonstration of the startup-script pattern; to have your OS actually run something like this at boot, register it with the OS's own mechanism (systemd, Task Scheduler, or launchd) instead of relying on this script being manually invoked.

#### Script

```python
#!/usr/bin/env python3
"""
execute_startup_script.py

Purpose:
    Demonstrate/execute a "startup script" pattern: run one or more
    setup commands in sequence when a system or application starts,
    stopping early if any command fails.

Usage:
    python execute_startup_script.py

    This runs a demo list of startup commands defined inside the script
    (edit the STARTUP_COMMANDS list below to customize for your use
    case, e.g. checking directories exist, setting env, etc.).

Expected Output:
    [Startup 1/3] echo Checking environment...
    STDOUT: Checking environment...
    [Startup 2/3] echo Creating temp directory...
    STDOUT: Creating temp directory...
    [Startup 3/3] echo Startup complete.
    STDOUT: Startup complete.
    All startup steps completed successfully.

Caution:
    - This script stops at the FIRST failing command (non-zero exit
      code) to avoid continuing in a broken state, similar to how a
      real startup/init script should behave.
    - This is a demonstration of the startup-script pattern in Python.
      To have your OS actually run this at boot, register it with your
      OS's own mechanism: systemd (Linux), Task Scheduler (Windows), or
      launchd (macOS) — see manage_system_services.py for a related
      example.
"""

import subprocess
import sys

# Edit this list to define your own startup sequence.
STARTUP_COMMANDS = [
    "echo Checking environment...",
    "echo Creating temp directory...",
    "echo Startup complete."
]


def run_startup_sequence(commands) -> bool:
    total = len(commands)
    for i, command in enumerate(commands, start=1):
        print(f"[Startup {i}/{total}] {command}")
        result = subprocess.run(command, shell=True, capture_output=True, text=True)
        if result.stdout.strip():
            print("STDOUT:", result.stdout.strip())
        if result.stderr.strip():
            print("STDERR:", result.stderr.strip())
        if result.returncode != 0:
            print(f"Startup step {i} failed with exit code {result.returncode}. Stopping.")
            return False
    print("All startup steps completed successfully.")
    return True


def main():
    success = run_startup_sequence(STARTUP_COMMANDS)
    sys.exit(0 if success else 1)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

The script iterates through `STARTUP_COMMANDS` in order, running each with `subprocess.run()` and checking `.returncode` immediately after. The moment any step returns non-zero, the loop returns `False` and no further steps run — the same fail-fast behavior a real init system uses.

```bash
python execute_startup_script.py
```

Customize the sequence by editing the `STARTUP_COMMANDS` list at the top of the file.

#### Sample Output

```bash
$ python execute_startup_script.py
[Startup 1/3] echo Checking environment...
STDOUT: Checking environment...
[Startup 2/3] echo Creating temp directory...
STDOUT: Creating temp directory...
[Startup 3/3] echo Startup complete.
STDOUT: Startup complete.
All startup steps completed successfully.
```

#### Variations / Flags

- Add a `--continue-on-error` flag that runs every step regardless of failures and reports a final summary, for situations where later steps are genuinely independent of earlier ones.
- Load `STARTUP_COMMANDS` from an external file (YAML/JSON) instead of hardcoding the list, so the sequence can be edited without touching the script.
- Wrap this script itself in a systemd `ExecStartPre=` directive or a Windows Task Scheduler action to actually run it at boot, rather than manually invoking it.

#### Common Pitfalls

- Editing `STARTUP_COMMANDS` to include steps with side effects that aren't idempotent — if the script is ever re-run after a partial failure, a step like "create directory" may now fail simply because it already exists, masking the real problem.
- Forgetting that `main()` calls `sys.exit(1)` on failure — if you import this script's functions elsewhere instead of running it as `__main__`, that exit call won't fire, which is usually what you want for reuse but is easy to overlook.
- Assuming this genuinely runs at system boot just because it's called a "startup script" — without registering it with systemd/launchd/Task Scheduler, it only runs when someone (or something) explicitly executes it.

### 2.17 Manage System Services

#### Use Case

Task-oriented: "Start, stop, restart, or check the status of a named OS service (nginx, ssh, a custom daemon) from a script," using whatever native service manager the current platform actually provides, instead of hand-rolling process management for something the OS already manages.

#### Prerequisites

Standard library only (`subprocess`, `platform`). Targets `systemctl` on Linux; best-effort support for `launchctl` on macOS and `sc` on Windows.

```bash
python3 --version   # no pip install needed; systemctl must exist on the target Linux host
```

#### Cautions

This can affect real system services. Stopping or restarting the wrong service (your SSH daemon, display manager, or network manager) can disrupt the system or lock you out of a remote session — double-check the service name before running start/stop/restart. Managing services typically requires elevated privileges; on Linux/macOS you may need `sudo`.

#### Script

```python
#!/usr/bin/env python3
"""
manage_system_service.py

Purpose:
    Start, stop, restart, or check the status of a system service,
    using the appropriate native tool for the current OS
    (systemctl on Linux, sc/net on Windows, launchctl on macOS - Linux
    is the primary supported target here via systemctl).

Usage:
    python manage_system_service.py status ssh
    python manage_system_service.py start ssh
    python manage_system_service.py stop ssh
    python manage_system_service.py restart ssh

    If no arguments are given, this script only prints usage help (it
    will NOT touch any real service without an explicit service name).

Expected Output:
    Running: systemctl status ssh
    <systemctl output describing the service state>

Caution:
    - THIS CAN AFFECT REAL SYSTEM SERVICES. Stopping or restarting the
      wrong service (e.g. your SSH daemon, display manager, or network
      manager) can disrupt your system or lock you out of a remote
      session. Double-check the service name before running start/
      stop/restart.
    - Managing services typically requires elevated privileges; on
      Linux/macOS you may need to run this script with sudo, e.g.:
          sudo python manage_system_service.py restart nginx
    - This script currently targets Linux's systemctl. On macOS/Windows
      you would need to adapt the underlying command (launchctl / sc).
"""

import platform
import subprocess
import sys


def manage_service(action: str, service_name: str) -> None:
    system = platform.system()
    if system == "Linux":
        command = f"systemctl {action} {service_name}"
    elif system == "Darwin":
        print("Note: macOS uses launchctl, which has a different syntax "
              "than systemctl. This demo will attempt a best-effort command.")
        command = f"sudo launchctl {action} {service_name}"
    elif system == "Windows":
        action_map = {"start": "start", "stop": "stop", "status": "query", "restart": "restart"}
        command = f"sc {action_map.get(action, action)} {service_name}"
    else:
        print(f"Unsupported OS: {system}")
        return

    print(f"Running: {command}")
    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    print(result.stdout if result.stdout else "(no output)")
    if result.stderr.strip():
        print("STDERR:", result.stderr.strip())
    print(f"Exit Code: {result.returncode}")


def print_usage():
    print("Usage: python manage_system_service.py <start|stop|restart|status> <service_name>")
    print("Example: python manage_system_service.py status ssh")
    print("No service name was given, so nothing was executed.")


def main():
    if len(sys.argv) >= 3:
        action = sys.argv[1].lower()
        service_name = sys.argv[2]
        if action not in ("start", "stop", "restart", "status"):
            print(f"Unknown action '{action}'. Use start, stop, restart, or status.")
            return
        manage_service(action, service_name)
    else:
        print_usage()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`platform.system()` picks which native command template to build (`systemctl` on Linux, `launchctl` on macOS, `sc` on Windows), then runs it via `subprocess.run()` and prints whatever the OS tool itself reports — this script is essentially a thin, cross-platform-aware wrapper, not a reimplementation of service management.

```bash
python manage_system_service.py status ssh
sudo python manage_system_service.py restart nginx
python manage_system_service.py                       # prints usage, touches nothing
```

#### Sample Output

```bash
$ python manage_system_service.py status ssh
Running: systemctl status ssh
● ssh.service - OpenBSD Secure Shell server
     Loaded: loaded (/lib/systemd/system/ssh.service; enabled)
     Active: active (running) since ...
Exit Code: 0
```

#### Variations / Flags

- Add a `--dry-run` flag that prints the command it *would* run without executing it — valuable given how disruptive the wrong `restart` can be.
- Add an explicit `--yes` confirmation requirement before `stop`/`restart` actions, following the pattern from 2.4, so a mistyped service name doesn't take something down unattended.
- Parse `systemctl status`'s output into a structured `{active: bool, enabled: bool}` result instead of just printing raw text, for use in health-check automation.

#### Common Pitfalls

- Running `stop` or `restart` on a service you're currently connected through (most classically, `sshd` over an SSH session) — this can lock you out immediately with no way to reconnect.
- Assuming the macOS `launchctl` command built here is correct for your specific service — `launchctl`'s syntax differs meaningfully from `systemctl`'s and this script's macOS branch is explicitly a best-effort placeholder, not a tested equivalent.
- Forgetting `sudo`/elevated privileges are required for most real service actions — without them, expect a permission-denied exit code rather than a silent no-op.

### 2.18 Detect Operating System

#### Use Case

Any automation script that needs to branch its behavior between Linux, macOS, and Windows — different service managers, different path separators, different shell syntax — starts with a reliable read of what OS it's actually running on.

#### Prerequisites

Standard library only (`platform`, `sys`). Works identically on Linux, macOS, and Windows — that's the entire point of the script.

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script only reads system information; it makes no changes, so it's always safe to run. Output format varies significantly between Linux, macOS, and Windows — don't assume the same fields exist identically on every platform if you build automation on top of this.

#### Script

```python
#!/usr/bin/env python3
"""
detect_os.py

Purpose:
    Detect and print details about the current operating system:
    name, release, version, machine architecture, and Python version.

Usage:
    python detect_os.py

Expected Output:
    OS Name:        Linux
    OS Release:     5.15.0-91-generic
    OS Version:     #101-Ubuntu SMP ...
    Machine Arch:   x86_64
    Python Version: 3.11.4

Caution:
    - This script only reads system information; it makes no changes,
      so it is always safe to run.
    - Output format varies significantly between Linux, macOS, and
      Windows; don't assume the same fields exist identically on every
      platform if you build automation on top of this.
"""

import platform
import sys


def detect_os() -> None:
    print(f"OS Name:        {platform.system()}")
    print(f"OS Release:     {platform.release()}")
    print(f"OS Version:     {platform.version()}")
    print(f"Machine Arch:   {platform.machine()}")
    print(f"Python Version: {platform.python_version()}")


def main():
    detect_os()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

Everything here comes from the standard library's `platform` module, which reads OS-reported values directly — no external calls, no shelling out, nothing that can fail due to a missing binary.

```bash
python detect_os.py
```

#### Sample Output

```bash
$ python detect_os.py
OS Name:        Linux
OS Release:     5.15.0-91-generic
OS Version:     #101-Ubuntu SMP Fri Jan 1 00:00:00 UTC 2026
Machine Arch:   x86_64
Python Version: 3.11.4
```

#### Variations / Flags

- Add a `--json` flag to emit the same fields as structured JSON, so other tools in a pipeline can consume it without text-parsing.
- Add a simplified `platform.system() in ("Linux", "Darwin", "Windows")` helper function that other scripts in this chapter (like `manage_system_service.py`) can import directly instead of re-detecting the OS each time.
- Add a WSL-detection check (`"microsoft" in platform.release().lower()`) since WSL reports as Linux but has some Windows-specific quirks worth flagging separately.

#### Common Pitfalls

- Assuming `platform.system()` returns anything other than exactly `"Linux"`, `"Darwin"` (macOS), or `"Windows"` — code that string-matches loosely (e.g. checking for `"mac"`) will miss the actual value.
- Parsing `platform.version()` expecting a consistent format across OSes — it's a free-form string whose contents differ wildly between platforms and even between Linux distributions.
- Using this script's output to make security or compatibility decisions without also checking the specific tool/library versions you actually depend on — OS detection alone doesn't guarantee a given command or package is available.

### 2.19 Detect Current User

#### Use Case

Any script that needs to know which OS account it's actually running as — to log an audit trail, decide whether it has the permissions it needs, or warn before doing something as `root`/`Administrator` by accident.

#### Prerequisites

Standard library only (`getpass`, `os`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # no pip install needed
```

#### Cautions

This script only reads information; it makes no system changes. Behavior can differ slightly between OSes: on some systems the username reflects the login user, while in certain sandboxed or containerized environments it may show a generic/service account name instead of a real person's name.

#### Script

```python
#!/usr/bin/env python3
"""
detect_current_user.py

Purpose:
    Detect and print the username of the account currently running
    this script.

Usage:
    python detect_current_user.py

Expected Output:
    Current User: your_username

Caution:
    - This script only reads information; it makes no system changes.
    - Behavior can differ slightly between OSes: on some systems the
      username may reflect the login user, while in certain sandboxed
      or containerized environments it may show a generic/service
      account name instead of a real person's name.
"""

import getpass
import os


def detect_current_user() -> None:
    try:
        user = getpass.getuser()
    except Exception:
        user = os.environ.get("USER") or os.environ.get("USERNAME") or "unknown"
    print(f"Current User: {user}")


def main():
    detect_current_user()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`getpass.getuser()` checks a small set of environment variables (`LOGNAME`, `USER`, `LNAME`, `USERNAME` depending on platform) and falls back to system account databases if needed. If that raises for any reason, the script falls back to reading `USER`/`USERNAME` directly.

```bash
python detect_current_user.py
```

#### Sample Output

```bash
$ python detect_current_user.py
Current User: alex
```

```bash
$ sudo python detect_current_user.py
Current User: root
```

#### Variations / Flags

- Add a check that warns loudly if the detected user is `root`/`Administrator`, as a lightweight safety gate before destructive scripts in this chapter run.
- Combine with `get_hostname.py` to print a single `user@hostname` identity line for logging.
- On Linux, cross-check against `os.getuid()`/`pwd.getpwuid()` if you need the numeric UID as well as the username, since `getpass.getuser()` only returns the name.

#### Common Pitfalls

- Trusting this value for security decisions in isolation — `getpass.getuser()` reflects environment variables first, which can in some contexts be spoofed by whoever controls the calling environment; for real authorization checks, use OS-level UID/permission checks instead.
- Expecting a "real" human name inside containers — Docker and similar environments commonly report a generic or root-like account name that has nothing to do with the person who built or launched the image.
- Assuming the current user always matches who owns the files the script is about to touch — ownership and "current user" are related but separate concepts, especially after a `sudo` or `su` context switch.

A one-liner that shows up constantly in logging, alerting, and multi-server automation: tagging output with the machine's hostname so you know which box a log line, error, or metric actually came from.

### 2.21 Get System Uptime

#### Use Case

A quick health signal — "how long has this machine been running since its last boot" — that shows up constantly in status dashboards, incident timelines ("did this coincide with a reboot?"), and general server health checks.

#### Prerequisites

Requires the third-party `psutil` library. Works on Linux, macOS, and Windows.

```bash
pip install psutil
```

#### Cautions

This script only reads information; it makes no system changes. In virtual machines or containers, boot time may reflect when the VM/container itself started, not the underlying physical host.

#### Script

```python
#!/usr/bin/env python3
"""
get_system_uptime.py

Purpose:
    Calculate and print how long the system has been running since its
    last boot (uptime), in a human-readable format.

Usage:
    python get_system_uptime.py

Expected Output:
    System Boot Time: 2026-07-14 08:15:32
    Uptime: 2 day(s), 3 hour(s), 12 minute(s)

Caution:
    - Requires the third-party 'psutil' library (see requirements.txt).
    - This script only reads information; it makes no system changes.
    - In virtual machines or containers, boot time may reflect when the
      VM/container itself started, not the underlying physical host.
"""

import sys
import time
import datetime

try:
    import psutil
except ImportError:
    print("Error: this script requires 'psutil'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)


def get_uptime() -> None:
    boot_timestamp = psutil.boot_time()
    boot_time = datetime.datetime.fromtimestamp(boot_timestamp)
    now = datetime.datetime.now()
    uptime_seconds = (now - boot_time).total_seconds()

    days, remainder = divmod(uptime_seconds, 86400)
    hours, remainder = divmod(remainder, 3600)
    minutes, _ = divmod(remainder, 60)

    print(f"System Boot Time: {boot_time.strftime('%Y-%m-%d %H:%M:%S')}")
    print(f"Uptime: {int(days)} day(s), {int(hours)} hour(s), {int(minutes)} minute(s)")


def main():
    get_uptime()


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`psutil.boot_time()` returns the system boot time as a Unix timestamp; the script converts it to a `datetime`, subtracts it from "now," and breaks the resulting seconds down into days/hours/minutes for a human-readable summary.

```bash
python get_system_uptime.py
```

#### Sample Output

```bash
$ python get_system_uptime.py
System Boot Time: 2026-07-14 08:15:32
Uptime: 2 day(s), 3 hour(s), 12 minute(s)
```

#### Variations / Flags

- Add a `--seconds` flag that also prints raw total seconds, for feeding into monitoring systems that expect a numeric value rather than a formatted string.
- Add a `--threshold-days N` flag that exits with a non-zero code if uptime is below N days, useful as a "this machine rebooted recently — investigate why" alert in a monitoring pipeline.
- Combine with `get_hostname.py` in a small fleet-wide script that SSHes to each host and reports uptime side by side.

#### Common Pitfalls

- Interpreting container uptime as host uptime — inside Docker/Kubernetes, `psutil.boot_time()` typically reflects the container's own start time (or the underlying VM's, depending on setup), not the physical server's actual uptime.
- Comparing uptime across machines with different clock synchronization — if a host's system clock is wrong or drifting, the calculated uptime will be wrong even though the underlying boot detection is correct.
- Assuming a short uptime always means a crash — scheduled maintenance reboots, OS updates, and routine patching all reset this number too, so pair it with logs before drawing conclusions.

### 2.22 Run Commands with Timeout

#### Use Case

Task-oriented: "Run this command, but don't let it hang the whole pipeline if it takes too long." Any automation that calls out to a network service, an external tool, or anything else with unpredictable latency needs a hard ceiling on how long it's willing to wait.

#### Prerequisites

Standard library only (`subprocess`). Works on Linux, macOS, and Windows.

```bash
python3 --version   # no pip install needed
```

#### Cautions

When a command times out, this script terminates the underlying process for you — any partial work that command was doing will be lost, and any files it had open may not be closed/flushed cleanly. Choose a timeout value generously if the command normally takes a variable amount of time (e.g. network calls under load).

#### Script

```python
#!/usr/bin/env python3
"""
run_command_with_timeout.py

Purpose:
    Run a command but automatically stop waiting for it (and treat it
    as timed out) if it takes longer than a specified number of seconds.

Usage:
    python run_command_with_timeout.py --timeout 3 --command "sleep 10"
    python run_command_with_timeout.py --timeout 5 --command "echo quick"

    If no arguments are given, a safe built-in demo runs a command that
    intentionally times out after 2 seconds.

Expected Output:
    Success case:
        STDOUT: quick
        Exit Code: 0
    Timeout case:
        Error: Command timed out after 2 seconds and was terminated.

Caution:
    - When a command times out, this script terminates the underlying
      process for you; any partial work that command was doing will be
      lost, and any files it had open may not be closed/flushed
      cleanly.
    - Choose a timeout value generously if the command normally takes
      a variable amount of time (e.g. network calls under load).
"""

import subprocess
import sys


def run_with_timeout(command: str, timeout_seconds: float) -> None:
    try:
        result = subprocess.run(
            command,
            shell=True,
            capture_output=True,
            text=True,
            timeout=timeout_seconds
        )
        print("STDOUT:", result.stdout.strip() if result.stdout else "(no output)")
        if result.stderr.strip():
            print("STDERR:", result.stderr.strip())
        print(f"Exit Code: {result.returncode}")
    except subprocess.TimeoutExpired:
        print(f"Error: Command timed out after {timeout_seconds} seconds and was terminated.")


def parse_args():
    args = sys.argv[1:]
    timeout = None
    command = None
    i = 0
    while i < len(args):
        if args[i] == "--timeout" and i + 1 < len(args):
            timeout = float(args[i + 1]); i += 2
        elif args[i] == "--command" and i + 1 < len(args):
            command = args[i + 1]; i += 2
        else:
            i += 1
    return timeout, command


def main():
    timeout, command = parse_args()
    if timeout is None or command is None:
        timeout, command = 2, "sleep 10"
        print("No arguments given, running demo mode (expects a timeout).")
    run_with_timeout(command, timeout)


if __name__ == "__main__":
    main()
```

#### How It Works & Running

`subprocess.run(..., timeout=timeout_seconds)` raises `subprocess.TimeoutExpired` if the command hasn't finished within the given window; Python automatically kills the underlying process when that happens, and the script catches the exception to report a clean message instead of a traceback.

```bash
python run_command_with_timeout.py --timeout 3 --command "sleep 10"
python run_command_with_timeout.py --timeout 5 --command "echo quick"
python run_command_with_timeout.py                                    # 2s demo timeout
```

#### Sample Output

```bash
$ python run_command_with_timeout.py --timeout 3 --command "sleep 10"
Error: Command timed out after 3.0 seconds and was terminated.
```

```bash
$ python run_command_with_timeout.py --timeout 5 --command "echo quick"
STDOUT: quick
Exit Code: 0
```

#### Variations / Flags

- Add a `--retries N` flag that re-attempts the command a fixed number of times after a timeout, with an optional backoff delay between attempts.
- Add a `--kill-signal` option for POSIX systems where `SIGTERM` isn't enough and the process needs `SIGKILL` to actually stop.
- Combine with `get_exit_status.py`'s success/failure interpretation to fold "timed out" in as its own explicit third outcome, distinct from "ran and failed."

#### Common Pitfalls

- Setting the timeout too aggressively for commands with naturally variable latency (DNS lookups, first-run cache misses, cold-start API calls) — leading to false "timeout" failures that are really just normal slow runs.
- Assuming the process is fully cleaned up the instant `TimeoutExpired` is raised — on some platforms, especially when `shell=True` spawns a child shell process, killing the reported PID doesn't always guarantee every grandchild process it spawned is also terminated.
- Not accounting for partial output — a command that times out mid-write may have already written incomplete data to a file or database before being killed, which this script doesn't attempt to detect or roll back.
