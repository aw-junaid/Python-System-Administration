## Chapter 14: Network Administration

Network administration is the backbone of every reliable infrastructure. Whether you are troubleshooting a downed service at 2 a.m., auditing which ports are exposed to the internet, or automating nightly file transfers between servers, a sysadmin's real leverage comes from scripting these tasks instead of repeating them by hand. This chapter collects fourteen practical, script-driven techniques that a working system administrator, DevOps engineer, or security analyst reaches for on a near-daily basis.

Each topic in this chapter is intentionally self-contained and lives in its own markdown file, complete with a working script, installation instructions, sample output, and a list of common pitfalls collected from real-world use. The goal is not just to hand you a script to copy-paste, but to explain *why* it's needed, *when* to reach for it, and *what* tends to go wrong when it's used carelessly in a production environment.

The chapter is organized into four broad categories: **diagnostics** (ping, port scanning, availability checks, connectivity tests), **information gathering** (DNS resolution, WHOIS lookups, public/local IP discovery), **file transfer automation** (FTP, SCP, SFTP), **remote automation** (SSH), and **security hardening** (firewall configuration). Together they form a toolkit that covers the majority of day-to-day network operations work.

Every script in this chapter favors the Python standard library wherever possible, since it ships with almost every OS-level Python install and requires zero extra setup. Where a third-party library is genuinely the better tool for the job (`dnspython` for DNS, `python-whois` for WHOIS, `paramiko` for SSH/SFTP, `requests` for HTTP-based IP lookups), that dependency is called out explicitly in the "Prerequisites" section of the relevant file, along with the exact `pip install` command needed. OS-specific behavior (Linux vs. macOS vs. Windows) is flagged wherever a script's underlying command or library differs by platform — this matters most for ping, firewall rules, and local IP discovery.

### 14.1 Ping Hosts

#### Use Case

Quickly verify that one or more hosts are alive and reachable over the network, and measure round-trip latency. Sysadmins reach for this before escalating an outage ticket, during initial troubleshooting of "the server is down" reports, or as a pre-flight check before running deployment scripts.

#### Prerequisites

Uses only the Python standard library (`subprocess`, `platform`, `argparse`) — no third-party packages required. Relies on the system's native `ping` binary, which ships by default on Linux, macOS, and Windows.

```bash
# Verify ping is available (should already be present on all major OSes)
ping -c 1 127.0.0.1      # Linux/macOS
ping -n 1 127.0.0.1      # Windows (run in cmd/PowerShell)

# No pip install needed — Python 3.8+ is the only requirement
python3 --version
```

#### Cautions

Some corporate firewalls and cloud security groups block ICMP entirely, so a "failed" ping does not always mean the host is down — always cross-check with a port-based check (see 14.2/14.3) before declaring a host unreachable.

#### Script

```python
#!/usr/bin/env python3
"""ping_hosts.py - Ping one or more hosts and report reachability + latency."""

import argparse
import platform
import subprocess
import sys


def ping_host(host: str, count: int = 4, timeout: int = 2) -> dict:
    system = platform.system().lower()

    if system == "windows":
        cmd = ["ping", "-n", str(count), "-w", str(timeout * 1000), host]
    else:  # linux / darwin
        cmd = ["ping", "-c", str(count), "-W", str(timeout), host]

    try:
        result = subprocess.run(
            cmd, capture_output=True, text=True, timeout=timeout * count + 5
        )
        alive = result.returncode == 0
        return {"host": host, "alive": alive, "output": result.stdout}
    except subprocess.TimeoutExpired:
        return {"host": host, "alive": False, "output": "Request timed out"}
    except FileNotFoundError:
        return {"host": host, "alive": False, "output": "ping binary not found"}


def main():
    parser = argparse.ArgumentParser(description="Ping one or more hosts.")
    parser.add_argument("hosts", nargs="+", help="Hostnames or IP addresses to ping")
    parser.add_argument("-c", "--count", type=int, default=4, help="Number of echo requests")
    parser.add_argument("-t", "--timeout", type=int, default=2, help="Timeout per request (seconds)")
    args = parser.parse_args()

    exit_code = 0
    for host in args.hosts:
        result = ping_host(host, args.count, args.timeout)
        status = "UP" if result["alive"] else "DOWN"
        print(f"[{status}] {result['host']}")
        if not result["alive"]:
            exit_code = 1

    sys.exit(exit_code)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

The script wraps the OS-native `ping` command via `subprocess.run`, choosing the correct flag syntax for Windows (`-n`, `-w`) versus Linux/macOS (`-c`, `-W`). It returns a non-zero exit code if any host is unreachable, so it can be dropped straight into CI/CD or cron health checks.

```bash
# Ping a single host with default settings (4 packets, 2s timeout each)
python3 ping_hosts.py example.com

# Ping multiple hosts at once
python3 ping_hosts.py example.com 8.8.8.8 192.168.1.1

# Send only 1 packet with a 1-second timeout (fast check)
python3 ping_hosts.py -c 1 -t 1 example.com

# Use in a shell script and check the exit code
python3 ping_hosts.py example.com && echo "Host reachable" || echo "Host unreachable"
```

#### Sample Output

```bash
$ python3 ping_hosts.py example.com 10.255.255.1
[UP] example.com
[DOWN] 10.255.255.1
```

#### Variations / Flags

- Add `--json` to emit structured output for log ingestion (`json.dumps(result)`).
- Add `--quiet` to suppress per-host output and only print a final summary count.
- Schedule with cron for periodic health checks: `*/5 * * * * python3 ping_hosts.py example.com >> /var/log/ping_check.log`.

#### Common Pitfalls

- **Permissions**: on some Linux distros, raw ICMP sockets require elevated privileges if you bypass the system `ping` binary and use raw sockets instead — this script avoids that by shelling out to `ping`, which is already correctly permissioned (setuid) on most systems.
- **Missing binary**: minimal Docker base images (e.g., `python:slim`) often don't include `ping` — install `iputils-ping` (Debian/Ubuntu) first.
- **Platform differences**: flag syntax differs between Windows and POSIX systems; hardcoding one will silently break on the other OS.
- **False negatives**: ICMP being blocked by a firewall is not the same as the host being down — always confirm with a TCP-level check when possible.

### 14.2 Scan Open Ports

#### Use Case

Discover which TCP ports are open and listening on a target host — essential for auditing what services are exposed before a security review, verifying a firewall change took effect, or confirming a newly deployed service is actually listening on the expected port.

#### Prerequisites

Uses only the Python standard library (`socket`, `concurrent.futures`, `argparse`). No third-party packages required. Works identically on Linux, macOS, and Windows since it relies on raw TCP sockets rather than an external binary.

```bash
python3 --version   # Python 3.8+ required for concurrent.futures.ThreadPoolExecutor
```

#### Cautions

Only scan hosts you own or are explicitly authorized to test — port scanning third-party systems without permission can violate the Computer Fraud and Abuse Act (US), the Computer Misuse Act (UK), or equivalent laws elsewhere, and may trigger IDS/IPS alerts or automatic IP bans.

#### Script

```python
#!/usr/bin/env python3
"""scan_ports.py - Scan a range of TCP ports on a target host."""

import argparse
import socket
from concurrent.futures import ThreadPoolExecutor, as_completed


COMMON_PORTS = {
    21: "FTP", 22: "SSH", 23: "Telnet", 25: "SMTP", 53: "DNS",
    80: "HTTP", 110: "POP3", 143: "IMAP", 443: "HTTPS",
    3306: "MySQL", 3389: "RDP", 5432: "PostgreSQL", 6379: "Redis",
    8080: "HTTP-Alt", 27017: "MongoDB",
}


def scan_port(host: str, port: int, timeout: float) -> tuple:
    with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as sock:
        sock.settimeout(timeout)
        result = sock.connect_ex((host, port))
        return port, result == 0


def scan_ports(host: str, start: int, end: int, timeout: float, workers: int):
    open_ports = []
    with ThreadPoolExecutor(max_workers=workers) as executor:
        futures = {
            executor.submit(scan_port, host, port, timeout): port
            for port in range(start, end + 1)
        }
        for future in as_completed(futures):
            port, is_open = future.result()
            if is_open:
                open_ports.append(port)
    return sorted(open_ports)


def main():
    parser = argparse.ArgumentParser(description="Scan TCP ports on a host.")
    parser.add_argument("host", help="Target hostname or IP address")
    parser.add_argument("--start-port", type=int, default=1)
    parser.add_argument("--end-port", type=int, default=1024)
    parser.add_argument("--timeout", type=float, default=0.5, help="Socket timeout in seconds")
    parser.add_argument("--workers", type=int, default=200, help="Concurrent scan threads")
    args = parser.parse_args()

    print(f"Scanning {args.host} ports {args.start_port}-{args.end_port} ...")
    open_ports = scan_ports(args.host, args.start_port, args.end_port, args.timeout, args.workers)

    if not open_ports:
        print("No open ports found.")
        return

    for port in open_ports:
        service = COMMON_PORTS.get(port, "unknown")
        print(f"  {port}/tcp  open  {service}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

Each port in the requested range is checked with `socket.connect_ex`, which returns `0` on a successful TCP handshake. A `ThreadPoolExecutor` runs many checks concurrently so scanning 1,024 ports takes seconds rather than minutes.

```bash
# Scan the default well-known port range (1-1024)
python3 scan_ports.py example.com

# Scan a specific range
python3 scan_ports.py 192.168.1.10 --start-port 1 --end-port 65535

# Faster scan with a shorter timeout and more threads (use on trusted LANs)
python3 scan_ports.py 192.168.1.10 --timeout 0.2 --workers 500
```

#### Sample Output

```bash
$ python3 scan_ports.py 192.168.1.10 --start-port 1 --end-port 1024
Scanning 192.168.1.10 ports 1-1024 ...
  22/tcp  open  SSH
  80/tcp  open  HTTP
  443/tcp  open  HTTPS
```

#### Variations / Flags

- Add `--top-ports` to only scan the entries in `COMMON_PORTS` for a quick triage scan.
- Add `--output results.json` to save findings for later diffing between scans.
- Add `--udp` support using `socket.SOCK_DGRAM` for a slower but useful UDP sweep (note: UDP scanning is inherently less reliable since closed ports may not respond at all).

#### Common Pitfalls

- **Permissions**: no elevated privileges are needed for this TCP-connect style scan (unlike raw SYN scans used by tools like `nmap -sS`), so it works fine as a non-root user.
- **Timeout too aggressive**: a `--timeout` under 0.1s on a high-latency or congested network will produce false negatives (open ports reported as closed).
- **Too many threads**: setting `--workers` extremely high (e.g., 5000+) can exhaust local ephemeral ports or trigger rate-limiting/blacklisting on the target.
- **Platform differences**: on some Windows setups, very high thread counts combined with default socket limits can raise `OSError: [WinError 10055]` — reduce `--workers` if this occurs.

### 14.3 Check Host Availability

#### Use Case

Continuously monitor whether critical hosts (a database server, a load balancer, a VPN gateway) are up, using a combined ping + TCP port check so a single blocked ICMP path doesn't produce a false "down" alert. Useful for lightweight uptime monitoring without standing up a full monitoring stack like Nagios or Prometheus.

#### Prerequisites

Uses the Python standard library only (`socket`, `subprocess`, `time`, `argparse`). No third-party packages needed. Works on Linux, macOS, and Windows.

```bash
python3 --version   # Python 3.8+
```

#### Cautions

Running this in an infinite loop against production hosts adds continuous network load — set `--interval` sensibly (30–60s is typical) and avoid running many overlapping instances against the same host.

#### Script

```python
#!/usr/bin/env python3
"""check_availability.py - Combined ping + port availability monitor."""

import argparse
import platform
import socket
import subprocess
import sys
import time
from datetime import datetime


def ping_alive(host: str, timeout: int = 2) -> bool:
    system = platform.system().lower()
    cmd = (
        ["ping", "-n", "1", "-w", str(timeout * 1000), host]
        if system == "windows"
        else ["ping", "-c", "1", "-W", str(timeout), host]
    )
    try:
        result = subprocess.run(cmd, capture_output=True, timeout=timeout + 3)
        return result.returncode == 0
    except (subprocess.TimeoutExpired, FileNotFoundError):
        return False


def port_alive(host: str, port: int, timeout: float = 2) -> bool:
    try:
        with socket.create_connection((host, port), timeout=timeout):
            return True
    except OSError:
        return False


def check_once(host: str, port: int) -> dict:
    icmp_ok = ping_alive(host)
    tcp_ok = port_alive(host, port) if port else None
    available = icmp_ok or (tcp_ok is True)
    return {
        "timestamp": datetime.now().isoformat(timespec="seconds"),
        "host": host,
        "icmp": icmp_ok,
        "tcp_port": port,
        "tcp_ok": tcp_ok,
        "available": available,
    }


def main():
    parser = argparse.ArgumentParser(description="Monitor host availability.")
    parser.add_argument("host", help="Hostname or IP to monitor")
    parser.add_argument("--port", type=int, default=443, help="TCP port to also check")
    parser.add_argument("--interval", type=int, default=30, help="Seconds between checks")
    parser.add_argument("--retries", type=int, default=3, help="Consecutive failures before alert")
    parser.add_argument("--once", action="store_true", help="Run a single check and exit")
    args = parser.parse_args()

    fail_streak = 0
    while True:
        result = check_once(args.host, args.port)
        status = "AVAILABLE" if result["available"] else "UNAVAILABLE"
        print(f"{result['timestamp']}  {args.host}  icmp={result['icmp']} "
              f"tcp:{args.port}={result['tcp_ok']}  -> {status}")

        if not result["available"]:
            fail_streak += 1
            if fail_streak >= args.retries:
                print(f"ALERT: {args.host} has failed {fail_streak} consecutive checks", file=sys.stderr)
        else:
            fail_streak = 0

        if args.once:
            sys.exit(0 if result["available"] else 1)

        time.sleep(args.interval)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

Each cycle runs an ICMP ping and a TCP connect attempt on a chosen port; a host is considered "available" if either succeeds. This handles environments where ICMP is blocked but the service port is open (or vice versa). Consecutive failures beyond `--retries` print an alert line to stderr, which you can pipe into a paging system.

```bash
# Run a single check and exit (useful for cron or CI gating)
python3 check_availability.py example.com --once

# Continuously monitor every 30 seconds, checking port 443
python3 check_availability.py example.com --port 443 --interval 30

# Monitor an internal database host on port 5432, alert after 5 failures
python3 check_availability.py db.internal.local --port 5432 --retries 5
```

#### Sample Output

```bash
$ python3 check_availability.py example.com --once
2026-07-18T10:15:02  example.com  icmp=True tcp:443=True  -> AVAILABLE
```

```bash
$ python3 check_availability.py db.internal.local --port 5432 --interval 10
2026-07-18T10:20:00  db.internal.local  icmp=False tcp:5432=False  -> UNAVAILABLE
2026-07-18T10:20:10  db.internal.local  icmp=False tcp:5432=False  -> UNAVAILABLE
2026-07-18T10:20:20  db.internal.local  icmp=False tcp:5432=False  -> UNAVAILABLE
ALERT: db.internal.local has failed 3 consecutive checks
```

#### Variations / Flags

- Add `--webhook <url>` to POST the alert JSON to Slack/PagerDuty/Teams instead of printing to stderr.
- Add `--log-file availability.log` to persist results for later trend analysis.
- Run under `systemd` as a long-lived service instead of cron, since this script itself loops on `--interval`.

#### Common Pitfalls

- **Running forever without `--once`**: forgetting this flag inside a cron job stacks up zombie processes — use `--once` for cron, and the built-in loop only for a dedicated long-running process/service.
- **Missing `ping` binary**: same caveat as 14.1 — minimal containers need `iputils-ping` installed.
- **Port choice matters**: checking a port that's intentionally closed (e.g., a firewalled admin port) will produce a false "unavailable" even though the host and its real services are fine — choose a port that's meant to be reachable.
- **Clock drift in logs**: if correlating this output across multiple machines, ensure NTP is synced or timestamps will be misleading.

### 14.4 Resolve DNS Records

#### Use Case

Look up A, AAAA, MX, TXT, NS, or CNAME records for a domain — used when troubleshooting email delivery (MX/TXT/SPF), verifying a DNS change has propagated, or confirming a domain points to the expected IP before a cutover.

#### Prerequisites

Requires the third-party `dnspython` package, since the standard library's `socket.gethostbyname` only resolves A records and cannot query MX/TXT/NS/CNAME directly.

```bash
pip install dnspython --break-system-packages   # Linux/macOS with externally-managed Python
pip install dnspython                           # Windows / virtualenv
```

Works on Linux, macOS, and Windows — `dnspython` is pure-Python and has no OS-specific dependencies.

#### Cautions

Results can be cached at multiple layers (OS resolver, local router, ISP resolver) — always cross-check against a public resolver like `8.8.8.8` if you suspect stale results after a recent DNS change.

#### Script

```python
#!/usr/bin/env python3
"""resolve_dns.py - Query DNS records for a domain."""

import argparse
import dns.resolver


RECORD_TYPES = ["A", "AAAA", "MX", "TXT", "NS", "CNAME"]


def resolve(domain: str, record_type: str, nameserver: str = None):
    resolver = dns.resolver.Resolver()
    if nameserver:
        resolver.nameservers = [nameserver]

    try:
        answers = resolver.resolve(domain, record_type)
        return [rdata.to_text() for rdata in answers]
    except dns.resolver.NoAnswer:
        return []
    except dns.resolver.NXDOMAIN:
        return None
    except dns.exception.DNSException as exc:
        return [f"ERROR: {exc}"]


def main():
    parser = argparse.ArgumentParser(description="Resolve DNS records for a domain.")
    parser.add_argument("domain", help="Domain name to query")
    parser.add_argument("--type", dest="record_type", default="A",
                         choices=RECORD_TYPES + ["ALL"], help="DNS record type")
    parser.add_argument("--nameserver", help="Use a specific resolver, e.g. 8.8.8.8")
    args = parser.parse_args()

    types_to_query = RECORD_TYPES if args.record_type == "ALL" else [args.record_type]

    for record_type in types_to_query:
        records = resolve(args.domain, record_type, args.nameserver)
        if records is None:
            print(f"{record_type}: domain does not exist (NXDOMAIN)")
            break
        if not records:
            print(f"{record_type}: no records found")
            continue
        for r in records:
            print(f"{record_type}: {r}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`dns.resolver.Resolver()` builds a resolver object; by default it uses the system's configured DNS servers, but `--nameserver` overrides that to query a specific resolver directly (bypassing local cache).

```bash
# Resolve the A record (default)
python3 resolve_dns.py example.com

# Resolve MX records (mail servers)
python3 resolve_dns.py example.com --type MX

# Query all common record types at once
python3 resolve_dns.py example.com --type ALL

# Bypass local cache and query Google's public resolver directly
python3 resolve_dns.py example.com --type A --nameserver 8.8.8.8
```

#### Sample Output

```bash
$ python3 resolve_dns.py example.com --type ALL
A: 93.184.216.34
AAAA: 2606:2800:220:1:248:1893:25c8:1946
MX: no records found
TXT: "v=spf1 -all"
NS: a.iana-servers.net.
NS: b.iana-servers.net.
CNAME: no records found
```

#### Variations / Flags

- Add `--json` to emit machine-readable output for feeding into monitoring dashboards.
- Add `--watch <seconds>` combined with a loop to detect DNS propagation in near-real time after a record change.
- For reverse DNS (IP → hostname), use `dns.reversename.from_address()` and query `PTR` instead.

#### Common Pitfalls

- **`ModuleNotFoundError: No module named 'dns'`**: this means `dnspython` isn't installed — note the package name (`dnspython`) differs from the import name (`dns`), which trips people up.
- **Stale results**: local OS or router DNS caching can return outdated records for minutes to hours after a change — always test with `--nameserver 8.8.8.8` to rule this out.
- **NXDOMAIN vs NoAnswer confusion**: NXDOMAIN means the domain doesn't exist at all; NoAnswer means the domain exists but has no records of the requested type — the script distinguishes these explicitly.
- **CNAME + other record conflicts**: a domain with a CNAME record generally cannot also have A/MX/TXT records at that same name per DNS spec — an empty result for other types alongside a CNAME is expected, not a bug.

### 14.5 Perform WHOIS Lookups

#### Use Case

Retrieve domain registration metadata — registrar, creation date, expiry date, and name servers — to confirm domain ownership, check when a domain is about to expire (to avoid an accidental lapse), or investigate a suspicious domain during a security incident.

#### Prerequisites

Requires the third-party `python-whois` package, which wraps the WHOIS protocol and handles parsing of the varied response formats returned by different registrars.

```bash
pip install python-whois --break-system-packages   # Linux/macOS externally-managed Python
pip install python-whois                           # Windows / virtualenv
```

Cross-platform (Linux/macOS/Windows) — `python-whois` communicates over TCP port 43 directly and does not depend on a system `whois` binary.

#### Cautions

WHOIS servers commonly rate-limit or temporarily block IPs that send many rapid queries — space out bulk lookups (e.g., one domain per second) and expect some registries (especially `.eu`, some ccTLDs) to redact registrant data under privacy regulations like GDPR.

#### Script

```python
#!/usr/bin/env python3
"""whois_lookup.py - Perform a WHOIS lookup for a domain."""

import argparse
import whois


def lookup(domain: str) -> dict:
    try:
        data = whois.whois(domain)
        return {
            "domain_name": data.domain_name,
            "registrar": data.registrar,
            "creation_date": data.creation_date,
            "expiration_date": data.expiration_date,
            "name_servers": data.name_servers,
            "status": data.status,
        }
    except Exception as exc:
        return {"error": str(exc)}


def main():
    parser = argparse.ArgumentParser(description="WHOIS lookup for a domain.")
    parser.add_argument("domain", help="Domain name to look up, e.g. example.com")
    args = parser.parse_args()

    result = lookup(args.domain)

    if "error" in result:
        print(f"WHOIS lookup failed: {result['error']}")
        return

    print(f"Domain:          {result['domain_name']}")
    print(f"Registrar:       {result['registrar']}")
    print(f"Created:         {result['creation_date']}")
    print(f"Expires:         {result['expiration_date']}")
    print(f"Status:          {result['status']}")
    print("Name Servers:")
    ns_list = result["name_servers"] or []
    if isinstance(ns_list, str):
        ns_list = [ns_list]
    for ns in ns_list:
        print(f"  - {ns}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`whois.whois(domain)` sends a raw WHOIS query to the appropriate registry server (it auto-detects which server to query based on the TLD) and parses the free-text response into structured fields.

```bash
# Basic lookup
python3 whois_lookup.py example.com

# Lookup a domain with a different TLD
python3 whois_lookup.py example.org
```

#### Sample Output

```bash
$ python3 whois_lookup.py example.com
Domain:          EXAMPLE.COM
Registrar:       RESERVED-Internet Assigned Numbers Authority
Created:         1995-08-14 04:00:00
Expires:         2026-08-13 04:00:00
Status:          ['clientDeleteProhibited', 'clientTransferProhibited']
Name Servers:
  - A.IANA-SERVERS.NET
  - B.IANA-SERVERS.NET
```

#### Variations / Flags

- Add `--expiring-in-days 30` combined with a loop over a domain list to build an automated "domains about to expire" report.
- Add `--raw` to print `data.text` (the unparsed WHOIS server response) when the parsed fields come back incomplete.
- Wrap in a `try/except` retry loop with exponential backoff for bulk lookups to gracefully handle rate-limiting.

#### Common Pitfalls

- **Rate limiting**: bulk-querying many domains back-to-back frequently triggers temporary blocks from registry WHOIS servers — add a `time.sleep(1)` between lookups.
- **Redacted data**: GDPR-covered registrars (many `.eu`, `.de`, and increasingly `.com` registrants using privacy proxies) will return `"REDACTED FOR PRIVACY"` instead of real registrant info — this is expected, not a script bug.
- **Inconsistent field names across TLDs**: some ccTLD registries return non-standard fields that `python-whois` cannot fully parse — fall back to `--raw` output in those cases.
- **Firewall blocking port 43**: corporate networks sometimes block outbound TCP/43 (the WHOIS protocol port) — if lookups hang or time out, test from a different network before assuming the script is broken.

### 14.6 Retrieve Public IP

#### Use Case

Determine the externally visible IP address of the current machine as seen by the internet — needed when whitelisting a server's egress IP with a third-party API, confirming a VPN/proxy is actually routing traffic, or diagnosing why a service reports connections from an unexpected address.

#### Prerequisites

Requires the third-party `requests` package (the standard library's `urllib.request` also works, but `requests` is simpler and included here for readability).

```bash
pip install requests --break-system-packages   # Linux/macOS externally-managed Python
pip install requests                           # Windows / virtualenv
```

Cross-platform (Linux/macOS/Windows) — this is a plain HTTPS request, no OS-specific behavior.

#### Cautions

This method depends on a third-party service being available; always have a fallback provider configured, and never assume the returned IP is static — many home/cloud connections use dynamic IPs that change over time (e.g., on DHCP lease renewal).

#### Script

```python
#!/usr/bin/env python3
"""public_ip.py - Retrieve this machine's public IP address."""

import argparse
import requests

PROVIDERS = [
    "https://api.ipify.org",
    "https://ifconfig.me/ip",
    "https://checkip.amazonaws.com",
]


def get_public_ip(timeout: float = 5) -> str:
    last_error = None
    for url in PROVIDERS:
        try:
            response = requests.get(url, timeout=timeout)
            response.raise_for_status()
            return response.text.strip()
        except requests.RequestException as exc:
            last_error = exc
            continue
    raise RuntimeError(f"All providers failed. Last error: {last_error}")


def main():
    parser = argparse.ArgumentParser(description="Retrieve the machine's public IP.")
    parser.add_argument("--timeout", type=float, default=5, help="Request timeout in seconds")
    args = parser.parse_args()

    try:
        ip = get_public_ip(args.timeout)
        print(f"Public IP: {ip}")
    except RuntimeError as exc:
        print(f"Error: {exc}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

The script queries a small list of public "what's my IP" services in order and returns the first successful response, providing basic fallback resilience if one provider is down or rate-limiting.

```bash

python3 public_ip.py

# Increase timeout on a slow/high-latency connection
python3 public_ip.py --timeout 10
```

#### Sample Output

```bash
$ python3 public_ip.py

```

#### Variations / Flags

- Add `--ipv6` by pointing at an IPv6-only endpoint (e.g., `https://api6.ipify.org`) to confirm IPv6 connectivity specifically.
- Add `--watch 60` in a loop to detect and log when a dynamic IP changes over time.
- Add `--geo` to also call an IP-geolocation API and print the approximate city/country for the returned IP.

#### Common Pitfalls

- **No internet access**: this script requires outbound internet connectivity — it will not work from an air-gapped or fully firewalled internal network by design.
- **Provider downtime**: any single "what's my IP" service can go down or rate-limit — this is why the script tries multiple providers in sequence rather than relying on just one.
- **Confusing public IP with local IP**: the public IP is the address seen by external servers (often your router's WAN address behind NAT), not the machine's own network interface address — see 14.7 for local IP retrieval.
- **Corporate proxy/VPN interference**: if the machine routes through a corporate proxy or VPN, the "public IP" returned will be the proxy/VPN exit node's IP, not the machine's own ISP-assigned address — this is expected behavior, not a bug.


### 14.7 Retrieve Local IP

#### Use Case

Determine the machine's internal/private IP address on the local network — needed when configuring a service to bind to a specific interface, setting up port forwarding on a home router, or generating a config file that needs to reference "this machine's address" on a LAN.

#### Prerequisites

Uses only the Python standard library (`socket`). No third-party packages required, no internet access required — this works entirely offline since it queries local network interface information.

```bash
python3 --version   # Python 3.8+
```

#### Cautions

A machine can have multiple local IPs (Ethernet, Wi-Fi, VPN tunnel, Docker bridge) — the technique below returns the IP associated with the interface that would be used to reach the given target, which is usually what you want, but it is not necessarily the "primary" interface in every multi-homed setup.

#### Script

```python
#!/usr/bin/env python3
"""local_ip.py - Retrieve this machine's local (LAN) IP address."""

import argparse
import socket


def get_local_ip(probe_target: str = "8.8.8.8", probe_port: int = 80) -> str:
    """Uses a UDP 'connect' (no packets actually sent) to determine
    which local interface/IP would be used to reach probe_target."""
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    try:
        sock.connect((probe_target, probe_port))
        return sock.getsockname()[0]
    except OSError:
        return "127.0.0.1"
    finally:
        sock.close()


def list_all_interfaces() -> list:
    hostname = socket.gethostname()
    try:
        _, _, ip_list = socket.gethostbyname_ex(hostname)
        return ip_list
    except socket.gaierror:
        return []


def main():
    parser = argparse.ArgumentParser(description="Retrieve local network IP address(es).")
    parser.add_argument("--all", action="store_true", help="List all detected local IPs")
    args = parser.parse_args()

    if args.all:
        ips = list_all_interfaces()
        print("Local IPs found via hostname resolution:")
        for ip in ips:
            print(f"  - {ip}")
        print(f"Primary outbound-route IP: {get_local_ip()}")
    else:
        print(f"Local IP: {get_local_ip()}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`get_local_ip()` opens a UDP socket and calls `connect()` toward a public IP (`8.8.8.8`) — no actual packet is sent for a UDP connect, but the OS still selects and binds a local interface/IP as if it would send one, which `getsockname()` then reveals. This is a well-known, reliable trick that avoids parsing `ifconfig`/`ipconfig` output.

```bash
# Get the primary local IP (the one used for outbound traffic)
python3 local_ip.py

# List all IPs bound to any local interface, plus the primary route IP
python3 local_ip.py --all
```

#### Sample Output

```bash
$ python3 local_ip.py
Local IP: 192.168.1.42
```

```bash
$ python3 local_ip.py --all
Local IPs found via hostname resolution:
  - 192.168.1.42
  - 172.17.0.1
Primary outbound-route IP: 192.168.1.42
```

#### Variations / Flags

- Add `--interface eth0` (Linux) and parse `/sys/class/net/eth0/` or use the `psutil` package for a more explicit, interface-name-based lookup instead of the route-guessing trick.
- Add `--ipv6` and use `socket.AF_INET6` with a probe target like `2001:4860:4860::8888` to get the local IPv6 address.
- On Windows, `--all` can be supplemented with `ipconfig` output parsing for adapter-friendly names.

#### Common Pitfalls

- **Multiple interfaces confusion**: on a machine with Docker, VPN clients, or multiple NICs, `--all` may list several unrelated IPs (e.g., a `172.17.0.1` Docker bridge address) — use the "primary outbound-route IP" line as the most reliable single answer.
- **No network connectivity**: if there is no route to the internet at all (fully isolated LAN), the UDP-connect trick still generally works as long as *some* default route exists; a machine with literally no network interfaces configured will fall back to `127.0.0.1`.
- **VPN interference**: when a VPN is active, the "primary" local IP returned may be the VPN tunnel's virtual interface rather than the physical Ethernet/Wi-Fi adapter — this is often exactly what's needed, but can surprise users expecting their LAN IP.
- **Not the same as public IP**: this script intentionally never leaves the local machine's routing table — it does not tell you anything about your public-facing address (see 14.6 for that).


### 14.8 Test Network Connectivity

#### Use Case

Run a single combined health check — ping, DNS resolution, and TCP port reachability — against a list of critical endpoints (e.g., internal DB, external payment gateway, DNS server) to quickly answer "is our network okay right now?" during an incident, or as a pre-deployment sanity check in a CI pipeline.

#### Prerequisites

Uses the Python standard library only (`socket`, `subprocess`, `platform`, `argparse`, `json`). No third-party packages required. Optionally reads a JSON config file listing endpoints to check.

```bash
python3 --version   # Python 3.8+
```

#### Cautions

Treat this as a triage tool, not a full monitoring replacement — for long-term uptime tracking and alerting, use a dedicated system (Prometheus/Grafana, Nagios, Datadog); this script is meant for quick, on-demand checks.

#### Script

```python
#!/usr/bin/env python3
"""test_connectivity.py - Combined ping/DNS/port connectivity check against multiple endpoints."""

import argparse
import json
import platform
import socket
import subprocess
import sys


def ping_alive(host: str, timeout: int = 2) -> bool:
    system = platform.system().lower()
    cmd = (
        ["ping", "-n", "1", "-w", str(timeout * 1000), host]
        if system == "windows"
        else ["ping", "-c", "1", "-W", str(timeout), host]
    )
    try:
        result = subprocess.run(cmd, capture_output=True, timeout=timeout + 3)
        return result.returncode == 0
    except (subprocess.TimeoutExpired, FileNotFoundError):
        return False


def dns_ok(host: str) -> bool:
    try:
        socket.gethostbyname(host)
        return True
    except socket.gaierror:
        return False


def port_ok(host: str, port: int, timeout: float = 2) -> bool:
    try:
        with socket.create_connection((host, port), timeout=timeout):
            return True
    except OSError:
        return False


def check_endpoint(name: str, host: str, port: int) -> dict:
    return {
        "name": name,
        "host": host,
        "port": port,
        "dns": dns_ok(host),
        "icmp": ping_alive(host),
        "tcp": port_ok(host, port) if port else None,
    }


DEFAULT_ENDPOINTS = [
    {"name": "Google DNS", "host": "8.8.8.8", "port": 53},
    {"name": "Public Internet (HTTPS)", "host": "example.com", "port": 443},
]


def main():
    parser = argparse.ArgumentParser(description="Test connectivity to multiple endpoints.")
    parser.add_argument("--config", help="Path to a JSON file with a list of endpoints")
    parser.add_argument("--json", action="store_true", help="Output raw JSON instead of a table")
    args = parser.parse_args()

    if args.config:
        with open(args.config) as f:
            endpoints = json.load(f)
    else:
        endpoints = DEFAULT_ENDPOINTS

    results = [check_endpoint(e["name"], e["host"], e.get("port", 0)) for e in endpoints]

    if args.json:
        print(json.dumps(results, indent=2))
        return

    all_ok = True
    for r in results:
        overall = r["dns"] and (r["icmp"] or r["tcp"])
        all_ok = all_ok and overall
        status = "OK" if overall else "FAIL"
        print(f"[{status}] {r['name']:<28} dns={r['dns']} icmp={r['icmp']} tcp={r['tcp']}")

    sys.exit(0 if all_ok else 1)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

For each endpoint the script checks DNS resolution, ICMP reachability, and (if a port is specified) TCP reachability, then combines these into a single pass/fail verdict per endpoint. A JSON config file lets you define a custom endpoint list for your specific environment.

```bash
# Run against the built-in default endpoints
python3 test_connectivity.py

# Run against a custom list of endpoints defined in a JSON file
python3 test_connectivity.py --config endpoints.json

# Get machine-readable JSON output for piping into another tool
python3 test_connectivity.py --config endpoints.json --json
```

Example `endpoints.json`:

```json
[
  {"name": "Internal DB", "host": "db.internal.local", "port": 5432},
  {"name": "Payment Gateway", "host": "api.stripe.com", "port": 443},
  {"name": "Internal DNS", "host": "10.0.0.2", "port": 53}
]
```

#### Sample Output

```bash
$ python3 test_connectivity.py --config endpoints.json
[OK]   Internal DB                  dns=True icmp=False tcp=True
[OK]   Payment Gateway              dns=True icmp=True tcp=True
[FAIL] Internal DNS                 dns=False icmp=False tcp=False
```

#### Variations / Flags

- Add `--parallel` using `concurrent.futures.ThreadPoolExecutor` to check all endpoints simultaneously instead of sequentially, speeding up large endpoint lists.
- Add `--interval 60` combined with a loop for continuous connectivity monitoring during an active incident.
- Pipe `--json` output into `jq` for quick command-line filtering: `python3 test_connectivity.py --json | jq '.[] | select(.tcp == false)'`.

#### Common Pitfalls

- **Config file typos**: a malformed JSON endpoints file will raise a `json.JSONDecodeError` — validate with `python3 -m json.tool endpoints.json` before running.
- **Port 0 / missing port**: if `port` is omitted or `0`, TCP checking is skipped entirely (`tcp` will show `None`) — this is intentional for ICMP-only targets, but easy to misread as a failure.
- **ICMP blocked but service actually fine**: a `FAIL` on `icmp` alongside `tcp=True` still counts as overall `OK` in this script's logic — read the per-field results, not just the pass/fail label, when diagnosing.
- **Running from a restricted network**: results will reflect the network the script is run from — running this from your laptop over Wi-Fi tells you nothing about connectivity from the actual production server experiencing issues.

### 14.9 Download Files via FTP

#### Use Case

Automate pulling files from a remote FTP server — common in legacy systems (older vendor integrations, some ISP/hosting file exchanges, EDI feeds) where SFTP isn't available and plain FTP is still the agreed transfer method.

#### Prerequisites

Uses only the Python standard library (`ftplib`). No third-party packages required.

```bash
python3 --version   # Python 3.8+
```

#### Cautions

Plain FTP transmits credentials and data unencrypted — never use it over an untrusted network or the public internet for sensitive data; prefer FTPS (`ftplib.FTP_TLS`) or SFTP (see 14.13) whenever the remote server supports it.

#### Script

```python
#!/usr/bin/env python3
"""ftp_download.py - Download a file from a remote FTP server."""

import argparse
import ftplib
import os
import sys


def download_file(host: str, user: str, password: str, remote_path: str,
                   local_path: str, port: int = 21, use_tls: bool = False) -> None:
    ftp_cls = ftplib.FTP_TLS if use_tls else ftplib.FTP
    with ftp_cls() as ftp:
        ftp.connect(host, port, timeout=15)
        ftp.login(user, password)
        if use_tls:
            ftp.prot_p()  # secure the data channel too

        with open(local_path, "wb") as f:
            ftp.retrbinary(f"RETR {remote_path}", f.write)


def main():
    parser = argparse.ArgumentParser(description="Download a file via FTP.")
    parser.add_argument("host", help="FTP server hostname or IP")
    parser.add_argument("remote_path", help="Path to the file on the FTP server")
    parser.add_argument("local_path", help="Where to save the file locally")
    parser.add_argument("--user", default="anonymous")
    parser.add_argument("--password", default=os.environ.get("FTP_PASSWORD", ""))
    parser.add_argument("--port", type=int, default=21)
    parser.add_argument("--tls", action="store_true", help="Use FTPS (FTP over TLS)")
    args = parser.parse_args()

    try:
        download_file(args.host, args.user, args.password, args.remote_path,
                       args.local_path, args.port, args.tls)
        print(f"Downloaded '{args.remote_path}' -> '{args.local_path}'")
    except ftplib.all_errors as exc:
        print(f"FTP download failed: {exc}", file=sys.stderr)
        sys.exit(1)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`ftplib.FTP` (or `FTP_TLS` for encrypted transfers) connects and authenticates, then `retrbinary` streams the remote file in binary chunks directly into a local file handle — this avoids loading the entire file into memory at once, which matters for large files.

```bash
# Anonymous download (no credentials needed)
python3 ftp_download.py ftp.example.com /pub/readme.txt ./readme.txt

# Authenticated download, reading the password from an environment variable
export FTP_PASSWORD='s3cret'
python3 ftp_download.py ftp.example.com /reports/daily.csv ./daily.csv --user reportuser

# Secure download using FTPS
python3 ftp_download.py ftp.example.com /data/export.zip ./export.zip --user svc_acct --tls
```

#### Sample Output

```bash
$ python3 ftp_download.py ftp.example.com /pub/readme.txt ./readme.txt
Downloaded '/pub/readme.txt' -> './readme.txt'
```

#### Variations / Flags

- Add `--resume` to check the local partial file size and use `REST` (restart) to resume an interrupted download.
- Wrap in a loop with `ftp.nlst(remote_dir)` to bulk-download every file in a remote directory.
- Add `--verify-checksum` to compare an MD5/SHA256 of the downloaded file against a manifest, if the server provides one.

#### Common Pitfalls

- **Passing credentials on the command line**: `--password` visible in shell history or `ps aux` output is a security risk — prefer the `FTP_PASSWORD` environment variable as shown above, or prompt interactively with `getpass`.
- **Passive vs active mode**: some servers/firewalls require passive mode (`ftp.set_pasv(True)`, which is the `ftplib` default) — if downloads hang indefinitely, explicitly toggle passive mode off/on to match the server's requirements.
- **Binary vs text mode**: always use `retrbinary` (not `retrlines`) for anything other than plain text files, or binary files (images, zips, PDFs) will get corrupted.
- **Unencrypted by default**: forgetting `--tls` on a server that supports FTPS means credentials and file contents travel in plaintext — confirm with your server admin whether TLS is available and required.

### 14.10 Upload Files via FTP

#### Use Case

Automate pushing files to a remote FTP server — common for delivering nightly report exports to a partner's legacy file-drop system, publishing a website update to a shared-hosting FTP account, or feeding a vendor's EDI pickup folder.

#### Prerequisites

Uses only the Python standard library (`ftplib`). No third-party packages required.

```bash
python3 --version   # Python 3.8+
```

#### Cautions

As with downloads, plain FTP uploads send credentials and file contents unencrypted — use FTPS (`--tls`) whenever the destination server supports it, and never hardcode passwords directly into scripts that get committed to version control.

#### Script

```python
#!/usr/bin/env python3
"""ftp_upload.py - Upload a file to a remote FTP server."""

import argparse
import ftplib
import os
import sys


def upload_file(host: str, user: str, password: str, local_path: str,
                 remote_path: str, port: int = 21, use_tls: bool = False) -> None:
    ftp_cls = ftplib.FTP_TLS if use_tls else ftplib.FTP
    with ftp_cls() as ftp:
        ftp.connect(host, port, timeout=15)
        ftp.login(user, password)
        if use_tls:
            ftp.prot_p()

        with open(local_path, "rb") as f:
            ftp.storbinary(f"STOR {remote_path}", f)


def main():
    parser = argparse.ArgumentParser(description="Upload a file via FTP.")
    parser.add_argument("host", help="FTP server hostname or IP")
    parser.add_argument("local_path", help="Path to the local file to upload")
    parser.add_argument("remote_path", help="Destination path on the FTP server")
    parser.add_argument("--user", default="anonymous")
    parser.add_argument("--password", default=os.environ.get("FTP_PASSWORD", ""))
    parser.add_argument("--port", type=int, default=21)
    parser.add_argument("--tls", action="store_true", help="Use FTPS (FTP over TLS)")
    args = parser.parse_args()

    if not os.path.isfile(args.local_path):
        print(f"Error: local file not found: {args.local_path}", file=sys.stderr)
        sys.exit(1)

    try:
        upload_file(args.host, args.user, args.password, args.local_path,
                     args.remote_path, args.port, args.tls)
        print(f"Uploaded '{args.local_path}' -> '{args.remote_path}'")
    except ftplib.all_errors as exc:
        print(f"FTP upload failed: {exc}", file=sys.stderr)
        sys.exit(1)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`ftplib.FTP.storbinary` streams a local file in binary chunks to the remote server using the `STOR` FTP command, writing directly from the open file handle so large files don't need to be fully buffered in memory first.

```bash
# Anonymous upload
python3 ftp_upload.py ftp.example.com ./report.csv /incoming/report.csv

# Authenticated upload with password from environment variable
export FTP_PASSWORD='s3cret'
python3 ftp_upload.py ftp.example.com ./export.zip /uploads/export.zip --user svc_acct

# Secure upload using FTPS
python3 ftp_upload.py ftp.example.com ./data.csv /secure/data.csv --user svc_acct --tls
```

#### Sample Output

```bash
$ python3 ftp_upload.py ftp.example.com ./report.csv /incoming/report.csv --user reportuser
Uploaded './report.csv' -> '/incoming/report.csv'
```

#### Variations / Flags

- Add `--mkdir` to create the remote destination directory first via `ftp.mkd()` if it doesn't already exist.
- Loop over `glob.glob("*.csv")` to bulk-upload every matching file in a local directory.
- Add `--verify` to re-download the file after upload and compare checksums, confirming a clean transfer.

#### Common Pitfalls

- **Uploading to a non-existent remote directory**: `ftplib` will raise `550 No such file or directory` if the destination folder doesn't exist — create it first with `ftp.mkd()` or confirm the path with your server admin.
- **File permission errors on the remote side**: some FTP accounts are restricted to specific subdirectories (`chroot` jails) — uploading outside that scope fails even with valid credentials.
- **Silent corruption from text-mode uploads**: always use `storbinary` (not `storlines`) for anything besides plain ASCII text, or binary files will be corrupted on arrival.
- **Large file timeouts**: very large uploads over a slow link can exceed the default socket timeout — increase the `timeout` value passed to `ftp.connect()` for multi-gigabyte transfers.

### 14.11 SSH Automation

#### Use Case

Run a shell command on one or many remote servers without manually opening an SSH session each time — used for fleet-wide status checks ("is disk usage okay on all 40 web nodes?"), rolling restarts, or pulling a log file from a specific host during an incident.

#### Prerequisites

Requires the third-party `paramiko` package, a pure-Python SSHv2 implementation.

```bash
pip install paramiko --break-system-packages   # Linux/macOS externally-managed Python
pip install paramiko                           # Windows / virtualenv
```

Cross-platform (Linux/macOS/Windows) — `paramiko` doesn't depend on the OS's native `ssh` binary.

#### Cautions

Never hardcode private key paths, passphrases, or passwords directly in scripts; and always verify host keys in production (avoid `AutoAddPolicy` for anything beyond a lab/dev environment) to protect against man-in-the-middle attacks.

#### Script

```python
#!/usr/bin/env python3
"""ssh_run.py - Execute a remote command over SSH on one or more hosts."""

import argparse
import sys
import paramiko


def run_command(host: str, user: str, command: str, key_path: str = None,
                 password: str = None, port: int = 22, timeout: int = 10) -> dict:
    client = paramiko.SSHClient()
    client.set_missing_host_key_policy(paramiko.AutoAddPolicy())  # lab/dev only, see cautions

    try:
        if key_path:
            client.connect(host, port=port, username=user, key_filename=key_path, timeout=timeout)
        else:
            client.connect(host, port=port, username=user, password=password, timeout=timeout)

        stdin, stdout, stderr = client.exec_command(command, timeout=timeout)
        exit_status = stdout.channel.recv_exit_status()
        return {
            "host": host,
            "exit_status": exit_status,
            "stdout": stdout.read().decode(errors="replace"),
            "stderr": stderr.read().decode(errors="replace"),
        }
    finally:
        client.close()


def main():
    parser = argparse.ArgumentParser(description="Run a command over SSH on one or more hosts.")
    parser.add_argument("hosts", nargs="+", help="One or more target hosts")
    parser.add_argument("--user", required=True, help="SSH username")
    parser.add_argument("--command", required=True, help="Command to execute remotely")
    parser.add_argument("--key", help="Path to private key file")
    parser.add_argument("--password", help="SSH password (prefer --key in production)")
    parser.add_argument("--port", type=int, default=22)
    args = parser.parse_args()

    exit_code = 0
    for host in args.hosts:
        try:
            result = run_command(host, args.user, args.command, args.key, args.password, args.port)
            print(f"--- {host} (exit={result['exit_status']}) ---")
            print(result["stdout"], end="")
            if result["stderr"]:
                print(f"[stderr] {result['stderr']}", file=sys.stderr)
            if result["exit_status"] != 0:
                exit_code = 1
        except paramiko.AuthenticationException:
            print(f"--- {host}: authentication failed ---", file=sys.stderr)
            exit_code = 1
        except Exception as exc:
            print(f"--- {host}: connection error: {exc} ---", file=sys.stderr)
            exit_code = 1

    sys.exit(exit_code)


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

`paramiko.SSHClient` handles the SSH handshake, authentication (key-based or password), and command execution; `exec_command` returns file-like objects for stdin/stdout/stderr, and `recv_exit_status()` blocks until the remote command finishes and reports its exit code.

```bash
# Run a command on a single host using key-based auth
python3 ssh_run.py web01.internal --user deploy --command "uptime" --key ~/.ssh/id_ed25519

# Run the same command across multiple hosts (fleet-wide check)
python3 ssh_run.py web01.internal web02.internal web03.internal \
    --user deploy --command "df -h /" --key ~/.ssh/id_ed25519

# Password auth (only recommended for quick lab testing)
python3 ssh_run.py testbox.local --user labuser --command "whoami" --password "s3cret"
```

#### Sample Output

```bash
$ python3 ssh_run.py web01.internal web02.internal --user deploy --command "uptime" --key ~/.ssh/id_ed25519
--- web01.internal (exit=0) ---
 10:32:01 up 45 days,  3:12,  1 user,  load average: 0.15, 0.22, 0.19
--- web02.internal (exit=0) ---
 10:32:02 up 12 days,  9:01,  1 user,  load average: 0.05, 0.08, 0.10
```

#### Variations / Flags

- Add `--parallel` using `concurrent.futures.ThreadPoolExecutor` to run against many hosts simultaneously rather than sequentially — critical when targeting dozens/hundreds of servers.
- Add `--sudo` to prefix the command with `sudo -S` and pipe a password via stdin for privileged operations.
- Replace `AutoAddPolicy` with `paramiko.RejectPolicy` plus a pre-populated `known_hosts` file for strict host-key verification in production use.

#### Common Pitfalls

- **`AutoAddPolicy` in production**: automatically trusting unknown host keys defeats SSH's protection against man-in-the-middle attacks — use it only for throwaway lab environments, never for production infrastructure.
- **Wrong key permissions**: private key files with overly permissive filesystem permissions (e.g., `644`) will be rejected by strict SSH servers — set `chmod 600 ~/.ssh/id_ed25519`.
- **Passphrase-protected keys**: `key_filename` alone doesn't handle an encrypted private key's passphrase — pass it via `client.connect(..., passphrase="...")` or use `paramiko.RSAKey.from_private_key_file(path, password=passphrase)`.
- **Long-running commands and timeouts**: the default `timeout` applies to the whole exec, not just connection setup — increase it for commands expected to run longer than a few seconds (e.g., a backup job).

### 14.12 SCP File Transfers

#### Use Case

Securely copy a file to or from a remote server over an encrypted SSH connection — the go-to method for moving a single config file, log bundle, or deployment artifact to/from a server when a full SFTP session or FTP setup would be overkill.

#### Prerequisites

Requires `paramiko` (for the underlying SSH transport) and the `scp` package (a thin wrapper implementing the SCP protocol on top of a paramiko transport).

```bash
pip install paramiko scp --break-system-packages   # Linux/macOS externally-managed Python
pip install paramiko scp                           # Windows / virtualenv
```

Cross-platform (Linux/macOS/Windows) — no dependency on the OS's native `scp` binary.

#### Cautions

The classic SCP protocol has known parsing edge cases and is considered legacy by OpenSSH (which now prefers SFTP internally) — for new automation, prefer SFTP (14.13) where possible, and reserve this script for environments where SCP is specifically required.

#### Script

```python
#!/usr/bin/env python3
"""scp_transfer.py - Copy files to/from a remote host via SCP over SSH."""

import argparse
import sys
import paramiko
from scp import SCPClient


def get_ssh_client(host: str, user: str, key_path: str = None,
                    password: str = None, port: int = 22) -> paramiko.SSHClient:
    client = paramiko.SSHClient()
    client.set_missing_host_key_policy(paramiko.AutoAddPolicy())  # lab/dev only
    if key_path:
        client.connect(host, port=port, username=user, key_filename=key_path, timeout=10)
    else:
        client.connect(host, port=port, username=user, password=password, timeout=10)
    return client


def main():
    parser = argparse.ArgumentParser(description="Copy files via SCP.")
    parser.add_argument("host", help="Remote host")
    parser.add_argument("source", help="Source path (local for upload, remote for download)")
    parser.add_argument("destination", help="Destination path")
    parser.add_argument("--direction", choices=["upload", "download"], required=True)
    parser.add_argument("--user", required=True)
    parser.add_argument("--key", help="Path to private key file")
    parser.add_argument("--password", help="SSH password (prefer --key in production)")
    parser.add_argument("--port", type=int, default=22)
    parser.add_argument("--recursive", action="store_true", help="Copy directories recursively")
    args = parser.parse_args()

    ssh = get_ssh_client(args.host, args.user, args.key, args.password, args.port)
    try:
        with SCPClient(ssh.get_transport()) as scp:
            if args.direction == "upload":
                scp.put(args.source, args.destination, recursive=args.recursive)
                print(f"Uploaded '{args.source}' -> {args.host}:'{args.destination}'")
            else:
                scp.get(args.source, args.destination, recursive=args.recursive)
                print(f"Downloaded {args.host}:'{args.source}' -> '{args.destination}'")
    except Exception as exc:
        print(f"SCP transfer failed: {exc}", file=sys.stderr)
        sys.exit(1)
    finally:
        ssh.close()


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

A `paramiko.SSHClient` establishes the encrypted SSH connection and authentication; `SCPClient` then wraps that transport to run `put` (upload) or `get` (download) operations, streaming file data over the same secure channel.

```bash
# Upload a single local file to a remote server
python3 scp_transfer.py web01.internal ./deploy.tar.gz /tmp/deploy.tar.gz \
    --direction upload --user deploy --key ~/.ssh/id_ed25519

# Download a remote log file to the local machine
python3 scp_transfer.py web01.internal /var/log/app.log ./app.log \
    --direction download --user deploy --key ~/.ssh/id_ed25519

# Recursively upload an entire local directory
python3 scp_transfer.py web01.internal ./release/ /opt/app/release/ \
    --direction upload --user deploy --key ~/.ssh/id_ed25519 --recursive
```

#### Sample Output

```bash
$ python3 scp_transfer.py web01.internal ./deploy.tar.gz /tmp/deploy.tar.gz --direction upload --user deploy --key ~/.ssh/id_ed25519
Uploaded './deploy.tar.gz' -> web01.internal:'/tmp/deploy.tar.gz'
```

#### Variations / Flags

- Add `--progress` using `SCPClient(transport, progress=callback)` to print a live percentage/byte-count progress bar for large transfers.
- Loop over a host list (as in 14.11) to push the same file to many servers in one run.
- For directory syncs where only changed files should transfer, prefer `rsync` over SSH instead of SCP — SCP always transfers the full file regardless of what changed.

#### Common Pitfalls

- **`AutoAddPolicy` in production**: same caution as 14.11 — blindly trusting unknown host keys is a security risk outside of lab/dev use.
- **Recursive flag mismatch**: forgetting `--recursive` when the source is a directory raises an error rather than silently copying — always confirm whether the source is a file or directory first.
- **Legacy SCP protocol quirks**: some newer OpenSSH server versions restrict or deprecate the legacy SCP protocol in favor of SFTP-based transfers; if `scp.get`/`scp.put` fails unexpectedly against a modern server, switch to the SFTP script in 14.13.
- **Destination directory must exist**: unlike some rsync-based tools, SCP will not auto-create missing destination directories — create them first via an SSH command (see 14.11) if needed.

### 14.13 SFTP Automation

#### Use Case

Script interactive-style file transfers over SFTP — listing remote directories, uploading/downloading individual files, or recursively syncing a whole directory tree — used for scheduled report deliveries to a partner's SFTP drop, pulling nightly data exports from a vendor, or backing up a remote application's upload folder.

#### Prerequisites

Requires the third-party `paramiko` package, which includes a full SFTP client implementation built on top of its SSH transport.

```bash
pip install paramiko --break-system-packages   # Linux/macOS externally-managed Python
pip install paramiko                           # Windows / virtualenv
```

Cross-platform (Linux/macOS/Windows) — no dependency on any native `sftp` binary.

#### Cautions

SFTP servers frequently `chroot` accounts to a specific subdirectory — attempting to navigate outside that path with `..` will fail even though the connection itself succeeds; always confirm the account's allowed root with the server owner beforehand.

#### Script

```python
#!/usr/bin/env python3
"""sftp_automation.py - List, upload, and download files over SFTP."""

import argparse
import os
import sys
import paramiko


def get_sftp_client(host: str, user: str, key_path: str = None,
                     password: str = None, port: int = 22):
    transport = paramiko.Transport((host, port))
    if key_path:
        pkey = paramiko.Ed25519Key.from_private_key_file(key_path)
        transport.connect(username=user, pkey=pkey)
    else:
        transport.connect(username=user, password=password)
    sftp = paramiko.SFTPClient.from_transport(transport)
    return sftp, transport


def list_dir(sftp, remote_path: str):
    for entry in sftp.listdir_attr(remote_path):
        kind = "d" if entry.st_mode and (entry.st_mode & 0o40000) else "f"
        print(f"{kind}  {entry.filename:<30} {entry.st_size} bytes")


def upload_recursive(sftp, local_dir: str, remote_dir: str):
    try:
        sftp.mkdir(remote_dir)
    except IOError:
        pass  # already exists
    for item in os.listdir(local_dir):
        local_path = os.path.join(local_dir, item)
        remote_path = f"{remote_dir}/{item}"
        if os.path.isdir(local_path):
            upload_recursive(sftp, local_path, remote_path)
        else:
            sftp.put(local_path, remote_path)


def main():
    parser = argparse.ArgumentParser(description="SFTP automation: list/get/put files.")
    parser.add_argument("host")
    parser.add_argument("--user", required=True)
    parser.add_argument("--key", help="Path to Ed25519 private key file")
    parser.add_argument("--password", help="SSH password (prefer --key in production)")
    parser.add_argument("--port", type=int, default=22)
    parser.add_argument("--action", choices=["list", "get", "put", "put-recursive"], required=True)
    parser.add_argument("--remote-path", required=True)
    parser.add_argument("--local-path", help="Required for get/put actions")
    args = parser.parse_args()

    sftp, transport = get_sftp_client(args.host, args.user, args.key, args.password, args.port)
    try:
        if args.action == "list":
            list_dir(sftp, args.remote_path)
        elif args.action == "get":
            sftp.get(args.remote_path, args.local_path)
            print(f"Downloaded {args.host}:'{args.remote_path}' -> '{args.local_path}'")
        elif args.action == "put":
            sftp.put(args.local_path, args.remote_path)
            print(f"Uploaded '{args.local_path}' -> {args.host}:'{args.remote_path}'")
        elif args.action == "put-recursive":
            upload_recursive(sftp, args.local_path, args.remote_path)
            print(f"Recursively uploaded '{args.local_path}' -> {args.host}:'{args.remote_path}'")
    except Exception as exc:
        print(f"SFTP operation failed: {exc}", file=sys.stderr)
        sys.exit(1)
    finally:
        sftp.close()
        transport.close()


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

A `paramiko.Transport` handles the low-level SSH connection and authentication; `paramiko.SFTPClient.from_transport` layers the SFTP subsystem on top, exposing familiar operations (`listdir_attr`, `get`, `put`, `mkdir`) that mirror a standard SFTP command-line session.

```bash
# List files in a remote directory
python3 sftp_automation.py sftp.partner.com --user reportuser --key ~/.ssh/id_ed25519 \
    --action list --remote-path /incoming

# Download a single file
python3 sftp_automation.py sftp.partner.com --user reportuser --key ~/.ssh/id_ed25519 \
    --action get --remote-path /incoming/data.csv --local-path ./data.csv

# Upload a single file
python3 sftp_automation.py sftp.partner.com --user reportuser --key ~/.ssh/id_ed25519 \
    --action put --local-path ./report.csv --remote-path /outgoing/report.csv

# Recursively upload an entire local directory tree
python3 sftp_automation.py sftp.partner.com --user reportuser --key ~/.ssh/id_ed25519 \
    --action put-recursive --local-path ./exports --remote-path /outgoing/exports
```

#### Sample Output

```bash
$ python3 sftp_automation.py sftp.partner.com --user reportuser --key ~/.ssh/id_ed25519 --action list --remote-path /incoming
f  daily_2026-07-17.csv          48213 bytes
f  daily_2026-07-18.csv          49810 bytes
d  archive                       4096 bytes
```

#### Variations / Flags

- Add `--filter "*.csv"` using `fnmatch` to only download files matching a pattern from a mixed-content directory.
- Add `--delete-after-download` to clean up processed files from the remote drop folder once safely transferred.
- Combine with 14.3's availability check as a pre-flight test before attempting the transfer, to fail fast with a clear message if the SFTP host is unreachable.

#### Common Pitfalls

- **Chrooted accounts**: many SFTP-only accounts are restricted to a subdirectory — using an absolute path like `/etc/passwd` will fail or resolve unexpectedly; always test with `--action list --remote-path .` first to see the account's actual root.
- **Key type mismatch**: this script assumes an Ed25519 key (`paramiko.Ed25519Key`) — for RSA keys use `paramiko.RSAKey.from_private_key_file` instead, or the connection will fail with a key-parsing error.
- **`mkdir` on existing directory**: `sftp.mkdir()` raises `IOError` if the directory already exists — the recursive upload function above deliberately swallows this specific error, but don't apply that same blanket suppression elsewhere without care.
- **Large recursive uploads and partial failures**: if a recursive upload is interrupted partway through, re-running it will re-upload already-transferred files (no built-in resume/skip logic) — add file-existence checks if this matters for your use case.

### 14.14 Configure Firewall Rules

#### Use Case

Programmatically allow or block a port or IP address at the OS firewall level — used when provisioning a new server (opening only the ports a service actually needs), responding to an active attack (quickly blocking a malicious source IP), or codifying firewall changes as part of a repeatable deployment script instead of manual, undocumented edits.

#### Prerequisites

Uses only the Python standard library (`subprocess`, `platform`, `argparse`) as a wrapper around native OS firewall tools: `ufw`/`iptables` on Linux, `netsh advfirewall` on Windows. No third-party Python packages required, but the underlying OS tool must be present and the script must be run with sufficient privileges.

```bash
# Linux (Debian/Ubuntu) - ufw is often pre-installed; if not:
sudo apt-get install ufw

# Linux (RHEL/CentOS) - iptables is typically pre-installed
sudo yum install iptables-services

# Windows - netsh ships with the OS by default, no install needed

python3 --version   # Python 3.8+
```

#### Cautions

Firewall changes can lock you out of a remote server if applied incorrectly (e.g., blocking your own SSH port) — always test rule changes on a console/out-of-band connection first, or add an explicit allow rule for your current management IP *before* applying broader restrictions. This script requires root/Administrator privileges to run.

#### Script

```python
#!/usr/bin/env python3
"""configure_firewall.py - Allow or block a port/IP using the native OS firewall."""

import argparse
import platform
import subprocess
import sys


def run(cmd: list) -> subprocess.CompletedProcess:
    return subprocess.run(cmd, capture_output=True, text=True)


def apply_rule_linux(action: str, port: int, protocol: str, source_ip: str, use_ufw: bool):
    if use_ufw:
        rule = ["sudo", "ufw"]
        verb = "allow" if action == "allow" else "deny"
        target = f"{port}/{protocol}" if not source_ip else f"from {source_ip} to any port {port} proto {protocol}"
        rule += [verb] + target.split()
        return run(rule)
    else:
        chain_action = "-A" if action == "allow" else "-A"  # both use ACCEPT/DROP via -j
        jump = "ACCEPT" if action == "allow" else "DROP"
        cmd = ["sudo", "iptables", chain_action, "INPUT", "-p", protocol, "--dport", str(port)]
        if source_ip:
            cmd += ["-s", source_ip]
        cmd += ["-j", jump]
        return run(cmd)


def apply_rule_windows(action: str, port: int, protocol: str, source_ip: str):
    rule_name = f"AutoRule_{protocol}_{port}_{action}"
    verb = "allow" if action == "allow" else "block"
    cmd = [
        "netsh", "advfirewall", "firewall", "add", "rule",
        f"name={rule_name}", "dir=in", f"action={verb}",
        f"protocol={protocol.upper()}", f"localport={port}",
    ]
    if source_ip:
        cmd.append(f"remoteip={source_ip}")
    return run(cmd)


def main():
    parser = argparse.ArgumentParser(description="Configure a firewall rule (allow/block a port or IP).")
    parser.add_argument("--action", choices=["allow", "block"], required=True)
    parser.add_argument("--port", type=int, required=True)
    parser.add_argument("--protocol", choices=["tcp", "udp"], default="tcp")
    parser.add_argument("--source-ip", help="Restrict the rule to a specific source IP/CIDR")
    parser.add_argument("--use-iptables", action="store_true",
                         help="Use raw iptables instead of ufw on Linux")
    args = parser.parse_args()

    system = platform.system().lower()

    if system == "linux":
        result = apply_rule_linux(args.action, args.port, args.protocol,
                                   args.source_ip, use_ufw=not args.use_iptables)
    elif system == "windows":
        result = apply_rule_windows(args.action, args.port, args.protocol, args.source_ip)
    else:
        print(f"Unsupported platform: {system} (macOS requires pfctl, not covered here)", file=sys.stderr)
        sys.exit(1)

    if result.returncode != 0:
        print(f"Firewall command failed:\n{result.stderr}", file=sys.stderr)
        sys.exit(1)

    print(f"Rule applied: {args.action} {args.protocol}/{args.port}"
          f"{f' from {args.source_ip}' if args.source_ip else ''}")


if __name__ == "__main__":
    main()
```

#### How It Works and How to Run

The script detects the current OS with `platform.system()` and dispatches to the correct native tool: `ufw` (simpler, recommended default on Ubuntu/Debian) or raw `iptables` (`--use-iptables`) on Linux, and `netsh advfirewall` on Windows. All variants require elevated privileges (`sudo` on Linux, Administrator/elevated PowerShell on Windows).

```bash
# Allow inbound TCP traffic on port 8080 (Linux, using ufw)
sudo python3 configure_firewall.py --action allow --port 8080 --protocol tcp

# Block a specific malicious IP entirely on port 22 (Linux, using iptables)
sudo python3 configure_firewall.py --action block --port 22 --protocol tcp \
    --source-ip 203.0.113.55 --use-iptables

# Allow inbound TCP traffic on port 3389 (Windows, run as Administrator)
python configure_firewall.py --action allow --port 3389 --protocol tcp
```

#### Sample Output

```bash
$ sudo python3 configure_firewall.py --action allow --port 8080 --protocol tcp
Rule applied: allow tcp/8080
```

```bash
$ sudo python3 configure_firewall.py --action block --port 22 --protocol tcp --source-ip 203.0.113.55 --use-iptables
Rule applied: block tcp/22 from 203.0.113.55
```

#### Variations / Flags

- Add `--dry-run` to print the exact `ufw`/`iptables`/`netsh` command that *would* run without executing it — highly recommended before applying rules on a production server.
- Add `--persist` to save `iptables` rules with `iptables-save > /etc/iptables/rules.v4` so they survive a reboot (raw iptables rules are otherwise ephemeral).
- Add `--list` to shell out to `sudo ufw status numbered` / `sudo iptables -L -n --line-numbers` / `netsh advfirewall firewall show rule name=all` to audit current rules before making changes.

#### Common Pitfalls

- **Locking yourself out**: blocking a port you're currently connected through (e.g., SSH on 22) will immediately drop your session with no way back in except console/out-of-band access — always allow-list your own management IP first.
- **Requires elevated privileges**: running without `sudo` (Linux) or an elevated prompt (Windows) fails silently or with a permissions error — the script does not auto-elevate itself.
- **`ufw` vs `iptables` rule conflicts**: mixing raw `iptables` rules with `ufw` (which manages its own iptables chains under the hood) can produce confusing, hard-to-debug interactions — pick one approach per host and stay consistent.
- **Rules not persisting after reboot**: raw `iptables` rules are in-memory only unless explicitly saved (see `--persist` above); `ufw` and `netsh` rules persist automatically, which is one reason `ufw` is the recommended default on Linux.
- **macOS not covered**: this script does not implement `pfctl` support for macOS — macOS firewall automation requires separate, more involved `pf.conf` rule management not included here.


#### Environment Assumptions

Unless a script file states otherwise, examples assume Python 3.8+ is installed and available on the `PATH` as `python3` (Linux/macOS) or `python` (Windows), and that you have permission to install packages with `pip`.
