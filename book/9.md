## Chapter 9: Task Scheduling

Every system administrator eventually hits the same wall: a task that needs to run without a human sitting there to click "go." Maybe it's a backup that has to fire at 2 AM, a health check that needs to poll a server every sixty seconds, or a flaky upload script that keeps failing on the first try but succeeds on the third. Manually babysitting these jobs doesn't scale — and forgetting to run one manually is how backups go missing and disks fill up unnoticed.

This chapter is about closing that gap with Python. Rather than reaching for a single framework, you'll build a toolbox: small, focused scripts that each solve one scheduling problem well. Some rely only on the standard library and run for as long as your terminal or process supervisor keeps them alive. Others hand the job off to the operating system itself — cron on Linux/macOS, Task Scheduler on Windows — so the schedule survives reboots and doesn't depend on a Python process staying resident. A few solve adjacent problems that show up constantly alongside scheduling: retrying a task that failed, or queuing a batch of jobs so they run one after another instead of stepping on each other.

By the end of the chapter you'll have working scripts for the eight most common task-scheduling patterns a sysadmin runs into, what each one is for, how to install and run it, and where it tends to bite people in production.

#### Two Philosophies: In-Process vs OS-Level Scheduling

Before diving into the scripts, it's worth understanding the fundamental split in this chapter. **In-process scheduling** (the `schedule` library, the periodic-execution script, the wait-until-a-time script) means a Python process stays running and fires jobs internally. This is simple and portable, but if the process crashes, dies with the terminal, or the machine reboots, the schedule dies with it. **OS-level scheduling** (cron, Windows Task Scheduler) hands the job off to the operating system. The OS wakes up at the right moment and launches your command itself — no long-running Python process required, and the schedule survives reboots. The tradeoff is platform lock-in: cron entries don't work on Windows, and `schtasks` entries don't work on Linux.

Neither approach is universally "better." A quick health check you want to babysit interactively is a fine fit for the in-process scripts. A nightly backup that absolutely must run whether or not anyone is logged in belongs in cron or Task Scheduler.


### 9.1 Schedule Jobs Using Schedule

#### Use Case

A sysadmin wants a lightweight, in-process way to run a recurring command — "ping this endpoint every 30 seconds", "rotate this log every hour" — without writing raw `time.sleep()` loops or touching cron. `schedule_with_library.py` wraps the popular `schedule` package so any shell command can be turned into a recurring job with one line at the terminal.

#### Prerequisites

- Python 3.7+
- Third-party package: `schedule`
- OS scope: cross-platform (Linux, macOS, Windows) — the scheduling logic is pure Python, not tied to any OS scheduler

```bash
pip install schedule
# or, using the project's requirements file:
pip install -r requirements.txt
```

#### Cautions

This script only fires jobs while it is actively running in the foreground (or under a process supervisor) — closing the terminal or killing the process stops all scheduling immediately, with no persistence or catch-up for missed runs.

#### Script

```python
#!/usr/bin/env python3
"""
schedule_with_library.py
------------------------------
Schedules and runs a command repeatedly using the 'schedule' library —
a lightweight, in-process Python job scheduler (no cron/OS integration
required). The script itself must keep running for jobs to fire.

Requires: pip install schedule

Usage:
    python schedule_with_library.py <command> --every <N> --unit <seconds|minutes|hours|days>

Example:
    python schedule_with_library.py "echo Hello" --every 10 --unit seconds
    python schedule_with_library.py "python3 backup.py" --every 1 --unit hours
"""

import argparse
import shlex
import subprocess
import sys
from datetime import datetime

try:
    import schedule
except ImportError:
    print("Error: the 'schedule' package is required.")
    print("Install it with: pip install -r requirements.txt")
    sys.exit(1)

import time


def run_command(command: str) -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    print(f"[{timestamp}] Running: {command}")

    try:
        result = subprocess.run(shlex.split(command), capture_output=True, text=True)
        if result.stdout:
            print(result.stdout.strip())
        if result.returncode != 0:
            print(f"[{timestamp}] Command exited with code {result.returncode}")
            if result.stderr:
                print(result.stderr.strip())
    except FileNotFoundError:
        print(f"[{timestamp}] Error: command not found: {command}")


def build_job(command: str, every: int, unit: str):
    job = getattr(schedule.every(every), unit)
    job.do(run_command, command)


def main():
    parser = argparse.ArgumentParser(description="Schedule a recurring command using the 'schedule' library.")
    parser.add_argument("command", help="Command to run (quote it if it has spaces/args)")
    parser.add_argument("--every", type=int, default=1, help="Run every N units (default: 1)")
    parser.add_argument("--unit", default="minutes",
                         choices=["seconds", "minutes", "hours", "days"],
                         help="Time unit for the interval (default: minutes)")
    args = parser.parse_args()

    build_job(args.command, args.every, args.unit)

    print(f"Scheduled '{args.command}' to run every {args.every} {args.unit}.")
    print("This script must keep running for the schedule to fire. Press Ctrl+C to stop.\n")

    try:
        while True:
            schedule.run_pending()
            time.sleep(1)
    except KeyboardInterrupt:
        print("\nScheduler stopped.")


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

`build_job()` calls `schedule.every(N)` to get a `Job` object, then dynamically looks up the unit attribute (`.seconds`, `.minutes`, `.hours`, or `.days`) with `getattr`, and finally calls `.do(run_command, command)` to register the callback. The main loop calls `schedule.run_pending()` once a second, which checks whether any registered job is due and executes it if so.

```bash
# Run a health check every 30 seconds
python schedule_with_library.py "curl -sf http://localhost:8080/health" --every 30 --unit seconds

# Run a backup every hour, on the hour cadence relative to start time
python schedule_with_library.py "python3 backup.py" --every 1 --unit hours

# Run daily maintenance every 1 day
python schedule_with_library.py "python3 cleanup.py" --every 1 --unit days
```

#### Sample Output

```
$ python schedule_with_library.py "echo Hello" --every 10 --unit seconds
Scheduled 'echo Hello' to run every 10 seconds.
This script must keep running for the schedule to fire. Press Ctrl+C to stop.

[2026-07-17 09:00:10] Running: echo Hello
Hello
[2026-07-17 09:00:20] Running: echo Hello
Hello
^C
Scheduler stopped.
```

#### Variations / Flags

- Add `.at("10:30")` support (schedule's own API) for a `--at HH:MM` flag to anchor daily/hourly jobs to a specific clock time.
- Add a `--jitter N` flag and call `time.sleep(random.uniform(0, N))` inside `run_command` to avoid a thundering-herd effect if you run several instances.
- Wrap the whole script in a `systemd` unit or `nohup ... &` so it survives a terminal closing (see Common Pitfalls below).

#### Common Pitfalls

- **Process death = schedule death.** Unlike cron, nothing here survives a reboot or a closed terminal. For anything that must run unattended long-term, prefer 9.2/9.3.
- **Drift on `.hours`/`.days` intervals.** The interval is relative to when the script started, not to wall-clock boundaries — restarting the script resets the cadence.
- **Missing package.** If `schedule` isn't installed, the script fails fast with a clear message rather than a raw `ModuleNotFoundError` traceback — but only if you actually read stdout, since some process supervisors swallow it.
- **Blocking commands.** `subprocess.run()` is synchronous, so a long-running command will delay the next scheduled check; it's not designed for jobs that overlap in time.

### 9.2 Schedule Jobs Using Cron

#### Use Case

A sysadmin needs a nightly backup, log rotation, or report-generation job to run reliably even if no one is logged in and no Python process stays resident — and to survive server reboots. `schedule_with_cron.py` manages entries in the current user's crontab programmatically, so cron jobs can be added, listed, and removed from a script instead of hand-editing `crontab -e`.

#### Prerequisites

- Python 3.7+
- Third-party package: `python-crontab`
- OS scope: **Linux/macOS only** — cron does not exist on Windows (use 9.3 there instead)
- The `cron` daemon must be installed and running on the host

```bash
pip install python-crontab
# or:
pip install -r requirements.txt

# Verify cron is available:
which crontab
systemctl status cron    # Debian/Ubuntu
systemctl status crond   # RHEL/CentOS/Fedora
```

#### Cautions

This script writes directly to your real crontab — a mistake here (wrong cron expression, wrong command path) creates a job that will actually run repeatedly on your system, potentially at 2 AM when no one is watching.

#### Script

```python
#!/usr/bin/env python3
"""
schedule_with_cron.py
---------------------------
Adds, lists, or removes an entry in the current user's crontab (Linux/macOS)
to run a command on a schedule using standard cron syntax. This script
manages OS-level cron jobs — it does not need to keep running itself.

Requires: pip install python-crontab
Platform: Linux/macOS only (cron is not available on Windows;
          see windows_task_scheduler.py for the Windows equivalent).

Usage:
    python schedule_with_cron.py add "<cron_expression>" "<command>" [--comment "label"]
    python schedule_with_cron.py list
    python schedule_with_cron.py remove --comment "label"

Example:
    python schedule_with_cron.py add "0 2 * * *" "python3 /home/user/backup.py" --comment "daily-backup"
    python schedule_with_cron.py list
    python schedule_with_cron.py remove --comment "daily-backup"
"""

import argparse
import platform
import sys

try:
    from crontab import CronTab
except ImportError:
    print("Error: the 'python-crontab' package is required.")
    print("Install it with: pip install -r requirements.txt")
    sys.exit(1)


def check_platform():
    if platform.system() == "Windows":
        print("Error: cron is not available on Windows.")
        print("Use windows_task_scheduler.py instead for scheduled tasks on Windows.")
        sys.exit(1)


def add_job(cron_expression: str, command: str, comment: str) -> None:
    cron = CronTab(user=True)

    job = cron.new(command=command, comment=comment or "")
    try:
        job.setall(cron_expression)
    except (KeyError, ValueError) as e:
        print(f"Error: invalid cron expression '{cron_expression}': {e}")
        sys.exit(1)

    if not job.is_valid():
        print(f"Error: cron expression '{cron_expression}' is not valid.")
        sys.exit(1)

    cron.write()
    print(f"Added cron job: '{cron_expression}' -> {command}")
    if comment:
        print(f"Comment/label: {comment}")
    print(f"Next run time: {job.schedule().get_next()}")


def list_jobs() -> None:
    cron = CronTab(user=True)
    jobs = list(cron)

    if not jobs:
        print("No cron jobs found for the current user.")
        return

    print(f"Cron jobs for current user ({len(jobs)}):\n")
    for job in jobs:
        status = "enabled" if job.is_enabled() else "disabled"
        print(f"[{status}] {job}")


def remove_jobs(comment: str) -> None:
    cron = CronTab(user=True)
    removed = cron.remove_all(comment=comment)
    cron.write()

    if removed:
        print(f"Removed {removed} job(s) with comment '{comment}'.")
    else:
        print(f"No jobs found with comment '{comment}'.")


def main():
    check_platform()

    parser = argparse.ArgumentParser(description="Manage cron jobs (add/list/remove).")
    subparsers = parser.add_subparsers(dest="action", required=True)

    add_parser = subparsers.add_parser("add", help="Add a new cron job")
    add_parser.add_argument("cron_expression", help="Cron schedule expression, e.g. '0 2 * * *'")
    add_parser.add_argument("command", help="Command to run")
    add_parser.add_argument("--comment", default="", help="Label to identify this job later")

    subparsers.add_parser("list", help="List all cron jobs for the current user")

    remove_parser = subparsers.add_parser("remove", help="Remove cron job(s) by comment/label")
    remove_parser.add_argument("--comment", required=True, help="Label of the job(s) to remove")

    args = parser.parse_args()

    if args.action == "add":
        add_job(args.cron_expression, args.command, args.comment)
    elif args.action == "list":
        list_jobs()
    elif args.action == "remove":
        remove_jobs(args.comment)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

`CronTab(user=True)` loads the current user's crontab. `add` creates a new job with `cron.new()`, applies the schedule with `job.setall(cron_expression)` (standard 5-field cron syntax: minute, hour, day-of-month, month, day-of-week), validates it, then persists with `cron.write()`. `remove` uses `cron.remove_all(comment=...)` to delete every job carrying that label in one call — which is why every job you add should carry a `--comment`, or you'll have no reliable way to remove it later.

```bash
# Add a daily 2 AM backup job, labeled for later removal
python schedule_with_cron.py add "0 2 * * *" "python3 /home/user/backup.py" --comment "daily-backup"

# Add a job that runs every 15 minutes
python schedule_with_cron.py add "*/15 * * * *" "curl -sf http://localhost/health" --comment "health-check"

# List every cron job for the current user
python schedule_with_cron.py list

# Remove a job by its label
python schedule_with_cron.py remove --comment "daily-backup"
```

#### Sample Output

```
$ python schedule_with_cron.py add "0 2 * * *" "python3 /home/user/backup.py" --comment "daily-backup"
Added cron job: '0 2 * * *' -> python3 /home/user/backup.py
Comment/label: daily-backup
Next run time: 2026-07-18 02:00:00

$ python schedule_with_cron.py list
Cron jobs for current user (1):

[enabled] 0 2 * * * python3 /home/user/backup.py # daily-backup

$ python schedule_with_cron.py remove --comment "daily-backup"
Removed 1 job(s) with comment 'daily-backup'.
```

#### Variations / Flags

- Add a `--disable`/`--enable` action that toggles `job.enable(False)`/`job.enable(True)` instead of removing jobs outright.
- Add validation with the `croniter` package (already in `requirements.txt`) to preview the next 5 run times before writing the job.
- Add a `--dry-run` flag that prints what would be added/removed without calling `cron.write()`.

#### Common Pitfalls

- **Permissions.** Writing to another user's crontab requires root and a different `CronTab(user="username")` call; running as the wrong user silently edits the wrong crontab.
- **PATH differences.** Cron runs with a minimal environment — commands that work fine in your interactive shell (relying on aliases or a PATH set in `.bashrc`) often fail silently under cron. Always use absolute paths to interpreters and scripts.
- **No output by default.** Cron normally emails job output to the local mail spool, which most systems don't have configured — redirect output explicitly (e.g. `>> /var/log/job.log 2>&1`) inside the `command` string if you need logs.
- **Forgetting `--comment`.** Without a label, `remove_jobs()` has nothing reliable to match against, and you'll need to edit the crontab by hand.
- **Windows.** This script hard-fails on Windows by design (`check_platform()`); that's expected behavior, not a bug — use 9.3 instead.

### 9.3 Windows Task Scheduler Automation

#### Use Case

A Windows sysadmin needs to create a recurring scheduled task — a nightly backup, an hourly log shipper — without opening the Task Scheduler GUI every time, and ideally from a script that can be checked into version control or run as part of a provisioning process. `windows_task_scheduler.py` wraps the built-in `schtasks` command-line tool to add, list, and remove scheduled tasks.

#### Prerequisites

- Python 3.7+ (standard library only — no pip installs required)
- OS scope: **Windows only** — this script wraps `schtasks`, which does not exist on Linux/macOS (use 9.2 there instead)
- No special install step; `schtasks` ships with every modern Windows version

```bash
# Nothing to install — verify schtasks is available:
schtasks /Query /FO LIST | more
```

#### Cautions

Creating a task with `/F` (force) silently overwrites any existing task of the same name — double-check `task_name` before running `add`, especially against a name you might already be using for something else.

#### Script

```python
#!/usr/bin/env python3
"""
windows_task_scheduler.py
-------------------------------
Adds, lists, or removes a Windows Task Scheduler task to run a command
on a schedule, using the built-in 'schtasks' command-line tool via subprocess.

Platform: Windows only.

Usage:
    python windows_task_scheduler.py add <task_name> <command> --schedule <DAILY|WEEKLY|HOURLY|ONCE> --time HH:MM
    python windows_task_scheduler.py list
    python windows_task_scheduler.py remove <task_name>

Example:
    python windows_task_scheduler.py add "DailyBackup" "C:\\Python312\\python.exe C:\\scripts\\backup.py" --schedule DAILY --time 02:00
    python windows_task_scheduler.py list
    python windows_task_scheduler.py remove "DailyBackup"
"""

import argparse
import platform
import subprocess
import sys


def check_platform():
    if platform.system() != "Windows":
        print("Error: this script only works on Windows (it wraps the 'schtasks' command).")
        print("Use schedule_with_cron.py for Linux/macOS, or schedule_with_library.py for a cross-platform option.")
        sys.exit(1)


def add_task(task_name: str, command: str, schedule_type: str, time_str: str) -> None:
    cmd = [
        "schtasks", "/Create",
        "/TN", task_name,
        "/TR", command,
        "/SC", schedule_type,
        "/ST", time_str,
        "/F",  # Force overwrite if it already exists
    ]

    print(f"Creating scheduled task '{task_name}'...")
    result = subprocess.run(cmd, capture_output=True, text=True)

    if result.returncode == 0:
        print(f"Task '{task_name}' created successfully.")
        print(result.stdout.strip())
    else:
        print("Error creating task:")
        print(result.stderr.strip())
        sys.exit(1)


def list_tasks() -> None:
    result = subprocess.run(["schtasks", "/Query", "/FO", "LIST"], capture_output=True, text=True)

    if result.returncode == 0:
        print(result.stdout)
    else:
        print("Error listing tasks:")
        print(result.stderr.strip())
        sys.exit(1)


def remove_task(task_name: str) -> None:
    result = subprocess.run(["schtasks", "/Delete", "/TN", task_name, "/F"], capture_output=True, text=True)

    if result.returncode == 0:
        print(f"Task '{task_name}' removed successfully.")
    else:
        print("Error removing task:")
        print(result.stderr.strip())
        sys.exit(1)


def main():
    check_platform()

    parser = argparse.ArgumentParser(description="Manage Windows Task Scheduler tasks (add/list/remove).")
    subparsers = parser.add_subparsers(dest="action", required=True)

    add_parser = subparsers.add_parser("add", help="Add a new scheduled task")
    add_parser.add_argument("task_name", help="Name for the scheduled task")
    add_parser.add_argument("command", help="Full command to run (quote if it has spaces)")
    add_parser.add_argument("--schedule", default="DAILY",
                             choices=["MINUTE", "HOURLY", "DAILY", "WEEKLY", "ONCE"],
                             help="Schedule type (default: DAILY)")
    add_parser.add_argument("--time", default="09:00", help="Start time in HH:MM 24-hour format (default: 09:00)")

    subparsers.add_parser("list", help="List all scheduled tasks")

    remove_parser = subparsers.add_parser("remove", help="Remove a scheduled task by name")
    remove_parser.add_argument("task_name", help="Name of the task to remove")

    args = parser.parse_args()

    if args.action == "add":
        add_task(args.task_name, args.command, args.schedule, args.time)
    elif args.action == "list":
        list_tasks()
    elif args.action == "remove":
        remove_task(args.task_name)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

Every action is a thin wrapper around a `schtasks` subcommand invoked through `subprocess.run()`. `add` maps directly onto `schtasks /Create /TN <name> /TR <command> /SC <schedule> /ST <time> /F`. `list` runs `schtasks /Query /FO LIST` and prints raw output. `remove` runs `schtasks /Delete /TN <name> /F`. `check_platform()` guards every invocation so the script fails immediately and clearly if run anywhere but Windows.

```bash
# Create a daily 2:00 AM backup task
python windows_task_scheduler.py add "DailyBackup" "C:\Python312\python.exe C:\scripts\backup.py" --schedule DAILY --time 02:00

# Create an hourly health check
python windows_task_scheduler.py add "HourlyCheck" "C:\Python312\python.exe C:\scripts\healthcheck.py" --schedule HOURLY --time 00:00

# List every scheduled task on the system
python windows_task_scheduler.py list

# Remove a task by name
python windows_task_scheduler.py remove "DailyBackup"
```

#### Sample Output

```
> python windows_task_scheduler.py add "DailyBackup" "C:\Python312\python.exe C:\scripts\backup.py" --schedule DAILY --time 02:00
Creating scheduled task 'DailyBackup'...
Task 'DailyBackup' created successfully.
SUCCESS: The scheduled task "DailyBackup" has successfully been created.

> python windows_task_scheduler.py remove "DailyBackup"
Task 'DailyBackup' removed successfully.
```

#### Variations / Flags

- Add a `--user`/`--password` pair to support `/RU` and `/RP` flags, letting the task run under a specific service account rather than the current user.
- Add a `--highest` flag mapped to `/RL HIGHEST` to run the task with elevated privileges.
- Parse `schtasks /Query /FO CSV /V` instead of `LIST` output for machine-readable task details.

#### Common Pitfalls

- **Requires elevated permissions for some tasks.** Creating tasks that run whether or not a user is logged in, or under SYSTEM, typically requires an Administrator command prompt — a normal user prompt will get an "Access is denied" error from `schtasks`.
- **Path quoting.** Windows paths with spaces (`C:\Program Files\...`) inside the `/TR` command need careful quoting; wrap the whole command string when calling this script from a shell.
- **`/F` overwrites silently.** There's no confirmation prompt — if a task with that name already exists, it's replaced without warning.
- **This is Windows-only by design.** Running it on Linux/macOS exits immediately via `check_platform()`; that's expected, not a bug.
- **Time zone/DST.** `schtasks` schedules run in the machine's local time zone; a task set for "02:00" will shift relative to UTC across a daylight-saving transition.

### 9.4 Execute Tasks Periodically

#### Use Case

A sysadmin needs the simplest possible "run this every N seconds" loop — no external scheduler concepts, no cron syntax, just a fixed interval and an optional run count. `execute_periodically.py` covers quick polling jobs (checking if a service is up, tailing a status endpoint) that you want to watch run live in a terminal.

#### Prerequisites

- Python 3.7+ (standard library only)
- OS scope: cross-platform (Linux, macOS, Windows)

```bash
# Nothing to install — pure standard library
python3 --version
```

#### Cautions

There is no overlap protection: if a run takes longer than `--interval` seconds, the next run still starts immediately after the previous one finishes rather than waiting for the original interval boundary, which can cause runs to bunch up under load.

#### Script

```python
#!/usr/bin/env python3
"""
execute_periodically.py
-----------------------------
Runs a command repeatedly at a fixed interval, forever (or for a limited
number of runs), using only Python's standard library — no external
scheduler needed. Simpler alternative to schedule_with_library.py for
basic "run every N seconds" needs.

Usage:
    python execute_periodically.py <command> --interval <seconds> [--count N]

Example:
    python execute_periodically.py "echo Hello" --interval 5
    python execute_periodically.py "python3 healthcheck.py" --interval 60 --count 10
"""

import argparse
import shlex
import subprocess
import sys
import time
from datetime import datetime


def execute_periodically(command: str, interval: int, count: int) -> None:
    run_number = 0

    print(f"Running '{command}' every {interval} second(s)"
          + (f", {count} time(s) total." if count > 0 else ", indefinitely.") + "\n")
    print("Press Ctrl+C to stop.\n")

    try:
        while count == 0 or run_number < count:
            run_number += 1
            timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
            print(f"[{timestamp}] Run {run_number}: {command}")

            try:
                result = subprocess.run(shlex.split(command), capture_output=True, text=True)
                if result.stdout:
                    print(result.stdout.strip())
                if result.returncode != 0:
                    print(f"Exit code: {result.returncode}")
                    if result.stderr:
                        print(result.stderr.strip())
            except FileNotFoundError:
                print(f"Error: command not found: {command}")
                sys.exit(1)

            if count == 0 or run_number < count:
                time.sleep(interval)

    except KeyboardInterrupt:
        print(f"\nStopped after {run_number} run(s).")
        return

    print(f"\nCompleted {run_number} run(s).")


def main():
    parser = argparse.ArgumentParser(description="Run a command repeatedly at a fixed interval.")
    parser.add_argument("command", help="Command to run (quote it if it has spaces/args)")
    parser.add_argument("--interval", type=int, required=True, help="Seconds to wait between runs")
    parser.add_argument("--count", type=int, default=0, help="Number of times to run, 0 = run forever (default: 0)")
    args = parser.parse_args()

    execute_periodically(args.command, args.interval, args.count)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

The core is a `while` loop guarded by `count == 0 or run_number < count` — a `count` of `0` means "run forever" until interrupted. Each iteration runs the command with `subprocess.run()`, prints stdout and any non-zero exit code, then sleeps for `interval` seconds before the next run (skipping the final sleep once the last run has completed). `Ctrl+C` raises `KeyboardInterrupt`, which is caught to print a clean summary instead of a traceback.

```bash
# Run forever, every 5 seconds
python execute_periodically.py "echo Hello" --interval 5

# Run exactly 10 times, once a minute (good for a bounded health-check burst)
python execute_periodically.py "python3 healthcheck.py" --interval 60 --count 10

# Poll a web endpoint every 15 seconds indefinitely
python execute_periodically.py "curl -sf http://localhost:8080/status" --interval 15
```

#### Sample Output

```
$ python execute_periodically.py "echo Hello" --interval 5 --count 3

Press Ctrl+C to stop.

[2026-07-17 09:00:00] Run 1: echo Hello
Hello
[2026-07-17 09:00:05] Run 2: echo Hello
Hello
[2026-07-17 09:00:10] Run 3: echo Hello
Hello

Completed 3 run(s).
```

#### Variations / Flags

- Add a `--jitter N` flag to randomize the sleep slightly and avoid synchronized load spikes across multiple hosts running the same script.
- Add a `--stop-on-failure` flag that exits the loop the first time `result.returncode != 0`, useful for "keep polling until this succeeds" workflows.
- Pipe output to a log file with standard shell redirection (`>> poll.log 2>&1`) for a long-running background poll.

#### Common Pitfalls

- **No overlap protection.** If the command itself takes longer than `--interval`, runs will queue up back-to-back rather than respecting the original cadence — this script is not suitable for jobs whose duration might exceed the interval.
- **Runs in the foreground.** Closing the terminal kills the loop; for anything that needs to survive a logout, background it explicitly (`nohup ... &` on Linux/macOS, or wrap it as a scheduled task/service).
- **`shlex.split()` and shell features.** Pipes, redirects, and shell globbing in the command string won't work as expected, since the command is executed directly via `subprocess.run()` rather than through a shell.
- **Infinite by default confusion.** Forgetting `--count` means the script runs forever — fine for a monitor, surprising if you only meant to run it a few times.

### 9.5 Execute Tasks at Specific Times

#### Use Case

A sysadmin needs a one-off job to fire at a specific clock time or exact future date — "run this backup at 11:30 tonight", "kick off this migration at 9:00 AM on August 1st" — without the overhead of registering a cron entry or a Windows scheduled task for something that will only ever run once.

#### Prerequisites

- Python 3.7+ (standard library only)
- OS scope: cross-platform (Linux, macOS, Windows)

```bash
# Nothing to install — pure standard library
python3 --version
```

#### Cautions

The script blocks the terminal (polling every 30 seconds) until the target time arrives — it is not a background daemon, so the process (and the machine) must stay running and awake the entire time you're waiting, which can be hours if you schedule something far in the future.

#### Script

```python
#!/usr/bin/env python3
"""
execute_at_specific_time.py
---------------------------------
Waits until a specific clock time (or a specific future date/time) and
then runs a command once. Useful for "run this at 3 AM tonight" style
one-off scheduling without needing cron or Task Scheduler.

Usage:
    python execute_at_specific_time.py <command> --time HH:MM
    python execute_at_specific_time.py <command> --datetime "YYYY-MM-DD HH:MM:SS"

Example:
    python execute_at_specific_time.py "python3 backup.py" --time 23:30
    # Waits until 23:30 today (or tomorrow if that time already passed today), then runs.

    python execute_at_specific_time.py "python3 report.py" --datetime "2026-08-01 09:00:00"
    # Waits until that exact date and time, then runs.
"""

import argparse
import shlex
import subprocess
import sys
import time
from datetime import datetime, timedelta


def compute_target_time(time_str: str = None, datetime_str: str = None) -> datetime:
    now = datetime.now()

    if datetime_str:
        try:
            target = datetime.strptime(datetime_str, "%Y-%m-%d %H:%M:%S")
        except ValueError:
            print("Error: --datetime must be in format 'YYYY-MM-DD HH:MM:SS'")
            sys.exit(1)

        if target <= now:
            print(f"Error: target datetime '{datetime_str}' is in the past.")
            sys.exit(1)

        return target

    else:
        try:
            hour, minute = map(int, time_str.split(":"))
        except ValueError:
            print("Error: --time must be in format 'HH:MM' (24-hour)")
            sys.exit(1)

        target = now.replace(hour=hour, minute=minute, second=0, microsecond=0)

        if target <= now:
            target += timedelta(days=1)  # If time already passed today, schedule for tomorrow

        return target


def wait_and_execute(command: str, target: datetime) -> None:
    now = datetime.now()
    wait_seconds = (target - now).total_seconds()

    print(f"Command:     {command}")
    print(f"Target time: {target}")
    print(f"Waiting:     {wait_seconds / 60:.1f} minute(s)")
    print("\nPress Ctrl+C to cancel.\n")

    try:
        while True:
            remaining = (target - datetime.now()).total_seconds()
            if remaining <= 0:
                break
            time.sleep(min(remaining, 30))
    except KeyboardInterrupt:
        print("\nCancelled before execution.")
        sys.exit(0)

    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    print(f"[{timestamp}] Target time reached. Running command now: {command}\n")

    try:
        result = subprocess.run(shlex.split(command), capture_output=True, text=True)
        if result.stdout:
            print(result.stdout.strip())
        if result.returncode != 0:
            print(f"Exit code: {result.returncode}")
            if result.stderr:
                print(result.stderr.strip())
    except FileNotFoundError:
        print(f"Error: command not found: {command}")
        sys.exit(1)

    print("\nDone.")


def main():
    parser = argparse.ArgumentParser(description="Wait until a specific time, then run a command once.")
    parser.add_argument("command", help="Command to run (quote it if it has spaces/args)")
    group = parser.add_mutually_exclusive_group(required=True)
    group.add_argument("--time", help="Time today/tomorrow in HH:MM 24-hour format")
    group.add_argument("--datetime", help="Exact date and time in 'YYYY-MM-DD HH:MM:SS' format")
    args = parser.parse_args()

    target = compute_target_time(time_str=args.time, datetime_str=args.datetime)
    wait_and_execute(args.command, target)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

`compute_target_time()` handles two mutually exclusive modes: `--time HH:MM` (today, or tomorrow if that clock time already passed) and `--datetime "YYYY-MM-DD HH:MM:SS"` (an exact moment, which must be in the future). `wait_and_execute()` then polls in a loop, sleeping in chunks of at most 30 seconds so `Ctrl+C` remains responsive, until the target time is reached, at which point it runs the command once via `subprocess.run()` and exits.

```bash
# Wait until 23:30 today (or tomorrow, if it's already past 23:30)
python execute_at_specific_time.py "python3 backup.py" --time 23:30

# Wait for an exact future date and time
python execute_at_specific_time.py "python3 report.py" --datetime "2026-08-01 09:00:00"

# Cancel a wait in progress
# (press Ctrl+C any time before the target time arrives)
```

#### Sample Output

```
$ python execute_at_specific_time.py "echo done" --time 09:05
Command:     echo done
Target time: 2026-07-17 09:05:00
Waiting:     3.2 minute(s)

Press Ctrl+C to cancel.

[2026-07-17 09:05:00] Target time reached. Running command now: echo done

done

Done.
```

#### Variations / Flags

- Add a `--repeat` flag that, after execution, recomputes the next occurrence of `--time` and loops instead of exiting — turning this into a simple "same time every day" scheduler.
- Add a `--timezone` flag (using `zoneinfo`) for environments where the server's local time isn't the time zone you're thinking in.
- Log the wait and execution to a file so you can confirm afterward that the job actually fired while you were away.

#### Common Pitfalls

- **The process must stay alive.** If your SSH session drops, or the laptop sleeps, the wait — and the scheduled run — is lost. Use `nohup`/`screen`/`tmux` on Linux/macOS for waits longer than a few minutes, or prefer cron/Task Scheduler (9.2/9.3) for anything that truly must fire unattended.
- **Past times roll to tomorrow silently.** `--time 09:00` requested at 09:05 today schedules for 09:00 *tomorrow* — easy to miss if you expected an immediate error.
- **`--datetime` in the past is rejected outright**, unlike `--time`, which just rolls forward — the two flags behave differently on this point.
- **Long waits tie up a terminal/process for hours**, which is fine for a quick one-off but wasteful compared to an OS-level scheduled job for anything routine.

### 9.6 Retry Failed Tasks

#### Use Case

A sysadmin has a flaky one-shot operation — an API call to a rate-limited service, a file upload over an unreliable network — that fails intermittently but is expected to succeed if given a few more attempts. `retry_failed_task.py` wraps any command with automatic retries and exponential backoff, so transient failures resolve themselves without a human re-running the command by hand.

#### Prerequisites

- Python 3.7+ (standard library only)
- OS scope: cross-platform (Linux, macOS, Windows)

```bash
# Nothing to install — pure standard library
python3 --version
```

#### Cautions

This script retries based purely on exit code — it will happily retry a command that "succeeds" with exit code 0 but produced wrong output, and will just as happily keep retrying a command that fails deterministically (a typo in a path, bad credentials) for the full `--max-retries` count, burning the entire backoff window for nothing.

#### Script

```python
#!/usr/bin/env python3
"""
retry_failed_task.py
---------------------------
Runs a command and automatically retries it if it fails (non-zero exit
code), using an exponential backoff delay between attempts. Different
from restart_failed_process.py (Resource Management module) in that this
is intended for one-shot tasks (like an API call or upload) that should
eventually succeed and then stop — not long-running services.

Usage:
    python retry_failed_task.py <command> [--max-retries 5] [--base-delay 2] [--max-delay 60]

Example:
    python retry_failed_task.py "curl -f https://example.com/api/ping"
    python retry_failed_task.py "python3 upload.py" --max-retries 8 --base-delay 3
"""

import argparse
import shlex
import subprocess
import sys
import time
from datetime import datetime


def retry_task(command: str, max_retries: int, base_delay: int, max_delay: int) -> int:
    attempt = 0

    while attempt < max_retries:
        attempt += 1
        timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        print(f"[{timestamp}] Attempt {attempt}/{max_retries}: {command}")

        try:
            result = subprocess.run(shlex.split(command), capture_output=True, text=True)
        except FileNotFoundError:
            print(f"Error: command not found: {command}")
            sys.exit(1)

        if result.stdout:
            print(result.stdout.strip())

        if result.returncode == 0:
            print(f"\nSuccess on attempt {attempt}.")
            return 0

        print(f"Attempt {attempt} failed with exit code {result.returncode}.")
        if result.stderr:
            print(result.stderr.strip())

        if attempt >= max_retries:
            break

        delay = min(base_delay * (2 ** (attempt - 1)), max_delay)
        print(f"Retrying in {delay} second(s)...\n")
        time.sleep(delay)

    print(f"\nGiving up after {max_retries} failed attempt(s).")
    return 1


def main():
    parser = argparse.ArgumentParser(description="Run a command and retry it (with exponential backoff) if it fails.")
    parser.add_argument("command", help="Command to run (quote it if it has spaces/args)")
    parser.add_argument("--max-retries", type=int, default=5, help="Maximum number of attempts (default: 5)")
    parser.add_argument("--base-delay", type=int, default=2, help="Initial retry delay in seconds (default: 2)")
    parser.add_argument("--max-delay", type=int, default=60, help="Maximum retry delay in seconds (default: 60)")
    args = parser.parse_args()

    exit_code = retry_task(args.command, args.max_retries, args.base_delay, args.max_delay)
    sys.exit(exit_code)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

Each attempt runs the command with `subprocess.run()` and checks `returncode`. On success (`0`), the script returns immediately with exit code `0`. On failure, it computes an exponential backoff delay — `base_delay * 2^(attempt-1)`, capped at `max_delay` — and sleeps before the next attempt. After `max_retries` failed attempts, it gives up and exits with code `1`, so the script's own exit status reflects overall success/failure and can be checked by whatever calls it.

```bash
# Retry a flaky API ping up to 5 times (default), starting at a 2s delay
python retry_failed_task.py "curl -f https://example.com/api/ping"

# Retry an upload up to 8 times, starting at a 3s delay, capped at 60s
python retry_failed_task.py "python3 upload.py" --max-retries 8 --base-delay 3

# A short, aggressive retry window for something that should resolve fast
python retry_failed_task.py "python3 quick_check.py" --max-retries 3 --base-delay 1 --max-delay 5
```

#### Sample Output

```
$ python retry_failed_task.py "curl -f https://example.com/api/ping" --max-retries 3 --base-delay 2
[2026-07-17 09:00:00] Attempt 1/3: curl -f https://example.com/api/ping
Attempt 1 failed with exit code 22.
curl: (22) The requested URL returned error: 503
Retrying in 2 second(s)...

[2026-07-17 09:00:02] Attempt 2/3: curl -f https://example.com/api/ping
{"status": "ok"}

Success on attempt 2.
```

#### Variations / Flags

- Add a `--jitter` flag that adds a small random offset to each delay, reducing the chance of many retrying clients hitting a recovering service at the exact same moment ("thundering herd").
- Add a `--retry-on-codes 1,22` flag to only retry on specific exit codes, leaving others (like a clear configuration error) to fail immediately without wasting the retry budget.
- Combine with 9.7 (`queue_background_jobs.py`) to retry each queued job individually rather than the whole queue.

#### Common Pitfalls

- **Retries a symptom, not a cause.** A deterministic failure (bad credentials, a missing file) will fail identically on every attempt — this script has no way to distinguish "will succeed if I wait" from "will never succeed," so it burns the full retry budget either way.
- **No cap on total wall-clock time**, only on attempt count — with a high `--max-retries` and `--max-delay`, the effective worst-case wait can stretch to minutes without an obvious warning up front.
- **Exit-code-only success check.** A command that returns `0` but produced empty or wrong output still counts as success; validate output yourself if that distinction matters.
- **Not for long-running services.** This is explicitly for one-shot tasks; a process that's supposed to stay running (a server, a daemon) needs a supervisor/restart tool instead, not a retry-until-success wrapper.

### 9.7 Queue Background Jobs

#### Use Case

A sysadmin has a batch of one-off tasks — several file conversions, a handful of uploads — that must run one at a time, in order, without overlapping or racing each other. `queue_background_jobs.py` implements a minimal persistent job queue backed by a local JSON file: add jobs from any terminal, then run a worker that drains the queue sequentially with full logging.

#### Prerequisites

- Python 3.7+ (standard library only)
- OS scope: cross-platform (Linux, macOS, Windows)
- Writes two files to the current directory: `job_queue.json` (the queue) and `job_queue.log` (the log)

```bash
# Nothing to install — pure standard library
python3 --version
```

#### Cautions

There is no file locking — if two `worker` processes run against the same `job_queue.json` at the same time, both can read the same "pending" job and process it twice; only run one worker per queue file at a time.

#### Script

```python
#!/usr/bin/env python3
"""
queue_background_jobs.py
------------------------------
A simple background job queue: add commands to a persistent queue (a local
JSON file), then run a worker that processes them one at a time in the
background, in order, logging results. Useful for queuing up several
one-off tasks (e.g. file conversions, uploads) to run sequentially without
overlapping.

Usage:
    python queue_background_jobs.py add "<command>"
    python queue_background_jobs.py list
    python queue_background_jobs.py worker [--once]

Example:
    python queue_background_jobs.py add "python3 convert.py file1.csv"
    python queue_background_jobs.py add "python3 convert.py file2.csv"
    python queue_background_jobs.py list
    python queue_background_jobs.py worker
    # Processes all queued jobs one by one, then exits.
    # Use without --once to keep watching for new jobs indefinitely.

Queue file: job_queue.json (created in the current directory)
Log file:   job_queue.log  (created in the current directory)
"""

import argparse
import json
import os
import shlex
import subprocess
import sys
import time
from datetime import datetime

QUEUE_FILE = "job_queue.json"
LOG_FILE = "job_queue.log"


def load_queue() -> list:
    if not os.path.exists(QUEUE_FILE):
        return []
    with open(QUEUE_FILE, "r", encoding="utf-8") as f:
        return json.load(f)


def save_queue(queue: list) -> None:
    with open(QUEUE_FILE, "w", encoding="utf-8") as f:
        json.dump(queue, f, indent=2)


def log(message: str) -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    line = f"[{timestamp}] {message}"
    print(line)
    with open(LOG_FILE, "a", encoding="utf-8") as f:
        f.write(line + "\n")


def add_job(command: str) -> None:
    queue = load_queue()
    job = {
        "id": len(queue) + 1 if not queue else max(j["id"] for j in queue) + 1,
        "command": command,
        "status": "pending",
        "added_at": datetime.now().isoformat(timespec="seconds"),
    }
    queue.append(job)
    save_queue(queue)
    print(f"Job #{job['id']} added to queue: {command}")


def list_jobs() -> None:
    queue = load_queue()
    if not queue:
        print("Queue is empty.")
        return

    print(f"Jobs in queue ({len(queue)}):\n")
    for job in queue:
        print(f"#{job['id']:<4} [{job['status']:<10}] {job['command']}")


def process_job(job: dict) -> None:
    log(f"Starting job #{job['id']}: {job['command']}")

    try:
        result = subprocess.run(shlex.split(job["command"]), capture_output=True, text=True)
        job["status"] = "completed" if result.returncode == 0 else "failed"
        job["exit_code"] = result.returncode

        if result.stdout:
            log(f"Job #{job['id']} output: {result.stdout.strip()}")
        if result.returncode != 0 and result.stderr:
            log(f"Job #{job['id']} error: {result.stderr.strip()}")

    except FileNotFoundError:
        job["status"] = "failed"
        job["exit_code"] = -1
        log(f"Job #{job['id']} failed: command not found")

    log(f"Job #{job['id']} finished with status: {job['status']}")


def run_worker(run_once: bool, poll_interval: int = 5) -> None:
    print(f"Worker started. Log file: {LOG_FILE}")
    if run_once:
        print("Mode: process current queue once, then exit.\n")
    else:
        print(f"Mode: continuously poll for new jobs every {poll_interval}s. Press Ctrl+C to stop.\n")

    try:
        while True:
            queue = load_queue()
            pending = [j for j in queue if j["status"] == "pending"]

            if not pending:
                if run_once:
                    print("No pending jobs. Exiting.")
                    return
                time.sleep(poll_interval)
                continue

            for job in pending:
                process_job(job)
                save_queue(queue)

            if run_once:
                print("\nAll queued jobs processed.")
                return

    except KeyboardInterrupt:
        print("\nWorker stopped.")


def main():
    parser = argparse.ArgumentParser(description="Queue and process background jobs sequentially.")
    subparsers = parser.add_subparsers(dest="action", required=True)

    add_parser = subparsers.add_parser("add", help="Add a command to the job queue")
    add_parser.add_argument("command", help="Command to queue (quote it if it has spaces/args)")

    subparsers.add_parser("list", help="List all jobs in the queue")

    worker_parser = subparsers.add_parser("worker", help="Process jobs from the queue")
    worker_parser.add_argument("--once", action="store_true", help="Process current queue then exit (default: keep polling)")
    worker_parser.add_argument("--interval", type=int, default=5, help="Poll interval in seconds when not using --once (default: 5)")

    args = parser.parse_args()

    if args.action == "add":
        add_job(args.command)
    elif args.action == "list":
        list_jobs()
    elif args.action == "worker":
        run_worker(args.once, args.interval)


if __name__ == "__main__":
    main()
```

#### How It Works & How to Run

`add` appends a job dict (auto-incrementing `id`, the raw command, a `pending` status, and a timestamp) to `job_queue.json`. `list` reads the file and prints each job's id, status, and command. `worker` loads all `pending` jobs and processes them one at a time via `process_job()`, which runs the command, records `completed` or `failed` plus the exit code, and writes everything to `job_queue.log`. With `--once`, the worker drains the current queue and exits; without it, the worker polls indefinitely for newly added jobs every `--interval` seconds (default 5).

```bash
# Queue up two jobs
python queue_background_jobs.py add "python3 convert.py file1.csv"
python queue_background_jobs.py add "python3 convert.py file2.csv"

# See what's queued
python queue_background_jobs.py list

# Drain the current queue once, then exit
python queue_background_jobs.py worker --once

# Or run a long-lived worker that keeps watching for new jobs
python queue_background_jobs.py worker --interval 10
```

#### Sample Output

```
$ python queue_background_jobs.py add "echo job1"
Job #1 added to queue: echo job1

$ python queue_background_jobs.py add "echo job2"
Job #2 added to queue: echo job2

$ python queue_background_jobs.py list
Jobs in queue (2):

#1    [pending   ] echo job1
#2    [pending   ] echo job2

$ python queue_background_jobs.py worker --once
Worker started. Log file: job_queue.log
Mode: process current queue once, then exit.

[2026-07-17 09:00:00] Starting job #1: echo job1
[2026-07-17 09:00:00] Job #1 output: job1
[2026-07-17 09:00:00] Job #1 finished with status: completed
[2026-07-17 09:00:00] Starting job #2: echo job2
[2026-07-17 09:00:00] Job #2 output: job2
[2026-07-17 09:00:00] Job #2 finished with status: completed

All queued jobs processed.
```

#### Variations / Flags

- Add a `retry` action that resets `failed` jobs back to `pending` so `worker` picks them up again — pairs naturally with 9.6's backoff logic for the retry itself.
- Add a `--priority N` flag to `add` and sort `pending` jobs by priority before processing.
- Swap the JSON file for a SQLite database if job volume grows large enough that repeatedly rewriting the whole file becomes a bottleneck.

#### Common Pitfalls

- **No file locking.** Running more than one `worker` process against the same queue file at once can process the same job twice, or corrupt `job_queue.json` if two processes save simultaneously.
- **Relative file paths.** `job_queue.json` and `job_queue.log` are created in whatever directory you happen to run the script from — running `add` and `worker` from different directories silently creates two unrelated queues.
- **A stuck job blocks everything behind it.** Since jobs run strictly in order and synchronously, one job that hangs (e.g. waiting on network input) stalls the entire rest of the queue.
- **Failed jobs stay `failed` forever** unless something explicitly resets their status — there's no automatic retry built in.

### 9.8 Distributed Task Scheduling (Celery/RQ)

#### Use Case

Everything in this chapter so far runs on a single machine. Once job volume grows — thousands of report generations, image conversions, or emails a day — a single-host queue (9.7) stops being enough: one slow job blocks everything behind it, and there's no way to spread the load across multiple machines. Distributed task queues solve this by handing jobs to a message broker (Redis or RabbitMQ), which any number of independent worker processes — on any number of machines — can pull from concurrently. This section covers the two most common Python options: **Celery**, the mature, feature-rich standard, and **RQ (Redis Queue)**, a much lighter alternative for teams that don't need Celery's full complexity.

#### Prerequisites

- Python 3.7+
- A running message broker: **Redis** (used by both examples below) — RabbitMQ is also supported by Celery but not shown here
- Third-party packages: `celery` and `redis` for the Celery example; `rq` and `redis` for the RQ example
- OS scope: cross-platform (Linux, macOS, Windows), though Redis itself is usually run via Docker or a Linux/macOS install; on Windows, Redis is typically run inside WSL or Docker

```bash
# Install Python dependencies
pip install celery redis   # for the Celery example
pip install rq redis       # for the RQ example

# Run Redis locally via Docker (simplest cross-platform option)
docker run -d --name redis -p 6379:6379 redis:7

# Or on Linux via package manager
sudo apt install redis-server && sudo systemctl start redis-server
```

#### Cautions

A worker that dies mid-task can leave a job in an ambiguous state depending on the broker's acknowledgment settings — always design tasks to be idempotent (safe to run twice) rather than assuming "exactly once" delivery, and never point a worker at a production broker until you've tested failure/retry behavior against a disposable Redis instance first.

---

#### Script: Celery (`celery_tasks.py` + `celery_producer.py`)

```python
#!/usr/bin/env python3
"""
celery_tasks.py
---------------
Defines the Celery application and its tasks. This is the module that
worker processes import and execute against. Run a worker with:

    celery -A celery_tasks worker --loglevel=info

Requires: pip install celery redis
Requires a running Redis instance (used here as both broker and result backend).
"""

from celery import Celery

app = Celery(
    "celery_tasks",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1",
)

app.conf.update(
    task_acks_late=True,          # only ack after the task finishes (safer on worker crash)
    worker_prefetch_multiplier=1, # avoid one worker hoarding many jobs at once
    task_time_limit=300,          # hard kill a task after 5 minutes
)


@app.task(bind=True, max_retries=3, default_retry_delay=10)
def convert_file(self, filename: str) -> str:
    """Example task: pretend to convert a file, retrying on failure."""
    try:
        # Replace this block with real work (image conversion, report generation, etc.)
        if not filename:
            raise ValueError("filename must not be empty")
        return f"Converted: {filename}"
    except Exception as exc:
        raise self.retry(exc=exc)


@app.task
def send_report(recipient: str, report_name: str) -> str:
    """Example task: pretend to send a report to a recipient."""
    return f"Sent '{report_name}' to {recipient}"
```

```python
#!/usr/bin/env python3
"""
celery_producer.py
-------------------
Submits jobs to the Celery queue defined in celery_tasks.py. This script
does NOT process jobs itself — it just enqueues them. A separate worker
process (started with `celery -A celery_tasks worker`) picks them up.

Usage:
    python celery_producer.py convert <filename>
    python celery_producer.py report <recipient> <report_name>
    python celery_producer.py status <task_id>

Example:
    python celery_producer.py convert file1.csv
```

##### How It Works & How to Run (Celery)

`celery_tasks.py` defines a Celery `app` pointed at Redis as both the broker (where jobs wait) and result backend (where results are stored). `@app.task` turns a plain function into a job type workers know how to run; `convert_file` demonstrates automatic retries via `self.retry()`. `celery_producer.py` never runs task code itself — calling `.delay(...)` just serializes the call and pushes it onto the Redis queue; a separate `celery worker` process, watching that queue, actually executes it.

```bash
# Terminal 1: start a worker (this process executes jobs)

python celery_producer.py convert file1.csv
```

#### Script: RQ / Redis Queue (`rq_tasks.py` + `rq_producer.py`)

```python
#!/usr/bin/env python3
"""
rq_tasks.py
-----------
Plain functions used as RQ jobs. Unlike Celery, RQ needs no special
decorator or app object — any importable function can be enqueued.

Requires: pip install rq redis
"""


def convert_file(filename: str) -> str:
    if not filename:
        raise ValueError("filename must not be empty")
    return f"Converted: {filename}"


def send_report(recipient: str, report_name: str) -> str:
    return f"Sent '{report_name}' to {recipient}"
```

```python
#!/usr/bin/env python3
"""
rq_producer.py
--------------
Submits jobs to an RQ queue backed by Redis. Start a worker separately with:

    rq worker default

Usage:
    python rq_producer.py convert <filename>
    python rq_producer.py report <recipient> <report_name>
    python rq_producer.py status <job_id>

Example:
    python rq_producer.py convert file1.csv
    python rq_producer.py status 3f2c1a9e-...
"""

import argparse

from redis import Redis
from rq import Queue
from rq.job import Job

import rq_tasks

redis_conn = Redis(host="localhost", port=6379, db=0)
queue = Queue("default", connection=redis_conn)


def submit_convert(filename: str) -> None:
    job = queue.enqueue(rq_tasks.convert_file, filename, retry=3)
    print(f"Submitted convert_file job. Job ID: {job.id}")


def submit_report(recipient: str, report_name: str) -> None:
    job = queue.enqueue(rq_tasks.send_report, recipient, report_name)
    print(f"Submitted send_report job. Job ID: {job.id}")


def check_status(job_id: str) -> None:
    job = Job.fetch(job_id, connection=redis_conn)
    print(f"Job {job_id} status: {job.get_status()}")
    if job.is_finished:
        print(f"Result: {job.result}")
    elif job.is_failed:
        print("Job failed. See worker logs for the traceback.")


def main():
    parser = argparse.ArgumentParser(description="Submit or check RQ jobs.")
    subparsers = parser.add_subparsers(dest="action", required=True)

    convert_parser = subparsers.add_parser("convert", help="Submit a convert_file job")
    convert_parser.add_argument("filename", help="File to (pretend to) convert")

    report_parser = subparsers.add_parser("report", help="Submit a send_report job")
    report_parser.add_argument("recipient", help="Recipient email/name")
    report_parser.add_argument("report_name", help="Name of the report")

    status_parser = subparsers.add_parser("status", help="Check a job's status by ID")
    status_parser.add_argument("job_id", help="RQ job ID returned at submit time")

    args = parser.parse_args()

    if args.action == "convert":
        submit_convert(args.filename)
    elif args.action == "report":
        submit_report(args.recipient, args.report_name)
    elif args.action == "status":
        check_status(args.job_id)


if __name__ == "__main__":
    main()
```

##### How It Works & How to Run (RQ)

RQ needs no decorators or app object — `queue.enqueue(function, *args)` serializes a reference to any importable function plus its arguments and pushes it onto a Redis-backed list named `"default"`. A separate `rq worker default` process pops jobs off that list and runs them. `Job.fetch(job_id, connection=...)` retrieves a job's current status (`queued`, `started`, `finished`, `failed`) and result at any time, from any process that can reach the same Redis instance.

```bash
# Terminal 1: start a worker
rq worker default

# Terminal 2: submit jobs
python rq_producer.py convert file1.csv
python rq_producer.py report "ops-team@example.com" "Monthly Usage"

# Check a submitted job's status/result
python rq_producer.py status 3f2c1a9e-7b44-4e21-9c2a-1a2b3c4d5e6f
```

#### Variations / Flags

- Add Celery Beat (`celery -A celery_tasks beat`) for recurring/scheduled tasks distributed across workers — the distributed equivalent of 9.1/9.2 combined.
- Use RQ Scheduler (`rq-scheduler` package) for delayed/recurring RQ jobs, since RQ's core doesn't include scheduling by itself.
- Run multiple workers (`celery -A celery_tasks worker --concurrency=4` or several `rq worker default` processes) to parallelize job processing across CPU cores or machines.
- Add a dead-letter queue pattern: route jobs that exhaust their retries to a separate queue for manual inspection instead of silently dropping them.

#### Common Pitfalls

- **Broker is a single point of failure.** If Redis/RabbitMQ goes down, both producers and workers lose the ability to enqueue or process jobs — plan for broker persistence/replication in production, not just the worker layer.
- **Task code must be importable by workers.** Both examples require the worker process to have access to the same task module (`celery_tasks.py` / `rq_tasks.py`) on its own filesystem or Python path — a common deployment mistake is updating the producer but forgetting to redeploy the worker.
- **Serialization limits.** Arguments and return values must be serializable (JSON by default in Celery, pickle by default in RQ) — passing complex objects like open file handles or database connections will fail.
- **At-least-once delivery, not exactly-once.** A worker crash after finishing a task but before acknowledging it can cause the same job to run twice — write task functions to be idempotent (e.g. check-then-act, or use unique keys) rather than assuming single execution.
- **No built-in scheduling in vanilla RQ.** Unlike Celery (which has Beat), plain RQ only handles "run this now" — recurring jobs need the separate `rq-scheduler` package.
- **This section could not be executed against a live broker in this environment** (no network access to install `celery`/`rq`/Redis or run a broker) — the code was validated for syntax and against documented APIs only. Run your own smoke test against a disposable Redis instance before trusting this in production.
