## Chapter 7: Resource Management

Every process a system administrator runs — a web server, a batch job, a data pipeline, a background daemon — competes for the same finite pool of CPU cycles, memory, disk space, and execution time. Left unchecked, a single runaway script can starve every other process on the box, fill a disk partition, or hang indefinitely while consuming resources that other jobs need. Resource management is the discipline of setting boundaries around what a process is *allowed* to do, watching what it *actually* does, and reacting automatically when something goes wrong.

This chapter walks through eight practical, script-driven techniques for managing resources on a Linux, macOS, or Windows host. Each technique is implemented as a small, self-contained Python utility that you can run directly from the command line, wire into a supervisor, or schedule with cron or Task Scheduler. None of them require a full monitoring stack or paid tooling — just Python, and in a few cases the cross-platform `psutil` library.

The chapter is organized into two halves. The first half (7.1–7.5) covers **enforcement**: how to cap the CPU time, memory, disk space, execution time, and scheduling priority a process is permitted to consume. The second half (7.6–7.8) covers **observability and recovery**: how to watch system resources in real time, detect when something is approaching exhaustion, and automatically restart a process that has crashed or hung.

Together, these tools form a lightweight but effective safety net. They won't replace a production-grade orchestrator like systemd, Kubernetes, or a proper APM solution — but they will get you most of the way there for scripts, cron jobs, and small services, and they'll teach you the underlying OS primitives (`rlimit`, `nice`, process signals) that those bigger systems are built on top of.

**A note on platform scope:** Several of these scripts rely on the POSIX `resource` module (`RLIMIT_CPU`, `RLIMIT_AS`), which is **not available on Windows**. Where this applies, the script detects the platform at runtime and either falls back to a softer alternative or clearly warns you that the hard limit could not be applied. Read the "Common pitfalls" section of each script's file before relying on it in production, especially across mixed-OS environments.

### 7.1 Set CPU Usage Limits

##### Use case

A sysadmin needs to run an untrusted or experimental script (a user-submitted job, a benchmark, a recursive algorithm with unknown complexity) and wants a hard guarantee that it cannot burn more than a fixed amount of **CPU time**, even if it enters an infinite loop or a runaway computation.

##### Prerequisites

Standard library only on Linux/macOS — no `pip install` required. Not supported on Windows (see cautions below).

```bash
# No installation needed on Linux/macOS.
# Just make sure you're running Python 3:
python3 --version
```

##### Cautions

`RLIMIT_CPU` measures actual **CPU time consumed**, not wall-clock time — a process that sleeps or blocks on I/O for an hour but only uses 2 seconds of CPU will not be killed. On Windows this limit cannot be enforced at all; the script detects this and runs the command unrestricted instead, so don't rely on this script for hard enforcement on Windows hosts.

##### Script

```python
#!/usr/bin/env python3
"""
set_cpu_limit.py
----------------------
Runs a command/process while limiting how much CPU time it can consume,
using the 'resource' module (RLIMIT_CPU) on Linux/macOS.

On Windows, hard CPU-time limits are not supported by the OS in the same
way; this script will inform you and fall back to monitoring instead.

Usage:
    python set_cpu_limit.py <cpu_seconds> -- <command> [args...]

Example:
    python set_cpu_limit.py 5 -- python3 -c "while True: pass"
    # The process will be killed by the OS once it uses 5 seconds of CPU time.
"""

import argparse
import platform
import subprocess
import sys


def set_cpu_limit_and_run(cpu_seconds: int, command: list) -> int:
    if platform.system() == "Windows":
        print("Warning: hard CPU-time limits (RLIMIT_CPU) are not supported on Windows.")
        print("Running the command WITHOUT a CPU limit. Consider using Job Objects for this on Windows.")
        result = subprocess.run(command)
        return result.returncode

    import resource

    def limit_cpu():
        # Soft limit == hard limit here; process receives SIGXCPU then SIGKILL.
        resource.setrlimit(resource.RLIMIT_CPU, (cpu_seconds, cpu_seconds))

    print(f"Running command with a CPU time limit of {cpu_seconds} second(s): {' '.join(command)}")

    try:
        result = subprocess.run(command, preexec_fn=limit_cpu)
        return result.returncode
    except FileNotFoundError:
        print(f"Error: command not found: {command[0]}")
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(
        description="Run a command with a CPU time limit.",
        usage="set_cpu_limit.py <cpu_seconds> -- <command> [args...]",
    )
    parser.add_argument("cpu_seconds", type=int, help="Maximum CPU time in seconds before the process is killed")
    parser.add_argument("command", nargs=argparse.REMAINDER, help="Command to run, after '--'")
    args = parser.parse_args()

    command = args.command
    if command and command[0] == "--":
        command = command[1:]

    if not command:
        print("Error: no command provided. Usage: set_cpu_limit.py <cpu_seconds> -- <command> [args...]")
        sys.exit(1)

    exit_code = set_cpu_limit_and_run(args.cpu_seconds, command)

    print(f"\nCommand finished with exit code: {exit_code}")
    if exit_code < 0:
        print(f"Process was terminated by signal {-exit_code} (likely due to exceeding the CPU limit).")


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script uses `resource.setrlimit(resource.RLIMIT_CPU, ...)` inside a `preexec_fn`, which runs in the child process right after `fork()` but before the target command executes. Once the child exceeds the CPU-second limit, the kernel sends it `SIGXCPU`, and if it doesn't exit, `SIGKILL` shortly after.

```bash
# Basic usage: limit a command to 5 seconds of CPU time
python3 set_cpu_limit.py 5 -- python3 -c "while True: pass"

# Limit a real workload, e.g. a compression job, to 30 CPU-seconds
python3 set_cpu_limit.py 30 -- gzip -9 large_file.tar

# Combine with a shell command via bash -c
python3 set_cpu_limit.py 10 -- bash -c "for i in {1..1000000}; do :; done"
```

##### Sample output

```text
$ python3 set_cpu_limit.py 5 -- python3 -c "while True: pass"
Running command with a CPU time limit of 5 second(s): python3 -c while True: pass

Command finished with exit code: -24
Process was terminated by signal 24 (likely due to exceeding the CPU limit).
```

(Signal 24 is `SIGXCPU` on most Linux systems.)

##### Variations / flags

- Wrap this script with `restart_failed_process.py` (7.8) to automatically relaunch a job that gets killed for exceeding its CPU budget.
- Add a `--soft-limit` argument to send `SIGXCPU` earlier than the hard kill, giving the process a chance to save state and exit cleanly.
- Schedule with cron for nightly batch jobs where you want a hard ceiling: `0 2 * * * /usr/bin/python3 /path/set_cpu_limit.py 3600 -- /path/nightly_job.sh`.

##### Common pitfalls

- **Windows has no equivalent** — the script silently runs unrestricted there; don't assume the limit is being enforced just because the script ran without error.
- **CPU time ≠ wall-clock time.** A process waiting on a slow network call can run for hours without tripping this limit — use `limit_execution_time.py` (7.4) if you need a wall-clock cutoff instead.
- **`preexec_fn` is not fork-safe with threads.** If the parent Python process is multi-threaded, mixing `preexec_fn` with threads can deadlock; this is a known CPython limitation, not a bug in this script.
- Multi-process workloads (e.g., a command that forks children) may not have the limit propagate the way you expect — `RLIMIT_CPU` applies per-process, not per-process-tree.

### 7.2 Limit Memory Usage

##### Use case

A sysadmin needs to run a memory-hungry or untrusted script (data processing job, third-party tool, user-submitted code) without risking it consuming all available RAM and triggering the kernel's OOM killer, which can take down unrelated processes on the same host.

##### Prerequisites

Standard library only on Linux/macOS — no `pip install` required. Not enforced on Windows.

```bash
# No installation needed on Linux/macOS.
python3 --version
```

##### Cautions

On **macOS**, `RLIMIT_AS` is accepted by the kernel but is known to be poorly enforced in many versions — don't treat this as a hard guarantee on Mac. On Windows the limit is skipped entirely and the script runs the command unrestricted. Setting the limit too low can cause the interpreter or dynamic linker itself to fail before your target program even starts.

##### Script

```python
#!/usr/bin/env python3
"""
limit_memory_usage.py
----------------------------
Runs a command/process while limiting how much memory (RAM) it can use,
using the 'resource' module (RLIMIT_AS) on Linux/macOS.

On Windows, hard memory limits via RLIMIT are not supported; this script
will inform you and fall back to running without a hard limit.

Usage:
    python limit_memory_usage.py <memory_mb> -- <command> [args...]

Example:
    python limit_memory_usage.py 200 -- python3 -c "x = 'a' * (500 * 1024 * 1024)"
    # The process will fail to allocate memory once it exceeds 200 MB.
"""

import argparse
import platform
import subprocess
import sys


def limit_memory_and_run(memory_mb: int, command: list) -> int:
    if platform.system() == "Windows":
        print("Warning: hard memory limits (RLIMIT_AS) are not supported on Windows.")
        print("Running the command WITHOUT a memory limit. Consider using Job Objects for this on Windows.")
        result = subprocess.run(command)
        return result.returncode

    import resource

    memory_bytes = memory_mb * 1024 * 1024

    def limit_memory():
        resource.setrlimit(resource.RLIMIT_AS, (memory_bytes, memory_bytes))

    print(f"Running command with a memory limit of {memory_mb} MB: {' '.join(command)}")

    try:
        result = subprocess.run(command, preexec_fn=limit_memory)
        return result.returncode
    except FileNotFoundError:
        print(f"Error: command not found: {command[0]}")
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(
        description="Run a command with a memory usage limit.",
        usage="limit_memory_usage.py <memory_mb> -- <command> [args...]",
    )
    parser.add_argument("memory_mb", type=int, help="Maximum memory in megabytes")
    parser.add_argument("command", nargs=argparse.REMAINDER, help="Command to run, after '--'")
    args = parser.parse_args()

    command = args.command
    if command and command[0] == "--":
        command = command[1:]

    if not command:
        print("Error: no command provided. Usage: limit_memory_usage.py <memory_mb> -- <command> [args...]")
        sys.exit(1)

    exit_code = limit_memory_and_run(args.memory_mb, command)

    print(f"\nCommand finished with exit code: {exit_code}")
    if exit_code != 0:
        print("Non-zero exit code may indicate the process hit the memory limit (e.g. MemoryError, SIGKILL, SIGSEGV).")


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`resource.setrlimit(resource.RLIMIT_AS, ...)` caps the total **address space** (virtual memory) the process may map, applied via `preexec_fn` in the child before the target command runs. Once the process tries to allocate beyond that ceiling, the allocation fails — typically surfacing as a `MemoryError` in Python programs or a segfault/abort in native ones.

```bash
# Basic usage: limit a Python one-liner to 200 MB
python3 limit_memory_usage.py 200 -- python3 -c "x = 'a' * (500 * 1024 * 1024)"

# Limit a real data-processing script to 1 GB
python3 limit_memory_usage.py 1024 -- python3 process_large_csv.py

# Limit a Node.js process to 512 MB
python3 limit_memory_usage.py 512 -- node server.js
```

##### Sample output

```text
$ python3 limit_memory_usage.py 200 -- python3 -c "x = 'a' * (500 * 1024 * 1024)"
Running command with a memory limit of 200 MB: python3 -c x = 'a' * (500 * 1024 * 1024)
Traceback (most recent call last):
  File "<string>", line 1, in <module>
MemoryError

Command finished with exit code: 1
Non-zero exit code may indicate the process hit the memory limit (e.g. MemoryError, SIGKILL, SIGSEGV).
```

##### Variations / flags

- Add a `--swap-ok` flag to switch between `RLIMIT_AS` (virtual memory, stricter) and `RLIMIT_RSS` (resident memory, looser and less reliably enforced on modern Linux kernels).
- Pair with `detect_resource_exhaustion.py` (7.7) to alert *before* a process is hard-killed, giving you a chance to intervene.
- On Linux, consider offering a cgroups-based variant (`systemd-run --scope -p MemoryMax=200M ...`) as a more robust alternative for production use.

##### Common pitfalls

- **Not enforced on Windows** — the script warns and proceeds without any limit; don't deploy this as your only safeguard on Windows hosts.
- **Unreliable on macOS** — `RLIMIT_AS` is frequently ignored or only partially enforced depending on the macOS/Python build.
- **Interpreter startup overhead counts against the limit.** Setting the limit too close to what the interpreter itself needs to boot (imports, shared libraries) can cause the target command to fail immediately, even before your code runs — leave headroom.
- A killed process may exit with `SIGKILL` (no traceback at all) rather than a clean `MemoryError`, depending on how abruptly the allocation failed — don't assume you'll always get a readable Python exception.

### 7.3 Limit Disk Usage

##### Use case

A sysadmin manages a shared upload directory, log folder, or cache path that has no OS-level quota configured, and needs a lightweight way to be warned — or to stop a companion process — before that directory fills the disk and takes down the whole system.

##### Prerequisites

Standard library only — no `pip install` required. Works identically on Linux, macOS, and Windows.

```bash
# No installation needed on any platform.
python3 --version
```

##### Cautions

This script cannot *prevent* writes at the OS level the way a real disk quota (`setquota`, ZFS quotas, Windows FSRM) can — it only detects and reports after the fact by periodically re-scanning the directory. On very large directories (millions of files), the recursive `os.walk` size scan can itself become slow and CPU-intensive.

##### Script

```python
#!/usr/bin/env python3
"""
limit_disk_usage.py
--------------------------
Monitors a directory's total disk usage and takes action (warn or block
further writes by exiting) if it exceeds a specified limit. Since Python
cannot enforce OS-level disk quotas without root/admin tools (like
'setquota'), this script works by periodically checking directory size
and alerting/stopping a companion process if the limit is exceeded.

Usage:
    python limit_disk_usage.py <directory_path> <limit_mb> [--interval 5] [--watch]

Example:
    python limit_disk_usage.py /home/user/uploads 500
    python limit_disk_usage.py /home/user/uploads 500 --watch --interval 10
"""

import argparse
import os
import sys
import time


def get_directory_size(path: str) -> int:
    total_size = 0
    for root, _, files in os.walk(path):
        for f in files:
            full_path = os.path.join(root, f)
            try:
                total_size += os.path.getsize(full_path)
            except OSError:
                continue
    return total_size


def check_once(path: str, limit_mb: int) -> bool:
    size_bytes = get_directory_size(path)
    size_mb = size_bytes / (1024 * 1024)
    percent = (size_mb / limit_mb) * 100 if limit_mb > 0 else 0

    print(f"Directory: {path}")
    print(f"Current size: {size_mb:.2f} MB")
    print(f"Limit:        {limit_mb} MB")
    print(f"Usage:        {percent:.1f}%")

    if size_mb > limit_mb:
        print("\nStatus: LIMIT EXCEEDED")
        return False
    elif percent >= 90:
        print("\nStatus: WARNING — approaching limit (90%+)")
        return True
    else:
        print("\nStatus: OK")
        return True


def watch(path: str, limit_mb: int, interval: int) -> None:
    print(f"Watching '{path}' every {interval} second(s). Press Ctrl+C to stop.\n")
    try:
        while True:
            within_limit = check_once(path, limit_mb)
            if not within_limit:
                print("\nDisk usage limit exceeded! Consider cleaning up files.")
            print("-" * 50)
            time.sleep(interval)
    except KeyboardInterrupt:
        print("\nStopped watching.")


def main():
    parser = argparse.ArgumentParser(description="Monitor and limit disk usage for a directory.")
    parser.add_argument("directory", help="Directory to monitor")
    parser.add_argument("limit_mb", type=int, help="Maximum allowed size in megabytes")
    parser.add_argument("--interval", type=int, default=5, help="Check interval in seconds when using --watch (default: 5)")
    parser.add_argument("--watch", action="store_true", help="Continuously monitor instead of checking once")
    args = parser.parse_args()

    if not os.path.isdir(args.directory):
        print(f"Error: '{args.directory}' is not a valid directory.")
        sys.exit(1)

    if args.watch:
        watch(args.directory, args.limit_mb, args.interval)
    else:
        within_limit = check_once(args.directory, args.limit_mb)
        sys.exit(0 if within_limit else 1)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script recursively walks the target directory with `os.walk`, summing `os.path.getsize()` for every file to compute total size, then compares that against your limit. In `--watch` mode it re-scans on a fixed interval; in single-check mode it exits with code `0` (within limit) or `1` (exceeded), making it easy to use in scripts and CI checks.

```bash
# One-time check
python3 limit_disk_usage.py /home/user/uploads 500

# Continuous monitoring, checking every 10 seconds
python3 limit_disk_usage.py /home/user/uploads 500 --watch --interval 10

# Use the exit code in a shell conditional
python3 limit_disk_usage.py /var/log/myapp 2048 && echo "OK" || echo "Over quota!"
```

##### Sample output

```text
$ python3 limit_disk_usage.py /home/user/uploads 500
Directory: /home/user/uploads
Current size: 462.18 MB
Limit:        500 MB
Usage:        92.4%

Status: WARNING — approaching limit (90%+)
```

```text
$ python3 limit_disk_usage.py /home/user/uploads 500 --watch --interval 10
Watching '/home/user/uploads' every 10 second(s). Press Ctrl+C to stop.

Directory: /home/user/uploads
Current size: 512.77 MB
Limit:        500 MB
Usage:        102.6%

Status: LIMIT EXCEEDED

Disk usage limit exceeded! Consider cleaning up files.
--------------------------------------------------
```

##### Variations / flags

- Add a `--delete-oldest` flag that automatically removes the oldest files once the limit is exceeded (a simple log-rotation strategy).
- Add a `--exclude` pattern argument to skip certain subdirectories (e.g., `.git`, `node_modules`) from the size calculation.
- Schedule with cron for periodic checks without needing `--watch` running continuously: `*/15 * * * * /usr/bin/python3 /path/limit_disk_usage.py /data/uploads 5000`.

##### Common pitfalls

- **This is detection, not prevention.** Nothing stops a write from happening between scan intervals — for true enforcement, use OS-level quotas (`setquota` on Linux, FSRM on Windows Server, ZFS/APFS quotas).
- **Symlinks and hardlinks can distort totals.** `os.walk` by default doesn't follow symlinked directories, but hardlinked files will be double-counted as if they were separate files.
- **Permission errors are silently skipped** — files the script can't read are excluded from the size total via the `except OSError: continue`, which can make usage look lower than it really is if permissions are inconsistent.
- Large directories (many files) make each scan slow; don't set `--interval` shorter than the time a single scan takes, or scans will pile up.


### 7.4 Limit Execution Time

##### Use case

A sysadmin needs to run a command that *should* finish quickly but occasionally hangs — a network call, a database query, a third-party CLI tool — and wants a guaranteed wall-clock cutoff so a stuck job doesn't block a pipeline or cron schedule indefinitely.

##### Prerequisites

Standard library only (`subprocess`, `argparse`) — no `pip install` required. Works identically on Linux, macOS, and Windows.

```bash
# No installation needed on any platform.
python3 --version
```

##### Cautions

`subprocess.run(..., timeout=...)` kills the direct child process, but **grandchild processes spawned by the command may survive** if the child doesn't propagate the termination signal to its own children (common with shell wrappers and process trees). Always verify no orphaned processes are left behind for commands that fork internally.

##### Script

```python
#!/usr/bin/env python3
"""
limit_execution_time.py
-----------------------------
Runs a command and forcibly terminates it if it runs longer than a
specified wall-clock time limit (a timeout), regardless of platform.

Usage:
    python limit_execution_time.py <timeout_seconds> -- <command> [args...]

Example:
    python limit_execution_time.py 10 -- ping -c 100 example.com
    # The command is killed if it runs longer than 10 seconds.
"""

import argparse
import subprocess
import sys


def run_with_timeout(timeout_seconds: int, command: list) -> int:
    print(f"Running command with a {timeout_seconds}-second time limit: {' '.join(command)}\n")

    try:
        result = subprocess.run(command, timeout=timeout_seconds)
        return result.returncode
    except subprocess.TimeoutExpired:
        print(f"\nProcess exceeded the {timeout_seconds}-second time limit and was terminated.")
        return 124  # Conventional timeout exit code (matches GNU 'timeout' command)
    except FileNotFoundError:
        print(f"Error: command not found: {command[0]}")
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(
        description="Run a command with a wall-clock execution time limit.",
        usage="limit_execution_time.py <timeout_seconds> -- <command> [args...]",
    )
    parser.add_argument("timeout_seconds", type=float, help="Maximum execution time in seconds")
    parser.add_argument("command", nargs=argparse.REMAINDER, help="Command to run, after '--'")
    args = parser.parse_args()

    command = args.command
    if command and command[0] == "--":
        command = command[1:]

    if not command:
        print("Error: no command provided. Usage: limit_execution_time.py <timeout_seconds> -- <command> [args...]")
        sys.exit(1)

    exit_code = run_with_timeout(args.timeout_seconds, command)

    print(f"\nCommand finished with exit code: {exit_code}")
    sys.exit(exit_code)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script delegates directly to `subprocess.run(command, timeout=timeout_seconds)`. If the process is still running when the timeout elapses, Python raises `subprocess.TimeoutExpired`, and the script reports it, returning exit code `124` — the same convention used by GNU's `timeout` utility, making it easy to script against.

```bash
# Kill a ping after 10 seconds
python3 limit_execution_time.py 10 -- ping -c 100 example.com

# Cap a database backup job at 5 minutes (300 seconds)
python3 limit_execution_time.py 300 -- pg_dump mydb -f backup.sql

# Fractional seconds are supported
python3 limit_execution_time.py 2.5 -- sleep 5
```

##### Sample output

```text
$ python3 limit_execution_time.py 2.5 -- sleep 5
Running command with a 2.5-second time limit: sleep 5


Process exceeded the 2.5-second time limit and was terminated.

Command finished with exit code: 124
```

```text
$ python3 limit_execution_time.py 10 -- echo "done quickly"
Running command with a 10-second time limit: echo done quickly

done quickly

Command finished with exit code: 0
```

##### Variations / flags

- Add a `--signal SIGTERM|SIGKILL` flag to choose how the process is terminated (Python's default `timeout` behavior sends `SIGKILL` on POSIX after the process doesn't respond).
- Combine with `restart_failed_process.py` (7.8) by setting its own `--timeout` flag, giving you both a wall-clock cutoff *and* automatic restarts in one supervision loop.
- Wrap in a cron entry to guarantee scheduled jobs never overlap: `*/5 * * * * /usr/bin/python3 /path/limit_execution_time.py 240 -- /path/sync_job.sh`.

##### Common pitfalls

- **Child processes of the command may survive termination.** If your command is a shell script that spawns background jobs, those grandchildren can be orphaned rather than killed — check with `ps` after a timeout if this matters to you.
- **Exit code 124 is a convention, not universal.** Some tools already use 124 for their own purposes; don't assume every 124 you see elsewhere came from this script.
- On Windows, killing a process tree behaves differently than POSIX signals — `subprocess`'s timeout handling still works, but downstream child-process cleanup is less reliable than on Linux/macOS.
- Setting the timeout too aggressively for naturally slow commands (large file transfers, compilations) will cause false-positive kills — always benchmark expected runtime before choosing a limit.


### 7.5 Set Process Priority

##### Use case

A sysadmin wants a background job (backup, log compression, batch analytics) to run without slowing down interactive or latency-sensitive processes on the same machine, by lowering its CPU scheduling priority — or conversely wants to boost a critical process's priority when the system is under load.

##### Prerequisites

Requires the cross-platform `psutil` library.

```bash
pip install psutil
# or, using the project's requirements file:
pip install -r requirements.txt
```

##### Cautions

Setting a **negative** niceness value (higher priority) on Linux/macOS typically requires root/sudo — the script will exit with a permission error if you're not privileged enough. On Windows, priority is mapped to the nearest matching *priority class*, which is coarser-grained than POSIX niceness, so exact values won't translate one-to-one.

##### Script

```python
#!/usr/bin/env python3
"""
set_process_priority.py
-----------------------------
Sets the CPU scheduling priority (niceness) of a running process by PID,
or launches a new command at a given priority.

On Linux/macOS, priority ("niceness") ranges from -20 (highest priority)
to 19 (lowest priority). Setting negative values usually requires root/sudo.

On Windows, priority classes are used instead (via psutil).

Requires: pip install psutil

Usage:
    python set_process_priority.py --pid <pid> --priority <value>
    python set_process_priority.py --priority <value> -- <command> [args...]

Example:
    python set_process_priority.py --pid 1234 --priority 10
    python set_process_priority.py --priority -5 -- python3 my_script.py
"""

import argparse
import platform
import subprocess
import sys

try:
    import psutil
except ImportError:
    print("Error: the 'psutil' package is required.")
    print("Install it with: pip install -r requirements.txt")
    sys.exit(1)


def set_priority_by_pid(pid: int, priority: int) -> None:
    try:
        proc = psutil.Process(pid)
    except psutil.NoSuchProcess:
        print(f"Error: no process found with PID {pid}.")
        sys.exit(1)

    try:
        if platform.system() == "Windows":
            priority_map = {
                -20: psutil.REALTIME_PRIORITY_CLASS,
                -15: psutil.HIGH_PRIORITY_CLASS,
                -5: psutil.ABOVE_NORMAL_PRIORITY_CLASS,
                0: psutil.NORMAL_PRIORITY_CLASS,
                10: psutil.BELOW_NORMAL_PRIORITY_CLASS,
                19: psutil.IDLE_PRIORITY_CLASS,
            }
            closest = min(priority_map.keys(), key=lambda k: abs(k - priority))
            proc.nice(priority_map[closest])
            print(f"Set process {pid} ({proc.name()}) to Windows priority class matching niceness ~{closest}.")
        else:
            proc.nice(priority)
            print(f"Set process {pid} ({proc.name()}) niceness to {priority}.")
    except psutil.AccessDenied:
        print("Error: permission denied. Setting this priority may require root/administrator privileges.")
        sys.exit(1)


def launch_with_priority(priority: int, command: list) -> None:
    print(f"Launching command with priority {priority}: {' '.join(command)}")

    try:
        process = subprocess.Popen(command)
    except FileNotFoundError:
        print(f"Error: command not found: {command[0]}")
        sys.exit(1)

    try:
        proc = psutil.Process(process.pid)
        if platform.system() == "Windows":
            priority_map = {
                -20: psutil.REALTIME_PRIORITY_CLASS,
                -15: psutil.HIGH_PRIORITY_CLASS,
                -5: psutil.ABOVE_NORMAL_PRIORITY_CLASS,
                0: psutil.NORMAL_PRIORITY_CLASS,
                10: psutil.BELOW_NORMAL_PRIORITY_CLASS,
                19: psutil.IDLE_PRIORITY_CLASS,
            }
            closest = min(priority_map.keys(), key=lambda k: abs(k - priority))
            proc.nice(priority_map[closest])
        else:
            proc.nice(priority)
        print(f"Process launched with PID {process.pid} at the requested priority.")
    except psutil.AccessDenied:
        print("Warning: could not set priority (permission denied). Process is still running at default priority.")

    process.wait()
    print(f"\nProcess finished with exit code: {process.returncode}")


def main():
    parser = argparse.ArgumentParser(description="Set process priority (niceness) for a running or new process.")
    parser.add_argument("--pid", type=int, default=None, help="PID of an existing process to re-prioritize")
    parser.add_argument("--priority", type=int, required=True,
                         help="Priority/niceness value (-20 highest to 19 lowest on Linux/macOS)")
    parser.add_argument("command", nargs=argparse.REMAINDER, help="Command to launch (if not using --pid), after '--'")
    args = parser.parse_args()

    if args.pid:
        set_priority_by_pid(args.pid, args.priority)
    else:
        command = args.command
        if command and command[0] == "--":
            command = command[1:]

        if not command:
            print("Error: provide either --pid <pid> or a command to launch after '--'.")
            sys.exit(1)

        launch_with_priority(args.priority, command)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script has two modes: re-prioritize an existing process by `--pid`, or launch a brand-new command already set to the desired priority via `--priority`. Under the hood, it uses `psutil.Process.nice()`, which calls the POSIX `setpriority()` syscall on Linux/macOS, or maps to a Windows priority class on that platform.

```bash
# Lower the priority of an already-running process (be nicer to other processes)
python3 set_process_priority.py --pid 1234 --priority 10

# Raise the priority of a critical process (requires sudo on Linux/macOS)
sudo python3 set_process_priority.py --pid 1234 --priority -5

# Launch a new command directly at low priority
python3 set_process_priority.py --priority 15 -- python3 batch_report.py
```

##### Sample output

```text
$ python3 set_process_priority.py --pid 1234 --priority 10
Set process 1234 (python3) niceness to 10.
```

```text
$ python3 set_process_priority.py --priority 15 -- python3 batch_report.py
Launching command with priority 15: python3 batch_report.py
Process launched with PID 5678 at the requested priority.

Process finished with exit code: 0
```

##### Variations / flags

- Add a `--io-priority` flag using `psutil.Process.ionice()` to additionally throttle disk I/O priority alongside CPU priority on Linux.
- Add a `--children` flag to recursively apply the same priority to all child processes via `psutil.Process.children(recursive=True)`.
- Combine with `monitor_system_resources.py` (7.6) to dynamically re-nice processes when overall CPU load crosses a threshold.

##### Common pitfalls

- **`--pid 0` is treated as "no PID given."** The script checks `if args.pid:`, and since `0` is falsy in Python, passing `--pid 0` silently falls through to command-launch mode instead of targeting PID 0 — usually harmless since PID 0 isn't a normal user process target, but worth knowing if you're scripting around it.
- **Raising priority (negative values) requires elevated privileges** on Linux/macOS; running without `sudo` will raise `psutil.AccessDenied`.
- **Windows priority classes are coarse.** Six classes must cover the full -20..19 range, so nearby values (e.g., 3 vs 8) may map to the same class — don't expect fine-grained control on Windows.
- Re-prioritizing a process you don't own, or a system process, can destabilize the OS scheduler — always double check the PID before applying an aggressive priority change.

### 7.6 Monitor System Resources

##### Use case

A sysadmin needs a quick, dependency-light dashboard to check what's happening on a machine right now — CPU load, memory pressure, disk usage, network throughput — without SSHing into a full monitoring stack or opening a GUI tool.

##### Prerequisites

Requires the cross-platform `psutil` library.

```bash
pip install psutil
# or:
pip install -r requirements.txt
```

##### Cautions

The first `cpu_percent(interval=1)` call blocks for one second by design (it needs a sampling window to compute a meaningful percentage) — don't be alarmed if the script pauses briefly before printing. Network totals are cumulative since boot, not per-interval, so in `--watch` mode they will only ever increase.

##### Script

```python
#!/usr/bin/env python3
"""
monitor_system_resources.py
---------------------------------
Displays live system resource usage: CPU, memory, disk, and network,
either once or continuously.

Requires: pip install psutil

Usage:
    python monitor_system_resources.py [--watch] [--interval 2]

Example:
    python monitor_system_resources.py
    python monitor_system_resources.py --watch --interval 5
"""

import argparse
import sys
import time
from datetime import datetime

try:
    import psutil
except ImportError:
    print("Error: the 'psutil' package is required.")
    print("Install it with: pip install -r requirements.txt")
    sys.exit(1)


def bytes_to_human(n: float) -> str:
    for unit in ["B", "KB", "MB", "GB", "TB"]:
        if n < 1024:
            return f"{n:.1f} {unit}"
        n /= 1024
    return f"{n:.1f} PB"


def print_snapshot() -> None:
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    cpu_percent = psutil.cpu_percent(interval=1)
    cpu_per_core = psutil.cpu_percent(interval=None, percpu=True)
    mem = psutil.virtual_memory()
    disk = psutil.disk_usage("/")
    net = psutil.net_io_counters()

    print(f"[{timestamp}] System Resource Snapshot\n")

    print(f"CPU Usage:     {cpu_percent}%")
    print(f"CPU Per Core:  {cpu_per_core}")

    print(f"\nMemory:        {bytes_to_human(mem.used)} / {bytes_to_human(mem.total)} used ({mem.percent}%)")
    print(f"Swap:          {bytes_to_human(psutil.swap_memory().used)} / {bytes_to_human(psutil.swap_memory().total)} used")

    print(f"\nDisk (/):      {bytes_to_human(disk.used)} / {bytes_to_human(disk.total)} used ({disk.percent}%)")

    print(f"\nNetwork Sent:  {bytes_to_human(net.bytes_sent)}")
    print(f"Network Recv:  {bytes_to_human(net.bytes_recv)}")


def main():
    parser = argparse.ArgumentParser(description="Monitor system CPU, memory, disk, and network usage.")
    parser.add_argument("--watch", action="store_true", help="Continuously monitor instead of a single snapshot")
    parser.add_argument("--interval", type=int, default=2, help="Refresh interval in seconds when using --watch (default: 2)")
    args = parser.parse_args()

    if args.watch:
        print(f"Monitoring system resources every {args.interval} second(s). Press Ctrl+C to stop.\n")
        try:
            while True:
                print_snapshot()
                print("=" * 60)
                time.sleep(max(0, args.interval - 1))  # cpu_percent(interval=1) already waits 1s
        except KeyboardInterrupt:
            print("\nStopped monitoring.")
    else:
        print_snapshot()


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`psutil` exposes cross-platform accessors for every metric shown: `cpu_percent()` for CPU load (system-wide and per-core), `virtual_memory()` and `swap_memory()` for RAM/swap, `disk_usage("/")` for the root filesystem, and `net_io_counters()` for cumulative network bytes. The `bytes_to_human` helper converts raw byte counts into readable KB/MB/GB units.

```bash
# Single snapshot
python3 monitor_system_resources.py

# Continuous dashboard, refreshing every 5 seconds
python3 monitor_system_resources.py --watch --interval 5

# Fast refresh for short-lived load spikes
python3 monitor_system_resources.py --watch --interval 1
```

##### Sample output

```text
$ python3 monitor_system_resources.py
[2026-07-17 10:42:03] System Resource Snapshot

CPU Usage:     23.4%
CPU Per Core:  [18.2, 27.6, 25.1, 22.9]

Memory:        6.1 GB / 16.0 GB used (38.1%)
Swap:          0.0 B / 2.0 GB used

Disk (/):      182.3 GB / 500.0 GB used (36.5%)

Network Sent:  1.2 GB
Network Recv:  4.8 GB
```

##### Variations / flags

- Add a `--json` flag to emit each snapshot as a JSON object, suitable for piping into `jq` or logging to a metrics collector.
- Add a `--process TOP_N` flag using `psutil.process_iter()` to list the top N processes by CPU or memory, turning this into a lightweight `top` replacement.
- Log snapshots to a CSV file over time for later trend analysis: `python3 monitor_system_resources.py --watch --interval 60 >> resource_log.txt`.

##### Common pitfalls

- **The first CPU reading always costs ~1 second** due to `cpu_percent(interval=1)`'s sampling requirement — don't run this in a tight loop expecting sub-second responsiveness.
- **Disk usage only checks `/`** (or `C:\` on Windows via psutil's path handling) — mounted secondary volumes are not included unless you modify the script to accept a path argument.
- **Network counters are cumulative since boot**, not per-interval — to get a rate (e.g., MB/s), you need to compute the delta between two consecutive snapshots yourself.
- Running with `--watch --interval 1` on a busy system generates a lot of terminal output quickly; consider redirecting to a file or increasing the interval for long monitoring sessions.

### 7.7 Detect Resource Exhaustion

##### Use case

A sysadmin wants an early-warning system that watches system-wide CPU, memory, and disk usage in the background and immediately flags when any of them crosses a dangerous threshold — catching resource exhaustion *before* it causes an outage, rather than debugging after the fact.

##### Prerequisites

Requires the cross-platform `psutil` library.

```bash
pip install psutil
# or:
pip install -r requirements.txt
```

##### Cautions

This script alerts by printing to stdout only — it does not page, email, or otherwise notify anyone by default, so it's only useful if something is actually watching its output (a terminal, a log aggregator, or a wrapper script that forwards alerts). Like `monitor_system_resources.py`, each check briefly blocks for 1 second while sampling CPU usage.

##### Script

```python
#!/usr/bin/env python3
"""
detect_resource_exhaustion.py
-----------------------------------
Continuously monitors CPU, memory, and disk usage and alerts (prints a
warning) when any of them crosses a configurable threshold — helping you
catch resource exhaustion before it causes an outage.

Requires: pip install psutil

Usage:
    python detect_resource_exhaustion.py [--cpu 90] [--memory 90] [--disk 90] [--interval 5]

Example:
    python detect_resource_exhaustion.py
    python detect_resource_exhaustion.py --cpu 80 --memory 85 --disk 95 --interval 10
"""

import argparse
import sys
import time
from datetime import datetime

try:
    import psutil
except ImportError:
    print("Error: the 'psutil' package is required.")
    print("Install it with: pip install -r requirements.txt")
    sys.exit(1)


def check_thresholds(cpu_limit: float, memory_limit: float, disk_limit: float) -> list:
    alerts = []

    cpu_percent = psutil.cpu_percent(interval=1)
    if cpu_percent >= cpu_limit:
        alerts.append(f"CPU usage is {cpu_percent}% (threshold: {cpu_limit}%)")

    mem = psutil.virtual_memory()
    if mem.percent >= memory_limit:
        alerts.append(f"Memory usage is {mem.percent}% (threshold: {memory_limit}%)")

    disk = psutil.disk_usage("/")
    if disk.percent >= disk_limit:
        alerts.append(f"Disk usage is {disk.percent}% (threshold: {disk_limit}%)")

    return alerts


def main():
    parser = argparse.ArgumentParser(description="Detect resource exhaustion (CPU, memory, disk) and alert.")
    parser.add_argument("--cpu", type=float, default=90, help="CPU usage percent threshold (default: 90)")
    parser.add_argument("--memory", type=float, default=90, help="Memory usage percent threshold (default: 90)")
    parser.add_argument("--disk", type=float, default=90, help="Disk usage percent threshold (default: 90)")
    parser.add_argument("--interval", type=int, default=5, help="Check interval in seconds (default: 5)")
    parser.add_argument("--once", action="store_true", help="Check once instead of continuously monitoring")
    args = parser.parse_args()

    if args.once:
        alerts = check_thresholds(args.cpu, args.memory, args.disk)
        if alerts:
            print("RESOURCE EXHAUSTION DETECTED:")
            for a in alerts:
                print(f"  - {a}")
            sys.exit(1)
        else:
            print("All resources are within normal thresholds.")
        return

    print(f"Monitoring for resource exhaustion every {args.interval} second(s).")
    print(f"Thresholds -> CPU: {args.cpu}%, Memory: {args.memory}%, Disk: {args.disk}%")
    print("Press Ctrl+C to stop.\n")

    try:
        while True:
            timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
            alerts = check_thresholds(args.cpu, args.memory, args.disk)

            if alerts:
                print(f"[{timestamp}] ALERT — resource exhaustion detected:")
                for a in alerts:
                    print(f"  - {a}")
            else:
                print(f"[{timestamp}] OK — all resources within thresholds.")

            time.sleep(max(0, args.interval - 1))
    except KeyboardInterrupt:
        print("\nStopped monitoring.")


if __name__ == "__main__":
    main()
```

##### How it works and how to run

`check_thresholds()` samples CPU (`psutil.cpu_percent`), memory (`psutil.virtual_memory().percent`), and disk (`psutil.disk_usage("/").percent`) and compares each against its configured threshold, collecting any that are breached into a list of alert strings. In continuous mode, the loop re-checks on a fixed interval; in `--once` mode it checks a single time and sets its exit code accordingly (`0` clean, `1` alert), making it easy to embed in cron or CI health checks.

```bash
# Continuous monitoring with default 90% thresholds
python3 detect_resource_exhaustion.py

# Custom thresholds, checking every 10 seconds
python3 detect_resource_exhaustion.py --cpu 80 --memory 85 --disk 95 --interval 10

# Single check suitable for cron (exit code signals alert state)
python3 detect_resource_exhaustion.py --once --cpu 85 --memory 90 --disk 90
```

##### Sample output

```text
$ python3 detect_resource_exhaustion.py --cpu 80 --memory 85 --disk 95 --interval 10
Monitoring for resource exhaustion every 10 second(s).
Thresholds -> CPU: 80.0%, Memory: 85.0%, Disk: 95.0%
Press Ctrl+C to stop.

[2026-07-17 10:55:12] OK — all resources within thresholds.
[2026-07-17 10:55:21] ALERT — resource exhaustion detected:
  - CPU usage is 91.2% (threshold: 80.0%)
  - Memory usage is 88.7% (threshold: 85.0%)
```

```text
$ python3 detect_resource_exhaustion.py --once
All resources are within normal thresholds.
```

##### Variations / flags

- Add a `--webhook URL` flag to POST alerts to Slack, PagerDuty, or a custom endpoint instead of only printing to stdout.
- Add a `--sustained N` flag requiring a threshold to be breached for N consecutive checks before alerting, to avoid noisy false positives from brief spikes.
- Schedule the `--once` mode with cron for lightweight periodic health checks without a long-running background process: `*/5 * * * * /usr/bin/python3 /path/detect_resource_exhaustion.py --once --cpu 90 --memory 90 --disk 90 || mail -s "Resource alert" admin@example.com`.

##### Common pitfalls

- **Alerts are print-only by default** — without piping to a log file, notification script, or terminal you're actively watching, alerts will be missed entirely.
- **Short spikes can trigger false alarms**, since each check is a point-in-time sample; a single CPU-heavy moment can cross the threshold even if overall load is healthy.
- **Disk checks only cover the root filesystem** (`/`) — a separate data volume filling up won't be detected unless you extend the script to check additional mount points.
- Running very short `--interval` values (e.g., 1 second) means the CPU sampling call itself becomes a meaningful fraction of your monitoring loop's own overhead.


### 7.8 Automatically Restart Failed Processes

##### Use case

A sysadmin runs a long-lived worker, small server, or scheduled job that occasionally crashes or hangs due to transient errors (a flaky network dependency, a memory spike, an unhandled exception) and wants it to come back automatically, without manual intervention, while avoiding an infinite "restart storm" if the process is fundamentally broken.

##### Prerequisites

Standard library only (`subprocess`, `argparse`, `time`) — no `pip install` required. Works identically on Linux, macOS, and Windows.

```bash
# No installation needed on any platform.
python3 --version
```

##### Cautions

Setting `--max-retries 0` means **unlimited retries** — if the underlying command is permanently broken (bad config, missing dependency), this will restart it forever and can flood your logs. Always pair a short `--delay` with a sane `--max-retries` in production rather than leaving both at extreme values.

##### Script

```python
#!/usr/bin/env python3
"""
restart_failed_process.py
-------------------------------
Runs a command and automatically restarts it if it crashes (exits with a
non-zero code) or if it hangs beyond an optional per-run timeout. Includes
a maximum retry count and a delay between restarts to avoid restart storms.

Usage:
    python restart_failed_process.py <command> [args...] [--max-retries 5] [--delay 3] [--timeout 0]

Example:
    python restart_failed_process.py python3 my_server.py
    python restart_failed_process.py python3 my_server.py --max-retries 10 --delay 5
    python restart_failed_process.py python3 flaky_job.py --timeout 30
"""

import argparse
import subprocess
import sys
import time
from datetime import datetime


def run_and_restart(command: list, max_retries: int, delay: int, timeout: int) -> None:
    attempt = 0

    while attempt < max_retries or max_retries == 0:
        attempt += 1
        timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        print(f"[{timestamp}] Starting process (attempt {attempt}): {' '.join(command)}")

        try:
            if timeout > 0:
                result = subprocess.run(command, timeout=timeout)
                exit_code = result.returncode
            else:
                process = subprocess.Popen(command)
                exit_code = process.wait()
        except FileNotFoundError:
            print(f"Error: command not found: {command[0]}")
            sys.exit(1)
        except subprocess.TimeoutExpired:
            print(f"Process exceeded the {timeout}-second timeout and was terminated.")
            exit_code = -1

        timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")

        if exit_code == 0:
            print(f"[{timestamp}] Process exited normally (code 0). Stopping — no restart needed.")
            return

        print(f"[{timestamp}] Process exited with code {exit_code}.")

        if max_retries != 0 and attempt >= max_retries:
            print(f"Maximum retry limit ({max_retries}) reached. Giving up.")
            sys.exit(1)

        print(f"Restarting in {delay} second(s)... (Ctrl+C to stop)\n")
        try:
            time.sleep(delay)
        except KeyboardInterrupt:
            print("\nStopped by user.")
            sys.exit(0)


def main():
    parser = argparse.ArgumentParser(
        description="Run a command and automatically restart it if it fails.",
        usage="restart_failed_process.py <command> [args...] [--max-retries N] [--delay N] [--timeout N]",
    )
    parser.add_argument("command", nargs=argparse.REMAINDER, help="Command to run (and restart on failure)")
    parser.add_argument("--max-retries", type=int, default=5,
                         help="Maximum number of restart attempts, 0 = unlimited (default: 5)")
    parser.add_argument("--delay", type=int, default=3, help="Seconds to wait before restarting (default: 3)")
    parser.add_argument("--timeout", type=int, default=0,
                         help="Per-run timeout in seconds; 0 = no timeout, run until it exits (default: 0)")
    args = parser.parse_args()

    command = args.command
    if command and command[0] == "--":
        command = command[1:]

    if not command:
        print("Error: no command provided.")
        print("Usage: restart_failed_process.py <command> [args...] [--max-retries N] [--delay N] [--timeout N]")
        sys.exit(1)

    run_and_restart(command, args.max_retries, args.delay, args.timeout)


if __name__ == "__main__":
    main()
```

##### How it works and how to run

The script loops, launching the target command each iteration with `subprocess.Popen` (or `subprocess.run(..., timeout=...)` if `--timeout` is set). A clean exit (code `0`) stops the loop immediately; any non-zero exit or a timeout increments the retry counter, waits `--delay` seconds, and tries again — up to `--max-retries` attempts, or forever if `--max-retries 0`.

```bash
# Basic supervision with default 5 retries, 3-second delay
python3 restart_failed_process.py python3 my_server.py

# More retries, longer delay between attempts
python3 restart_failed_process.py python3 my_server.py --max-retries 10 --delay 5

# Kill and restart a job that hangs past 30 seconds per attempt
python3 restart_failed_process.py python3 flaky_job.py --timeout 30

# Unlimited retries for a critical long-running service (use with caution)
python3 restart_failed_process.py python3 worker.py --max-retries 0 --delay 10
```

  ...

Restarting in 2 second(s)... (Ctrl+C to stop)

  ...

##### Variations / flags

- Add a `--backoff exponential` flag to increase the delay after each failed attempt (e.g., 2s, 4s, 8s...) instead of a fixed delay, reducing load during sustained outages.
- Add a `--notify-on-giveup` flag to send an email/webhook alert once `--max-retries` is exhausted, so a human is informed the process needs manual attention.
- Run as a systemd service or Windows Scheduled Task for persistent supervision across reboots, rather than a one-off terminal invocation.

##### Common pitfalls

- **`--max-retries 0` means unlimited**, not zero attempts — a common source of confusion; if you want the script to attempt exactly once with no retries, use `--max-retries 1`.
- **Fixed delay can contribute to restart storms** under sustained failure, hammering a dependency (e.g., a database) that's still recovering — consider exponential backoff for production use.
- **`--timeout` here caps each individual run**, not the total supervision time — a process that keeps timing out will keep restarting indefinitely up to `--max-retries`, consuming that budget quickly.
- The script's own `Ctrl+C` handling only interrupts the *sleep between* restarts; if you need to stop a currently-running child immediately, you'll need to interrupt during execution too, which may leave the child process needing manual cleanup depending on the OS and signal propagation.
