## Chapter 22: Container & Virtualization Automation

Modern infrastructure rarely runs on bare metal alone. Somewhere behind almost every production deployment sits a container, a virtual machine, or an orchestrator quietly making sure the right code is running in the right place with the right amount of CPU and memory. For a long time, managing that layer meant living in a terminal, typing `docker run` a hundred slightly different ways, copy-pasting `kubectl` incantations from old notes, or clicking through a hypervisor's GUI at 2 AM because a VM snapshot needed to be restored before a demo. That works, until it doesn't scale — until you need to do the same ten-step deployment across fifteen environments, or reproduce a bug that only shows up when a container is memory-starved, or tear down and rebuild a lab environment every single morning.

This chapter is about automating that layer instead of hand-operating it. Every script here wraps a well-established tool — the Docker Engine API, the Kubernetes API, Helm, libvirt, LXD, Vagrant — behind a small, scriptable Python command-line interface. None of these scripts reinvent container or VM technology; they orchestrate it, so that the tasks a systems engineer performs daily (spin up a container, check if it's healthy, cap its memory, snapshot a VM before a risky change, clean up disk space littered with dangling images) become single, repeatable, loggable commands instead of manual multi-step rituals.

The chapter is organized around the two dominant virtualization models in use today: containers and full virtual machines, plus the orchestration layers built on top of both.

#### Containers: Docker and the Docker ecosystem

The bulk of this chapter (Sections 22.1–22.9, 22.18–22.19) covers Docker, because Docker remains the default packaging and runtime format for most modern applications. We start with the basics — building an image from a Dockerfile, running a container with the right ports, volumes, and environment variables — and move outward to the operational concerns that separate a toy deployment from a production one: attaching health checks so orchestrators know when a service is actually ready, capping CPU and memory so one runaway container can't starve its neighbors, wiring up custom networks so containers can (or can't) talk to each other, and managing the volumes that hold state across restarts.

We also cover the plumbing that keeps a Docker host healthy over time — pruning unused images, containers, and volumes before disk space becomes a 3 AM page — and the everyday debugging workflow of tailing logs and running ad hoc commands inside a live container. Rounding out the Docker section, we cover registry operations: authenticating to Docker Hub, ECR, or a private registry, and pulling or pushing images as part of a build/release pipeline. Docker Compose gets its own section because multi-container applications, not single containers, are the norm in real deployments — Compose lets a whole stack be brought up, inspected, and torn down as one unit.

#### Kubernetes and Helm

Sections 22.15–22.17 move up a layer, into orchestration. Kubernetes takes the primitives Docker gives us — images, containers, networks, volumes — and turns them into a declarative system: you describe the desired state (how many replicas, which image version, what resources) and the cluster continuously works to match it. We cover direct pod management for quick, imperative operations, Deployment management for the rolling-update workflow that production services actually use, and Helm for templated, versioned, repeatable application packaging. These scripts don't replace `kubectl` or `helm` — they wrap them so the same operations can be triggered from a larger automation pipeline, a scheduled job, or another Python program.

#### Full virtualization: libvirt, VM lifecycle, and Vagrant

Sections 22.10–22.14 step back from containers entirely and address full virtual machines — the heavier-weight but more isolated alternative, still essential for testing kernel-level changes, running different operating systems side by side, or satisfying compliance requirements that containers can't. We cover provisioning a new VM through libvirt/KVM, cloning an existing VM for rapid environment replication, and capturing and restoring point-in-time snapshots — the single most useful safety net when testing anything destructive. Vagrant gets its own section as the higher-level developer-facing tool that wraps a hypervisor (VirtualBox, libvirt) with a declarative `Vagrantfile`, and LXC/LXD is covered as a middle ground: system containers that behave much more like lightweight VMs than application containers.

### 22.1 Build Docker Images Programmatically

`docker_build_image.py` — Build a Docker image from a Dockerfile and build context without typing out `docker build` by hand, and capture the build log and resulting image ID in a form another script can consume.

#### Use case

A CI pipeline or release script needs to build an image, inject version-specific build arguments (like a Git commit hash or semantic version), and know programmatically whether the build succeeded or failed — not just watch text scroll by in a terminal.

#### Prerequisites

- Docker Engine or Docker Desktop installed and **running** on the machine executing this script
- The user/service account running the script must have permission to talk to the Docker socket (on Linux: be a member of the `docker` group, or run with `sudo`)
- A build context directory containing a valid `Dockerfile` (or a custom-named one passed via `--dockerfile`)
- OS scope: Linux, macOS, Windows (anywhere Docker Engine/Desktop runs)

```bash
pip install docker
```

#### Cautions

`--no-cache` forces a full rebuild of every layer, which is slower and pulls fresh base images — use it deliberately, not as a default. Build arguments passed via `--buildarg` are visible in image history/metadata, so never pass secrets that way.

#### Script

```python
#!/usr/bin/env python3
"""
docker_build_image.py
Build a Docker image programmatically using the Docker SDK for Python
(no need to run `docker build` manually).

USAGE
    python docker_build_image.py --path ./myapp --tag myapp:latest
    python docker_build_image.py --path ./myapp --tag myapp:latest \
        --dockerfile Dockerfile.prod --buildarg VERSION=1.0 --buildarg ENV=prod --no-cache

REQUIREMENTS
    - Docker Engine/Desktop installed and RUNNING on this machine
    - The account running this script must have permission to talk to the
      Docker socket (on Linux, be in the `docker` group or run with sudo)
    - pip install -r requirements.txt   (installs the `docker` package)

EXPECTED OUTPUT
    Streamed build log lines from the Docker daemon, ending with:
        Image built successfully: myapp:latest (id=sha256:xxxxxxx)
    On failure, the build log is printed and the script exits with code 1.
"""

import argparse
import sys

import docker
from docker.errors import APIError, BuildError


def parse_build_args(pairs):
    result = {}
    for pair in pairs or []:
        if "=" not in pair:
            print(f"Ignoring malformed --buildarg '{pair}', expected KEY=VALUE")
            continue
        k, v = pair.split("=", 1)
        result[k] = v
    return result


def build_image(path, tag, dockerfile, buildargs, nocache):
    client = docker.from_env()
    print(f"Building image '{tag}' from context '{path}' ...")
    try:
        image, logs = client.images.build(
            path=path,
            tag=tag,
            dockerfile=dockerfile,
            buildargs=buildargs,
            nocache=nocache,
            rm=True,
        )
        for chunk in logs:
            if "stream" in chunk:
                sys.stdout.write(chunk["stream"])
        print(f"\nImage built successfully: {tag} (id={image.short_id})")
        return image
    except BuildError as e:
        print(f"Build failed: {e}")
        for line in e.build_log:
            if "stream" in line:
                sys.stdout.write(line["stream"])
        sys.exit(1)
    except APIError as e:
        print(f"Docker API error: {e}")
        sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Build a Docker image programmatically.")
    parser.add_argument("--path", required=True, help="Build context directory (contains the Dockerfile)")
    parser.add_argument("--tag", required=True, help="Image tag, e.g. myapp:latest")
    parser.add_argument("--dockerfile", default="Dockerfile", help="Dockerfile name relative to --path")
    parser.add_argument("--buildarg", action="append", help="Build arg KEY=VALUE, repeatable")
    parser.add_argument("--no-cache", action="store_true", help="Disable the build cache")
    args = parser.parse_args()

    buildargs = parse_build_args(args.buildarg)
    build_image(args.path, args.tag, args.dockerfile, buildargs, args.no_cache)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script wraps `docker.images.build()` from the Docker SDK, which talks to the daemon over the same API `docker build` uses under the hood. `--path` is the build context (everything the Dockerfile can `COPY` from); `--dockerfile` lets you point at an alternate Dockerfile (e.g. `Dockerfile.prod`) without renaming files. Repeatable `--buildarg KEY=VALUE` flags become the `buildargs` dict passed to the SDK, letting you inject values like a version string at build time.

```bash
# Standard build
python docker_build_image.py --path ./myapp --tag myapp:latest

# Production build with build args and no cache
python docker_build_image.py --path ./myapp --tag myapp:latest \
    --dockerfile Dockerfile.prod --buildarg VERSION=1.0 --buildarg ENV=prod --no-cache
```

#### Sample output

```
Building image 'myapp:latest' from context './myapp' ...
Step 1/6 : FROM python:3.11-slim
 ---> 2af23f00e5a1
Step 2/6 : WORKDIR /app
 ---> Using cache
 ---> b1e9a0c3d2f4
...

Image built successfully: myapp:latest (id=sha256:9f8a7b6c5d4e)
```

#### Variations / flags

- Add a `--platform linux/arm64` passthrough for cross-building multi-arch images (requires Buildx-backed daemon).
- Add a `--quiet` mode that suppresses streamed log lines and only prints the final result, for use in scripted pipelines.
- Wire this into a CI step that tags the image with `$CI_COMMIT_SHA` via `--buildarg`.

#### Common pitfalls

- **Docker daemon not running** — the script will fail immediately with a connection error; always confirm `docker info` succeeds first.
- **Permission denied on socket** — on Linux, the invoking user must be in the `docker` group, or every invocation needs `sudo`.
- **Wrong build context** — `--path` must contain everything referenced by `COPY`/`ADD`; a Dockerfile living outside the context directory will fail to resolve paths.
- **Silent build arg typos** — malformed `--buildarg` values (missing `=`) are skipped with a warning, not treated as fatal — easy to miss in long log output.

### 22.2 Run Docker Containers

`docker_run_container.py` — Launch, stop, and remove a Docker container, with custom port mappings, bind-mounted volumes, and environment variables, all from a single scriptable command.

#### Use case

An engineer needs to spin up a dependency (a database, a cache, an internal API) for local development or an integration test run, wire it to a specific host port and a persistent data directory, and later tear it down cleanly — without hand-typing a long `docker run` command each time.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Permission to access the Docker socket (Linux: `docker` group membership, or `sudo`)
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

`rm --force` will forcibly kill and remove a container even mid-write, which can corrupt data in bind-mounted volumes if the process was writing at the time. Publishing a port with `--port` exposes that service on the host's network interface — don't expose sensitive services (databases, admin panels) on `0.0.0.0` without a firewall in front of them.

#### Script

```python
#!/usr/bin/env python3
"""
docker_run_container.py
Launch and manage a container's lifecycle (run, stop, remove) with custom
ports, volumes, and environment variables via the Docker SDK for Python.

USAGE
    Run a container:
        python docker_run_container.py run --image nginx:latest --name web \
            --port 8080:80 --volume /host/data:/data --env KEY=VALUE --detach

    Stop a container:
        python docker_run_container.py stop --name web

    Remove a container:
        python docker_run_container.py rm --name web --force

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    run   -> prints the new container's id, name and status ("running")
    stop  -> prints "Container 'web' stopped."
    rm    -> prints "Container 'web' removed."
"""

import argparse

import docker
from docker.errors import APIError, NotFound


def parse_kv_list(pairs):
    result = {}
    for pair in pairs or []:
        if "=" not in pair:
            continue
        k, v = pair.split("=", 1)
        result[k] = v
    return result


def parse_ports(pairs):
    # host:container -> {"container/tcp": host}
    ports = {}
    for pair in pairs or []:
        host, container = pair.split(":")
        ports[f"{container}/tcp"] = int(host)
    return ports


def parse_volumes(pairs):
    # host:container[:mode] -> {"host": {"bind": "container", "mode": "rw"}}
    volumes = {}
    for pair in pairs or []:
        parts = pair.split(":")
        host, container = parts[0], parts[1]
        mode = parts[2] if len(parts) > 2 else "rw"
        volumes[host] = {"bind": container, "mode": mode}
    return volumes


def run_container(args):
    client = docker.from_env()
    try:
        container = client.containers.run(
            image=args.image,
            name=args.name,
            ports=parse_ports(args.port),
            volumes=parse_volumes(args.volume),
            environment=parse_kv_list(args.env),
            detach=True,
        )
        container.reload()
        print(f"Container started: id={container.short_id} name={container.name} status={container.status}")
    except APIError as e:
        print(f"Failed to run container: {e}")


def stop_container(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
        container.stop()
        print(f"Container '{args.name}' stopped.")
    except NotFound:
        print(f"Container '{args.name}' not found.")


def remove_container(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
        container.remove(force=args.force)
        print(f"Container '{args.name}' removed.")
    except NotFound:
        print(f"Container '{args.name}' not found.")


def main():
    parser = argparse.ArgumentParser(description="Manage the lifecycle of Docker containers.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_run = sub.add_parser("run", help="Run a new container")
    p_run.add_argument("--image", required=True)
    p_run.add_argument("--name", required=True)
    p_run.add_argument("--port", action="append", help="host:container, repeatable")
    p_run.add_argument("--volume", action="append", help="host:container[:mode], repeatable")
    p_run.add_argument("--env", action="append", help="KEY=VALUE, repeatable")
    p_run.add_argument("--detach", action="store_true", help="(kept for CLI familiarity; always detached)")
    p_run.set_defaults(func=run_container)

    p_stop = sub.add_parser("stop", help="Stop a running container")
    p_stop.add_argument("--name", required=True)
    p_stop.set_defaults(func=stop_container)

    p_rm = sub.add_parser("rm", help="Remove a container")
    p_rm.add_argument("--name", required=True)
    p_rm.add_argument("--force", action="store_true")
    p_rm.set_defaults(func=remove_container)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The `run` subcommand parses repeatable `--port`, `--volume`, and `--env` flags into the dict shapes the Docker SDK expects, then calls `client.containers.run(detach=True)`. `stop` and `rm` look the container up by name and act on it, catching `NotFound` so a typo doesn't produce a raw traceback.

```bash
# Run nginx, mapping host port 8080 to container port 80, mounting a data dir, and setting an env var
python docker_run_container.py run --image nginx:latest --name web \
    --port 8080:80 --volume /host/data:/data --env KEY=VALUE --detach

# Stop it
python docker_run_container.py stop --name web

# Remove it (force even if still running)
python docker_run_container.py rm --name web --force
```

#### Sample output

```
Container started: id=a3f9c21e0b7d name=web status=running
```

```
Container 'web' stopped.
```

```
Container 'web' removed.
```

#### Variations / flags

- Add a `--restart-policy always` flag mapped to the SDK's `restart_policy` parameter for services that should survive host reboots.
- Add a `--network` flag so the container joins a specific custom network at launch instead of the default bridge.
- Wrap `run` + `check_health` (Section 22.18) into a single pipeline: launch, then poll until healthy before proceeding.

#### Common pitfalls

- **Port already in use** — if the host port in `--port` is already bound by another process, `run` will fail with an `APIError`; check with `docker ps` or `lsof -i :<port>` first.
- **Volume path doesn't exist on host** — Docker will silently create an empty directory at that path rather than error, which can mask a typo in `--volume`.
- **Name collision** — reusing a `--name` that belongs to a stopped-but-not-removed container fails with "container name already in use"; run `rm` first.
- **Windows path syntax** — bind mount paths on Windows need drive-letter-aware formatting (e.g. `//c/host/data:/data` in some shells) which differs from the Linux/macOS examples shown here.

### 22.3 Manage Docker Volumes

`docker_manage_volumes.py` — Create, list, inspect, remove, and prune Docker volumes, the mechanism Docker uses for data that needs to outlive a container.

#### Use case

A team runs a database in a container and needs its data to survive container restarts and upgrades. Someone needs to create a named volume ahead of time, verify it exists with the right driver, and periodically prune orphaned volumes left behind by removed containers before they quietly consume all available disk space.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Permission to access the Docker socket
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

`remove --force` and especially `prune` are destructive and **irreversible** — once a volume's data is deleted there is no undo. Always confirm the volume isn't attached to a container you still need before removing or pruning, since `prune` only removes volumes with no attached containers, but a container you've stopped-and-forgotten still counts as "attached."

#### Script

```python
#!/usr/bin/env python3
"""
docker_manage_volumes.py
Programmatically create, inspect, list, and remove Docker volumes used for
persistent storage in containerized applications.

USAGE
    python docker_manage_volumes.py create --name app_data
    python docker_manage_volumes.py list
    python docker_manage_volumes.py inspect --name app_data
    python docker_manage_volumes.py remove --name app_data
    python docker_manage_volumes.py prune

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    create   -> "Volume 'app_data' created (driver=local)."
    list     -> a table of existing volume names and drivers
    inspect  -> the raw JSON metadata of the volume (mountpoint, labels, etc.)
    remove   -> "Volume 'app_data' removed."
    prune    -> number of unused volumes deleted and space reclaimed
"""

import argparse
import json

import docker
from docker.errors import APIError, NotFound


def create_volume(args):
    client = docker.from_env()
    volume = client.volumes.create(name=args.name, driver=args.driver)
    print(f"Volume '{volume.name}' created (driver={args.driver}).")


def list_volumes(args):
    client = docker.from_env()
    volumes = client.volumes.list()
    if not volumes:
        print("No volumes found.")
        return
    print(f"{'NAME':30} {'DRIVER':10}")
    for v in volumes:
        print(f"{v.name:30} {v.attrs.get('Driver', ''):10}")


def inspect_volume(args):
    client = docker.from_env()
    try:
        volume = client.volumes.get(args.name)
        print(json.dumps(volume.attrs, indent=2))
    except NotFound:
        print(f"Volume '{args.name}' not found.")


def remove_volume(args):
    client = docker.from_env()
    try:
        volume = client.volumes.get(args.name)
        volume.remove(force=args.force)
        print(f"Volume '{args.name}' removed.")
    except NotFound:
        print(f"Volume '{args.name}' not found.")
    except APIError as e:
        print(f"Could not remove volume: {e}")


def prune_volumes(args):
    client = docker.from_env()
    result = client.volumes.prune()
    reclaimed = result.get("SpaceReclaimed", 0)
    deleted = result.get("VolumesDeleted", []) or []
    print(f"Pruned {len(deleted)} unused volume(s), reclaimed {reclaimed} bytes.")


def main():
    parser = argparse.ArgumentParser(description="Manage Docker volumes programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_create = sub.add_parser("create")
    p_create.add_argument("--name", required=True)
    p_create.add_argument("--driver", default="local")
    p_create.set_defaults(func=create_volume)

    p_list = sub.add_parser("list")
    p_list.set_defaults(func=list_volumes)

    p_inspect = sub.add_parser("inspect")
    p_inspect.add_argument("--name", required=True)
    p_inspect.set_defaults(func=inspect_volume)

    p_remove = sub.add_parser("remove")
    p_remove.add_argument("--name", required=True)
    p_remove.add_argument("--force", action="store_true")
    p_remove.set_defaults(func=remove_volume)

    p_prune = sub.add_parser("prune")
    p_prune.set_defaults(func=prune_volumes)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Each subcommand maps directly to a method on `client.volumes`: `create()`, `list()`, `get()` (for inspect/remove), and `prune()`. `inspect` dumps the full raw attribute dictionary as JSON so it can be piped into `jq` or parsed by another tool.

```bash
# Create a named volume
python docker_manage_volumes.py create --name app_data

# List all volumes
python docker_manage_volumes.py list

# Inspect one volume's metadata
python docker_manage_volumes.py inspect --name app_data

# Remove it
python docker_manage_volumes.py remove --name app_data

# Prune all unused volumes
python docker_manage_volumes.py prune
```

#### Sample output

```
Volume 'app_data' created (driver=local).
```

```
NAME                           DRIVER    
app_data                       local     
postgres_data                  local     
```

```json
{
  "CreatedAt": "2026-07-17T10:00:00Z",
  "Driver": "local",
  "Labels": {},
  "Mountpoint": "/var/lib/docker/volumes/app_data/_data",
  "Name": "app_data",
  "Options": {},
  "Scope": "local"
}
```

```
Pruned 3 unused volume(s), reclaimed 47185920 bytes.
```

#### Variations / flags

- Add a `--label` filter to `list` and `prune` so only volumes matching a specific project or team label are affected.
- Add a `--dry-run` flag to `prune` that lists what *would* be deleted without actually deleting anything — valuable before running against a shared host.
- Schedule `prune` as a weekly cron job on build/CI hosts where volumes accumulate quickly.

#### Common pitfalls

- **Removing a volume still in use** — Docker will refuse to remove a volume attached to any container (running or stopped) unless `--force` is passed, and even then only in narrower cases; the safer fix is to remove the container first.
- **Prune doesn't touch attached-but-stopped containers' volumes** — people often expect `prune` to clean up "everything unused," but a volume tied to a stopped container is technically still attached and won't be pruned.
- **Bind mounts vs named volumes confusion** — this script only manages named/managed volumes, not host bind mounts configured directly in `docker run --volume /host/path:/container/path`.

### 22.4 Manage Docker Networks

`docker_manage_networks.py` — Create, list, inspect, remove Docker networks, and connect or disconnect containers from them, to control which containers can talk to each other and to the outside world.

#### Use case

A multi-service application needs its backend and database containers isolated on a private, internal-only network that the public-facing web container cannot reach directly, while the web container itself needs to be reachable from the internet. A sysadmin needs to create that segmented topology and adjust it as services are added or retired.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Permission to access the Docker socket
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

An `--internal` network blocks outbound internet access for every container connected to it — including package downloads and DNS lookups to external hosts — so don't use it for a container that needs to reach out to third-party APIs. Removing a network that still has containers connected to it will fail; disconnect them first.

#### Script

```python
#!/usr/bin/env python3
"""
docker_manage_networks.py
Create, list, inspect, and remove Docker networks, and connect/disconnect
containers to control communication between containers and the outside world.

USAGE
    python docker_manage_networks.py create --name backend --driver bridge --internal
    python docker_manage_networks.py list
    python docker_manage_networks.py connect --network backend --container web
    python docker_manage_networks.py disconnect --network backend --container web
    python docker_manage_networks.py remove --name backend

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    create      -> "Network 'backend' created (driver=bridge, internal=True)."
    list        -> table of network names, drivers and scope
    connect     -> "Connected container 'web' to network 'backend'."
    disconnect  -> "Disconnected container 'web' from network 'backend'."
    remove      -> "Network 'backend' removed."
"""

import argparse

import docker
from docker.errors import APIError, NotFound


def create_network(args):
    client = docker.from_env()
    network = client.networks.create(
        name=args.name, driver=args.driver, internal=args.internal
    )
    print(f"Network '{network.name}' created (driver={args.driver}, internal={args.internal}).")


def list_networks(args):
    client = docker.from_env()
    networks = client.networks.list()
    print(f"{'NAME':25} {'DRIVER':10} {'SCOPE':10}")
    for n in networks:
        print(f"{n.name:25} {n.attrs.get('Driver',''):10} {n.attrs.get('Scope',''):10}")


def connect(args):
    client = docker.from_env()
    try:
        network = client.networks.get(args.network)
        network.connect(args.container)
        print(f"Connected container '{args.container}' to network '{args.network}'.")
    except (NotFound, APIError) as e:
        print(f"Could not connect: {e}")


def disconnect(args):
    client = docker.from_env()
    try:
        network = client.networks.get(args.network)
        network.disconnect(args.container)
        print(f"Disconnected container '{args.container}' from network '{args.network}'.")
    except (NotFound, APIError) as e:
        print(f"Could not disconnect: {e}")


def remove_network(args):
    client = docker.from_env()
    try:
        network = client.networks.get(args.name)
        network.remove()
        print(f"Network '{args.name}' removed.")
    except NotFound:
        print(f"Network '{args.name}' not found.")
    except APIError as e:
        print(f"Could not remove network: {e}")


def main():
    parser = argparse.ArgumentParser(description="Manage Docker networks programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_create = sub.add_parser("create")
    p_create.add_argument("--name", required=True)
    p_create.add_argument("--driver", default="bridge")
    p_create.add_argument("--internal", action="store_true", help="Isolate network from external access")
    p_create.set_defaults(func=create_network)

    p_list = sub.add_parser("list")

    p_remove = sub.add_parser("remove")
    p_remove.add_argument("--name", required=True)
    p_remove.set_defaults(func=remove_network)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### Variations / flags

- Add a `--subnet 172.28.0.0/16` flag mapped to the SDK's IPAM config for environments that need predictable container IP ranges.

#### Common pitfalls

- **Removing a network with connected containers** — fails with an `APIError`; disconnect every container first, including stopped ones.
- **Using `--internal` and expecting DNS to external hosts to work** — it won't; internal networks block all egress, which is the point, but is a common source of confusion when debugging "container can't reach the internet."

### 22.5 Docker Compose Orchestration

`docker_compose_orchestration.py` — Bring up, inspect, stream logs from, and tear down an entire multi-container application defined in a `docker-compose.yml` file, driven from Python via the `python-on-whales` wrapper around the `docker compose` CLI.

#### Use case

A development team's application is made up of several services — a web server, an API, a database, a cache — that need to be started together, in the right order, sharing a network. Instead of manually running `docker compose up` and remembering follow-up commands, this script exposes that whole lifecycle as one CLI, suitable for calling from a larger setup script or CI job.

#### Prerequisites

- Docker Engine/Desktop installed and running, with the Compose plugin available (verify with `docker compose version`)
- A valid `docker-compose.yml` file in the project directory
- OS scope: Linux, macOS, Windows

```bash
pip install python-on-whales
```

#### Cautions

`down --volumes` deletes the named volumes created by the stack along with the containers — any database data stored there is gone unless it was backed up or stored in an external bind mount. Running `up --build` rebuilds images every time, which is slower and will invalidate layer caches if the Dockerfile or context changed since the last build.

#### Script

```python
#!/usr/bin/env python3
"""
docker_compose_orchestration.py
Define and control the entire lifecycle of a multi-service application
programmatically, driving Docker Compose through the python-on-whales library
(a Python wrapper around the `docker compose` CLI).

USAGE
    python docker_compose_orchestration.py up --file docker-compose.yml
    python docker_compose_orchestration.py ps --file docker-compose.yml
    python docker_compose_orchestration.py logs --file docker-compose.yml --service web
    python docker_compose_orchestration.py down --file docker-compose.yml

REQUIREMENTS
    - Docker Engine/Desktop installed and running, with the Compose plugin
      available (`docker compose version` should work in your terminal)
    - A valid docker-compose.yml file in your project
    - pip install -r requirements.txt   (installs python-on-whales)

EXPECTED OUTPUT
    up     -> builds/pulls images, starts all services, prints running service names
    ps     -> table of service name, status and exposed ports
    logs   -> streamed log lines for the requested service (or all services)
    down   -> stops and removes containers/networks created by `up`
"""

import argparse

from python_on_whales import DockerClient


def up(args):
    docker = DockerClient(compose_files=[args.file])
    docker.compose.up(detach=True, build=args.build)
    print("Services started:")
    for c in docker.compose.ps():
        print(f"  - {c.name} ({c.state.status})")


def down(args):
    docker = DockerClient(compose_files=[args.file])
    docker.compose.down(volumes=args.volumes)
    print("Compose stack stopped and removed.")


def ps(args):
    docker = DockerClient(compose_files=[args.file])
    containers = docker.compose.ps()
    if not containers:
        print("No running services.")
        return
    print(f"{'SERVICE':20} {'STATUS':15} {'PORTS'}")
    for c in containers:
        ports = ", ".join(f"{p}" for p in (c.network_settings.ports or {}))
        print(f"{c.name:20} {c.state.status:15} {ports}")


def logs(args):
    docker = DockerClient(compose_files=[args.file])
    services = [args.service] if args.service else None
    for line in docker.compose.logs(services=services, follow=args.follow, stream=True):
        print(line)


def main():
    parser = argparse.ArgumentParser(description="Control a Docker Compose stack programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_up = sub.add_parser("up", help="Start all services defined in the compose file")
    p_up.add_argument("--file", default="docker-compose.yml")
    p_up.add_argument("--build", action="store_true", help="Rebuild images before starting")
    p_up.set_defaults(func=up)

    p_down = sub.add_parser("down", help="Stop and remove the compose stack")
    p_down.add_argument("--file", default="docker-compose.yml")
    p_down.add_argument("--volumes", action="store_true", help="Also remove named volumes")
    p_down.set_defaults(func=down)

    p_ps = sub.add_parser("ps", help="List running services")
    p_ps.add_argument("--file", default="docker-compose.yml")
    p_ps.set_defaults(func=ps)

    p_logs = sub.add_parser("logs", help="Stream logs for a service (or all)")
    p_logs.add_argument("--file", default="docker-compose.yml")
    p_logs.add_argument("--service", help="Specific service name; omit for all services")
    p_logs.add_argument("--follow", action="store_true", help="Keep streaming new log lines")
    p_logs.set_defaults(func=logs)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`python-on-whales` builds a `DockerClient` scoped to a specific `docker-compose.yml`, then exposes `.compose.up()`, `.compose.down()`, `.compose.ps()`, and `.compose.logs()` as Python methods that shell out to the `docker compose` CLI under the hood while returning structured Python objects instead of raw text.

```bash
# Start the whole stack, rebuilding images first
python docker_compose_orchestration.py up --file docker-compose.yml

# Check what's running
python docker_compose_orchestration.py ps --file docker-compose.yml

# Follow logs for just the 'web' service
python docker_compose_orchestration.py logs --file docker-compose.yml --service web --follow

# Tear everything down, including volumes
python docker_compose_orchestration.py down --file docker-compose.yml
```

#### Sample output

```
Services started:
  - myapp-web-1 (running)
  - myapp-db-1 (running)
  - myapp-cache-1 (running)
```

```
SERVICE              STATUS          PORTS
myapp-web-1          running         8080/tcp
myapp-db-1           running         5432/tcp
myapp-cache-1        running         6379/tcp
```

```
Compose stack stopped and removed.
```

#### Variations / flags

- Add a `--scale web=3` passthrough for horizontally scaling a specific service.
- Add a `restart` subcommand wrapping `docker.compose.restart(services=[...])` for restarting one service without touching the rest of the stack.
- Chain `up` into a smoke-test script that waits for `ps` to show all services `running` before proceeding, similar to the polling pattern in Section 22.18.

#### Common pitfalls

- **Compose v1 vs v2 confusion** — this script requires the newer `docker compose` (v2, a Docker CLI plugin), not the older standalone `docker-compose` (v1) binary; `docker compose version` should return a v2 result.
- **Relative paths in `--file`** — the compose file's relative build contexts and volume paths are resolved relative to the file's own directory, which can surprise you if you run the script from somewhere else.
- **Stale containers after a Dockerfile change** — `up` without `--build` will keep using the previously built image even if the Dockerfile changed; always pass `--build` after making source or Dockerfile edits.

### 22.6 Prune Unused Docker Resources

`docker_prune_resources.py` — Clean up dangling images, stopped containers, unused networks, and orphaned volumes in one pass to reclaim disk space on a Docker host.

#### Use case

A build server that runs many `docker build`s a day accumulates dangling image layers and stopped containers until disk space runs out and builds start failing. An operator (or a scheduled job) needs a single command to clean all of that up and report exactly how much space was reclaimed.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Permission to access the Docker socket
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

Pruning is destructive: stopped containers, dangling (and with `--all-images`, *all* unused) images, unused networks, and unattached volumes are permanently deleted. Never run `--all-images` on a host where you rely on cached "just in case" images you haven't tagged as in-use — they'll be gone.

#### Script

```python
#!/usr/bin/env python3
"""
docker_prune_resources.py
Automatically clean up dangling images, stopped containers, unused networks,
and orphaned volumes to free disk space.

USAGE
    python docker_prune_resources.py --all
    python docker_prune_resources.py --containers --images
    python docker_prune_resources.py --volumes --networks

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    A summary for each resource type pruned, e.g.:
        Containers pruned: 3 removed
        Images pruned: 5 removed, 214582112 bytes reclaimed
        Volumes pruned: 1 removed, 10485760 bytes reclaimed
        Networks pruned: 2 removed
        Total space reclaimed: 225067872 bytes
"""

import argparse

import docker


def prune_containers(client):
    result = client.containers.prune()
    removed = result.get("ContainersDeleted", []) or []
    print(f"Containers pruned: {len(removed)} removed")
    return result.get("SpaceReclaimed", 0)


def prune_images(client, dangling_only):
    filters = {"dangling": True} if dangling_only else {}
    result = client.images.prune(filters=filters)
    removed = result.get("ImagesDeleted", []) or []
    space = result.get("SpaceReclaimed", 0)
    print(f"Images pruned: {len(removed)} removed, {space} bytes reclaimed")
    return space


def prune_volumes(client):
    result = client.volumes.prune()
    removed = result.get("VolumesDeleted", []) or []
    space = result.get("SpaceReclaimed", 0)
    print(f"Volumes pruned: {len(removed)} removed, {space} bytes reclaimed")
    return space


def prune_networks(client):
    result = client.networks.prune()
    removed = result.get("NetworksDeleted", []) or []
    print(f"Networks pruned: {len(removed)} removed")
    return 0


def main():
    parser = argparse.ArgumentParser(description="Prune unused Docker resources.")
    parser.add_argument("--all", action="store_true", help="Prune containers, images, volumes and networks")
    parser.add_argument("--containers", action="store_true")
    parser.add_argument("--images", action="store_true")
    parser.add_argument("--volumes", action="store_true")
    parser.add_argument("--networks", action="store_true")
    parser.add_argument("--all-images", action="store_true", help="Prune ALL unused images, not just dangling ones")
    args = parser.parse_args()

    if not any([args.all, args.containers, args.images, args.volumes, args.networks]):
        parser.error("Specify --all or at least one of --containers/--images/--volumes/--networks")

    client = docker.from_env()
    total = 0

    if args.all or args.containers:
        total += prune_containers(client)
    if args.all or args.images:
        total += prune_images(client, dangling_only=not args.all_images)
    if args.all or args.volumes:
        total += prune_volumes(client)
    if args.all or args.networks:
        total += prune_networks(client)

    print(f"Total space reclaimed: {total} bytes")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

Each resource type has its own `.prune()` method on the Docker SDK client, each returning a dict with a `*Deleted` list and, for images/volumes, a `SpaceReclaimed` byte count. The script accumulates the total across whichever categories were selected and prints one combined summary at the end.

```bash
# Prune everything
python docker_prune_resources.py --all

# Prune just containers and dangling images
python docker_prune_resources.py --containers --images

# Prune volumes and networks only
python docker_prune_resources.py --volumes --networks

# Prune ALL unused images, not just dangling ones
python docker_prune_resources.py --images --all-images
```

#### Sample output

```
Containers pruned: 3 removed
Images pruned: 5 removed, 214582112 bytes reclaimed
Volumes pruned: 1 removed, 10485760 bytes reclaimed
Networks pruned: 2 removed
Total space reclaimed: 225067872 bytes
```

#### Variations / flags

- Add a `--dry-run` flag that lists what each `.prune()` call *would* remove using filter previews, without executing the deletion — useful before running on a shared or production host.
- Schedule `--all` (without `--all-images`, to be conservative) as a nightly cron job on CI/build hosts:

```bash
# crontab entry: prune every night at 2am
0 2 * * * /usr/bin/python3 /opt/scripts/docker_prune_resources.py --all >> /var/log/docker-prune.log 2>&1
```

- Add a `--older-than 24h` filter (Docker supports `until` filters) to only prune resources older than a given age, protecting recently built images still under test.

#### Common pitfalls

- **`--all-images` removing images you still need** — any image not currently backing a running container is fair game, including ones you built yesterday for tomorrow's release; tag anything you want to keep and check before running this in `--all-images` mode.
- **Networks in use aren't pruned, silently** — `prune_networks` only removes networks with zero connected containers; if you expected a network to disappear and it didn't, check for a stopped container still attached to it.
- **Running this against a production host** — pruning on a host actively serving traffic can remove images needed for an imminent rollback; restrict `--all-images` to build/CI hosts, not production nodes.

### 22.7 Monitor Container Logs

`docker_monitor_logs.py` — Stream and optionally filter real-time log output from a running (or stopped) container, for live debugging and monitoring.

#### Use case

A service in a container is misbehaving and an engineer needs to watch its logs live as requests come in, filtering out the noise to find only the `ERROR` lines, without opening a separate terminal and running raw `docker logs -f` plus a manual `grep`.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- The target container must already exist (running or stopped — stopped containers still have historical logs to read)
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

`--follow` blocks and streams indefinitely until interrupted (`Ctrl+C`) or the container stops — don't run it unattended in a script expecting a return value without a timeout wrapper. Filtering with `--filter` happens client-side after the daemon has already sent the full unfiltered stream, so it doesn't reduce the amount of data actually transferred from the daemon.

#### Script

```python
#!/usr/bin/env python3
"""
docker_monitor_logs.py
Stream and filter real-time log output from a running container, useful for
debugging and monitoring.

USAGE
    python docker_monitor_logs.py --name web --follow
    python docker_monitor_logs.py --name web --tail 100
    python docker_monitor_logs.py --name web --follow --filter ERROR

REQUIREMENTS
    - Docker Engine/Desktop installed and running, with the target container
      already created (running or stopped)
    - pip install -r requirements.txt

EXPECTED OUTPUT
    Log lines from the container are printed to stdout as they arrive.
    If --filter is given, only lines containing that substring (case
    insensitive) are printed, e.g.:
        [2026-07-17 10:00:01] ERROR: connection refused
    Press Ctrl+C to stop following.
"""

import argparse

import docker
from docker.errors import NotFound


def monitor(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
    except NotFound:
        print(f"Container '{args.name}' not found.")
        return

    print(f"Streaming logs for '{args.name}' (tail={args.tail}, follow={args.follow}) ...")
    try:
        for raw_line in container.logs(stream=True, follow=args.follow, tail=args.tail):
            line = raw_line.decode("utf-8", errors="replace").rstrip()
            if args.filter and args.filter.lower() not in line.lower():
                continue
            print(line)
    except KeyboardInterrupt:
        print("\nStopped following logs.")


def main():
    parser = argparse.ArgumentParser(description="Stream and filter logs from a Docker container.")
    parser.add_argument("--name", required=True, help="Container name or id")
    parser.add_argument("--tail", default="all", help="Number of lines to show from the end of the logs")
    parser.add_argument("--follow", action="store_true", help="Keep streaming new log lines")
    parser.add_argument("--filter", help="Only show lines containing this substring")
    args = parser.parse_args()
    monitor(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`container.logs(stream=True, ...)` returns a generator yielding raw byte chunks as the daemon produces them. Each chunk is decoded to text, and if `--filter` is set, lines not containing the (case-insensitive) substring are skipped before printing.

```bash
# Follow logs live
python docker_monitor_logs.py --name web --follow

# Show the last 100 lines only, don't follow
python docker_monitor_logs.py --name web --tail 100

# Follow logs, but only show lines mentioning ERROR
python docker_monitor_logs.py --name web --follow --filter ERROR
```

#### Sample output

```
Streaming logs for 'web' (tail=all, follow=True) ...
[2026-07-17 10:00:01] INFO: server started on port 80
[2026-07-17 10:00:05] INFO: request GET /health -> 200
[2026-07-17 10:00:12] ERROR: connection refused to upstream db
[2026-07-17 10:00:12] ERROR: request GET /api/users -> 500
^C
Stopped following logs.
```

#### Variations / flags

- Add a `--since 10m` flag mapped to the SDK's `since` parameter to only show logs from the last N minutes, avoiding a full-history dump on containers with heavy log volume.
- Add a `--regex` mode using `re.search` instead of a plain substring match for more precise filtering.
- Pipe output into a log aggregator or `tee` it to a file for later analysis: `python docker_monitor_logs.py --name web --follow | tee web.log`.

#### Common pitfalls

- **`--tail all` on a chatty, long-running container** — can print an enormous backlog before getting to the live tail; use a numeric `--tail` (e.g. `100`) in day-to-day debugging.
- **Log driver mismatch** — if the container's logging driver isn't `json-file` or `journald` (e.g. it's configured for `syslog` or a remote driver), `container.logs()` may return nothing; check the container's `--log-driver` setting.
- **Non-UTF-8 log output** — binary or non-UTF-8 bytes in application logs are replaced with a placeholder character (`errors="replace"`) rather than crashing the script, which can slightly obscure genuinely garbled output.

### 22.8 Execute Commands Inside Containers

`docker_exec_command.py` — Run an arbitrary shell command inside the filesystem of an already-running container, for quick administration or troubleshooting without opening an interactive shell.

#### Use case

A container is running but something looks off — an engineer wants to quickly check a config file's contents, list a directory, or check an environment variable inside the container, without attaching a full interactive `docker exec -it ... bash` session.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- The target container must be **running** (not just created or stopped)
- The command being run must actually exist inside the container's image (e.g. `curl` won't work if the image never installed it)
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

Commands run with `--user root` have full root privileges inside the container's namespace — treat this the same as any other root shell. This script does not provide an interactive TTY, so commands expecting interactive input (password prompts, interactive editors) will hang or fail; it's meant for single-shot, non-interactive commands.

#### Script

```python
#!/usr/bin/env python3
"""
docker_exec_command.py
Run a shell command directly inside a running container's filesystem, for
quick administration or troubleshooting tasks.

USAGE
    python docker_exec_command.py --name web --cmd "ls -la /app"
    python docker_exec_command.py --name web --cmd "cat /etc/os-release" --user root
    python docker_exec_command.py --name web --cmd "env" --workdir /app

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - The target container must already be running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    The stdout/stderr produced by the command inside the container, followed
    by the exit code, e.g.:
        total 24
        drwxr-xr-x 3 root root 4096 Jul 17 10:00 .
        ...
        Exit code: 0
"""

import argparse
import shlex

import docker
from docker.errors import APIError, NotFound


def exec_command(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
    except NotFound:
        print(f"Container '{args.name}' not found.")
        return

    if container.status != "running":
        print(f"Container '{args.name}' is not running (status={container.status}).")
        return

    cmd = shlex.split(args.cmd)
    try:
        result = container.exec_run(
            cmd,
            user=args.user or "",
            workdir=args.workdir or None,
            environment=None,
        )
        output = result.output.decode("utf-8", errors="replace")
        print(output)
        print(f"Exit code: {result.exit_code}")
    except APIError as e:
        print(f"Exec failed: {e}")


def main():
    parser = argparse.ArgumentParser(description="Execute a command inside a running Docker container.")
    parser.add_argument("--name", required=True, help="Container name or id")
    parser.add_argument("--cmd", required=True, help="Command to run, e.g. \"ls -la /app\"")
    parser.add_argument("--user", help="Run the command as this user, e.g. root")
    parser.add_argument("--workdir", help="Working directory inside the container")
    args = parser.parse_args()
    exec_command(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`shlex.split()` breaks the `--cmd` string into an argument list the way a shell would, respecting quoting. `container.exec_run()` then runs that command inside the container's namespace via the Docker Exec API and returns both the combined stdout/stderr output and the process's exit code.

```bash
# List files in /app
python docker_exec_command.py --name web --cmd "ls -la /app"

# Check the OS release info as root
python docker_exec_command.py --name web --cmd "cat /etc/os-release" --user root

# Dump environment variables from a specific working directory
python docker_exec_command.py --name web --cmd "env" --workdir /app
```

#### Sample output

```
total 24
drwxr-xr-x 3 root root 4096 Jul 17 10:00 .
drwxr-xr-x 1 root root 4096 Jul 17 09:55 ..
-rw-r--r-- 1 root root  512 Jul 17 09:58 app.py
-rw-r--r-- 1 root root  128 Jul 17 09:58 requirements.txt

Exit code: 0
```

```
NAME="Ubuntu"
VERSION="22.04.4 LTS (Jammy Jellyfish)"
ID=ubuntu

Exit code: 0
```

#### Variations / flags

- Add a `--env KEY=VALUE` repeatable flag (the underlying `exec_run` already supports an `environment` dict, currently hardcoded to `None` in this script) to inject temporary environment variables for the executed command only.
- Add a `--detach` mode using `exec_run(detach=True)` for fire-and-forget commands where you don't need to wait for output.
- Wrap this in a health-check script that runs an internal diagnostic command and parses the exit code to feed into monitoring.

#### Common pitfalls

- **Container not running** — `exec` only works on running containers; the script checks `container.status` first and exits cleanly rather than throwing a raw Docker API error.
- **Command not found inside the image** — if the base image is a minimal distro (e.g. `alpine` or `distroless`), common tools like `curl`, `bash`, or `ps` may simply not be installed; check the image's package manager or Dockerfile.
- **Quoting pitfalls with `shlex.split`** — complex commands involving pipes (`|`) or shell redirection (`>`) won't work as a single `--cmd` string, since `shlex.split` doesn't invoke a shell; wrap such commands as `sh -c "cmd1 | cmd2"` instead.
- **Permission errors on file access** — running as the container's default (often non-root) user can produce "permission denied" on paths owned by root inside the image; add `--user root` when appropriate.

### 22.9 Pull/Push Docker Images to Registries

`docker_registry_transfer.py` — Authenticate to a Docker registry (Docker Hub, AWS ECR, or any private registry) and pull or push images programmatically, streaming progress as it happens.

#### Use case

A deployment pipeline needs to log into a private registry using credentials injected from a secrets manager, pull the latest version of a base image, or push a freshly built image up to that registry as part of a release — all without a human typing `docker login` interactively.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Valid credentials for the target registry (for AWS ECR: a temporary password generated via `aws ecr get-login-password --region <region>`)
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

**Never hardcode real credentials directly in a script or commit them to version control.** Always pass credentials via environment variables or a secrets manager. Pushing to a public registry (like Docker Hub without a private repository) makes that image's contents world-readable — double check the repository's visibility settings before pushing anything containing internal code or secrets baked into layers.

#### Script

```python
#!/usr/bin/env python3
"""
docker_registry_transfer.py
Authenticate to a Docker registry (Docker Hub, AWS ECR, or any private
registry) and pull or push images programmatically.

USAGE
    Login:
        python docker_registry_transfer.py login --registry docker.io \
            --username myuser --password mypassword

    Pull:
        python docker_registry_transfer.py pull --image nginx:latest

    Push (tag your local image with the registry path first, e.g.
    myrepo/myapp:latest, then push it):
        python docker_registry_transfer.py push --image myrepo/myapp:latest

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - Valid credentials for the target registry
      (for AWS ECR, generate a temporary password with:
       aws ecr get-login-password --region <region>)
    - pip install -r requirements.txt

EXPECTED OUTPUT
    login -> "Login succeeded for registry 'docker.io' as 'myuser'."
    pull  -> streamed pull progress, then "Pulled image: nginx:latest"
    push  -> streamed push progress, then "Pushed image: myrepo/myapp:latest"

CAUTION
    Never hardcode real credentials in scripts or commit them to version
    control. Prefer environment variables or a secrets manager, e.g.:
        python docker_registry_transfer.py login --registry docker.io \
            --username myuser --password "$DOCKER_PASSWORD"
"""

import argparse

import docker
from docker.errors import APIError


def login(args):
    client = docker.from_env()
    try:
        resp = client.login(username=args.username, password=args.password, registry=args.registry)
        print(f"Login succeeded for registry '{args.registry}' as '{args.username}'. ({resp.get('Status', '')})")
    except APIError as e:
        print(f"Login failed: {e}")


def pull(args):
    client = docker.from_env()
    try:
        for line in client.api.pull(args.image, stream=True, decode=True):
            if "status" in line:
                progress = line.get("progress", "")
                print(f"{line['status']} {progress}")
        print(f"Pulled image: {args.image}")
    except APIError as e:
        print(f"Pull failed: {e}")


def push(args):
    client = docker.from_env()
    try:
        for line in client.api.push(args.image, stream=True, decode=True):
            if "status" in line:
                progress = line.get("progress", "")
                print(f"{line['status']} {progress}")
        print(f"Pushed image: {args.image}")
    except APIError as e:
        print(f"Push failed: {e}")


def main():
    parser = argparse.ArgumentParser(description="Pull/push Docker images to/from a registry.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_login = sub.add_parser("login")
    p_login.add_argument("--registry", required=True, help="e.g. docker.io or <account>.dkr.ecr.<region>.amazonaws.com")
    p_login.add_argument("--username", required=True)
    p_login.add_argument("--password", required=True)
    p_login.set_defaults(func=login)

    p_pull = sub.add_parser("pull")
    p_pull.add_argument("--image", required=True, help="e.g. nginx:latest or registry/repo:tag")
    p_pull.set_defaults(func=pull)

    p_push = sub.add_parser("push")
    p_push.add_argument("--image", required=True, help="Must already be tagged with the registry path")
    p_push.set_defaults(func=push)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`login` calls the SDK's top-level `client.login()`, which stores registry auth in the same credential store the `docker` CLI itself uses. `pull` and `push` use the lower-level `client.api` interface with `stream=True, decode=True` so each progress event (layer download/upload percentage) is yielded as a parsed dict and printed as it arrives, mirroring what you'd see in a terminal running `docker pull`/`docker push` directly.

```bash
# Log in to Docker Hub (use an env var for the password, never a literal)
python docker_registry_transfer.py login --registry docker.io \
    --username myuser --password "$DOCKER_PASSWORD"

# Pull an image
python docker_registry_transfer.py pull --image nginx:latest

# Push a locally built, already-tagged image
python docker_registry_transfer.py push --image myrepo/myapp:latest
```

```

```

```

Pulling fs layer 
Downloading [==================>                                ]  12.3MB/33.7MB
Downloading [===================================>               ]  25.1MB/33.7MB
Pull complete 
Pulled image: nginx:latest
```

```
Pushing [==========>                                        ]  8.192MB/40.1MB
Pushing [========================>                          ]  19.5MB/40.1MB
Pushed

```

#### Variations / flags

- For AWS ECR, generate the password first and feed it directly into the `login` command:

```bash
PASSWORD=$(aws ecr get-login-password --region us-east-1)
python docker_registry_transfer.py login \
    --registry 123456789012.dkr.ecr.us-east-1.amazonaws.com \
    --username AWS --password "$PASSWORD"
```

- Add a `--all-tags` flag to `pull` mapped to the SDK's `all_tags=True` to fetch every tag of a repository at once.
- Add retry-with-backoff logic around `pull`/`push` for flaky network conditions common in CI runners.

#### Common pitfalls

- **Credentials in shell history** — passing `--password` directly on the command line leaves it in your shell history; always source it from an environment variable or a secrets manager instead.
- **Forgetting to tag before push** — `push` will fail if `--image` isn't already tagged with the target registry's path (e.g. pushing `myapp:latest` to Docker Hub under a specific account requires it be tagged `myuser/myapp:latest` first via a separate `docker tag` or SDK call).
- **ECR password expiry** — AWS ECR tokens from `get-login-password` expire (typically after 12 hours); a long-running pipeline needs to refresh and re-login periodically, not log in once at the start.
- **Rate limiting on Docker Hub** — anonymous or free-tier pulls are rate-limited; authenticate even for pulls if you're hitting Hub frequently in CI.

### 22.10 Create Virtual Machines (libvirt/VirtualBox)

`vm_create_libvirt.py` — Provision a complete virtual machine (CPU, RAM, and a fresh disk image) through the libvirt hypervisor API, booting from an installer ISO on KVM/QEMU.

#### Use case

A sysadmin needs to spin up a test VM running a different Linux distribution or kernel version than the host, to validate a driver, test a kernel-level change, or reproduce an OS-specific bug that a container simply can't isolate.

#### Prerequisites

- A working libvirt/KVM hypervisor host — **Linux only**
- `qemu-img` available on PATH (used to create the disk image)
- System libvirt development headers, needed to build the Python bindings

```bash
sudo apt install qemu-kvm libvirt-daemon-system libvirt-clients virtinst libvirt-dev
pip install libvirt-python
```

- The invoking user should be in the `libvirt` group, or the script needs to run with `sudo`

#### Cautions

This script talks directly to the hypervisor and creates **real disk images and VM definitions** on the host — double-check `--disk-path` and available disk space before running, since a botched `--disk-size-gb` can silently eat a large chunk of the host's storage.

#### Script

```python
#!/usr/bin/env python3
"""
vm_create_libvirt.py
Provision a complete virtual machine (CPU, RAM, disk) through the libvirt
hypervisor API. Works with KVM/QEMU on Linux.

USAGE
    python vm_create_libvirt.py --name test-vm --vcpus 2 --memory-mb 2048 \
        --disk-path /var/lib/libvirt/images/test-vm.qcow2 --disk-size-gb 20 \
        --iso /var/lib/libvirt/images/ubuntu-22.04.iso

REQUIREMENTS
    - A working libvirt/KVM hypervisor host (Linux only). Install with:
        sudo apt install qemu-kvm libvirt-daemon-system libvirt-clients virtinst
    - System libvirt development headers for the Python bindings:
        sudo apt install libvirt-dev
    - The user running this script should be in the 'libvirt' group,
      or you run it with sudo
    - pip install -r requirements.txt   (installs libvirt-python)
    - qemu-img available on PATH (used to create the disk image)

EXPECTED OUTPUT
    A raw disk image is created at --disk-path, the VM is defined and started,
    and the script prints:
        Disk image created: /var/lib/libvirt/images/test-vm.qcow2 (20G)
        Domain 'test-vm' defined and started (state=running)

CAUTION
    This script talks directly to your hypervisor and will create real disk
    images and VM definitions on the host. Double-check --disk-path and
    available disk space before running.
"""

import argparse
import subprocess
import sys

import libvirt

DOMAIN_XML_TEMPLATE = """
<domain type='kvm'>
  <name>{name}</name>
  <memory unit='MiB'>{memory_mb}</memory>
  <vcpu>{vcpus}</vcpu>
  <os>
    <type arch='x86_64'>hvm</type>
    <boot dev='cdrom'/>
    <boot dev='hd'/>
  </os>
  <features><acpi/><apic/></features>
  <devices>
    <disk type='file' device='disk'>
      <driver name='qemu' type='qcow2'/>
      <source file='{disk_path}'/>
      <target dev='vda' bus='virtio'/>
    </disk>
    {cdrom_xml}
    <interface type='network'>
      <source network='default'/>
      <model type='virtio'/>
    </interface>
    <graphics type='vnc' port='-1' autoport='yes'/>
    <console type='pty'/>
  </devices>
</domain>
"""

CDROM_XML_TEMPLATE = """
    <disk type='file' device='cdrom'>
      <driver name='qemu' type='raw'/>
      <source file='{iso_path}'/>
      <target dev='sda' bus='sata'/>
      <readonly/>
    </disk>
"""


def create_disk(disk_path, size_gb):
    subprocess.run(
        ["qemu-img", "create", "-f", "qcow2", disk_path, f"{size_gb}G"],
        check=True,
    )
    print(f"Disk image created: {disk_path} ({size_gb}G)")


def create_vm(args):
    create_disk(args.disk_path, args.disk_size_gb)

    cdrom_xml = CDROM_XML_TEMPLATE.format(iso_path=args.iso) if args.iso else ""
    domain_xml = DOMAIN_XML_TEMPLATE.format(
        name=args.name,
        memory_mb=args.memory_mb,
        vcpus=args.vcpus,
        disk_path=args.disk_path,
        cdrom_xml=cdrom_xml,
    )

    conn = libvirt.open(args.connect)
    if conn is None:
        print(f"Failed to open connection to {args.connect}")
        sys.exit(1)

    try:
        domain = conn.defineXML(domain_xml)
        domain.create()
        state, _ = domain.state()
        state_name = {1: "running", 3: "paused", 5: "shutoff"}.get(state, str(state))
        print(f"Domain '{args.name}' defined and started (state={state_name})")
    finally:
        conn.close()


def main():
    parser = argparse.ArgumentParser(description="Create a VM via libvirt.")
    parser.add_argument("--name", required=True)
    parser.add_argument("--vcpus", type=int, default=2)
    parser.add_argument("--memory-mb", type=int, default=2048)
    parser.add_argument("--disk-path", required=True)
    parser.add_argument("--disk-size-gb", type=int, default=20)
    parser.add_argument("--iso", help="Path to installer ISO (optional)")
    parser.add_argument("--connect", default="qemu:///system", help="libvirt connection URI")
    args = parser.parse_args()
    create_vm(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script first shells out to `qemu-img` to create an empty qcow2 disk image of the requested size, then builds a libvirt domain XML definition from a template (name, memory, vCPUs, disk path, and an optional CD-ROM device pointing at an installer ISO), and calls `conn.defineXML()` followed by `domain.create()` to define and immediately boot the VM.

```bash
python vm_create_libvirt.py --name test-vm --vcpus 2 --memory-mb 2048 \
    --disk-path /var/lib/libvirt/images/test-vm.qcow2 --disk-size-gb 20 \
    --iso /var/lib/libvirt/images/ubuntu-22.04.iso
```

#### Sample output

```
Disk image created: /var/lib/libvirt/images/test-vm.qcow2 (20G)
Domain 'test-vm' defined and started (state=running)
```

#### Variations / flags

- Add a `--network <name>` flag to attach to a custom libvirt network instead of the hardcoded `default` NAT network.
- Add a `--graphics none` mode for headless VMs managed purely over serial console, useful for automated provisioning without VNC.
- Follow VM creation with `virsh console test-vm` or a VNC client pointed at the auto-assigned port to complete an unattended install, or supply a `cloud-init` seed ISO instead of an interactive installer for fully automated provisioning.

#### Common pitfalls

- **Not in the `libvirt` group** — connecting to `qemu:///system` without proper group membership or `sudo` fails silently with `conn` returning `None`; check `groups $(whoami)` includes `libvirt` or `kvm`.
- **Insufficient disk space** — `qemu-img create` on qcow2 is sparse (thin-provisioned) by default, so the file won't immediately consume the full `--disk-size-gb`, but it *can* grow to that size — make sure the host has room.
- **Missing hardware virtualization support** — KVM requires CPU virtualization extensions (Intel VT-x / AMD-V) enabled in BIOS/firmware; without them, `domain.create()` will fail.
- **Name collision** — defining a domain with a name that already exists in libvirt will fail; check `virsh list --all` first or catch the `libvirtError`.

### 22.11 Clone Virtual Machines

`vm_clone.py` — Duplicate an entire libvirt VM instance, including its disk(s) and configuration, for rapid environment replication, by wrapping the `virt-clone` utility.

#### Use case

A team maintains a "golden image" VM with a base OS and common tooling pre-installed, and needs to spin up several near-identical copies for different test scenarios without repeating the OS install and configuration steps each time.

#### Prerequisites

- A working libvirt/KVM hypervisor host — **Linux only**
- `virt-clone` installed (part of the `virtinst` package)
- Sufficient free disk space to hold a full copy of the source VM's disk(s)

```bash
sudo apt install virtinst
```

#### Cautions

The source VM **should be shut off** before cloning — cloning a running VM risks copying a disk image that's still being actively written to, producing a corrupted or inconsistent clone.

#### Script

```python
#!/usr/bin/env python3
"""
vm_clone.py
Duplicate an entire libvirt VM instance, including its disks and settings,
for rapid environment replication. Wraps the battle-tested `virt-clone`
utility so disk copying and MAC address regeneration are handled correctly.

USAGE
    python vm_clone.py --original test-vm --clone test-vm-copy
    python vm_clone.py --original test-vm --clone test-vm-copy \
        --new-disk-path /var/lib/libvirt/images/test-vm-copy.qcow2

REQUIREMENTS
    - A working libvirt/KVM hypervisor host (Linux only)
    - virt-clone (part of virtinst) installed:
        sudo apt install virtinst
    - Sufficient disk space to hold a full copy of the source VM's disk(s)
    - pip install -r requirements.txt

EXPECTED OUTPUT
    virt-clone's own progress output while it copies the disk image(s),
    followed by:
        Clone complete: 'test-vm-copy' cloned from 'test-vm'

CAUTION
    The source VM should be shut off before cloning to avoid copying a disk
    that is still being written to.
"""

import argparse
import subprocess
import sys


def clone_vm(args):
    cmd = ["virt-clone", "--original", args.original, "--name", args.clone]
    if args.new_disk_path:
        cmd += ["--file", args.new_disk_path]
    else:
        cmd += ["--auto-clone"]

    print(f"Running: {' '.join(cmd)}")
    result = subprocess.run(cmd, capture_output=True, text=True)
    print(result.stdout)
    if result.returncode != 0:
        print(result.stderr)
        print(f"Clone failed with exit code {result.returncode}")
        sys.exit(result.returncode)

    print(f"Clone complete: '{args.clone}' cloned from '{args.original}'")


def main():
    parser = argparse.ArgumentParser(description="Clone a libvirt VM using virt-clone.")
    parser.add_argument("--original", required=True, help="Name of the source VM")
    parser.add_argument("--clone", required=True, help="Name for the new cloned VM")
    parser.add_argument("--new-disk-path", help="Explicit path for the cloned disk (optional; auto-generated if omitted)")
    args = parser.parse_args()
    clone_vm(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script is a thin wrapper around the `virt-clone` CLI. If `--new-disk-path` isn't given, it passes `--auto-clone`, letting `virt-clone` pick a sensible new disk path and automatically regenerate the clone's MAC address and UUID so it doesn't collide with the original on the same network.

```bash
# Auto-generate the clone's disk path
python vm_clone.py --original test-vm --clone test-vm-copy

# Specify an explicit disk path for the clone
python vm_clone.py --original test-vm --clone test-vm-copy \
    --new-disk-path /var/lib/libvirt/images/test-vm-copy.qcow2
```

#### Sample output

```
Running: virt-clone --original test-vm --name test-vm-copy --auto-clone

Allocating 'test-vm-copy.qcow2'                    |  20 GB  00:00:14     

Clone 'test-vm-copy' created successfully.
Clone complete: 'test-vm-copy' cloned from 'test-vm'
```

#### Variations / flags

- Add a `--force` flag mapped to `virt-clone --force-copy` if disk copy-on-write acceleration should be bypassed (e.g. cloning to a different storage pool/filesystem).
- Loop this script to spin up a batch of numbered clones (`test-vm-copy-1`, `test-vm-copy-2`, ...) for a load-testing farm sharing the same golden image.
- Combine with `vm_snapshot.py` (Section 22.12): clone a VM, then immediately snapshot the clone as a clean baseline before running destructive tests against it.

#### Common pitfalls

- **Cloning a running VM** — produces a warning or outright failure from `virt-clone`; always `virsh shutdown <name>` (or `virsh destroy` for a hard stop) first.
- **Insufficient disk space** — `virt-clone` performs a full disk copy by default (not a copy-on-write overlay), so cloning a 20 GB disk needs at least another 20 GB free; check `df -h` before cloning large VMs.
- **Name or MAC collisions on the network** — if `--auto-clone` isn't used and MAC regeneration is skipped, two VMs with identical MAC addresses on the same network segment can cause intermittent connectivity issues.
- **`virt-clone` not installed** — a common miss on minimal libvirt setups; the `virtinst` package is separate from the core `libvirt-daemon-system` package.

### 22.12 Snapshot Virtual Machines

`vm_snapshot.py` — Capture, list, revert to, and delete point-in-time snapshots of a libvirt VM, the safety net for testing risky changes.

#### Use case

Before applying a potentially destructive OS update, config change, or software install inside a test VM, an engineer takes a snapshot. If the change goes badly, they revert to the snapshot in seconds instead of rebuilding the VM from scratch.

#### Prerequisites

- A working libvirt/KVM hypervisor host — **Linux only**
- System libvirt development headers for the Python bindings
- The target domain must already exist (see Section 22.10, `vm_create_libvirt.py`)
- The VM's disk must use the **qcow2** format for internal (default) snapshots to work

```bash
sudo apt install libvirt-dev
pip install libvirt-python
```

#### Cautions

Internal (default) snapshots are stored inside the qcow2 disk file itself; reverting to a snapshot discards any state changes made after that snapshot was taken, permanently, unless another snapshot was taken first to preserve that state.

#### Script

```python
#!/usr/bin/env python3
"""
vm_snapshot.py
Capture, list, and restore point-in-time snapshots of a libvirt VM, useful
for testing and rollback scenarios.

USAGE
    python vm_snapshot.py create --domain test-vm --name before-update
    python vm_snapshot.py list --domain test-vm
    python vm_snapshot.py revert --domain test-vm --name before-update
    python vm_snapshot.py delete --domain test-vm --name before-update

REQUIREMENTS
    - A working libvirt/KVM hypervisor host (Linux only)
    - System libvirt development headers:
        sudo apt install libvirt-dev
    - pip install -r requirements.txt   (installs libvirt-python)
    - The target domain must already exist (see vm_create_libvirt.py)

EXPECTED OUTPUT
    create  -> "Snapshot 'before-update' created for domain 'test-vm'."
    list    -> a list of snapshot names for the domain
    revert  -> "Domain 'test-vm' reverted to snapshot 'before-update'."
    delete  -> "Snapshot 'before-update' deleted."

CAUTION
    Internal (default) snapshots are stored inside the qcow2 disk file itself
    and require the VM's disk to use the qcow2 format.
"""

import argparse
import sys

import libvirt

SNAPSHOT_XML_TEMPLATE = """
<domainsnapshot>
  <name>{name}</name>
  <description>Snapshot created by vm_snapshot.py</description>
</domainsnapshot>
"""


def get_domain(conn, name):
    try:
        return conn.lookupByName(name)
    except libvirt.libvirtError:
        print(f"Domain '{name}' not found.")
        sys.exit(1)


def create_snapshot(conn, args):
    domain = get_domain(conn, args.domain)
    xml = SNAPSHOT_XML_TEMPLATE.format(name=args.name)
    domain.snapshotCreateXML(xml, 0)
    print(f"Snapshot '{args.name}' created for domain '{args.domain}'.")


def list_snapshots(conn, args):
    domain = get_domain(conn, args.domain)
    names = domain.snapshotListNames()
    if not names:
        print(f"No snapshots found for domain '{args.domain}'.")
        return
    for n in names:
        print(n)


def revert_snapshot(conn, args):
    domain = get_domain(conn, args.domain)
    snapshot = domain.snapshotLookupByName(args.name, 0)
    domain.revertToSnapshot(snapshot, 0)
    print(f"Domain '{args.domain}' reverted to snapshot '{args.name}'.")


def delete_snapshot(conn, args):
    domain = get_domain(conn, args.domain)
    snapshot = domain.snapshotLookupByName(args.name, 0)
    snapshot.delete(0)
    print(f"Snapshot '{args.name}' deleted.")


def main():
    parser = argparse.ArgumentParser(description="Manage libvirt VM snapshots.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_create = sub.add_parser("create")
    p_create.add_argument("--domain", required=True)
    p_create.add_argument("--name", required=True)
    p_create.set_defaults(func=create_snapshot)

    p_list = sub.add_parser("list")
    p_list.add_argument("--domain", required=True)
    p_list.set_defaults(func=list_snapshots)

    p_revert = sub.add_parser("revert")
    p_revert.add_argument("--domain", required=True)
    p_revert.add_argument("--name", required=True)
    p_revert.set_defaults(func=revert_snapshot)

    p_delete = sub.add_parser("delete")
    p_delete.add_argument("--domain", required=True)
    p_delete.add_argument("--name", required=True)
    p_delete.set_defaults(func=delete_snapshot)

    parser.add_argument("--connect", default="qemu:///system", help="libvirt connection URI")
    args = parser.parse_args()

    conn = libvirt.open(args.connect)
    if conn is None:
        print(f"Failed to open connection to {args.connect}")
        sys.exit(1)
    try:
        args.func(conn, args)
    finally:
        conn.close()


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`create` builds a minimal `<domainsnapshot>` XML document and calls `domain.snapshotCreateXML()`. `list` calls `domain.snapshotListNames()` to enumerate existing snapshots by name. `revert` looks up a snapshot by name and calls `domain.revertToSnapshot()`, rolling the VM's disk and (if the VM was running) memory state back to that point. `delete` removes a snapshot definition (and its associated internal qcow2 data) permanently.

```bash
# Take a snapshot before a risky change
python vm_snapshot.py create --domain test-vm --name before-update

# List all snapshots for this VM
python vm_snapshot.py list --domain test-vm

# Something went wrong -- revert
python vm_snapshot.py revert --domain test-vm --name before-update

# Clean up an old snapshot no longer needed
python vm_snapshot.py delete --domain test-vm --name before-update
```

```

after-package-install
```


#### Variations / flags

- Add a `--description "text"` flag to attach meaningful notes to each snapshot, since the current template hardcodes a generic description.
- Wrap `create` into a pre-deployment script that automatically snapshots a fleet of test VMs before a batch config push runs against them.
- For external (file-based) snapshots instead of internal qcow2 snapshots — useful for backing up snapshot data separately from the live disk — extend the XML template with `<disks><disk name='vda' snapshot='external'/></disks>`.

#### Common pitfalls

- **Non-qcow2 disks** — internal snapshots silently fail or are rejected on raw-format disks; check the VM's disk driver type (`qemu... type='qcow2'`) in its domain XML before relying on this workflow.
- **Snapshot sprawl** — each internal snapshot adds overhead to the qcow2 file and slightly degrades disk I/O performance the more snapshots accumulate; delete snapshots you no longer need rather than letting them pile up indefinitely.
- **Reverting discards forward progress** — reverting to an old snapshot throws away every change made since, with no separate confirmation step in this script; consider snapshotting the current state before reverting if there's any chance you'll want it back.
- **Domain not found** — a typo in `--domain` produces a clear "not found" message and a clean exit here, rather than a raw libvirt exception, but double-check `virsh list --all` for the exact registered name.

### 22.13 Manage Vagrant Environments

`vagrant_manage.py` — Bring up, check the status of, provision, halt, and destroy Vagrant-managed development VMs programmatically.

#### Use case

A development team shares a `Vagrantfile`-defined environment (a consistent OS, dependencies, and network setup) for local development. A new team member, or a CI job validating the environment definition itself, needs to bring the box up, confirm it started correctly, and tear it down again — scripted, not manual.

#### Prerequisites

- Vagrant installed and on PATH
- A hypervisor provider installed — VirtualBox, libvirt, or another Vagrant-supported provider
- A `Vagrantfile` present in the target directory
- OS scope: Linux, macOS, Windows (Vagrant itself is cross-platform; the underlying provider may not be)

```bash
pip install python-vagrant
```

#### Cautions

`destroy` permanently deletes the VM and its disk — any files or state inside the guest that weren't synced to a host folder or committed elsewhere are gone. Vagrant's `up` will silently do nothing useful if run against a directory without a valid `Vagrantfile`; always confirm `--dir` is correct.

#### Script

```python
#!/usr/bin/env python3
"""
vagrant_manage.py
Control Vagrant development environments programmatically: spin up,
provision, halt, or destroy Vagrant boxes.

USAGE
    python vagrant_manage.py up --dir ./my-vagrant-project
    python vagrant_manage.py status --dir ./my-vagrant-project
    python vagrant_manage.py provision --dir ./my-vagrant-project
    python vagrant_manage.py halt --dir ./my-vagrant-project
    python vagrant_manage.py destroy --dir ./my-vagrant-project --force

REQUIREMENTS
    - Vagrant installed and on PATH (https://www.vagrantup.com/downloads)
    - A hypervisor provider installed (VirtualBox, libvirt, etc.)
    - A Vagrantfile present in --dir
    - pip install -r requirements.txt   (installs python-vagrant)

EXPECTED OUTPUT
    up         -> Vagrant's own boot log, ending with "Environment is up."
    status     -> current machine status, e.g. "default: running"
    provision  -> provisioner output, ending with "Provisioning complete."
    halt       -> "Environment halted."
    destroy    -> "Environment destroyed."
"""

import argparse

import vagrant


def up(v, args):
    v.up(provision=args.provision)
    print("Environment is up.")


def status(v, args):
    for s in v.status():
        print(f"{s.name}: {s.state}")


def provision(v, args):
    v.provision()
    print("Provisioning complete.")


def halt(v, args):
    v.halt()
    print("Environment halted.")


def destroy(v, args):
    v.destroy()
    print("Environment destroyed.")


def main():
    parser = argparse.ArgumentParser(description="Manage Vagrant environments programmatically.")
    parser.add_argument("command", choices=["up", "status", "provision", "halt", "destroy"])
    parser.add_argument("--dir", required=True, help="Path to the directory containing the Vagrantfile")
    parser.add_argument("--provision", action="store_true", help="Run provisioners when bringing the machine up")
    parser.add_argument("--force", action="store_true", help="(kept for CLI familiarity; python-vagrant destroy is non-interactive)")
    args = parser.parse_args()

    v = vagrant.Vagrant(root=args.dir, quiet_stdout=False, quiet_stderr=False)

    dispatch = {
        "up": up,
        "status": status,
        "provision": provision,
        "halt": halt,
        "destroy": destroy,
    }
    dispatch[args.command](v, args)


if __name__ == "__main__":
    main()
```


#### Variations / flags

- Add a `--provider libvirt` passthrough for teams running Vagrant on Linux hosts with libvirt instead of VirtualBox.
- Add an `ssh-config` subcommand wrapping `v.ssh_config()` to programmatically retrieve connection details for feeding into other automation (e.g. Ansible's dynamic inventory).

### 22.14 LXC/LXD Container Management

`lxc_manage.py` — Launch, list, stop, start, and delete lightweight LXD system containers — full OS environments with near-native performance — using the `pylxd` library.

#### Use case

An engineer needs a lightweight, fast-booting environment that behaves like a full Linux OS (with its own init system, multiple processes, systemd services) rather than the single-process-focused model of a Docker application container — for example, to test a systemd service unit or replicate a full server environment for training or CI purposes.

#### Prerequisites

- LXD installed and initialized on the host
- The invoking user in the `lxd` group, or run with `sudo`
- LXD's local UNIX socket reachable (default: `/var/snap/lxd/common/lxd/unix.socket`)
- OS scope: **Linux only**

```bash
sudo snap install lxd
sudo lxd init
pip install pylxd
```

#### Cautions

`delete --force` will stop a running container before deleting it, discarding all in-container state permanently — there's no confirmation prompt in this script, so double-check `--name` before running delete.

#### Script

```python
#!/usr/bin/env python3
"""
lxc_manage.py
Manage lightweight LXD system containers (full OS environments with
near-native performance) programmatically via the pylxd library.

USAGE
    python lxc_manage.py launch --name test-ct --image ubuntu:22.04
    python lxc_manage.py list
    python lxc_manage.py stop --name test-ct
    python lxc_manage.py start --name test-ct
    python lxc_manage.py delete --name test-ct --force

REQUIREMENTS
    - LXD installed and initialized on this machine:
        sudo snap install lxd
        sudo lxd init
    - Your user must be in the 'lxd' group, or run with sudo
    - pip install -r requirements.txt   (installs pylxd)
    - LXD's local UNIX socket must be reachable (default: /var/snap/lxd/common/lxd/unix.socket)

EXPECTED OUTPUT
    launch -> "Container 'test-ct' launched from image 'ubuntu:22.04' (status=Running)."
    list   -> table of container names and statuses
    stop   -> "Container 'test-ct' stopped."
    start  -> "Container 'test-ct' started."
    delete -> "Container 'test-ct' deleted."
"""

import argparse
import sys

import pylxd
from pylxd.exceptions import LXDAPIException, NotFound


def get_client():
    try:
        return pylxd.Client()
    except Exception as e:
        print(f"Could not connect to the local LXD daemon: {e}")
        sys.exit(1)


def launch(args):
    client = get_client()
    config = {
        "name": args.name,
        "source": {"type": "image", "alias": args.image},
    }
    container = client.containers.create(config, wait=True)
    container.start(wait=True)
    container.sync()
    print(f"Container '{args.name}' launched from image '{args.image}' (status={container.status}).")


def list_containers(args):
    client = get_client()
    containers = client.containers.all()
    if not containers:
        print("No containers found.")
        return
    print(f"{'NAME':25} {'STATUS':15}")
    for c in containers:
        print(f"{c.name:25} {c.status:15}")


def stop(args):
    client = get_client()
    try:
        container = client.containers.get(args.name)
        container.stop(wait=True)
        print(f"Container '{args.name}' stopped.")
    except NotFound:
        print(f"Container '{args.name}' not found.")


def start(args):
    client = get_client()
    try:
        container = client.containers.get(args.name)
        container.start(wait=True)
        print(f"Container '{args.name}' started.")
    except NotFound:
        print(f"Container '{args.name}' not found.")


def delete(args):
    client = get_client()
    try:
        container = client.containers.get(args.name)
        if container.status == "Running" and args.force:
            container.stop(wait=True)
        container.delete(wait=True)
        print(f"Container '{args.name}' deleted.")
    except NotFound:
        print(f"Container '{args.name}' not found.")
    except LXDAPIException as e:
        print(f"Could not delete container (is it still running? use --force): {e}")


def main():
    parser = argparse.ArgumentParser(description="Manage LXC/LXD containers programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_launch = sub.add_parser("launch")
    p_launch.add_argument("--name", required=True)
    p_launch.add_argument("--image", required=True, help="Image alias, e.g. ubuntu:22.04")
    p_launch.set_defaults(func=launch)

    p_list = sub.add_parser("list")
    p_list.set_defaults(func=list_containers)

    p_stop = sub.add_parser("stop")
    p_stop.add_argument("--name", required=True)
    p_stop.set_defaults(func=stop)

    p_start = sub.add_parser("start")
    p_start.add_argument("--name", required=True)
    p_start.set_defaults(func=start)

    p_delete = sub.add_parser("delete")
    p_delete.add_argument("--name", required=True)
    p_delete.add_argument("--force", action="store_true", help="Stop the container first if it is running")
    p_delete.set_defaults(func=delete)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`pylxd.Client()` connects to the local LXD daemon over its UNIX socket. `launch` creates a container from a remote image alias (e.g. `ubuntu:22.04` from the default LXD image server) and immediately starts it. `stop`/`start`/`delete` look the container up by name and act on its lifecycle; `delete --force` stops a running container first since LXD refuses to delete a running one outright.

```bash

python lxc_manage.py launch --name test-ct --image ubuntu:22.04

# List all containers and their status
python lxc_manage.py list

# Stop it
python lxc_manage.py stop --name test-ct

# Start it again
python lxc_manage.py start --name test-ct

# Delete it, stopping first if still running
python lxc_manage.py delete --name test-ct --force
```

#### Sample output

```

```

```
NAME                      STATUS         
test-ct                   Running        
old-test-box              Stopped        
```

```
Container 'test-ct' stopped.
```

```
Container 'test-ct' started.
```

```

```

#### Variations / flags

- Add a `--profile` flag to launch containers with a specific LXD profile applied (custom resource limits, network config, or device passthrough).
- Add an `exec` subcommand wrapping `container.execute([...])` for running commands inside the system container, similar to Section 22.8's Docker equivalent.
- Combine with cloud-init-style provisioning via LXD's `user.user-data` config key for automated in-container setup at launch time.

#### Common pitfalls

- **Not in the `lxd` group** — `pylxd.Client()` fails to connect to the socket without proper group membership; `newgrp lxd` or a fresh login session after `usermod -aG lxd $USER` is often needed.
- **Image alias not found remotely** — image aliases like `ubuntu:22.04` are resolved against LXD's configured remote image servers; if the remote is unreachable or misconfigured, `launch` will hang or fail — check `lxc remote list`.
- **Confusing LXC with Docker semantics** — LXD containers behave like lightweight VMs (full init system, multiple services, persistent state by default) rather than Docker's single-process application container model; scripts and workflows built around Docker assumptions won't map directly.
- **Deleting without `--force` while running** — fails with an `LXDAPIException` rather than silently stopping the container first; the flag must be passed explicitly.

### 22.15 Kubernetes Pod Management

`k8s_pod_management.py` — Create, list, check status of, and delete individual pods within a Kubernetes cluster, using the official Kubernetes Python client.

#### Use case

An engineer needs to quickly spin up a one-off pod for debugging (e.g. a throwaway pod running a network utility image to test connectivity from inside the cluster), check whether it's actually running, and clean it up afterward — without hand-typing several `kubectl` commands.

#### Prerequisites

- A reachable Kubernetes cluster and a valid kubeconfig (default: `~/.kube/config`, or set via the `KUBECONFIG` environment variable)
- Sufficient RBAC permissions to create/list/delete pods in the target namespace
- OS scope: Linux, macOS, Windows (anywhere `kubectl`/kubeconfig access works)

```bash
pip install kubernetes
```

#### Cautions

Direct pod creation (as opposed to a Deployment, Section 22.16) means Kubernetes will **not** automatically recreate the pod if it crashes or the node it's on fails — this script is best suited for short-lived, disposable debugging pods, not production workloads.

#### Script

```python
#!/usr/bin/env python3
"""
k8s_pod_management.py
Create, monitor, and terminate pods within a Kubernetes cluster using the
official Kubernetes Python client.

USAGE
    python k8s_pod_management.py create --name test-pod --image nginx:latest --namespace default
    python k8s_pod_management.py list --namespace default
    python k8s_pod_management.py status --name test-pod --namespace default
    python k8s_pod_management.py delete --name test-pod --namespace default

REQUIREMENTS
    - A reachable Kubernetes cluster and a valid kubeconfig
      (default location: ~/.kube/config, or set KUBECONFIG)
    - Sufficient RBAC permissions to create/list/delete pods in the namespace
    - pip install -r requirements.txt   (installs the `kubernetes` package)

EXPECTED OUTPUT
    create -> "Pod 'test-pod' created in namespace 'default'."
    list   -> table of pod names, phases and node assignments
    status -> "Pod 'test-pod' phase: Running"
    delete -> "Pod 'test-pod' deleted from namespace 'default'."
"""

import argparse

from kubernetes import client, config
from kubernetes.client.rest import ApiException


def load_client():
    config.load_kube_config()
    return client.CoreV1Api()


def create_pod(args):
    v1 = load_client()
    pod = client.V1Pod(
        metadata=client.V1ObjectMeta(name=args.name),
        spec=client.V1PodSpec(
            containers=[client.V1Container(name=args.name, image=args.image)]
        ),
    )
    try:
        v1.create_namespaced_pod(namespace=args.namespace, body=pod)
        print(f"Pod '{args.name}' created in namespace '{args.namespace}'.")
    except ApiException as e:
        print(f"Failed to create pod: {e}")


def list_pods(args):
    v1 = load_client()
    pods = v1.list_namespaced_pod(namespace=args.namespace)
    if not pods.items:
        print(f"No pods found in namespace '{args.namespace}'.")
        return
    print(f"{'NAME':30} {'PHASE':12} {'NODE'}")
    for p in pods.items:
        node = p.spec.node_name or "-"
        print(f"{p.metadata.name:30} {p.status.phase:12} {node}")


def pod_status(args):
    v1 = load_client()
    try:
        pod = v1.read_namespaced_pod_status(name=args.name, namespace=args.namespace)
        print(f"Pod '{args.name}' phase: {pod.status.phase}")
    except ApiException as e:
        print(f"Failed to read pod status: {e}")


def delete_pod(args):
    v1 = load_client()
    try:
        v1.delete_namespaced_pod(name=args.name, namespace=args.namespace)
        print(f"Pod '{args.name}' deleted from namespace '{args.namespace}'.")
    except ApiException as e:
        print(f"Failed to delete pod: {e}")


def main():
    parser = argparse.ArgumentParser(description="Manage Kubernetes pods programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_create = sub.add_parser("create")
    p_create.add_argument("--name", required=True)
    p_create.add_argument("--image", required=True)
    p_create.add_argument("--namespace", default="default")
    p_create.set_defaults(func=create_pod)

    p_list = sub.add_parser("list")
    p_list.add_argument("--namespace", default="default")
    p_list.set_defaults(func=list_pods)

    p_status = sub.add_parser("status")
    p_status.add_argument("--name", required=True)
    p_status.add_argument("--namespace", default="default")
    p_status.set_defaults(func=pod_status)

    p_delete = sub.add_parser("delete")
    p_delete.add_argument("--name", required=True)
    p_delete.add_argument("--namespace", default="default")
    p_delete.set_defaults(func=delete_pod)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`config.load_kube_config()` reads the local kubeconfig to establish cluster credentials, exactly as `kubectl` does. `client.CoreV1Api()` exposes the low-level pod API; `create_pod` builds a minimal `V1Pod` object with a single container and submits it via `create_namespaced_pod()`. `list`, `status`, and `delete` are thin wrappers around their respective `CoreV1Api` methods.

```
other-app-7f9c8d-x2j4k         Running      node-2
```

#### Variations / flags

- Add a `--command` flag to override the container's entrypoint, useful for spinning up a debug pod running `sleep infinity` to `kubectl exec` into later.
- Add a `--labels key=value` repeatable flag so pods created by this script can be filtered/selected later via label selectors.

#### Common pitfalls

- **Pod stuck in `Pending`** — usually means the cluster has no node with enough free resources to schedule it, or the image can't be pulled; `kubectl describe pod <name>` gives more detail than this script's simple phase check.

### 22.16 Deploy to Kubernetes Clusters

`k8s_deploy.py` — Apply a Kubernetes Deployment manifest, check rollout status, scale replicas, and delete deployments, using the official Kubernetes Python client.

#### Use case

A release pipeline needs to apply an updated Deployment manifest (new image version, new resource requests) to a cluster, wait until the rollout completes, and optionally scale the number of replicas up or down in response to traffic — the standard production deployment lifecycle.

#### Prerequisites

- A reachable Kubernetes cluster and a valid kubeconfig
- A Deployment manifest YAML file (`apiVersion: apps/v1`, `kind: Deployment`)
- OS scope: Linux, macOS, Windows

```bash
pip install kubernetes pyyaml
```

#### Cautions

`delete` removes the Deployment and, by default Kubernetes garbage collection behavior, its managed ReplicaSets and Pods — traffic to that service stops immediately unless another Deployment or service is already covering it. Applying a manifest with `replace_namespaced_deployment` overwrites the *entire* deployment spec, not just the fields you changed — any manual `kubectl edit` changes made directly against the live object since the last apply will be lost.

#### Script

```python
#!/usr/bin/env python3
"""
k8s_deploy.py
Apply a Kubernetes Deployment manifest and roll out updates to applications
running on a cluster, using the official Kubernetes Python client.

USAGE
    python k8s_deploy.py apply --file deployment.yaml --namespace default
    python k8s_deploy.py rollout-status --name my-app --namespace default
    python k8s_deploy.py scale --name my-app --replicas 5 --namespace default
    python k8s_deploy.py delete --name my-app --namespace default

REQUIREMENTS
    - A reachable Kubernetes cluster and a valid kubeconfig
    - A Deployment manifest YAML file (apiVersion: apps/v1, kind: Deployment)
    - pip install -r requirements.txt   (installs kubernetes + PyYAML)

EXPECTED OUTPUT
    apply           -> "Deployment 'my-app' created." or "... updated."
    rollout-status  -> "Deployment 'my-app': 3/3 replicas ready."
    scale           -> "Deployment 'my-app' scaled to 5 replicas."
    delete          -> "Deployment 'my-app' deleted from namespace 'default'."
"""

import argparse

import yaml
from kubernetes import client, config
from kubernetes.client.rest import ApiException


def load_apps_client():
    config.load_kube_config()
    return client.AppsV1Api()


def apply_deployment(args):
    apps_v1 = load_apps_client()
    with open(args.file) as f:
        manifest = yaml.safe_load(f)

    name = manifest["metadata"]["name"]
    namespace = args.namespace
    try:
        apps_v1.read_namespaced_deployment(name=name, namespace=namespace)
        apps_v1.replace_namespaced_deployment(name=name, namespace=namespace, body=manifest)
        print(f"Deployment '{name}' updated.")
    except ApiException as e:
        if e.status == 404:
            apps_v1.create_namespaced_deployment(namespace=namespace, body=manifest)
            print(f"Deployment '{name}' created.")
        else:
            print(f"Failed to apply deployment: {e}")


def rollout_status(args):
    apps_v1 = load_apps_client()
    try:
        dep = apps_v1.read_namespaced_deployment_status(name=args.name, namespace=args.namespace)
        ready = dep.status.ready_replicas or 0
        desired = dep.spec.replicas
        print(f"Deployment '{args.name}': {ready}/{desired} replicas ready.")
    except ApiException as e:
        print(f"Failed to read rollout status: {e}")


def scale(args):
    apps_v1 = load_apps_client()
    body = {"spec": {"replicas": args.replicas}}
    try:
        apps_v1.patch_namespaced_deployment_scale(name=args.name, namespace=args.namespace, body=body)
        print(f"Deployment '{args.name}' scaled to {args.replicas} replicas.")
    except ApiException as e:
        print(f"Failed to scale deployment: {e}")


def delete_deployment(args):
    apps_v1 = load_apps_client()
    try:
        apps_v1.delete_namespaced_deployment(name=args.name, namespace=args.namespace)
        print(f"Deployment '{args.name}' deleted from namespace '{args.namespace}'.")
    except ApiException as e:
        print(f"Failed to delete deployment: {e}")


def main():
    parser = argparse.ArgumentParser(description="Deploy and manage Kubernetes Deployments programmatically.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_apply = sub.add_parser("apply")
    p_apply.add_argument("--file", required=True, help="Path to a Deployment manifest YAML file")
    p_apply.add_argument("--namespace", default="default")
    p_apply.set_defaults(func=apply_deployment)

    p_status = sub.add_parser("rollout-status")
    p_status.add_argument("--name", required=True)
    p_status.add_argument("--namespace", default="default")
    p_status.set_defaults(func=rollout_status)

    p_scale = sub.add_parser("scale")
    p_scale.add_argument("--name", required=True)
    p_scale.add_argument("--replicas", type=int, required=True)
    p_scale.add_argument("--namespace", default="default")
    p_scale.set_defaults(func=scale)

    p_delete = sub.add_parser("delete")
    p_delete.add_argument("--name", required=True)
    p_delete.add_argument("--namespace", default="default")
    p_delete.set_defaults(func=delete_deployment)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`apply` reads and parses the YAML manifest, then tries to read an existing Deployment of the same name — if found, it replaces it (an update); if not found (a 404 `ApiException`), it creates a new one. `rollout-status` reads the Deployment's live status subresource to compare ready vs. desired replica counts. `scale` patches just the `spec.replicas` field via the dedicated scale subresource, which is more surgical than replacing the whole spec.

```bash
# Apply a manifest -- creates or updates as needed
python k8s_deploy.py apply --file deployment.yaml --namespace default

# Check rollout progress
python k8s_deploy.py rollout-status --name my-app --namespace default

# Scale up to 5 replicas
python k8s_deploy.py scale --name my-app --replicas 5 --namespace default

# Remove the deployment entirely
python k8s_deploy.py delete --name my-app --namespace default
```

```

```

```

```

#### Variations / flags

- Add a polling loop around `rollout-status` that blocks until `ready == desired`, turning this into a true "wait for rollout to finish" gate for a CI/CD pipeline.
- Add a `--dry-run` flag using the Kubernetes API's built-in dry-run query parameter to validate a manifest without actually applying it.
- Combine with `helm_deploy.py` (Section 22.17) — this script is better suited for direct manifest control, while Helm is better for templated, versioned application packages; many teams use both depending on the service.

#### Common pitfalls

- **`replace` overwriting unmanaged fields** — if anything (an autoscaler, a manual `kubectl patch`) changed fields not present in your local manifest, `replace_namespaced_deployment` wipes those changes out; consider a proper server-side apply/patch approach for GitOps-style workflows that need to coexist with other controllers.
- **Rollout status doesn't mean "the app is healthy"** — `ready_replicas` reflects the Kubernetes readiness probe passing, not that the application is functionally correct; pair this with your own smoke tests.
- **Namespace must already exist** — unlike `helm install`, this script doesn't create the namespace automatically; run `kubectl create namespace <ns>` first if needed.
- **Manifest missing `metadata.name`** — `apply_deployment` will raise a `KeyError` if the YAML doesn't define `metadata.name`; validate the manifest structure before feeding it in.

### 22.17 Helm Chart Automation

`helm_deploy.py` — Package and deploy applications to Kubernetes using Helm charts, driven from Python as a thin, scriptable wrapper around the `helm` CLI.

#### Use case

A platform team maintains a reusable Helm chart for a common application pattern (e.g. a web service plus its database dependency), and different application teams need to install, upgrade, list, and uninstall their own releases of that chart with different values — without every team needing to remember Helm's full CLI syntax.

#### Prerequisites

- Helm 3 installed and on PATH
- A reachable Kubernetes cluster and a valid kubeconfig
- A Helm chart directory or packaged `.tgz` chart
- OS scope: Linux, macOS, Windows

```bash
# Install Helm (Linux example; see helm.sh for other platforms)
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
```

No extra Python packages are strictly required beyond the standard library for this script as written, though `pip install pyyaml` is useful if you extend it to parse values files directly.

#### Cautions

`uninstall` removes the entire release, including everything Helm created for it (Deployments, Services, ConfigMaps, etc.) — there's no confirmation step in this script. `--set key=value` overrides on the command line are easy to lose track of over time; prefer checked-in values files for anything beyond quick one-off overrides.

#### Script

```python
#!/usr/bin/env python3
"""
helm_deploy.py
Package and deploy applications to Kubernetes using Helm charts, managed
and driven from Python (a thin, scriptable wrapper around the `helm` CLI).

USAGE
    python helm_deploy.py install --release my-release --chart ./mychart --namespace default
    python helm_deploy.py upgrade --release my-release --chart ./mychart --set image.tag=1.2.0
    python helm_deploy.py list --namespace default
    python helm_deploy.py uninstall --release my-release --namespace default

REQUIREMENTS
    - Helm 3 installed and on PATH (https://helm.sh/docs/intro/install/)
    - A reachable Kubernetes cluster and a valid kubeconfig
    - A Helm chart directory or packaged .tgz chart
    - pip install -r requirements.txt   (no extra Python packages strictly
      required beyond the standard library, but PyYAML is included for
      parsing --set-file/values files if you extend this script)

EXPECTED OUTPUT
    install    -> Helm's own install summary, ending with "Release 'my-release' installed."
    upgrade    -> Helm's own upgrade summary, ending with "Release 'my-release' upgraded."
    list       -> table of releases (name, namespace, status, chart, app version)
    uninstall  -> "Release 'my-release' uninstalled."
"""

import argparse
import subprocess
import sys


def run_helm(args_list):
    print(f"Running: helm {' '.join(args_list)}")
    result = subprocess.run(["helm"] + args_list, capture_output=True, text=True)
    print(result.stdout)
    if result.returncode != 0:
        print(result.stderr)
    return result.returncode


def install(args):
    cmd = ["install", args.release, args.chart, "--namespace", args.namespace, "--create-namespace"]
    for kv in args.set or []:
        cmd += ["--set", kv]
    rc = run_helm(cmd)
    if rc == 0:
        print(f"Release '{args.release}' installed.")
    else:
        sys.exit(rc)


def upgrade(args):
    cmd = ["upgrade", args.release, args.chart, "--namespace", args.namespace]
    if args.install:
        cmd.append("--install")
    for kv in args.set or []:
        cmd += ["--set", kv]
    rc = run_helm(cmd)
    if rc == 0:
        print(f"Release '{args.release}' upgraded.")
    else:
        sys.exit(rc)


def list_releases(args):
    cmd = ["list", "--namespace", args.namespace]
    run_helm(cmd)


def uninstall(args):
    cmd = ["uninstall", args.release, "--namespace", args.namespace]
    rc = run_helm(cmd)
    if rc == 0:
        print(f"Release '{args.release}' uninstalled.")
    else:
        sys.exit(rc)


def main():
    parser = argparse.ArgumentParser(description="Deploy applications to Kubernetes via Helm.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_install = sub.add_parser("install")
    p_install.add_argument("--release", required=True)
    p_install.add_argument("--chart", required=True, help="Chart directory or .tgz path")
    p_install.add_argument("--namespace", default="default")
    p_install.add_argument("--set", action="append", help="key=value, repeatable")
    p_install.set_defaults(func=install)

    p_upgrade = sub.add_parser("upgrade")
    p_upgrade.add_argument("--release", required=True)
    p_upgrade.add_argument("--chart", required=True)
    p_upgrade.add_argument("--namespace", default="default")
    p_upgrade.add_argument("--install", action="store_true", help="Install the release if it does not exist")
    p_upgrade.add_argument("--set", action="append", help="key=value, repeatable")
    p_upgrade.set_defaults(func=upgrade)

    p_list = sub.add_parser("list")
    p_list.add_argument("--namespace", default="default")
    p_list.set_defaults(func=list_releases)

    p_uninstall = sub.add_parser("uninstall")
    p_uninstall.add_argument("--release", required=True)
    p_uninstall.add_argument("--namespace", default="default")
    p_uninstall.set_defaults(func=uninstall)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`run_helm()` shells out to the `helm` binary via `subprocess.run`, capturing both stdout and stderr and printing Helm's own native output. `install` adds `--create-namespace` so the target namespace is created automatically if it doesn't exist yet. `upgrade --install` mirrors Helm's own "upgrade or install" idiom for idempotent deploy scripts.

```bash
# Fresh install
python helm_deploy.py install --release my-release --chart ./mychart --namespace default

# Upgrade with a value override (e.g. a new image tag)
python helm_deploy.py upgrade --release my-release --chart ./mychart --set image.tag=1.2.0

# List releases in a namespace
python helm_deploy.py list --namespace default

# Remove a release entirely
python helm_deploy.py uninstall --release my-release --namespace default
```

#### Sample output

```

NAME: my-release
LAST DEPLOYED: Fri Jul 17 10:00:00 2026
NAMESPACE: default
STATUS: deployed
REVISION: 1

Release 'my-release' installed.
```

```

my-release    default     2          deployed   mychart-0.1.0    1.2.0
```


#### Variations / flags

- Add a `--values values-prod.yaml` passthrough flag (`-f`) for teams that prefer checked-in values files over ad hoc `--set` overrides.
- Add a `rollback` subcommand wrapping `helm rollback <release> <revision>` for quickly reverting a bad upgrade.
- Add a `--dry-run` passthrough for validating a chart's rendered manifests before actually applying them: `helm install ... --dry-run --debug`.

#### Common pitfalls

- **Chart not found** — `--chart` must point at a valid chart directory (containing `Chart.yaml`) or a packaged `.tgz`; a typo'd relative path is a common source of confusing Helm errors.
- **`upgrade` without `--install` on a release that doesn't exist yet** — fails outright; use `upgrade --install` for idempotent "make sure this is deployed" scripts that should work whether or not the release already exists.
- **`--set` type coercion surprises** — Helm's `--set` parses values as YAML types, so `--set replicaCount=3` becomes the integer `3`, but `--set version=1.20` can be misinterpreted as a float and truncate trailing zeros; use `--set-string` for values that must remain literal strings.
- **Namespace mismatches between `list` and where the release actually lives** — Helm releases are namespace-scoped; forgetting `--namespace` (or using the wrong one) makes an existing release look like it's missing.

### 22.18 Container Health Checks

`container_health_check.py` — Attach a Docker `HEALTHCHECK` to a container at launch and poll its reported health status until it becomes healthy (or times out), to confirm a service is actually operational, not just "running."

#### Use case

A deployment script starts a container and needs to know, definitively, that the service inside it is ready to receive traffic — not just that the container process has started, since a web server process starting doesn't mean the app has finished booting, connected to its database, or warmed its cache yet.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- The container's image must already have the tool used in `--test` available (e.g. `curl` or `wget`) — otherwise the health check will always report failure
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

If the image doesn't include the binary referenced in `--test` (a common gotcha with minimal/distroless images), the health check will perpetually fail even though the service itself may be working fine — verify the test command actually runs inside the target image first.

#### Script

```python
#!/usr/bin/env python3
"""
container_health_check.py
Define a Docker HEALTHCHECK on a running container's image and verify
readiness/liveness by polling the container's reported health status, to
confirm a service is actually operational (not just "running").

USAGE
    Run a container with a health check attached:
        python container_health_check.py run --image nginx:latest --name web \
            --test "curl -f http://localhost/ || exit 1" \
            --interval 10 --timeout 5 --retries 3 --start-period 5

    Poll the health status of an existing container until healthy or timeout:
        python container_health_check.py check --name web --wait 60

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - The container's image must have the tool used in --test available
      (e.g. curl or wget) or the health check will always fail
    - pip install -r requirements.txt

EXPECTED OUTPUT
    run   -> "Container 'web' started with health check attached."
    check -> polls every 2 seconds and prints status transitions, e.g.:
                [0s] status=starting
                [10s] status=healthy
             then "Container 'web' is healthy." or exits with an error after
             the --wait timeout if it never becomes healthy.
"""

import argparse
import sys
import time

import docker
from docker.errors import APIError, NotFound


def run_with_healthcheck(args):
    client = docker.from_env()
    healthcheck = {
        "test": ["CMD-SHELL", args.test],
        "interval": args.interval * 1_000_000_000,
        "timeout": args.timeout * 1_000_000_000,
        "retries": args.retries,
        "start_period": args.start_period * 1_000_000_000,
    }
    try:
        client.containers.run(
            image=args.image,
            name=args.name,
            detach=True,
            healthcheck=healthcheck,
        )
        print(f"Container '{args.name}' started with health check attached.")
    except APIError as e:
        print(f"Failed to start container: {e}")
        sys.exit(1)


def check_health(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
    except NotFound:
        print(f"Container '{args.name}' not found.")
        sys.exit(1)

    elapsed = 0
    poll_interval = 2
    while elapsed <= args.wait:
        container.reload()
        health = container.attrs.get("State", {}).get("Health", {})
        status = health.get("Status", "unknown")
        print(f"[{elapsed}s] status={status}")
        if status == "healthy":
            print(f"Container '{args.name}' is healthy.")
            return
        if status == "unhealthy":
            print(f"Container '{args.name}' is unhealthy.")
            sys.exit(1)
        time.sleep(poll_interval)
        elapsed += poll_interval

    print(f"Timed out after {args.wait}s waiting for '{args.name}' to become healthy.")
    sys.exit(1)


def main():
    parser = argparse.ArgumentParser(description="Attach and verify Docker container health checks.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_run = sub.add_parser("run", help="Run a container with a HEALTHCHECK attached")
    p_run.add_argument("--image", required=True)
    p_run.add_argument("--name", required=True)
    p_run.add_argument("--test", required=True, help="Shell command; exit 0 = healthy, non-zero = unhealthy")
    p_run.add_argument("--interval", type=int, default=30, help="Seconds between checks")
    p_run.add_argument("--timeout", type=int, default=10, help="Seconds before a check itself times out")
    p_run.add_argument("--retries", type=int, default=3, help="Consecutive failures before marking unhealthy")
    p_run.add_argument("--start-period", type=int, default=0, help="Grace period in seconds before checks count")
    p_run.set_defaults(func=run_with_healthcheck)

    p_check = sub.add_parser("check", help="Poll an existing container's health status")
    p_check.add_argument("--name", required=True)
    p_check.add_argument("--wait", type=int, default=60, help="Max seconds to wait for a healthy status")
    p_check.set_defaults(func=check_health)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`run` builds a Docker `healthcheck` dict from CLI flags — converting human-friendly seconds into the nanoseconds the Docker API expects — and passes it to `client.containers.run()`. The `test` command follows Docker's `CMD-SHELL` convention: exit code 0 means healthy, non-zero means unhealthy. `check` polls an existing container's `State.Health.Status` field every 2 seconds, printing each transition, until it sees `healthy`, `unhealthy`, or the `--wait` timeout is reached.

```bash
# Run nginx with a health check that curls localhost
python container_health_check.py run --image nginx:latest --name web \
    --test "curl -f http://localhost/ || exit 1" \
    --interval 10 --timeout 5 --retries 3 --start-period 5

# Poll until it reports healthy, waiting up to 60 seconds
python container_health_check.py check --name web --wait 60
```

#### Sample output

```
Container 'web' started with health check attached.
```

```
[0s] status=starting
[2s] status=starting
[10s] status=healthy
Container 'web' is healthy.
```

#### Variations / flags

- Add a `--exit-nonzero-on-timeout` toggle (already the default behavior via `sys.exit(1)`) that CI systems can key off of to fail a pipeline stage cleanly.
- Wire `check` directly after `docker_run_container.py run` (Section 22.2) in a deployment script so nothing proceeds until the new container is confirmed healthy.
- Add a `--webhook-url` flag that posts a notification if a container flips to `unhealthy` during long-running monitoring.

#### Common pitfalls

- **Test command tool missing from the image** — the single most common failure mode; `curl`/`wget` aren't installed in many minimal base images (e.g. `alpine` without extra packages, `distroless`), so the health check perpetually fails even though the app itself is fine.
- **`start_period` too short** — services with a slow boot (JVM apps, anything doing migrations on startup) need a `--start-period` long enough to cover that boot time, or they'll be marked unhealthy before they've had a chance to come up.
- **Nanosecond conversion confusion** — the Docker API's healthcheck fields are in nanoseconds; passing raw seconds without the `* 1_000_000_000` conversion (easy to forget if extending this script) silently creates an effectively-zero interval/timeout.
- **Interpreting "starting" as a failure** — a container legitimately spends time in the `starting` status during its `start_period`; don't treat that as unhealthy in downstream automation that keys off this script's output.

### 22.19 Resource Limit Enforcement on Containers

`container_resource_limits.py` — Set, update, and inspect CPU and memory constraints on Docker containers, either at launch time or on an already-running container, to prevent one service from starving resources needed by others on the same host.

#### Use case

Several unrelated services share the same Docker host. One of them has a memory leak or a runaway process that, left unconstrained, could consume all available RAM and CPU and take down every other container on the host. An operator needs to cap that container's resource usage — ideally before it ever launches, but also live if a problem is discovered after the fact.

#### Prerequisites

- Docker Engine or Docker Desktop installed and running
- Permission to access the Docker socket
- OS scope: Linux, macOS, Windows

```bash
pip install docker
```

#### Cautions

Setting a `--memory` limit lower than what the application actually needs at steady state will cause the kernel's OOM killer to terminate the container's main process — sometimes repeatedly, if Docker's restart policy brings it back into the same limit. Always benchmark actual memory usage under realistic load before locking in a hard limit.

#### Script

```python
#!/usr/bin/env python3
"""
container_resource_limits.py
Set CPU and memory constraints on Docker containers, either at launch time
or on an already-running container, to prevent one service from starving
resources needed by others on the same host.

USAGE
    Run a new container with limits:
        python container_resource_limits.py run --image nginx:latest --name web \
            --cpus 1.5 --memory 512m --memory-swap 1g

    Update limits on an already-running container:
        python container_resource_limits.py update --name web --cpus 0.5 --memory 256m

    Inspect current limits:
        python container_resource_limits.py inspect --name web

REQUIREMENTS
    - Docker Engine/Desktop installed and running
    - pip install -r requirements.txt

EXPECTED OUTPUT
    run     -> "Container 'web' started with cpus=1.5 memory=512m."
    update  -> "Container 'web' updated with cpus=0.5 memory=256m."
    inspect -> current NanoCpus/CpuQuota and Memory limit values in bytes, e.g.:
                 CPU quota: 50000 / period: 100000 (~0.5 CPUs)
                 Memory limit: 268435456 bytes (256.0 MB)
"""

import argparse
import sys

import docker
from docker.errors import APIError, NotFound


def parse_memory(mem_str):
    """Return the human-readable string as-is; the Docker SDK accepts
    strings like '512m', '1g' directly for mem_limit/memswap_limit."""
    return mem_str


def cpus_to_quota(cpus):
    period = 100_000
    return int(cpus * period), period


def run_with_limits(args):
    client = docker.from_env()
    nano_cpus = int(args.cpus * 1_000_000_000) if args.cpus else None
    try:
        client.containers.run(
            image=args.image,
            name=args.name,
            detach=True,
            nano_cpus=nano_cpus,
            mem_limit=args.memory,
            memswap_limit=args.memory_swap,
        )
        print(f"Container '{args.name}' started with cpus={args.cpus} memory={args.memory}.")
    except APIError as e:
        print(f"Failed to start container: {e}")
        sys.exit(1)


def update_limits(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
    except NotFound:
        print(f"Container '{args.name}' not found.")
        sys.exit(1)

    kwargs = {}
    if args.cpus:
        kwargs["nano_cpus"] = int(args.cpus * 1_000_000_000)
    if args.memory:
        kwargs["mem_limit"] = args.memory
    if args.memory_swap:
        kwargs["memswap_limit"] = args.memory_swap

    try:
        container.update(**kwargs)
        print(f"Container '{args.name}' updated with cpus={args.cpus} memory={args.memory}.")
    except APIError as e:
        print(f"Could not update container: {e}")
        sys.exit(1)


def inspect_limits(args):
    client = docker.from_env()
    try:
        container = client.containers.get(args.name)
    except NotFound:
        print(f"Container '{args.name}' not found.")
        sys.exit(1)

    host_config = container.attrs.get("HostConfig", {})
    nano_cpus = host_config.get("NanoCpus", 0)
    cpu_quota = host_config.get("CpuQuota", 0)
    cpu_period = host_config.get("CpuPeriod", 0)
    memory = host_config.get("Memory", 0)

    if nano_cpus:
        print(f"CPU limit: {nano_cpus / 1_000_000_000} CPUs (NanoCpus)")
    elif cpu_quota and cpu_period:
        print(f"CPU quota: {cpu_quota} / period: {cpu_period} (~{cpu_quota / cpu_period} CPUs)")
    else:
        print("CPU limit: none set")

    if memory:
        print(f"Memory limit: {memory} bytes ({memory / (1024 * 1024):.1f} MB)")
    else:
        print("Memory limit: none set")


def main():
    parser = argparse.ArgumentParser(description="Set CPU and memory limits on Docker containers.")
    sub = parser.add_subparsers(dest="command", required=True)

    p_run = sub.add_parser("run")
    p_run.add_argument("--image", required=True)
    p_run.add_argument("--name", required=True)
    p_run.add_argument("--cpus", type=float, help="Number of CPUs, e.g. 1.5")
    p_run.add_argument("--memory", help="Memory limit, e.g. 512m, 1g")
    p_run.add_argument("--memory-swap", help="Total memory + swap limit, e.g. 1g")
    p_run.set_defaults(func=run_with_limits)

    p_update = sub.add_parser("update")
    p_update.add_argument("--name", required=True)
    p_update.add_argument("--cpus", type=float, help="Number of CPUs, e.g. 0.5")
    p_update.add_argument("--memory", help="Memory limit, e.g. 256m")
    p_update.add_argument("--memory-swap", help="Total memory + swap limit, e.g. 512m")
    p_update.set_defaults(func=update_limits)

    p_inspect = sub.add_parser("inspect")
    p_inspect.add_argument("--name", required=True)
    p_inspect.set_defaults(func=inspect_limits)

    args = parser.parse_args()
    args.func(args)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`run` converts a fractional `--cpus` value (e.g. `1.5`) into `nano_cpus`, the unit the Docker Engine API actually uses internally (CPUs × 1,000,000,000). `--memory`/`--memory-swap` strings like `512m` or `1g` are passed straight through, since the Docker SDK itself accepts these human-readable forms. `update` applies new limits to an already-running container live, without needing to stop and restart it. `inspect` reads back the container's `HostConfig` to report whatever limits are currently in effect, whether set via `NanoCpus` or the older `CpuQuota`/`CpuPeriod` pair.

```bash
# Launch with a 1.5 CPU and 512MB memory cap (1GB total including swap)
python container_resource_limits.py run --image nginx:latest --name web \
    --cpus 1.5 --memory 512m --memory-swap 1g

# Tighten limits on a running container without restarting it
python container_resource_limits.py update --name web --cpus 0.5 --memory 256m

# Check current limits
python container_resource_limits.py inspect --name web
```

#### Sample output

```
Container 'web' started with cpus=1.5 memory=512m.
```

```
Container 'web' updated with cpus=0.5 memory=256m.
```

```

Memory limit: 268435456 bytes (256.0 MB)
```

#### Variations / flags

- Add a `--cpuset-cpus 0,1` flag to pin a container to specific physical CPU cores, useful for latency-sensitive workloads that benefit from cache locality.
- Add a `--memory-reservation` (soft limit) flag distinct from the hard `--memory` limit, letting Docker reclaim memory under host pressure before hitting the hard OOM boundary.
- Wire `inspect` into a periodic audit script that flags any container running with no CPU/memory limits at all on a shared host.

#### Common pitfalls

- **Memory limit set below actual working-set size** — triggers the OOM killer inside the container's cgroup, which looks like a crash rather than an obvious "out of memory" error unless you check `docker inspect` for `OOMKilled: true`.
- **`--memory-swap` less than or equal to `--memory`** — Docker interprets equal values as "no additional swap allowed"; if `--memory-swap` is omitted entirely, Docker defaults it to double the memory limit, which can be surprising.
- **`update` limits not persisting across container recreation** — `update` only affects the live container; if it's later removed and recreated (e.g. via a Compose `up` or Deployment rollout) without the same limits baked into the launch config, the constraint disappears.
- **Fractional CPU confusion** — `--cpus 1.5` means "the equivalent of one and a half CPU cores over time," not a guarantee of a specific core; on a busy host, a container can still experience CPU throttling even under its nominal limit if overall host contention is high.
