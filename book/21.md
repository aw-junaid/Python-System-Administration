## Chapter 21: Automation Best Practices

Every system administrator eventually writes a script that outlives its author's memory of it. It runs quietly on a cron job for two years, survives three server migrations, and then one day fails at 3 a.m. because a hardcoded path didn't exist on the new box, or a bare `except:` swallowed the one error message that would have explained everything. This chapter is about the difference between a script that "works on my machine, once" and automation that a team can actually depend on.

None of the practices here are exotic. They are the accumulated scar tissue of production automation: catch the exception you expect, not every exception that could ever occur; log what happened instead of printing and hoping someone is watching the terminal; retry transient failures with backoff instead of hammering a struggling service; keep secrets out of source code; make scripts safe to re-run; and write code in small, testable pieces instead of one 400-line function nobody wants to touch. Individually, each idea is almost obvious. Together, they are what separates a one-off hack from infrastructure.

The twelve topics in this chapter are organized to build on each other. Exception handling, logging, and retry strategies form the reliability core — how a script behaves when something goes wrong. Secure credential handling and idempotent automation address safety — how a script behaves around sensitive data and repeated execution. Modular scripting, configuration management, and testing address maintainability — how a script behaves as it grows and as other people start relying on it. Packaging, cross-platform scripting, performance optimization, and documentation generation address distribution — how a script stops being "your script" and becomes a tool your team can install, run anywhere, and understand without reading the source.

#### Why this matters in practice

Automation scripts have an unusual lifecycle compared to application code. They are often written quickly, under time pressure, to solve an immediate operational problem — clean up old log files, rotate a credential, restart a hung service, sync a directory to backup storage. Because the initial version works, it gets deployed. Because it gets deployed, it gets depended upon. Because it gets depended upon, it eventually gets inherited by someone who did not write it and has no context for its assumptions.

The practices in this chapter exist specifically to survive that handoff. A script with specific exception handling tells the next engineer exactly what failure modes were anticipated. A script with proper logging leaves a trail when it fails unattended overnight. A script with externalized configuration can move from a developer's laptop to a staging server to production without a single code change. A script with tests can be modified with confidence instead of fear. None of this is about elegance for its own sake — it is about reducing the number of times a human being has to get paged, SSH into a machine, and reverse-engineer their own automation from six months ago.

### 21.1 Exception Handling

#### Use case

A nightly script reads a config file, computes an average from metrics, converts a string setting to an integer, and writes a report. A sysadmin needs each of these steps to fail loudly and specifically when something expected goes wrong (file missing, divide by zero, bad input) while still letting a truly unexpected bug — like a typo causing a `KeyError` — surface as a real traceback instead of vanishing silently.

#### Prerequisites

- Python 3.6+ (uses f-strings)
- Only standard library: `json`, `logging`, `os` — no `pip install` required
- OS scope: Linux, macOS, Windows (all paths used are illustrative; the script doesn't require the paths to actually exist)

```bash
python3 --version   # confirm 3.6+
# no external dependencies to install
```

#### Cautions

- The script deliberately tries to read `/root/forbidden/report.txt`, a path that doesn't exist on most systems — this is intentional and demonstrates a handled `FileNotFoundError`, not a bug.
- Do not copy the "ANTI-PATTERN" comment block into real code; it exists only to illustrate what *not* to do.

#### Script

```python
#!/usr/bin/env python3
"""
exception_handling.py

Demonstrates proper exception handling: catching SPECIFIC exception
types instead of a bare `except:`, preserving tracebacks with
`raise ... from e`, and only handling the errors you actually expect.
"""

import json
import logging
import os

logging.basicConfig(level=logging.INFO, format="%(levelname)s: %(message)s")
logger = logging.getLogger("exception_handling_demo")


def read_config_file(path: str) -> dict:
    """Read and parse a JSON config file, handling only expected failures."""
    try:
        with open(path, "r", encoding="utf-8") as f:
            return json.load(f)
    except FileNotFoundError:
        logger.warning("Config file not found at '%s'; using defaults.", path)
        return {}
    except json.JSONDecodeError as e:
        logger.error("Config file '%s' contains invalid JSON: %s", path, e)
        raise ValueError(f"Could not parse config file: {path}") from e
    except PermissionError:
        logger.error("No permission to read config file '%s'.", path)
        raise


def divide_metrics(total: float, count: int) -> float:
    """Compute an average, handling only the specific error we expect."""
    try:
        return total / count
    except ZeroDivisionError:
        logger.warning("Count was zero; returning 0.0 instead of dividing.")
        return 0.0
    except TypeError as e:
        logger.error("Invalid types passed to divide_metrics: %s", e)
        raise


def parse_int_setting(raw_value: str) -> int:
    """Convert a string setting to int, handling only bad-format input."""
    try:
        return int(raw_value)
    except ValueError as e:
        logger.error("Setting %r is not a valid integer.", raw_value)
        raise ValueError(f"Expected an integer, got: {raw_value!r}") from e


def write_report(path: str, content: str) -> None:
    """Write a report file, handling only permission/path errors."""
    try:
        with open(path, "w", encoding="utf-8") as f:
            f.write(content)
    except PermissionError:
        logger.error("No permission to write to '%s'.", path)
        raise
    except FileNotFoundError:
        logger.error("Directory for '%s' does not exist.", path)
        raise
    except OSError as e:
        logger.error("OS error while writing '%s': %s", path, e)
        raise


def run_task(name: str, func, *args, **kwargs) -> bool:
    """Run a task, printing success/failure without hiding unexpected errors."""
    try:
        result = func(*args, **kwargs)
        logger.info("Task %-22s -> OK (result=%r)", name, result)
        return True
    except (FileNotFoundError, ValueError, PermissionError, OSError) as e:
        logger.info("Task %-22s -> HANDLED FAILURE (%s: %s)", name, type(e).__name__, e)
        return False


def main():
    print("Exception Handling Demo")
    print("=" * 40)

    results = []
    results.append(run_task("read_missing_config", read_config_file, "/nonexistent/config.json"))
    results.append(run_task("divide_by_zero", divide_metrics, 100, 0))
    results.append(run_task("parse_bad_int", parse_int_setting, "not-a-number"))
    results.append(run_task("write_to_bad_path", write_report, "/root/forbidden/report.txt", "data"))

    succeeded = sum(1 for r in results if r)
    print("=" * 40)
    print(f"Summary: {succeeded}/{len(results)} tasks completed without needing a handled failure.")
    print("(All failures above were caught by SPECIFIC exception types, not a bare 'except:'.)")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

The script has no command-line flags — it runs a fixed sequence of four demo tasks through a shared `run_task()` wrapper that only catches the exception types it explicitly lists.

```bash
python3 exception_handling.py
```

#### Sample output

```

========================================

INFO: Task read_missing_config    -> OK (result={})
WARNING: Count was zero; returning 0.0 instead of dividing.
INFO: Task divide_by_zero         -> OK (result=0.0)
ERROR: Setting 'not-a-number' is not a valid integer.
INFO: Task parse_bad_int          -> HANDLED FAILURE (ValueError: Expected an integer, got: 'not-a-number')
ERROR: Directory for '/root/forbidden/report.txt' does not exist.
INFO: Task write_to_bad_path      -> HANDLED FAILURE (FileNotFoundError: [Errno 2] No such file or directory: '/root/forbidden/report.txt')

```

#### Variations / flags

- Swap `logging.basicConfig(level=logging.INFO, ...)` to `level=logging.DEBUG` to see more granular internal messages if you add `logger.debug()` calls.
- Add a `--config` argument (via `argparse`) to point `read_config_file` at a real path instead of the hardcoded nonexistent one.
- Wrap `run_task` calls in a loop over a list of `(name, func, args)` tuples if you have more than four tasks — keeps `main()` short as the script grows.

#### Common pitfalls

- **Bare `except:` in "real" scripts** — silently catches `KeyboardInterrupt` and `SystemExit`, making a script impossible to Ctrl-C and hiding real bugs. Always name the exception type(s).
- **Catching `Exception` broadly "to be safe"** — nearly as bad as bare `except:`; it still hides unanticipated bugs, just with extra steps.
- **Forgetting `raise ... from e`** — without it, the original traceback context is lost when you raise a new exception, making root-cause debugging much harder.
- **Permission errors on `/root/...` paths** — if you're running as `root` yourself, the `write_to_bad_path` demo may behave differently since root can bypass some permission checks; the `FileNotFoundError` branch (directory doesn't exist) will still fire.

### 21.2 Logging Best Practices


#### Use case

A sysadmin runs a nightly backup job via cron. Nobody is watching the console at 3 a.m., so when the job partially fails, the only way to find out what happened is a log file. The script needs INFO-level milestones on screen (when run manually) plus a full DEBUG-level history on disk that doesn't grow forever and eat the disk.

#### Prerequisites

- Python 3.6+
- Only standard library: `argparse`, `logging`, `logging.handlers`, `os`, `random`, `sys`
- OS scope: Linux, macOS, Windows

```bash
python3 --version   # confirm 3.6+
# no external dependencies to install
```

#### Cautions

- The script uses `random.random()` to simulate transient copy failures, so exact WARNING/ERROR lines will vary slightly run to run — this is expected, not a bug.
- Running the script repeatedly will keep appending to `automation.log` until it hits the 1MB rotation threshold; delete it if you want a clean slate.

#### Script

```python
#!/usr/bin/env python3
"""
logging_best_practices.py

Demonstrates proper use of Python's `logging` module: appropriate log
levels (DEBUG/INFO/WARNING/ERROR/CRITICAL), a console handler AND a
rotating file handler, structured formatting, and avoiding `print()`
for anything other than final user-facing output.
"""

import argparse
import logging
import logging.handlers
import os
import random
import sys


def build_logger(log_file: str, verbose: bool) -> logging.Logger:
    logger = logging.getLogger("automation")
    logger.setLevel(logging.DEBUG)  # capture everything; handlers filter what's shown
    logger.handlers.clear()

    formatter = logging.Formatter(
        fmt="%(asctime)s | %(levelname)-8s | %(name)s | %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    console_handler = logging.StreamHandler(sys.stdout)
    console_handler.setLevel(logging.DEBUG if verbose else logging.INFO)
    console_handler.setFormatter(formatter)
    logger.addHandler(console_handler)

    file_handler = logging.handlers.RotatingFileHandler(
        log_file, maxBytes=1_000_000, backupCount=5, encoding="utf-8"
    )
    file_handler.setLevel(logging.DEBUG)
    file_handler.setFormatter(formatter)
    logger.addHandler(file_handler)

    return logger


def simulate_backup_task(logger: logging.Logger, files):
    """A small fake automation task demonstrating each log level's purpose."""
    logger.info("Backup task started for %d file(s).", len(files))

    backed_up = 0
    for f in files:
        logger.debug("Preparing to back up file: %s", f)

        if f.endswith(".tmp"):
            logger.warning("Skipping temp file (recoverable, non-fatal): %s", f)
            continue

        if random.random() < 0.2:
            logger.error("Failed to copy file (will be retried by caller): %s", f)
            continue

        logger.debug("Copied %s successfully.", f)
        backed_up += 1

    if backed_up == 0:
        logger.critical("No files were backed up at all — backup job effectively failed.")
    else:
        logger.info("Backup task finished: %d/%d file(s) backed up.", backed_up, len(files))


def main():
    parser = argparse.ArgumentParser(description="Demonstrate logging best practices with a simulated backup task.")
    parser.add_argument("--log-file", default="automation.log", help="Path to the rotating log file (default: automation.log)")
    parser.add_argument("--verbose", action="store_true", help="Show DEBUG-level messages on the console too")
    args = parser.parse_args()

    logger = build_logger(args.log_file, args.verbose)

    sample_files = ["report.csv", "cache.tmp", "database.sql", "session.tmp", "photos.zip"]
    simulate_backup_task(logger, sample_files)

    print()
    print(f"Full DEBUG-level log written to: {os.path.abspath(args.log_file)}")
    print("Run again with --verbose to also see DEBUG messages on the console.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`build_logger()` attaches two handlers to one logger: a console handler (INFO by default) and a `RotatingFileHandler` (always DEBUG) that caps each log file at 1MB and keeps 5 rotated backups.

```bash
# Default: INFO+ on console, DEBUG+ in automation.log
python3 logging_best_practices.py

# Show DEBUG-level messages on console too
python3 logging_best_practices.py --verbose

# Use a custom log file path
python3 logging_best_practices.py --log-file /var/log/myapp/backup.log --verbose
```

#### Sample output

```
2026-07-18 05:33:20 | INFO     | automation | Backup task started for 5 file(s).
2026-07-18 05:33:20 | WARNING  | automation | Skipping temp file (recoverable, non-fatal): cache.tmp
2026-07-18 05:33:20 | WARNING  | automation | Skipping temp file (recoverable, non-fatal): session.tmp
2026-07-18 05:33:20 | INFO     | automation | Backup task finished: 3/5 file(s) backed up.

Full DEBUG-level log written to: /home/user/automation.log
Run again with --verbose to also see DEBUG messages on the console.
```

With `--verbose`, you'll additionally see lines like `DEBUG | automation | Preparing to back up file: report.csv` on the console.

#### Variations / flags

- `--log-file automation.log` — change where the rotating log file is written; point it at `/var/log/...` for real deployments (ensure write permissions).
- `--verbose` — mirrors DEBUG-level detail to the console; useful when troubleshooting interactively instead of tailing the file.
- Change `maxBytes=1_000_000, backupCount=5` to tune rotation size/retention for your disk budget.
- Schedule with cron for a real nightly job:
  ```bash
  # crontab -e
  0 2 * * * /usr/bin/python3 /opt/scripts/logging_best_practices.py --log-file /var/log/backup.log
  ```

#### Common pitfalls

- **Using `print()` in library/automation code** — output is lost once the terminal or SSH session closes; always log instead, and reserve `print()` for final human-facing summaries.
- **Setting the logger's own level too high** — if `logger.setLevel(logging.INFO)` is set instead of `DEBUG`, the file handler will never receive DEBUG messages regardless of its own level, because filtering happens at the logger first.
- **Forgetting `logger.handlers.clear()`** when a script or module is re-imported/re-run in the same process (e.g. in tests) — old handlers accumulate and duplicate every log line.
- **Log rotation permission errors** — if the log file path is owned by another user (e.g. root-owned `/var/log/`), a non-privileged cron user will fail silently or raise `PermissionError`; ensure the running user owns the log directory.

### 21.3 Retry Strategies


#### Use case

A sysadmin's script calls an internal REST API to create a maintenance ticket, but the API occasionally returns a transient `503 Service Unavailable` under load. The script needs to retry those specific failures with increasing delay (so it doesn't pile onto an already-struggling service) while immediately giving up on permanent errors like bad authentication — and it needs the retried operation itself to be safe to repeat.

#### Prerequisites

- Python 3.6+
- Only standard library: `argparse`, `functools`, `logging`, `random`, `time`
- OS scope: Linux, macOS, Windows

```bash
python3 --version   # confirm 3.6+
# no external dependencies to install
```

#### Cautions

- `call_flaky_api` uses `random.random()` to simulate a ~40% success rate per attempt; occasionally it will exhaust all 5 attempts and report failure — this is expected, not a bug in the script.
- Real backoff delays involve `time.sleep()`; running with a high `--max-attempts` and low `--base-delay` is fine, but a high `--base-delay` will make the demo take noticeably longer to finish.

#### Script

```python
#!/usr/bin/env python3
"""
retry_strategies.py

Demonstrates a production-style retry strategy: exponential backoff
WITH random jitter (to avoid "thundering herd" retries), a maximum
attempt cap, only retrying on specific transient exceptions, and
designing the retried operation to be IDEMPOTENT using an
idempotency key.
"""

import argparse
import functools
import logging
import random
import time

logging.basicConfig(level=logging.INFO, format="%(asctime)s | %(levelname)s | %(message)s", datefmt="%H:%M:%S")
logger = logging.getLogger("retry_demo")


class TransientError(Exception):
    """Represents a retryable, temporary failure (e.g. network blip, 503)."""


class PermanentError(Exception):
    """Represents a non-retryable failure (e.g. 400 Bad Request, bad auth)."""


def retry_with_backoff(max_attempts=5, base_delay=0.3, max_delay=8.0, retry_on=(TransientError,)):
    """Decorator implementing exponential backoff with full jitter."""
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            attempt = 1
            while True:
                try:
                    return func(*args, **kwargs)
                except retry_on as e:
                    if attempt >= max_attempts:
                        logger.error("Giving up after %d attempts: %s", attempt, e)
                        raise
                    exp_delay = min(max_delay, base_delay * (2 ** (attempt - 1)))
                    jittered_delay = random.uniform(0, exp_delay)
                    logger.warning(
                        "Attempt %d/%d failed (%s). Retrying in %.2fs...",
                        attempt, max_attempts, e, jittered_delay,
                    )
                    time.sleep(jittered_delay)
                    attempt += 1
        return wrapper
    return decorator


@retry_with_backoff(max_attempts=5, base_delay=0.3, retry_on=(TransientError,))
def call_flaky_api(success_probability=0.4):
    """Simulates a remote call that fails transiently most of the time."""
    if random.random() < success_probability:
        return {"status": "ok", "data": 42}
    raise TransientError("simulated 503 Service Unavailable")


_processed_requests = {}  # simulates a server-side dedupe store, keyed by idempotency key


def create_order_idempotent(order_payload: dict, idempotency_key: str) -> dict:
    """Simulates a POST /orders call that is safe to retry via an idempotency key."""
    if idempotency_key in _processed_requests:
        logger.info("Idempotency key %s already processed — returning cached result, no duplicate created.", idempotency_key)
        return _processed_requests[idempotency_key]

    logger.info("Processing NEW order for idempotency key %s ...", idempotency_key)
    result = {"order_id": f"ORD-{len(_processed_requests) + 1:04d}", "payload": order_payload}
    _processed_requests[idempotency_key] = result
    return result


def main():
    parser = argparse.ArgumentParser(description="Demonstrate retry with exponential backoff, jitter, and idempotency.")
    parser.add_argument("--max-attempts", type=int, default=5, help="Max retry attempts (default: 5)")
    parser.add_argument("--base-delay", type=float, default=0.3, help="Base delay in seconds for backoff (default: 0.3)")
    args = parser.parse_args()

    global call_flaky_api
    call_flaky_api = retry_with_backoff(
        max_attempts=args.max_attempts, base_delay=args.base_delay, retry_on=(TransientError,)
    )(call_flaky_api.__wrapped__)

    print("Part 1: Retry with exponential backoff + jitter")
    print("-" * 50)
    try:
        result = call_flaky_api(success_probability=0.4)
        print(f"SUCCESS: {result}")
    except TransientError as e:
        print(f"FAILED after all retries: {e}")

    print()
    print("Part 2: Idempotent retry (safe to call twice with same key)")
    print("-" * 50)
    key = "order-abc-123"
    first = create_order_idempotent({"item": "widget", "qty": 2}, key)
    print(f"First call result:  {first}")
    second = create_order_idempotent({"item": "widget", "qty": 2}, key)
    print(f"Retry call result:  {second}")
    print(f"Same order returned both times: {first == second} (no duplicate order was created)")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`retry_with_backoff` is a decorator factory: it wraps a function so that only exceptions listed in `retry_on` trigger a retry, with delay computed as `random.uniform(0, min(max_delay, base_delay * 2**(attempt-1)))` — full jitter, which avoids synchronized retry storms across many clients.

```bash
# Default: up to 5 attempts, 0.3s base delay
python3 retry_strategies.py

# More attempts, faster base delay
python3 retry_strategies.py --max-attempts 8 --base-delay 0.1

# Fewer attempts, slower base delay (more likely to exhaust retries)
python3 retry_strategies.py --max-attempts 2 --base-delay 0.5
```

#### Sample output

```
Part 1: Retry with exponential backoff + jitter
--------------------------------------------------
SUCCESS: {'status': 'ok', 'data': 42}

Part 2: Idempotent retry (safe to call twice with same key)
--------------------------------------------------
05:33:25 | INFO | Processing NEW order for idempotency key order-abc-123 ...
First call result:  {'order_id': 'ORD-0001', 'payload': {'item': 'widget', 'qty': 2}}
05:33:25 | INFO | Idempotency key order-abc-123 already processed — returning cached result, no duplicate created.
Retry call result:  {'order_id': 'ORD-0001', 'payload': {'item': 'widget', 'qty': 2}}
Same order returned both times: True (no duplicate order was created)
```

On a run where the API fails repeatedly, you'll instead see WARNING lines counting up attempts before either a late success or a final `FAILED after all retries` message.

#### Variations / flags

- `--max-attempts N` — cap total attempts; lower it for calls where giving up fast matters more than eventual success.
- `--base-delay S` — tune how aggressive the initial backoff is; higher values are gentler on a struggling downstream service.
- Add `max_delay` tuning as a flag too, to cap how long any single retry wait can get regardless of attempt count.
- Swap `retry_on=(TransientError,)` for a tuple including real exceptions your HTTP client raises (e.g. `requests.exceptions.ConnectionError`, or specific 5xx status handling).

#### Common pitfalls

- **Retrying non-idempotent operations** — retrying a raw "create order" call without an idempotency key can create duplicate orders/charges on the very failures that require a retry in the first place; always pair retries with idempotency for anything that mutates state.
- **Retrying permanent errors** — catching a broad exception type means retrying things like bad authentication or malformed requests, wasting time and delaying the real error message.
- **No jitter** — plain exponential backoff without randomization causes many clients to retry in sync ("thundering herd"), which can re-overwhelm a recovering service.
- **Unbounded `max_delay`** — without a cap, exponential growth can produce multi-minute waits on later attempts; always set a ceiling.

### 21.4 Secure Credential Handling

#### Use case

A sysadmin's deployment script needs an API token and a database password to connect to a service. Hardcoding either directly in the script means the secret ends up in git history, in Slack when someone pastes the script, and in shell history when it's `cat`-ed for debugging. The script instead needs to load both from environment variables (backed by a real secrets manager in production) and refuse to run — loudly — if either is missing.

#### Prerequisites

- Python 3.6+
- Standard library: `os`, `sys`
- Optional (recommended for local dev): `python-dotenv`, to auto-load a local `.env` file

```bash
python3 --version   # confirm 3.6+
pip install python-dotenv   # optional, for .env file support
```

- OS scope: Linux, macOS, Windows

#### Cautions

- **Never commit a `.env` file** containing real secrets — always add `.env` to `.gitignore` before creating one.
- This script's `connect_to_service()` function is a simulation; it does not make any real network connection, but the masking pattern shown (`mask_secret`) should be applied to any real logging of credentials.

#### Script

```python
#!/usr/bin/env python3
"""
secure_credential_handling.py

Demonstrates SECURE credential handling: never hardcoding secrets in
source code, loading them from environment variables (optionally via
a local, git-ignored .env file for development), masking secrets in
any logged/printed output, and failing fast with a clear error if a
required secret is missing.
"""

import os
import sys

try:
    from dotenv import load_dotenv
    HAS_DOTENV = True
except ImportError:
    HAS_DOTENV = False


REQUIRED_SECRETS = ["API_TOKEN", "DB_PASSWORD"]


def mask_secret(value: str, keep=4) -> str:
    """Return a masked version of a secret, safe to print/log."""
    if value is None:
        return "<not set>"
    if len(value) <= keep * 2:
        return "*" * len(value)
    return f"{value[:keep]}{'*' * (len(value) - keep * 2)}{value[-keep:]}"


def load_environment():
    """Load a local .env file for development, if python-dotenv is available."""
    if HAS_DOTENV:
        loaded = load_dotenv()
        if loaded:
            print("Loaded local .env file for development.")
    else:
        print("(Tip: install python-dotenv to auto-load a local .env file: pip install python-dotenv)")


def get_required_secret(name: str) -> str:
    """Fetch a required secret from the environment, failing fast if missing."""
    value = os.environ.get(name)
    if not value:
        raise EnvironmentError(
            f"Required secret '{name}' is not set. "
            f"Set it as an environment variable (or in a local .env file "
            f"that is listed in .gitignore) before running this script. "
            f"In production, this should come from your secrets manager "
            f"or vault (e.g. AWS Secrets Manager, HashiCorp Vault, Azure Key Vault)."
        )
    return value


def connect_to_service(api_token: str, db_password: str):
    """Simulates using credentials, logging only masked versions."""
    print(f"Connecting with API_TOKEN={mask_secret(api_token)}  DB_PASSWORD={mask_secret(db_password)}")
    print("(In a real script, the raw values would be used here to authenticate, "
          "but NEVER printed, logged, or committed to source control.)")


def main():
    load_environment()

    print()
    print("Secure Credential Handling Demo")
    print("=" * 45)

    missing = [name for name in REQUIRED_SECRETS if not os.environ.get(name)]
    if missing:
        print(f"ERROR: missing required secret(s): {', '.join(missing)}")
        print()
        print("To try this script with real values, either:")
        print("  1) Export them in your shell:")
        for name in missing:

### 21.5 Idempotent Automation

#### Use case

A sysadmin's provisioning script needs to run every time a server boots (or every time a config-management tool like Ansible re-applies its playbook). It must be safe to run the same script a hundred times in a row: creating a directory that already exists shouldn't error, appending a config line that's already there shouldn't duplicate it, and reprocessing a file that hasn't changed shouldn't waste time or overwrite good output with identical output.

#### Prerequisites

- Python 3.6+
- Standard library only: `hashlib`, `json`, `os`
- OS scope: Linux, macOS, Windows

```bash
python3 --version   # confirm 3.6+
# no external dependencies to install
```

#### Cautions

- The script creates a real directory (`idempotent_demo/`) and files in your current working directory — run it somewhere you don't mind leaving small test artifacts, or clean up afterward with `rm -rf idempotent_demo`.
- Run it **twice in a row** to actually see the idempotency proof; a single run only shows the "changes made" path.

#### Script

```python
#!/usr/bin/env python3
"""
idempotent_automation.py

Demonstrates designing automation that is IDEMPOTENT — safe to run
over and over with the same end result. Covers three patterns:
"ensure directory exists", "ensure a config line is present exactly
once", and "only do work if the output is missing/outdated" using a
checksum/state file.
"""

import hashlib
import json
import os


WORKSPACE = "idempotent_demo"
CONFIG_FILE = os.path.join(WORKSPACE, "app.conf")
STATE_FILE = os.path.join(WORKSPACE, ".state.json")
DATA_FILE = os.path.join(WORKSPACE, "input.txt")
PROCESSED_FILE = os.path.join(WORKSPACE, "processed.txt")

DESIRED_CONFIG_LINE = "feature_flag_enabled=true"


def ensure_directory(path: str) -> bool:
    """Create a directory only if it doesn't already exist. Returns True if it made a change."""
    if os.path.isdir(path):
        return False
    os.makedirs(path, exist_ok=True)
    return True


def ensure_line_in_file(path: str, line: str) -> bool:
    """Ensure a specific line exists in a file EXACTLY once. Safe to call repeatedly."""
    existing_lines = []
    if os.path.isfile(path):
        with open(path, "r", encoding="utf-8") as f:
            existing_lines = [l.rstrip("\n") for l in f]

    if line in existing_lines:
        return False

    with open(path, "a", encoding="utf-8") as f:
        f.write(line + "\n")
    return True


def file_checksum(path: str) -> str:
    with open(path, "rb") as f:
        return hashlib.sha256(f.read()).hexdigest()


def load_state() -> dict:
    if os.path.isfile(STATE_FILE):
        with open(STATE_FILE, "r", encoding="utf-8") as f:
            return json.load(f)
    return {}


def save_state(state: dict) -> None:
    with open(STATE_FILE, "w", encoding="utf-8") as f:
        json.dump(state, f, indent=2)


def process_if_changed(input_path: str, output_path: str) -> bool:
    """Only re-process input_path if its checksum has changed since the last successful run."""
    state = load_state()
    current_checksum = file_checksum(input_path)

    if state.get("input_checksum") == current_checksum and os.path.isfile(output_path):
        return False

    with open(input_path, "r", encoding="utf-8") as f:
        content = f.read()
    with open(output_path, "w", encoding="utf-8") as f:
        f.write(content.upper())

    state["input_checksum"] = current_checksum
    save_state(state)
    return True


def main():
    print("Idempotent Automation Demo")
    print("=" * 40)

    changed = []

    if ensure_directory(WORKSPACE):
        changed.append(f"created directory '{WORKSPACE}/'")

    if ensure_line_in_file(CONFIG_FILE, DESIRED_CONFIG_LINE):
        changed.append(f"added config line to '{CONFIG_FILE}'")

    if not os.path.isfile(DATA_FILE):
        with open(DATA_FILE, "w", encoding="utf-8") as f:
            f.write("hello idempotent world\n")
        changed.append(f"created sample input file '{DATA_FILE}'")

    if process_if_changed(DATA_FILE, PROCESSED_FILE):
        changed.append(f"processed '{DATA_FILE}' -> '{PROCESSED_FILE}'")

    if changed:
        print("Changes made this run:")
        for c in changed:
            print(f"  - {c}")
    else:
        print("No changes needed — system was already in the desired state.")
        print("(This is the goal: running this script again is always safe.)")

    print()
    print(f"Workspace: {os.path.abspath(WORKSPACE)}")
    print("Run this script again to see idempotency in action — it should report no changes.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

No flags — run it, then run it again immediately to see the difference.

```bash

# Second run: should report no changes at all

# Clean up the demo workspace when done

```

#### Variations / flags

- Change the contents of `DATA_FILE` between runs (e.g. `echo "new content" >> idempotent_demo/input.txt`) and re-run — `process_if_changed` will detect the checksum mismatch and reprocess only that step, leaving the directory/config-line steps untouched.
- Swap the checksum algorithm in `file_checksum` (e.g. `hashlib.md5`) if SHA-256 is overkill for your use case — for pure change-detection (not security), MD5 is faster and adequate.
- Wrap `ensure_directory` / `ensure_line_in_file` into a general-purpose "state enforcement" library shared across multiple provisioning scripts.

#### Common pitfalls

- **Using `open(path, "a")` (append mode) unconditionally** without checking whether content already exists — the single most common way idempotency breaks; always check first, then act.
- **Checksums computed on files that include timestamps or run-specific data** — if `input.txt` itself embeds a "generated at" timestamp, its checksum will differ every run even when nothing meaningful changed, defeating the point of `process_if_changed`.
- **State file corruption or deletion** — if `.state.json` is deleted but `processed.txt` still exists, the script will treat the input as changed and reprocess it; this is safe (idempotent) but worth knowing so you're not surprised by unexpected reprocessing.
- **Directory creation race conditions** — in highly concurrent automation (multiple processes provisioning the same path simultaneously), prefer `os.makedirs(path, exist_ok=True)` (already used here) over checking-then-creating separately, which has a TOCTOU race.

### 21.6 Modular Scripting

#### Use case

A sysadmin needs a quick system-report tool (OS, Python version, CPU count, disk usage, working directory) for health checks across a fleet of servers. Written as one giant function, none of the individual data-gathering pieces can be reused elsewhere or tested without running the whole thing. Written modularly, `gather_disk_info()` alone can be imported into a separate disk-alerting script without dragging along the rest.

#### Prerequisites

- Python 3.7+ (uses `dataclasses`, added in 3.7)
- Standard library only: `os`, `platform`, `shutil`, `dataclasses`, `typing`
- OS scope: Linux, macOS, Windows (disk usage path `"."` resolves correctly cross-platform via `shutil.disk_usage`)

```bash
python3 --version   # confirm 3.7+
# no external dependencies to install
```

#### Cautions

- `gather_disk_info(".")` reports usage for whatever drive/filesystem the current working directory is on — on Windows this will be the drive letter of your CWD, not necessarily `C:`.
- `os.cpu_count()` can return `None` on some restricted/containerized environments; the script guards this with `or 0`.

#### Script

```python
#!/usr/bin/env python3
"""
modular_scripting.py

Demonstrates breaking a "monolithic" automation script into small,
reusable, independently-testable functions and a class — instead of
one giant script with everything inline.
"""

import os
import platform
import shutil
from dataclasses import dataclass, field
from typing import Callable, Dict


def gather_platform_info() -> Dict[str, str]:
    """Single responsibility: describe the OS/platform."""
    return {
        "system": platform.system(),
        "release": platform.release(),
        "machine": platform.machine(),
    }


def gather_python_info() -> Dict[str, str]:
    """Single responsibility: describe the Python runtime."""
    return {
        "version": platform.python_version(),
        "implementation": platform.python_implementation(),
    }


def gather_cpu_info() -> Dict[str, int]:
    """Single responsibility: describe CPU resources."""
    return {"logical_cpus": os.cpu_count() or 0}


def gather_disk_info(path: str = ".") -> Dict[str, int]:
    """Single responsibility: describe disk usage for a given path."""
    total, used, free = shutil.disk_usage(path)
    gb = 1024 ** 3
    return {
        "total_gb": round(total / gb, 1),
        "used_gb": round(used / gb, 1),
        "free_gb": round(free / gb, 1),
    }


def gather_cwd_info() -> Dict[str, str]:
    """Single responsibility: describe the current working directory."""
    return {"cwd": os.getcwd()}


@dataclass
class SystemReportBuilder:
    """Coordinates independent data-gathering functions into one report."""

    sections: Dict[str, Callable[[], dict]] = field(default_factory=lambda: {
        "Platform": gather_platform_info,
        "Python": gather_python_info,
        "CPU": gather_cpu_info,
        "Disk (current path)": gather_disk_info,
        "Working Directory": gather_cwd_info,
    })

    def build(self) -> Dict[str, dict]:
        return {name: fn() for name, fn in self.sections.items()}

    def render_text(self) -> str:
        report = self.build()
        lines = ["System Report", "=" * 40]
        for section, data in report.items():
            lines.append(f"\n[{section}]")
            for key, value in data.items():
                lines.append(f"  {key}: {value}")
        return "\n".join(lines)


def main():
    builder = SystemReportBuilder()
    print(builder.render_text())
    print()
    print("Each section above came from its own small, reusable function —")
    print("e.g. gather_disk_info() can be imported and reused in other scripts.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

No flags — `SystemReportBuilder` composes five independent functions into one report via its `sections` dict, which can be extended without modifying any existing function.

```bash
python3 modular_scripting.py
```

To reuse a single piece in another script:

```bash
python3 -c "from modular_scripting import gather_disk_info; print(gather_disk_info('/'))"
```

#### Sample output

```

========================================

[Platform]
  system: Linux
  release: 6.18.5
  machine: x86_64

[Python]
  version: 3.12.3
  implementation: CPython

[CPU]
  logical_cpus: 1

[Disk (current path)]
  total_gb: 252.0
  used_gb: 8.6
  free_gb: 10.0

[Working Directory]
  cwd: /home/user/scripts

Each section above came from its own small, reusable function —
e.g. gather_disk_info() can be imported and reused in other scripts.
```

Exact values (release string, CPU count, disk sizes, cwd) will reflect your own machine.

#### Variations / flags

- Add a new section by writing one more small function and adding it to `sections`, e.g. `gather_network_info()` for IP addresses — no existing code needs to change.
- Pass a custom `sections` dict to `SystemReportBuilder(sections={...})` to build a report with only a subset of checks, useful for a lightweight health-check variant.
- Add `--json` output by serializing `builder.build()` with `json.dumps(..., indent=2)` instead of `render_text()`, for feeding into monitoring pipelines.

#### Common pitfalls

- **One giant `main()` with everything inline** — the anti-pattern this script avoids; makes unit testing nearly impossible since you can't isolate "just the disk check" without running the whole script and its side effects.
- **Functions with hidden shared state** — each `gather_*` function here takes no arguments (except `gather_disk_info`'s optional `path`) and returns a plain dict, making them trivially testable with `assert gather_cpu_info()["logical_cpus"] >= 0`; avoid reaching into global variables from within these functions.
- **Overusing classes for stateless operations** — note the gathering functions are plain functions, not methods; only the *coordination* (`SystemReportBuilder`) benefits from being a class. Don't force single-responsibility functions into classes unnecessarily.
- **`shutil.disk_usage` on a path that doesn't exist** — raises `FileNotFoundError`; if extending `gather_disk_info` to check arbitrary paths, validate the path exists first or handle the exception per the patterns in 21.1.

### 21.7 Configuration Management


#### Use case

A sysadmin's script currently has `retries = 3` and `"https://api.example.com"` buried in the code. Moving it from staging to production means editing source code and redeploying just to change a URL or a timeout. The script instead needs a config file that ships with sensible defaults, plus the ability to override individual values per-environment via environment variables (common in Docker/Kubernetes/CI) without maintaining separate config files per environment.

#### Prerequisites

- Python 3.6+
- Recommended: `PyYAML` for full YAML parsing (the script degrades gracefully to a minimal hand-rolled parser without it, but with reduced fidelity)
- OS scope: Linux, macOS, Windows

```bash
python3 --version   # confirm 3.6+
pip install pyyaml   # recommended
```

#### Cautions

- Without PyYAML installed, the fallback config writer/reader is a simplified `key: value` format — it does **not** support nested YAML structures, lists, or proper type coercion beyond what `ENV_OVERRIDES` casts explicitly. Install PyYAML for anything beyond the flat demo config.
- Running the script for the first time creates `config.yaml` in the current directory; delete it if you want to see the "no config file found" path again.

#### Script

```python
#!/usr/bin/env python3
"""
configuration_management.py

Demonstrates externalizing "magic numbers" and hardcoded strings into
a configuration file, with sane defaults, environment-variable
overrides for deployment-specific values, and validation.
"""

import argparse
import os
import sys

try:
    import yaml
    HAS_YAML = True
except ImportError:
    HAS_YAML = False


DEFAULT_CONFIG = {
    "app_name": "backup-automation",
    "max_retries": 3,
    "timeout_seconds": 30,
    "api_base_url": "https://api.example.com",
    "log_level": "INFO",
    "batch_size": 100,
}

ENV_OVERRIDES = {
    "APP_MAX_RETRIES": ("max_retries", int),
    "APP_TIMEOUT_SECONDS": ("timeout_seconds", int),
    "APP_API_BASE_URL": ("api_base_url", str),
    "APP_LOG_LEVEL": ("log_level", str),
    "APP_BATCH_SIZE": ("batch_size", int),
}


def write_default_config(path: str) -> None:
    if HAS_YAML:
        with open(path, "w", encoding="utf-8") as f:
            yaml.safe_dump(DEFAULT_CONFIG, f, sort_keys=False)
    else:
        with open(path, "w", encoding="utf-8") as f:
            for key, value in DEFAULT_CONFIG.items():
                f.write(f"{key}: {value}\n")


def load_config_file(path: str) -> dict:
    if not os.path.isfile(path):
        print(f"No config file found at '{path}'; creating one with defaults.")
        write_default_config(path)
        return dict(DEFAULT_CONFIG)

    if HAS_YAML:
        with open(path, "r", encoding="utf-8") as f:
            data = yaml.safe_load(f) or {}
    else:
        print("(Tip: install PyYAML for full YAML support: pip install pyyaml)")
        data = {}
        with open(path, "r", encoding="utf-8") as f:
            for line in f:
                if ":" in line:
                    key, _, value = line.partition(":")
                    data[key.strip()] = value.strip()

    merged = dict(DEFAULT_CONFIG)
    merged.update(data)
    return merged


def apply_env_overrides(config: dict) -> dict:
    """Environment variables take precedence over the config file."""
    applied = []
    for env_var, (key, caster) in ENV_OVERRIDES.items():
        if env_var in os.environ:
            try:
                config[key] = caster(os.environ[env_var])
                applied.append(env_var)
            except ValueError:
                print(f"WARNING: could not parse {env_var}={os.environ[env_var]!r} as {caster.__name__}; ignoring.")
    return config, applied


def validate_config(config: dict) -> None:
    if config["max_retries"] < 0:
        raise ValueError("max_retries must be >= 0")
    if config["timeout_seconds"] <= 0:
        raise ValueError("timeout_seconds must be > 0")
    if config["batch_size"] <= 0:
        raise ValueError("batch_size must be > 0")


def run_task(config: dict) -> None:
    print(f"\nRunning '{config['app_name']}' against {config['api_base_url']} "
          f"(timeout={config['timeout_seconds']}s, batch_size={config['batch_size']})")
    for attempt in range(1, config["max_retries"] + 1):
        print(f"  attempt {attempt}/{config['max_retries']} ... (simulated)")
    print("Task complete.")


def main():
    parser = argparse.ArgumentParser(description="Demonstrate configuration management for automation scripts.")
    parser.add_argument("--config", default="config.yaml", help="Path to the YAML config file (default: config.yaml)")
    args = parser.parse_args()

    print("Configuration Management Demo")
    print("=" * 40)

    config = load_config_file(args.config)
    config, overridden = apply_env_overrides(config)

    try:
        validate_config(config)
    except ValueError as e:
        print(f"ERROR: invalid configuration: {e}", file=sys.stderr)
        sys.exit(1)

    print(f"\nEffective configuration (from '{args.config}', env overrides: {overridden or 'none'}):")
    for key, value in config.items():
        source = "env var" if key in [ENV_OVERRIDES[e][0] for e in overridden] else "config file/default"
        print(f"  {key}: {value}   [{source}]")

    run_task(config)


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`load_config_file` creates `config.yaml` with defaults on first run, merges any partial file over the defaults, then `apply_env_overrides` lets specific environment variables override individual keys without touching the file.

```bash

# Point at a custom config path
python3 configuration_management.py --config custom_config.yaml

# Override a single value without editing the file

# Override multiple values (e.g. for a staging deployment)
APP_API_BASE_URL="https://staging.api.example.com" APP_TIMEOUT_SECONDS=60 python3 configuration_management.py
```

#### Sample output

```

========================================
No config file found at 'config.yaml'; creating one with defaults.

Effective configuration (from 'config.yaml', env overrides: none):
  app_name: backup-automation   [config file/default]
  max_retries: 3   [config file/default]
  timeout_seconds: 30   [config file/default]
  api_base_url: https://api.example.com   [config file/default]
  log_level: INFO   [config file/default]
  batch_size: 100   [config file/default]

Running 'backup-automation' against https://api.example.com (timeout=30s, batch_size=100)
  attempt 1/3 ... (simulated)

```

With `APP_MAX_RETRIES=10`, the `max_retries` line instead reads `max_retries: 10   [env var]` and the task loop runs 10 simulated attempts.

#### Variations / flags

- `--config custom_config.yaml` — maintain separate named configs per environment (`config.dev.yaml`, `config.prod.yaml`) while sharing the same script.
- Add new keys by extending both `DEFAULT_CONFIG` and, if you want env-var overrides for them, `ENV_OVERRIDES`.
- Add stricter validation in `validate_config` (e.g. URL format checking) as the config surface grows.
- Combine with 21.4 (Secure Credential Handling) — keep secrets in environment variables/`.env`, and non-secret tuning knobs in this YAML file; don't mix the two.

#### Common pitfalls

- **Hardcoding literals directly in business logic** (`for attempt in range(3)`) — the exact anti-pattern this script avoids; makes every environment-specific tweak require a code change and redeploy.
- **PyYAML not installed in production** — the fallback parser is intentionally minimal and will mis-handle anything beyond flat `key: value` pairs; always pin `pyyaml` in your `requirements.txt` for real deployments.
- **Env var type coercion failures** — setting `APP_MAX_RETRIES=abc` triggers a caught `ValueError` with a warning and the value is ignored (falls back to file/default) rather than crashing; verify the warning appears if you're debugging an override that "isn't taking."
- **Config drift between file and environment** — because env vars silently take precedence, a value in `config.yaml` can look right while the *effective* running value differs; the "effective configuration" printout with `[env var]`/`[config file/default]` tags exists specifically to make this visible — don't skip printing it in your own scripts.

### 21.8 Testing Automation Scripts

#### Use case

A sysadmin has a script that wraps the `df` command to report disk usage as part of a monitoring tool. Testing it "for real" means the test's result depends on the actual disk usage of whatever machine runs the test suite — impossible to assert a fixed expected value, slow, and non-portable to CI runners with different disk layouts. Mocking `subprocess.run` lets the test suite verify the *parsing logic* and *error handling* deterministically, in milliseconds, without running `df` at all.

#### Prerequisites

- Python 3.6+
- Standard library: `shutil`, `subprocess`, `sys`
- For running the tests: `pytest` (the script's test functions are skipped gracefully if pytest isn't installed)

```bash
python3 --version   # confirm 3.6+
pip install pytest
```

- OS scope: the automation function itself has a Windows fallback (via `shutil.disk_usage`) for systems without `df`; the test suite runs identically on Linux, macOS, and Windows since `subprocess.run` is mocked, not actually executed.

#### Cautions

- Running the script **directly** (`python3 testing_automation_scripts.py`) executes a real `df` command against your actual filesystem — this is expected and different from running the test suite.
- Running with **pytest** never touches the real filesystem or shells out — if you see real disk percentages during a `pytest` run, something is wrong with the mock setup.

#### Script

```python
#!/usr/bin/env python3
"""
testing_automation_scripts.py

Demonstrates writing unit tests for automation code that shells out
via `subprocess`, using `pytest` and `unittest.mock` to avoid actually
running real commands during tests.
"""

import shutil
import subprocess
import sys


def get_disk_usage_percent(path: str = ".") -> int:
    """
    A small automation function that wraps `subprocess` to call the
    'df' command (falls back to shutil on platforms without df) and
    returns disk usage as a percentage. This is the function under test.
    """
    try:
        result = subprocess.run(
            ["df", "-P", path],
            capture_output=True,
            text=True,
            timeout=5,
            check=True,
        )
    except FileNotFoundError:
        total, used, _free = shutil.disk_usage(path)
        return int(used / total * 100)
    except subprocess.CalledProcessError as e:
        raise RuntimeError(f"'df' command failed with exit code {e.returncode}: {e.stderr}") from e
    except subprocess.TimeoutExpired as e:
        raise RuntimeError(f"'df' command timed out: {e}") from e

    lines = result.stdout.strip().splitlines()
    if len(lines) < 2:
        raise RuntimeError(f"Unexpected 'df' output: {result.stdout!r}")
    fields = lines[1].split()
    capacity_str = fields[4].rstrip("%")
    return int(capacity_str)


def main():
    try:
        pct = get_disk_usage_percent(".")
        print(f"Disk usage for current path: {pct}%")
    except RuntimeError as e:
        print(f"Could not determine disk usage: {e}", file=sys.stderr)
        sys.exit(1)

#### Common pitfalls

- **Testing against real subprocess output** — makes tests slow, flaky across environments (different disk sizes, different `df` output formats on BSD vs. GNU), and sometimes destructive if the command under test has side effects; always mock `subprocess.run` for unit tests, and reserve real-command execution for a separate, clearly-labeled integration test suite.
- **Patching the wrong target** — `patch("subprocess.run", ...)` patches the name as looked up in the `subprocess` module; if the code under test did `from subprocess import run` instead, you'd need `patch("testing_automation_scripts.run", ...)` instead. Always patch where the name is *used*, not where it's *defined*.
- **Forgetting `capture_output=True, text=True`** parity between the real call and the mock's `MagicMock` attributes — if the mock doesn't set `.stdout` to a string, `.strip().splitlines()` will fail with an `AttributeError` on the `MagicMock` itself.
- **`pytest` not installed** — the test functions are wrapped in `if HAS_PYTEST:` so the file still imports and `main()` still runs without pytest present; but obviously no tests will execute until it's installed.

### 21.9 Packaging Automation Tools

#### Use case

A sysadmin has written a useful internal CLI tool as a single script and now wants to share it with the rest of the team. Copying the raw `.py` file into everyone's home directory doesn't scale: there's no versioning, no dependency management, and no clean way to invoke it as a real command (`my-internal-tool --example hello`) instead of `python3 /some/path/tool.py`. This script scaffolds the modern, minimal package layout needed to fix that.

#### Prerequisites

- Python 3.8+ (matches the `requires-python` set in the generated `pyproject.toml`)
- Standard library only for the scaffolder itself: `argparse`, `os`
- To actually install the generated package: `pip` with build support for `pyproject.toml` (standard in any reasonably current `pip`)

```bash
python3 --version   # confirm 3.8+
pip --version        # confirm pip is available
```

- OS scope: Linux, macOS, Windows (the generated package and its `pip install -e .` step work identically on all three)

#### Cautions

- The script **refuses to run** if the target output directory already exists, to avoid silently overwriting an existing package — remove or rename the directory first if you need to regenerate it.
- The generated package is a skeleton with a placeholder `--example` CLI flag; you'll need to replace `cli.py`'s contents with real functionality before distributing it for actual use.

#### Script

```python
#!/usr/bin/env python3
"""
packaging_automation_tools.py

Demonstrates packaging an internal automation tool for distribution,
by SCAFFOLDING a minimal, correct, installable Python package.
"""

import argparse
import os


PYPROJECT_TEMPLATE = """\
[build-system]
requires = ["setuptools>=68.0"]
build-backend = "setuptools.build_meta"

[project]
name = "{dist_name}"
version = "0.1.0"

    {{ name = "Your Team", email = "automation@example.com" }}
]

    # list runtime dependencies here, e.g. "requests>=2.31",
]

[project.scripts]
{cli_command} = "{package_name}.cli:main"

[tool.setuptools.packages.find]
where = ["src"]
"""

INIT_TEMPLATE = '''"""{package_name} — internal automation tool."""

__version__ = "0.1.0"
'''

CLI_TEMPLATE = '''"""Command-line entry point for {package_name}."""

import argparse


def main():
    parser = argparse.ArgumentParser(prog="{cli_command}", description="{description}")
    parser.add_argument("--example", help="An example option — replace with real functionality.")
    args = parser.parse_args()

    print(f"{{parser.prog}} ran successfully. example={{args.example}}")


if __name__ == "__main__":
    main()
'''

README_TEMPLATE = """\
# {dist_name}

Internal automation tool: {description}

## Install (editable, for development)

```bash
pip install -e .
```

#### Install (for internal distribution)

Publish this package to your internal package index (e.g. Artifactory,
Nexus, devpi, or a private GitHub/GitLab package registry), then:

```bash
pip install {dist_name} --index-url https://your-internal-index/simple/
```

#### Usage

```bash
{cli_command} --example hello
```
"""


def to_package_name(dist_name: str) -> str:
    """Convert a dash-separated distribution name to a valid Python package name."""
    return dist_name.replace("-", "_")


def scaffold_package(dist_name: str, description: str, outdir: str) -> None:
    package_name = to_package_name(dist_name)
    cli_command = dist_name

    src_dir = os.path.join(outdir, "src", package_name)
    os.makedirs(src_dir, exist_ok=True)

    with open(os.path.join(outdir, "pyproject.toml"), "w", encoding="utf-8") as f:
        f.write(PYPROJECT_TEMPLATE.format(
            dist_name=dist_name, description=description,
            cli_command=cli_command, package_name=package_name,
        ))

    with open(os.path.join(outdir, "README.md"), "w", encoding="utf-8") as f:
        f.write(README_TEMPLATE.format(dist_name=dist_name, description=description, cli_command=cli_command))

    with open(os.path.join(src_dir, "__init__.py"), "w", encoding="utf-8") as f:
        f.write(INIT_TEMPLATE.format(package_name=package_name))

    with open(os.path.join(src_dir, "cli.py"), "w", encoding="utf-8") as f:
        f.write(CLI_TEMPLATE.format(package_name=package_name, cli_command=cli_command, description=description))


def main():
    parser = argparse.ArgumentParser(description="Scaffold an installable package for an internal automation tool.")
    parser.add_argument("--name", default="my-internal-tool", help="Distribution name, dash-separated (default: my-internal-tool)")
    parser.add_argument("--description", default="A reusable internal automation CLI.", help="Short package description")
    parser.add_argument("--outdir", default=None, help="Output folder (default: same as --name, underscored)")
    args = parser.parse_args()

    outdir = args.outdir or to_package_name(args.name)

    if os.path.exists(outdir):
        print(f"ERROR: output folder '{outdir}' already exists. Choose a different --outdir or remove it first.")
        return

    scaffold_package(args.name, args.description, outdir)

    print("Packaging Automation Tools Demo")
    print("=" * 40)
    print(f"Scaffolded installable package at: {os.path.abspath(outdir)}")
    print()
    print("Structure created:")
    for root, _dirs, files in os.walk(outdir):
        for fname in files:
            print(" ", os.path.relpath(os.path.join(root, fname), outdir))
    print()
    print("Next steps:")
    print(f"  cd {outdir}")
    print("  pip install -e .")
    print(f"  {args.name} --example hello")


if __name__ == "__main__":
    main()
```

#### Variations / flags

- `--name` — sets both the distribution name (in `pyproject.toml`) and the installed CLI command name; use dashes (`my-tool`), the script converts to underscores automatically for the importable package name (`my_tool`).
- `--description` — short text embedded in both `pyproject.toml` and the generated `README.md`.
- `--outdir` — where to write the scaffold; defaults to the underscored version of `--name`.
- After scaffolding, add real dependencies to the `dependencies = [...]` list in `pyproject.toml` (e.g. `"requests>=2.31"`) before running `pip install -e .` again.
- For internal distribution beyond one machine, `python -m build` the package and upload the resulting wheel to an internal index (Artifactory/Nexus/devpi) as noted in the generated README.

#### Common pitfalls

- **Running the scaffolder against an existing directory** — the script exits early with an error rather than overwriting; this is intentional, but it means re-running with the same `--name`/`--outdir` twice requires manually removing the old directory first.
- **Forgetting `-e` on `pip install`** — installing without `-e` (editable mode) copies the code at install time; further edits to `cli.py` won't take effect until you reinstall. Use `-e` during development.
- **CLI command name collisions** — if `--name` matches an existing command already on `PATH` (e.g. naming your tool `test` or `backup`), `pip install -e .` will shadow it only within environments where this package is installed; choose a distinctive name.
- **Missing `build-system` table** — if you hand-edit `pyproject.toml` and remove or misconfigure the `[build-system]` section, `pip install -e .` will fail with a backend error; keep it intact unless you know you're swapping build backends (e.g. to `hatchling` or `poetry-core`).

### 21.10 Cross-Platform Scripting


#### Use case

A sysadmin's team runs the same automation scripts across a mix of Linux servers, developer Macs, and a few Windows machines used by junior staff. A script hardcoding `"/dev/null"` or joining paths with `"/"` will silently break the moment someone runs it on Windows. This script demonstrates the portable equivalents that work identically everywhere, plus the one legitimate pattern for branching on OS (choosing the right `ping` flags).

#### Prerequisites

- Python 3.4+ (uses `pathlib`, added in 3.4)
- Standard library only: `os`, `platform`, `subprocess`, `sys`, `pathlib`, `tempfile`
- OS scope: Linux, macOS, Windows — this script's entire purpose is running identically on all three

```bash
python3 --version   # confirm 3.4+
# no external dependencies to install
```

#### Cautions

- The demo command (`ping`) may not exist in minimal/containerized environments (common in CI runners and some Docker images) — the script handles this gracefully and reports it, it is not a failure of the portability pattern itself.
- `subprocess.run(..., timeout=5)` will raise `subprocess.TimeoutExpired` if `ping` hangs; this is caught only as `FileNotFoundError` in the current script, so a genuinely hanging `ping` would still raise — acceptable for a demo, but worth hardening in production code per the retry/exception patterns in 21.1 and 21.3.

#### Script

```python
#!/usr/bin/env python3
"""
cross_platform_scripting.py

Demonstrates writing automation that works correctly on Windows,
macOS, and Linux without modification: using `pathlib` instead of
hardcoded path separators, `os.devnull` instead of a hardcoded null
device path, detecting the OS only when truly necessary.
"""

import os
import platform
import subprocess
import sys
from pathlib import Path


def build_portable_path() -> Path:
    """pathlib's `/` operator uses the correct separator for the current OS."""
    return Path("automation_workspace") / "logs" / "app.log"


def get_null_device() -> str:
    """os.devnull is '/dev/null' on Unix and 'nul' on Windows — never hardcode either."""
    return os.devnull


def get_path_separator() -> str:
    """os.pathsep is ':' on Unix and ';' on Windows (used to split $PATH / %PATH%)."""
    return os.pathsep


def run_command_discarding_output(command: list) -> int:
    """Run a command and discard its output, portably, using os.devnull."""
    with open(os.devnull, "w") as devnull:
        try:
            result = subprocess.run(command, stdout=devnull, stderr=devnull, timeout=5)
            return result.returncode
        except FileNotFoundError:
            return -1


def get_temp_dir_portable() -> Path:
    """Use tempfile for a portable temp directory instead of hardcoding /tmp."""
    import tempfile
    return Path(tempfile.gettempdir())


def detect_os_when_necessary() -> str:
    """
    Most code should be OS-agnostic via pathlib/os.*; only branch on the
    OS name for the rare cases that truly need it.
    """
    system = platform.system()
    if system == "Windows":
        ping_count_flag = "-n"
    else:
        ping_count_flag = "-c"
    return f"{system} detected -> would use 'ping {ping_count_flag} 1 host' for a single ping"


def main():
    print("Cross-Platform Scripting Demo")
    print("=" * 40)
    print(f"Detected OS:        {platform.system()} ({platform.platform()})")
    print(f"Path separator:     {os.sep!r}   (used by pathlib automatically)")
    print(f"PATH list sep:      {get_path_separator()!r}   (os.pathsep, splits $PATH / %PATH%)")
    print(f"Null device:        {get_null_device()!r}   (os.devnull)")
    print(f"Portable path:      {build_portable_path()}")
    print(f"Portable temp dir:  {get_temp_dir_portable()}")
    print(f"OS-specific branch: {detect_os_when_necessary()}")

    ping_target = "127.0.0.1"
    if platform.system() == "Windows":
        command = ["ping", "-n", "1", ping_target]
    else:
        command = ["ping", "-c", "1", ping_target]

    print(f"\nRunning a portable, output-discarding command: {' '.join(command)}")
    rc = run_command_discarding_output(command)
    if rc == 0:
        print("Command succeeded (output was discarded via os.devnull, not a hardcoded path).")
    elif rc == -1:
        print("'ping' not available in this environment — that's fine, the portability pattern still applies.")
    else:
        print(f"Command exited with code {rc}.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

No flags — the script self-detects the OS and prints the portable equivalents for each platform-sensitive value, then runs a small demo command that discards its output via `os.devnull`.

```bash

```

#### Variations / flags

- Add a `--target` argument for the ping demo instead of hardcoding `127.0.0.1`.
- Extend `detect_os_when_necessary()` with more OS-specific branches as genuinely needed (e.g. choosing between `systemctl` on Linux and `sc.exe` on Windows for service management) — but keep the branch as narrow as possible; only the parts that truly differ should be inside the `if`.
- Replace the `ping` demo with any other command whose output you want to discard portably, such as a health-check script.

#### Common pitfalls

- **String-concatenating paths** (`"logs" + "/" + "app.log"`) — breaks on Windows; always use `pathlib.Path` with the `/` operator or `os.path.join`.
- **Hardcoding `/dev/null`** — silently breaks automation the first time it runs on Windows; use `os.devnull` everywhere output needs to be discarded.
- **Hardcoding `/tmp` for temporary files** — doesn't exist on Windows; use `tempfile.gettempdir()` instead.
- **Over-branching on OS** — writing `if platform.system() == "Windows": ... else: ...` around code that could have simply used a portable stdlib function (like `pathlib`) adds unnecessary complexity and doubles the surface area to test; reserve OS branches for cases with no portable equivalent, like choosing between genuinely different command-line tools.
- **`ping` missing in containers** — many minimal Docker images and CI runners don't ship `ping`; the script's `FileNotFoundError` handling covers this, but don't assume common CLI tools are always present in automation environments.

### 21.11 Performance Optimization

#### Use case

A sysadmin's log-processing script has gotten slow, and there are several plausible culprits. Rather than guessing and rewriting code speculatively, the script needs to be profiled first to find the *actual* bottleneck — which in this common case turns out to be checking "is this hostname in my allow-list?" against a Python list inside a loop, an easy mistake to make when the list was written quickly and never revisited as the data grew.

#### Prerequisites

- Python 3.x (uses `cProfile`, `pstats`, `io`, `time` — all standard library)
- No `pip install` required
- OS scope: Linux, macOS, Windows

```bash
python3 --version
# no external dependencies to install
```

#### Cautions

- The unoptimized demo (`slow_membership_checks`) is deliberately slow (O(n²)-ish behavior with `DATA_SIZE = 20_000` and `LOOKUP_COUNT = 20_000`) and will take on the order of a second or more depending on your CPU — this is intentional, to make the profiling difference visible.
- The script writes `profile_slow.prof` and `profile_fast.prof` to the current directory; these are binary profile dumps intended for further inspection with tools like `snakeviz`, not meant to be read directly.

#### Script

```python
#!/usr/bin/env python3
"""
performance_optimization.py

Demonstrates profiling a script with `cProfile` to FIND bottlenecks
before optimizing, then shows a concrete before/after optimization
for repeated O(n) membership checks against a list vs. a set.
"""

import cProfile
import io
import pstats
import time


DATA_SIZE = 20_000
LOOKUP_COUNT = 20_000


def slow_membership_checks():
    """O(n) scan per check against a LIST -> O(n*m) total."""
    allow_list = list(range(DATA_SIZE))
    found = 0
    for i in range(LOOKUP_COUNT):
        if i in allow_list:
            found += 1
    return found


def fast_membership_checks():
    """Same logical result, but using a set for O(1) average-case membership checks."""
    allow_set = set(range(DATA_SIZE))
    found = 0
    for i in range(LOOKUP_COUNT):
        if i in allow_set:
            found += 1
    return found


def profile_function(func, label: str, save_to: str = None) -> float:
    """Run cProfile on `func`, print a sorted report, and return elapsed wall time."""
    profiler = cProfile.Profile()

    start = time.perf_counter()
    profiler.enable()
    func()
    profiler.disable()
    elapsed = time.perf_counter() - start

    print(f"\n{'=' * 60}\nProfile: {label}  (wall time: {elapsed:.4f}s)\n{'=' * 60}")
    stream = io.StringIO()
    stats = pstats.Stats(profiler, stream=stream).sort_stats("cumulative")
    stats.print_stats(5)
    print(stream.getvalue())

    if save_to:
        profiler.dump_stats(save_to)

    return elapsed


def main():
    print("Performance Optimization Demo (cProfile)")

    slow_time = profile_function(slow_membership_checks, "UNOPTIMIZED (list membership checks)", save_to="profile_slow.prof")
    fast_time = profile_function(fast_membership_checks, "OPTIMIZED (set membership checks)", save_to="profile_fast.prof")

    speedup = slow_time / fast_time if fast_time > 0 else float("inf")
    print("=" * 60)
    print(f"Summary: optimized version was ~{speedup:.1f}x faster "
          f"({slow_time:.4f}s -> {fast_time:.4f}s) for the same {LOOKUP_COUNT:,} lookups.")
    print("Detailed profiles saved to profile_slow.prof / profile_fast.prof")
    print("(Tip: inspect them visually with: pip install snakeviz && snakeviz profile_slow.prof)")


if __name__ == "__main__":
    main()
```

Exact timings and the speedup multiplier will vary by CPU, but the *shape* of the result — a two-to-three-order-of-magnitude difference — is consistent and is the whole point of the demo.

#### Variations / flags

- Change `DATA_SIZE` and `LOOKUP_COUNT` to see how the gap between list and set membership checks scales — larger `DATA_SIZE` widens the gap further, since list lookup is O(n) but set lookup stays O(1).
- Use `stats.sort_stats("tottime")` instead of `"cumulative"` to rank by time spent *in* each function specifically, excluding time in functions it calls — useful when profiling something with a deeper call chain than this single-function demo.
- Increase `stats.print_stats(5)` to show more or fewer rows of the profile report.
- For scripts with many functions, profile the whole `main()` instead of individual functions to get a full call-graph view, then drill into the worst offender.

#### Common pitfalls

- **Optimizing before profiling** — the exact anti-pattern this script demonstrates avoiding; without data, it's easy to "optimize" a function that wasn't actually the bottleneck while leaving the real slow path untouched.
- **List membership checks (`x in some_list`) in a hot loop** — one of the most common accidental performance bugs in automation scripts (e.g. checking hostnames against an allow-list while processing every line of a large log file); converting the list to a `set` (or `frozenset` if it's static) is usually a one-line fix with a massive payoff.
- **Profiling overhead skewing results for very fast code** — `cProfile` adds per-call overhead, which can make trivially fast functions look artificially slower relative to their true cost; for micro-benchmarking very small operations, use the `timeit` module instead of `cProfile`.
- **Forgetting profiling is per-process, not per-thread by default** — `cProfile` in its default deterministic mode profiles the main thread; if your automation spawns worker threads, you'll need `threading`-aware profiling or `yappi` to get accurate multi-threaded profiles.

### 21.12 Documentation Generation

#### Use case

A sysadmin has written a handful of internal automation modules with proper docstrings (parameters, return values, behavior) but no one on the team reads the raw source to find out how to use them. The script scaffolds a minimal Sphinx project pointed at a sample module, and — if Sphinx is installed — builds real HTML documentation automatically, so the docs stay derived directly from the code instead of a separately maintained wiki page that goes stale.

#### Prerequisites

- Python 3.6+
- Standard library only for scaffolding: `argparse`, `os`, `shutil`, `subprocess`
- Required to actually **build** the HTML docs: `sphinx`

```bash
python3 --version   # confirm 3.6+
pip install sphinx
```

- OS scope: Linux, macOS, Windows (Sphinx and `sphinx-build` work identically on all three; the script locates `sphinx-build` via `shutil.which`, which respects the OS's `PATH` mechanism)

#### Cautions

- If Sphinx isn't installed, the script still scaffolds the full project structure and prints the exact commands to build it manually later — this is a graceful degradation, not a failure.
- The generated `conf.py` inserts `sys.path.insert(0, os.path.abspath("../.."))` so Sphinx's `autodoc` extension can import `sample_module.py`; if you point this scaffold at your own real module instead, make sure the equivalent path actually resolves to where your module lives.

#### Script

```python
#!/usr/bin/env python3
"""
documentation_generation.py

Demonstrates auto-generating HTML documentation from Python docstrings
using Sphinx: scaffolds a minimal Sphinx project, points it at a
sample module full of docstrings, and (if Sphinx is installed) builds
the final HTML docs automatically.
"""

import argparse
import os
import shutil
import subprocess
import sys


SAMPLE_MODULE = '''"""
sample_module

A tiny example module with real docstrings, used to demonstrate
Sphinx's ability to auto-generate documentation from docstrings
(via the autodoc extension) instead of maintaining separate docs
by hand.
"""


def add(a: int, b: int) -> int:
    """
    Add two integers together.

    :param a: The first number.
    :param b: The second number.
    :return: The sum of ``a`` and ``b``.
    """
    return a + b


class Greeter:
    """A simple greeter class, used to demonstrate class + method docs."""

    def __init__(self, name: str):
        """
        :param name: The name to greet.
        """
        self.name = name

    def greet(self) -> str:
        """Return a friendly greeting string for :attr:`name`."""
        return f"Hello, {self.name}!"
'''

CONF_PY_TEMPLATE = '''# Minimal Sphinx configuration, generated by documentation_generation.py
import os
import sys
sys.path.insert(0, os.path.abspath("../.."))

project = "{project_name}"
extensions = ["sphinx.ext.autodoc", "sphinx.ext.napoleon"]
html_theme = "alabaster"
'''

INDEX_RST_TEMPLATE = '''{title}
{underline}

.. automodule:: sample_module
   :members:
   :undoc-members:
   :show-inheritance:
'''


def scaffold_sphinx_project(outdir: str, project_name: str) -> str:
    source_dir = os.path.join(outdir, "source")
    os.makedirs(source_dir, exist_ok=True)

    with open(os.path.join(outdir, "sample_module.py"), "w", encoding="utf-8") as f:
        f.write(SAMPLE_MODULE)

    with open(os.path.join(source_dir, "conf.py"), "w", encoding="utf-8") as f:
        f.write(CONF_PY_TEMPLATE.format(project_name=project_name))

    title = f"{project_name} Documentation"
    with open(os.path.join(source_dir, "index.rst"), "w", encoding="utf-8") as f:
        f.write(INDEX_RST_TEMPLATE.format(title=title, underline="=" * len(title)))

    return source_dir


def build_docs(outdir: str, source_dir: str) -> bool:
    """Run sphinx-build to generate HTML docs. Returns True on success."""
    build_dir = os.path.join(outdir, "build", "html")
    sphinx_build = shutil.which("sphinx-build")
    if not sphinx_build:
        return False

    try:
        subprocess.run(
            [sphinx_build, "-b", "html", source_dir, build_dir],
            check=True,
            capture_output=True,
            text=True,
            cwd=outdir,
        )
        return True
    except subprocess.CalledProcessError as e:
        print("sphinx-build failed:")
        print(e.stdout)
        print(e.stderr)
        return False


def main():
    parser = argparse.ArgumentParser(description="Scaffold and build Sphinx documentation from docstrings.")
    parser.add_argument("--outdir", default="docs_demo", help="Output folder (default: docs_demo)")
    parser.add_argument("--project-name", default="SampleAutomationTool", help="Sphinx project name")
    args = parser.parse_args()

    print("Documentation Generation Demo (Sphinx)")
    print("=" * 45)

    source_dir = scaffold_sphinx_project(args.outdir, args.project_name)
    print(f"Scaffolded Sphinx project at: {os.path.abspath(args.outdir)}")
    print(f"  - sample module with docstrings: {args.outdir}/sample_module.py")
    print(f"  - Sphinx config:                 {source_dir}/conf.py")
    print(f"  - Sphinx index:                  {source_dir}/index.rst")

    built = build_docs(args.outdir, source_dir)

    if built:
        html_index = os.path.abspath(os.path.join(args.outdir, "build", "html", "index.html"))
        print(f"\nHTML documentation built successfully:")
        print(f"  {html_index}")
        print("Open that file in a browser to see the docs generated from the docstrings above.")
    else:
        print("\nSphinx is not installed (or sphinx-build was not found on PATH), so the HTML build was skipped.")
        print("Install it and build manually with:")
        print("  pip install sphinx")
        print(f"  sphinx-build -b html {source_dir} {os.path.join(args.outdir, 'build', 'html')}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run

`scaffold_sphinx_project()` writes a sample module plus a minimal `conf.py`/`index.rst` pair using Sphinx's `automodule` directive; `build_docs()` then shells out to `sphinx-build` if it's found on `PATH`.

```bash
# Default output folder (docs_demo/)
python3 documentation_generation.py

# Custom output folder and project name
python3 documentation_generation.py --outdir mydocs --project-name "BackupToolDocs"

# If Sphinx wasn't installed at scaffold time, build manually afterward:
pip install sphinx
sphinx-build -b html docs_demo/source docs_demo/build/html
```

#### Variations / flags

- `--outdir` — choose where the Sphinx project and generated docs live; useful for keeping docs alongside a real project instead of the demo default.
- `--project-name` — sets the title shown in the generated documentation's index page.
- Point `conf.py`'s `sys.path.insert` and `index.rst`'s `.. automodule::` directive at your **own** real module (instead of the bundled `sample_module`) to document actual production code — the scaffold is a starting template, not a fixed target.
- Add `sphinx.ext.viewcode` to the `extensions` list in `conf.py` to include clickable source-code links alongside the rendered docs.
- Swap `html_theme = "alabaster"` for `"sphinx_rtd_theme"` (`pip install sphinx-rtd-theme`) for a more polished, ReadTheDocs-style look.

#### Common pitfalls

- **Docstrings that don't follow a Sphinx-compatible format** — `autodoc` renders whatever docstring text exists, but structured fields like `:param:` and `:return:` (as used in `sample_module.py`) render far more usefully than a single unstructured paragraph; consider adopting Google-style or NumPy-style docstrings with the `napoleon` extension (already enabled in the generated `conf.py`) if your team prefers that format.
- **`sys.path` not pointing at the real module** — the single most common reason `automodule` fails with "no module named X" is that Sphinx's `conf.py` can't import the target module; verify the relative path math from `source/conf.py` to wherever your actual code lives.
- **Stale generated docs** — HTML output in `build/html` is a point-in-time snapshot; wire `sphinx-build` into a CI job or pre-commit hook if you want documentation that's guaranteed to reflect the latest docstrings rather than a specific point in the past.
- **`sphinx-build` not on `PATH` despite `pip install sphinx`** — common in virtual environments that aren't activated, or when Sphinx was installed for a different Python interpreter than the one running this script; verify with `which sphinx-build` (or `where sphinx-build` on Windows) before assuming installation failed.
