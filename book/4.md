## Chapter 4: Configuration Management

Every application, service, and cron job a sysadmin touches is driven by
configuration somewhere — a JSON blob, an INI file inherited from a
Windows-era service, a YAML manifest for a container, a `.env` file
nobody remembers writing. This chapter is a working toolkit for the
day-to-day mechanics of handling that configuration safely: reading it,
parsing it in whatever format it happens to be in, checking that the
values actually make sense, combining several files into one, generating
new files from scratch, editing a single value without hand-rolling a
text editor session over SSH, and — critically — backing it up and
restoring it when a change goes wrong.

None of these fourteen scripts are framework code. Each one is a small,
single-purpose command-line tool built entirely on Python's standard
library, with the exception of the YAML and TOML parsers, which lean on
the well-known third-party packages `PyYAML` and `tomli`. Every script
can be run with zero arguments to execute a self-contained demo, so you
can see exactly what it does before pointing it at a real file. Every
script has also been executed end-to-end while writing this chapter to
confirm its documented output is accurate, not assumed.

**Safety posture:** Read-only tools (4.1–4.8) never touch the files they inspect. Tools
that write (4.9–4.14) generally **overwrite their target file without
a confirmation prompt** — that's a deliberate design choice to keep
the scripts scriptable and cron-friendly, but it means the burden of
safety falls on you. Run `backup_config.py` (4.11) before you run
`modify_config_file.py` (4.10), `merge_config_files.py` (4.8) with
`--output`, or `generate_config_file.py` (4.9) against a file you
care about.

**Dependencies:** Twelve of the fourteen scripts use only the Python standard library
and run on any Python 3.8+ install with no `pip install` at all. The
two exceptions are the YAML parser/generator (4.4, 4.9's `--format
yaml`) and the TOML parser (4.5 on Python 3.10 and earlier), which
need `PyYAML` and `tomli` respectively. Each of those sections
includes its own install instructions and a `requirements.txt`.

### 4.1 Read Configuration Files

#### Use case

Before parsing a config file in a specific format, you often just need
to *look at it* — check it exists, see its raw contents with line
numbers, and confirm you're pointed at the right file. A sysadmin
reaches for this when troubleshooting "why isn't my service picking up
this config" and needs a fast, format-agnostic dump of a file's exact
contents straight from the shell.

#### Prerequisites

Standard library only — no installation required. Runs on any Python
3.6+ on Linux, macOS, or Windows.

```bash
python3 --version   # 3.6 or newer is sufficient
```

#### Cautions

Large files are read fully into memory in one call to `readlines()`;
this is fine for typical config files but not appropriate for
multi-gigabyte logs. Non-UTF-8 bytes are replaced rather than raising
an error, so unusual encodings may display as `�` characters instead
of failing loudly.

#### Script

```python
#!/usr/bin/env python3
"""
read_config_file.py

Purpose:
    Read a plain configuration/text file from disk and print its raw
    contents with line numbers. This is a generic, format-agnostic
    starting point before you parse a specific format (INI/JSON/YAML/
    TOML/XML) with the other scripts in this folder.

Usage:
    python3 read_config_file.py <path_to_file>

    If no path is given, this script creates and reads a small demo
    config file (demo_config.txt) in the current directory.
"""

import os
import sys

DEMO_FILE = "demo_config.txt"
DEMO_CONTENT = (
    "# Demo configuration file\n"
    "debug = true\n"
    "max_connections = 100\n"
)


def read_config(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    with open(path, "r", encoding="utf-8", errors="replace") as f:
        lines = f.readlines()

    print(f"Reading file: {path}")
    print("-" * 30)
    for i, line in enumerate(lines, start=1):
        print(f"{i:3}: {line.rstrip()}")
    print("-" * 30)
    print(f"Total lines: {len(lines)}")


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and reading a demo config file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE
    read_config(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

The script checks whether the target file exists with
`os.path.isfile`, then reads it in one shot with `errors="replace"` so
a bad encoding never crashes the script. Each line is printed with a
right-aligned line number, matching what you'd see in an editor's
gutter, followed by a total line count.

```bash
# Run with no arguments — creates and reads a demo file
python3 read_config_file.py

# Read a specific file
python3 read_config_file.py /etc/myapp/config.ini

# Read a file with a space in the path (quote it)
python3 read_config_file.py "/opt/my app/settings.conf"
```

#### Sample output

```text
$ python3 read_config_file.py
No file path given, creating and reading a demo config file.

Reading file: demo_config.txt
------------------------------
  1: # Demo configuration file
  2: debug = true
  3: max_connections = 100
------------------------------
Total lines: 3
```

This was captured from a real run of the uploaded script — confirmed
correct.

#### Variations / flags

This script takes a single positional path and has no optional flags.
Reasonable extensions if you need more:

- Add a `--tail N` flag to print only the last `N` lines, useful for
  log-like config files.
- Pipe the output through `grep` for a quick filtered view:
  `python3 read_config_file.py app.conf | grep -i port`.
- Wrap it in a `watch` command to monitor a config file for changes:
  `watch -n 2 python3 read_config_file.py app.conf`.

#### Common pitfalls

- **Permissions**: if the file is owned by another user (e.g. a
  service account) you may need `sudo` to read it, or the script will
  print a Python `PermissionError` traceback rather than the friendly
  "file not found" message, since that check only covers existence,
  not readability.
- **Relative paths**: running the script from a different working
  directory than you expect silently reads (or creates) the demo file
  in the wrong place — use absolute paths when scripting this.
- **Windows path quoting**: backslash paths like
  `C:\config\app.ini` should be passed with forward slashes or a raw
  string in code that calls this script, though on the command line
  passing them directly usually works fine.

### 4.2 Parse INI Files

#### Use case

INI files are still everywhere in legacy Linux daemons, Windows
services, and tools like `pip.ini`, `.gitconfig`, and `my.cnf`. A
sysadmin reaches for this when they need to quickly confirm what
sections and key/value pairs a service is actually going to read at
startup, without opening the file in an editor and manually parsing
`[section]` headers by eye.

#### Prerequisites

Standard library only (`configparser`) — no installation required.
Runs on Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

`configparser` treats every value as a string — `port = 8080` parses
as the string `"8080"`, not an integer. If you need typed values, pair
this with `validate_config_values.py` (section 4.7). Duplicate section
names or malformed syntax raise a `configparser.Error`, which this
script catches and reports cleanly rather than crashing.

#### Script

```python
#!/usr/bin/env python3
"""
parse_ini_config.py

Purpose:
    Parse an INI-format configuration file (sections with key=value
    pairs) using Python's built-in configparser module, and print its
    structure.

Usage:
    python3 parse_ini_config.py <path_to_file.ini>

    If no path is given, this script creates and parses a demo
    demo_config.ini file in the current directory.
"""

import configparser
import os
import sys

DEMO_FILE = "demo_config.ini"
DEMO_CONTENT = (
    "[server]\n"
    "host = 127.0.0.1\n"
    "port = 8080\n"
    "\n"
    "[database]\n"
    "name = mydb\n"
    "user = admin\n"
)


def parse_ini(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    parser = configparser.ConfigParser()
    try:
        parser.read(path, encoding="utf-8")
    except configparser.Error as e:
        print(f"Error parsing INI file: {e}")
        return

    print(f"Sections found: {parser.sections()}\n")
    for section in parser.sections():
        print(f"[{section}]")
        for key, value in parser.items(section):
            print(f"  {key} = {value}")
        print()


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and parsing a demo INI file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE
    parse_ini(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`configparser.ConfigParser().read()` loads the file and splits it into
sections; `parser.sections()` lists them, and `parser.items(section)`
iterates each key/value pair inside a section. Parse errors are caught
with a targeted `except configparser.Error` so a malformed file
reports a clear message instead of a traceback.

```bash
# Run with no arguments — creates and parses a demo INI file
python3 parse_ini_config.py

# Parse a specific INI file
python3 parse_ini_config.py /etc/myapp/settings.ini

# Parse a Windows-style config
python3 parse_ini_config.py "C:\Program Files\App\config.ini"
```

#### Sample output

```text
$ python3 parse_ini_config.py
No file path given, creating and parsing a demo INI file.

Sections found: ['server', 'database']

[server]
  host = 127.0.0.1
  port = 8080

[database]
  name = mydb
  user = admin
```

Verified against a real run of the uploaded script — output matches
exactly.

#### Variations / flags

No built-in flags beyond the positional path. Useful extensions:

- Add a `--section NAME` flag to print only one section, handy for
  large multi-service INI files.
- Combine with `validate_config_values.py` to add type checking, since
  `configparser` alone won't tell you `port` should be an integer.
- Use `parser.get(section, key, fallback=...)` in your own scripts
  built on this pattern to safely handle missing keys.

#### Common pitfalls

- **Interpolation surprises**: `configparser` by default interprets
  `%` characters specially (for `%(name)s` interpolation); a raw `%`
  in a value (e.g. a password containing `%`) will raise an error
  unless you configure `interpolation=None`.
- **Case sensitivity**: section names are case-sensitive by default,
  but key names are lowercased automatically — `Port` and `port` in
  the same section will collide.
- **Missing `DEFAULT` section handling**: values in a `[DEFAULT]`
  section apply to every other section but won't show up when you
  iterate `parser.sections()` directly, which can be confusing when
  auditing output.

### 4.3 Parse JSON Configuration

#### Use case

JSON is the default config format for most modern web services, Node
tooling, and REST-based systems. A sysadmin reaches for this when a
service fails to start with a vague "config error" and they need to
confirm the JSON is syntactically valid and see its structure
pretty-printed, without hunting for a missing comma by eye.

#### Prerequisites

Standard library only (`json`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

This script only reads and reports; it never modifies the file. JSON
does not support comments or trailing commas — a config file
containing either (common if someone hand-edited it or copied from a
JSONC example) will fail to parse, though this script reports that
clearly rather than crashing with a raw traceback. Very large JSON
files are loaded entirely into memory.

#### Script

```python
#!/usr/bin/env python3
"""
parse_json_config.py

Purpose:
    Parse a JSON configuration file using Python's built-in json module
    and pretty-print its structure.

Usage:
    python3 parse_json_config.py <path_to_file.json>

    If no path is given, this script creates and parses a demo
    demo_config.json file in the current directory.
"""

import json
import os
import sys

DEMO_FILE = "demo_config.json"
DEMO_CONTENT = {
    "server": {
        "host": "127.0.0.1",
        "port": 8080
    },
    "debug": True
}


def parse_json(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    try:
        with open(path, "r", encoding="utf-8") as f:
            data = json.load(f)
    except json.JSONDecodeError as e:
        print(f"Error parsing JSON file: {e}")
        return

    print("Parsed configuration:")
    print(json.dumps(data, indent=2))
    if isinstance(data, dict):
        print(f"\nTop-level keys: {list(data.keys())}")


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and parsing a demo JSON file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                json.dump(DEMO_CONTENT, f, indent=2)
        path = DEMO_FILE
    parse_json(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`json.load()` parses the file directly into native Python types
(`dict`, `list`, `str`, `int`, `bool`, `None`). Any syntax problem
raises `json.JSONDecodeError`, which is caught and reported with the
exact line/column the standard library reports. The parsed structure
is re-serialized with `json.dumps(..., indent=2)` for a clean,
consistently indented view regardless of how the original file was
formatted.

```bash
# Run with no arguments — creates and parses a demo JSON file
python3 parse_json_config.py

# Parse a specific JSON config
python3 parse_json_config.py /etc/myapp/config.json

# Quickly validate JSON syntax in a CI step
python3 parse_json_config.py config.json || exit 1
```

#### Sample output

```text
$ python3 parse_json_config.py
No file path given, creating and parsing a demo JSON file.

Parsed configuration:
{
  "server": {
    "host": "127.0.0.1",
    "port": 8080
  },
  "debug": true
}

Top-level keys: ['server', 'debug']
```

Captured from an actual run of the uploaded script — matches exactly.

#### Variations / flags

No built-in flags beyond the positional path. Useful extensions:

- Add a `--compact` flag that calls `json.dumps(data, separators=(",", ":"))`
  for a single-line, whitespace-free view.
- Pipe through `jq` for further filtering once you've confirmed the
  file parses: `python3 parse_json_config.py config.json | jq .server`.
- Wrap this pattern in a pre-commit hook to reject malformed JSON
  config files before they're committed.

#### Common pitfalls

- **Trailing commas**: `{"debug": true,}` is invalid JSON and will
  raise a decode error, even though many editors' auto-formatting
  makes this an easy typo to introduce.
- **`NaN`/`Infinity`**: Python's `json` module accepts these as
  extensions by default even though they aren't valid JSON per the
  spec, which can mask a portability problem if the file needs to be
  read by a strict parser elsewhere.
- **Encoding assumptions**: the script opens files as UTF-8; a JSON
  file saved with a BOM (common from some Windows editors) can cause a
  decode error that looks like a syntax problem at position 0.

### 4.4 Parse YAML Configuration

#### Use case

YAML is the dominant format for Kubernetes manifests, Docker Compose,
Ansible playbooks, and CI pipelines (GitHub Actions, GitLab CI). A
sysadmin reaches for this when a deploy fails because of a YAML
indentation error and they need a fast way to confirm what the parser
actually sees, since a single misplaced space can silently change the
structure rather than raising an obvious error.

#### Prerequisites

Requires the third-party `PyYAML` library. Install it with pip before
running:

```bash
pip install pyyaml --break-system-packages
# or, using the provided requirements file:
pip install -r requirements.txt --break-system-packages
```

Runs on Linux, macOS, and Windows on any Python 3.6+; the `--break-system-packages`
flag is only needed on systems (like Debian/Ubuntu 23.04+) with
externally-managed Python environments — a plain `pip install pyyaml`
is enough elsewhere, and using a virtual environment avoids the flag
entirely.

#### Cautions

This script deliberately uses `yaml.safe_load()`, not `yaml.load()`.
`safe_load()` only constructs plain Python types (dicts, lists,
strings, numbers, booleans) and cannot execute arbitrary Python
objects embedded in the file — do not change this to `yaml.load()` on
files you don't fully trust, since that API can be used to execute
arbitrary code via crafted YAML tags.

#### Script

```python
#!/usr/bin/env python3
"""
parse_yaml_config.py

Purpose:
    Parse a YAML configuration file using the third-party PyYAML
    library and pretty-print its structure.

Usage:
    python3 parse_yaml_config.py <path_to_file.yaml>

    If no path is given, this script creates and parses a demo
    demo_config.yaml file in the current directory.
"""

import os
import sys

try:
    import yaml
except ImportError:
    print("Error: this script requires 'PyYAML'. Install with:")
    print("    pip install -r requirements.txt")
    sys.exit(1)

DEMO_FILE = "demo_config.yaml"
DEMO_CONTENT = (
    "server:\n"
    "  host: 127.0.0.1\n"
    "  port: 8080\n"
    "debug: true\n"
)


def parse_yaml(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    try:
        with open(path, "r", encoding="utf-8") as f:
            data = yaml.safe_load(f)
    except yaml.YAMLError as e:
        print(f"Error parsing YAML file: {e}")
        return

    print("Parsed configuration:")
    print(data)
    if isinstance(data, dict):
        print(f"\nTop-level keys: {list(data.keys())}")


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and parsing a demo YAML file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE
    parse_yaml(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

The script attempts `import yaml` first and exits with a clear install
instruction if `PyYAML` isn't present, rather than a raw
`ModuleNotFoundError`. `yaml.safe_load()` parses the file into native
Python types; parse errors surface as `yaml.YAMLError`, which is
caught and reported. The result is printed with Python's default
`repr`-style formatting (not re-serialized as YAML), which is
intentional — it shows you exactly what data types Python parsed
your file into.

```bash
# Run with no arguments — creates and parses a demo YAML file
python3 parse_yaml_config.py

# Parse a specific YAML file
python3 parse_yaml_config.py docker-compose.yml

# Validate a Kubernetes manifest parses cleanly before applying it
python3 parse_yaml_config.py deployment.yaml
```

#### Sample output

```text
$ python3 parse_yaml_config.py
No file path given, creating and parsing a demo YAML file.

Parsed configuration:
{'server': {'host': '127.0.0.1', 'port': 8080}, 'debug': True}

Top-level keys: ['server', 'debug']
```

Verified with `PyYAML` installed and a real run of the uploaded
script — output confirmed correct.

#### Variations / flags

No built-in flags beyond the positional path. Useful extensions:

- Add a `--pretty` flag that re-serializes with `yaml.dump(data,
  default_flow_style=False)` instead of Python's `repr` formatting, for
  output that looks like the original file.
- Chain with `parse_json_config.py`-style output by converting the
  parsed dict with `json.dumps(data, indent=2)`, useful when piping
  into JSON-aware tools like `jq`.
- Use `yaml.safe_load_all()` instead of `safe_load()` to support
  multi-document YAML files (separated by `---`), common in
  Kubernetes manifests that bundle several resources in one file.

#### Common pitfalls

- **Tabs vs. spaces**: YAML forbids tab characters for indentation;
  a file edited with a tab-inserting editor will fail to parse with an
  error that isn't always obvious about the real cause.
- **Implicit typing surprises**: YAML auto-converts unquoted values
  like `yes`, `no`, `on`, `off`, and `null` into booleans/`None` — a
  hostname value of `no` (unlikely, but a version string like `1.20`
  is a real example) can silently become the wrong type unless quoted.
- **Missing PyYAML in production**: since this is a third-party
  dependency, forgetting to include it in a container image or
  virtualenv requirements file is the most common real-world failure
  mode — the script's explicit `ImportError` handling exists precisely
  to catch this early.

### 4.5 Parse TOML Configuration

#### Use case

TOML has become the standard for Python packaging (`pyproject.toml`)
and is also common in Rust (`Cargo.toml`) and Hugo site configs. A
sysadmin reaches for this when auditing a `pyproject.toml` or similar
build/tooling config, needing a quick structural dump without manually
tracing through nested `[tool.x.y]` table headers.

#### Prerequisites

On Python 3.11 and newer, this script uses the built-in `tomllib`
module — no installation needed. On Python 3.10 and earlier, it falls
back to the third-party `tomli` library:

```bash
python3 --version                 # check your version first
pip install tomli --break-system-packages     # only needed on Python < 3.11
# or:
pip install -r requirements.txt --break-system-packages
```

#### Cautions

This script only reads and parses; it never writes to the file.
`tomllib`/`tomli` require the file to be opened in binary mode — the
script handles this for you, so if you adapt this code elsewhere,
remember TOML parsing does not accept a text-mode file handle.

#### Script

```python
#!/usr/bin/env python3
"""
parse_toml_config.py

Purpose:
    Parse a TOML configuration file (e.g. pyproject.toml-style files)
    and pretty-print its structure.

Usage:
    python3 parse_toml_config.py <path_to_file.toml>

    If no path is given, this script creates and parses a demo
    demo_config.toml file in the current directory.
"""

import json
import os
import sys

try:
    import tomllib as toml_reader  # Python 3.11+
    _BINARY_MODE = True
except ImportError:
    try:
        import tomli as toml_reader  # Python <=3.10 fallback
        _BINARY_MODE = True
    except ImportError:
        print("Error: this script requires 'tomli' on Python <3.11. Install with:")
        print("    pip install -r requirements.txt")
        sys.exit(1)

DEMO_FILE = "demo_config.toml"
DEMO_CONTENT = (
    "debug = true\n\n"
    "[server]\n"
    "host = \"127.0.0.1\"\n"
    "port = 8080\n"
)


def parse_toml(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    try:
        with open(path, "rb") as f:
            data = toml_reader.load(f)
    except Exception as e:  # tomllib/tomli raise their own decode errors
        print(f"Error parsing TOML file: {e}")
        return

    print("Parsed configuration:")
    print(json.dumps(data, indent=2, default=str))
    if isinstance(data, dict):
        print(f"\nTop-level keys: {list(data.keys())}")


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and parsing a demo TOML file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE
    parse_toml(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

The script first tries `import tomllib` (standard library since Python
3.11), and falls back to the third-party `tomli` package on older
Python versions — both share the same `load()` API, so the rest of
the script doesn't need to know which one is active. The file is
opened in binary mode as required, parsed into a dict, and printed via
`json.dumps(..., default=str)` so TOML-specific types like dates
render as readable strings instead of raising a JSON serialization
error.

```bash
# Run with no arguments — creates and parses a demo TOML file
python3 parse_toml_config.py

# Parse a real pyproject.toml
python3 parse_toml_config.py pyproject.toml

# Parse a Cargo.toml from a Rust project
python3 parse_toml_config.py Cargo.toml
```

  "debug": true,

#### Variations / flags

No built-in flags beyond the positional path. Useful extensions:

- Add a `--section server` flag to print only a single table using

- Extract just the dependency list from a `pyproject.toml`:

- Use this as a pre-flight check in a `Makefile` target before running
  a build tool that depends on strict TOML syntax.

#### Common pitfalls

- **Python version drift**: a script written and tested against
  Python 3.11+'s `tomllib` will fail with an unhelpful `ModuleNotFoundError`
  on Python 3.10 unless `tomli` is installed — this script guards
  against that explicitly, so keep that fallback if you adapt the code.
- **Binary mode is mandatory**: unlike JSON or YAML, calling
  `open(path, "r")` and passing a text handle to `tomllib.load()` will
  raise a `TypeError`; always use `"rb"`.
- **Dates and times**: TOML natively supports date/time literals which
  Python parses into `datetime.date`/`datetime.datetime` objects —
  these aren't JSON-serializable by default, which is why this script
  passes `default=str` to `json.dumps()`.


### 4.6 Parse XML Configuration

#### Use case

XML configuration is still common in Java-based enterprise software
(Tomcat's `server.xml`, Maven's `pom.xml`), Windows app config
(`app.config`, `web.config`), and older network appliances. A
sysadmin reaches for this when they need to walk an XML config's
element tree quickly to confirm nesting and values, without manually
tracing angle brackets in a wall of text.

#### Prerequisites

Standard library only (`xml.etree.ElementTree`) — no installation
required. Runs on Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

This script only reads and parses; it makes no changes to the file.
`xml.etree.ElementTree`, used here with its default settings, is safe
against the most dangerous XML attacks such as external entity
expansion ("XXE") — but that safety only holds if you stick to the
standard library defaults as this script does. Avoid swapping in a
third-party XML parser with entity resolution enabled when working
with untrusted files. Deeply nested or attribute-heavy XML will need
custom traversal logic beyond this simple demo printer, which does not
print element attributes at all, only tag names and text content.

#### Script

```python
#!/usr/bin/env python3
"""
parse_xml_config.py

Purpose:
    Parse an XML configuration file using Python's built-in
    xml.etree.ElementTree module and print its tree structure.

Usage:
    python3 parse_xml_config.py <path_to_file.xml>

    If no path is given, this script creates and parses a demo
    demo_config.xml file in the current directory.
"""

import os
import sys
import xml.etree.ElementTree as ET

DEMO_FILE = "demo_config.xml"
DEMO_CONTENT = (
    "<configuration>\n"
    "  <server>\n"
    "    <host>127.0.0.1</host>\n"
    "    <port>8080</port>\n"
    "  </server>\n"
    "  <debug>true</debug>\n"
    "</configuration>\n"
)


def print_element(element, indent=0):
    label = element.tag
    if len(element) == 0:
        text = (element.text or "").strip()
        print("  " * indent + f"{label}: {text}")
    else:
        print("  " * indent + label)
        for child in element:
            print_element(child, indent + 1)


def parse_xml(path: str) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    try:
        tree = ET.parse(path)
    except ET.ParseError as e:
        print(f"Error parsing XML file: {e}")
        return

    root = tree.getroot()
    print(f"Root tag: {root.tag}")
    print_element(root)


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, creating and parsing a demo XML file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE
    parse_xml(path)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`ET.parse()` builds an in-memory element tree; `tree.getroot()` gives
the top-level element. `print_element()` recurses through the tree: if
an element has no children, it's treated as a leaf and its stripped
text is printed on one line (`tag: value`); if it has children, the
tag name is printed alone and the function recurses one indent level
deeper for each child.

```bash
# Run with no arguments — creates and parses a demo XML file
python3 parse_xml_config.py

# Parse a Tomcat server.xml
python3 parse_xml_config.py /opt/tomcat/conf/server.xml

# Parse a Maven pom.xml to sanity-check its structure
python3 parse_xml_config.py pom.xml
```

#### Sample output

```text
$ python3 parse_xml_config.py
No file path given, creating and parsing a demo XML file.

Root tag: configuration
configuration
  server
    host: 127.0.0.1
    port: 8080
  debug: true
```

Captured from a real run of the uploaded script — confirmed correct.

#### Variations / flags

No built-in flags beyond the positional path. Useful extensions:

- Extend `print_element` to also print each element's `.attrib`
  dictionary, since the current version deliberately ignores XML
  attributes (e.g. `<server env="prod">`) and only shows tag/text.
- Add an `--xpath EXPR` flag using `root.findall(EXPR)` to jump
  straight to a specific nested element instead of printing the whole
  tree.
- Combine with `validate_config_values.py`-style rules by converting
  the parsed tree into a dict first (there's no built-in XML-to-dict
  conversion in the standard library, so this needs custom code).

#### Common pitfalls

- **Attributes are invisible in this output**: values stored as XML
  attributes (`<host name="db1"/>`) rather than element text won't
  show up at all with this script's current traversal — check the raw
  file with `read_config_file.py` (4.1) if output looks incomplete.
- **Namespaces**: XML files using namespaces (`xmlns="..."`) will show
  tag names like `{http://example.com/ns}server` instead of plain
  `server`, which can look confusing until you know to expect it.
- **Malformed XML from hand-edits**: a single unclosed tag anywhere in
  a large file causes the entire parse to fail — `ET.ParseError`
  reports a line and column number, which is the fastest way to
  locate the break.

### 4.7 Validate Configuration Values

#### Use case

A config file can be perfectly valid JSON and still be wrong — a
missing `timeout` key, a `port` stored as the string `"8080a"` instead
of an integer, a value out of an acceptable range. A sysadmin reaches
for this as a pre-deploy gate: run it in a CI pipeline or a pre-start
hook to catch bad configuration before a service starts with it and
fails in a more confusing way at runtime.

#### Prerequisites

Standard library only (`json`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

This is a small, illustrative validator, not a general schema
framework — the `RULES` dictionary inside the script is hard-coded to
one example schema (`host`, `port`, `debug`, `timeout`) and needs to
be edited to match your own configuration's required keys, types, and
ranges before it's useful against a real config file. For complex,
reusable schemas, prefer a dedicated library such as `jsonschema` or
`pydantic` instead of hand-editing this script. It only reads and
reports; it never modifies the file, even when issues are found.

#### Script

```python
#!/usr/bin/env python3
"""
validate_config_values.py

Purpose:
    Validate a JSON configuration file's values against a simple set of
    rules: required keys, expected types, and (optionally) allowed
    ranges — reporting every problem found rather than stopping at the
    first one.

Usage:
    python3 validate_config_values.py <path_to_file.json>

    If no path is given, this script validates a demo config (which
    intentionally includes one bad value, to show validation catching
    a real problem).
"""

import json
import os
import sys

DEMO_FILE = "demo_config_to_validate.json"
# Demo file intentionally has a bad "port" value and is missing "timeout"
DEMO_CONTENT = {
    "host": "127.0.0.1",
    "port": "8080a",
    "debug": True
}

# Define your validation rules here: key -> (expected_type, required, min, max)
RULES = {
    "host": {"type": str, "required": True},
    "port": {"type": int, "required": True, "min": 1, "max": 65535},
    "debug": {"type": bool, "required": True},
    "timeout": {"type": int, "required": True, "min": 1, "max": 300},
}


def validate(data: dict, rules: dict) -> list:
    issues = []
    for key, rule in rules.items():
        if key not in data:
            if rule.get("required"):
                issues.append(f"'{key}' is missing a required key.")
            continue

        value = data[key]
        expected_type = rule["type"]
        if not isinstance(value, expected_type):
            issues.append(
                f"'{key}' should be of type {expected_type.__name__}, "
                f"got {type(value).__name__} ({value!r})"
            )
            continue

        if "min" in rule and value < rule["min"]:
            issues.append(f"'{key}' value {value} is below minimum {rule['min']}.")
        if "max" in rule and value > rule["max"]:
            issues.append(f"'{key}' value {value} is above maximum {rule['max']}.")
    return issues


def main():
    if len(sys.argv) > 1:
        path = sys.argv[1]
    else:
        print("No file path given, validating a demo config (with an intentional error).\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                json.dump(DEMO_CONTENT, f, indent=2)
        path = DEMO_FILE

    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    with open(path, "r", encoding="utf-8") as f:
        try:
            data = json.load(f)
        except json.JSONDecodeError as e:
            print(f"Error parsing JSON file: {e}")
            return

    print(f"Validating: {path}\n")
    issues = validate(data, RULES)

    if issues:
        for issue in issues:
            print(f"ISSUE: {issue}")
        print(f"\nValidation FAILED with {len(issues)} issue(s).")
    else:
        print("Validation PASSED. All required keys present with correct types.")


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`validate()` walks the `RULES` dictionary key by key (not the config
file's keys), so it inherently checks for *missing* required keys as
well as wrong types and out-of-range values on keys that *are*
present. Every problem found is appended to an `issues` list rather
than raising on the first one, so a single run reports the complete
picture of what's wrong with a file.

```bash
# Run with no arguments — validates a demo file with a known bad value
python3 validate_config_values.py

# Validate a real config file against the hard-coded RULES
python3 validate_config_values.py /etc/myapp/config.json

# Use as a CI gate — non-zero exit on failure requires a small edit,
# since the script currently always exits 0; add `sys.exit(1)` in the
# `if issues:` branch if you want that behavior for a pipeline.
python3 validate_config_values.py config.json
```

#### Sample output

```text
$ python3 validate_config_values.py
No file path given, validating a demo config (with an intentional error).

Validating: demo_config_to_validate.json

ISSUE: 'port' should be of type int, got str ('8080a')
ISSUE: 'timeout' is missing a required key.

Validation FAILED with 2 issue(s).
```

And against a config that satisfies every rule:

```text
$ python3 validate_config_values.py good.json

Validation PASSED. All required keys present with correct types.
```

against the built-in demo, one against a valid file
(`{"host": "127.0.0.1", "port": 443, "debug": true, "timeout": 30}`)
— confirmed correct.

#### Variations / flags

- Edit the `RULES` dictionary to match your own schema — this is the
  primary way to "configure" this script, since it has no
  command-line flags for defining rules.
- Add a `--strict` flag that also flags *unexpected* keys present in
  the file but absent from `RULES`, catching typos like `debgu`.
- Add `sys.exit(1)` when `issues` is non-empty so the script can gate
  a CI pipeline or a systemd `ExecStartPre=` check.

#### Common pitfalls

- **`bool` is a subtype of `int` in Python**: `isinstance(True, int)`
  returns `True`, which means a rule expecting `int` will silently
  accept a boolean value — order your `RULES` checks carefully if this
  matters for your schema (this script's default rules don't hit that
  edge case, but a custom schema checking a numeric field could).
- **Forgetting to update `RULES` for a new key**: the validator will
  happily ignore any key present in the config file but not listed in
  `RULES` — it validates the schema you defined, not the file's actual
  contents, so a leftover unused key won't be flagged.
- **JSON-only**: this script assumes the config file is JSON; point it
  at an INI or YAML file and it will fail at the `json.load()` step
  with a parse error, not a helpful "wrong format" message.

### 4.8 Merge Multiple Configuration Files

#### Use case

A common deployment pattern is a base config file plus one or more
environment-specific override files (`base.json` + `production.json`).
A sysadmin reaches for this when they need to combine those files into
a single effective configuration — for example to hand to a service
that only accepts one config file, or to preview exactly what values
will win before deploying.

#### Prerequisites

Standard library only (`json`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

The merge is **deep for nested dictionaries but shallow for lists** —
a list value in a later file completely replaces a list in an earlier
file rather than concatenating them. If you use `--output`, an
existing file at that path is overwritten without a prompt; run
`backup_config.py` (4.11) first if the destination matters. Files are
merged strictly in the order given on the command line, so always put
your base/default config first and environment-specific overrides
after it.

#### Script

```python
#!/usr/bin/env python3
"""
merge_config_files.py

Purpose:
    Merge two or more JSON configuration files into one, with later
    files overriding matching keys from earlier files (deep merge for
    nested dictionaries).

Usage:
    python3 merge_config_files.py base.json override.json > merged.json
    python3 merge_config_files.py base.json override.json --output merged.json

    If no files are given, this script merges two small built-in demo
    configs and prints the result (base + environment-specific override).
"""

import json
import os
import sys

DEMO_BASE = {
    "server": {"host": "127.0.0.1", "port": 8080},
    "debug": True
}
DEMO_OVERRIDE = {
    "server": {"port": 9090},
    "debug": False
}


def deep_merge(base: dict, override: dict) -> dict:
    result = dict(base)
    for key, value in override.items():
        if key in result and isinstance(result[key], dict) and isinstance(value, dict):
            result[key] = deep_merge(result[key], value)
        else:
            result[key] = value
    return result


def load_json(path: str) -> dict:
    with open(path, "r", encoding="utf-8") as f:
        return json.load(f)


def main():
    args = [a for a in sys.argv[1:] if not a.startswith("--")]
    output_path = None
    if "--output" in sys.argv:
        idx = sys.argv.index("--output")
        if idx + 1 < len(sys.argv):
            output_path = sys.argv[idx + 1]

    if len(args) >= 2:
        merged = {}
        for path in args:
            if not os.path.isfile(path):
                print(f"Error: file not found: {path}")
                return
            try:
                data = load_json(path)
            except json.JSONDecodeError as e:
                print(f"Error parsing {path}: {e}")
                return
            merged = deep_merge(merged, data)
    else:
        print("No files given (need at least 2), merging demo configs.\n")
        merged = deep_merge(DEMO_BASE, DEMO_OVERRIDE)

    output_text = json.dumps(merged, indent=2)

    if output_path:
        with open(output_path, "w", encoding="utf-8") as f:
            f.write(output_text + "\n")
        print(f"Merged configuration written to: {output_path}")
    else:
        print("Merged configuration:")
        print(output_text)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`deep_merge()` copies the base dict, then walks the override dict key
by key: if both the base and override values for a key are
dictionaries, it recurses to merge them; otherwise the override value
simply replaces the base value. The script folds any number of
positional file arguments together in sequence with successive calls
to `deep_merge`, so three or more files work the same way as two.

```bash
# Run with no arguments — merges two small built-in demo configs
python3 merge_config_files.py

# Merge base + one override, printed to stdout
python3 merge_config_files.py base.json production.json

# Merge base + one override, redirected to a new file via shell
python3 merge_config_files.py base.json production.json > merged.json

# Merge and write directly with --output instead of shell redirection
python3 merge_config_files.py base.json production.json --output merged.json

# Merge three files: base, then region override, then host-specific override
python3 merge_config_files.py base.json region.json host.json --output final.json
```

#### Sample output

```text
$ python3 merge_config_files.py
No files given (need at least 2), merging demo configs.

Merged configuration:
{
  "server": {
    "host": "127.0.0.1",
    "port": 9090
  },
  "debug": false
}
```

Here `port` and `debug` from the override replaced the base file's
values, while `host` was kept from the base file since the override
didn't specify it — deep merge in action. Captured from a real run of
the uploaded script — confirmed correct.

#### Variations / flags

- `--output PATH` — write the merged result to a file instead of
  printing to stdout (demonstrated above).
- Without `--output`, use standard shell redirection (`>`) to the same
  effect, which also lets you pipe the result into another tool.
- Add a `--list-strategy concat` flag to opt into concatenating list
  values instead of replacing them, for schemas where that behavior
  is preferred (e.g. merging lists of allowed IPs).

#### Common pitfalls

- **File order matters**: `merge_config_files.py override.json
  base.json` merges backwards — base values will win over the
  override, silently producing the opposite of the intended result.
- **Lists don't concatenate**: a base file with
  `"allowed_ips": ["10.0.0.1"]` and an override with
  `"allowed_ips": ["10.0.0.2"]` results in only `10.0.0.2` surviving,
  not both — a common surprise for anyone expecting array merging.
- **JSON only**: all input files must be valid JSON; this script does
  not merge YAML or INI files directly, even though the same deep-merge
  logic would apply once parsed — you'd need to parse with
  `parse_yaml_config.py`-style code first and adapt this script.

### 4.9 Generate Configuration Files

#### Use case

Spinning up a new environment, a new service instance, or bootstrapping
a project template often means producing a fresh, correctly-formatted
config file from scratch. A sysadmin reaches for this when they need
to generate a starter JSON, YAML, or INI config from a known-good
Python dictionary rather than hand-typing brackets and worrying about
formatting.

#### Prerequisites

JSON and INI output need only the standard library. YAML output
requires `PyYAML`:

```bash
python3 --version                              # 3.6 or newer
pip install pyyaml --break-system-packages     # only needed for --format yaml
# or:
pip install -r requirements.txt --break-system-packages
```

#### Cautions

**If the output file already exists, it is overwritten without a
prompt.** Run `backup_config.py` (4.11) first if you're regenerating a
config you care about. The dictionary of values generated
(`DEFAULT_CONFIG`) is hard-coded inside the script — edit it directly
to match what your own project actually needs before relying on this
for anything beyond a demo or template.

#### Script

```python
#!/usr/bin/env python3
"""
generate_config_file.py

Purpose:
    Generate a new configuration file from a Python dictionary, in
    your choice of format: JSON, YAML, or INI.

Usage:
    python3 generate_config_file.py --format json --output config.json
    python3 generate_config_file.py --format yaml --output config.yaml
    python3 generate_config_file.py --format ini  --output config.ini

    If no arguments are given, this script generates a demo JSON config
    file (generated_demo_config.json) in the current directory.
"""

import configparser
import json
import os
import sys

DEFAULT_CONFIG = {
    "server": {
        "host": "127.0.0.1",
        "port": 8080
    },
    "debug": False
}


def generate_json(config: dict) -> str:
    return json.dumps(config, indent=2)


def generate_yaml(config: dict) -> str:
    try:
        import yaml
    except ImportError:
        print("Error: YAML output requires 'PyYAML'. Install with:")
        print("    pip install -r requirements.txt")
        sys.exit(1)
    return yaml.dump(config, default_flow_style=False, sort_keys=False)


def generate_ini(config: dict) -> str:
    parser = configparser.ConfigParser()
    for section, values in config.items():
        if isinstance(values, dict):
            parser[section] = {k: str(v) for k, v in values.items()}
        else:
            # top-level scalar values go under a generic [general] section
            if "general" not in parser:
                parser["general"] = {}
            parser["general"][section] = str(values)
    import io
    buf = io.StringIO()
    parser.write(buf)
    return buf.getvalue()


def parse_args():
    args = sys.argv[1:]
    fmt = None
    output = None
    i = 0
    while i < len(args):
        if args[i] == "--format" and i + 1 < len(args):
            fmt = args[i + 1].lower(); i += 2
        elif args[i] == "--output" and i + 1 < len(args):
            output = args[i + 1]; i += 2
        else:
            i += 1
    return fmt, output


def main():
    fmt, output = parse_args()

    if not fmt:
        fmt = "json"
        output = output or "generated_demo_config.json"
        print("No --format given, generating a demo JSON config file.\n")

    generators = {"json": generate_json, "yaml": generate_yaml, "ini": generate_ini}
    if fmt not in generators:
        print(f"Error: unsupported format '{fmt}'. Choose from: json, yaml, ini")
        return

    content = generators[fmt](DEFAULT_CONFIG)

    if not output:
        output = f"generated_config.{fmt}"

    if os.path.isfile(output):
        print(f"Note: '{output}' already exists and will be overwritten.")

    with open(output, "w", encoding="utf-8") as f:
        f.write(content)

    print(f"Generated {fmt} config:")
    print(content)
    print(f"Written to: {output}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`parse_args()` does a small manual scan of `sys.argv` for `--format`
and `--output` rather than using `argparse`. Based on `--format`, one
of three generator functions turns the same `DEFAULT_CONFIG`
dictionary into text: `generate_json` uses `json.dumps`, `generate_yaml`
uses `yaml.dump` (importing `PyYAML` lazily, only when YAML output is
requested), and `generate_ini` builds a `configparser.ConfigParser`,
mapping top-level scalar values (like `debug`) into a generic
`[general]` section since INI has no concept of a bare top-level key.

```bash
# Run with no arguments — generates a demo JSON file
python3 generate_config_file.py

# Generate a JSON config at a specific path
python3 generate_config_file.py --format json --output config.json

# Generate a YAML config (requires PyYAML)
python3 generate_config_file.py --format yaml --output config.yaml

# Generate an INI config
python3 generate_config_file.py --format ini --output config.ini
```

#### Sample output

```text

  host: 127.0.0.1
  port: 8080
debug: false

Written to: gen.yaml
```

```text
$ python3 generate_config_file.py --format ini --output gen.ini

[server]
host = 127.0.0.1
port = 8080

[general]

Written to: gen.ini
```

`configparser.write()` naturally produces after the last section.

#### Variations / flags

- `--format {json,yaml,ini}` — choose the output format (default:
  `json` when omitted, as part of demo mode).
- `--output PATH` — choose the destination file path (default:
  `generated_config.<format>`).
- Edit `DEFAULT_CONFIG` directly to template out your own project's
  starter configuration instead of the demo `server`/`debug` example.
- Add a `--force` / `--no-overwrite` pair of flags to make the
  overwrite-without-prompting behavior opt-in rather than default.

#### Common pitfalls

- **Silent overwrite**: this is the single biggest risk with this
  script — running it twice against the same `--output` path replaces
  the file with no confirmation, only a `Note: ... already exists`
  message that's easy to miss in scrolled terminal output.
- **INI can't represent nested structures beyond one level**: if
  `DEFAULT_CONFIG` had a doubly-nested dict (e.g.
  `{"server": {"tls": {"enabled": true}}}`), `generate_ini` would
  raise an error trying to stringify a dict as a value — INI's
  `[section]` model only supports one level of nesting.
- **YAML boolean casing**: `yaml.dump` writes Python's `False` as
  lowercase `false`, matching YAML convention — if you hand-edit the
  generated file later, remember unquoted `False`/`True` (capitalized)
  are treated as plain strings by strict YAML 1.2 parsers, not
  booleans.

### 4.10 Modify Existing Configuration Files

#### Use case

Sometimes you just need to flip one setting — bump a port number,

file. A sysadmin reaches for this to make a single, precise, scriptable
change to one key in a JSON config, including nested keys via dot
notation like `server.port`.

#### Prerequisites

Standard library only (`json`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

**This script overwrites the original file with the modified version
— there is no prompt and no built-in undo.** Always run
`backup_config.py` (4.11) on a file first if it matters. Values are
auto-converted (`"true"`/`"false"` become booleans, numeric strings
become `int`/`float`, everything else stays a string) — if you
genuinely need the literal string `"true"` rather than the boolean
`true`, this simple script cannot preserve that distinction. Dot
notation only works for nested dictionaries, not list indices — you
can't target something like `servers.0.port`.

#### Script

```python
#!/usr/bin/env python3
"""
modify_config_file.py

Purpose:
    Modify a single key's value in an existing JSON configuration file,
    using dot notation to reach nested keys, then save the file back.

Usage:
    python3 modify_config_file.py <path.json> --key server.port --value 9090
    python3 modify_config_file.py <path.json> --key debug --value true

    If no arguments are given, this script creates a demo config file,
    modifies one of its values, and shows the before/after result.
"""

import json
import os
import sys

DEMO_FILE = "demo_config_to_modify.json"
DEMO_CONTENT = {
    "server": {"host": "127.0.0.1", "port": 8080},
    "debug": False
}


def convert_value(raw: str):
    if raw.lower() == "true":
        return True
    if raw.lower() == "false":
        return False
    try:
        if "." in raw:
            return float(raw)
        return int(raw)
    except ValueError:
        return raw


def set_nested_value(data: dict, dotted_key: str, value) -> None:
    keys = dotted_key.split(".")
    current = data
    for key in keys[:-1]:
        if key not in current or not isinstance(current[key], dict):
            current[key] = {}
        current = current[key]
    current[keys[-1]] = value


def parse_args():
    args = sys.argv[1:]
    path = None
    key = None
    value = None
    positional = []
    i = 0
    while i < len(args):
        if args[i] == "--key" and i + 1 < len(args):
            key = args[i + 1]; i += 2
        elif args[i] == "--value" and i + 1 < len(args):
            value = args[i + 1]; i += 2
        else:
            positional.append(args[i]); i += 1
    if positional:
        path = positional[0]
    return path, key, value


def main():
    path, key, raw_value = parse_args()

    if not path or not key or raw_value is None:
        print("No arguments given, running demo mode.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                json.dump(DEMO_CONTENT, f, indent=2)
        path, key, raw_value = DEMO_FILE, "server.port", "9090"

    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    with open(path, "r", encoding="utf-8") as f:
        try:
            data = json.load(f)
        except json.JSONDecodeError as e:
            print(f"Error parsing JSON file: {e}")
            return

    print("Before:")
    print(json.dumps(data, indent=2))

    value = convert_value(raw_value)
    set_nested_value(data, key, value)

    print(f"\nAfter setting '{key}' = {value!r}:")
    print(json.dumps(data, indent=2))

    with open(path, "w", encoding="utf-8") as f:
        json.dump(data, f, indent=2)

    print(f"\nSaved to: {path}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`set_nested_value()` splits the dotted key on `.` and walks into the
dict one level at a time, creating intermediate dictionaries if a
level doesn't exist yet, then sets the final key to the converted
value. `convert_value()` does lightweight type coercion so
`--value 9090` becomes the integer `9090` rather than the string
`"9090"`. The script prints the full file before and after the change
for a visible diff, then overwrites the original file with the result.

```bash

python3 modify_config_file.py

# Set a top-level key
python3 modify_config_file.py config.json --key debug --value true

# Set a nested key via dot notation
python3 modify_config_file.py config.json --key server.port --value 9090

# Set a key that creates a new nested section that didn't exist before
python3 modify_config_file.py config.json --key cache.ttl --value 300
```

    "host": "127.0.0.1",

#### Variations / flags

- `--key DOTTED.PATH` — the key to modify, using `.` to reach nested
  dictionaries (required, along with `--value`).
- `--value RAW` — the new value as a string; auto-converted to bool,
  int, float, or left as a string.
- Add a `--dry-run` flag that prints the before/after diff but skips
  the final `json.dump()` write, letting you preview a change safely.
- Add a `--type {str,int,float,bool}` flag to force a specific type
  instead of relying on automatic conversion, solving the "literal
  string true" problem noted above.

#### Common pitfalls

- **No confirmation before overwrite**: unlike a text editor, there's
  no "save" step to back out of — the file is overwritten the moment
  the script finishes running.
- **Auto-conversion can pick the wrong type**: a value like `"007"`
  (e.g. an area code stored as a string) becomes the integer `7`
  through `int("007")`, silently dropping the leading zeros — verify
  the "After" output before trusting the auto-conversion for
  string-like numeric values.
- **Creating new keys instead of typo-catching**: if you mistype a key
  name (`sever.port` instead of `server.port`), the script happily
  creates a brand-new `sever` section rather than warning you the
  intended key wasn't found — always check the "Before"/"After" diff
  printed to the console.

### 4.11 Backup Configuration Files

#### Use case

Before touching a config file with `modify_config_file.py` (4.10),
`generate_config_file.py` (4.9), or any manual edit, a sysadmin wants
a safe restore point. This script creates a timestamped copy of a
config file so a mistake can be undone with `restore_config.py`
(4.12) instead of reconstructing the file from memory.

#### Prerequisites

Standard library only (`shutil`, `datetime`) — no installation
required. Runs on Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

By default, backups are placed **alongside the original file**, in
the same directory, with a timestamp suffix — use `--backup-dir` to
store them somewhere else instead (e.g. a dedicated `backups/` folder
outside your deployment path). This script does **not** delete old
backups automatically; running it repeatedly accumulates one backup
file per run, so clean up old backups periodically if disk space
matters. It only copies files — it never modifies the original.

#### Script

```python
#!/usr/bin/env python3
"""
backup_config.py

Purpose:
    Create a timestamped backup copy of a configuration file, so you
    have a safe restore point before modifying it.

Usage:
    python3 backup_config.py <path_to_config_file>
    python3 backup_config.py <path_to_config_file> --backup-dir backups

    If no path is given, this script creates a small demo config file
    and backs it up as an example.
"""

import datetime
import os
import shutil
import sys

DEMO_FILE = "demo_config_to_backup.json"
DEMO_CONTENT = '{\n  "server": {"host": "127.0.0.1", "port": 8080}\n}\n'


def backup_file(path: str, backup_dir: str = None) -> None:
    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    timestamp = datetime.datetime.now().strftime("%Y%m%d_%H%M%S")
    filename = os.path.basename(path)
    backup_name = f"{filename}.{timestamp}.bak"

    if backup_dir:
        os.makedirs(backup_dir, exist_ok=True)
        backup_path = os.path.join(backup_dir, backup_name)
    else:
        backup_path = os.path.join(os.path.dirname(path) or ".", backup_name)

    shutil.copy2(path, backup_path)
    print(f"Backed up '{path}' to '{backup_path}'")


def parse_args():
    args = sys.argv[1:]
    path = None
    backup_dir = None
    positional = []
    i = 0
    while i < len(args):
        if args[i] == "--backup-dir" and i + 1 < len(args):
            backup_dir = args[i + 1]; i += 2
        else:
            positional.append(args[i]); i += 1
    if positional:
        path = positional[0]
    return path, backup_dir


def main():
    path, backup_dir = parse_args()

    if not path:
        print("No file path given, creating and backing up a demo config file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE

    backup_file(path, backup_dir)


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`shutil.copy2()` copies the file's contents *and* its metadata
(modification time, permission bits), which is preferable to a plain
`shutil.copy()` for backup purposes since it preserves when the
original was last actually changed. The backup filename is built as
`<original_name>.<YYYYMMDD_HHMMSS>.bak`, so backups sort chronologically
by filename and never collide as long as you don't run the script
twice within the same second.

```bash
# Run with no arguments — creates and backs up a demo config file
python3 backup_config.py

# Back up a specific file, next to the original
python3 backup_config.py /etc/myapp/config.json

# Back up to a dedicated backups directory instead
python3 backup_config.py /etc/myapp/config.json --backup-dir /etc/myapp/backups

# Back up right before a scripted change
python3 backup_config.py config.json && python3 modify_config_file.py config.json --key debug --value false
```

#### Sample output

```text
$ python3 backup_config.py
No file path given, creating and backing up a demo config file.

Backed up 'demo_config_to_backup.json' to './demo_config_to_backup.json.20260717_182558.bak'
```

```text
$ python3 backup_config.py demo_config_to_backup.json --backup-dir backups
Backed up 'demo_config_to_backup.json' to 'backups/demo_config_to_backup.json.20260717_182609.bak'
```

Both captured from real runs of the uploaded script — confirmed
correct (timestamps will naturally differ on each run).

#### Variations / flags

- `--backup-dir DIR` — store the backup in a specific directory
  instead of next to the original file (the directory is created if
  it doesn't exist).
- Add a `--keep N` flag to automatically prune older backups beyond
  the most recent `N`, addressing the "no automatic cleanup" caution
  above.
- Schedule this with cron before a nightly config sync job:
  `0 2 * * * /usr/bin/python3 /opt/scripts/backup_config.py /etc/myapp/config.json --backup-dir /var/backups/myapp`.

#### Common pitfalls

- **Accumulating backups silently**: since nothing is ever deleted
  automatically, a config file that's modified frequently (e.g. via a
  script that backs up before every change) can quietly fill a disk
  over weeks or months — monitor `--backup-dir` disk usage or add
  rotation.
- **Backup directory permissions**: if `--backup-dir` points somewhere
  the running user can't write (e.g. a system directory), `os.makedirs`
  will raise a `PermissionError` traceback rather than a friendly
  message — run with appropriate privileges or point at a writable path.
- **Same-second collisions**: the timestamp has one-second resolution;
  running the script twice in rapid succession (e.g. in a tight loop)
  can produce the same backup filename and silently overwrite the
  first backup with the second.

### 4.12 Restore Configuration Files

#### Use case

When a config change goes wrong — a bad value causes a service to
crash, or a generated file overwrote something important — a
sysadmin needs to roll back to a known-good backup fast. This script
is the other half of the 4.11 backup workflow: it copies a previously
created `.bak` file back over the live config, undoing the change.

#### Prerequisites

Standard library only (`shutil`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

**This overwrites the destination file without prompting** — make
absolutely sure you're restoring to the correct path before running
it, especially in a script or automation context. This script does
**not** validate that the backup file is a well-formed config in your
expected format before restoring it; it performs a plain file copy.
Verify the backup's contents first with `read_config_file.py` (4.1)
if you're unsure what's actually in it. If the destination file
doesn't exist yet, it's simply created from the backup — restoring
onto a "clean slate" works fine.

#### Script

```python
#!/usr/bin/env python3
"""
restore_config.py

Purpose:
    Restore a configuration file from a previously created backup (see
    backup_config.py), overwriting the current version with the backed
    up one.

Usage:
    python3 restore_config.py <path_to_backup_file> <path_to_restore_to>

    If no arguments are given, this script runs an end-to-end demo:
    it creates a config, backs it up, modifies the original, then
    restores it from the backup so you can see the original content
    return.
"""

import os
import shutil
import sys

DEMO_ORIGINAL = "demo_config_to_restore.json"
DEMO_CONTENT_ORIGINAL = '{\n  "server": {"host": "127.0.0.1", "port": 8080}\n}\n'
DEMO_CONTENT_MODIFIED = '{\n  "server": {"host": "0.0.0.0", "port": 9999}\n}\n'


def restore_file(backup_path: str, restore_to: str) -> None:
    if not os.path.isfile(backup_path):
        print(f"Error: backup file not found: {backup_path}")
        return
    shutil.copy2(backup_path, restore_to)
    print(f"Restored '{backup_path}' -> '{restore_to}'")


def run_demo():
    import datetime
    print("No arguments given, running a full backup+restore demo.\n")

    # 1. Create an "original" config
    with open(DEMO_ORIGINAL, "w", encoding="utf-8") as f:
        f.write(DEMO_CONTENT_ORIGINAL)
    print(f"1. Created original config: {DEMO_ORIGINAL}")

    # 2. Back it up
    timestamp = datetime.datetime.now().strftime("%Y%m%d_%H%M%S")
    backup_path = f"{DEMO_ORIGINAL}.{timestamp}.bak"
    shutil.copy2(DEMO_ORIGINAL, backup_path)
    print(f"2. Backed up to: {backup_path}")

    # 3. Simulate a bad change to the original
    with open(DEMO_ORIGINAL, "w", encoding="utf-8") as f:
        f.write(DEMO_CONTENT_MODIFIED)
    print(f"3. Modified original (simulating an unwanted change).")

    # 4. Restore from the backup
    restore_file(backup_path, DEMO_ORIGINAL)

    with open(DEMO_ORIGINAL, "r", encoding="utf-8") as f:
        print("\nFinal restored content:")
        print(f.read())


def main():
    args = sys.argv[1:]
    if len(args) >= 2:
        restore_file(args[0], args[1])
    else:
        run_demo()


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`restore_file()` is a thin wrapper around `shutil.copy2()`, copying
the backup file's contents and metadata directly over the destination
path — conceptually the exact reverse direction of `backup_config.py`.
With no arguments, `run_demo()` walks through the entire
backup-then-restore lifecycle in one run: create a file, back it up,
deliberately corrupt the original, then restore it, printing the final
content so you can see the rollback actually worked.

```bash
# Run with no arguments — full backup+restore demo, prints before/after
python3 restore_config.py

# Restore a specific backup over the live config
python3 restore_config.py config.json.20260717_101530.bak config.json

# Restore to a different path than the original (e.g. staging a copy)
python3 restore_config.py config.json.20260717_101530.bak /tmp/config_review.json
```

#### Sample output

```text
$ python3 restore_config.py
No arguments given, running a full backup+restore demo.

1. Created original config: demo_config_to_restore.json
2. Backed up to: demo_config_to_restore.json.20260717_182600.bak
3. Modified original (simulating an unwanted change).
Restored 'demo_config_to_restore.json.20260717_182600.bak' -> 'demo_config_to_restore.json'

{
  "server": {"host": "127.0.0.1", "port": 8080}
}
```

Captured from a real run of the uploaded script — the final content

change, confirming the restore worked as documented.

#### Variations / flags

- Positional arguments only: `<backup_path> <restore_to>` — no named
  flags in the current version.
- Add a `--confirm` flag that prints a diff between the current
  destination and the backup and requires a keypress before
  overwriting, turning off the "no prompt" behavior for interactive use.
- Add a `--list-backups DIR` mode that scans a directory for
  `*.bak` files matching a given base filename and lets you pick one
  interactively, instead of needing to know the exact timestamp.

#### Common pitfalls

- **Wrong argument order**: the backup path comes first, the
  destination second — `restore_config.py config.json backup.bak`
  would overwrite your good backup with the (possibly bad) live
  config, the opposite of the intended operation.
- **No content validation**: restoring a `.bak` file that happens to
  be truncated, corrupted, or from an incompatible schema version will
  succeed silently — the script doesn't parse or validate the backup's
  format before copying it over the destination.
- **Losing track of which backup is "good"**: since `backup_config.py`
  accumulates one file per run with no automatic labeling beyond a
  timestamp, restoring the wrong `.bak` file (an older one, from
  before a legitimate change you wanted to keep) is an easy mistake —
  check the backup's modification time or content first.

### 4.13 Manage Application Settings

#### Use case

Beyond raw file parsing, applications often need a small persistent
settings store — user preferences, feature toggles, a theme choice —
that's simple to read and write from the command line or from other
scripts, with sane defaults created automatically the first time it
runs. A sysadmin or developer reaches for this pattern when building
or debugging that kind of lightweight settings layer, backed by a
plain JSON file rather than a database.

#### Prerequisites

Standard library only (`json`) — no installation required. Runs on
Linux, macOS, and Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

Settings are stored in `app_settings.json` **in the current working
directory** by default — delete that file if you want to reset to
defaults, and be aware that running this script from different
directories will read/write different settings files, which can be
confusing. This script **overwrites `app_settings.json` on every
`set` call** — run `backup_config.py` (4.11) first if you want a
safety copy of settings you care about. Values passed via `set` are
auto-converted (`true`/`false` become `bool`, numeric strings become
`int`/`float`); everything else is stored as a plain string.

#### Script

```python
#!/usr/bin/env python3
"""
manage_app_settings.py

Purpose:
    A small reusable "settings manager" for an application: load
    settings from a JSON file (creating sane defaults if missing),
    get/set individual values, and save changes back to disk.

Usage:
    python3 manage_app_settings.py get theme
    python3 manage_app_settings.py set theme dark
    python3 manage_app_settings.py list

    If no arguments are given, this script runs a demo: it creates a
    default settings file, sets a value, and lists all settings.
"""

import json
import os
import sys

SETTINGS_FILE = "app_settings.json"
DEFAULT_SETTINGS = {
    "theme": "light",
    "language": "en",
    "notifications_enabled": True
}


def load_settings() -> dict:
    if not os.path.isfile(SETTINGS_FILE):
        save_settings(DEFAULT_SETTINGS)
        return dict(DEFAULT_SETTINGS)
    with open(SETTINGS_FILE, "r", encoding="utf-8") as f:
        return json.load(f)


def save_settings(settings: dict) -> None:
    with open(SETTINGS_FILE, "w", encoding="utf-8") as f:
        json.dump(settings, f, indent=2)


def convert_value(raw: str):
    if raw.lower() == "true":
        return True
    if raw.lower() == "false":
        return False
    try:
        if "." in raw:
            return float(raw)
        return int(raw)
    except ValueError:
        return raw


def cmd_get(key: str) -> None:
    settings = load_settings()
    if key in settings:
        print(f"{key} = {settings[key]}")
    else:
        print(f"Setting '{key}' not found.")


def cmd_set(key: str, raw_value: str) -> None:
    settings = load_settings()
    value = convert_value(raw_value)
    settings[key] = value
    save_settings(settings)
    print(f"Updated '{key}' = {value!r}")


def cmd_list() -> None:
    settings = load_settings()
    print("Current application settings:")
    for key, value in settings.items():
        print(f"  {key} = {value}")


def main():
    args = sys.argv[1:]

    if not args:
        print("No arguments given, running demo mode.\n")
        cmd_list()
        print()
        cmd_set("theme", "dark")
        print()
        cmd_list()
        return

    action = args[0].lower()
    if action == "get" and len(args) >= 2:
        cmd_get(args[1])
    elif action == "set" and len(args) >= 3:
        cmd_set(args[1], args[2])
    elif action == "list":
        cmd_list()
    else:
        print("Usage:")
        print("  python3 manage_app_settings.py get <key>")
        print("  python3 manage_app_settings.py set <key> <value>")
        print("  python3 manage_app_settings.py list")


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`load_settings()` is the core of the pattern: if `app_settings.json`
doesn't exist yet, it's created with `DEFAULT_SETTINGS` and that
default dict is returned; otherwise the existing file is read and
returned as-is. `get`, `set`, and `list` are simple subcommands
dispatched on `sys.argv[1]`; `set` re-saves the entire settings file
after updating one key, since this is a "load whole file, modify,
save whole file" pattern rather than an in-place edit.

```bash
# Run with no arguments — demo: list defaults, set theme=dark, list again
python3 manage_app_settings.py

# Get a single setting
python3 manage_app_settings.py get theme

# Set a setting (auto-converted: true/false, numbers, or string)
python3 manage_app_settings.py set theme dark
python3 manage_app_settings.py set notifications_enabled false
python3 manage_app_settings.py set retry_count 5

# List every current setting
python3 manage_app_settings.py list
```

#### Sample output

```text
$ python3 manage_app_settings.py get theme
theme = dark

$ python3 manage_app_settings.py set language fr
Updated 'language' = 'fr'

$ python3 manage_app_settings.py list
Current application settings:
  theme = dark
  language = fr
  notifications_enabled = True
```

Captured from real, chained runs of the uploaded script (following an
earlier demo run that had already set `theme` to `dark`) — confirmed
correct, including the auto-conversion of `set language fr` storing
the plain string `'fr'`.

#### Variations / flags

- `get <key>` — print a single setting's current value.
- `set <key> <value>` — update a setting, with automatic type
  conversion, and persist the change immediately.
- `list` — print every current setting.
- Add a `--file PATH` global flag so the settings file location isn't
  hard-coded to `app_settings.json` in the current directory, useful
  for managing settings for multiple app instances.
- Add a `delete <key>` subcommand to remove a setting entirely and
  fall back to the default on next `get`.

#### Common pitfalls

- **Working-directory dependence**: because `SETTINGS_FILE` is a
  relative path, running this script from two different directories
  (e.g. a cron job's working directory vs. an interactive shell) reads
  and writes two entirely different settings files without warning.
- **No merge on missing keys**: if `DEFAULT_SETTINGS` gains a new key
  in a future version of the script but an existing `app_settings.json`
  predates that change, `load_settings()` returns the file's contents
  as-is — the new default key won't appear until you either delete the
  file or explicitly `set` it once.
- **Overwrite on every `set`**: like several other scripts in this
  chapter, there's no locking — running two `set` commands concurrently
  (e.g. from two scripts at once) can result in a lost update, since
  the second `save_settings()` call overwrites the whole file based on
  whatever it read at the start of its own run.

### 4.14 Load Environment Variables

#### Use case

Many applications read secrets and settings (database URLs, API keys,
feature flags) from environment variables rather than a config file
directly, following the twelve-factor app convention, with a local
`.env` file used to populate those variables during development. A
sysadmin or developer reaches for this when they need to load a
`.env` file into a Python process's environment without pulling in a
third-party dependency like `python-dotenv`.

#### Prerequisites

Standard library only (`os`) — no installation required, deliberately
avoiding a dependency on `python-dotenv`. Runs on Linux, macOS, and
Windows on any Python 3.6+.

```bash
python3 --version   # 3.6 or newer
```

#### Cautions

**Environment variables set by this script only affect this Python
process** (and any child processes it launches afterward, e.g. via
`subprocess`) — they do not persist in your shell/terminal once the
script exits, which surprises people expecting `export`-like behavior.
By default, this script does **not** overwrite a variable that's
already set in the real environment, matching common `.env`-loader
conventions — pass `--override` to force overwriting existing values.
Do not commit real `.env` files containing secrets to version control;
this script is for *loading* them locally, not generating or sharing
them. The parser is intentionally minimal: it supports simple
`KEY=VALUE` lines and `#` comments, but not multi-line values or the
interpolation features some third-party `.env` libraries support.

#### Script

```python
#!/usr/bin/env python3
"""
load_env_file.py

Purpose:
    Load key=value pairs from a .env-style file into the current
    process's environment variables, without requiring any third-party
    library (a minimal hand-rolled parser).

Usage:
    python3 load_env_file.py <path_to_.env_file>

    If no path is given, this script creates and loads a demo .env
    file in the current directory.
"""

import os
import sys

DEMO_FILE = ".env"
DEMO_CONTENT = (
    "# Demo environment file\n"
    "DEBUG=true\n"
    "DATABASE_URL=postgres://localhost/mydb\n"
    "SECRET_KEY=demo-secret-value\n"
)


def parse_env_file(path: str) -> dict:
    variables = {}
    with open(path, "r", encoding="utf-8") as f:
        for line in f:
            line = line.strip()
            if not line or line.startswith("#") or "=" not in line:
                continue
            key, _, value = line.partition("=")
            key = key.strip()
            value = value.strip().strip('"').strip("'")
            variables[key] = value
    return variables


def load_into_environment(variables: dict, override: bool = False) -> None:
    for key, value in variables.items():
        if override or key not in os.environ:
            os.environ[key] = value


def main():
    override = "--override" in sys.argv
    positional = [a for a in sys.argv[1:] if not a.startswith("--")]

    if positional:
        path = positional[0]
    else:
        print("No file path given, creating and loading a demo .env file.\n")
        if not os.path.isfile(DEMO_FILE):
            with open(DEMO_FILE, "w", encoding="utf-8") as f:
                f.write(DEMO_CONTENT)
        path = DEMO_FILE

    if not os.path.isfile(path):
        print(f"Error: file not found: {path}")
        return

    variables = parse_env_file(path)
    load_into_environment(variables, override=override)

    print(f"Loaded {len(variables)} variable(s) from {path}:")
    for key, value in variables.items():
        print(f"  {key}={value}")

    print("\nVerifying via os.environ:")
    for key in list(variables.keys())[:2]:
        print(f"  os.environ['{key}'] = {os.environ.get(key)}")


if __name__ == "__main__":
    main()
```

#### How it works and how to run it

`parse_env_file()` reads the file line by line, skipping blank lines
and `#`-prefixed comments, and splits each remaining line on the first
`=` with `str.partition`, stripping surrounding whitespace and
matching quote characters from the value. `load_into_environment()`
then writes each parsed variable into `os.environ`, respecting the
"don't overwrite existing values" default unless `--override` is
passed. Because this only mutates the current process's environment,
it's most useful either at the very top of a longer Python script (so
subsequent code in the same process sees the variables) or before
launching a child process that inherits the environment.

```bash
# Run with no arguments — creates and loads a demo .env file
python3 load_env_file.py

# Load a specific .env file
python3 load_env_file.py .env.production

# Load and force-override any variables already set in the shell
python3 load_env_file.py .env --override

# Typical use: source this at the top of a larger deployment script
python3 -c "
import load_env_file, os
load_env_file.load_into_environment(load_env_file.parse_env_file('.env'))
print(os.environ['DATABASE_URL'])
"
```

#### Variations / flags

- `--override` — force-overwrite any environment variable that
  already exists, instead of skipping it.
- A single positional `<path>` argument for the `.env` file to load;
  defaults to `.env` in the current directory in demo mode.
- Add a `--quiet` flag to suppress printing loaded values to stdout,
  since echoing secrets like `SECRET_KEY` to the console/log is a
  real risk in shared CI environments.
- Add multi-line value support (values wrapped in matching triple
  quotes) if your `.env` files need to store things like a multi-line
  private key.

#### Common pitfalls

- **Expecting shell persistence**: the most common misunderstanding —
  running `python3 load_env_file.py` does *not* make the variables
  available in your interactive shell afterward, since a Python
  subprocess cannot modify its parent shell's environment; you need to
  `source` a real `.env` file with shell syntax, or run your
  application *from inside* the same Python process that loaded it.
- **Printing secrets to stdout**: the default output echoes every
  loaded value, including `SECRET_KEY` in the demo — be cautious
  running this against a real `.env` file in a shared terminal, logged
  CI job, or screen-shared session.
- **No support for multi-line or quoted-with-equals values**: a value
  containing a literal `=` after the first one is fine (only the
  *first* `=` splits key from value), but multi-line values or values
  needing shell-style escaping are not supported by this minimal
  parser — use `python-dotenv` if you need that.
