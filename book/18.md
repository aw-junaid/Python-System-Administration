## Chapter 18: Automation Utilities

Every sysadmin and backend engineer eventually writes the same fifteen scripts twice: once badly, at 2 a.m., during an incident, and once properly, months later, after the third time it bit them in production. This chapter collects the properly-written versions. They are the connective tissue of automation work — not the business logic itself, but the scaffolding that makes business logic survive contact with flaky networks, slow disks, overloaded CPUs, and the general unreliability of distributed systems.

None of these scripts depend on a framework. Most use nothing beyond the Python standard library; the few that touch Redis degrade gracefully to an in-process fallback when Redis isn't available, so every demo in this chapter runs out of the box with a plain `python3 script.py`. That's deliberate. A retry decorator you have to `pip install` six packages to try is a retry decorator nobody actually tries.

The chapter is organized roughly by the shape of the problem each utility solves: making unreliable operations reliable (retries, backoff, timeouts), giving humans visibility into long-running work (progress bars, spinners), controlling how much work happens at once (rate limiting, parallel execution, threading, multiprocessing, async), and managing the resources and state that automation scripts accumulate (job queues, caching, config validation, cleanup). Read them in order for a narrative arc, or jump straight to the topic you need — each section stands alone.

### 18.1 Retry Failed Operations

#### Use case

A sysadmin needs a database connection, an HTTP call, or a file-lock acquisition to survive the occasional transient failure — a dropped connection, a momentary 503, a lock held by another process for half a second — without hand-writing a `for` loop and `try/except` around every call site in the codebase.

#### Prerequisites

Pure standard library. No installation needed beyond Python 3.7+ (uses `functools.wraps`). Works identically on Linux, macOS, and Windows.

```bash
python3 --version   # 3.7 or newer is all that's required
```

#### Cautions

Retrying blindly on *any* exception (the default `exceptions=(Exception,)`) can mask real bugs — a `TypeError` from a coding mistake will get silently retried three times instead of failing fast. Always narrow `exceptions` to the specific failure modes you expect (e.g. `ConnectionError`, `TimeoutError`).

#### Script

```python
#!/usr/bin/env python3
"""
retry_failed_operations.py
--------------------
A simple @retry decorator: catches specified exceptions and retries the
wrapped function up to N times with a fixed delay between attempts.
"""

import functools
import time


def retry(times=3, delay=1.0, exceptions=(Exception,)):
    """
    Decorator factory. Retries the wrapped function up to `times` times,
    waiting `delay` seconds between attempts, only for the given
    `exceptions`. Re-raises the last exception if all attempts fail.
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as exc:
                    last_exc = exc
                    print(f"[retry] Attempt {attempt}/{times} for '{func.__name__}' "
                          f"failed: {exc!r}")
                    if attempt < times:
                        time.sleep(delay)
            print(f"[retry] All {times} attempts failed for '{func.__name__}'.")
            raise last_exc
        return wrapper
    return decorator


if __name__ == "__main__":
    call_count = {"n": 0}

    @retry(times=4, delay=0.3, exceptions=(RuntimeError,))
    def unstable_operation():
        call_count["n"] += 1
        if call_count["n"] < 3:
            raise RuntimeError(f"simulated failure #{call_count['n']}")
        return "success"

    print("Demo: retrying an operation that fails twice before succeeding.\n")
    result = unstable_operation()
    print(f"\nFinal result: {result}")
    print(f"Total calls made: {call_count['n']}")
```

#### How It Works & How to Run

`retry(times, delay, exceptions)` is a decorator factory: it returns a `decorator` that wraps your function in a loop. On each failed attempt within the allowed `exceptions`, it logs the attempt and sleeps `delay` seconds before trying again; on the final failed attempt it re-raises the original exception so callers still see a real error, not a silent `None`.

```bash
# Run the built-in demo (no flags — configuration lives in the decorator call)
python3 retry_failed_operations.py

# Use it as a library in your own script
python3 -c "
from retry_failed_operations import retry

@retry(times=5, delay=2.0, exceptions=(ConnectionError,))
def flaky_call():
    ...
"
```

#### Sample Output

```
Demo: retrying an operation that fails twice before succeeding.

[retry] Attempt 1/4 for 'unstable_operation' failed: RuntimeError('simulated failure #1')
[retry] Attempt 2/4 for 'unstable_operation' failed: RuntimeError('simulated failure #2')

Final result: success
Total calls made: 3
```

#### Variations / Flags

- Add a `--dry-run` mode to your call site by wrapping the decorated function in a check that logs "would retry" instead of executing, useful when testing retry policy without hitting a real service.
- Schedule a script that uses this decorator with cron for periodic flaky jobs: `*/5 * * * * /usr/bin/python3 /opt/scripts/sync_job.py >> /var/log/sync_job.log 2>&1`.
- Swap the fixed `delay` for a callable to get per-attempt custom delays without moving to full backoff (see 18.15 if you want exponential growth).

#### Common Pitfalls

- Leaving `exceptions=(Exception,)` at its default retries on *everything*, including programmer errors like `AttributeError` — narrow it.
- Wrapping a function that has side effects on every call (e.g. "charge the customer") without idempotency guards means a retry after a false failure can duplicate the effect.
- No maximum total wall-clock time is enforced — `times=10, delay=30` will block the caller for up to five minutes; make sure calling code (e.g. a request handler) can tolerate that.

### 18.2 Timeout Execution

#### Use case

A sysadmin needs to call a function — a third-party library call, a legacy synchronous API client, a subprocess wrapper — that has no timeout parameter of its own, and must guarantee the caller isn't blocked forever if it hangs.

#### Prerequisites

Pure standard library (`threading`, `functools`). No installation required. Works on Linux, macOS, and Windows, though see the caution below about what "timeout" actually means here.

```bash
python3 --version   # 3.x, no third-party packages needed
```

#### Cautions

This does **not** kill the underlying work — Python has no safe cross-platform API to forcibly terminate a thread. The wrapped function keeps running in a background daemon thread even after `TimeoutError_` is raised to the caller. For code that must actually stop (e.g. an infinite CPU-bound loop), run it in a separate process instead (see 18.9) and terminate the process.

#### Script

```python
#!/usr/bin/env python3
"""
timeout_execution.py
--------------------
Kill/interrupt a function's execution if it runs longer than a given
threshold. Uses a background thread + a sentinel, since Python cannot
force-kill another thread; instead this races the target function
against a timer and raises TimeoutError in the caller if the timer
wins.
"""

import functools
import threading


class TimeoutError_(TimeoutError):
    pass


def timeout(seconds):
    """
    Decorator factory. Runs the wrapped function in a daemon thread and
    raises TimeoutError_ in the calling thread if it doesn't finish
    within `seconds`.
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            result = {}
            exc_holder = {}
            done = threading.Event()

            def target():
                try:
                    result["value"] = func(*args, **kwargs)
                except Exception as exc:  # noqa: BLE001
                    exc_holder["exc"] = exc
                finally:
                    done.set()

            worker = threading.Thread(target=target, daemon=True)
            worker.start()
            finished_in_time = done.wait(timeout=seconds)

            if not finished_in_time:
                raise TimeoutError_(
                    f"'{func.__name__}' did not complete within {seconds}s"
                )
            if "exc" in exc_holder:
                raise exc_holder["exc"]
            return result["value"]
        return wrapper
    return decorator


if __name__ == "__main__":
    import time

    @timeout(seconds=1)
    def fast_task():
        time.sleep(0.3)
        return "fast task finished"

    @timeout(seconds=1)
    def slow_task():
        time.sleep(3)
        return "slow task finished"  # never reached in time

    print("Demo 1: a task that finishes within the timeout")
    print(" ->", fast_task())

    print("\nDemo 2: a task that exceeds the timeout")
    try:
        slow_task()
    except TimeoutError_ as e:
        print(" -> Caught expected error:", e)
```

#### How It Works & How to Run

`@timeout(seconds)` runs the wrapped call in a daemon thread and blocks the caller on `threading.Event.wait(timeout=seconds)`. If the event fires before the timeout, the real result (or exception) is propagated back to the caller normally; if it doesn't, the caller gets `TimeoutError_` while the background thread is abandoned to finish or die on its own.

```bash
# Run the built-in demo
python3 timeout_execution.py

# Adjust the threshold inline when using as a library
python3 -c "
from timeout_execution import timeout, TimeoutError_
import time

@timeout(seconds=2)
def maybe_slow():
    time.sleep(5)

try:
    maybe_slow()
except TimeoutError_ as e:
    print('timed out:', e)
"
```

#### Sample Output

```
Demo 1: a task that finishes within the timeout
 -> fast task finished

Demo 2: a task that exceeds the timeout
 -> Caught expected error: 'slow_task' did not complete within 1s
```

#### Variations / Flags

- Lower `seconds` to a fraction (e.g. `timeout(seconds=0.5)`) for tight health-check style calls.
- Combine with 18.1's `@retry` (apply `@timeout` first, then `@retry`) to retry an operation that occasionally hangs rather than just erroring.
- For CPU-bound work that truly must stop, replace the `threading.Thread` in this pattern with a `multiprocessing.Process` and call `.terminate()` on timeout — the trade-off is process-startup overhead per call.

#### Common Pitfalls

- Assuming the background thread stops when `TimeoutError_` is raised — it doesn't, and if it eventually writes to shared state after the caller has moved on, that's a race condition to watch for.
- Daemon threads are killed abruptly when the main process exits, which can leave partially-written files or unflushed I/O if the timed-out function was mid-write.
- Very short timeouts on functions with real per-call overhead (thread creation itself takes time) can trigger false-positive timeouts under system load.

### 18.3 Progress Indicators

#### Use case

A sysadmin runs a batch job over a known number of items — migrating 10,000 rows, resizing 500 images, processing a directory of log files — and wants a live percentage bar in the terminal without pulling in `tqdm` as a dependency.

#### Prerequisites

Pure standard library (`sys`, `time`). No installation required. Works on Linux, macOS, and Windows terminals that support carriage-return (`\r`) line overwriting — most do, including standard `cmd.exe` and PowerShell.

```bash
python3 --version
```

#### Cautions

Piping output to a file or a non-TTY consumer (e.g. `python3 script.py > log.txt`) will still write the `\r`-laden progress updates into the file, producing a messy, hard-to-read log. Detect `stream.isatty()` if you need clean output in both interactive and redirected contexts.

#### Script

```python
#!/usr/bin/env python3
"""
progress_indicator.py
--------------------
A dependency-free iterable wrapper that prints a percentage-complete
progress bar as you iterate, similar in spirit to `tqdm` but with zero
external dependencies.
"""

import sys
import time


def progress(iterable, label="Progress", bar_width=30, stream=sys.stdout):
    """
    Wrap any sized iterable. Prints a live-updating progress bar of the
    form: "Label [###------] 40% (4/10)" and a newline once complete.
    """
    try:
        total = len(iterable)
    except TypeError:
        iterable = list(iterable)
        total = len(iterable)

    if total == 0:
        stream.write(f"{label}: nothing to do\n")
        return

    for index, item in enumerate(iterable, start=1):
        yield item
        pct = index / total
        filled = int(bar_width * pct)
        bar = "#" * filled + "-" * (bar_width - filled)
        stream.write(f"\r{label} [{bar}] {pct*100:5.1f}% ({index}/{total})")
        stream.flush()
        if index == total:
            stream.write("\n")


if __name__ == "__main__":
    print("Demo: iterating over 20 items with a live progress bar\n")
    for _ in progress(range(20), label="Working"):
        time.sleep(0.05)
    print("Done.")
```

#### How It Works & How to Run

`progress()` is a generator that wraps any sized iterable (or consumes it into a list if it lacks `len()`, e.g. a plain generator). Each time it yields the next item to your `for` loop, it also overwrites the current terminal line with an updated bar using `\r` plus `stream.flush()`, so the bar animates in place rather than scrolling.

```bash
# Run the built-in demo
python3 progress_indicator.py

# Typical library usage over a real batch job
python3 -c "
from progress_indicator import progress
import time

files = ['a.txt', 'b.txt', 'c.txt']
for f in progress(files, label='Processing'):
    time.sleep(0.2)  # stand-in for real work
"
```

#### Sample Output

```
Demo: iterating over 20 items with a live progress bar

Working [#######################] 100.0% (20/20)
Done.
```
*(In a real terminal, the bar overwrites itself in place as it climbs from 0% to 100%; the snippet above shows only the final resting line.)*

#### Variations / Flags

- Pass `bar_width=50` for a wider bar on modern terminals, or `bar_width=10` for narrow CI log windows.
- Guard for redirected output: `stream = sys.stdout if sys.stdout.isatty() else open(os.devnull, "w")` to silence the bar entirely when logging to a file.
- Wrap it around `os.walk()` results or a database cursor's `fetchall()` for file-migration and ETL-style scripts.

#### Common Pitfalls

- Calling `len()` implicitly assumes the iterable is sized or fully materializable in memory — passing a huge lazy generator will force it into a list, which can blow up memory for very large datasets.
- Interleaving other `print()` calls inside the loop body will break the in-place bar rendering, since those writes don't start with `\r` and will land in the middle of the bar line.
- Not calling `stream.flush()` (if you customize this) can leave the bar looking frozen on some buffered terminals.

### 18.4 Spinner Animations

#### Use case

A sysadmin kicks off a task with no known total — waiting for a remote server to come up, waiting on an external API with no progress signal, waiting for a lock — and wants a simple "still working" animation instead of a silent, possibly-frozen-looking terminal.

#### Prerequisites

Pure standard library (`threading`, `itertools`, `sys`, `time`). No installation required. Works on Linux, macOS, and Windows terminals supporting `\r` line overwriting.

```bash
python3 --version
```

#### Cautions

The spinner runs on a background daemon thread; if your main code also writes to `stdout` while the spinner is active, output will interleave and corrupt the animation. Stop the spinner (or use it only around silent work) before printing anything else.

#### Script

```python
#!/usr/bin/env python3
"""
spinner_animation.py
--------------------
An animated spinner for tasks with an unknown duration. Runs on a
background thread so it can animate while your main code does its
work, and can also be used as a context manager.
"""

import itertools
import sys
import threading
import time


class Spinner:
    FRAMES = ["|", "/", "-", "\\"]

    def __init__(self, message="Working", interval=0.1, stream=sys.stdout):
        self.message = message
        self.interval = interval
        self.stream = stream
        self._stop_event = threading.Event()
        self._thread = None

    def _spin(self):
        for frame in itertools.cycle(self.FRAMES):
            if self._stop_event.is_set():
                break
            self.stream.write(f"\r{self.message}... {frame}")
            self.stream.flush()
            time.sleep(self.interval)

    def start(self):
        self._stop_event.clear()
        self._thread = threading.Thread(target=self._spin, daemon=True)
        self._thread.start()
        return self

    def stop(self, final_message="done"):
        self._stop_event.set()
        if self._thread is not None:
            self._thread.join()
        clear_len = len(self.message) + len(final_message) + 8
        self.stream.write("\r" + " " * clear_len + "\r")
        self.stream.write(f"{self.message}... {final_message}\n")
        self.stream.flush()

    def __enter__(self):
        self.start()
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.stop("failed" if exc_type else "done")
        return False  # don't suppress exceptions


if __name__ == "__main__":
    print("Demo: spinner running for 2 seconds via context manager\n")
    with Spinner("Waiting for indeterminate task"):
        time.sleep(2)
    print("Demo finished.")
```

#### How It Works & How to Run

`Spinner.start()` launches a daemon thread that cycles through `FRAMES` and rewrites the current line with `\r` every `interval` seconds. `Spinner.stop()` sets a `threading.Event`, joins the thread so it's fully stopped before returning, and prints a final resting message (`done` or `failed`). Used as a context manager, entering the `with` block starts the spinner and exiting it stops the spinner automatically — printing `failed` if the block raised.

```bash
# Run the built-in demo
python3 spinner_animation.py

# Manual start/stop instead of context manager
python3 -c "
from spinner_animation import Spinner
import time

s = Spinner('Connecting to server')
s.start()
time.sleep(3)
s.stop()
"
```

#### Sample Output

```
Demo: spinner running for 2 seconds via context manager

Waiting for indeterminate task... done
Demo finished.
```
*(In a live terminal, the frame character `|`, `/`, `-`, `\` cycles in place for the 2-second duration before settling on the final `done` line.)*

#### Variations / Flags

- Pass `interval=0.05` for a faster spin on responsive terminals, or `interval=0.3` to reduce CPU wake-ups on constrained systems.
- Call `.stop("failed")` explicitly from an `except` block if you're using manual `start()`/`stop()` rather than the context manager, to signal failure in the final line.
- Swap `FRAMES` for a different character set (e.g. Braille dot spinners `"⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏"`) for a more modern look on UTF-8 terminals.

#### Common Pitfalls

- Printing from the main thread while the spinner is running garbles the line — stop the spinner first, or buffer your own output until after `.stop()`.
- Non-TTY output (redirected to a file or CI log) will fill the log with repeated `\r`-separated frames; check `stream.isatty()` and skip the spinner (falling back to a single "working..." line) when it's `False`.
- Forgetting to call `.stop()` on a manually-started spinner leaves the daemon thread spinning silently until the process exits — harmless but wasteful; prefer the context-manager form.

### 18.5 Retry Decorators

#### Use case

A sysadmin needs a retry that works across both sync and async codebases (common in modern services that mix blocking scripts with `asyncio` workers), and sometimes needs to retry based on a *bad result* — e.g. polling a status endpoint until it says `"ready"` — not just on a raised exception.

#### Prerequisites

Pure standard library (`asyncio`, `functools`, `inspect`, `time`). No installation required. The async path needs Python 3.5+ for `async`/`await` syntax; Python 3.7+ recommended.

```bash
python3 --version
```

#### Cautions

`result_ok` is only checked when the call *doesn't* raise — if the function both raises sometimes and returns bad values other times, make sure both `exceptions` and `result_ok` are set, or one failure mode will slip through unretried.

#### Script

```python
#!/usr/bin/env python3
"""
retry_decorator.py
--------------------
A more configurable @retry annotation: supports both sync and async
functions, a custom on_failure callback per attempt, and an optional
predicate to decide whether a *result* (not just an exception) counts
as a failure worth retrying.
"""

import asyncio
import functools
import inspect
import time


def retry(times=3, delay=1.0, exceptions=(Exception,), result_ok=None, on_failure=None):
    """
    Decorator factory usable on both sync and async callables.

    times       -- max attempts
    delay       -- seconds to wait between attempts
    exceptions  -- exception types that trigger a retry
    result_ok   -- optional callable(result) -> bool; if it returns
                   False, the result is treated as a failure and retried
                   even though no exception was raised
    on_failure  -- optional callable(attempt, exc_or_result) called after
                   each failed attempt, useful for logging/metrics
    """
    def decorator(func):
        if inspect.iscoroutinefunction(func):
            @functools.wraps(func)
            async def async_wrapper(*args, **kwargs):
                last_exc = None
                for attempt in range(1, times + 1):
                    try:
                        result = await func(*args, **kwargs)
                        if result_ok is not None and not result_ok(result):
                            if on_failure:
                                on_failure(attempt, result)
                            if attempt < times:
                                await asyncio.sleep(delay)
                            continue
                        return result
                    except exceptions as exc:
                        last_exc = exc
                        if on_failure:
                            on_failure(attempt, exc)
                        if attempt < times:
                            await asyncio.sleep(delay)
                if last_exc:
                    raise last_exc
                raise RuntimeError(f"'{func.__name__}' never returned an acceptable result")
            return async_wrapper

        @functools.wraps(func)
        def sync_wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(1, times + 1):
                try:
                    result = func(*args, **kwargs)
                    if result_ok is not None and not result_ok(result):
                        if on_failure:
                            on_failure(attempt, result)
                        if attempt < times:
                            time.sleep(delay)
                        continue
                    return result
                except exceptions as exc:
                    last_exc = exc
                    if on_failure:
                        on_failure(attempt, exc)
                    if attempt < times:
                        time.sleep(delay)
            if last_exc:
                raise last_exc
            raise RuntimeError(f"'{func.__name__}' never returned an acceptable result")
        return sync_wrapper
    return decorator


if __name__ == "__main__":
    def log_failure(attempt, info):
        print(f"[retry_decorator] attempt {attempt} did not succeed: {info!r}")

    counter = {"n": 0}

    @retry(times=4, delay=0.2, result_ok=lambda r: r == "ready", on_failure=log_failure)
    def poll_status():
        counter["n"] += 1
        return "pending" if counter["n"] < 3 else "ready"

    print("Demo 1 (sync, result-based retry): polling until status == 'ready'\n")
    print("Result:", poll_status())

    async def async_demo():
        attempts = {"n": 0}

        @retry(times=3, delay=0.2, exceptions=(ValueError,), on_failure=log_failure)
        async def flaky_async_call():
            attempts["n"] += 1
            if attempts["n"] < 2:
                raise ValueError("simulated async failure")
            return "async success"

        return await flaky_async_call()

    print("\nDemo 2 (async, exception-based retry):\n")
    result = asyncio.run(async_demo())
    print("Result:", result)
```

#### How It Works & How to Run

`retry()` inspects the wrapped function with `inspect.iscoroutinefunction()` at decoration time and returns either an `async_wrapper` (using `await` and `asyncio.sleep`) or a `sync_wrapper` (using plain blocking `time.sleep`) — the calling code doesn't need to know or care which path was chosen. On each attempt, if `result_ok` is provided and returns `False` for the returned value, that counts as a failure and the loop retries exactly as it would for a caught exception.

```bash
# Run both the sync and async demos
python3 retry_decorator.py

# Sync result-based retry as a library
python3 -c "
from retry_decorator import retry

@retry(times=5, delay=1.0, result_ok=lambda r: r is not None)
def fetch_maybe_none():
    ...
"
```

#### Sample Output

```
Demo 1 (sync, result-based retry): polling until status == 'ready'

[retry_decorator] attempt 1 did not succeed: 'pending'
[retry_decorator] attempt 2 did not succeed: 'pending'
Result: ready

Demo 2 (async, exception-based retry):

[retry_decorator] attempt 1 did not succeed: ValueError('simulated async failure')
Result: async success
```

#### Variations / Flags

- Combine `exceptions` and `result_ok` together to retry on both raised errors and "successful but wrong" responses in the same decorator.
- Use `on_failure` to push attempt counts into a metrics system (e.g. StatsD, Prometheus pushgateway) instead of just printing.
- For long-running polling loops, raise `delay` and lower `times`, or switch to `retry_with_backoff` (18.15) so polling interval grows instead of hammering the endpoint at a fixed cadence.

#### Common Pitfalls

- Forgetting that `result_ok=None` (the default) disables result-based checking entirely — a function that "succeeds" but returns garbage will not be retried unless you supply a predicate.
- Mixing sync and async incorrectly: calling the async-wrapped version without `await` returns an unresolved coroutine object instead of the real result, silently.
- If `times=1`, no retry ever actually happens and the decorator just adds overhead — make sure `times` is intentionally set above 1.

### 18.6 Rate Limiting

#### Use case

A sysadmin's script calls a third-party API with a hard rate limit (e.g. "5 requests per second") and needs to stay under it automatically — without manually calculating sleep intervals or risking a burst that trips the provider's throttle.

#### Prerequisites

Pure standard library (`functools`, `threading`, `time`). No installation required. Thread-safe by design, so it's also safe to share a single `TokenBucket` across multiple worker threads.

```bash
python3 --version
```

#### Cautions

`TokenBucket` limits *your process's* call rate, not the API's actual quota if other processes or machines are also calling it — for a rate limit shared across multiple hosts, you need a centralized limiter (e.g. Redis-backed) instead of this in-process one.

#### Script

```python
#!/usr/bin/env python3
"""
rate_limiting.py
--------------------
A token-bucket rate limiter to throttle API calls (or any repeated
operation) to at most `rate` events per `per_seconds`, while still
allowing short bursts up to the bucket's capacity.
"""

import functools
import threading
import time


class TokenBucket:
    def __init__(self, rate, per_seconds=1.0, capacity=None):
        """
        rate        -- tokens added per `per_seconds`
        per_seconds -- time window the rate applies to
        capacity    -- max tokens the bucket can hold (defaults to `rate`,
                       i.e. no more than one window's worth of burst)
        """
        self.rate = rate
        self.per_seconds = per_seconds
        self.capacity = capacity if capacity is not None else rate
        self._tokens = float(self.capacity)
        self._lock = threading.Lock()
        self._last_refill = time.monotonic()

    def _refill(self):
        now = time.monotonic()
        elapsed = now - self._last_refill
        added = elapsed * (self.rate / self.per_seconds)
        if added > 0:
            self._tokens = min(self.capacity, self._tokens + added)
            self._last_refill = now

    def try_acquire(self, tokens=1):
        """Non-blocking: returns True if tokens were available and consumed."""
        with self._lock:
            self._refill()
            if self._tokens >= tokens:
                self._tokens -= tokens
                return True
            return False

    def acquire(self, tokens=1):
        """Blocking: waits until enough tokens are available, then consumes them."""
        while True:
            with self._lock:
                self._refill()
                if self._tokens >= tokens:
                    self._tokens -= tokens
                    return
                deficit = tokens - self._tokens
                wait_time = deficit / (self.rate / self.per_seconds)
            time.sleep(max(wait_time, 0.001))


def rate_limited(rate, per_seconds=1.0, capacity=None):
    """Decorator: throttle calls to a function using a shared TokenBucket."""
    bucket = TokenBucket(rate=rate, per_seconds=per_seconds, capacity=capacity)

    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            bucket.acquire()
            return func(*args, **kwargs)
        return wrapper
    return decorator


if __name__ == "__main__":
    print("Demo: allowing 3 calls/second, firing 8 calls back-to-back\n")

    @rate_limited(rate=3, per_seconds=1.0)
    def call_api(n):
        print(f"  call {n} executed at t={time.monotonic() - start:.2f}s")

    start = time.monotonic()
    for i in range(1, 9):
        call_api(i)
    print("\nNotice calls are spaced out once the initial burst capacity is used.")
```

#### How It Works & How to Run

`TokenBucket` starts full (`capacity` tokens) and refills continuously based on elapsed wall-clock time (`rate / per_seconds` tokens per second). `acquire()` blocks, sleeping and re-checking, until enough tokens exist, then consumes them; `try_acquire()` is the non-blocking variant for code that wants to skip or queue rather than wait. `rate_limited()` wraps a function with a shared bucket so every call blocks until a token is free.

#### Variations / Flags

- Set `capacity` higher than `rate` (e.g. `TokenBucket(rate=5, per_seconds=1.0, capacity=20)`) to allow larger bursts followed by a steady-state rate.
- Use `try_acquire()` inside a queue-and-drain loop to build a "best effort, drop if over limit" pattern instead of blocking.
- Share one `TokenBucket` instance across multiple `ThreadPoolExecutor` workers (18.7) to enforce a global rate limit across concurrent callers.

#### Common Pitfalls

- Creating a *new* `TokenBucket` per call (instead of one shared instance) defeats the whole purpose — the limiter must persist across calls to track consumed tokens.
- `acquire()` uses `time.sleep()` in a loop, which blocks the calling thread; in `asyncio` code, use an async-native limiter instead or run this in a thread pool via `loop.run_in_executor`.
- A very small `capacity` relative to `rate` can make the limiter feel stricter than intended, since it removes the ability to burst even briefly.

### 18.7 Parallel Execution

#### Use case

A sysadmin needs to fetch 50 URLs, ping 200 hosts, or read a directory of files, and doing it one-at-a-time sequentially wastes most of the time waiting on I/O rather than doing useful work.

#### Prerequisites

Pure standard library (`concurrent.futures`). No installation required. Best suited to I/O-bound work; for CPU-bound work, see 18.9 (multiprocessing) instead, since threads can't run CPU-bound Python bytecode in parallel due to the GIL.

```bash
python3 --version
```

#### Cautions

Exceptions inside `func(item)` propagate by default and abort the whole batch unless you pass `on_error`. For long batches against unreliable endpoints, always supply an `on_error` handler so one bad item doesn't discard results already computed for the others.

#### Script

```python
#!/usr/bin/env python3
"""
parallel_execution.py
--------------------
Map a function over an iterable concurrently using
concurrent.futures.ThreadPoolExecutor -- ideal for I/O-bound work
(network calls, file I/O) where threads spend most of their time
waiting rather than computing.
"""

import time
from concurrent.futures import ThreadPoolExecutor, as_completed


def parallel_map(func, items, max_workers=4, on_error=None):
    """
    Run `func(item)` for every item in `items` concurrently.
    Returns a list of results in the SAME ORDER as the input items.
    If a call raises and `on_error` is given, `on_error(item, exc)` is
    called and that slot is filled with None; otherwise the exception
    propagates.
    """
    results = [None] * len(items)
    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        future_to_index = {
            executor.submit(func, item): idx for idx, item in enumerate(items)
        }
        for future in as_completed(future_to_index):
            idx = future_to_index[future]
            try:
                results[idx] = future.result()
            except Exception as exc:
                if on_error is not None:
                    on_error(items[idx], exc)
                    results[idx] = None
                else:
                    raise
    return results


if __name__ == "__main__":
    def simulated_io_task(n):
        time.sleep(0.3)  # pretend this is a network request
        return n * n

    items = list(range(1, 9))

    print(f"Demo: squaring {len(items)} numbers, each simulating 0.3s of I/O\n")
    start = time.monotonic()
    results = parallel_map(simulated_io_task, items, max_workers=8)
    elapsed = time.monotonic() - start

    print(f"Input:   {items}")
    print(f"Results: {results}")
    print(f"\nElapsed: {elapsed:.2f}s (sequential would take ~{0.3*len(items):.1f}s)")
```

#### How It Works & How to Run

`parallel_map` submits every item to a `ThreadPoolExecutor` up front, then drains completed futures with `as_completed()` (which yields futures as they finish, not in submission order) while writing each result back into its original index so the returned list matches input order regardless of completion order.

```bash

python3 -c "

def fetch(url):
    return urllib.request.urlopen(url, timeout=5).status

urls = ['https://example.com'] * 5
print(parallel_map(fetch, urls, max_workers=5))
"
```

#### Sample Output

```
Demo: squaring 8 numbers, each simulating 0.3s of I/O

Input:   [1, 2, 3, 4, 5, 6, 7, 8]
Results: [1, 4, 9, 16, 25, 36, 49, 64]

Elapsed: 0.30s (sequential would take ~2.4s)
```

#### Variations / Flags

- Raise `max_workers` for highly I/O-bound, low-CPU tasks (e.g. `max_workers=50` for simple HTTP GETs) — threads are cheap relative to processes.
- Pass `on_error=lambda item, exc: log.warning(f"{item} failed: {exc}")` to keep processing the rest of the batch and inspect failures afterward.
- Combine with 18.6's `TokenBucket` inside `func` to rate-limit concurrent calls against an API with a strict quota.

#### Common Pitfalls

- Setting `max_workers` too high against a target service can look like a denial-of-service attack from that service's perspective — respect published rate limits.
- CPU-bound `func` (heavy computation, not I/O) will *not* speed up with threads due to the GIL — use `multiprocessing_utils.py` (18.9) instead.
- Not handling `on_error` means a single failing item raises and discards all other completed results in that batch, even ones that succeeded.

### 18.8 Threading

#### Use case

A sysadmin needs finer control than `ThreadPoolExecutor` offers: a counter that's safe to increment from many threads at once (e.g. tallying processed records), or a manual producer/consumer worker pool for a long-running pipeline where items may arrive continuously rather than as one fixed batch.

#### Prerequisites

Pure standard library (`queue`, `threading`, `time`). No installation required. Standard on all platforms Python supports.

```bash
python3 --version
```

#### Cautions

`SafeCounter` protects only the counter itself — it does not make any *other* shared state you touch inside worker threads safe. Every additional piece of shared mutable state needs its own lock (or should be avoided in favor of returning values through the `Queue`).

#### Script

```python
#!/usr/bin/env python3
"""
threading_utils.py
--------------------
Helpers for handling I/O-bound concurrent workloads safely with the
`threading` module directly. Includes a thread-safe Counter using a
Lock, and a worker-pool pattern using a Queue for producer/consumer
workloads.
"""

import queue
import threading
import time


class SafeCounter:
    """A counter safe to increment from multiple threads at once."""

    def __init__(self, start=0):
        self._value = start
        self._lock = threading.Lock()

    def increment(self, amount=1):
        with self._lock:
            self._value += amount
            return self._value

    @property
    def value(self):
        with self._lock:
            return self._value


def run_worker_pool(worker_fn, items, num_workers=4):
    """
    Classic producer/consumer pattern: put all `items` on a Queue, then
    start `num_workers` threads that each pull items off the queue and
    call `worker_fn(item)` until the queue is empty. Blocks until all
    items are processed.
    """
    q = queue.Queue()
    for item in items:
        q.put(item)

    def worker():
        while True:
            try:
                item = q.get_nowait()
            except queue.Empty:
                return
            try:
                worker_fn(item)
            finally:
                q.task_done()

    threads = [threading.Thread(target=worker, daemon=True) for _ in range(num_workers)]
    for t in threads:
        t.start()
    q.join()
    for t in threads:
        t.join()


if __name__ == "__main__":
    print("Demo 1: SafeCounter incremented from 10 threads, 1000 times each\n")
    counter = SafeCounter()

    def bump():
        for _ in range(1000):
            counter.increment()

    threads = [threading.Thread(target=bump) for _ in range(10)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()

    print(f"Expected: 10000, Actual: {counter.value}  "
          f"({'correct -- no race condition' if counter.value == 10000 else 'MISMATCH'})")

    print("\nDemo 2: worker pool processing 10 items with 3 workers\n")
    processed = SafeCounter()

    def process_item(item):
        time.sleep(0.05)
        processed.increment()
        print(f"  processed item {item} on {threading.current_thread().name}")

    run_worker_pool(process_item, list(range(1, 11)), num_workers=3)
    print(f"\nTotal processed: {processed.value}")
```

#### How It Works & How to Run

`SafeCounter` wraps a plain integer with a `threading.Lock`, so `increment()` is atomic even when called concurrently — without the lock, `+=` is not atomic in Python and concurrent increments can silently lose updates. `run_worker_pool` loads every item onto a thread-safe `queue.Queue`, starts `num_workers` daemon threads that each drain the queue with `get_nowait()` until it's empty, and blocks the caller with `q.join()` until every item has been marked done via `task_done()`.

```bash

python3 -c "

def send_email(addr):
    print('sending to', addr)

run_worker_pool(send_email, ['a@x.com', 'b@x.com', 'c@x.com'], num_workers=2)
"
```

#### Sample Output

```
Demo 1: SafeCounter incremented from 10 threads, 1000 times each

Expected: 10000, Actual: 10000  (correct -- no race condition)

Demo 2: worker pool processing 10 items with 3 workers

  processed item 1 on Thread-1 (worker)
  processed item 2 on Thread-2 (worker)
  processed item 3 on Thread-3 (worker)
  processed item 4 on Thread-1 (worker)
  ...
Total processed: 10
```
*(Exact thread names and interleaving order vary between runs — that's expected with concurrent threads.)*

#### Variations / Flags

- Increase `num_workers` for more I/O-bound parallelism, matching the same "more threads than CPU cores is fine" guidance as `parallel_map` (18.7).
- Feed `run_worker_pool`'s queue continuously from a separate producer thread (rather than pre-filling it) for a true streaming pipeline — just don't call `q.join()` until you've signaled the producer is done.
- Use `SafeCounter` anywhere you need a running total across threads: bytes downloaded, records migrated, errors seen.

#### Common Pitfalls

- Forgetting the lock and incrementing a plain `int` from multiple threads is a classic race condition — it usually "mostly works" in casual testing and then silently under-counts in production under real concurrency.
- `run_worker_pool` blocks until *all* items are processed; if `worker_fn` can hang, combine it with the `@timeout` decorator from 18.2 inside `worker_fn` to bound each item's processing time.
- Daemon threads don't prevent the process from exiting — if the main thread crashes before `q.join()` returns, in-flight work is lost silently.

### 18.9 Multiprocessing

#### Use case

A sysadmin needs to crunch genuinely CPU-heavy work — image resizing, hashing large files, numeric computation — across multiple CPU cores. Threads can't help here because of the GIL; this needs real OS processes.

#### Prerequisites

Pure standard library (`multiprocessing`). No installation required. The worker function **must** be importable at module level (not defined inside `if __name__ == "__main__":` or as a lambda/closure), because `multiprocessing` pickles it to send to child processes. On Windows, scripts using `multiprocessing.Pool` must guard the entry point with `if __name__ == "__main__":`, which the script below already does.

```bash
python3 --version
python3 -c "from multiprocessing import cpu_count; print(cpu_count(), 'cores available')"
```

#### Cautions

Each worker process has its own memory space — arguments and return values are pickled and copied between processes, so this has real overhead per call and is only worth it when the per-item work is substantial (milliseconds or more, not microseconds). Also, spawning many processes consumes real OS resources; don't set `workers` far beyond `cpu_count()`.

#### Script

```python
#!/usr/bin/env python3
"""
multiprocessing_utils.py
--------------------
Parallelize CPU-bound work across real OS processes using
`multiprocessing`, bypassing the GIL entirely.
"""

import time
from multiprocessing import Pool, cpu_count


def parallel_map_processes(func, items, workers=None):
    """
    Run `func(item)` for every item in `items`, distributed across a
    process pool. Returns results in the same order as the input.
    `workers` defaults to the machine's CPU count.
    """
    workers = workers or cpu_count()
    with Pool(processes=workers) as pool:
        return pool.map(func, items)


def _cpu_bound_work(n):
    """A deliberately expensive pure-CPU computation for the demo."""
    total = 0
    for i in range(n):
        total += i * i
    return total


if __name__ == "__main__":
    items = [3_000_000] * 8

    print(f"Demo: {len(items)} CPU-bound tasks (summing squares up to {items[0]:,})")
    print(f"Machine has {cpu_count()} CPU core(s) available.\n")

    start = time.monotonic()
    sequential_results = [_cpu_bound_work(n) for n in items]
    sequential_time = time.monotonic() - start
    print(f"Sequential: {sequential_time:.2f}s")

    start = time.monotonic()
    parallel_results = parallel_map_processes(_cpu_bound_work, items)
    parallel_time = time.monotonic() - start
    print(f"Multiprocessing (Pool): {parallel_time:.2f}s")

    assert sequential_results == parallel_results, "results should match"
    speedup = sequential_time / parallel_time if parallel_time > 0 else float("inf")
    print(f"\nResults match. Speedup: {speedup:.2f}x "
          f"(actual speedup depends on available CPU cores)")
```

#### How It Works & How to Run

`parallel_map_processes` opens a `multiprocessing.Pool` with `workers` (or all CPU cores if unspecified) and distributes `items` across them via `pool.map`, which handles pickling arguments to each child process and collecting results back in input order. The `with Pool(...)` block ensures workers are cleanly terminated afterward.

```bash
# Run the built-in demo (compares sequential vs. process-pool timing)
python3 multiprocessing_utils.py

# Use a fixed worker count instead of all cores
python3 -c "
from multiprocessing_utils import parallel_map_processes, _cpu_bound_work
print(parallel_map_processes(_cpu_bound_work, [1_000_000]*4, workers=2))
"
```

#### Sample Output

```
Demo: 8 CPU-bound tasks (summing squares up to 3,000,000)
Machine has 8 CPU core(s) available.

Sequential: 1.86s
Multiprocessing (Pool): 0.41s

Results match. Speedup: 4.54x (actual speedup depends on available CPU cores)
```
*(On a single-core machine or container, speedup will be close to 1x or even slightly negative due to process-spawn overhead — the pattern is still correct, the hardware just isn't parallel.)*

#### Variations / Flags

- Set `workers=cpu_count() - 1` to leave one core free for the OS and other processes on a shared machine.
- Use `pool.imap()` instead of `pool.map()` inside a custom version if you want to start processing results as they complete rather than waiting for the whole batch.
- For very large item lists, pass `chunksize` to `pool.map(func, items, chunksize=N)` to reduce inter-process communication overhead.

#### Common Pitfalls

- Defining the worker function as a local closure or lambda breaks pickling with a `PicklingError` — it must be a top-level, importable function, as `_cpu_bound_work` is here.
- On Windows (and when using the `spawn` start method generally), forgetting the `if __name__ == "__main__":` guard causes the child processes to re-import and re-execute the whole script recursively.
- Using multiprocessing for I/O-bound work wastes the overhead of process creation for no benefit — prefer threads (18.7, 18.8) or asyncio (18.10) for I/O-bound tasks instead.

### 18.10 Async Programming

#### Use case

A sysadmin's script already uses `asyncio` (e.g. an async HTTP client like `aiohttp` or `httpx`) and needs to fire off many coroutines concurrently while capping how many run at once — `asyncio.gather` alone has no built-in concurrency limit, which matters against a rate-limited API.

#### Prerequisites

Pure standard library (`asyncio`, `time`). No installation required. Python 3.7+ recommended for `asyncio.run()`.

```bash
python3 --version
```

#### Cautions

`gather_limited` still schedules *all* coroutines up front (they're created as objects immediately); the semaphore only limits how many run concurrently, not how many exist in memory at once. For an extremely large number of items, generate coroutines lazily instead of building the full list first.

#### Script

```python
#!/usr/bin/env python3
"""
async_utils.py
--------------------
Non-blocking concurrency helpers built on `asyncio`. Includes a
gather-with-concurrency-limit helper (asyncio.gather alone has no
built-in way to cap how many coroutines run at once).
"""

import asyncio
import time


async def gather_limited(coros, limit=5):
    """
    Run a list of coroutines concurrently, but never more than `limit`
    at once. Returns results in the same order as the input list.
    """
    semaphore = asyncio.Semaphore(limit)

    async def run_with_limit(coro):
        async with semaphore:
            return await coro

    return await asyncio.gather(*(run_with_limit(c) for c in coros))


async def with_timeout(coro, seconds):
    """
    Await a coroutine but cancel it and raise asyncio.TimeoutError if it
    doesn't finish within `seconds`. Thin, explicit wrapper around
    asyncio.wait_for for readability in calling code.
    """
    return await asyncio.wait_for(coro, timeout=seconds)


if __name__ == "__main__":
    async def fake_network_call(n, duration=0.5):
        await asyncio.sleep(duration)
        return f"result-{n}"

    async def main():
        print("Demo 1: 10 'network calls' with concurrency capped at 3\n")
        start = time.monotonic()
        coros = [fake_network_call(i) for i in range(10)]
        results = await gather_limited(coros, limit=3)
        elapsed = time.monotonic() - start
        print(f"Results: {results}")
        print(f"Elapsed: {elapsed:.2f}s "
              f"(unlimited concurrency would take ~0.5s, "
              f"fully sequential would take ~{0.5*10:.1f}s)\n")

        print("Demo 2: a coroutine that exceeds its timeout\n")
        try:
            await with_timeout(fake_network_call("slow", duration=2), seconds=0.5)
        except asyncio.TimeoutError:
            print("Caught expected asyncio.TimeoutError")

    asyncio.run(main())
```

#### How It Works & How to Run

`gather_limited` wraps each coroutine in a helper that first acquires an `asyncio.Semaphore(limit)` before awaiting it, so at most `limit` coroutines are ever running concurrently at once, then feeds all the wrapped coroutines to `asyncio.gather()`, which preserves input order in the returned results list. `with_timeout` is a thin, self-documenting wrapper over `asyncio.wait_for`.

```bash
    async with session.get(url) as resp:
        return resp.status

# (assuming an aiohttp ClientSession named 'session' and a list of 'urls')
# asyncio.run(gather_limited([fetch(session, u) for u in urls], limit=10))
"
```

#### Sample Output

```

Results: ['result-0', 'result-1', 'result-2', 'result-3', 'result-4', 'result-5', 'result-6', 'result-7', 'result-8', 'result-9']
Elapsed: 2.00s (unlimited concurrency would take ~0.5s, fully sequential would take ~5.0s)

Demo 2: a coroutine that exceeds its timeout

Caught expected asyncio.TimeoutError
```

#### Variations / Flags

- Lower `limit` to match an API's documented concurrent-connection cap (e.g. `limit=2` for a strict provider).
- Combine `with_timeout` around each individual coroutine inside `gather_limited` for per-item timeouts, not just a timeout on the whole batch.
- Wrap `gather_limited` results with `return_exceptions=True`-style handling (pass exceptions through instead of raising) by adding a `try/except` inside `run_with_limit` if one failing call shouldn't cancel the rest.

#### Common Pitfalls

- Calling a coroutine function without `await` (or without passing it through `asyncio.run`/`gather`) creates a coroutine object that never actually runs — a common source of "nothing happened" bugs.
- Mixing blocking calls (e.g. `time.sleep()`, `requests.get()`) inside an `async def` function blocks the entire event loop, defeating the purpose of async — use `asyncio.sleep()` and an async HTTP client instead.
- `asyncio.wait_for`'s cancellation on timeout doesn't guarantee the underlying task stops instantly if it's not cooperating with cancellation (e.g. it's stuck in blocking code) — the coroutine must actually `await` something to be cancellable.

### 18.11 Job Queues

#### Use case

A sysadmin needs to hand work off between processes — a web app enqueues "send this email" jobs, a separate worker process dequeues and sends them — without standing up the full complexity of Celery or RabbitMQ for what is, at its core, a durable FIFO list.

#### Prerequisites

Requires a running Redis server and the `redis` Python package for real (multi-process, persistent) use. Without `redis` installed, the script still runs its demo using a single-process in-memory fallback so you can see the API, but that fallback is **not** a substitute for real Redis in production.

```bash
# Install the client library
pip install redis

# Install and start a local Redis server (Debian/Ubuntu)
sudo apt-get install redis-server
sudo systemctl start redis-server

# macOS (Homebrew)
brew install redis
brew services start redis

# Verify it's reachable
redis-cli ping   # should print PONG
```

#### Cautions

`RedisJobQueue.dequeue(timeout=0)` blocks the calling thread/process forever if the queue stays empty — make sure the worker process is expected to run indefinitely (e.g. as a systemd service or supervisor-managed process), not as a one-shot script, or set a nonzero timeout and handle the `None` return.

#### Script

```python
#!/usr/bin/env python3
"""
job_queue.py
--------------------
Minimal task-queue orchestration helpers for Redis-backed job queues.
Provides an enqueue/dequeue API that mirrors what you'd do with
RabbitMQ (via `pika`) or Celery, but without those heavier
dependencies.
"""

import json
import time
import queue as _queue

try:
    import redis  # type: ignore
    REDIS_AVAILABLE = True
except ImportError:
    REDIS_AVAILABLE = False


class RedisJobQueue:
    """A simple FIFO job queue backed by a Redis list (LPUSH/BRPOP)."""

    def __init__(self, queue_name="default", redis_url="redis://localhost:6379/0"):
        if not REDIS_AVAILABLE:
            raise RuntimeError(
                "The 'redis' package is not installed. Run: pip install redis"
            )
        self.queue_name = queue_name
        self.client = redis.from_url(redis_url)

    def enqueue(self, job: dict):
        """Push a JSON-serializable job onto the queue."""
        self.client.lpush(self.queue_name, json.dumps(job))

    def dequeue(self, timeout=0):
        """
        Block up to `timeout` seconds (0 = wait forever) for a job.
        Returns the deserialized job dict, or None if timed out.
        """
        result = self.client.brpop(self.queue_name, timeout=timeout)
        if result is None:
            return None
        _, raw = result
        return json.loads(raw)

    def queue_length(self):
        return self.client.llen(self.queue_name)


class InMemoryJobQueue:
    """
    Single-process fallback with the same enqueue/dequeue API, used only
    when `redis` isn't installed. Not shared across processes -- use
    RedisJobQueue for anything real.
    """

    def __init__(self, queue_name="default"):
        self.queue_name = queue_name
        self._q = _queue.Queue()

    def enqueue(self, job: dict):
        self._q.put(json.dumps(job))

    def dequeue(self, timeout=0):
        try:
            raw = self._q.get(timeout=timeout if timeout else None)
        except _queue.Empty:
            return None
        return json.loads(raw)

    def queue_length(self):
        return self._q.qsize()


def get_job_queue(queue_name="default", redis_url="redis://localhost:6379/0"):
    """Factory: returns a real RedisJobQueue if redis is available and
    reachable, otherwise falls back to InMemoryJobQueue with a warning."""
    if REDIS_AVAILABLE:
        try:
            q = RedisJobQueue(queue_name, redis_url)
            q.client.ping()
            return q
        except Exception as exc:
            print(f"[job_queue] Could not reach Redis at {redis_url} ({exc}). "
                  f"Falling back to in-memory queue.")
    else:
        print("[job_queue] 'redis' package not installed. "
              "Falling back to in-memory queue. Run: pip install redis")
    return InMemoryJobQueue(queue_name)


if __name__ == "__main__":
    print("Demo: enqueue 5 jobs, then process them via dequeue()\n")

    q = get_job_queue(queue_name="demo-jobs")

    for i in range(1, 6):
        q.enqueue({"job_id": i, "task": "send_report", "created": time.time()})
    print(f"Enqueued 5 jobs. Queue length: {q.queue_length()}\n")

    processed = 0
    while True:
        job = q.dequeue(timeout=1)
        if job is None:
            break
        print(f"  processed job {job['job_id']} ({job['task']})")
        processed += 1

    print(f"\nProcessed {processed} job(s). Queue length now: {q.queue_length()}")
```

#### How It Works & How to Run

`RedisJobQueue.enqueue` serializes a job dict to JSON and `LPUSH`es it onto a Redis list; `dequeue` uses `BRPOP`, Redis's *blocking* right-pop, so a worker can wait efficiently for new work without polling. `get_job_queue()` is a factory that tries to connect to Redis and transparently falls back to `InMemoryJobQueue` (built on the standard-library `queue.Queue`) if Redis isn't installed or reachable — so the same calling code works in both environments.

#### Variations / Flags

- Run multiple worker processes against the same `queue_name` for simple horizontal scaling — Redis's `BRPOP` guarantees each job is delivered to only one waiting consumer.
- Point `redis_url` at a remote managed Redis instance (e.g. `redis://user:pass@host:6379/0`) for production deployments instead of `localhost`.
- Wrap `dequeue`'s processing step in the `@retry` decorator from 18.1 or 18.15 so a job that fails mid-processing can be requeued rather than lost.

#### Common Pitfalls

- The in-memory fallback silently hides the fact that Redis isn't configured — always check for the `[job_queue] 'redis' package not installed` warning in logs before assuming jobs are durable across process restarts.
- `BRPOP` with `timeout=0` blocks forever; a worker script with this setting needs to be run as a long-lived service, not invoked from cron expecting it to exit.
- Jobs are plain JSON with no acknowledgment/retry semantics — if a worker crashes mid-processing after `dequeue()` but before finishing the job, that job is gone; for guaranteed-delivery semantics, use Redis Streams or a dedicated queue system instead.

### 18.12 Cache Results

#### Use case

A sysadmin has an expensive function — a slow database aggregation, a third-party API lookup, a heavy computation — that gets called repeatedly with the same arguments, and wants to avoid redoing the work either within a single process run or across separate runs and worker processes.

#### Prerequisites

`functools.lru_cache` (used by `memoize_lru`) is pure standard library. The Redis-backed path (`memoize_redis`) needs the `redis` package and a running Redis server; without either, it automatically falls back to an in-process dict with manual TTL handling.

```bash
# For the Redis-backed decorator
pip install redis

sudo apt-get install redis-server && sudo systemctl start redis-server   # Linux
brew install redis && brew services start redis                          # macOS
redis-cli ping   # should print PONG
```

#### Cautions

Both decorators key the cache off the function's *arguments*, serialized with `json.dumps`. Arguments that aren't JSON-serializable (custom objects, open file handles) will raise a `TypeError` when `memoize_redis` tries to build the cache key — keep decorated functions' arguments to plain JSON-friendly types (str, int, float, bool, list, dict).

#### Script

```python
#!/usr/bin/env python3
"""
cache_results.py
--------------------
Memoize function return values two ways:

  1. In-process, via functools.lru_cache (fast, zero setup, but cache is
     lost when the process exits and isn't shared across processes).
  2. Cross-process, via a Redis-backed cache decorator (persists between
     runs and is shared across workers, at the cost of needing a Redis
     server + the `redis` package).
"""

import functools
import json
import time

try:
    import redis  # type: ignore
    REDIS_AVAILABLE = True
except ImportError:
    REDIS_AVAILABLE = False


def memoize_lru(maxsize=128):
    """Thin, explicit wrapper around functools.lru_cache for consistency
    with the other decorators in this file."""
    def decorator(func):
        return functools.lru_cache(maxsize=maxsize)(func)
    return decorator


def memoize_redis(ttl_seconds=300, redis_url="redis://localhost:6379/0", key_prefix=None):
    """
    Cache a function's return value in Redis, keyed by its arguments,
    for `ttl_seconds`. Falls back to an in-process dict (with the same
    TTL behavior) if redis isn't installed/reachable, so the decorator
    is always safe to use.
    """
    def decorator(func):
        prefix = key_prefix or func.__name__
        local_cache = {}
        client = None
        if REDIS_AVAILABLE:
            try:
                client = redis.from_url(redis_url)
                client.ping()
            except Exception:
                client = None

        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            cache_key = f"{prefix}:{json.dumps([args, kwargs], sort_keys=True, default=str)}"

            if client is not None:
                cached = client.get(cache_key)
                if cached is not None:
                    return json.loads(cached)
                result = func(*args, **kwargs)
                client.setex(cache_key, ttl_seconds, json.dumps(result))
                return result

            # in-process fallback with manual TTL
            now = time.monotonic()
            if cache_key in local_cache:
                value, expires_at = local_cache[cache_key]
                if now < expires_at:
                    return value
            result = func(*args, **kwargs)
            local_cache[cache_key] = (result, now + ttl_seconds)
            return result

        return wrapper
    return decorator


if __name__ == "__main__":
    call_count = {"lru": 0, "redis": 0}

    @memoize_lru(maxsize=32)
    def slow_square(n):
        call_count["lru"] += 1
        time.sleep(0.2)
        return n * n

    print("Demo 1: functools.lru_cache-based memoization\n")
    print("First call (n=5):", slow_square(5), f"(function body ran {call_count['lru']}x)")
    print("Second call (n=5):", slow_square(5), f"(function body ran {call_count['lru']}x -- cached)")

    @memoize_redis(ttl_seconds=5)
    def slow_lookup(user_id):
        call_count["redis"] += 1
        time.sleep(0.2)
        return {"user_id": user_id, "name": f"user-{user_id}"}

    print("\nDemo 2: Redis-backed memoization (falls back to in-process if no Redis)\n")
    if not REDIS_AVAILABLE:
        print("[note] 'redis' package not installed -- using in-process TTL fallback for this demo.")
    print("First call (id=42):", slow_lookup(42), f"(function body ran {call_count['redis']}x)")
    print("Second call (id=42):", slow_lookup(42), f"(function body ran {call_count['redis']}x -- cached)")
```

#### How It Works & How to Run

`memoize_lru` is a thin naming wrapper around the standard library's `functools.lru_cache`, keeping the API consistent with `memoize_redis`. `memoize_redis` builds a cache key by JSON-serializing the call's `args`/`kwargs`, then checks Redis for that key via `GET`; on a miss it calls the real function and stores the JSON-serialized result with `SETEX` (set-with-expiry) for automatic TTL-based invalidation. If Redis isn't reachable at decoration time, it transparently swaps to an in-process dict with the same expiry semantics computed against `time.monotonic()`.

```bash

python3 -c "

@memoize_lru(maxsize=256)
def expensive_calc(n):
    return sum(range(n))

print(expensive_calc(1000000))
print(expensive_calc(1000000))  # cached, instant
"
```

#### Sample Output

```

First call (n=5): 25 (function body ran 1x)
Second call (n=5): 25 (function body ran 1x -- cached)

Demo 2: Redis-backed memoization (falls back to in-process if no Redis)

```

#### Variations / Flags

- Set `maxsize=None` on `memoize_lru` for an unbounded cache when memory isn't a concern and the input space is small and finite.
- Lower `ttl_seconds` on `memoize_redis` for data that changes frequently (e.g. `ttl_seconds=30` for near-real-time dashboards), or raise it for expensive, rarely-changing lookups (e.g. `ttl_seconds=86400` for a daily-refreshed reference table).
- Pass a custom `key_prefix` when multiple decorated functions could otherwise collide on argument-only keys (e.g. two functions both called with `(42,)`).

#### Common Pitfalls

- `functools.lru_cache` requires all arguments to be hashable — passing a `list` or `dict` argument raises `TypeError: unhashable type`; use `memoize_redis` (which JSON-serializes instead) if you need mutable-looking argument types.
- The in-process fallback in `memoize_redis` is *not* shared across separate processes, silently defeating the "cross-process cache" purpose if Redis is down — monitor for the fallback warning in production logs.
- Caching functions with side effects (e.g. one that also sends a notification) means the second call's side effect never fires — only cache pure, side-effect-free lookups.

### 18.13 Configuration Validation

#### Use case

A sysadmin defines a settings/config object (host, port, feature flags, credentials) for a script or service and wants it to fail loudly and immediately at startup with a clear error message if a value is the wrong type or out of range — rather than surfacing a confusing error deep inside the application hours later.

#### Prerequisites

Pure standard library (`dataclasses`, `typing`). No installation required. This is a lightweight, dependency-free alternative to Pydantic for simple validation needs; reach for Pydantic instead if you need nested models, coercion (not just validation), or JSON-schema export.

```bash
python3 --version   # 3.7+ required for dataclasses
```

#### Cautions

Validation happens in `__post_init__`, which only runs on construction — mutating an attribute after the object is created (`cfg.port = 99999`) bypasses validation entirely, since plain dataclasses don't intercept attribute assignment by default.

#### Script

```python
#!/usr/bin/env python3
"""
config_validation.py
--------------------
Enforce strong typing and constraints on runtime configuration/settings
using plain dataclasses -- no external dependency required.
"""

import dataclasses
import typing


class Field:
    """Marks a dataclass field with an optional default and validator."""

    def __init__(self, default=dataclasses.MISSING, validator=None):
        self.default = default
        self.validator = validator


def validated_config(cls):
    """
    Class decorator: turns a plain annotated class into a dataclass that
    validates, on construction:
      - every field's runtime type against its annotation
      - any `Field(validator=...)` constraint
    Raises TypeError for type mismatches and ValueError for failed
    validators, both with a clear message pointing at the offending
    field.
    """
    annotations = typing.get_type_hints(cls)
    validators = {}
    dataclass_fields = {}

    for name in annotations:
        raw_default = getattr(cls, name, dataclasses.MISSING)
        if isinstance(raw_default, Field):
            validators[name] = raw_default.validator
            dataclass_fields[name] = raw_default.default
        else:
            dataclass_fields[name] = raw_default

    for name, default in dataclass_fields.items():
        setattr(cls, name, default)

    # dataclasses.dataclass() only wires a call to __post_init__ into the
    # generated __init__ if __post_init__ already exists on the class AT
    # DECORATION TIME -- so this must be attached before dataclass(cls).
    original_post_init = getattr(cls, "__post_init__", None)

    def __post_init__(self):
        for name, expected_type in annotations.items():
            value = getattr(self, name)
            if not _type_matches(value, expected_type):
                raise TypeError(
                    f"{cls.__name__}.{name} expected {expected_type}, "
                    f"got {type(value).__name__} ({value!r})"
                )
            validator = validators.get(name)
            if validator is not None and not validator(value):
                raise ValueError(
                    f"{cls.__name__}.{name} failed validation: {value!r}"
                )
        if original_post_init:
            original_post_init(self)

    cls.__post_init__ = __post_init__
    dc = dataclasses.dataclass(cls)
    return dc


def _type_matches(value, expected_type):
    origin = typing.get_origin(expected_type)
    if origin is typing.Union:  # handles Optional[...]
        return any(_type_matches(value, t) for t in typing.get_args(expected_type))
    if origin is not None:
        return isinstance(value, origin)
    return isinstance(value, expected_type)


if __name__ == "__main__":
    @validated_config
    class AppConfig:
        host: str
        port: int = Field(default=8080, validator=lambda v: 1 <= v <= 65535)
        debug: bool = False

    print("Demo 1: valid configuration\n")
    cfg = AppConfig(host="0.0.0.0", port=9000, debug=True)
    print(" ->", cfg)

    print("\nDemo 2: wrong type for 'port'\n")
    try:
        AppConfig(host="0.0.0.0", port="not-a-number")
    except TypeError as e:
        print(" -> Caught expected TypeError:", e)

    print("\nDemo 3: 'port' fails its validator (out of range)\n")
    try:
        AppConfig(host="0.0.0.0", port=99999)
    except ValueError as e:
        print(" -> Caught expected ValueError:", e)
```

#### How It Works & How to Run

`@validated_config` reads a class's type annotations with `typing.get_type_hints`, separates out any `Field(default=..., validator=...)` markers into a side table, then attaches a generated `__post_init__` **before** calling `dataclasses.dataclass(cls)` — this ordering matters, since `dataclass()` only wires `__post_init__` into the generated `__init__` if it's already present on the class at decoration time. The generated `__post_init__` checks every field's actual runtime value against its declared type (via `isinstance`, with special handling for `Union`/`Optional`) and then runs any custom `validator` callable, raising `TypeError` or `ValueError` respectively with a message naming the offending field.

```bash

python3 -c "

@validated_config
class DbConfig:
    host: str
    pool_size: int = Field(default=10, validator=lambda v: 1 <= v <= 100)

cfg = DbConfig(host='db.internal', pool_size=25)
print(cfg)
"
```

#### Sample Output

```
Demo 1: valid configuration

 -> AppConfig(host='0.0.0.0', port=9000, debug=True)

Demo 2: wrong type for 'port'

 -> Caught expected TypeError: AppConfig.port expected <class 'int'>, got str ('not-a-number')

Demo 3: 'port' fails its validator (out of range)

 -> Caught expected ValueError: AppConfig.port failed validation: 99999
```

#### Variations / Flags

- Use `typing.Optional[int]` (i.e. `typing.Union[int, None]`) for fields that may legitimately be `None`, since `_type_matches` already unwraps `Union` types.
- Load values from environment variables or a YAML/JSON file into the constructor call (`AppConfig(**parsed_dict)`) so validation runs automatically against externally-sourced config.
- Add a class-level `validator` that cross-checks *multiple* fields together (e.g. "if `debug` is True, `port` must be in the dev range") by extending the custom `__post_init__` logic.

#### Common Pitfalls

- Mutating an already-constructed instance's attribute skips validation entirely — if strict validation must hold for the object's whole lifetime, freeze it with `dataclasses.dataclass(cls, frozen=True)` (note: `frozen=True` changes assignment semantics elsewhere in your code too).
- Forgetting `typing.get_type_hints` needs real type annotations, not just default values — a field declared only as `port = 8080` without `port: int` won't be validated at all.
- Booleans are a subclass of `int` in Python, so a type annotation of `int` will happily accept `True`/`False` — annotate as `bool` explicitly if that distinction matters for your config.

### 18.14 Automatic Cleanup

#### Use case

A sysadmin opens a resource — a database connection, a file handle, a temp directory for intermediate build artifacts — and needs it reliably closed or removed even if the code using it raises an exception halfway through, without wrapping every call site in a manual `try/finally`.

#### Prerequisites

Pure standard library (`shutil`, `tempfile`). No installation required. Works identically on Linux, macOS, and Windows (temp directory location is chosen automatically per-OS by `tempfile.mkdtemp`).

```bash
python3 --version
```

#### Cautions

`temp_directory`'s cleanup uses `shutil.rmtree(..., ignore_errors=True)`, which silently swallows any error during deletion (e.g. a file still open elsewhere, or a permissions issue) — if you need to know whether cleanup actually succeeded, don't rely on this pattern as-is; check `os.path.exists()` afterward or drop `ignore_errors=True`.

#### Script

```python
#!/usr/bin/env python3
"""
auto_cleanup.py
--------------------
Context managers that guarantee resource cleanup via __exit__, even if
the code inside the `with` block raises an exception. Includes a
generic ManagedResource wrapper plus a ready-made temp-directory
example.
"""

import shutil
import tempfile


class ManagedResource:
    """
    Generic context manager: calls `acquire()` on entry and
    `release(resource)` on exit -- exception or not.

    acquire  -- callable() -> resource
    release  -- callable(resource) -> None
    """

    def __init__(self, acquire, release):
        self.acquire = acquire
        self.release = release
        self.resource = None

    def __enter__(self):
        self.resource = self.acquire()
        return self.resource

    def __exit__(self, exc_type, exc_val, exc_tb):
        print(f"[auto_cleanup] Releasing resource "
              f"({'after exception' if exc_type else 'normally'})...")
        self.release(self.resource)
        return False  # never suppress the exception


class temp_directory:
    """Context manager that creates a temp directory and guarantees its
    removal on exit, exception or not."""

    def __enter__(self):
        self.path = tempfile.mkdtemp(prefix="auto_cleanup_demo_")
        print(f"[auto_cleanup] Created temp directory: {self.path}")
        return self.path

    def __exit__(self, exc_type, exc_val, exc_tb):
        print(f"[auto_cleanup] Removing temp directory: {self.path}")
        shutil.rmtree(self.path, ignore_errors=True)
        return False


if __name__ == "__main__":
    print("Demo 1: ManagedResource cleans up even when the block raises\n")

    class FakeConnection:
        def __init__(self):
            self.open = True

    def acquire_conn():
        print("  acquiring fake connection")
        return FakeConnection()

    def release_conn(conn):
        conn.open = False
        print("  fake connection closed:", not conn.open)

    try:
        with ManagedResource(acquire_conn, release_conn) as conn:
            print("  using connection...")
            raise RuntimeError("simulated failure while using the resource")
    except RuntimeError as e:
        print(f"  caught expected exception: {e}")

    print("\nDemo 2: temp_directory is removed automatically\n")
    with temp_directory() as path:
        import os
        with open(os.path.join(path, "example.txt"), "w") as f:
            f.write("hello")
        print("  wrote a file inside:", os.listdir(path))

    import os
    print(f"\nDirectory still exists after the block? {os.path.exists(path)}")
```

#### How It Works & How to Run

`ManagedResource.__enter__` calls the `acquire` callable and hands its return value to the `with ... as` variable; `__exit__` always calls `release(resource)` regardless of whether the block raised, then returns `False` so the original exception (if any) still propagates to the caller rather than being swallowed. `temp_directory` follows the same pattern hard-coded to `tempfile.mkdtemp()`/`shutil.rmtree()` for the common case of scratch-space cleanup.

```bash

with ManagedResource(lambda: open('data.txt', 'w'), lambda f: f.close()) as f:
    f.write('hello')
"

# temp_directory around a build step
python3 -c "

import subprocess

with temp_directory() as build_dir:
    subprocess.run(['echo', 'building in', build_dir])
"
```

#### Sample Output

```

  acquiring fake connection
  using connection...
[auto_cleanup] Releasing resource (after exception)...
  fake connection closed: True
  caught expected exception: simulated failure while using the resource

Demo 2: temp_directory is removed automatically

[auto_cleanup] Created temp directory: /tmp/auto_cleanup_demo_z50o0yef
  wrote a file inside: ['example.txt']
[auto_cleanup] Removing temp directory: /tmp/auto_cleanup_demo_z50o0yef

Directory still exists after the block? False
```

#### Variations / Flags

- Wrap `ManagedResource` around any acquire/release pair: `ManagedResource(lambda: pool.get_connection(), lambda c: pool.release(c))` for a connection pool, or `ManagedResource(lambda: lock.acquire(), lambda _: lock.release())` for a manual lock.
- Drop `ignore_errors=True` from `temp_directory.__exit__` (or catch and re-raise the `shutil.Error`) if you want cleanup failures to be visible instead of silent.
- Combine with `@contextlib.contextmanager` as an alternative, more concise way to write the same acquire/yield/release pattern for simple one-off cases.

#### Common Pitfalls

- `ignore_errors=True` in `shutil.rmtree` hides real problems (e.g. a file locked by another process on Windows) — cleanup can silently fail and leave stale directories accumulating on disk over time.
- `ManagedResource.release` must itself not raise, or that new exception will replace/mask the original exception from inside the `with` block — wrap `release` internals in their own `try/except` if the release step can fail.
- Nesting many `ManagedResource` context managers by hand gets unwieldy fast; for more than two or three resources, consider `contextlib.ExitStack` instead.

### 18.15 Retry with Exponential Backoff

#### Use case

A sysadmin's script talks to a service that's currently struggling (rate-limited, overloaded, or recovering from an outage), and needs a retry strategy where the delay between attempts grows — not stays fixed — so that many clients retrying simultaneously don't all hit the service again at the exact same moment and prolong the outage (the classic "thundering herd" problem).

#### Prerequisites

Pure standard library (`functools`, `random`, `time`). No installation required.

```bash
python3 --version
```

#### Cautions

Setting `jitter=False` across a fleet of identical clients recreates the thundering-herd problem this pattern exists to solve — every client backs off on the exact same schedule and retries in lockstep. Leave `jitter=True` (the default) for anything running as more than one instance.

#### Script

```python
#!/usr/bin/env python3
"""
retry_backoff.py
--------------------
Retry decorator that progressively increases the delay between attempts
(exponential backoff), optionally with random jitter, to avoid a
"thundering herd" of clients all retrying at the exact same moment
against a struggling service.
"""

import functools
import random
import time


def retry_with_backoff(times=5, base_delay=0.5, max_delay=30.0,
                        multiplier=2.0, jitter=True, exceptions=(Exception,)):
    """
    Decorator factory. Retries up to `times` attempts. The delay before
    attempt N is `min(max_delay, base_delay * multiplier**(N-1))`, with
    up to 50% random jitter added/subtracted if `jitter=True` to spread
    out retries across many clients.
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as exc:
                    last_exc = exc
                    if attempt == times:
                        break
                    delay = min(max_delay, base_delay * (multiplier ** (attempt - 1)))
                    if jitter:
                        delay = delay * random.uniform(0.5, 1.5)
                    print(f"[retry_backoff] Attempt {attempt}/{times} for "
                          f"'{func.__name__}' failed: {exc!r} -- retrying in {delay:.2f}s")
                    time.sleep(delay)
            print(f"[retry_backoff] All {times} attempts failed for '{func.__name__}'.")
            raise last_exc
        return wrapper
    return decorator


if __name__ == "__main__":
    call_count = {"n": 0}

    @retry_with_backoff(times=5, base_delay=0.2, max_delay=3.0, jitter=False,
                         exceptions=(ConnectionError,))
    def unreliable_service():
        call_count["n"] += 1
        if call_count["n"] < 4:
            raise ConnectionError(f"service unavailable (attempt {call_count['n']})")
        return "connected"

    print("Demo: exponential backoff (0.2s, 0.4s, 0.8s, 1.6s...) with no jitter\n")
    result = unreliable_service()
    print(f"\nFinal result: {result}")
    print(f"Total calls made: {call_count['n']}")
```

#### How It Works & How to Run

On each failed attempt, the delay before the *next* attempt is computed as `min(max_delay, base_delay * multiplier ** (attempt - 1))` — so it doubles (with the default `multiplier=2.0`) on every retry, capped at `max_delay` so it never grows unbounded. When `jitter=True`, the computed delay is multiplied by a random factor between 0.5 and 1.5, spreading out when different client instances actually retry even if they all failed at the same moment.

```bash
    ...
```

#### Variations / Flags

- Raise `max_delay` (e.g. `max_delay=300.0`) for background jobs that can tolerate long waits, or lower it (e.g. `max_delay=5.0`) for interactive requests where the user is waiting.
- Set `multiplier=1.5` for gentler backoff growth, or `multiplier=3.0` for aggressive backoff against services known to need long recovery windows.
- Schedule the whole script (not just individual calls) with cron as a coarser-grained retry: `*/10 * * * * /usr/bin/python3 /opt/scripts/sync_with_backoff.py` lets the OS-level scheduler handle retries across script invocations too.

#### Common Pitfalls

- Confusing `retry_with_backoff` (this section) with the plain fixed-delay `retry` from 18.1 — reach for backoff whenever the failure might be *load-related* (the service degrades further under retry pressure), and fixed-delay for truly transient, load-independent blips.
- Disabling jitter in any multi-instance deployment (multiple pods, multiple workers) undoes the thundering-herd protection this pattern is built for.
- `max_delay` caps the *per-attempt* delay, not the total retry budget — with `times=10` and `max_delay=30`, worst case is still several minutes of blocking; make sure the calling code (e.g. an HTTP request handler with its own timeout) can tolerate that.
