## Chapter 17: Package Management

Every Python project eventually runs into the same wall: a script that worked perfectly on your machine breaks on a colleague's laptop, or on the CI server, or six months later on your own machine after an "innocent" system update. The cause is almost always the same — packages. Different versions installed, different versions missing, or no isolation between projects at all. This chapter is about taking manual control of that problem instead of hoping `pip install` quietly does the right thing.

A sysadmin or automation engineer rarely installs one package by hand and calls it done. In practice, package management is a recurring maintenance loop: provision an isolated environment, install what a project needs, keep track of what's outdated, upgrade safely, remove what's no longer needed, and — critically — capture the exact state of an environment so it can be rebuilt identically on another machine. Every script in this chapter automates one link in that loop using Python's `subprocess` module to drive `pip`, `venv`, or a modern resolver like Poetry or Pipenv, rather than asking you to remember the right flags every time.

The scripts are deliberately built as command-line tools you can drop into a `scripts/` folder and call from cron, a CI pipeline, a Makefile, or just your own terminal. None of them require third-party libraries beyond the tools they're wrapping (`pip`, `venv`, optionally `poetry`/`pipenv`) — everything else comes from the Python standard library (`subprocess`, `argparse`, `json`, `pathlib`). That means they run anywhere Python 3 runs, with no extra setup beyond having `pip` itself available, which every modern Python installation ships with by default.

### 17.1 Install Python Packages


#### Use Case
A sysadmin provisioning a new server, a CI job bootstrapping a build container, or a developer onboarding onto a project all need the same thing: install a known list of packages reliably, target the correct interpreter (system, user, or a specific venv), and get a clean pass/fail report instead of scrolling through raw pip logs.

#### Prerequisites
No third-party libraries are required — the script only uses the Python standard library (`argparse`, `subprocess`, `sys`). It calls out to `pip`, which ships with every standard Python 3 installation. Works identically on Linux, macOS, and Windows.

```bash
# Confirm pip is available (it ships with Python by default)
python3 -m pip --version

# If missing on Linux, install it via your package manager, e.g. Debian/Ubuntu:
sudo apt install python3-pip
```

#### Cautions
Running with `--upgrade` can change versions already relied on by other projects sharing the same interpreter — always prefer a virtual environment (see 17.5) over installing into the system Python. Installing with `--user` on a machine with multiple Python versions can also lead to confusing "which install won" bugs.

#### Script

```python
#!/usr/bin/env python3
"""
install_packages.py
--------------------
Auto-provision Python packages using `pip install` run inside a subprocess.

Usage:
    python install_packages.py requests flask "django>=4.0"
    python install_packages.py --user requests
    python install_packages.py --python /path/to/venv/bin/python requests

Exit codes:
    0  -> all packages installed successfully
    1  -> one or more packages failed to install
    2  -> bad arguments / nothing to install
"""

import argparse
import subprocess
import sys


def build_pip_command(python_exe, packages, user_flag, upgrade_flag, extra_index):
    cmd = [python_exe, "-m", "pip", "install"]
    if user_flag:
        cmd.append("--user")
    if upgrade_flag:
        cmd.append("--upgrade")
    if extra_index:
        cmd.extend(["--extra-index-url", extra_index])
    cmd.extend(packages)
    return cmd


def install_packages(packages, python_exe=sys.executable, user_flag=False,
                      upgrade_flag=False, extra_index=None):
    """Install a list of package specifiers. Returns (succeeded, failed) lists."""
    succeeded, failed = [], []

    for pkg in packages:
        cmd = build_pip_command(python_exe, [pkg], user_flag, upgrade_flag, extra_index)
        print(f"[install] Running: {' '.join(cmd)}")
        try:
            result = subprocess.run(
                cmd,
                capture_output=True,
                text=True,
                timeout=300,
            )
        except subprocess.TimeoutExpired:
            print(f"[install] TIMEOUT while installing '{pkg}'")
            failed.append(pkg)
            continue
        except FileNotFoundError:
            print(f"[install] ERROR: interpreter '{python_exe}' not found")
            failed.append(pkg)
            continue

        if result.returncode == 0:
            print(f"[install] OK: {pkg}")
            succeeded.append(pkg)
        else:
            print(f"[install] FAILED: {pkg}")
            print(result.stderr.strip()[-800:])  # last part of stderr is usually the reason
            failed.append(pkg)

    return succeeded, failed


def main():
    parser = argparse.ArgumentParser(description="Install Python packages via pip.")
    parser.add_argument("packages", nargs="*", help="Package specifiers, e.g. requests 'flask>=2.0'")
    parser.add_argument("--user", action="store_true", help="Install into user site-packages")
    parser.add_argument("--upgrade", action="store_true", help="Upgrade if already installed")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    parser.add_argument("--extra-index", default=None, help="Extra package index URL (private repos, etc.)")
    args = parser.parse_args()

    if not args.packages:
        print("No packages specified. Example: python install_packages.py requests flask")
        sys.exit(2)

    succeeded, failed = install_packages(
        args.packages,
        python_exe=args.python,
        user_flag=args.user,
        upgrade_flag=args.upgrade,
        extra_index=args.extra_index,
    )

    print("\n--- Summary ---")
    print(f"Succeeded ({len(succeeded)}): {', '.join(succeeded) or 'none'}")
    print(f"Failed    ({len(failed)}): {', '.join(failed) or 'none'}")

    sys.exit(1 if failed else 0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script builds one `pip install` command per package (rather than one command for the whole list) so that a single bad package specifier doesn't abort the entire batch — each install attempt is isolated, timed out at 300 seconds, and its result recorded independently.

```bash
# Install one package into the current interpreter
python3 install_packages.py requests

# Install multiple packages, one of them version-pinned
python3 install_packages.py requests flask "django>=4.0"

# Target a specific virtual environment's interpreter
python3 install_packages.py --python ./myenv/bin/python requests

# Install into user site-packages instead of system-wide
python3 install_packages.py --user requests

# Force-upgrade an already-installed package
python3 install_packages.py --upgrade requests

# Use a private/extra package index
python3 install_packages.py --extra-index https://pypi.company.com/simple requests
```

#### Sample Output

```bash
$ python3 install_packages.py --python testenv/bin/python six
[install] Running: testenv/bin/python -m pip install six
[install] OK: six

--- Summary ---
Succeeded (1): six
Failed    (0): none
```

Verified in a sandbox: this exact command was run against a freshly created venv and completed successfully with the summary shown above.

#### Variations / Flags
- `--user` — install into the per-user site-packages directory instead of the interpreter's main site-packages.
- `--upgrade` — force pip to upgrade the package if a version is already installed.
- `--extra-index URL` — point pip at a private or mirrored package index in addition to PyPI.
- Add a `--dry-run` flag yourself by wrapping `build_pip_command` output in a print-only branch if you want to preview commands before they run.
- Schedule a "provision this list every boot" job with cron by pointing at a fixed package list: `@reboot python3 install_packages.py requests flask >> /var/log/install_packages.log 2>&1`.

#### Common Pitfalls
- **Permissions**: installing system-wide without `--user` or a venv on Linux/macOS often needs `sudo` or fails with a permissions error — prefer a venv instead of escalating privileges.
- **Missing interpreter**: passing a bad `--python` path fails with `FileNotFoundError`, caught and reported per-package rather than crashing the whole run.
- **Platform differences**: package names and available wheels can differ across OSes (some packages ship no Windows wheel and require a compiler); a failure on one platform doesn't necessarily mean the same package fails elsewhere.
- **Silent version conflicts**: installing without `--upgrade` when an older, incompatible version is already present will report success even though the version constraint you wanted wasn't actually satisfied — always cross-check with `check_outdated.py` (17.4) afterward if versions matter.

### 17.2 Upgrade Packages


#### Use Case
A sysadmin running monthly maintenance on a fleet of servers, or a developer keeping a project's dependencies current ahead of a security audit, needs a single command that finds what's stale and upgrades it — with the option to preview the change first instead of upgrading blind.

#### Prerequisites
Standard library only (`argparse`, `json`, `subprocess`, `sys`) plus `pip` itself, which is bundled with Python 3. No extra installation needed on Linux, macOS, or Windows.

```bash
# Confirm pip is available
python3 -m pip --version

# Optional: make sure pip's own outdated-list feature works
python3 -m pip list --outdated
```

#### Cautions
Upgrading "everything outdated" in one pass can introduce breaking API changes, especially for packages with major version bumps. Always run with `--dry-run` first on anything beyond a throwaway environment, and pair this with `export_requirements.py` (17.7) beforehand so you can roll back to a known-good pin list if an upgrade breaks something.

#### Script

```python
#!/usr/bin/env python3
"""
upgrade_packages.py
--------------------
Detect outdated packages (via `pip list --outdated --format=json`) and
batch-upgrade them via `pip install --upgrade`.

Usage:
    python upgrade_packages.py                # upgrade everything outdated
    python upgrade_packages.py requests numpy  # upgrade only these, if outdated
    python upgrade_packages.py --dry-run       # show what would be upgraded
    python upgrade_packages.py --python /path/to/venv/bin/python

Exit codes:
    0 -> success (including "nothing to upgrade")
    1 -> one or more upgrades failed
"""

import argparse
import json
import subprocess
import sys


def get_outdated(python_exe):
    cmd = [python_exe, "-m", "pip", "list", "--outdated", "--format=json"]
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=120)
    if result.returncode != 0:
        print("[upgrade] Could not query outdated packages:")
        print(result.stderr.strip())
        sys.exit(1)
    try:
        return json.loads(result.stdout or "[]")
    except json.JSONDecodeError:
        print("[upgrade] Unexpected pip output, aborting.")
        sys.exit(1)


def upgrade(python_exe, names, dry_run=False):
    if not names:
        print("[upgrade] Nothing to upgrade.")
        return []

    failed = []
    for name in names:
        cmd = [python_exe, "-m", "pip", "install", "--upgrade", name]
        if dry_run:
            print(f"[dry-run] Would run: {' '.join(cmd)}")
            continue
        print(f"[upgrade] Running: {' '.join(cmd)}")
        result = subprocess.run(cmd, capture_output=True, text=True, timeout=300)
        if result.returncode == 0:
            print(f"[upgrade] OK: {name}")
        else:
            print(f"[upgrade] FAILED: {name}")
            print(result.stderr.strip()[-800:])
            failed.append(name)
    return failed


def main():
    parser = argparse.ArgumentParser(description="Batch-upgrade outdated Python packages.")
    parser.add_argument("packages", nargs="*", help="Optional subset of package names to consider")
    parser.add_argument("--dry-run", action="store_true", help="List what would be upgraded, do nothing")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    args = parser.parse_args()

    outdated = get_outdated(args.python)
    outdated_names = {pkg["name"] for pkg in outdated}

    if args.packages:
        targets = [p for p in args.packages if p in outdated_names]
        skipped = [p for p in args.packages if p not in outdated_names]
        if skipped:
            print(f"[upgrade] Already up to date / not installed, skipping: {', '.join(skipped)}")
    else:
        targets = sorted(outdated_names)

    if not targets:
        print("[upgrade] Nothing outdated. All good.")
        sys.exit(0)

    print(f"[upgrade] {len(targets)} package(s) to upgrade: {', '.join(targets)}")
    failed = upgrade(args.python, targets, dry_run=args.dry_run)

    print("\n--- Summary ---")
    print(f"Attempted: {len(targets)}  Failed: {len(failed)}")
    sys.exit(1 if failed else 0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script first calls `pip list --outdated --format=json` to get a machine-readable list of stale packages, filters it against any package names you passed on the command line (or takes everything if you passed none), then loops through and runs `pip install --upgrade` per package — same "isolate each failure" design as `install_packages.py`.

```bash
# Upgrade every outdated package for the current interpreter
python3 upgrade_packages.py

# Upgrade only specific packages, if and only if they're outdated
python3 upgrade_packages.py requests numpy

# Preview what would happen without changing anything
python3 upgrade_packages.py --dry-run

# Target a specific venv's interpreter
python3 upgrade_packages.py --python ./myenv/bin/python
```

#### Sample Output

```bash
$ python3 upgrade_packages.py --python testenv/bin/python --dry-run
[upgrade] 1 package(s) to upgrade: pip
[dry-run] Would run: testenv/bin/python -m pip install --upgrade pip

--- Summary ---
Attempted: 1  Failed: 0
```

Verified in a sandbox venv: `pip` itself showed up as outdated (24.0 installed vs. a newer release available), and `--dry-run` correctly listed the command without executing it.

#### Variations / Flags
- `--dry-run` — already built in; always run this first in production.
- Pass explicit package names to scope the upgrade instead of upgrading the whole environment.
- Schedule with cron for routine maintenance, e.g. a monthly job: `0 3 1 * * python3 upgrade_packages.py --python /opt/venvs/prod/bin/python >> /var/log/upgrade.log 2>&1`.
- Combine with `export_requirements.py` (17.7) immediately before upgrading, so a bad upgrade can be rolled back with `install_from_requirements.py` (17.8).

#### Common Pitfalls
- **Major version jumps**: `pip list --outdated` doesn't distinguish a patch release from a breaking major version bump — review the target list before running without `--dry-run`.
- **Permissions**: upgrading system-wide packages without a venv can require elevated privileges on Linux/macOS, or fail silently against a read-only site-packages directory.
- **Network/index timeouts**: the outdated-check itself hits PyPI (or your configured index); a slow or unreachable index will cause the 120-second timeout on `get_outdated` to trigger.
- **Interpreter mismatch**: forgetting `--python` means the script upgrades whatever interpreter is running the script itself, which may not be the project's virtual environment.

### 17.3 Remove Packages

#### Use Case
A sysadmin cleaning up an environment full of leftover, no-longer-needed dependencies (or removing a package that turned out to be the wrong fix) needs an uninstall path that won't nuke `pip` itself by a typo, and that reports exactly what happened for an audit trail.

#### Prerequisites
Standard library only (`argparse`, `subprocess`, `sys`). Requires `pip` to be present on the target interpreter — no additional installation needed on Linux, macOS, or Windows.

```bash
# Confirm pip is available
python3 -m pip --version
```

#### Cautions
Uninstalling a package that other installed packages depend on can silently break those dependents — `pip uninstall` does not do dependency-aware removal. Always check what depends on a package (e.g. with `pip show <package>`) before removing it from a shared or production environment.

#### Script

```python
#!/usr/bin/env python3
"""
remove_packages.py
--------------------
Safely uninstall one or more Python packages via `pip uninstall`.

"Safely" here means:
  - Requires explicit confirmation (unless --yes is passed)
  - Refuses to uninstall a small protected list of critical packages
    (pip, setuptools, wheel) unless --force is given
  - Reports clearly which packages were removed vs skipped vs failed

Usage:
    python remove_packages.py requests flask
    python remove_packages.py requests --yes
    python remove_packages.py pip --force --yes

Exit codes:
    0 -> all requested removals succeeded (or nothing to do)
    1 -> one or more removals failed
    2 -> bad arguments
"""

import argparse
import subprocess
import sys

PROTECTED = {"pip", "setuptools", "wheel"}


def confirm(packages):
    names = ", ".join(packages)
    reply = input(f"About to uninstall: {names}\nProceed? [y/N]: ").strip().lower()
    return reply == "y"


def remove_packages(packages, python_exe=sys.executable, force=False):
    to_remove = []
    skipped = []

    for pkg in packages:
        if pkg.lower() in PROTECTED and not force:
            skipped.append(pkg)
        else:
            to_remove.append(pkg)

    if skipped:
        print(f"[remove] Skipping protected package(s) (use --force to override): {', '.join(skipped)}")

    succeeded, failed = [], []
    for pkg in to_remove:
        cmd = [python_exe, "-m", "pip", "uninstall", "-y", pkg]
        print(f"[remove] Running: {' '.join(cmd)}")
        result = subprocess.run(cmd, capture_output=True, text=True, timeout=120)
        if result.returncode == 0:
            print(f"[remove] OK: {pkg}")
            succeeded.append(pkg)
        else:
            print(f"[remove] FAILED: {pkg}")
            print(result.stderr.strip()[-800:])
            failed.append(pkg)

    return succeeded, failed, skipped


def main():
    parser = argparse.ArgumentParser(description="Safely uninstall Python packages.")
    parser.add_argument("packages", nargs="*", help="Package names to remove")
    parser.add_argument("--yes", action="store_true", help="Skip the confirmation prompt")
    parser.add_argument("--force", action="store_true", help="Allow removing protected packages (pip, setuptools, wheel)")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    args = parser.parse_args()

    if not args.packages:
        print("No packages specified. Example: python remove_packages.py requests flask")
        sys.exit(2)

    if not args.yes and not confirm(args.packages):
        print("Aborted, no changes made.")
        sys.exit(0)

    succeeded, failed, skipped = remove_packages(args.packages, python_exe=args.python, force=args.force)

    print("\n--- Summary ---")
    print(f"Removed ({len(succeeded)}): {', '.join(succeeded) or 'none'}")
    print(f"Failed  ({len(failed)}): {', '.join(failed) or 'none'}")
    print(f"Skipped ({len(skipped)}): {', '.join(skipped) or 'none'}")

    sys.exit(1 if failed else 0)


if __name__ == "__main__":
    main()
```

#### How It Works
Every requested package is first checked against a small `PROTECTED` set (`pip`, `setuptools`, `wheel`). Anything protected is skipped unless `--force` is passed. Unless `--yes` is also passed, the script pauses for an interactive `y/N` confirmation before touching anything, then uninstalls each remaining package individually with `pip uninstall -y`, tallying successes and failures separately.

```bash
# Interactive removal, will prompt for confirmation
python3 remove_packages.py requests flask

# Non-interactive removal (e.g. for use in scripts/CI)
python3 remove_packages.py requests --yes

# Attempt to remove a protected package (will be skipped)
python3 remove_packages.py pip --yes

# Force-remove a protected package (dangerous, rarely needed)
python3 remove_packages.py pip --force --yes

# Target a specific virtual environment
python3 remove_packages.py requests --yes --python ./myenv/bin/python
```

#### Sample Output

```bash
$ python3 remove_packages.py six --yes --python testenv/bin/python
[remove] Running: testenv/bin/python -m pip uninstall -y six
[remove] OK: six

--- Summary ---
Removed (1): six
Failed  (0): none
Skipped (0): none

$ python3 remove_packages.py pip --yes --python testenv/bin/python
[remove] Skipping protected package(s) (use --force to override): pip

--- Summary ---
Removed (0): none
Failed  (0): none
Skipped (1): pip
```

Both cases were verified directly: a normal package (`six`) was removed cleanly, and an attempt to remove `pip` without `--force` was correctly skipped and reported rather than silently ignored.

#### Variations / Flags
- `--yes` — skip the interactive confirmation prompt (needed for cron/CI use, since there's no terminal to answer `y/N`).
- `--force` — allow removal of protected packages; use sparingly and never in an automated job.
- Add a `--dry-run` flag yourself by printing the `pip uninstall` command without calling `subprocess.run`, to preview a batch removal safely.
- Schedule periodic cleanup with cron once you're confident in a fixed removal list: `0 4 * * 0 python3 remove_packages.py old-package --yes >> /var/log/remove.log 2>&1`.

#### Common Pitfalls
- **Forgetting `--yes` in automation**: without it, `input()` will hang indefinitely waiting for a prompt that no cron job or CI runner can answer.
- **Breaking dependents**: removing a package that other installed packages import will not raise an error at uninstall time — the breakage only shows up later, at import time.
- **Case sensitivity on the protected list**: the check is done via `.lower()`, so `Pip`, `PIP`, and `pip` are all treated as protected — but a package with a similar-but-different name (e.g. `pip-tools`) is not protected and will be removed without `--force`.
- **Wrong interpreter**: forgetting `--python` removes the package from whichever interpreter is running the script, not necessarily the target project's venv.

### 17.4 Check Outdated Packages

#### Use Case
A sysadmin doing a routine environment audit, or a developer deciding whether an upgrade cycle is needed before a release, wants a quick, scriptable answer to "what's stale right now?" — as a human-readable table, as raw JSON for other tooling, or saved to a file for a report.

#### Prerequisites
Standard library only (`argparse`, `json`, `subprocess`, `sys`) plus `pip`, which ships with Python 3 by default. No extra installation required on Linux, macOS, or Windows.

```bash
# Confirm pip's outdated-list feature works directly
python3 -m pip list --outdated --format=json
```

#### Cautions
This script is read-only and safe to run at any time — it changes nothing. The only caution is trusting the "latest" version blindly: `pip`'s outdated check reports whatever the configured index (usually PyPI) considers newest, which may include pre-release or yanked versions depending on your pip configuration.

#### Script

```python
#!/usr/bin/env python3
"""
check_outdated.py
--------------------
Programmatic inventory of stale (outdated) requirements using
`pip list --outdated --format=json`.

Usage:
    python check_outdated.py
    python check_outdated.py --json                # raw JSON output
    python check_outdated.py --save outdated.json   # also write results to a file
    python check_outdated.py --python /path/to/venv/bin/python

Exit codes:
    0 -> ran successfully (0 or more outdated packages found)
    1 -> failed to query pip
"""

import argparse
import json
import subprocess
import sys


def check_outdated(python_exe=sys.executable):
    cmd = [python_exe, "-m", "pip", "list", "--outdated", "--format=json"]
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=120)
    if result.returncode != 0:
        print("[check] Could not query pip:")
        print(result.stderr.strip())
        sys.exit(1)
    try:
        return json.loads(result.stdout or "[]")
    except json.JSONDecodeError:
        print("[check] Unexpected pip output.")
        sys.exit(1)


def print_table(outdated):
    if not outdated:
        print("Everything is up to date.")
        return
    name_w = max(len(p["name"]) for p in outdated) + 2
    cur_w = max(len(p["version"]) for p in outdated) + 2
    print(f"{'PACKAGE'.ljust(name_w)}{'CURRENT'.ljust(cur_w)}{'LATEST'}")
    for p in outdated:
        print(f"{p['name'].ljust(name_w)}{p['version'].ljust(cur_w)}{p['latest_version']}")
    print(f"\n{len(outdated)} package(s) outdated.")


def main():
    parser = argparse.ArgumentParser(description="Inventory of outdated Python packages.")
    parser.add_argument("--json", action="store_true", help="Print raw JSON instead of a table")
    parser.add_argument("--save", metavar="FILE", help="Also write the JSON result to a file")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    args = parser.parse_args()

    outdated = check_outdated(args.python)

    if args.json:
        print(json.dumps(outdated, indent=2))
    else:
        print_table(outdated)

    if args.save:
        with open(args.save, "w") as f:
            json.dump(outdated, f, indent=2)
        print(f"\nSaved to {args.save}")

    sys.exit(0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script calls `pip list --outdated --format=json`, parses the JSON, and either prints a formatted table (default), dumps raw JSON with `--json`, or writes the JSON to a file with `--save` for downstream tooling — any combination of these can be used together in one run.

```bash
# Default human-readable table for the current interpreter
python3 check_outdated.py

# Raw JSON, useful for piping into jq or another script
python3 check_outdated.py --json

# Save the result to a file (in addition to printing the table)
python3 check_outdated.py --save outdated.json

# Target a specific virtual environment
python3 check_outdated.py --python ./myenv/bin/python
```

#### Sample Output

```bash
$ python3 check_outdated.py --python testenv/bin/python
PACKAGECURRENTLATEST
pip  24.0  26.1.2

1 package(s) outdated.

$ python3 check_outdated.py --python testenv/bin/python --json
[
  {
    "name": "pip",
    "version": "24.0",
    "latest_version": "26.1.2",
    "latest_filetype": "wheel"
  }
]
```

Both output modes were verified against a real sandbox venv. Note the table header in the first example: `PACKAGECURRENTLATEST` runs together with no spacing — see Common Pitfalls below, this is a genuine formatting bug in the script as written, not a copy error in this doc.

#### Variations / Flags
- `--json` — machine-readable output, ideal for feeding into `jq`, a monitoring dashboard, or another Python script.
- `--save FILE` — persist the JSON snapshot for historical tracking or a scheduled report.
- Schedule a weekly audit with cron and email/Slack the result: `0 8 * * 1 python3 check_outdated.py --save /var/log/outdated-$(date +\%F).json`.
- Feed the output directly into `upgrade_packages.py` (17.2) by extracting package names from the JSON.

#### Common Pitfalls
- **Header alignment bug**: `print_table` sizes each column's width from the longest *data* value in that column (e.g. `len("pip")`), not from the header word itself (`"PACKAGE"`, `"CURRENT"`, `"LATEST"`). When a header word is longer than every value in its column — which is common when only one or two short-named packages are outdated — `ljust()` has nothing to pad, and the header prints as one run-together string like `PACKAGECURRENTLATEST`. The data rows below the header are usually fine since real package names/versions tend to be longer. If you rely on this table for a report, either use `--json` and format it yourself, or widen `name_w`/`cur_w` to also account for `len("PACKAGE")` and `len("CURRENT")`.
- **Network dependency**: this is not truly "offline" — pip still needs to reach its configured package index to know what's outdated; a firewall or offline mirror without index metadata will cause the check to hang or fail within the 120-second timeout.
- **Interpreter mismatch**: omitting `--python` checks whichever interpreter is running the script itself, not necessarily the project venv you meant to audit.
- **Pre-release noise**: if your pip config allows pre-releases, "latest_version" may point to an alpha/beta/rc build rather than the last stable release.

### 17.5 Create Virtual Environments

#### Use Case
A sysadmin spinning up a fresh environment for each new project (or a CI pipeline provisioning a clean sandbox per build) needs isolation guaranteed — no shared site-packages, no version collisions between unrelated projects, and no risk of polluting the system Python.

#### Prerequisites
Entirely standard library (`argparse`, `subprocess`, `sys`, `venv`, `pathlib`) — no pip install needed for the script itself. The `venv` module ships with Python 3 on all platforms, though some minimal Linux distros split it into a separate OS package.

```bash
# Debian/Ubuntu sometimes ships venv separately from the base python3 package
sudo apt install python3-venv

# macOS / Windows: venv is included with the standard python3 installer, no extra step
python3 -m venv --help
```

#### Cautions
Using `--clear` deletes the entire contents of the target directory before rebuilding it — double-check the path before running this against anything containing data you didn't put there yourself (e.g. never point it at a shared or system directory).

#### Script

```python
#!/usr/bin/env python3
"""
create_venv.py
--------------------
Build an isolated Python virtual environment using the standard-library
`venv` module.

Usage:
    python create_venv.py myenv
    python create_venv.py myenv --python python3.11
    python create_venv.py myenv --with-pip-upgrade
    python create_venv.py myenv --clear     # wipe and recreate if it exists

Exit codes:
    0 -> venv created successfully
    1 -> creation failed
    2 -> bad arguments
"""

import argparse
import subprocess
import sys
import venv
from pathlib import Path


def create_venv(target_dir, clear=False, upgrade_pip=False):
    target_dir = Path(target_dir)
    print(f"[venv] Creating virtual environment at: {target_dir.resolve()}")

    builder = venv.EnvBuilder(
        system_site_packages=False,
        clear=clear,
        symlinks=(sys.platform != "win32"),
        with_pip=True,
        upgrade_deps=upgrade_pip,
    )
    try:
        builder.create(str(target_dir))
    except Exception as exc:
        print(f"[venv] FAILED: {exc}")
        return False

    # Report the interpreter path so the user knows what to run next
    if sys.platform == "win32":
        py_path = target_dir / "Scripts" / "python.exe"
    else:
        py_path = target_dir / "bin" / "python"

    print(f"[venv] Done. Interpreter located at: {py_path}")
    print("[venv] Use activate_venv.py to see the activation command for your shell.")
    return True


def main():
    parser = argparse.ArgumentParser(description="Create an isolated Python virtual environment.")
    parser.add_argument("path", help="Directory to create the venv in, e.g. ./myenv")
    parser.add_argument("--clear", action="store_true", help="Delete contents of an existing venv dir first")
    parser.add_argument("--with-pip-upgrade", dest="upgrade_pip", action="store_true",
                         help="Upgrade pip/setuptools/wheel inside the new venv")
    args = parser.parse_args()

    if not args.path:
        print("You must specify a target directory. Example: python create_venv.py myenv")
        sys.exit(2)

    ok = create_venv(args.path, clear=args.clear, upgrade_pip=args.upgrade_pip)
    sys.exit(0 if ok else 1)


if __name__ == "__main__":
    main()
```

#### How It Works
The script wraps `venv.EnvBuilder` directly (rather than shelling out to `python -m venv`), which lets it configure options like `system_site_packages=False` (real isolation), `symlinks` (used on POSIX, copied on Windows), and `upgrade_deps` (updates pip/setuptools/wheel inside the new venv immediately) all in one call. Note: despite the `--python python3.11` example in the docstring, the current script body does not actually implement a `--python` argument — it always uses whichever interpreter is running the script itself to build the venv.

```bash
# Create a new venv in ./myenv
python3 create_venv.py myenv

# Wipe and recreate an existing venv directory
python3 create_venv.py myenv --clear

# Create it and immediately upgrade pip/setuptools/wheel inside it
python3 create_venv.py myenv --with-pip-upgrade
```

#### Sample Output

```bash
$ python3 create_venv.py testenv
[venv] Creating virtual environment at: /home/user/project/testenv
[venv] Done. Interpreter located at: testenv/bin/python
[venv] Use activate_venv.py to see the activation command for your shell.
```

Verified directly in a sandbox: the venv was created successfully, and `testenv/bin/python` existed and ran afterward exactly as reported.

#### Variations / Flags
- `--clear` — delete and rebuild an existing venv directory instead of failing on a collision.
- `--with-pip-upgrade` — upgrade `pip`, `setuptools`, and `wheel` inside the venv as part of creation, so you start from current tooling rather than whatever was bundled with your Python installer.
- To actually support choosing a specific Python version (as the docstring example suggests), you would need to add a `--python` argument and invoke that interpreter's own `-m venv` via `subprocess` instead of using the currently-running interpreter's `venv.EnvBuilder` — the script as given does not yet do this.
- Combine with `install_from_requirements.py` (17.8) right after creation to provision a project's dependencies in one pipeline.

#### Common Pitfalls
- **Missing `python3-venv` on minimal Linux images**: some distros (notably Debian/Ubuntu minimal/slim containers) omit `venv` from the base `python3` package, causing `ModuleNotFoundError` or a "not found" error until `python3-venv` is installed separately.
- **`--clear` deletes data**: it wipes the target directory's contents — always confirm the path before use, especially in scripted/automated contexts.
- **Docstring/implementation mismatch**: the module docstring shows a `--python python3.11` example, but there is no such argument wired up in `main()` — running that exact example will fail with an "unrecognized arguments" error from `argparse`.
- **Symlink vs. copy differences**: on Windows the interpreter is copied into the venv (no symlinks), which makes the venv larger and means it won't automatically reflect changes to the system Python the way a symlinked POSIX venv can.

### 17.6 Activate Virtual Environments

#### Use Case
A sysadmin writing an automation script that needs to use a project's venv, without wanting to actually `source` an activate script inside another script (which wouldn't survive past the subprocess anyway), needs a reliable way to find the right interpreter and invoke it directly.

#### Prerequisites
Standard library only (`argparse`, `subprocess`, `sys`, `pathlib`). Assumes a venv already exists — create one first with `create_venv.py` (17.5). Works on Linux, macOS, and Windows, with platform-specific paths handled internally (`bin/` vs. `Scripts/`).

```bash
# No installation needed - just make sure a venv exists first
python3 create_venv.py myenv
```

#### Cautions
A subprocess genuinely cannot change the environment variables of the shell that launched it — this is a fundamental limitation of how processes work, not a bug in this script. Don't expect `python3 activate_venv.py myenv` alone to leave your current shell "activated"; you must run the printed `source ...` command yourself, or use `--run` to execute something inside the venv without activating at all.

#### Script

```python
#!/usr/bin/env python3
"""
activate_venv.py
--------------------
A subprocess cannot change the activation state of your parent shell
(that is a shell-only concept), so this script instead LOCATES the
correct interpreter and activation script inside bin/ (POSIX) or
Scripts/ (Windows) and prints the exact command you need to run.

Usage:
    python activate_venv.py myenv
    python activate_venv.py myenv --run "pip -V"   # run a command using that venv's python

Exit codes:
    0 -> venv found and info printed (or command ran successfully)
    1 -> venv / interpreter not found, or --run command failed
"""

import argparse
import subprocess
import sys
from pathlib import Path


def locate_interpreter(venv_dir):
    venv_dir = Path(venv_dir)
    if sys.platform == "win32":
        py = venv_dir / "Scripts" / "python.exe"
        activate = venv_dir / "Scripts" / "activate.bat"
        activate_cmd = str(activate)
    else:
        py = venv_dir / "bin" / "python"
        activate = venv_dir / "bin" / "activate"
        activate_cmd = f"source {activate}"
    return py, activate, activate_cmd


def main():
    parser = argparse.ArgumentParser(
        description="Locate a venv's interpreter and print its activation command."
    )
    parser.add_argument("path", help="Path to the virtual environment directory")
    parser.add_argument("--run", metavar="CMD",
                         help="Run CMD using this venv's python -m pip / python directly, e.g. 'pip -V'")
    args = parser.parse_args()

    py, activate, activate_cmd = locate_interpreter(args.path)

    if not py.exists():
        print(f"[activate] No interpreter found at {py}. Did you create the venv with create_venv.py?")
        sys.exit(1)

    print(f"[activate] Interpreter: {py}")
    if activate.exists():
        print(f"[activate] To activate this venv in your shell, run:\n    {activate_cmd}")
    else:
        print("[activate] No activate script found (unusual) -- you can still call the interpreter directly.")

    if args.run:
        full_cmd = [str(py)] + args.run.split()
        print(f"[activate] Running inside venv: {' '.join(full_cmd)}")
        result = subprocess.run(full_cmd, capture_output=True, text=True)
        print(result.stdout)
        if result.returncode != 0:
            print(result.stderr)
            sys.exit(1)

    sys.exit(0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script builds the expected interpreter and activation-script paths for the current platform (`bin/python` + `bin/activate` on POSIX, `Scripts/python.exe` + `Scripts/activate.bat` on Windows), checks that the interpreter actually exists, and prints the correct shell command to activate it. If `--run` is given, it instead builds `[venv_python] + run_command.split()` and executes that directly via `subprocess.run`, bypassing activation entirely.

```bash
# Show the interpreter path and the activation command for your shell
python3 activate_venv.py myenv

# Run a one-off command using the venv's python, without activating
python3 activate_venv.py myenv --run "-m pip -V"

# Then, to actually activate the venv in your current shell, copy-paste the printed command:
source myenv/bin/activate
```

#### Sample Output

```bash
$ python3 activate_venv.py testenv
[activate] Interpreter: testenv/bin/python
[activate] To activate this venv in your shell, run:
    source testenv/bin/activate

$ python3 activate_venv.py testenv --run "-m pip -V"
[activate] Interpreter: testenv/bin/python
[activate] To activate this venv in your shell, run:
    source testenv/bin/activate
[activate] Running inside venv: testenv/bin/python -m pip -V
pip 24.0 from /home/user/project/testenv/lib/python3.12/site-packages/pip (python 3.12)
```

Both modes were verified directly against a real venv created with `create_venv.py` — the interpreter path resolved correctly, and `--run` executed `pip -V` inside the venv without ever activating it in the shell.

#### Variations / Flags
- `--run CMD` — execute any command using the venv's own interpreter (e.g. `"-m pip install requests"`, `"-m pytest"`) without needing to activate the venv first — ideal for CI or cron, where there's no interactive shell to `source` anything in.
- The `--run` command is split naively on whitespace (`args.run.split()`), so it works for simple flag-based commands but won't correctly handle arguments that need quoting/escaping (e.g. a file path containing spaces).
- Wrap this in a Makefile target (`make shell`) that prints the activation command, or in a shell function that captures and evaluates it automatically: `eval "$(python3 activate_venv.py myenv | grep source | xargs)"`.

#### Common Pitfalls
- **Expecting real activation from the script itself**: running `python3 activate_venv.py myenv` does *not* change your current shell's `PATH` or `VIRTUAL_ENV` — you must copy-paste and run the printed `source` command yourself, since a child process cannot mutate its parent's environment.
- **Naive command splitting with `--run`**: `args.run.split()` breaks on any whitespace, so `--run "install 'package name'"`-style quoting will not behave as expected; keep `--run` commands simple (single flags/arguments without embedded spaces).
- **Wrong path on Windows**: the script correctly branches on `sys.platform == "win32"`, but only if you run it *from* Windows — running this script inside WSL (which reports as Linux) against a venv created on native Windows will look in the wrong subdirectory (`bin/` instead of `Scripts/`).
- **venv not yet created**: pointing this at a directory that isn't actually a venv (or one that failed to create fully) reports a missing interpreter and exits with code 1 rather than silently continuing.

### 17.7 Export requirements.txt

#### Use Case
A sysadmin who just got a project's environment working correctly needs to capture that exact state before anything changes — so it can be checked into version control, handed to a teammate, or used to rebuild an identical environment on another machine or in CI.

#### Prerequisites
Standard library only (`argparse`, `subprocess`, `sys`). Requires `pip`, bundled with Python 3 by default. Works the same on Linux, macOS, and Windows.

```bash
# Confirm pip freeze works directly
python3 -m pip freeze
```

#### Cautions
`pip freeze` captures *everything* currently installed, including transitive dependencies you never explicitly asked for — this is usually what you want for exact reproducibility, but it means the resulting file can be long and includes packages that aren't your project's direct dependencies.

#### Script

```python
#!/usr/bin/env python3
"""
export_requirements.py
--------------------
Equivalent of `pip freeze`, scoped to a specific project/venv interpreter,
written out to a requirements.txt-style file.

Usage:
    python export_requirements.py                       # freeze current interpreter to requirements.txt
    python export_requirements.py --output deps.txt
    python export_requirements.py --python ./myenv/bin/python --output requirements.txt

Exit codes:
    0 -> exported successfully
    1 -> pip freeze failed
"""

import argparse
import subprocess
import sys


def freeze(python_exe):
    cmd = [python_exe, "-m", "pip", "freeze"]
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=120)
    if result.returncode != 0:
        print("[export] pip freeze failed:")
        print(result.stderr.strip())
        sys.exit(1)
    return result.stdout


def main():
    parser = argparse.ArgumentParser(description="Export requirements.txt from an environment.")
    parser.add_argument("--output", default="requirements.txt", help="Output file path")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    args = parser.parse_args()

    print(f"[export] Freezing packages from: {args.python}")
    contents = freeze(args.python)

    lines = [l for l in contents.splitlines() if l.strip()]
    with open(args.output, "w") as f:
        f.write(contents)

    print(f"[export] Wrote {len(lines)} package pin(s) to {args.output}")
    sys.exit(0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script runs `pip freeze` against the interpreter you specify (or the current one by default), captures the output, and writes it verbatim to the target file — it also counts non-blank lines to report how many packages were pinned, without altering the actual freeze output that gets written to disk.

```bash
# Freeze the current interpreter's packages to ./requirements.txt
python3 export_requirements.py

# Freeze to a custom filename
python3 export_requirements.py --output deps.txt

# Freeze a specific virtual environment's packages
python3 export_requirements.py --python ./myenv/bin/python --output requirements.txt
```

#### Sample Output

```bash
$ python3 export_requirements.py --python testenv/bin/python --output req_test.txt
[export] Freezing packages from: testenv/bin/python
[export] Wrote 1 package pin(s) to req_test.txt

$ cat req_test.txt
six==1.17.0
```

Verified directly: after installing `six` into a sandbox venv, this export correctly produced a one-line `requirements.txt` pinning the exact installed version.

#### Variations / Flags
- `--output FILE` — write to any filename/path instead of the default `requirements.txt`.
- `--python PATH` — scope the freeze to a specific venv's interpreter rather than whichever Python is running the script.
- Pair with `check_outdated.py` (17.4) beforehand as a "before" snapshot, then diff two exports across time to track how a project's dependency tree has changed.
- Schedule a nightly export as a lightweight backup: `0 2 * * * python3 export_requirements.py --python /opt/venvs/prod/bin/python --output /backups/requirements-$(date +\%F).txt`.

#### Common Pitfalls
- **Includes everything, not just direct dependencies**: `pip freeze` output has no concept of "I explicitly asked for this" vs. "this got pulled in as a dependency" — for a cleaner, hand-curated requirements file, you'll need to maintain that distinction yourself (or use a tool like `pip-tools`/`pipreqs`, or Poetry/Pipenv from 17.9, which do track direct vs. transitive dependencies).
- **VCS and editable installs**: packages installed via `pip install -e .` or directly from a git URL show up in `pip freeze` output in a form that isn't always portable to another machine without the same local path or repo access.
- **Overwriting silently**: the script opens the output file in `"w"` mode, so re-running it will silently overwrite a previous export with no backup — keep it under version control if you want history.
- **Wrong interpreter**: omitting `--python` freezes whatever interpreter is running the script itself, which may be the system Python rather than the project's venv.

### 17.8 Install from requirements.txt

#### Use Case
A sysadmin setting up a new server, a CI job building a fresh container, or a teammate cloning a repository for the first time all need the exact same dependency set another machine had — this script is the mirror image of `export_requirements.py` (17.7): one exports the state, the other reproduces it.

#### Prerequisites
Standard library only (`argparse`, `subprocess`, `sys`, `pathlib`). Requires `pip`, bundled with Python 3 by default. Identical behavior on Linux, macOS, and Windows.

```bash
# Confirm pip can install from a requirements file directly
python3 -m pip install -r requirements.txt --dry-run 2>/dev/null || echo "note: --dry-run support varies by pip version"
```

#### Cautions
Installing from an untrusted or unreviewed requirements file runs arbitrary code from PyPI (or wherever the packages come from) on your machine — always review the contents of a `requirements.txt` you didn't write yourself before installing it, especially if it references a custom `--extra-index`.

#### Script

```python
#!/usr/bin/env python3
"""
install_from_requirements.py
--------------------
Reproduce an environment reliably from a requirements.txt (or similarly
formatted) lock file, via `pip install -r`.

Usage:
    python install_from_requirements.py requirements.txt
    python install_from_requirements.py requirements.txt --python ./myenv/bin/python
    python install_from_requirements.py requirements.txt --no-deps

Exit codes:
    0 -> installed successfully
    1 -> pip install failed
    2 -> requirements file not found / bad arguments
"""

import argparse
import subprocess
import sys
from pathlib import Path


def install_from_file(req_file, python_exe=sys.executable, no_deps=False, extra_index=None):
    cmd = [python_exe, "-m", "pip", "install", "-r", req_file]
    if no_deps:
        cmd.append("--no-deps")
    if extra_index:
        cmd.extend(["--extra-index-url", extra_index])

    print(f"[install-req] Running: {' '.join(cmd)}")
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=900)
    print(result.stdout)
    if result.returncode != 0:
        print(result.stderr.strip()[-1500:])
    return result.returncode == 0


def main():
    parser = argparse.ArgumentParser(description="Install packages from a requirements.txt file.")
    parser.add_argument("requirements", help="Path to requirements.txt")
    parser.add_argument("--python", default=sys.executable, help="Path to target python interpreter")
    parser.add_argument("--no-deps", action="store_true", help="Do not install transitive dependencies")
    parser.add_argument("--extra-index", default=None, help="Extra package index URL")
    args = parser.parse_args()

    req_path = Path(args.requirements)
    if not req_path.is_file():
        print(f"[install-req] File not found: {req_path}")
        sys.exit(2)

    ok = install_from_file(str(req_path), python_exe=args.python, no_deps=args.no_deps,
                            extra_index=args.extra_index)
    if ok:
        print("[install-req] Environment reproduced successfully.")
    else:
        print("[install-req] Installation failed. See error output above.")
    sys.exit(0 if ok else 1)


if __name__ == "__main__":
    main()
```

#### How It Works
The script first checks that the requirements file actually exists (failing fast with exit code 2 if not), then runs a single `pip install -r <file>` call with a generous 900-second timeout to accommodate large dependency trees, optionally adding `--no-deps` or `--extra-index-url` if requested.

```bash
# Install everything pinned in requirements.txt into the current interpreter
python3 install_from_requirements.py requirements.txt

# Install into a specific virtual environment
python3 install_from_requirements.py requirements.txt --python ./myenv/bin/python

# Install only the exact pins listed, with no transitive dependency resolution
python3 install_from_requirements.py requirements.txt --no-deps

# Install using a private/extra package index
python3 install_from_requirements.py requirements.txt --extra-index https://pypi.company.com/simple
```

#### Sample Output

```bash
$ python3 install_from_requirements.py req_test2.txt --python testenv/bin/python
[install-req] Running: testenv/bin/python -m pip install -r req_test2.txt
Collecting urllib3 (from -r req_test2.txt (line 1))
  Downloading urllib3-2.7.0-py3-none-any.whl.metadata (6.9 kB)
Downloading urllib3-2.7.0-py3-none-any.whl (131 kB)
Installing collected packages: urllib3
Successfully installed urllib3-2.7.0

[install-req] Environment reproduced successfully.
```

Verified directly: a simple one-line requirements file (`urllib3`) was installed cleanly into a sandbox venv using this exact script and flag combination.

#### Variations / Flags
- `--no-deps` — install exactly what's pinned with no transitive resolution; useful when your requirements file already lists every package (direct and transitive) and you want to guarantee pip won't pull in a newer version of an unlisted dependency.
- `--extra-index URL` — reach a private or mirrored package index in addition to PyPI.
- `--python PATH` — target a specific venv's interpreter rather than whichever Python runs the script.
- Chain this immediately after `create_venv.py` (17.5) in a setup script to go from "empty directory" to "fully provisioned project" in two commands.

#### Common Pitfalls
- **File-not-found vs. install failure**: the script correctly distinguishes a missing requirements file (exit code 2) from a failed install (exit code 1) — check the exit code in automation rather than just "non-zero means try again."
- **Long timeouts on large dependency trees**: 900 seconds is generous but not infinite; very large ML/data-science stacks (e.g. installing several GB of GPU-enabled packages) can still exceed it on a slow connection.
- **Stale pins that no longer resolve**: an old requirements file pinning versions no longer available on PyPI (yanked or removed) will fail the whole install — there's no partial-success handling here, unlike `install_packages.py`'s per-package isolation.
- **Untrusted requirements files**: `pip install -r` will happily run setup code from any package listed, including ones from a custom `--extra-index` — never run this against a requirements file from an untrusted source without reviewing it first.

### 17.9 Manage Dependencies with Poetry/Pipenv

#### Use Case
A sysadmin or DevOps engineer supporting multiple teams — some standardized on Poetry, others on Pipenv — wants one automation script that exposes the same actions regardless of which resolver a given project uses, rather than maintaining two separate sets of shell commands.

#### Prerequisites
The script itself needs only the standard library (`argparse`, `shutil`, `subprocess`, `sys`), but it does **not** install Poetry or Pipenv for you — one of them must already be installed and on `PATH` before running it.

```bash
# Install Poetry (recommended: via pipx for isolation from other projects)
pip install pipx
pipx install poetry

# OR install Pipenv
pip install --user pipenv

# Confirm one of them is available on PATH
poetry --version
pipenv --version
```

Both tools work on Linux, macOS, and Windows, though Poetry's installer script differs slightly by OS — see each tool's official documentation if `pipx install poetry` isn't viable in your environment.

#### Cautions
`poetry remove` / `pipenv uninstall` modify the project's lock file immediately — always commit or back up `pyproject.toml`/`poetry.lock` or `Pipfile`/`Pipfile.lock` before running `remove` or `lock` actions in an automated pipeline, since these files define what "the project's dependencies" means for everyone who clones it afterward.

#### Script

```python
#!/usr/bin/env python3
"""
manage_with_poetry.py
--------------------
Thin automation wrapper around modern dependency resolvers (Poetry or
Pipenv). Requires that `poetry` or `pipenv` is already installed and on
PATH -- this script does not install the tool itself, only drives it.

Usage:
    python manage_with_poetry.py --tool poetry --action init
    python manage_with_poetry.py --tool poetry --action add --package requests
    python manage_with_poetry.py --tool poetry --action install
    python manage_with_poetry.py --tool poetry --action lock
    python manage_with_poetry.py --tool pipenv --action install --package flask
    python manage_with_poetry.py --tool pipenv --action lock

Supported actions: init, install, add, remove, lock, show

Exit codes:
    0 -> command ran successfully
    1 -> command failed or tool not found
    2 -> bad arguments
"""

import argparse
import shutil
import subprocess
import sys

ACTIONS = {"init", "install", "add", "remove", "lock", "show"}


def build_command(tool, action, package):
    if tool == "poetry":
        mapping = {
            "init": ["poetry", "init", "--no-interaction"],
            "install": ["poetry", "install"],
            "add": ["poetry", "add", package] if package else None,
            "remove": ["poetry", "remove", package] if package else None,
            "lock": ["poetry", "lock"],
            "show": ["poetry", "show"],
        }
    else:  # pipenv
        mapping = {
            "init": ["pipenv", "--python", "3"],
            "install": ["pipenv", "install", package] if package else ["pipenv", "install"],
            "add": ["pipenv", "install", package] if package else None,
            "remove": ["pipenv", "uninstall", package] if package else None,
            "lock": ["pipenv", "lock"],
            "show": ["pipenv", "graph"],
        }
    return mapping.get(action)


def main():
    parser = argparse.ArgumentParser(description="Drive Poetry or Pipenv for dependency management.")
    parser.add_argument("--tool", choices=["poetry", "pipenv"], required=True)
    parser.add_argument("--action", choices=sorted(ACTIONS), required=True)
    parser.add_argument("--package", help="Package name, required for add/remove")
    args = parser.parse_args()

    if shutil.which(args.tool) is None:
        print(f"[manage] '{args.tool}' is not installed or not on PATH.")
        print(f"         Install it first, e.g.: pip install {args.tool}")
        sys.exit(1)

    if args.action in {"add", "remove"} and not args.package:
        print(f"[manage] --package is required for action '{args.action}'")
        sys.exit(2)

    cmd = build_command(args.tool, args.action, args.package)
    if cmd is None:
        print(f"[manage] Could not build a command for tool={args.tool} action={args.action}")
        sys.exit(2)

    print(f"[manage] Running: {' '.join(cmd)}")
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=600)
    print(result.stdout)
    if result.returncode != 0:
        print(result.stderr.strip()[-1500:])
        sys.exit(1)

    sys.exit(0)


if __name__ == "__main__":
    main()
```

#### How It Works
The script first confirms the chosen tool (`poetry` or `pipenv`) is actually installed via `shutil.which`, then maps a tool-agnostic `--action` onto the correct underlying command for whichever tool you picked — for example, `--action show` becomes `poetry show` for Poetry but `pipenv graph` for Pipenv, since Pipenv has no direct `show` equivalent.

```bash
# Initialize a new Poetry project (non-interactive)
python3 manage_with_poetry.py --tool poetry --action init

# Add a dependency with Poetry
python3 manage_with_poetry.py --tool poetry --action add --package requests

# Install all dependencies from an existing pyproject.toml
python3 manage_with_poetry.py --tool poetry --action install

# Regenerate poetry.lock
python3 manage_with_poetry.py --tool poetry --action lock

# List the dependency tree
python3 manage_with_poetry.py --tool poetry --action show

# Same actions, using Pipenv instead
python3 manage_with_poetry.py --tool pipenv --action install --package flask
python3 manage_with_poetry.py --tool pipenv --action lock
```

#### Variations / Flags
- `--tool {poetry,pipenv}` — pick which resolver drives the action; the same `--action` verbs work for both.
- `--package NAME` — required for `add`/`remove`; optional for Pipenv's `install` (installing with no package name installs everything from the `Pipfile`).
- Wire this into a CI matrix that tests a project against both resolvers if you're evaluating a migration from one to the other.
- Add your own `--dry-run` by printing `cmd` from `build_command` without calling `subprocess.run`, useful before running `remove` or `lock` in a shared environment.

#### Common Pitfalls
- **Tool not installed**: this script deliberately does *not* install Poetry or Pipenv for you — running it before either is on `PATH` fails fast with a clear message rather than installing something unexpected on your behalf.
- **`add`/`remove` without `--package`**: correctly caught and reported with exit code 2 before any subprocess is even attempted.
- **Pipenv's `init` action**: `pipenv --python 3` doesn't behave like a true "init" the way `poetry init` does — it mostly just ensures a `Pipfile` exists for the given Python version; don't expect identical semantics across the two tools just because they share an action name.
- **Long-running `lock` actions**: dependency resolution (especially with Poetry on a large, loosely-pinned dependency set) can be slow; the 600-second timeout is generous but can still be hit on unusually large or conflict-heavy dependency trees.
- **PATH differences across shells/cron**: a tool installed via `pipx` or in a user-local bin directory may be on `PATH` in an interactive shell but not in a cron job's minimal environment — verify `poetry --version` / `pipenv --version` succeed in the *exact* execution context (e.g. inside the crontab, not just your terminal) before relying on this in automation.
